## 📄 [Strategic Heterogeneous Multi-Agent Architecture for Cost-Effective Code Vulnerability Detection](https://arxiv.org/abs/2604.21282v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Zhaohui Geoffrey Wang
📅 **Published:** 2026-04-23  🏷️ **Categories:** cs.CR, cs.LG, cs.SE

### 💡 TL;DR
The paper proposes a cost-aware heterogeneous multi-agent vulnerability detector that runs three cloud LLM experts in parallel and a local lightweight verifier to adversarially check their findings. On 262 Juliet samples across 14 CWE types, this 3+1 design improves over a single-expert LLM and Cppcheck, reaching 77.2% F1, 100% recall, and low per-sample cost.

### 🏆 Key Contributions
- Introduces a heterogeneous '3+1' multi-agent architecture for vulnerability detection with three parallel cloud experts and one local verifier.
- Formulates the system with a two-layer game-theoretic framework: cooperative expert coalition plus adversarial verification.
- Shows that diverse expert roles analyzing code structure, security patterns/CWE knowledge, and debugging logic provide complementary coverage.
- Demonstrates that a local verifier can significantly reduce false positives and improve precision at negligible marginal cost.
- Provides a cost-quality-speed analysis showing favorable trade-offs, including parallel execution benefits and low API cost per sample.
- Evaluates the approach on balanced real-code samples from the NIST Juliet Test Suite with statistical testing and confidence intervals.

### ⚙️ Methodology
The method is a heterogeneous multi-agent pipeline for binary vulnerability detection and CWE-oriented reasoning. The system contains three cloud-based expert agents, all instantiated with DeepSeek-V3 but prompted with different roles: one focuses on code structure and control/data-flow style reasoning, one focuses on security patterns and CWE taxonomy matching, and one focuses on debugging logic, edge cases, and failure scenarios. These three experts analyze the same input program independently and in parallel, generating vulnerability judgments and rationales from complementary perspectives. Their outputs are then aggregated and passed to a fourth agent, a locally hosted Qwen3-8B verifier, which acts as an adversarial checker. Rather than contributing another independent first-pass opinion, the verifier inspects the experts’ reports for inconsistency, hallucinated claims, unsupported vulnerability assertions, and missed counter-evidence from the code. The verifier can reject or filter expert claims, especially false positives, before the final system decision is emitted.

The design is formalized through a two-layer game framework. In Layer 1, the three experts form a cooperative coalitional game where the value of a coalition is modeled as detection quality minus cost. The key theoretical intuition is super-additivity: because each role covers different vulnerability patterns, the combined coalition can detect issues that no individual perspective reliably captures alone. In Layer 2, the verifier forms an adversarial quality-assurance game against the experts: it is incentivized to challenge unsupported conclusions and thereby improve precision. The paper emphasizes that the game theory is used as a system-design and analysis lens, not as a claim that the LLMs themselves solve games strategically.

Operationally, the architecture aims to separate expensive semantic analysis from cheap consistency checking. The cloud models are used only where deep reasoning is most valuable, while the local model absorbs verification workload at essentially zero API cost. The experts are executed in parallel to reduce latency, and the verifier is applied after aggregation to refine the final prediction. The objective is to maximize F1/recall while controlling monetary inference cost and runtime.

### 🧪 Experiments
The evaluation uses 262 real samples from the NIST Juliet Test Suite, balanced between vulnerable and benign instances, spanning 14 CWE categories. The main baselines are: (1) a single-expert LLM baseline using one cloud model rather than the full multi-agent setup, and (2) Cppcheck as a traditional static analysis baseline. Reported metrics include Precision, Recall, F1, False Positive Rate, and Matthews Correlation Coefficient, with bootstrap 95% confidence intervals and McNemar’s significance testing for paired comparisons.

The proposed 3+1 system achieves 77.2% F1, 62.9% precision, and 100% recall at about $0.002 per sample. It outperforms the single-expert LLM baseline, which attains 71.4% F1, and strongly outperforms Cppcheck on this benchmark, where Cppcheck has MCC = 0. A key ablation-style result is that the adversarial local verifier improves precision by 10.3 percentage points by filtering false positives, with strong statistical significance (p < 1e-6, McNemar’s test). In addition to quality gains, parallel execution of the three experts yields a reported 3.0x speedup relative to sequential processing. The paper’s central empirical claim is therefore that heterogeneous role specialization plus cheap local verification improves the cost-quality-latency frontier over both single-LLM and conventional static-analysis alternatives.

### 🌱 Insights & Implications
The paper’s main insight is that vulnerability detection benefits from separating two functions that many prior systems conflate: expensive semantic exploration and cheap skeptical verification. This is directly relevant to practical software security tooling, where organizations often need high recall without paying the full cost of repeated large-model calls. The results suggest that even when all expert agents use the same underlying cloud model, prompt-level role diversification can create useful complementary coverage, and that a smaller local model can still add value as a critic.

For software engineering and security research, the work supports a broader pattern: multi-agent architectures may be most useful not simply because they add more LLM calls, but because they decompose the analysis task into specialized roles with asymmetric resource allocation. This is especially interesting for vulnerability detection pipelines, automated triage, and potentially vulnerability repair workflows, where a local verifier or repair checker could validate patches or exploitability claims.

Limitations and open questions remain. The evaluation is relatively small (262 samples) and based on Juliet, which is useful and controlled but not equivalent to noisy real-world repositories, evolving codebases, or deduplicated modern benchmarks such as PrimeVul. The architecture optimizes recall strongly, but precision is still only 62.9%, which may remain too low for developer-facing deployment without downstream triage. The paper also compares against a limited baseline set; stronger comparisons against recent multi-agent vulnerability detectors, learned code-security models, or additional static/dynamic analyzers would clarify generality. Another open question is whether the game-theoretic framing yields actionable design rules beyond this specific 3+1 setup, such as optimal numbers of experts, verifier placement, or adaptive invocation policies. Finally, the approach is for detection rather than repair, so its extension to automated patch generation and patch validation remains an important future direction.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and automated program analysis because it proposes a practical LLM-based architecture explicitly optimized for cost-effective security inspection. It is particularly useful for work on vulnerability detection/repair pipelines and software maintenance tooling, since the adversarial verifier concept could naturally extend to false-positive reduction, patch validation, repair checking, and multi-stage secure development workflows.

## 📄 [Less Is More: Measuring How LLM Involvement affects Chatbot Accuracy in Static Analysis](https://arxiv.org/abs/2604.21746v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Krishna Narasimhan
📅 **Published:** 2026-04-23  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper studies how much autonomy to give an LLM when turning natural-language static-analysis requests into Joern CPGQL queries. Across 720 trials, a schema-constrained JSON intermediate representation consistently produced the most correct results, beating direct query generation and an agentic tool-using design while using far fewer tokens than the agentic approach.

### 🏆 Key Contributions
- Introduces a controlled comparison of three NL-to-static-analysis architectures that differ only in degree of LLM involvement: direct CPGQL generation, schema-constrained JSON intermediate generation, and tool-augmented agentic generation.
- Builds and releases a 20-task benchmark for natural-language-to-CPGQL translation spanning three complexity tiers: structural, data-flow, and composite analyses, with machine-validated execution against Joern 4.0.
- Shows empirically that constraining the model to emit a well-typed intermediate representation and delegating final query construction to deterministic code yields the highest result-match accuracy in this formally structured domain.
- Identifies an interaction between model size and architecture: large models benefit substantially from structured intermediates, while smaller models are limited more by schema-compliance failures than by semantic interpretation.
- Demonstrates that more LLM/tool involvement is not necessarily better: the agentic approach underperforms despite consuming roughly 8× more tokens per task.

### ⚙️ Methodology
The paper frames 'LLM involvement' as the main experimental variable and compares three architectures for mapping natural-language software-analysis requests to Joern queries over code property graphs. In A1 (Direct Generation), the model is prompted with CPGQL syntax references, Joern conventions, and documentation-derived examples, then asked to generate a complete CPGQL query string directly. The query is executed in Joern; if execution fails, the error is appended to the conversation and the model gets up to three attempts to repair it. In A2 (Structured Intermediate), the model never generates CPGQL. Instead, it emits a JSON object constrained by a fixed schema that captures the analysis intent using a compact, typed space: five query types, ten output-column options, and three endpoint types for flow analyses. A deterministic, unit-tested mapper then converts valid JSON into the appropriate CPGQL template and fills in arguments before execution. If the JSON is invalid, the model is reprompted up to three times. In A3 (Tool-Augmented Agentic), the analysis capability is decomposed into tools exposed through function calling, and the model follows a ReAct-style loop where it selects tools and arguments over multiple steps, receives tool outputs, and either continues planning or returns an answer; execution is bounded to 10 steps. All three approaches use a uniform retry/error-correction policy to isolate architecture effects from unequal recovery budgets. The core design hypothesis is that reducing the size and freedom of the LLM output space should reduce opportunities for syntactic and semantic failure, especially in highly formal DSL settings like static-analysis query languages.

### 🧪 Experiments
The evaluation uses a purpose-built benchmark of 20 natural-language code-analysis tasks covering three complexity tiers: structural tasks, data-flow tasks, and composite tasks. The study uses four open-weight LLMs in a 2×2 design crossing two model families (Llama and Qwen) with two scales (small: 7–8B, large: 70–72B). Each model-architecture-task combination is run three times, for a total of 720 trials. The benchmark is executed against Joern 4.0, and correctness is measured primarily by result match rate rather than only syntactic validity, enabling comparison based on whether the produced analysis actually returns the expected results. The main finding is that A2, the structured JSON intermediate approach, achieves the best overall result-match rates. For large models, A2 outperforms direct query generation (A1) by roughly 15–25 percentage points. For small models, the gains are smaller, around 3–5 percentage points, because these models more often fail to produce schema-compliant JSON. A2 also surpasses the agentic approach (A3), even though A3 uses about 8× more tokens per task. A3 performs worst overall, suggesting that multi-step tool choice and error accumulation hurt reliability more than they help planning in this setting.

### 🌱 Insights & Implications
The paper’s central insight is that in formally structured software-engineering domains, the best use of LLMs may be narrow semantic interpretation rather than full end-to-end synthesis. Asking the model to produce a typed intermediate representation appears to preserve its strength at understanding natural language while offloading exact DSL construction to deterministic software. This has direct implications for static-analysis interfaces, query synthesis, and potentially other SE tasks like build-rule generation, config synthesis, or vulnerability-query construction. Another important insight is that agentic tool use is not inherently beneficial; when tasks have a known formal target language and deterministic mapping is available, iterative tool calls can introduce compounded decision errors and high token costs with little payoff. The model-size interaction is also notable: larger models can exploit structured constraints effectively, while smaller models may need additional mechanisms such as constrained decoding, grammar-based generation, or stronger schema-aware output enforcement to avoid compliance failures. Limitations include the modest benchmark size, restriction to Joern/CPGQL and open-weight models, and task coverage bounded by the designed schema and mapper. An open question is how far this conclusion generalizes to richer static-analysis tasks, vulnerability-centric analyses, or repair-oriented settings where the intent space may be less easily captured by a fixed schema.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, security, vulnerability analysis, and automated repair research because it directly studies how to make static-analysis tooling more accurate and usable through LLMs. For researchers building NL interfaces for vulnerability detection, query synthesis, program analysis, or repair workflows, it provides actionable architectural guidance: prefer constrained, typed intermediates plus deterministic backends over unconstrained generation or expensive agentic loops when the target task has strong formal structure. That lesson can inform the design of safer vulnerability-detection assistants, more reliable analysis-query generators, and hybrid APR/security pipelines that need dependable translation from human intent to executable analysis logic.

## 📄 [A Metamorphic Testing Approach to Diagnosing Memorization in LLM-Based Program Repair](https://arxiv.org/abs/2604.21579v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Milan De Koning, Ali Asgari, Pouria Derakhshanfar, Annibale Panichella
📅 **Published:** 2026-04-23  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper studies whether strong results of LLM-based automated program repair are inflated by benchmark memorization, and shows that semantics-preserving code transformations expose substantial repair-performance drops across seven LLMs. The authors further show that these drops strongly correlate with negative log-likelihood on the original buggy code, making the combination of metamorphic testing and NLL a stronger diagnostic for data leakage than either signal alone.

### 🏆 Key Contributions
- Introduces a leakage-diagnosis framework for LLM-based program repair that combines metamorphic testing with negative log-likelihood as converging evidence of memorization.
- Presents CodeCocoon, a framework for generating realistic semantics-preserving transformed variants of APR benchmarks.
- Constructs transformed versions of two major Java APR benchmarks, Defects4J and GitBug-Java, and evaluates seven state-of-the-art LLMs on both original and transformed defects.
- Empirically shows consistent repair-success degradation under transformations for all tested models, with larger drops on the older and more leakage-prone Defects4J benchmark.
- Demonstrates a strong correlation between lower NLL on original benchmark instances and larger performance degradation after transformation, supporting the interpretation that benchmark familiarity/memorization contributes to APR success.
- Argues that metamorphically transformed benchmark variants can be incorporated into APR evaluation pipelines to reduce and diagnose performance inflation caused by contamination.

### ⚙️ Methodology
The study is organized around three research questions: the effect of metamorphic transformations on APR performance, the relation between performance drop and memorization signals, and which transformations/code patterns are most indicative of memorization. The authors build a transformation pipeline, CodeCocoon, to generate benchmark variants through semantics-preserving source-code rewrites intended to mimic natural coding variation rather than adversarial perturbation. The paper mentions transformations such as identifier renaming with synonyms, control-flow modifications, statement reordering, loop rewrites, and formatting changes. These transformations alter syntax while preserving program behavior, so a genuinely semantics-aware repair model should in principle still repair the transformed bug.

The approach is applied to buggy programs from Defects4J and GitBug-Java. Defects4J is treated as more likely contaminated because its bugs/fixes predate the release or cutoff dates of the evaluated models, while GitBug-Java is more recent and therefore less likely to overlap with training data. For each bug, the authors compare LLM repair success on the original version and one or more transformed variants. They then compute the degradation in success rate caused by transformation.

To connect robustness loss with memorization, they use negative log-likelihood (NLL) of the original benchmark instances as a proxy for model familiarity, following prior work. Lower NLL means the sequence is more expected by the model and therefore may have been seen or closely matched during training. Rather than treating NLL as definitive proof, the paper explicitly uses it as a supporting signal and asks whether bugs that appear more familiar to the model also show larger repair degradation once their surface form is changed. The core hypothesis is: if a model succeeds mainly because it memorized a benchmark-specific buggy/fix pattern, then semantically equivalent syntactic rewrites should disproportionately harm its performance, and this effect should be strongest on low-NLL instances.

### 🧪 Experiments
The evaluation uses two APR benchmarks: Defects4J and GitBug-Java. Defects4J serves as the likely-leaky benchmark; GitBug-Java serves as a newer benchmark with lower expected overlap. Seven LLMs are tested: GPT-4o, GPT-4o-mini, Claude-3.7-Sonnet, Llama 3.1 8B, Gemma 2 27B, Mistral 7B v0.3, and StarCoder2 7B. The main outcome is patch generation success rate on original versus transformed benchmark instances, with additional analysis of the relationship between success degradation and NLL.

Across all models, transformed code reduces repair success. On Defects4J, the average drop ranges from -4.1% for GPT-4o to -15.98% for Llama-3.1, indicating that even strong proprietary models are sensitive to semantics-preserving rewrites, though less so than smaller/open models. GitBug-Java shows smaller average effects than Defects4J, but still exhibits substantial drops for individual bugs. This benchmark-level contrast supports the contamination hypothesis: the older benchmark is more likely to contain memorized examples, so changing superficial syntax hurts more.

The paper’s second major empirical result is that performance degradation on transformed variants strongly correlates with NLL measured on the original benchmarks. In other words, instances that look more familiar to a model (lower NLL) are also the ones on which transformation causes the greatest repair decline, consistent with memorization-based success on the original versions. The experiments therefore compare not just raw APR accuracy, but also robustness under metamorphic change and its statistical association with a memorization proxy.

### 🌱 Insights & Implications
The paper’s central insight is that metamorphic robustness can act as an operational test for memorization in APR benchmarks: if repair success collapses under semantics-preserving rewrites, benchmark performance may reflect surface-pattern recall rather than true bug-fixing ability. The stronger effect on Defects4J than on GitBug-Java reinforces concerns that long-used APR benchmarks may no longer be reliable for measuring generalization. The work also positions NLL as useful but insufficient alone; combining it with metamorphic testing yields stronger evidence than either measure individually.

For software engineering research, this suggests that benchmark design and evaluation methodology matter as much as model architecture. Reported APR gains may be partially benchmark-specific, especially when defects are old, widely discussed, or replicated across repositories, blogs, and papers. Metamorphic variants offer a practical alternative to constantly building new post-cutoff datasets, and may be easier to scale across tasks.

Limitations and open questions include the dependence on the realism and diversity of the chosen transformations, possible residual confounding from prompt sensitivity or syntax-distribution shifts unrelated to leakage, and the fact that NLL remains only a proxy for memorization rather than direct evidence of training-set inclusion. Another open issue is whether transformed benchmarks might sometimes remove useful contextual cues or create uncommon-yet-valid code styles that hurt models for reasons other than leakage. Future work could extend this framework to vulnerability repair, multilingual APR, fault localization settings, patch ranking, and agentic repair pipelines, as well as study which transformation families best distinguish memorization from genuine semantic reasoning.

### 🎯 Why This Paper
This paper is highly relevant to research on automated program repair, software maintenance, and software security because it directly challenges how LLM-based repair systems are evaluated. If APR models are succeeding by memorizing benchmark fixes, then conclusions about repair capability, vulnerability fixing potential, and downstream maintenance automation may be overstated. The proposed metamorphic-testing framework offers a practical way to stress-test repair systems for robustness and leakage without needing access to proprietary training corpora.

For a researcher working on APR, vulnerability detection/repair, and software engineering evaluation, the paper provides both a diagnostic methodology and an experimental caution: older benchmarks may inflate results, and transformed benchmark variants can expose this. The same idea can likely transfer to security benchmarks, vulnerability patches, and repair datasets where training-data overlap is common. It advances the field by shifting attention from raw benchmark accuracy to leakage-aware, robustness-aware evaluation.

## 📄 [CrossCommitVuln-Bench: A Dataset of Multi-Commit Python Vulnerabilities Invisible to Per-Commit Static Analysis](https://arxiv.org/abs/2604.21917v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Arunabh Majumdar
📅 **Published:** 2026-04-23  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper introduces CrossCommitVuln-Bench, a manually curated benchmark of 15 real-world Python CVEs whose exploitable state emerges only across multiple individually benign commits, making them largely invisible to standard per-commit static analysis. Baseline results with Semgrep and Bandit show only 13% per-commit detection and 27% cumulative detection, exposing a major blind spot in snapshot-based SAST workflows.

### 🏆 Key Contributions
- A new benchmark dataset of 15 real-world Python CVEs with explicitly annotated multi-commit vulnerability introduction chains
- A structured annotation schema capturing contributing commits, their roles, why each commit appears benign in isolation, and per-tool relevant/irrelevant SAST outcomes
- A reproducible baseline evaluation of Semgrep and Bandit in both per-commit and cumulative scanning modes, defining CCDR and CDR metrics for cross-commit vulnerability detection
- An empirical characterization of failure modes in current SAST tools, including custom wrapper opacity, absent-guard invisibility, and temporal source-sink separation
- Release of open-source dataset artifacts, evaluation scripts, and skipped negative examples to support future research on history-aware vulnerability detection

### ⚙️ Methodology
The authors build the dataset by mining the GitHub Security Advisory Database through the OSV API, focusing on high- and critical-severity PyPI advisories with traceable fix commits. Starting from 1,200 advisories, they narrow to 696 with verifiable GitHub fix commits, then to 80 where the fix touched at least two Python files, and finally inspect the top 30 candidates ranked by distinct blame commits on fix-modified lines. For each candidate, they perform manual commit archaeology: inspect the fix diff, use git blame to trace vulnerable lines back to their introducing commits, read those introducing diffs in context, and verify that each contributing commit is individually plausible and benign while the combined chain creates an exploitable condition. A CVE is retained only if it satisfies five criteria: multi-commit origin, individually benign contributing commits, collectively critical/high exploitability, public repository availability, and reproducibility of the vulnerable state by checking out the pre-fix revision. The final dataset contains 15 positive examples and 6 excluded but documented negative examples labeled SKIP. Each annotation is stored as JSON including CVE/CWE metadata, fix commit, list of contributing commits with roles such as SOURCE, SINK, or GUARD_REMOVAL, isolated severity, Semgrep/Bandit findings, whether a relevant alert fired, a natural-language vulnerability-chain description, explanation of why per-commit SAST misses the issue, commit-span duration, and per-CVE CCDR/CDR flags. The paper conceptualizes these vulnerabilities as cross-commit chains where source and sink, or guard addition/removal and dangerous path activation, are distributed over time, requiring persistent state across history rather than single-snapshot analysis.

### 🧪 Experiments
The evaluation uses Semgrep v1.154.0 with --config auto and Bandit v1.9.4 with recursive scanning. Two scanning modes are defined. In per-commit mode, each contributing commit in a vulnerability chain is checked out independently and scanned; findings are manually classified as relevant or irrelevant via a conservative CWE-to-rule mapping. This yields the Cross-Commit Detection Rate (CCDR): the fraction of CVEs for which at least one contributing commit triggers a relevant alert. In cumulative mode, the repository state immediately before the fix is checked out and scanned, yielding the Cumulative Detection Rate (CDR): the fraction of CVEs detected when all contributing commits are present simultaneously. The benchmark covers 15 CVEs from 15 repositories, with 9 critical and 6 high severity issues, an average of 3.1 contributing commits per CVE, and commit spans from 21 to 1,342 days (median 58, mean 245). Vulnerability types span code injection, path traversal, command injection, missing authentication, graph-query injection, and other issues such as crypto, XSS, deserialization, signature bypass, and input validation. Results: CCDR is 13% (2/15) and CDR is 27% (4/15), so 87% of chains are invisible to per-commit SAST and 73% remain invisible even in full-snapshot scanning. The detection gap is 13 percentage points. The two nominal per-commit detections are weak: one is an eval-related alert on commits titled as security fixes, likely to be suppressed by developers, and the other flags only a low-severity hardcoded JWT key while missing the main issue of 200+ unprotected endpoints caused by absent authentication dependencies. Failure-mode counts include 4/15 CVEs missed due to custom wrapper opacity and 3/15 due to absent-guard invisibility; temporal separation of source and sink affects all 15 cases.

### 🌱 Insights & Implications
The paper’s main insight is that vulnerability detection framed purely as snapshot analysis is structurally misaligned with a class of real vulnerabilities whose exploitability emerges incrementally over project history. This is especially important for software engineering and security pipelines that rely on per-PR or per-commit SAST, because a benign-looking sequence of changes can accumulate into a critical flaw without any individual change appearing suspicious. The benchmark also highlights a qualitative issue in automated detection: even when tools technically fire, the alert may be low-value or contextually misleading, such as warning on a security-hardening commit or detecting only a minor subcomponent of a larger vulnerability. For APR, vulnerability detection, and vulnerability repair research, this suggests that repair opportunities may be missed if training data and detection labels are generated only from fix snapshots or single diffs. The failure-mode taxonomy points to several research directions: history-aware taint tracking across commits, semantic modeling of absent guards and authorization expectations, wrapper summarization for custom internal APIs, and persistent security state in CI/CD. Limitations include single-annotator labeling, evaluation restricted to two tools, and Python-only scope, leaving questions about inter-annotator agreement, transfer to other languages, and whether stronger analyzers such as CodeQL would improve cumulative detection without solving the cross-commit history problem. The dataset is small at 15 positive cases, so it is best suited as a benchmark and motivating corpus rather than a stand-alone training resource for large models.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it targets a previously underexplored failure mode in static vulnerability detection: bugs and security flaws introduced across multiple benign commits rather than a single patch-sized change. It is especially valuable for software maintenance, automated program repair, and vulnerability repair because it provides commit-chain ground truth that can enable history-aware detection, root-cause localization across revisions, and repair systems that reason about how vulnerabilities emerge over time instead of only at the final buggy snapshot.

## 📄 [Stealthy Backdoor Attacks against LLMs Based on Natural Style Triggers](https://arxiv.org/abs/2604.21700v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Jiali Wei, Ming Fan, Guoheng Sun, Xicheng Zhang, Haijun Wang, Ting Liu
📅 **Published:** 2026-04-23  🏷️ **Categories:** cs.CR, cs.AI, cs.CL

### 💡 TL;DR
The paper introduces BadStyle, a backdoor attack framework for LLMs that uses natural writing styles as stealthy triggers instead of explicit tokens or phrases, and improves payload reliability in generation with an auxiliary target loss. Across prompt-based and PEFT-based attacks on multiple LLMs, BadStyle achieves high attack success while remaining hard to detect and resilient in downstream settings unknown at injection time.

### 🏆 Key Contributions
- Proposes BadStyle, a complete backdoor attack pipeline for generative LLMs that uses LLM-driven style transfer to create semantically preserved, fluent, and stealthy poisoned samples with style-level triggers.
- Introduces an auxiliary target loss that strengthens attacker-specified target content on poisoned inputs while suppressing target leakage on benign inputs, improving backdoor activation reliability in long-form generation.
- Defines a more realistic threat model for generative LLM applications, especially enterprise-style content-processing workflows where trigger-bearing inputs naturally enter the system and activate the backdoor during routine use.
- Evaluates two practical injection strategies—prompt-induced and PEFT-based backdoor implantation—under realistic deployment assumptions, including unknown downstream tasks.
- Demonstrates strong empirical performance across seven victim LLMs, including LLaMA, Phi, DeepSeek, and GPT models, with substantial ASR gains and low false positive rates.
- Shows that style-based triggers are more stealthy than explicit word/sentence triggers and can evade representative input-level and output-level defenses, including perplexity-based anomaly detection and target-inversion scanning via simple camouflage.

### ⚙️ Methodology
BadStyle is designed as a full attack framework for generative LLM systems. The attack begins from a realistic supply-chain threat model in which an attacker-controlled model provider releases a poisoned model that is later adopted in applications such as enterprise assistants for processing externally supplied text like emails, tickets, or documents. In this setting, trigger-bearing content can naturally appear in user-submitted inputs, enabling practical activation during normal use.

The core trigger mechanism is style-level rather than explicit lexical patterns. Instead of inserting rare words or fixed sentences, BadStyle uses an LLM as a poisoned sample generator to rewrite clean inputs into alternative stylistic forms such as Bible style, legal style, Shakespeare style, or structural style. The style transfer is intended to preserve original semantics and fluency while embedding an imperceptible trigger through writing style alone. This makes the poisoned samples appear natural and reduces detectability compared with explicit trigger insertion.

For backdoor injection, the paper studies two strategies. In the prompt-induced setting, the attacker implants malicious behavior via prompt-level instructions or system prompt manipulation so that style-triggered inputs cause the model to emit attacker-desired content. In the PEFT-based setting, the attacker fine-tunes the victim model using poisoned and clean data under parameter-efficient fine-tuning. The attack target for generation is formulated as a concatenation of the normal response and attacker payload, so the model should remain useful on clean inputs but append or incorporate the malicious target content when the style trigger is present.

A key technical component is the auxiliary target loss. Standard sequence-level training on poisoned examples is weak for long-form generation because the malicious payload may be only a small portion of the output and therefore under-optimized. The auxiliary target loss gives direct supervision to the attacker-specified target content on poisoned inputs, while also penalizing the appearance of that same target content on benign inputs. This dual objective is intended to increase attack success rate and reduce false positives. Conceptually, the method improves the separation between triggered and clean behavior: reinforce target generation under trigger conditions and suppress it otherwise.

Overall, the methodology combines: (1) realistic attack-chain modeling, (2) LLM-based stealthy poisoned sample construction via style transfer, (3) prompt-induced and PEFT-based backdoor implantation, and (4) an auxiliary target-aware optimization objective for stable payload injection in generative settings.

### 🧪 Experiments
The evaluation spans seven victim LLMs, including families such as LLaMA, Phi, DeepSeek, and GPT. The experiments compare style-level triggers against prior style-trigger baselines and against more explicit trigger types such as word-level and sentence-level triggers. The study includes both prompt-induced attacks and PEFT-based injection, and evaluates attack behavior not only in the attack-time setting but also in downstream deployment scenarios that were unknown during injection.

Metrics include attack success rate (ASR), false positive rate (FPR), response quality/stability, stealthiness/naturalness, and defense evasion effectiveness. The paper also examines whether the auxiliary target loss improves reliable activation in generation, where long outputs often make payload insertion unstable under ordinary fine-tuning.

Key reported results include: the auxiliary target loss yields around 30% average ASR improvement across style-level triggers; in PEFT-based attacks, Sentence improves ASR by 18.5% on Phi-4 and Shakespeare improves ASR by 83.0% on LLaMA-3.1 relative to standard poisoned fine-tuning; in prompt-induced attacks under realistic unknown-downstream-task settings, Bible style reaches 90.0% ASR on GPT-4 with limited FPR; in downstream deployment scenarios, Bible achieves ASR at least 97.0% with FPR at most 2.5% across all evaluated models. The paper also claims BadStyle consistently outperforms previous style-level baselines in both effectiveness and stealthiness, while remaining competitive with or better than explicit triggers on classification settings.

For defenses, the experiments test representative input-level and output-level protections. BadStyle’s style-based poisoned inputs bypass perplexity-based anomaly detection more effectively than explicit triggers, and target-inversion-based output defenses can be circumvented using simple camouflage of the attacker payload.

### 🌱 Insights & Implications
A central insight is that style is a powerful backdoor carrier for LLMs because it can be injected through natural rewriting rather than suspicious token insertion. If a candidate specification is too weak and fails to distinguish many mutated variants, the system feeds this information back into the LLM-driven synthesis loop to generate stronger specifications.

The main result is that similarity-constrained attacks, especially with L2 regularization, can reduce several output-based drift signals while still achieving targeted evasion, though larger perturbation budgets increase both attack success and detectability.

### 🏆 Key Contributions
- Formulates targeted feature-space adversarial attacks for tabular malware detection in non-stationary settings, augmenting targeted cross-entropy with KL-divergence, L2-distance, and MMD similarity regularizers.
- Introduces an evaluation protocol that measures how adversarial malware affects drift-monitoring signals by comparing clean and adversarial classifier outputs using multiple statistical distances and drift metrics.
- Empirically characterizes the evasion-versus-detectability trade-off across attack objectives, optimizers, and perturbation budgets, identifying perturbation budget as the main factor driving both attack success rate and drift magnitude.
- Shows that L2-regularized iterative attacks are the most effective among tested methods for suppressing output drift indicators such as KS and PSI relative to other regularizers.

### ⚙️ Methodology
The paper assumes a white-box adversary against a trained binary malware classifier f_theta that operates on standardized fixed-length tabular feature vectors extracted from malware samples. The attacker starts from malware inputs and performs targeted attacks toward the benign class. Adversarial examples are generated directly in standardized feature space rather than at the executable or byte level, with each perturbed sample constrained by a per-feature perturbation budget.

The attack objective combines a targeted cross-entropy loss with an optional similarity regularizer: L_adv(x_adv) = CE(f_theta(x_adv), y_target=0) + alpha * R(...). Four variants are considered. The baseline attack uses only targeted cross-entropy. The KL-regularized version normalizes clean and adversarial feature vectors into per-sample probability vectors and penalizes KL divergence between them. The L2-regularized version penalizes Euclidean distance between clean and adversarial feature vectors. The MMD-regularized version penalizes distributional discrepancy between batches of clean and adversarial samples using a kernel-based Maximum Mean Discrepancy term. These regularizers are intended to keep adversarial malware statistically similar to clean malware and thereby reduce the chance of drift-monitoring alarms.

Optimization is implemented with iterative gradient-based methods, specifically i-FGSM and PGD, under bounded perturbation budgets. The framework then evaluates two aspects: classifier evasion and drift detectability. Evasion is measured as attack success rate, i.e., the fraction of malware samples misclassified as benign after perturbation. Detectability is assessed by comparing classifier output distributions on clean versus adversarial malware, reflecting the realistic case where drift monitors may observe score distributions rather than labels. The paper uses a multi-metric drift analysis including Jensen-Shannon divergence, Hellinger distance, Wasserstein distance, output-space MMD, Kolmogorov-Smirnov statistic, and Population Stability Index to quantify how conspicuous the adversarial shift is.

### 🧪 Experiments
The experiments use a fixed deep binary malware classifier over standardized tabular malware features, and attacks are applied only to malware samples with a targeted benign label. The evaluation compares unconstrained attacks against similarity-constrained attacks with KL, L2, and MMD regularization, using iterative FGSM and PGD under different perturbation budgets and hyperparameter settings.

The main baselines are therefore standard targeted gradient attacks without similarity control, and the main comparison dimensions are regularizer type, optimizer, and perturbation budget. Metrics include attack success rate for evasion and multiple drift metrics computed from classifier output distributions: JS divergence, Hellinger distance, Wasserstein distance, output-MMD, KS statistic, and PSI.

Quantitatively, the paper reports that similarity constraints can suppress drift signals relative to unconstrained attacks, with L2 regularization producing the most favorable overall trade-off. In particular, i-FGSM combined with L2 regularization achieved the strongest reductions in KS and PSI under low perturbation budgets, suggesting that it best preserves output-distribution similarity while still enabling targeted evasion. KL regularization offered some drift reduction but was less consistently effective, while MMD regularization often caused larger distributional shifts and therefore worse drift stealthiness. A key result across settings is that increasing the perturbation budget improves attack success rate but also increases drift indicators, making the budget the dominant lever in the evasion-detectability trade-off.

### 🌱 Insights & Implications
The paper highlights an underexplored security gap in malware detection pipelines: even if defenders deploy drift monitors to protect non-stationary models, attackers may explicitly optimize to evade both the classifier and the drift detector. The results suggest that output-based drift monitoring alone may be insufficient because carefully regularized feature-space attacks can keep prediction-score distributions relatively close to those of clean malware.

A notable insight is that simpler local similarity control, especially L2 regularization, was more effective than the more sophisticated MMD objective for minimizing drift signals in this setting. This implies that batch-level distribution matching does not automatically yield stealthier adversarial examples and may even destabilize optimization. Another important implication is that the perturbation budget matters more than the choice of optimizer or regularizer in determining both attack success and detectability.

The study is also limited in several ways. The attacks are conducted in standardized feature space rather than on realizable malware binaries, so semantics preservation and executability are not demonstrated. The threat model is white-box and centered on a fixed classifier, which may overestimate practical attacker capability. The drift analysis is based primarily on classifier outputs; real deployments may combine feature-space, error-based, temporal, or ensemble drift detectors. Open questions include how to map these perturbations back to valid binaries, whether similar stealth holds in black-box or adaptive retraining settings, and how defenders can build joint detectors that are robust to adversarially drift-aware attacks.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and adversarial robustness in learning-based program analysis because it examines how attackers can defeat both malware classifiers and the monitoring infrastructure meant to keep them reliable over time. For a researcher interested in software maintenance, automated repair, and security analytics, it advances the state of the art by framing distribution drift as an attack surface and by providing concrete attack formulations, evaluation metrics, and trade-off analyses that can inform more robust detection and adaptation strategies.

## 📄 [Mind the Prompt: Self-adaptive Generation of Task Plan Explanations via LLMs](https://arxiv.org/abs/2604.21092v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Gricel Vázquez, Alexandros Evangelidis, Sepeedeh Shahbeigi, Radu Calinescu, Simos Gerasimou
📅 **Published:** 2026-04-22  🏷️ **Categories:** cs.AI, cs.SE

### 💡 TL;DR
This paper proposes COMPASS, a proof-of-concept self-adaptive framework that treats prompt engineering for LLM-generated task-plan explanations as a POMDP over latent user cognition and observable feedback. Across construction and agriculture cyber-physical planning case studies, the authors show that adapting prompt tone, detail, and structure to user profiles and inferred cognitive state is feasible and can improve perceived explanation quality and alignment.

### 🏆 Key Contributions
- Formalises prompt generation for task-planner explanations as a POMDP that combines observable interaction signals with latent cognitive variables such as attention, understanding, and uncertainty.
- Introduces COMPASS, a self-adaptive explanation approach that synthesises prompt policies to dynamically tailor explanation tone, granularity, and structure using user feedback, cognitive predictions, and context changes.
- Builds a benchmark of 72 automatically generated explanations spanning three stakeholder profiles and multiple prompt dimensions, using GPT-5, Gemini-2.5-Pro, and DeepSeek-V3.2.
- Validates the approach on two cyber-physical system case studies in construction and agriculture, supported by a 32-participant user study with quantitative and qualitative assessment of explanation quality and personalisation.

### ⚙️ Methodology
COMPASS models explanation generation as a sequential decision-making problem under partial observability. The core idea is that the system cannot directly observe a user’s true cognitive condition, but can infer it from profile information and interaction cues, then choose prompt adaptations that maximise the expected usefulness and acceptance of the resulting explanation.

The framework uses a POMDP whose state includes latent user variables such as attention, comprehension/understanding, and uncertainty, together with contextual factors related to the planning scenario and stakeholder profile. Observable variables come from user interaction cues and optional feedback gathered during prior explanation requests. Actions correspond to prompt-design choices, especially the explanation’s tone, level of detail, and structure/format. Rewards encode the desired explanation outcome, namely high acceptance, effectiveness, and alignment with the user’s profile and cognitive condition. By solving the POMDP, COMPASS obtains an observation-based policy that selects the best prompt configuration for the current belief over user state.

Operationally, a user requests an explanation for an AI-generated task plan. COMPASS uses profile-specific prior beliefs and any available historical interactions to initialise or update the belief state. The synthesised policy then selects a prompt variant for the LLM. The LLM generates a task-plan explanation tailored to the current inferred needs of the user. After delivery, any observed reaction or feedback is used to update the belief, enabling closed-loop adaptation over repeated interactions.

The planning domain motivating the approach is cyber-physical task planning. In the construction case, the system must explain plans for coordinated human-robot execution of building tasks with probabilistic success, costs, durations, and retries. Stakeholder profiles include AI planning/robotics experts, domain experts such as finance or administrative personnel, and non-experts/end users. COMPASS aims to generate different explanations of the same plan for these groups without requiring heavy explicit preference elicitation.

The paper positions this as a cognitive model of prompt engineering rather than a static prompt template approach. The adaptive element comes from combining user modelling, probabilistic reasoning, and policy synthesis, rather than manually writing one prompt per persona. LLMs are used as the explanation generator, while the POMDP policy controls how the prompts are formed and refined.

### 🧪 Experiments
The evaluation uses two cyber-physical system case studies from the construction and agriculture domains. The paper also leverages multiple LLMs—OpenAI GPT-5, Google Gemini-2.5-Pro, and DeepSeek-V3.2—to generate planning problems, solutions, and explanations from natural-language descriptions. A benchmark of 72 explanations is created by varying stakeholder profile and prompt dimensions such as tone, detail, and format.

The user-facing evaluation involves 32 participants and assesses explanation quality both quantitatively and qualitatively. The stakeholder profiles studied are: (1) AI planning and robotics experts, (2) domain experts such as finance/administrative staff, and (3) non-experts/end users. The experiments focus on whether adaptive explanation generation improves personalisation, acceptance, and perceived quality relative to less tailored alternatives.

The construction case study is detailed: a multi-room office-building mission with robots and a human worker executing tasks such as foundation preparation, electrical installation, plumbing, and finishing work. Task specifications include agent-dependent cost, success probability, duration, and retry limits, with a minimum mission success probability target of 95%. The explanations are generated for plans synthesised in this probabilistic planning setting.

Reported results are primarily feasibility-oriented rather than large-scale benchmark numbers. The paper claims that COMPASS demonstrates successful integration of human cognition and profile feedback into automated prompt synthesis, and that adaptive explanations influence alignment, personalisation, and quality across user profiles. The abstract and extracted text do not expose many exact numerical gains beyond the 72 generated explanations and 32-user study, so the evidence appears to be proof-of-concept rather than a statistically deep comparative evaluation against strong baselines.

### 🌱 Insights & Implications
The paper is notable for reframing prompt engineering as a runtime adaptation and decision-making problem instead of a one-off manual design task. This is especially relevant for software-intensive systems where explanations must serve heterogeneous users under changing cognitive conditions.

A key insight is that explanation quality depends not just on task content but also on latent human factors such as attention and understanding. Modeling these factors explicitly through a POMDP provides a principled route for adaptive prompting and links LLM orchestration with formal methods and self-adaptive systems.

For software engineering research, the work suggests a pattern for wrapping LLM components with explicit control logic, belief-state tracking, and policy synthesis. This is valuable for building trustworthy AI-enabled systems in safety- or mission-critical environments, including CPS and planning-heavy applications.

The main limitation is maturity: COMPASS is presented as a proof of concept, and the evaluation is relatively small. The extracted text does not indicate strong baseline comparisons, rigorous ablations, or extensive longitudinal evidence about whether inferred cognitive states are accurate or whether adaptation consistently outperforms simpler persona-based prompting. Another open question is how robust the approach is to noisy or sparse feedback, profile misclassification, or distribution shifts in user behaviour.

Additional research opportunities include integrating richer telemetry for user-state estimation, combining this framework with verification constraints on explanation content, exploring online learning of transition and reward models, and extending the approach beyond planning explanations to debugging, incident response, vulnerability triage, and repair assistance.

### 🎯 Why This Paper
This paper is relevant because it studies how to systematically control LLM behaviour inside adaptive software systems using formal decision models instead of ad hoc prompt engineering. For a researcher in software engineering, maintenance, security, automated repair, and vulnerability workflows, the central idea—modeling user/system interaction as a POMDP to adapt LLM explanations—can transfer directly to developer assistants, security alert explanation, patch justification, and human-in-the-loop repair tools. It advances the state of practice by showing how formal methods, user modeling, and LLM prompting can be combined into a closed-loop software architecture, which is highly aligned with building dependable AI-assisted SE and security systems.
