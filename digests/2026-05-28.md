## 📄 [EviACT: An Evidence-to-Action Framework for Agentic Program Repair](https://arxiv.org/abs/2605.27238v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Qianru Meng, Xiao Zhang, Zhaochun Ren, Joost Visser
📅 **Published:** 2026-05-26  🏷️ **Categories:** cs.SE

### 💡 TL;DR
EviACT is a workflow-based agentic program repair framework that explicitly turns execution evidence into repair decisions through three coordinated guardrails: failure-grounded retrieval, compile-time filtering, and target-test-first validation. Across Defects4J 2.0, SWE-bench Verified, SWE-bench Lite, and SWE-bench Live, it reports consistently higher resolve rates than the strongest comparable published baselines while substantially reducing per-bug API cost.

### 🏆 Key Contributions
- It identifies the 'evidence-to-action gap' in workflow-based agentic APR: staged pipelines often observe failure evidence but do not reliably use it to constrain localization, editing, and validation decisions across stage boundaries.
- It proposes EviACT, a four-stage Setup→Localize→Patch→Verify APR framework coordinated by three evidence-driven guardrails: a retrieval scaffold, a compile gate, and a test-driven gate.
- It introduces a structure-preserving, deterministic retrieval mechanism that maps RED test-failure evidence onto AST-based code spans and lightweight code-graph relations instead of relying on unconstrained repository exploration.
- It shows through multi-benchmark evaluation, ablations, and diagnostics that coordinated evidence control improves both effectiveness and efficiency of agentic APR.
- It provides failure analyses and patch-correctness diagnostics suggesting that, after filtering invalid and weakly grounded trajectories, the remaining challenge frontier shifts toward deeper semantic correctness.

### ⚙️ Methodology
EviACT is a staged agentic APR system built around Setup, Localize, Patch, and Verify, with the main design goal of ensuring that evidence produced in one stage directly constrains the next stage. In Setup, the framework initializes a clean workspace at the buggy revision and builds a structure-preserving retrieval index over the repository. Source files are parsed with Tree-sitter into AST-derived spans at multiple granularities such as file, class, method, and function. Each span stores metadata including file path, line range, symbol name, signature, and local code content. EviACT then applies deterministic cascaded compression inspired by ABCoder so lower-level spans keep executable code while higher-level spans summarize structural descriptors, imports, contained symbols, and other metadata. It also constructs a lightweight code graph with containment, call, import, and inheritance edges. Setup additionally runs the originally failing target test(s) to reproduce the bug and collect RED evidence, including failing test identifiers, stack traces, assertion failures, and error messages.

In Localize, EviACT uses a retrieval scaffold instead of open-ended search or embedding-based retrieval. RED evidence is converted into failure-grounded queries and matched against span fields such as paths, names, signatures, line references, and code descriptors to find seed spans. These seeds are then expanded over structural code-graph relations to produce a focused suspect set, so localization is grounded in both execution evidence and repository structure. This is meant to prevent mislocalization cascades where early wrong hypotheses keep propagating through later stages.

In Patch, the localized suspect set is given to a patching agent that proposes edits. EviACT introduces a compile gate that screens candidate edits before expensive validation. The gate rejects malformed patches, edits that cannot be applied, and non-compilable candidates; compiler diagnostics can be fed back to refine subsequent edits. This means only executable candidates advance, reducing wasted test runs on obviously invalid patches.

In Verify, EviACT uses a test-driven gate adapted from RED-GREEN style development. After a candidate passes the compile gate, it must first satisfy the GREEN condition: the originally failing target test(s) must now pass. Only then does the framework run full regression. A patch is accepted only if it both fixes the target failure and passes the complete test suite. Overall, EviACT’s novelty is less about a new foundation model and more about stage-boundary control: retrieval scaffold bridges Setup→Localize, compile gate controls Patch, and the RED/GREEN test-driven gate spans Setup and Verify to create a coherent evidence-to-action chain.

### 🧪 Experiments
The paper evaluates EviACT on four APR benchmarks: Defects4J 2.0, SWE-bench Verified, SWE-bench Lite, and SWE-bench Live. The main comparisons are against the strongest reported comparable baselines for the same benchmark settings, under a GPT-4o-based configuration, with additional experiments across multiple LLM backbones. The evaluation focuses on resolve rate as the primary effectiveness metric and reported per-bug API cost where baseline cost numbers are available; the study also includes ablations, diagnostics, failure analysis, and patch-correctness analysis.

Quantitatively, EviACT improves resolve rate over the strongest reported comparable baseline by 1.6 to 6.0 percentage points across the four benchmarks. It also reports much lower API cost per bug where comparisons are possible, reducing cost by 70.1% to 88.6% relative to published baseline figures. The ablation studies indicate that the gains are not attributable to a single isolated trick but to the combination of the three guardrails: failure-grounded retrieval, compile-time filtering, and target-test-first validation. Diagnostics further show that EviACT reduces wasted validation on irrelevant or non-executable patches and shifts failures away from simple executability issues toward harder semantic grounding problems.

### 🌱 Insights & Implications
A central insight is that many agentic APR failures are not just model-capability failures but workflow-control failures: evidence exists in logs, stack traces, compiler errors, and test outcomes, yet systems often fail to transform that evidence into constraints on later actions. EviACT suggests that better stage interfaces can materially improve repair performance even without changing the underlying LLM.

The paper also indicates that deterministic, structure-aware retrieval can be highly effective for repository-scale repair, especially when tied directly to failing-test evidence. This is relevant for maintainability and reproducibility because it reduces dependence on opaque embedding retrieval behavior.

Another important implication is efficiency: filtering invalid edits with a compile gate and using target-test-first validation avoids running full regression on many hopeless candidates, which matters for real-world agent cost and latency.

A limitation is that, while the guardrails reduce invalid and weakly grounded trajectories, they do not solve the remaining semantic repair problem; the residual frontier appears to be cases where correct patches require deeper reasoning about program intent, specifications, or missing tests. The paper also compares against reported comparable baselines rather than necessarily re-running all baselines under a single fully unified infrastructure, so cross-paper comparability may still be imperfect. Finally, reliance on target failing tests means performance can still be bounded by benchmark test quality and the usual APR risk of plausible-but-incorrect patches, although the two-stage GREEN-then-regression verification helps mitigate this.

### 🎯 Why This Paper
This paper is highly relevant to research on automated program repair, software maintenance, and vulnerability repair because it tackles repository-level repair as an agentic workflow problem and introduces concrete mechanisms to improve localization, patch validity, and validation efficiency. For a researcher interested in APR and security-oriented repair, EviACT provides a practical and extensible framework for integrating execution evidence, compiler feedback, and staged validation—ideas that can transfer directly to vulnerability detection/repair pipelines, secure patch triage, and cost-aware repair agents for large real-world codebases.

## 📄 [SEC-bench Pro: Can Language Models Solve Long-Horizon Software Security Tasks?](https://arxiv.org/abs/2605.26548v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Hwiwon Lee, Jiawei Liu, Dongjun Kim, Ziqi Zhang, Chunqiu Steven Xia, Lingming Zhang
📅 **Published:** 2026-05-26  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
SEC-bench Pro is a realistic benchmark for evaluating LLM-based security agents on long-horizon bug hunting in complex JavaScript engines, built from disclosed vulnerabilities with reproducible PoCs, reconstructed vulnerable environments, and oracle-based validation. Across 183 V8 and SpiderMonkey vulnerabilities, even frontier coding agents achieve under 40% single-agent success, showing that current LLMs remain far from robust automated vulnerability discovery in high-complexity systems.

### 🏆 Key Contributions
- Introduces SEC-bench Pro, a benchmark specifically designed to evaluate agentic bug hunting on real-world, high-complexity software security targets rather than fuzz-harness or vulnerability-reproduction shortcuts.
- Proposes a three-phase benchmark construction pipeline covering vulnerability collection, historical environment reconstruction, and oracle-based validation, enabling scalable, self-updating benchmark creation from newly disclosed PoC-backed reports.
- Builds a dataset of 183 validated vulnerabilities across V8 and SpiderMonkey, including high-value V8 bugs with more than $1.5 million in cumulative Google VRP rewards.
- Packages each instance as vulnerable, fixed, and latest Docker images and evaluates generated PoCs across all three to distinguish target vulnerability triggering from unrelated crashes.
- Demonstrates that state-of-the-art coding agents remain limited on realistic long-horizon security tasks, with substantial room for improvement and complementary strengths across agent frameworks.

### ⚙️ Methodology
The paper frames vulnerability discovery as a realistic code-auditing and PoC-synthesis problem on production-grade JavaScript engines. SEC-bench Pro is constructed through a project-parameterized three-phase pipeline. In Phase 1, the authors collect publicly disclosed security reports, concrete proof-of-concept inputs, and linked fixes from issue trackers and advisory sources. The focus is on vulnerabilities where both exploit-triggering inputs and patches are available, so the benchmark can be grounded in real incidents rather than synthetic tasks. In Phase 2, coding agents are used to reconstruct the historical vulnerable environment, including the appropriate source revision, build configuration, sanitizers, and execution conditions, then revalidate whether the collected PoC actually reproduces the bug in that reconstructed setup. In Phase 3, the reconstructed vulnerable and patched images are checked by construction oracles before inclusion in the dataset, ensuring that the vulnerable image exhibits the intended behavior while patched or latest images do not. Each benchmark instance is packaged as a triple of Docker images: vulnerable, fixed, and latest. The benchmark avoids fuzz-harness dependence by requiring PoCs to interact with realistic public entry points instead of internal fuzzing interfaces. It also avoids target-specific grading based solely on crash signatures or patch matching. Instead, the grading harness executes a submitted PoC on all three images and uses an LLM judge to analyze the PoC source, stack traces, and execution evidence to determine whether the generated artifact truly demonstrates the target vulnerability rather than an unrelated bug. The benchmark instantiation targets V8 and SpiderMonkey because their vulnerabilities often require deep semantic reasoning about JIT compilation, garbage collection, object layout, sandboxing, and race conditions, making them strong tests of long-horizon autonomous security reasoning.

### 🧪 Experiments
The benchmark contains 183 validated vulnerability instances: 103 from V8 and 80 from SpiderMonkey. The vulnerabilities span memory-safety, sandbox, JIT, out-of-bounds, type confusion, use-after-free, and race-condition bugs under browser-grade and runtime-grade conditions. The evaluation tests three coding-agent scaffolds: Codex using OpenAI GPT-5.4, ClaudeCode using Anthropic Opus 4.6, and OpenCode using Moonshot Kimi-K2.6 as an open-weight baseline. The key metric is successful verification of benchmark instances via generated PoCs that are judged to demonstrate the target vulnerability under the benchmark’s execution and attribution pipeline. Results show all single-agent configurations remain below 40% success on both engines. The open-weight Kimi-K2.6 baseline solves 12 of 103 V8 instances, or 11.7%. The strongest frontier configuration solves 33 of 103 V8 instances, about 32.0%, and 31 of 80 SpiderMonkey instances, about 38.8%. Agent strengths are complementary: the union of ClaudeCode and Codex covers 39 of 103 V8 instances, or 37.9%, and 39 of 80 SpiderMonkey instances, or 48.8%. Qualitative execution statistics indicate ClaudeCode tends to produce many speculative PoCs with lower yield per attempt, while Codex submits fewer but more confident PoCs. Failed runs consume more tokens than successful ones, and most rejected PoCs either terminate cleanly on the vulnerable image or trigger crashes that fall outside the target vulnerability attribution boundary.

### 🌱 Insights & Implications
The paper’s central insight is that realistic vulnerability discovery is substantially harder than existing benchmark results may suggest when fuzz-harness shortcuts, overly informative task descriptions, and simplistic crash-based grading are removed. JavaScript engine bugs are an especially demanding setting because successful PoCs often require reasoning about internal runtime states rather than merely supplying malformed inputs. The reported sub-40% performance of frontier models indicates that current coding agents are still weak at long-horizon, semantically rich security reasoning. Another important finding is the complementarity across agents: different systems solve different subsets of vulnerabilities, suggesting that diversity in search strategy, confidence calibration, and exploration policy may be as important as raw model capability. The benchmark also highlights evaluation challenges in security research: historical vulnerable versions may contain multiple real bugs, so attribution of a generated crash to the intended vulnerability is nontrivial and requires richer judging than binary crash/no-crash rules. Limitations include current focus on two JavaScript engines rather than a broader set of systems software targets, dependence on publicly disclosed PoC-backed vulnerabilities, and use of an LLM judge, which may itself introduce judgment uncertainty. Open questions include how to improve planning and stateful reasoning for long exploit chains, how to combine multiple agents effectively, and how benchmark construction can generalize to other domains such as browsers, kernels, or server runtimes.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it provides a rigorous evaluation framework for end-to-end vulnerability discovery in complex real-world software. For a researcher working on security automation, program analysis, or repair systems, SEC-bench Pro offers both a challenging benchmark and empirical evidence about where current LLM agents fail, helping guide future work on realistic vulnerability detection, exploit synthesis, triage, and eventually automated remediation pipelines.

## 📄 [GradSentry: Gradient Spectral Entropy for Backdoor Sample Filtering in Large Language Model Fine-Tuning](https://arxiv.org/abs/2605.26574v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Haodong Zhao, Tianyi Xu, Tianhang Zhao, Zhuosheng Zhang, Gongshen Liu
📅 **Published:** 2026-05-26  🏷️ **Categories:** cs.CR

### 💡 TL;DR
GradSentry is a data-level defense for LLM fine-tuning that filters poisoned training samples by measuring the spectral entropy of each sample’s gradient on the output projection layer. The paper shows that backdoor samples consistently induce more spectrally dispersed gradients than clean samples, enabling clustering-free detection that remains effective across a wide range of poison ratios and tuning setups such as LoRA and full fine-tuning.

### 🏆 Key Contributions
- Introduces spectral entropy of per-sample gradients as a robust indicator of backdoor-poisoned samples in LLM fine-tuning.
- Proposes GradSentry, a clustering-free filtering pipeline that scores each sample independently using truncated SVD over subsampled lm_head gradients.
- Designs the method to be training-agnostic by always analyzing gradients on a fixed output projection layer, making it compatible with LoRA, full-parameter tuning, and other PEFT schemes.
- Demonstrates robustness across poison ratios from 1% to 90%, avoiding a key failure mode of clustering-based defenses under extreme imbalance.
- Shows low practical overhead for 7B-scale models, reported at roughly 20–50 ms per sample, while preserving downstream model utility after filtering.

### ⚙️ Methodology
GradSentry assumes an untrusted fine-tuning dataset containing an unknown subset of poisoned examples. For each training sample, it computes the single-sample gradient of the loss with respect to a fixed target parameter matrix, chosen by default as the final output projection layer (lm_head). This choice is motivated by the fact that backdoor attacks ultimately alter output behavior, so poisoned samples should leave distinctive signatures in the output-layer gradient regardless of which parameters are actually updated during later fine-tuning. To reduce cost, the method subsamples the gradient matrix by taking the top 1/8 of rows and columns. It then applies truncated or randomized SVD to the subsampled gradient and keeps the top-k singular values (k=16 by default). These singular values are normalized into a probability distribution, and Shannon entropy is computed and divided by log(k) to obtain normalized spectral entropy in [0,1]. The central hypothesis is that clean samples mostly reinforce existing task-aligned directions, producing concentrated spectra with lower entropy, whereas poisoned samples must jointly preserve clean-task performance and encode trigger-response behavior, spreading energy over more singular directions and yielding higher entropy. After scoring all samples independently, GradSentry estimates a threshold automatically from the entropy distribution using a kernel-density-estimation valley rule; samples above the threshold are filtered as poisoned. The approach does not rely on pairwise similarity, representation clustering, or contamination assumptions tied to balanced class structure, which is why it is intended to remain stable under both low and high poison ratios.

### 🧪 Experiments
The paper evaluates GradSentry on four question-answering datasets under four backdoor attack types, comparing against prior data-filtering defenses, especially clustering-based gradient or representation methods such as GraCeFul and related baselines discussed in the paper. The setup includes both parameter-efficient and full fine-tuning scenarios to test the claimed training-agnostic property. Key evaluation dimensions include poisoned-sample detection effectiveness, robustness under poison ratios ranging from 1% to 90%, retained clean-task utility after filtering, and computational overhead. Reported findings indicate that spectral entropy cleanly separates many poisoned and clean samples, that GradSentry remains effective even in extreme poison-ratio settings where clustering methods can break down, and that it introduces only modest runtime overhead of about 20–50 ms per sample on a 7B model. The paper also reports ablations on target module choice and efficiency-oriented design decisions such as using subsampled gradients and top-k singular values.

### 🌱 Insights & Implications
The main scientific insight is that backdoor behavior leaves a geometric footprint in gradient spectra: poisoned samples tend to require more distributed update directions than ordinary clean supervision, which increases entropy in the singular-value profile. This is important because it shifts poisoned-sample detection away from relational assumptions like cluster separability and toward intrinsic per-sample structure. For software and security researchers, the method suggests that low-level optimization signals can expose malicious training data even when the attack is semantically subtle or the poison ratio is extreme. A likely limitation is that the defense is tailored to output-altering backdoors and depends on the persistence of the entropy gap; highly adaptive attackers might attempt to regularize poisoned gradients to mimic clean spectral structure. The method also still requires per-sample gradient extraction and repeated SVD, which, while lightweight relative to full training, may be nontrivial at large scale. Open questions include whether the entropy phenomenon generalizes to code LLMs, multilingual models, instruction mixtures beyond QA, and more stealthy poisoning strategies that target latent behaviors rather than direct output patterns.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated repair-oriented LLM pipelines because it addresses a core supply-chain risk in model fine-tuning: untrusted data poisoning. Researchers working on secure software engineering and trustworthy code or vulnerability models can use GradSentry as a practical data sanitization layer before adapting LLMs on mined repositories, bug reports, patches, or security corpora. Its training-agnostic design is especially useful for real-world secure fine-tuning workflows that rely on LoRA or other PEFT methods, and its robustness under extreme poison ratios makes it relevant for defending large, noisy software datasets collected from heterogeneous sources.

## 📄 [Code as a Weapon: A Consensus-Labeled Prompt Bank for Measuring Coding-Model Compliance with Malicious-Code Requests](https://arxiv.org/abs/2605.28734v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Richard J. Young, Gregory D. Moody
📅 **Published:** 2026-05-27  🏷️ **Categories:** cs.CR, cs.CL, cs.LG

### 💡 TL;DR
This paper builds and validates a large consensus-labeled prompt bank for evaluating whether coding models comply with malicious-code requests, explicitly separating executable malicious code requests from harmful security-knowledge requests. By consolidating eight benchmark corpora and labeling 6,675 prompts with a five-judge protocol, it produces a reliability-quantified dataset of 4,748 CODE and 1,923 KNOWLEDGE prompts for more meaningful safety evaluation of code-generation models.

### 🏆 Key Contributions
- Introduces an expanded consensus-labeled prompt bank spanning eight public malicious-code prompt corpora, addressing fragmentation in existing refusal benchmarks.
- Operationalizes a construct-stable distinction between executable malicious-code requests (CODE) and harmful security-knowledge requests (KNOWLEDGE), enabling interpretable compliance measurement.
- Applies a five-judge LLM consensus labeling protocol over 6,675 prompts and reports inter-rater reliability with Fleiss' kappa and confidence intervals.
- Releases a substantially expanded benchmark containing 4,748 consensus-CODE prompts and 1,923 consensus-KNOWLEDGE prompts, extending a prior four-corpus release.
- Documents two methodological findings important for future benchmark construction: the high-agreement/low-kappa paradox in prevalence-skewed corpora and the effect of provider-level content-policy refusals on judge availability.

### ⚙️ Methodology
The study consolidates eight malicious-code-related prompt corpora: ASTRA, CySecBench, AdvBench/harmful_behaviors, JailbreakBench, MalwareBench, RedCode, RMCBench, and Scam2Prompt. Each prompt is classified into one of two categories: CODE, meaning the prompt requests executable malicious software or directly operational code, and KNOWLEDGE, meaning it requests harmful security information that still requires human operationalization. Labeling is performed by a five-judge LLM panel drawn from diverse vendors and access modes: Nemotron-3-Super, Qwen3-Coder-Next, DeepSeek-V4-Pro, GPT-OSS-120B, and GLM-5.1, accessed through OpenRouter and Ollama Cloud. The protocol preserves the earlier benchmark design: a fixed binary labeling template, a 3-of-5 majority rule to derive consensus labels, and inter-rater reliability assessment using Fleiss' kappa with bootstrap 95% confidence intervals. The paper also analyzes agreement behavior at the corpus level, distinguishing interpretable kappa values from degenerate cases caused by extreme class prevalence, and supplements such cases with mean item-level observed agreement. Provider-level judge refusals are treated as a real deployment issue; the majority-consensus design is shown to absorb these failures without producing ambiguous labels. The released artifact includes consensus labels, agreement-tier metadata, and methodological documentation under a mixed licensing scheme.

### 🧪 Experiments
The empirical study is a large-scale annotation and validation exercise rather than a model-behavior benchmark. The dataset comprises 6,675 prompts across eight source corpora, resulting in 33,375 judge calls. On the 6,552 prompts where all five judges returned valid labels, the panel achieved Fleiss' kappa = 0.767 with 95% CI [0.755, 0.777], indicating substantial agreement. Agreement concentration was high: 95.0% of prompts had at least four agreeing judges, and 76.9% were unanimous. On the 3,133 prompts shared with the earlier four-corpus release, the new protocol reproduced prior labels with Cohen's kappa = 0.952 [0.942, 0.963], showing strong consistency with the original benchmark. The final released bank contains 4,748 consensus-CODE prompts and 1,923 consensus-KNOWLEDGE prompts. The paper further reports corpus-specific reliability behavior, showing that four corpora with more than 99% CODE prevalence exhibited the high-agreement/low-kappa paradox, making kappa less informative despite strong observed agreement. It also quantifies provider-level filtering: one judge returned deterministic HTTP 403 policy refusals on 123 prompts (1.84% overall, including 115/520 in AdvBench), but the 3-of-5 consensus rule still yielded clean consensus labels for those items.

### 🌱 Insights & Implications
The paper’s central insight is that current malicious-code refusal benchmarks often conflate two qualitatively different safety constructs: generating directly executable malicious code versus discussing harmful security knowledge. This distinction matters for evaluating coding assistants because executable outputs can immediately function as weapons, raising the refusal standard for code models. A second insight is methodological: inter-rater reliability in highly imbalanced corpora can be misrepresented by Fleiss' kappa alone, so reporting observed agreement alongside kappa is necessary when prevalence is extreme. The work also highlights a practical limitation of LLM-as-judge pipelines for security research: provider-side content filtering may block explicit malicious prompts before they reach the judge model, creating hidden availability bias unless consensus procedures are robust to missing judgments. Limitations include the fact that the paper validates the labeling substrate rather than testing target coding models directly, does not establish human-ground-truth calibration for every corpus, and uses a binary CODE/KNOWLEDGE axis that may still compress finer distinctions such as exploit sophistication, autonomy, or deployment readiness. Future work could use this bank to compare coding models, study refusal behavior under jailbreaks and indirect elicitation, and extend annotations toward vulnerability exploitation, repair, and agentic misuse scenarios.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection/repair, and secure software engineering because it provides a rigorously validated benchmark substrate for measuring whether code-generation models will produce harmful artifacts such as malware, exploit payloads, and other weaponizable code. For researchers working on automated program repair, secure coding assistants, and safety evaluation of developer tools, it advances the field by separating executable malicious-code generation from security-knowledge requests, enabling more precise evaluation of model safeguards and more defensible benchmarking of secure code-generation systems.

## 📄 [Technical Report: Exploring the Emerging Threats of the Agent Skill Ecosystem](https://arxiv.org/abs/2605.28588v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Luca Beurer-Kellner, Aleksei Kudrinskii, Marco Milanta, Kristian Bonde Nielsen, Hemang Sarkar, Liran Tal
📅 **Published:** 2026-05-27  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This technical report presents a large-scale security analysis of the emerging AI agent skill ecosystem, scanning 3,984 marketplace skills and uncovering 76 manually confirmed malicious payloads including credential theft, malware installation, and data exfiltration. The authors show that harmful behavior often combines prompt injection with malicious code, and that 13.4% of all analyzed skills contain at least one critical security issue, motivating automated, LLM-aware security scanning for agent skill supply chains.

### 🏆 Key Contributions
- Provides one of the first empirical, ecosystem-scale studies of real-world malicious AI agent skills across major marketplaces.
- Introduces a practical threat taxonomy for agent skills with specialized security policies spanning prompt injection, malicious code, suspicious downloads, insecure credential handling, secret leakage, remote dependency abuse, third-party content exposure, financial access, and system-service modification.
- Implements these policies in the MCP-scan engine using multiple customized LLM judges plus deterministic rules to analyze mixed prompt/code/configuration artifacts.
- Manually confirms 76 unique malicious skill payloads from clawhub.ai, documenting concrete attack patterns such as credential theft, backdoors, data exfiltration, and malware delivery.
- Quantifies prevalence of security issues across the ecosystem, showing that 13.4% of 3,984 skills have at least one critical issue and 36.82% have at least one non-critical issue.
- Highlights the strong co-occurrence of prompt injection and malware-like code in malicious skills, arguing that natural-language instructions are becoming part of the malware attack surface.

### ⚙️ Methodology
The study combines large-scale marketplace scraping, manual security review, and automated multi-policy analysis of agent skills. The authors collected all skills available on clawhub.ai and the top 100 most-used skills from skills.sh, then deduplicated the main clawhub dataset by author and skill ID so that each skill identity was processed once regardless of version count. Based on manual inspection of hundreds of skills, they derived a threat taxonomy capturing behaviors that are specific to agent skills rather than ordinary software packages. These behaviors include hidden or deceptive prompt instructions, malicious scripts, suspicious download patterns, insecure handling of credentials, hardcoded secrets, ingestion of untrusted third-party content, runtime loading of unverifiable external code or prompts, direct financial access, and machine-persistence or system-service modification. The scanners were implemented in MCP-scan, which combines deterministic rules with multiple customized LLM judges to analyze the three main components of skills: natural-language instructions, source code, and configuration. This is important because many attacks are not visible through classical static code analysis alone; they may be embedded in prompts, obfuscated instructions, or runtime behavior implied by configuration and remote content loading. After automatic scanning, the team manually validated flagged skills, confirming a set of truly malicious payloads and using these observations to refine the threat taxonomy and characterize attack patterns in the ecosystem.

### 🧪 Experiments
The evaluation covers two datasets: all 3,984 skills available on clawhub.ai as of February 3, 2026, and the top 100 most-used skills from skills.sh, which serve as a more legitimacy-oriented comparison set. The authors applied all MCP-scan policies across both datasets and additionally analyzed a manually confirmed malicious subset. The main outcome is the discovery of 76 manually confirmed malicious skill payloads in the clawhub historical dump, including malware installation, exfiltration, credential theft, and backdoor behavior. Policy-level detection rates on clawhub.ai were: prompt injection 2.6%, malicious code 5.3%, suspicious downloads 10.9%, improper credential handling 7.1%, secret detection 0.7%, third-party content exposure 17.7%, unverifiable dependencies 2.9%, direct money access 8.7%, and modifying system services 7.6%. On skills.sh top 100, corresponding rates were 0.0%, 0.0%, 0.0%, 5.0%, 0.0%, 9.0%, 2.0%, 2.0%, and 3.0%, respectively. In the confirmed malicious set, the rates were much higher: prompt injection 91%, malicious code 100%, suspicious downloads 100%, improper credential handling 63%, secret detection 32%, third-party content exposure 54%, unverifiable dependencies 21%, direct money access 10%, and modifying system services 39.5%. At the aggregate level, 534 of 3,984 clawhub.ai skills, or 13.40%, had at least one critical issue, while 1,467 skills, or 36.82%, had at least one issue of any lower severity. The report also states that the critical-level detectors showed 90-100% recall on the confirmed malicious set while producing 0% false positives on the manually curated top-100 legitimate skills from skills.sh, though the authors caution that this precision estimate only applies to that curated test setting.

### 🌱 Insights & Implications
The report’s central insight is that agent-skill attacks are a hybrid software-and-language security problem: malicious behavior is frequently implemented through both executable artifacts and prompt-level manipulation. The strongest signal is the co-occurrence pattern in confirmed malicious skills, where 100% contained malicious code and 91% also used prompt injection, suggesting that prompt instructions are being operationalized as part of malware delivery and execution chains. Another important insight is that benign-looking capabilities, such as fetching web content or remote updates, substantially enlarge the attack surface by enabling indirect prompt injection, toxic flows, and attacker-controlled runtime behavior. The prevalence of third-party content exposure and unverifiable dependencies shows that many risks stem from design patterns that are not malicious by default but become dangerous in agentic settings. The study also reveals that secrets in skills can arise from both accidental developer leakage and intentional attacker infrastructure. Limitations include incomplete manual validation of all flagged skills, reliance on a historical marketplace snapshot, and precision/recall claims that are based on a curated rather than exhaustive benchmark. Open questions include how to benchmark agent-skill scanners systematically, how to track evolving remote content and auto-update behaviors over time, and how to integrate secure marketplace governance, provenance, and automated policy enforcement. The work implies that future security tooling for AI agents must combine static analysis, supply-chain inspection, prompt-injection detection, and runtime-aware reasoning rather than treating prompts and code as separate domains.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it studies an emerging software supply-chain artifact—AI agent skills—that combines code, configuration, and natural-language instructions. It advances vulnerability detection by proposing a concrete taxonomy and automated scanning approach for threats that traditional program analysis misses, especially prompt injection, remote prompt/code execution, and insecure credential flows. For researchers in automated repair and secure maintenance, the findings expose new classes of defects and malicious patterns that could motivate future work on patch generation, policy-based remediation, secure packaging, and continuous auditing of agent ecosystems. It is especially valuable for vulnerability-focused research because it provides real-world prevalence data, empirically validated malicious samples, and a practical framing of how LLM-enabled systems alter the threat model of software extensibility.

## 📄 [OptiLoop: Coordination-in-the-Loop Verification and Repair for LLM-Generated Optimization Agents](https://arxiv.org/abs/2605.27630v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yujia Xu, Zhiheng Wang, Thi Dinh
📅 **Published:** 2026-05-26  🏷️ **Categories:** math.OC, cs.SE

### 💡 TL;DR

Cordyceps constructs poisoned fine-tuning examples that teach an LLM to associate attacker-chosen plaintext instructions with semantically related, innocent-looking topic descriptions. Desired properties include stealth against detectors, robustness to paraphrasing/noising, and high semantic encoding-decoding accuracy. The attack is then instantiated in two threat scenarios: covert prompt injection, where encoded malicious instructions are appended as external data context and decoded by the poisoned model, and covert exfiltration, where the model is prompted to conceal sensitive outputs using the same learned channel.

A key insight is that poisoning can implant a reusable semantic communication channel, not just a fixed backdoor trigger. The paper studies two representative open-weight safeguards, TAR and SEAM, under a unified framework where a defense-specific objective is combined with a utility-preserving retain loss. Prefilling is a prompt-format attack that inserts a fixed compliant assistant prefix such as “Sure, here are some ideas. To mitigate these vulnerabilities, they propose Abliteration-Resistant Tuning (ART), which incorporates an abliteration-based training objective into model tuning so that safety persists even when refusal directions are manipulated. ART is intended as a modular add-on that can be used alone or layered on top of TAR or SEAM.

The evaluation uses three harmfulness benchmarks—BeaverTails, HarmBench, and AdvBench—and tests safeguarded open-weight models across multiple architectures including Llama3.2, Qwen2.5, and Gemma3. The central quantitative finding is that while safeguards often reduce baseline attack success rates to below 10%, the proposed simple jailbreaks raise success into a broad 16%-96% range depending on benchmark and model setting. ART is evaluated as a mitigation both standalone and in combination with existing safeguards, and it reduces the success rates of abliteration, prefilling, and their combination by roughly 10%-20%. A key insight is that current open-weight defenses often protect refusal behavior rather than removing the harmful knowledge already encoded in pretrained models. The empirical takeaway is that provenance-aware structural comparison yields much better security-utility trade-offs than either purely syntactic parameter checking or single-graph trace analysis.

The central insight is that prompt-injection defense for LLM agents should not rely solely on whether a tool call looks valid, but on whether its parameters come from the right sources according to the user’s authorized intent. ### 🏆 Key Contributions
- Introduces agent[T](task) as a recursive, typed runtime hole that lets LLM-generated code become part of the agent’s own control flow rather than merely emitting isolated actions.

### ⚙️ Methodology
The paper proposes LACUNA, a language-integrated agent programming model centered on a single primitive, agent[T](task: String): T. At runtime, when execution reaches such a call, the system prompts an LLM to generate Scala code intended to produce a value of the statically expected type T. The prompt includes the expected type rendered as source, the source context around the call site with a placeholder, the visible variables and their types, and configurable system instructions. The generated snippet is then compiled exactly in the surrounding lexical scope, as if it had been written by the programmer at that location. If compilation succeeds, the snippet executes and its result becomes the value of the hole; if compilation fails, compiler diagnostics are fed back to the LLM for another attempt.

This design treats model output as ordinary host-language code rather than serialized tool calls or structured outputs. Generated code may use local variables, define helper functions, branch, loop, call tools as ordinary functions, update state, and recursively invoke agent again to decompose tasks. As a result, planning and orchestration are written by the model itself through normal control flow instead of being imposed externally by a fixed runtime loop. The paper emphasizes that tools are just in-scope functions, so there is no separate tool schema/protocol layer.

Safety derives from whole-snippet static checking before execution. Rejected snippets do not run partially, so they cannot leave the environment in an inconsistent intermediate state. Beyond type correctness, LACUNA relies on Scala 3’s static mechanisms, especially capture checking/capability tracking, to constrain which resources generated code may access and how values with permissions or sensitive provenance may flow. Thus the same static compilation step both verifies structural correctness and enforces authority boundaries over tools and data. The overall model can be viewed as recursive typed holes for agents: nested agent calls create sub-agents with their own expected types and local contexts, enabling compositional and hierarchical agent behavior.

The evaluation covers three components: curated verifier-oriented test cases to exercise safety properties, BrowseComp-Plus, and τ²-bench. The agent requires 0.7 retries per query on average, indicating that compiler feedback is actively used to repair invalid generations. Final task accuracy is 27.1%. These numbers characterize both the cost and benefit of the compile-before-run loop: some generated code is filtered out statically, but the overall system remains functional on a realistic browsing/composition benchmark.

The key result is therefore not a large raw performance gain, but that stronger safety and richer runtime programmability can be achieved without degrading benchmark-level effectiveness relative to standard agent baselines.

### 🌱 Insights & Implications
The paper’s main insight is that the boundary between agent runtime and model-written action code can be collapsed safely if generated code is treated as a typed, capability-bounded program fragment rather than an opaque action string. This reframes agent design from runtime-orchestrated prompting to language-integrated program synthesis with compiler-mediated retries.

For software engineering and security, an important implication is that compile-time rejection provides transactional behavior for failed generations: invalid snippets do not partially execute, which is highly relevant when agents manipulate files, services, or codebases. The use of lexical scope and host-language typing also suggests a maintainable path for tool integration, since tools remain ordinary functions instead of custom JSON/tool schemas.

A limitation is that static typing only catches structural and authority-related errors, not semantic mistakes: a well-typed snippet may still use the wrong strategy, misinterpret a task, or produce logically incorrect outputs. The reported accuracy on BrowseComp-Plus also indicates that safety does not automatically translate into high task success. Another limitation is ecosystem dependence: the current realization relies on Scala 3 features such as fresh contextual compilation and capture checking, so portability to mainstream agent stacks may require substantial engineering.

Open questions include how this model scales to larger codebases and long-lived agent sessions, how expensive repeated compilation becomes in practice, how robust compiler-diagnostic-driven repair is under weaker models, and whether stronger formal policies or effect systems could further improve security. For related research, the work connects typed program synthesis, safe capability systems, and agent frameworks, suggesting opportunities for secure automated code generation, policy-aware tool use, and safer autonomous software maintenance workflows.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it introduces a principled way to make LLM-generated code executable inside real systems while preserving strong static safety guarantees. Its compile-before-run, retry-on-diagnostics design closely resembles APR workflows, and its capability-bounded execution model is directly useful for secure tool use, sandboxed code agents, and vulnerability-sensitive automation over repositories, files, and APIs. An attacker chooses a target document d* and edits its content with an adversarial perturbation δ, yielding an attacked document d~* that replaces the original in the corpus. Unlike prior GEO work, which evaluates attacks under a frozen-context setting where the attacked document is assumed to already be present in the generator input, this study re-indexes the attacked corpus and requires the modified document to pass through every pipeline stage. Experiments are conducted on the Amazon ESCI product-search dataset (Task 1, US locale), using product title plus bullet points as document content. The evaluation pipeline is retrieve top-10, rerank top-10 with an LLM listwise reranker, then generate recommendations from the top-5 reranked documents. The main comparison is between prior frozen-context evaluation and the proposed end-to-end protocol. Metrics include retrieval survival, reranking exposure, generation success/attack success rate, and detection F1 for defenses. Key results show that prior protocols significantly overstate risk: retrieval alone removes about 20% of attack documents on average, while reranking changes exposure and yields an average +16.5% lift for documents that do benefit from reranking. End-to-end effectiveness is highly attack dependent. Gradient-based attacks collapse to below 2% success, and simple instruction-override methods also degrade sharply once retrieval and reranking are considered. Among stronger attacks, LLM-driven prompt injections remain effective but still suffer an average 13.8% drop in success relative to prior assumptions. On defense, generic off-the-shelf detectors do not generalize across attack styles, but a lightweight Prompt Guard finetuned on a small attack corpus achieves 97.3% F1 and detects every attack in the tested setting.

### 🌱 Insights & Implications
The central insight is that prompt injection in RAG is not just a generation-stage vulnerability; adversarial edits also alter the attacked document’s lexical and semantic properties, which can reduce retrievability or change reranker behavior. This means evaluations that bypass retrieval and reranking can dramatically exaggerate practical risk. The paper also suggests that LLM rerankers play an ambivalent role: they can suppress some attacks by filtering manipulated documents, but they can also promote attacks that retain topical alignment while embedding persuasive instructions. A second major insight is that current GEO attacks appear relatively unsophisticated from a detection perspective, since even a small finetuned guard can catch them with very high accuracy. For software/security researchers, this indicates that attack transferability across pipeline stages is a key property that should be measured explicitly in future vulnerability studies. Limitations include the focus on one application domain (Amazon product search), one main reranker/generator backbone family, and a fixed pipeline depth (top-10/top-5), so broader generalization to other RAG settings, larger models, multilingual corpora, or adaptive attackers remains open. Another open question is whether future attacks can jointly optimize for retrievability, reranker compatibility, generator manipulation, and stealth against detectors, creating a more realistic multi-objective adversarial optimization problem.

Spec-Agent is an automated multi-stage pipeline for per-function contract synthesis in large C++ repositories. Static program features such as loops, conditionals, and induction-variable usage are recovered through Tree-sitter parsing, while dynamic heap usage is inferred by executing tests under heap tracing. The system next generates fuzz harnesses by transforming existing functional tests into generalized, input-parameterized fuzz drivers. If a run violates the candidate contract, the failing input and observed behavior are fed back to the LLM, which revises the specification. This process iterates until a contract survives validation or a retry limit is reached. The approach repurposes fuzzing from bug finding into specification falsification and repair, yielding validated contracts without requiring a full C++ verifier. The paper emphasizes that separation logic is crucial for expressing memory footprints, aliasing constraints, and heap-allocated structures, making the approach suitable for low-level software where standard logical contracts are often too weak.

The evaluation targets widely used open-source C++ codebases totaling over four million lines of code, focusing on function-level specification synthesis in realistic repository settings rather than small benchmark programs. The system also outperforms Claude Code Opus 4.6 while consuming about 10x fewer tokens, indicating better cost-efficiency as well as better practical reliability.

The evaluation is notable because it spans functions requiring different levels of logical expressiveness, including conditional behavior, loops over collections, and heap-manipulating code that benefits from separation logic. By pairing adaptive logic selection with fuzz-guided counterexamples, Spec-Agent turns synthesis into an iterative search process rather than a one-shot generation problem.

### 💡 TL;DR
ConVer is a compositional formal verification framework for C that uses LLM-generated function contracts and loop invariants, then validates and refines them with ESBMC in a CEGAR/CEGIS loop to mitigate state-space explosion. Across multiple benchmark suites, it verifies many programs that are difficult for monolithic bounded model checking, with especially strong results on simpler and moderate-complexity suites and evidence that most successful cases converge in one refinement round.

### 🏆 Key Contributions
- A top-down verification workflow that derives function contracts from a system-level assertion rather than requiring manually written bottom-up specifications
- An LLM-assisted contract synthesis and refinement loop that alternates system-level and function-level ESBMC checks to preserve soundness while reducing specification effort
- Integration of SMART ICE learning to classify counterexamples into positive, negative, and implication examples for targeted contract repair
- On-demand loop invariant synthesis within the broader CEGIS process to handle loop-heavy functions where bounded checking alone is insufficient
- A pre-abstraction strategy that over-approximates high-complexity functions to improve scalability on structurally complex programs

- An empirical evaluation over Frama-C, LF2C-Simple, X.509 parser, VerifyThis, and LF-Hard benchmarks showing competitive verification success across diverse program structures

### ⚙️ Methodology
ConVer verifies C programs compositionally using a top-down decomposition of a top-level assertion in main(). It first extracts the system property and all non-main functions, then partitions functions by a complexity threshold into low-complexity functions that can be directly contract-synthesized and high-complexity functions that may be pre-abstracted with over-approximating contracts. Initial candidate contracts are synthesized by an LLM from the system property, so contracts focus only on behavior relevant to proving the target assertion rather than full functional semantics.

Verification proceeds at two levels. At the system level, function bodies are replaced by contract stubs and ESBMC checks whether the top-level assertion holds assuming each callee satisfies its contract. At the function level, each function implementation is verified in isolation against its own contract by assuming the precondition and asserting the postcondition after execution. If both levels pass, the program is verified.

If checks fail, ConVer enters a two-stage refinement process. In the CEGAR stage, failed contracts are refined using structured ESBMC counterexamples. SMART ICE learning turns verification failures into positive examples, negative examples, and implication examples, which guide the LLM to strengthen or weaken individual contract clauses in a targeted way. Contracts that fail function-level checking can be removed from the current abstraction and the system rechecked with only surviving contracts. If refinement stagnates, ConVer applies a delta-debugging-like stagnation breaker that incrementally removes problematic postcondition clauses from stagnating contracts until verification progresses.

If CEGAR still does not converge, ConVer escalates to a CEGIS stage that synthesizes contracts against the accumulated example set. For loops and recursion-heavy code where bounded reasoning is inadequate, ConVer integrates loop invariant synthesis on demand, using ESBMC’s support for invariants to establish inductive correctness rather than relying only on loop unrolling. Throughout the pipeline, the LLM is used only for proposing annotations; soundness comes from ESBMC’s validation of contracts and invariants, not from trusting the model output itself.

On the Frama-C benchmark of 45 simple C programs, ConVer achieves 82% to 96% verification success depending on the LLM backend. For programs that converged, 93% to 95% required only a single CEGAR/CEGIS iteration, indicating that initial LLM-generated contracts were often close to sufficient.

On LF2C-Simple, a suite of 17 programs, success ranges from 82% to 88%, showing that the approach transfers beyond toy examples to reactive-system-derived C models. On the X.509 parser benchmark of 6 programs, success is lower at 33% to 50%, reflecting the difficulty of parser logic and likely more complex data/control interactions. On the VerifyThis suite of 11 recursive and loop-intensive programs, the Pre-Abstraction strategy achieves 55% to 64% success, suggesting that abstraction is important for harder programs with deeper reasoning requirements.

The paper also introduces ESBMC-LF to transpile 24 Lingua Franca verifier benchmarks into C, producing the LF-Hard suite. ConVer verifies 67% of these overall, demonstrating scalability to more complex generated models. Comparisons are made against state-of-the-art verification approaches/tooling, though the paper excerpt emphasizes benchmark-wise success rates and convergence behavior more than a full table of per-baseline metrics. The core evaluation measures are verification success and convergence/refinement efficiency across backends and benchmark types.

### 🌱 Insights & Implications
The paper’s main insight is that top-down, property-directed contract generation can substantially reduce the annotation burden of compositional verification while still preserving soundness through external checking. Rather than requiring complete specifications for every function, ConVer only asks for contracts strong enough to prove the target system property, which is a practical reframing for scalable verification.

A second insight is that LLMs are useful when embedded inside a sound verification loop instead of being trusted directly. The strong one-iteration convergence on simpler suites suggests that modern models can often infer adequate contracts from code and assertions, but the lower success on X.509 and only moderate success on VerifyThis indicate clear limits when reasoning becomes heavily structural, recursive, or invariant-dependent.

SMART ICE learning appears valuable because not all counterexamples are equally informative; classifying failures into positive, negative, and implication examples provides a more structured signal for refinement than raw traces alone. The pre-abstraction strategy also seems important for scaling, but it introduces a tradeoff: over-approximation can improve tractability while risking weaker contracts that require further repair.

Limitations include dependence on LLM quality, benchmark-sensitive performance, and potential difficulties with rich specifications that ESBMC’s contract mechanism cannot easily express, such as quantified array properties. The approach is also centered on functional correctness assertions rather than directly targeting security properties, though many safety/security conditions can be encoded as assertions. Open questions include how robust the method is on real industrial codebases with pointer aliasing, concurrency, or complex heap manipulation; how to better synthesize contracts for parsers and low-level systems code; and how to combine this framework with vulnerability-specific specifications, symbolic summaries, or automated repair loops.

Intuitively, if two or more generated programs are near-duplicates at the implementation level, the reward penalizes them, encouraging the model to spread probability mass over distinct solution patterns rather than repeatedly emitting the same implementation. The paper targets a core mismatch between conventional coverage-guided fuzzers and RPKI validation: existing fuzzers assume one self-contained input per execution, while RPKI validation requires repositories containing hundreds of interdependent objects linked by certificates, hashes, manifests, CRLs, URIs, serials, and signatures. CAT addresses this with a batched, non-sequential fuzzing architecture. Instead of validating one mutated object per run, CAT executes many candidate objects inside a shared repository context, amortizing expensive setup costs such as trust-chain construction, object loading, and cryptographic validation support. Because batching destroys the straightforward mapping from coverage to a single test case, the authors introduce object-specific coverage attribution. They use function-level side channels during execution to infer which object in a batch is responsible for newly observed behavior, enabling the fuzzer to score, retain, and mutate useful inputs despite multi-object execution. The paper reports attribution accuracy above 99%, which is critical because inaccurate attribution would corrupt the feedback loop.

For input generation, CAT includes a schema-free ASN.1 mutation framework. It parses arbitrary ASN.1 DER objects into labeled trees without relying on hard-coded templates or per-schema generators. This representation lets CAT mutate both syntax and semantics: changing field values, altering structure, crossing type boundaries, and exploring protocol edge cases that template-bound fuzzers typically miss. Since many naïve mutations would immediately fail due to broken lengths, dependent fields, hashes, or signatures, CAT repairs dependent metadata and re-signs mutated objects so that repositories remain cryptographically valid enough to reach deep validation logic. The system also includes a custom DER encoder capable of re-encoding even structurally damaged objects, which is important for testing parser robustness as well as semantic validators. Overall, the methodology combines batching for scale, side-channel-based attribution for coverage guidance, and cryptography-aware ASN.1 mutation for deep path reachability.

### 🧪 Experiments
The evaluation is conducted on all five major RPKI validator implementations. CAT is compared against standard and prior RPKI fuzzing approaches, including libFuzzer and sequential fuzzing baselines. The campaign runs 300 million tests in total. The primary metrics are throughput, code-path exploration, and vulnerability yield. CAT achieves a reported 66x throughput improvement over sequential fuzzing by batching inputs and amortizing repository setup costs. In terms of exploration, it covers 24% to 47% more unique code paths than libFuzzer and previous work, demonstrating that the batched design does not merely run faster but also reaches deeper validator logic. The system’s per-object coverage attribution is reported to exceed 99% accuracy, supporting the validity of its guidance mechanism. Security outcomes are substantial: CAT uncovers 21 previously unknown vulnerabilities across the tested validators, with 8 CVEs already assigned and CVSS scores ranging from 7.5 to 9.8. The discovered bugs include a buffer overflow, denial-of-service conditions, and exploitable repository-poisoning logic flaws, with the paper stating that one memory-safety bug was developed into remote code execution against an RPKI server.

### 🌱 Insights & Implications
A key insight is that coverage-guided fuzzing can be extended beyond the traditional one-input-per-execution model if the system can recover accurate per-input credit assignment. This is especially important for security-critical software that validates sets of mutually dependent artifacts rather than isolated files. Another important takeaway is that preserving enough semantic and cryptographic validity is necessary to reach the most interesting code in RPKI validators; purely random byte-level mutation would mostly die in early parsing or signature checks. The paper also shows that high-value vulnerabilities in routing-security infrastructure are not limited to parser crashes: deep logic flaws such as repository poisoning and chain-validation issues are practically discoverable when fuzzers can maintain valid repository state.

For related research, the work suggests a general pattern for fuzzing cryptography-heavy ecosystems such as DNSSEC, TLS certificate validation, and other trust-chain-based systems. Likely limitations include the engineering complexity of deriving reliable attribution side channels, the dependence on protocol-specific repair and re-signing logic to maintain validity, and possible challenges in generalizing repository construction or oracle design to other domains. Open questions include how well the attribution method scales to even larger or more concurrent validation pipelines, whether additional semantic invariants could be automatically inferred rather than hard-coded, and how to combine this approach with symbolic execution or learned mutation strategies for even deeper path exploration.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated repair-oriented research because it proposes a new fuzzing architecture for a hard class of targets: multi-file, cryptographically linked validation software. It advances the state of the art in practical vulnerability discovery for complex security-critical systems, offers reusable ideas for fuzzing ASN.1/X.509-style infrastructures, and provides techniques that could inspire future work on secure software maintenance, regression testing, validator hardening, and automated vulnerability reproduction or repair in protocol implementations.

## 📄 [ChainCaps: Composition-Safe Tool-Using Agents via Monotonic Capability Attenuation](https://arxiv.org/abs/2605.26542v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Xiaochong Jiang, Shiqi Yang, Ziwei Li, Lifei Liu, Haoran Yu, Yichen Liu
📅 **Published:** 2026-05-26  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
ChainCaps is a runtime enforcement mechanism for tool-using LLM agents that prevents 'permission laundering' by attaching sink-specific capability budgets to data and monotonically shrinking those budgets as tools compose values. Implemented as a transparent MCP proxy, it cuts attack success from 25-68% to 0-4.8% across 82 tasks and five frontier models while preserving 96-100% benign task completion, with manifest quality emerging as the main practical bottleneck.

### 🏆 Key Contributions
- Introduces permission laundering as a distinct composition-safety failure mode in tool-using agents, where individually authorized tool calls combine into an unauthorized end-to-end action.
- Formulates composition safety using monotonic capability attenuation: every value carries a sink-specific authority budget, and derived values inherit the intersection of contributing inputs' budgets and tool pass-through limits.
- Proves a non-amplification guarantee: without explicit declassification, tool composition cannot create authority for a sink that none of the inputs already possessed.
- Implements ChainCaps as a transparent MCP proxy requiring no modification to agents or tool servers, with support for manifests, fail-closed handling of unknown sinks, session isolation, signed one-shot declassification tokens, and manifest linting.
- Evaluates the approach on live frontier-model agents and replay-based attack traces, showing strong attack blocking, high benign-task preservation, and better performance than scalar-IFC and per-function-isolation baselines.
- Identifies manifest quality, rather than runtime overhead or model compatibility, as the dominant deployment challenge for composition-safe agent enforcement.

### ⚙️ Methodology
ChainCaps targets explicit-flow safety in dynamically assembled tool chains. Its core abstraction is a sink privilege represented as an operation-scope pair, such as HTTP send to a URL prefix or file write to a directory subtree. Each runtime value is tagged with a budget: a downward-closed set of sink privileges that specifies where that value is still allowed to flow. Source values are initialized with budgets defined by trusted manifests. Each tool manifest specifies at least three semantic properties: the effect family the tool may exercise, a pass-through budget limiting the authority of tool outputs, and a way to derive the call-specific sink requirement from invocation arguments. During execution, when a tool consumes one or more input values and produces an output, ChainCaps computes the output budget as the intersection of the tool's pass-through budget and all input budgets. This intersection rule ensures monotonic capability attenuation: values may preserve or lose downstream authority, but they cannot gain new authority through transformations, summarization, concatenation, or mixing. Before an effectful sink call, the proxy checks whether the required sink privilege is contained in the budgets of all contributing values; if not, the action is blocked. The system supports explicit, controlled exceptions through signed one-shot declassification tokens, allowing deliberate policy overrides without weakening the default invariant. The implementation is a transparent MCP proxy placed between the agent and tool servers, so it observes tool calls and data movement at the protocol boundary and enforces policy without changing model internals or server code. Additional engineering features include session isolation, fail-closed behavior for unknown sinks, and manifest linting to help operators specify source/sink roles correctly. The formal guarantee is a non-amplification property: assuming trusted manifests and proxy-visible data movement, composition alone cannot introduce new sink authority absent from the originating values.

### 🧪 Experiments
The evaluation uses an 82-task stress-test suite executed with five frontier models from three providers, with three runs per condition. The task mix includes both benign workflows and adversarial or unsafe multi-tool compositions designed to trigger permission laundering, such as reading restricted data, transforming it, and attempting exfiltration through another tool. The primary live-deployment metrics are attack success rate and benign completion rate. Without defense, attack success ranges from 25% to 68% depending on model and condition; with ChainCaps, this drops to 0% to 4.8%. At the same time, benign task completion remains high at 96% to 100%, showing low disruption to intended workflows. The paper also includes replay-based experiments on logged traces to enable controlled comparison against alternative defenses. In these replays, ChainCaps outperforms a scalar information-flow control baseline and a per-function isolation baseline, indicating that sink-specific budgets capture composition risk more precisely than coarse labels or isolated call boundaries. A manifest-quality sweep further shows that policy specification quality dominates outcomes: expert-authored manifests achieve 100% attack blocking, while naive manifests drop to 27.3%. The paper also reports residual failures in scenarios like script indirection and shell-pipe laundering, which fall outside what an application-layer MCP proxy can directly observe.

### 🌱 Insights & Implications
The paper's main insight is that the safety problem in tool-using agents is not just per-tool authorization but end-to-end authority propagation across dynamically chosen workflows. Sink-specific budgets are more expressive than scalar secrecy labels for practical agent deployments because they capture exactly which external destinations, file regions, or execution scopes a value may still reach. The results suggest that runtime composition safety can be achieved with very high compatibility using a proxy-based design, making the approach appealing for real-world enterprise agent deployments. At the same time, the work makes clear that policy manifests are the weak link: the formal guarantee is only as good as the correctness and granularity of source/sink annotations and pass-through declarations. Important limitations remain. ChainCaps only claims explicit-flow protection under trusted manifests and proxy-visible data movement; it does not address covert channels, implicit flows through hidden model state, compromised tools, OS-level side effects, or data movement that bypasses the proxy. Residual attack cases involving script indirection and shell pipelines underscore this boundary. Open research questions include how to automatically infer or verify manifests, how to combine application-layer capability attenuation with lower-layer sandboxing or OS controls, and how to extend the model to richer declassification, implicit-flow reasoning, and complex multi-agent environments.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it addresses a concrete, emerging security failure mode in deployed LLM-based software systems: unsafe composition of individually permitted tool invocations. For research on automated program repair and vulnerability repair, ChainCaps offers a principled runtime mitigation that can contain exploitation even when underlying agent logic or tool orchestration is flawed. For vulnerability detection work, it provides a precise threat model, a formal enforcement invariant, and an experimentally validated benchmark setup for multi-tool security failures. More broadly, it advances secure agent engineering by showing how to retrofit composition-safe enforcement into existing MCP ecosystems without modifying agents or services, which is directly useful for building and maintaining trustworthy tool-augmented software systems.
