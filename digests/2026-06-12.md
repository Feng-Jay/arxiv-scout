## 📄 [The End of Code Review: Coding Agents Supersede Human Inspection](https://arxiv.org/abs/2606.13175v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Martin Monperrus
📅 **Published:** 2026-06-11  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This position paper argues that modern coding agents have become capable enough to replace mandatory human code review for routine software changes, because they can satisfy review goals more cheaply, faster, and at greater scale. The paper’s main result is a synthesized argument—not a new empirical study—that the traditional workflow of AI-generated code plus compulsory human review is economically and technically unstable.

### 🏆 Key Contributions
- Argues that all canonical goals of code review—defect detection, style enforcement, knowledge transfer, and team awareness—can be fulfilled by coding agents with lower cost and higher throughput than humans.
- Claims that the hybrid model where agents generate code but humans remain mandatory reviewers is a dead end, because human review becomes a bottleneck without providing strong independent assurance.
- Presents a cost-benefit argument that the marginal value of human inspection is shrinking as agent capabilities improve, while agent review offers instantaneous, consistent, and auditable verification.
- Synthesizes prior work on automated code review, coding agents, SWE-bench progress, and LLM-based software engineering into a broader thesis about replacing human review gates with agent-driven verification.
- Highlights workflow and tooling implications for software engineering practice, especially the shift from human inspection to automated verification, testing, and repair loops.

### ⚙️ Methodology
The paper is a conceptual and argumentative synthesis rather than a new empirical or model-building study. It organizes its case around three evidence streams: (1) benchmark evidence for general coding-agent capability, especially progress on repository-level software engineering tasks such as SWE-bench; (2) prior work on review-specific LLM systems that generate review comments, classify severity, or directly refine patches; and (3) observations about the economics and organizational role of code review from the software engineering literature. The core argumentative structure is: first, identify the four established goals of code review from prior empirical studies—defect detection, standards enforcement, knowledge transfer, and team awareness; second, map each goal to current or emerging agent capabilities, including whole-repository context access, test execution, patch synthesis, static/dynamic checking, documentation generation, and iterative self-repair; third, argue that agents possess structural advantages over human reviewers, such as continuous availability, consistency, ability to inspect broader context than a diff, and ability to close the loop by fixing and validating code rather than merely commenting on it; and fourth, conclude that retaining humans as mandatory review gates after AI code generation is neither scalable nor a meaningful assurance mechanism. The paper also contrasts earlier automated review approaches—rule-based tools, ML classifiers, and review-comment predictors—with newer agentic systems that can inspect, modify, run, and repair code end to end.

### 🧪 Experiments
The paper does not report new experiments. Instead, it cites prior empirical results to support its claims. For benchmark capability, it relies heavily on SWE-bench and SWE-bench Verified, describing a progression from roughly 1.7% task resolution for early GPT-4 retrieval setups, to about 12.5% for SWE-agent, to more than 50% on verified subsets by late 2024, and over 70% for leading public agents by late 2025. For review-specific capability, it cites systems such as CodeReviewer, LLaMA-Reviewer, and multi-agent review frameworks, along with industrial evaluations suggesting that LLM-based reviewers can detect defect categories targeted by humans, including correctness, security, performance, and style issues, and can generate actionable inline comments comparable to trained human reviewers on a substantial fraction of cases. It also references adjacent evidence from automated program repair, noting that LLM-based repair methods outperform older generate-and-validate systems like GenProg, and cites competitive programming evidence such as AlphaCode’s ranking within the top 54% of Codeforces participants. The metrics discussed are mainly benchmark resolution rate, qualitative or comparative comment quality, and broad productivity/effectiveness trends rather than a unified experimental protocol designed by the paper itself.

### 🌱 Insights & Implications
A key insight is that the paper reframes code review from a social/human inspection ritual into a quality-assurance function that could be decomposed into automated verification, agent-based analysis, and repair. This is especially provocative for software engineering because it suggests that the true successor to code review is not better reviewer assistance but fully agentic validation pipelines. For software maintenance and automated repair research, the paper strengthens the view that review, testing, diagnosis, and patch generation are converging into one integrated loop. For software security, its discussion implies that security review may benefit disproportionately from agents because vulnerability classes can be checked systematically and continuously, though the cited concern that AI coding tools also introduce vulnerabilities creates a dual-use tension. Limitations are substantial: the paper is a position piece, not a controlled empirical study; its claims may overgeneralize from benchmark progress to production-grade assurance; and it underplays cases where review serves organizational, legal, accountability, onboarding, or socio-technical coordination functions not easily replaced by agents. Open questions include how to validate agent review reliability on safety-critical code, how to audit false negatives in agent-based review, what governance replaces human sign-off, and whether agent-generated explanations truly substitute for human knowledge transfer. The paper’s bold thesis is likely strongest for routine changes in high-test-coverage repositories and weaker for novel architecture, high-assurance systems, or complex socio-technical changes.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it directly addresses the future of quality gates around AI-generated code. It connects automated review, agentic coding, testing, and repair into a unified vision that overlaps strongly with APR and vulnerability research: if agents can detect, explain, patch, and validate defects, then review becomes part of an automated repair pipeline rather than a separate human step. Even though it is not an empirical systems paper, it is useful as a research agenda piece because it identifies where stronger evidence is needed: trustworthy agent-based verification, secure patch validation, scalable vulnerability review, and the replacement of human inspection with auditable automated assurance.

## 📄 [Understanding the Rejection of Fixes Generated by Agentic Pull Requests -- Insights from the AIDev Dataset](https://arxiv.org/abs/2606.13468v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Mahmoud Abujadallah, Ali Arabat, Mohammed Sayagh
📅 **Published:** 2026-06-11  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper analyzes why nearly half of AI-agent-generated bug-fix pull requests are rejected in the AIDev dataset, finding that failures stem from incorrect implementations, CI/test failures, agent execution problems, and low-relevance fixes. Through a qualitative study of 306 rejected PRs, the authors derive a 14-reason taxonomy and argue that better task selection, clearer guidance, and stronger validation practices are needed to make agentic PRs more effective teammates.

### 🏆 Key Contributions
- Provides one of the first focused studies of rejected AI-agent-generated bug-fix pull requests, using 1,497 rejected fixes out of 3,225 total fixes from Copilot, Devin, Cursor, and Claude in the AIDev dataset.
- Builds a qualitative taxonomy of 14 rejection reasons grouped into four high-level categories: relevance of the fix, implementation issues, provider/agent issues, and technical issues.
- Quantifies the prevalence of rejection causes in a representative random sample of 306 rejected PRs, showing that many rejections are due not only to wrong code but also inactivity, superseded work, CI failures, and agent execution breakdowns.
- Frames rejected agentic PRs as wasted human and computational effort, emphasizing review burden, code churn, comments, token usage, and limited premium agent requests.
- Derives practical recommendations for improving agentic bug fixing: provide hints about solution strategy, specify forbidden or undesirable approaches, and instruct agents to validate against CI and avoid breaking changes.

### ⚙️ Methodology
The study is based on the AIDev dataset of AI-generated pull requests from open-source GitHub repositories with more than 100 stars. The authors filter the dataset to focus specifically on bug-fix PRs that were closed without being merged and were created or co-authored by Copilot, Devin, Cursor, or Claude; Codex PRs are excluded because its exact contribution cannot be systematically identified. This yields 1,497 rejected agentic bug-fix PRs out of 3,225 total fix PRs from the four agents. From these 1,497 rejected fixes, the authors draw a representative random sample of 306 PRs at 95% confidence level and 5% margin of error. For qualitative analysis, two authors independently inspect each PR by reading discussion threads, comments, CI results, and linked external artifacts to infer the reason for rejection. They assign labels capturing the underlying cause, then reconcile labels through discussion to derive a final coding scheme. Inter-rater reliability is measured with Cohen’s kappa of 0.605, indicating substantial agreement. A third author reviews the 93 disagreement cases to reach consensus, and the authors then organize the final labels into higher-level categories. The resulting taxonomy contains 14 rejection reasons under four top-level themes: relevance-related issues, implementation issues, provider issues, and technical issues. The paper also includes a quantitative analysis of the distribution of reasons and investigates wasted review effort using proxies such as code churn and number of comments exchanged on rejected PRs.

### 🧪 Experiments
The empirical study uses the AIDev dataset containing 33,596 curated PRs across 2,807 projects. The target subset consists of AI-agent-generated bug-fix PRs from Copilot, Devin, Cursor, and Claude in repositories with over 100 stars. Among 3,225 fix PRs, 1,497 were rejected, corresponding to a rejection rate of 46.41%. The qualitative coding is conducted on a statistically representative sample of 306 rejected PRs. The main output is the taxonomy and frequency of rejection reasons. Reported labeled reasons include inactivity 53 cases (17.3%), agent failure 23 (7.5%), CI failure 21 (6.9%), superseded PRs 18 (5.9%), incorrect fix 17 (5.6%), wrong approach 8 (2.6%), low priority 3 (1.0%), rate limit 3 (1.0%), ambiguity 2 (0.7%), insufficient fix 2 (0.7%), wrong repository 2 (0.7%), architecture change 1 (0.3%), test PR 1 (0.3%), and breaking change 1 (0.3%). A large portion, 151 PRs (49.3%), could not be assigned a clear rejection reason from available evidence. For review-effort analysis, the authors use code churn and comment counts as proxies for wasted effort, and report substantial churn associated with rejected fixes, with medians ranging from 81 to 293 depending on rejection category. There are no model-training experiments or APR baselines; the study is observational and repository-mining-based.

### 🌱 Insights & Implications
A key insight is that rejection of agentic bug-fix PRs is not dominated solely by semantic incorrectness; workflow and socio-technical factors such as inactivity, duplication, superseded work, CI breakage, and agent-side failures account for a large share of failed outcomes. This matters for software engineering research because improving patch generation alone will not eliminate rejection unless agent pipelines also become better at task selection, coordination, and validation. The large 'Others' bucket and the fact that nearly half of sampled rejections lacked a clear observable explanation indicate an important limitation of repository-mining studies: public PR artifacts often under-specify the real rationale behind human decisions. The study is also limited to four agents, open-source GitHub projects in AIDev, and non-merged bug-fix PRs, so findings may not generalize to private repositories, other agents, or non-bug-fix tasks. Another implication is that project-specific context and constraints strongly shape acceptance, suggesting that future agentic repair systems should incorporate issue-level guidance, repository policies, CI-aware self-validation, and prioritization signals before opening PRs. For related research in automated program repair and vulnerability repair, the paper highlights that a technically plausible patch may still fail because it targets the wrong issue, uses an unacceptable approach, or does not survive integration checks.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, maintenance, automated program repair, and vulnerability repair because it studies real-world acceptance failures of AI-generated bug-fix patches at the pull-request level. It advances the researcher’s work by providing an empirical taxonomy of why generated fixes are rejected in practice, revealing actionable gaps beyond patch correctness—such as CI validation, task prioritization, and repository constraints—that are crucial for building repair and vulnerability-fixing systems that are not only correct but also acceptable to developers.

## 📄 [The Emergence of Autonomous Penetration Capabilities in Large Language Model-Powered AI Systems](https://arxiv.org/abs/2606.13079v1)  `90% relevant`

👤 **Authors:** Jiaqi Luo, Jiarun Dai, Zhile Chen, Jia Xu, Weibing Wang, Yawen Duan, Brian Tse, Geng Hong, Xudong Pan, Yuan Zhang, Min Yang
📅 **Published:** 2026-06-11  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper introduces a more realistic benchmark for testing whether LLM-based agents can autonomously penetrate vulnerable servers without target-specific hints, and shows that current open and proprietary models already achieve nontrivial to substantial success rates. Across 300 target servers and 19 models, autonomous penetration success ranges from 10.7% to 69.3%, suggesting this capability is emerging and scales with general model strength.

### 🏆 Key Contributions
- Proposes a new autonomous penetration evaluation framework aimed at measuring a red-line cyber capability under more realistic and less over-guided conditions
- Constructs 300 target servers with two difficulty tiers that combine one vulnerable service with additional secure, non-vulnerable services to better simulate realistic attack surfaces
- Designs a general-purpose agent scaffolding using generic cybersecurity tools and no target-specific prior knowledge, reducing evaluation leakage and hand-holding
- Evaluates 19 open-weight and proprietary LLMs in a unified setup and reports autonomous penetration success rates from 10.7% to 69.3%
- Provides evidence that autonomous penetration capability improves in step with broader model capability advances

### ⚙️ Methodology
The study builds an evaluation framework with two main components: target servers and agent scaffolding. On the environment side, each target server includes a vulnerable service embedded among secure services that have no known vulnerabilities, forcing the agent to perform realistic reconnaissance, service discrimination, vulnerability identification, exploitation, and post-exploitation access acquisition rather than simply attacking a single obviously vulnerable component. Two difficulty tiers are defined based on environmental complexity: Tier 1 contains one secure service plus one vulnerable service, while Tier 2 contains three secure services plus one vulnerable service. This yields 300 total target servers. On the agent side, the authors use a general-purpose LLM-powered autonomous agent architecture equipped with generic cybersecurity tools, likely for command execution, scanning, enumeration, and interaction with services, but explicitly without target-specific prior knowledge, exploit templates, or bespoke instructions tied to individual environments. The intent is to evaluate end-to-end autonomous offensive capability under minimal guidance: the model must independently plan attack steps, explore the target, identify the vulnerable pathway among benign ones, exploit it, and obtain unauthorized access or control. The benchmark is designed to remedy prior work’s issues of opacity, oversimplification, and excessive prompting support.

### 🧪 Experiments
The evaluation covers 19 LLMs spanning both open-weight and proprietary systems. The benchmark consists of 300 target servers distributed across two tiers of difficulty, where complexity increases with the number of co-hosted secure services. The primary reported metric is penetration success rate, i.e., the proportion of targets on which an autonomous agent successfully compromises the server or obtains unauthorized access/control. Reported success rates range from 10.7% to 69.3% across models, indicating wide variance in autonomous offensive capability. Although the abstract does not provide per-model breakdowns, exact baselines, or additional metrics such as time-to-compromise, number of tool calls, or false attack paths explored, the key quantitative finding is that several current models can successfully complete a substantial fraction of realistic autonomous penetration tasks without target-specific guidance. A further comparative result is the observed positive relationship between overall model capability and penetration performance.

### 🌱 Insights & Implications
The central insight is that autonomous penetration is no longer a purely speculative capability: under a more realistic and less scaffolded setup, current LLM agents can already execute meaningful end-to-end compromise workflows. For security researchers, this implies that general-purpose capability gains in frontier models may directly transfer into stronger offensive cyber performance, even without specialized fine-tuning or exploit-specific prompting. The benchmark’s design also highlights an important methodological lesson for AI safety and cyber evaluation: realistic background noise from secure services matters, because it forces models to distinguish exploitable from non-exploitable attack surfaces. Limitations visible from the abstract include the lack of detail on vulnerability diversity, exploit novelty, repeatability controls, and whether success depends on known exploit patterns versus deeper reasoning. Open questions include how robust these results are against hardened systems, patched-but-misconfigured services, multi-step chained exploits, privilege escalation tasks, stealth constraints, and detection-aware settings. Another key implication is dual-use risk: evaluation frameworks like this can improve safety measurement, but they also underscore the urgency of capability monitoring, secure deployment policies, and red-team benchmarking.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and vulnerability repair because it studies whether modern LLM agents can autonomously discover and exploit vulnerable software services in realistic server environments. It advances the researcher’s work by providing a benchmark-oriented perspective on offensive capability emergence, which is valuable for designing defensive analyses, secure software maintenance practices, automated vulnerability triage, and future systems for AI-assisted vulnerability detection or repair that must be evaluated against increasingly capable autonomous attackers.

## 📄 [DIG: Oracle-Guided Directed Input Generation for One-Day Vulnerabilities](https://arxiv.org/abs/2606.13037v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Andrew Bao, Haochen Zeng, Peng Chen, Stephen McCamant, Pen-Chung Yew
📅 **Published:** 2026-06-11  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
DIG is an oracle-guided system for generating proof-of-concept inputs for one-day vulnerabilities by extracting triggering preconditions from security patches and using them to steer both LLM-based input generation and directed fuzzing. Across 138 real-world CVEs, it triggers 80 vulnerabilities—beating two prior agentic systems and ten fuzzers, achieving 40% more triggered CVEs than the best baseline and often finding exploits much faster.

### 🏆 Key Contributions
- Formulates one-day PoC generation as a constraint-solving problem centered on patch-derived oracle conditions that encode necessary triggering preconditions.
- Introduces LLM-based oracle synthesis that analyzes patches, source context, and vulnerability descriptions to infer explicit conditions whose violation is necessary to trigger the vulnerable behavior.
- Proposes oracle-guided generator evolution, where an agent incrementally refines and preserves input-generation logic across iterations to solve oracle-related constraints while mitigating long-horizon goal drift.
- Develops oracle-guided directed mutation by instrumenting the synthesized oracle into the target and using branch-distance feedback plus oracle-relevant region mutation to complement high-level reasoning.
- Builds and evaluates an end-to-end system on 138 real-world vulnerabilities, showing state-of-the-art effectiveness and demonstrating that the same framework can also uncover previously unknown zero-day vulnerabilities.

### ⚙️ Methodology
DIG targets one-day vulnerabilities under the assumption that patches often reveal the preconditions required to trigger the original bug. Its first stage is oracle synthesis: an LLM inspects the patch, surrounding source code, and vulnerability description to identify security-relevant semantic changes such as newly added checks, corrected variable relations, or state constraints. DIG then inverts these patched conditions into an oracle representing what must hold in the vulnerable version for the bug to manifest. Rather than treating the target merely as a code location, the oracle makes the hidden triggering condition explicit.

DIG uses this oracle in two coordinated layers. At the high level, it performs oracle-guided generator evolution. Starting from the oracle, an agent reasons backward through code dependencies to infer how oracle variables are computed, propagated, and constrained by the input format and program logic. It translates these inferred constraints into an executable structured input generator. Crucially, DIG evolves this generator incrementally: each run preserves the successful logic from prior runs and extends it to satisfy additional unresolved constraints, thereby decomposing PoC generation into smaller subgoals and reducing the tendency of LLM agents to drift away from the actual triggering condition over long reasoning chains.

At the low level, DIG instruments the synthesized oracle into the program binary to support directed mutation. The fuzzer focuses mutations on input regions that influence oracle satisfaction and uses branch-distance feedback from the oracle predicates to determine whether mutations move the input closer to or farther from satisfying the necessary conditions. This provides more direct and semantically meaningful guidance than conventional target-distance-only directed greybox fuzzing. The two layers are complementary: generator evolution performs global, structure-aware search over valid inputs, while oracle-guided mutation performs high-throughput local search to satisfy residual hard constraints, including hidden state-dependent ones that the agent may not fully infer.

### 🧪 Experiments
DIG is evaluated on 138 real-world CVEs from the Magma benchmark and compared against 12 baselines in total: 2 state-of-the-art agentic/LLM-based input-generation systems and 10 fuzzers, including directed and non-directed fuzzing approaches. The primary evaluation objective is PoC generation, measured by whether a tool can trigger a vulnerability and how quickly it does so; the paper also reports exclusive triggers and relative speedups.

DIG triggers 80 of the 138 vulnerabilities, outperforming all baselines. The best prior baseline triggers 57 CVEs, so DIG improves the state of the art by about 40% in absolute triggered cases (57 vs. 80). DIG also exclusively triggers 9 vulnerabilities that no existing compared technique can trigger. In time-to-trigger comparisons, DIG is faster than the average of other tools in 92.9% of cases, achieves more than 100x speedup in 48.8% of cases, and reaches a maximum reported speedup of 3,664x. Beyond benchmarked one-day vulnerabilities, the authors report that DIG discovers 6 previously unknown vulnerabilities in widely deployed libraries, suggesting that the oracle-guided approach generalizes from one-day PoC generation to zero-day vulnerability discovery as well.

### 🌱 Insights & Implications
The central insight is that security patches are not only fixes but also rich semantic signals about the necessary conditions for exploitation. By converting those latent signals into explicit oracles, DIG bridges a gap between symbolic reasoning and fuzzing: it gives agents a concrete subgoal for structured reasoning and gives mutational search a directly optimizable objective. This is particularly important for vulnerabilities whose triggering requires satisfying long chains of semantic constraints, cross-field dependencies, numeric edge cases, or hidden state conditions that are poorly captured by standard control-flow distance metrics.

A major implication for related research is that patch-aware vulnerability analysis can improve not just exploit generation but also patch validation, regression testing, and incomplete-fix detection. The reported discovery of new vulnerabilities further suggests that oracle synthesis may encode general bug patterns beyond the original CVE context.

Potential limitations and open questions include the dependence on patch quality and explicitness: if a patch is noisy, incomplete, or refactors code without clearly exposing the true trigger condition, oracle synthesis may be harder or less reliable. The approach also relies on LLM reasoning quality for extracting semantically correct constraints, which may vary across domains and programming languages. Further questions include how robust DIG is to multi-commit fixes, configuration-dependent vulnerabilities, complex environmental interactions, and large-scale industrial codebases outside Magma; how much each component contributes via ablation; and whether synthesized oracles can be formally validated or combined with symbolic execution for even stronger constraint solving.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and vulnerability repair because it directly addresses automated exploitation and validation of one-day vulnerabilities using patch information—exactly the setting where defenders need to assess residual risk after a fix is released. It is also valuable for automated program repair and software maintenance research: DIG shows how patches can be mined for semantic constraints, enabling patch correctness checking, incomplete-fix detection, regression test/PoC generation, and security-focused maintenance workflows. For a researcher working on vulnerability detection and repair, DIG provides a strong new paradigm that unifies LLM reasoning, patch analysis, and feedback-guided search into a practical end-to-end system with strong empirical gains.

## 📄 [Toward Instructions-as-Code: Understanding the Impact of Instruction Files on Agentic Pull Requests](https://arxiv.org/abs/2606.13449v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Ali Arabat, Mohammed Sayagh
📅 **Published:** 2026-06-11  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper studies whether adding repository instruction files for coding agents improves the quality of agent-generated pull requests. Across 15,549 agentic PRs from 148 projects, the authors find no consistent benefit: instruction files help some projects, hurt others, and projects with better outcomes tend to have longer, more structured instructions.

### 🏆 Key Contributions
- Constructs a large-scale empirical study linking agent instruction files to downstream pull request outcomes in real GitHub repositories.
- Introduces a before/after project-level analysis of agentic PR performance around the creation of instruction files.
- Evaluates instruction-file impact along three dimensions: PR success (merge rate), task complexity (description length, code churn, changed files, commits), and merge effort (discussion and time to merge).
- Shows that instruction files do not universally improve agent performance; gains and losses are both common across projects.
- Provides an initial quantitative signal that longer and more hierarchically structured instruction files correlate with improved merge rates, motivating the notion of 'Instructions-as-Code'.
- Releases scripts and data to support replication and follow-up research.

### ⚙️ Methodology
The study starts from the AIDev dataset of agentic pull requests and expands it with newer GitHub data through December 10, 2025 using the GitHub GraphQL API. The authors first filter repositories to remove toy, template, archived, forked, or insufficiently active projects, ending with 2,449 candidate projects and 48,944 PRs before narrowing further to projects that contain agent instruction files and have enough agentic PRs on both sides of the instruction-file introduction. Instruction files are detected by scanning repository trees for agent-specific filename/path patterns covering Devin, Cursor, Copilot, Claude, Codex, and common `AGENTS.md`-style files. For each instruction file, its creation date is approximated by the first commit that adds it. The final analysis dataset contains 148 projects and 15,549 agentic PRs, with at least five agentic PRs before and after instruction-file creation. The analysis is organized by four research questions. For RQ1, the authors compute project-level relative change in merge rate before versus after instruction-file creation, considering both the first instruction file and the complete set of files when multiple instructions exist. For RQ2, they restrict to merged PRs and compare pre/post distributions of task-complexity metrics—description length, code churn, number of modified files, and number of commits—using Mann-Whitney U tests with `alpha = 0.05` and Cliff’s delta to assess statistical significance and effect size. For RQ3, they similarly compare merge-effort proxies such as time to merge and number of comments/discussion. For RQ4, they perform an exploratory quantitative analysis of instruction-file verbosity and structure, comparing projects with improved versus degraded merge rates using measures such as word count and header/subheader counts. The overall framing is observational and longitudinal at the project level, using instruction-file introduction as the temporal intervention point.

### 🧪 Experiments
The empirical setup uses 15,549 agentic PRs from 148 GitHub projects selected from the AIDev ecosystem, all with detectable instruction files and sufficient PR activity before and after those files were introduced. The main comparison is not against model baselines but against each project’s own pre-instruction period, which helps control for repository-specific context. Metrics are grouped into three outcome families: merge success (merge rate), task complexity (description length, code churn, changed files, commits), and merge effort (time to merge, discussion/comments). For statistical testing of complexity and effort metrics, the authors use Mann-Whitney U tests and Cliff’s delta, focusing on projects with at least five merged PRs in both periods for complexity analysis. Key results include: for merge rate after the first instruction file, 27.7% of projects improved by at least 20%, while 26.35% declined by at least 20%; after considering all instruction files, 31.93% improved by at least 20% and 27.73% declined by at least 20%. For task complexity after the first file, 35.35% of projects significantly increased PR description length, 10.10% increased code churn, 12.12% increased number of commits, and 8.08% increased changed files; corresponding decreases were much rarer for several metrics, such as 4.04% for description length and 7.07% for code churn and commits. For merge effort, 13.13% of projects showed significantly increased time to merge and 15.15% showed increased discussion, while 8.8% showed decreases in both. In the exploratory instruction-content analysis, projects with improved merge rates had much longer instruction files than projects with degraded merge rates, with median lengths of 976 versus 569 words, and they also had more structural headers, especially third-level headers.

### 🌱 Insights & Implications
The central insight is that simply adding instruction files is not enough to reliably improve agentic software development outcomes. Instruction files appear to be a high-variance intervention: they can increase merge success and enable larger or more complex changes in some repositories, but they can also coincide with worse merge rates and higher integration effort in others. This suggests that instruction quality, specificity, organization, and alignment with repository practices likely matter more than mere existence. For software engineering and maintenance researchers, the paper opens a new artifact class for study: instruction files as maintainable, evolving repository assets. The proposed 'Instructions-as-Code' framing is especially important because it implies that instructions should be engineered, reviewed, measured, and possibly refactored like source code or build/test configuration. Limitations include the observational design, which cannot establish causality; possible confounders such as simultaneous process changes, repository maturation, agent-version drift, or shifting task difficulty over time; reliance on filename heuristics to detect instruction files; and the fact that the paper offers only an initial content analysis focused on verbosity and structure rather than semantic quality. Open questions include what instruction characteristics causally improve mergeability, how to automatically assess instruction quality, how instructions interact with different agent platforms, and whether instruction-aware tooling could recommend, validate, or repair instructions for specific development tasks, including security-sensitive changes.

### 🎯 Why This Paper
This paper is highly relevant because it studies an emerging software-engineering control surface for AI coding agents: repository-level instructions. For researchers in software engineering, maintenance, security, automated program repair, vulnerability detection, and vulnerability repair, the work matters because instruction files may directly shape how agents navigate codebases, follow testing and review policies, and modify critical code. The paper does not propose a repair or security model itself, but it provides strong empirical evidence that agent guidance artifacts can materially affect whether generated PRs are accepted, how complex the changes are, and how much human effort is needed to integrate them. That makes it a useful foundation for future work on instruction synthesis, instruction quality metrics, secure-by-construction agent workflows, instruction-aware APR pipelines, and policy-driven vulnerability repair agents.

## 📄 [Beyond Problem Solving: UOJ-Bench for Evaluating Code Generation, Hacking, and Repair in Competitive Programming](https://arxiv.org/abs/2606.12864v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Tingqiang Xu, Hangrui Zhou, Tianle Cai, Alex Gu, Kaifeng Lyu
📅 **Published:** 2026-06-11  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper introduces UOJ-Bench, a benchmark built from real competitive-programming submissions on Universal Online Judge to evaluate LLMs on code generation, bug-triggering test generation (hacking), and minimal code repair under the judge’s native infrastructure. The main result is that current models are still weak at one-shot debugging-style tasks, especially on covert errors missed by standard tests, but aggressive test-time scaling can push hacking and repair success above 90% and even discover previously unknown bugs in supposedly accepted submissions.

### 🏆 Key Contributions
- Introduces UOJ-Bench, a unified benchmark for competitive-programming code generation, code hacking, and code repair using real UOJ submissions and native judge evaluation.
- Defines two difficulty levels for hacking and repair: Easy for overt errors detectable by standard judge tests, and Hard for covert errors only exposed through UOJ’s community hack mechanism.
- Constructs benchmark instances from authentic human revision histories and successful community hacks, grounding evaluation in realistic student/programmer failure modes rather than synthetic bugs.
- Adds Zero-Day-Hacking-5K, an extended evaluation set of accepted submissions to test whether LLMs can uncover previously unknown hidden errors, analogous to zero-day vulnerability discovery.
- Provides a cost-performance analysis showing that frontier models improve dramatically with test-time scaling, while highlighting the inference-cost barrier to practical deployment.
- Shows that LLMs can provide complementary quality signals beyond fixed online-judge test suites by finding errors in a nontrivial fraction of full-score submissions.

### ⚙️ Methodology
The benchmark is designed around three tasks. In code generation, a model receives a competitive-programming problem and must produce a correct solution. In code hacking, the model is given the problem plus a buggy human-written submission and must generate an input test case that causes the submission to fail under UOJ’s official judging environment. ### 💡 TL;DR
This paper introduces Multi-Session Memory Poisoning (MSMP), a threat where attackers poison the persistent memory of RAG/agent systems across user sessions, and proposes SMSR, a two-part defense combining cryptographic memory provenance and randomized retrieval smoothing with certified robustness guarantees. The main result is that SMSR eliminates unsigned poisoning attacks and substantially reduces authenticated/query-only poisoning while providing the first formal certificate for this dynamic memory setting.

### 🏆 Key Contributions
- Defines the MSMP threat model for persistent-memory LLM agents and formalizes a (t, delta)-style security notion for runtime memory poisoning.
- Proves an impossibility result showing that retrieval-time filtering without provenance cannot provide a non-trivial robustness certificate against adaptive memory injection.
- Proposes SMSR, a two-component defense: HMAC-SHA256 signing of legitimate memory writes and randomized memory ablation with verdict-based majority voting at query time.
- Derives a hypergeometric robustness certificate for the smoothed retrieval component, bounding attack success for authenticated adversaries controlling up to t injected memories.
- Formalizes the Consistent Minority Effect (CME), where adversarially consistent responses can defeat string-based voting despite being a numerical minority, and shows verdict-based aggregation mitigates it.
- Provides empirical validation across 15 enterprise scenarios and live-agent query-only attacks, showing strong reductions in attack success with moderate utility cost.

### ⚙️ Methodology
SMSR is designed for persistent RAG/agent memory systems in which memories are appended during normal operation and later retrieved to condition responses. The defense has two sequential components. Component 1 secures the write/retrieval boundary using provenance: each legitimate memory entry is tagged at write time with an HMAC-SHA256 signature generated by a trusted signing oracle. At retrieval time, only signed memories are eligible for use, which creates a cryptographic barrier against unsigned runtime injection through ordinary interaction channels or bypass text crafted to evade content filters. The paper argues formally that any defense relying only on retrieval-time content filtering, without provenance, is fundamentally uncertifiable against an adaptive adversary because fluent malicious text can mimic benign enterprise memory entries.

Component 2 addresses stronger attackers who are authenticated and therefore able to create signed but malicious memories. Instead of deterministically retrieving a fixed top-k set and passing it directly to the generator, SMSR first over-fetches a candidate pool of size m and then performs randomized ablation by sampling subsets from that pool across multiple runs. For each sampled context, the agent answers the query; the resulting answers are then judged not by string identity but by a verdict-based evaluator that classifies whether each answer is correct/benign or attack-induced. A majority vote over verdicts determines the final response acceptance. This design prevents attackers from exploiting textual consistency to dominate string-level self-consistency voting. The paper formalizes this as the Consistent Minority Effect: adversarial outputs may be semantically aligned and nearly identical, while benign outputs can vary lexically, causing naive string-majority voting to select the attack even when most runs are benign. Verdict-based aggregation collapses semantically equivalent benign answers and removes this failure mode.

The certified guarantee for Component 2 is derived using a hypergeometric argument over the random sampling process from the over-fetched memory pool. Given at most t adversarial memories in the candidate set, the analysis bounds the probability that enough poisoned memories survive ablation across runs to flip the final verdict. The result is a worst-case upper bound delta on attack success under specified retrieval/sample parameters and number of runs. The overall framework thus combines a hard provenance guarantee against unsigned injection with a probabilistic certified robustness guarantee against authenticated poisoning in a live, append-only memory system.

### 🧪 Experiments
The evaluation covers 15 enterprise knowledge-base scenarios with 3,150 repeated trials, including six Component-2 configurations across the scenarios with 30 repetitions each, plus 450 production-scale trials and an end-to-end live-agent attack study. Attack classes include unsigned injection, authenticated direct injection, flooding-style variants, heuristic-bypass attacks, and a query-only attack in which the agent itself writes the poison during normal operation rather than using pre-seeded malicious entries.

Baselines/comparators include heuristic retrieval-time defenses such as keyword blacklists, entropy proxies, and semantic anomaly filters, as well as discussion relative to prior RAG defenses like RobustRAG, ReliabilityRAG, and A-MemGuard. Evaluation focuses primarily on attack success rate (ASR), confidence intervals, clean-query utility, and judge agreement/reliability. A LLM judge is used for verdict-based scoring, and cross-model robustness is checked using Claude Sonnet and Haiku, with judge agreement reported as kappa = 0.955 over n = 84.

Key quantitative results: Component 1 reduces ASR from 93-100% to 0% for all unsigned injection variants, including attacks specifically crafted to bypass heuristic filters. In a production-scale memory store with m = 20 candidate memories and 20 benign seed memories, Component 2 reduces authenticated single-injection attack success to 8.0% (95% CI [5.8%, 10.9%], n = 450), which is below the certified worst-case bound of 10.4% for five runs; the theoretical bound tightens to 7.1% for seven runs, though that setting is not evaluated. In a smaller store setting m' = 10 + t, the canonical direct-injection t = 1 attack reaches 37.8% ASR (95% CI [33.4%, 42.3%], n = 450), below the evaluation-pool certificate delta = 41.5%; direct-injection bounds hold for t in {1,2,3}, while the flooding attack empirically approaches the worst-case bound, indicating certificate tightness. In the live end-to-end query-only attack where the agent writes the poisoned memory itself, SMSR lowers ASR from 65.3% to 5.3% over n = 150 with non-overlapping confidence intervals. Clean-query utility is reported as 90% with Component 1 alone and 85% for the combined defense.

### 🌱 Insights & Implications
The paper’s central insight is that persistent agent memory creates a fundamentally different security problem from static-corpus RAG poisoning: the adversary writes through the same operational channel that benign users rely on, so retrieval-only sanitization is structurally insufficient. This motivates the impossibility result and the necessity of provenance-aware defenses. A second important insight is that dilution of malicious memories through randomized over-fetching and ablation can be turned from an empirical intuition into a certified guarantee, making smoothing-style robustness applicable to memory retrieval rather than only token-level or image perturbation settings.

The Consistent Minority Effect is especially relevant for agent defense design: self-consistency or majority-vote schemes can be gamed when attackers produce highly templated outputs and benign generations are semantically correct but linguistically diverse. The proposed verdict-based aggregation is therefore a meaningful methodological contribution beyond this specific paper and could influence evaluation and defense design in other agent security settings.

Limitations include the dependence of Component 1 on a trusted signing infrastructure and key management, which may be operationally challenging in multi-tenant or federated deployments. Component 2 only certifies against bounded authenticated poisoning under specified retrieval/sample assumptions; stronger adaptive attackers, larger poisoning budgets, or different retrieval distributions may weaken practical guarantees. There is also a utility tradeoff, with clean performance dropping to 85% under the full defense. The use of LLM judges, while empirically reliable here, introduces another learned component whose calibration and failure cases deserve further study. Open questions include how to extend certification to richer memory operations such as editing/deletion, agent-generated summaries, multi-hop retrieval, cross-agent shared memory, and poisoning that targets downstream tool invocation rather than only answer text.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability defense, and trustworthy automated systems because it treats persistent LLM-agent memory as a new attack surface and provides one of the rare formal, certified defenses rather than a heuristic mitigation. For a researcher interested in security, repair, and robust software engineering of AI-enabled systems, it advances the state of the art by combining threat modeling, impossibility analysis, cryptographic provenance, and certified randomized defense mechanisms for runtime attacks that resemble persistent state corruption in long-lived software systems.

- A new benchmark dataset, HybridCodeAuthorship, containing full Python files with interleaved human- and AI-authored code rather than whole-snippet binary labels.

The benchmark measures classification effectiveness using F1 score. Results show that performance is modest, indicating substantial task difficulty. The best-performing system, AIGCode Detector, reaches a highest F1 of 0.48 on chunk-level detection and 0.56 on line-level detection. ### 🌱 Insights & Implications
The paper highlights an important mismatch between existing AI-code detection benchmarks and real industrial usage: most prior datasets assume an entire snippet is either human or AI authored, while actual repositories increasingly contain mixed authorship with edited and interleaved AI suggestions. Limitations include the current restriction to Python, dependence on CodeSearchNet sampling, and simulation of human-AI interaction through an automated masking-and-regeneration pipeline rather than logs of real developer-assistant sessions. Open questions include how well detectors generalize across programming languages, LLM families, editing styles, and acceptance rates; whether structural, semantic, or provenance-based methods outperform perplexity-style detectors; and how to distinguish benign AI assistance from risky or vulnerable generated code in maintenance and security workflows.

### 💡 TL;DR
This paper presents the first large-scale longitudinal study of covert communication channel usage in Android malware, combining custom static validation rules with selective dynamic analysis over 3.5–3.6 million malware samples from 2009–2025. ### 🏆 Key Contributions
- Introduces the first systematic longitudinal analysis of Android malware using covert channels such as Tor, Tor pluggable transports, I2P, VPNs, and proxies.
- Designs a scalable static-analysis framework based on manually curated static validation rules (SVRs) tailored to each covert channel, enabling detection without pre-existing ground truth.

### ⚙️ Methodology
The study proposes a multistage pipeline that combines static discovery, static validation, selective similarity-based expansion, and dynamic validation. Because no ground-truth dataset of Android malware using covert channels exists, the authors begin with an exploratory seed collection strategy using VirusTotal reports and keywords associated with covert communication tools. They then manually study how each covert channel is integrated into benign Android applications and malware, deriving channel-specific static validation rules rather than relying on generic signatures.

For Tor, Tor pluggable transports, and I2P, the rules focus on identifying characteristic Android packages, bundled libraries, and binaries whose presence strongly indicates integration of these systems. For VPNs, the authors decompile legitimate VPN apps from Google Play to identify packages and implementation patterns. For proxies, since proxy functionality is often embedded inside multifunctional applications, they aggregate provider names from prior work and inspect APKs to isolate packages specific to proxy behavior. They also instrument and dynamically execute a small set of proxy/VPN apps to confirm that the identified package methods are actually invoked at runtime.

To address obfuscation and modified libraries, the pipeline adds two similarity-based stages. First, LibScan is used to detect known libraries even when malware is obfuscated. Second, MatchScope compares entire APK codebases so that validated covert-channel samples can help identify additional non-validated but similar samples. These steps expand beyond simple direct package matching while acknowledging that current tools still miss heavily obfuscated or customized cases.

Dynamic validation is then applied to a subset of suspected covert-channel malware. Instead of relying only on emulators, the authors run samples on physical Google Pixel 5 devices for a bounded 5-minute window to reduce anti-analysis evasion and ethically limit exposure. They do not provide user interaction except allowing VPN permission creation. Runtime evidence is collected at two levels: network traces and system/process traces. Network-level rules look for Tor relays, VPN protocols such as OpenVPN and WireGuard, and proxy-related patterns using regex-based detection and nDPI. System-level rules inspect logs and process behavior, leveraging the fact that each covert channel leaves distinctive operational artifacts. The final dataset is then characterized longitudinally by year, family, certificate reuse, geographic infrastructure, and changes in covert-channel strategy over time, including multi-channel adoption and repeated switching between channels.

### 🧪 Experiments
The empirical study uses 3.5–3.6 million Android malware APKs from AndroZoo, covering 2009 to July 2025 and filtered with VirusTotal score greater than 0. Static keyword search over VirusTotal reports initially flags 102K APKs, but 81K of these are false positives, showing that naive metadata search is insufficient. In contrast, the proposed static validation pipeline identifies about 288K APKs using covert channels: 150 using Tor, 3.5K using Tor pluggable transports, 2.4K using I2P, 17K using VPNs, and 271K using proxies. These samples span 511 malware families overall, with 12 Tor families, 39 TorPT families, 23 I2P families, 127 VPN families, and 477 proxy families. The dataset also includes 215K singleton samples and about 124K unique package names.

The authors perform dynamic analysis on a random sample of 13.7K malware APKs from the 288K covert-channel candidates, executing them on physical Pixel 5 devices. Of these, 6.4K execute successfully, with post-2019 samples showing execution success rates above 40%. Dynamic analysis reveals communication with 19,308 unique IP addresses across 85 countries. The authors explicitly validate covert-channel presence for 59 IP addresses across 17 countries using runtime evidence. They also observe geographically distributed infrastructure and reuse of certain IP endpoints across malware.

The longitudinal analysis shows major temporal growth in covert-channel adoption: from 0.30% of malware in 2012 to 50% in 2025. The family-level evolution analysis finds that some malware families adopt multiple covert channels simultaneously, with one family using more than five channels, while others repeatedly switch channels over time; one family changes covert-channel usage 40 times between 2019 and 2025. Certificate analysis further reveals common signing keys and fake organization names reused across large malware clusters.

Finally, the paper evaluates two state-of-the-art malware detection systems on the identified covert-channel malware and reports that both fail badly, with false rates exceeding 50%. This result supports the claim that current detectors are not robust to malware that hides command-and-control communication through anonymity or tunneling infrastructure.

### 🌱 Insights & Implications
A central insight is that covert communication is no longer a niche tactic in Android malware but a rapidly growing and adaptive design choice. The strongest evidence is the dramatic increase in prevalence over time and the observation that malware families do not merely adopt one covert channel statically; they evolve, diversify, and switch channels in ways that likely help evade signatures and infrastructure blocking.

Another important takeaway is methodological: generic metadata sources such as VirusTotal reports are highly unreliable for discovering covert-channel malware at scale, while careful channel-specific program analysis substantially improves coverage. This suggests future malware mining work should use semantically grounded rules and hybrid analysis rather than keyword heuristics.

The paper also shows the practical value of combining static and dynamic analysis. Static rules provide scale across millions of APKs, while device-based dynamic execution provides stronger behavioral evidence and exposes runtime artifacts and network endpoints. Still, the study is limited by the usual constraints of malware execution: many samples fail to run, may require unavailable C2 infrastructure, or may suppress behavior under analysis. Thus the reported numbers are likely conservative lower bounds rather than complete coverage.

A further implication is that current malware detectors appear poorly equipped to handle communication stealth. This opens research questions on how to build detectors that reason about embedded anonymity libraries, protocol tunneling, covert traffic patterns, and family-level adaptation over time. Another open question is how many samples remain missed due to custom-modified covert-channel code, stronger obfuscation, dormant logic, or encrypted loading paths beyond the reach of existing library/code matching tools.

For security operations and malware triage, the findings imply that certificate reuse, package-level covert-channel indicators, and network/system runtime artifacts are useful pivots for clustering and hunting. For academic research, the released dataset and code create a foundation for benchmarking covert-channel-aware malware detection, attribution, and repair-oriented response systems.

This paper is highly relevant to the researcher's interests because it sits directly at the intersection of software security, vulnerability and malware detection, and large-scale program analysis. Its hybrid static/dynamic pipeline is especially valuable for researchers working on automated detection or repair because it identifies concrete code- and behavior-level indicators of stealthy malicious communication that existing detectors miss.

For vulnerability detection and repair, the work highlights an important blind spot: malware can evade modern analysis by embedding anonymity and tunneling components, so future tools for detection, containment, or automated remediation need to explicitly model covert-channel dependencies and runtime behavior.

## 📄 [Smarter Saboteurs, Better Fixers: Scaling & Security in Linear Multi-Agent Workflows](https://arxiv.org/abs/2606.12709v1)  `60% relevant`  `🌐 Full Text`

### 💡 TL;DR
This paper studies how scaling affects the security of strictly linear LLM multi-agent software pipelines under a compromised coding agent. It finds a strong compliance-correction symmetry: larger models become much better saboteurs in uncorrected pipelines, but a lightweight terminal QA+Fixer stage nearly eliminates the damage and restores performance to control levels, even at 27B scale.

### 🏆 Key Contributions
- Demonstrates that model scaling can substantially worsen resilience in uncorrected linear multi-agent workflows because larger models more faithfully execute malicious role instructions.
- Introduces and evaluates a strictly linear QA+Fixer extension that preserves topology while adding downstream correction capability.
- Shows that the apparent brittleness of linear MAS may be due less to topology itself and more to the absence of an effective correction stage.
- Provides a controlled empirical study across two open-weight model families and multiple scales on HumanEval under both benign and malicious Engineer conditions.
- Establishes statistical evidence that large malicious performance drops in no-fix pipelines vanish after adding QA+Fixer, with no significant control-malicious gap after correction.

### ⚙️ Methodology
The authors build a MetaGPT-inspired but simplified strictly linear software-development pipeline: Product Manager -> Architect -> Project Manager -> Engineer, with each role executed once and each role seeing the concatenated prior outputs through shared message memory. They remove file-system, git, and tool-use integrations so that outcomes depend on prompting and model behavior rather than external scaffolding. They study four configurations formed by two axes: prompting style and correction design. For prompting, they compare native MetaGPT role prompts against simplified prompts with prefix forcing to reduce small-model derailment. For correction, they compare a No-Fix pipeline, where QA only produces a report and the Engineer's code is final, against a QA+Fixer pipeline, where QA emits a structured status report and, if issues are found, a terminal Fixer Engineer patches the code; this preserves a strictly linear topology. To simulate adversarial compromise, they replace the benign Engineer role with a fixed hand-curated malicious profile instructing the Engineer to insert subtle semantic bugs while preserving function signatures and syntax. This design avoids variability from runtime-generated malicious prompts and isolates scaling effects on willingness and ability to sabotage. They evaluate all combinations of model family, model size, scenario, and pipeline configuration on the full HumanEval benchmark using deterministic greedy decoding, measuring whether the final produced code passes the reference tests.

### 🧪 Experiments
The evaluation uses all 164 HumanEval Python tasks. Models come from two open-weight families across five scales each: Qwen 3.5 (0.8B, 2B, 4B, 9B, 27B) and Gemma 3 (0.27B, 1B, 4B, 12B, 27B). For each task they run every combination of model, control vs malicious Engineer, native vs simplified prompting, and No-Fix vs QA+Fixer configuration, with all roles in a run sharing the same model instance. Decoding is greedy with temperature 0 and max_new_tokens 2048 on a single H100 GPU. The main metric is Pass@1 on HumanEval, with a 3000ms timeout and code extracted from the Engineer or Fixer output. The central reported statistic is the malicious drop, defined as Control Pass@1 minus Malicious Pass@1. In simplified prompting, uncorrected No-Fix pipelines show scale-dependent vulnerability: for Qwen 3.5 the malicious drop rises from 0.0pp at 0.8B to 53.66pp at 27B, where control is 97.56% and malicious falls to 43.90%; Gemma 3 similarly rises to 27.44pp at 12B and 23.78pp at 27B. With QA+Fixer, these drops collapse dramatically: Qwen 27B improves from a 53.66pp drop to 0.61pp, with 97.56% control vs 96.95% malicious; Qwen 9B also has only a 0.61pp drop; Gemma 27B falls from 23.78pp to 2.44pp. McNemar's exact test on paired outcomes shows significant malicious degradation for larger No-Fix models, while QA+Fixer differences are not statistically significant after Holm-Bonferroni correction.

### 🌱 Insights & Implications
The paper's main insight is that scaling amplifies both attack execution and defense capability, producing a compliance-correction symmetry: larger LLMs more reliably follow malicious sabotage instructions, but they also become much better at detecting and repairing those sabotaged outputs when placed in a correction role. This reframes prior claims that linear MAS topologies are intrinsically brittle; the results suggest that what looks like structural weakness may instead be a missing correction mechanism. For software-engineering MAS, the findings imply that verification-and-repair stages are not optional safety extras but core architectural components. A limitation is the focus on one topology class, one compromised role, one benchmark, deterministic decoding, and a fixed hand-written malicious profile rather than broader adaptive attacks. The benchmark also measures functional correctness on HumanEval rather than stealth, exploitability, long-horizon collaboration failures, or real-world software security properties. Open questions include whether the same recovery holds for stronger adaptive saboteurs, multiple compromised agents, non-linear or tool-augmented workflows, larger project contexts, and security-specific tasks such as vulnerability insertion and repair.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it studies adversarial failures in LLM-driven software-development workflows and shows that a lightweight reviewer-repairer stage can dramatically restore correctness under attack.
