## 📄 [Tailored Prompts, Targeted Protection: Vulnerability-Specific LLM Analysis for Smart Contracts](https://arxiv.org/abs/2605.03697v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Xing Zhang, Keyu Zhang, Taohong Zhu, Anbang Ruan
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
The paper presents a practical LLM-based framework for smart contract vulnerability detection that combines AST-guided context extraction with vulnerability-specific prompts and few-shot chain-of-thought prompting. Using a newly released real-world dataset of 31,165 annotated cases from 3,200+ projects across 15 blockchain platforms, it reports strong average detection performance with 0.92 positive recall and 0.85 negative recall across 13 vulnerability categories.

### 🏆 Key Contributions
- A large-scale, professionally annotated smart contract vulnerability dataset with 31,165 instances from over 3,200 real-world projects spanning 15 major blockchain platforms, released as an evaluation resource.
- An end-to-end LLM-based vulnerability detection framework that uses project preprocessing and AST-based structural analysis to extract precise, vulnerability-relevant context rather than sending whole codebases to the model.
- Customized vulnerability-specific prompt templates for 13 prevalent smart contract vulnerability categories, improving adaptability over one-size-fits-all prompting.
- Use of few-shot in-context learning with chain-of-thought prompting to enhance general-purpose LLM reasoning for smart contract auditing without task-specific fine-tuning.
- Empirical evidence that careful context construction and tailored prompts can achieve high practical effectiveness, with average positive recall of 0.92 and average negative recall of 0.85.

### ⚙️ Methodology
The framework is organized around three goals: defining what an LLM detector must understand, delivering precise context, and enhancing reasoning for vulnerability detection. First, the authors build a real-world corpus from major smart contract auditing platforms across EVM-compatible chains and analyze vulnerability patterns and auditing requirements. For detection, they start with project preprocessing to filter a project’s contracts down to security-relevant components, reducing noise from entire codebases. They then perform AST-based code analysis to recover structural relationships and extract vulnerability-critical program information, producing a structured representation that captures contract organization, syntactic structure, and semantically relevant evidence for a target vulnerability. Instead of using a universal prompt, they instantiate customized detectors for 13 vulnerability categories through vulnerability-aware prompt design. The structured code context is assembled into prompts together with category-specific instructions. To improve reasoning by general-purpose LLMs, the system further augments prompts with few-shot in-context examples and chain-of-thought style guidance so the model analyzes evidence step by step before outputting a decision. The overall pipeline therefore consists of data collection and empirical analysis, project preprocessing, AST-based context construction, prompt assembly and enrichment, LLM inference, and structured vulnerability report generation.

### 🧪 Experiments
The evaluation is grounded in a newly constructed dataset containing 31,165 professionally annotated vulnerability instances from more than 3,200 real-world smart contract projects across 15 major blockchain platforms, including Ethereum and Binance Smart Chain. The method targets 13 common vulnerability categories and evaluates practical detection behavior in terms of both finding vulnerable cases and correctly rejecting benign ones. The paper emphasizes recall-oriented results, reporting an average positive recall of 0.92 and an average negative recall of 0.85 across categories; it also states that the approach can achieve up to 0.90 recall on real-world test contracts for some settings/categories. The experimental comparison appears centered on prompt and context design choices rather than fine-tuned models, showing that AST-based precise context delivery plus vulnerability-specific prompting and few-shot CoT prompting materially improves real-world LLM vulnerability analysis. The provided excerpt does not include full baseline tables, exact model names, or per-category metric breakdowns, but the central quantitative result is strong balanced effectiveness on realistic annotated data.

### 🌱 Insights & Implications
A key insight is that for LLM-based security analysis, context quality matters more than simply providing more code: precise AST-filtered, vulnerability-relevant evidence helps reduce distraction and improves reasoning. Another important takeaway is that smart contract vulnerabilities are heterogeneous enough that category-specific detectors and prompts are preferable to generic auditing prompts. The work also suggests that general-purpose LLMs can be made practically useful for security auditing through lightweight prompt engineering and few-shot reasoning, avoiding the cost of task-specific fine-tuning. For software security research, this points toward a hybrid direction combining static program analysis for context selection with LLM reasoning for semantic judgment. Limitations and open questions include the incomplete details in the excerpt about exact baselines, precision/false-positive tradeoffs beyond negative recall, model robustness across unseen platforms or newly emerging vulnerability classes, and whether chain-of-thought prompting remains stable and reproducible across models. Another open issue is how this approach extends from detection to actionable repair or patch generation, which is highly relevant for automated remediation pipelines.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and automated analysis because it proposes a scalable LLM-plus-program-analysis framework for finding real security flaws in smart contracts, a security-critical software domain. It is especially useful for researchers interested in vulnerability detection and repair workflows: the AST-based context construction, vulnerability-specific prompting, and large annotated dataset can inform similar approaches for secure code review, neural/static hybrid analyzers, APR systems that need precise bug localization context, and future work on vulnerability-specific repair generation.

## 📄 [ARISE: A Repository-level Graph Representation and Toolset for Agentic Fault Localization and Program Repair](https://arxiv.org/abs/2605.03117v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Shahd Seddik, Fatemeh Fard
📅 **Published:** 2026-05-04  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
ARISE augments an LLM-based repair agent with a repository-level program graph that includes statement nodes and intra-procedural def-use edges, exposed through query tools for structural navigation and data-flow slicing. On SWE-bench Lite, this improves fine-grained fault localization substantially and raises end-to-end repair Pass@1 from 17.3% to 22.0% over the SWE-agent baseline.

### 🏆 Key Contributions
- A multi-granularity repository graph that links files, classes, functions, and statements, and adds statement-level def-use relations to capture intra-procedural value flow.
- A three-tier tool API that lets an agent query the graph at architectural, code-unit, and statement/data-flow levels, making backward, forward, and bidirectional slicing a first-class primitive.
- An agentic APR integration of the graph and tools into SWE-agent, showing that structured data-flow evidence improves function- and line-level fault localization under fixed context budgets.
- Controlled ablations demonstrating that the main gains come from the data-flow graph itself rather than merely adding extra tools, and that raw structured slice outputs are sufficient without natural-language summarization.
- A framework-agnostic graph builder and slicing toolset designed as a reusable plug-in for future repository-level debugging and repair research.

### ⚙️ Methodology
ARISE is built by extending SWE-agent with a repository analysis backend and graph-query tools. The system first statically parses a Python repository and constructs a heterogeneous, multi-granularity program graph. At the structural level, the graph contains repository entities such as files, classes, functions, imports, and call/dependency relations. At the finer level, each function is decomposed into statement nodes, and intra-procedural definition-use edges are added between statements to represent where variables are defined and where those values are later consumed. This design intentionally targets the semantic gap left by structure-only graphs: an agent can not only find a relevant function, but also ask which statements influence or depend on a variable involved in the bug.

The graph is exposed to the LLM through a three-tier tool API. Tier 0 provides high-level repository and architectural navigation, such as listing modules or traversing broad structural relations. Tier 1 supports code-unit retrieval and local neighborhood exploration around files, classes, and functions. Tier 2 provides statement-level reasoning, especially data-flow slicing. The slicing tool accepts a variable or statement context and a slice direction (backward, forward, or bidirectional) and returns structured outputs identifying the relevant defining and consuming statements. This makes data-flow slicing an explicit agent action rather than a hidden preprocessing step.

Within the agent loop, the backbone model uses ordinary SWE-agent capabilities for planning, navigation, editing, and testing, but can now call ARISE tools to iteratively narrow the search space. A typical workflow is: inspect issue description, query the repository graph for candidate modules/functions, retrieve local structural context, invoke slicing on suspicious variables or statements, inspect the resulting slice, then edit the implicated code and validate with tests. The tool outputs are structured rather than free-form prose; one ablation tests whether a natural-language summarization layer is needed and finds it is not. The overall design is decoupled from SWE-agent internals so that other tool-using APR agents can adopt the graph builder and API.

### 🧪 Experiments
The evaluation uses SWE-bench Lite with 300 real GitHub issues drawn from 11 Python repositories, and Qwen2.5-Coder-32B-Instruct as the backbone model. The primary comparison is against the unmodified SWE-agent baseline. The paper studies both fault localization and end-to-end repair, motivated by three research questions: whether ARISE improves localization, whether those gains translate into repair, and which components drive the gains.

For localization, the key metrics are Function Recall@1 and Line Recall@1, measuring whether the top-ranked localized function or line matches the ground-truth buggy region. ARISE improves Function Recall@1 by 17.0 percentage points and Line Recall@1 by 15.0 points over SWE-agent, indicating that the graph tools especially help at the fine granularity where prior systems struggle.

For repair, the main metric is Pass@1 on SWE-bench Lite, based on whether the generated patch passes the benchmark’s validation tests. ARISE achieves 22.0% Pass@1, corresponding to 66 solved issues out of 300, compared with 17.3% for SWE-agent, a gain of 4.7 percentage points. The paper argues that this improvement is directly explained by stronger localization, consistent with prior work identifying localization as the main bottleneck in APR.

The ablations isolate the effect of ARISE components. A key result is that adding the tool schema alone does not explain the improvement; the substantial gains appear when the data-flow graph and slicing capability are present. Another ablation evaluates an optional natural-language explanation layer for slice results and finds that large code models can reason effectively over the structured slice output directly, so summarization is unnecessary overhead. The paper also contrasts text-only or structure-only conditions against the full ARISE system to show that statement-level def-use information is the decisive factor.

### 🌱 Insights & Implications
The paper’s main insight is that repository-level APR is constrained less by raw model size than by the precision of the evidence presented to the model. Structural repository graphs help with navigation, but they are insufficient for function- and line-level localization because they do not explain how values propagate within procedures. Adding statement-level def-use edges closes this gap and yields measurable downstream repair gains.

A second insight is methodological: program analysis outputs should be exposed as queryable agent primitives, not merely precomputed features. ARISE shows that interactive slicing is useful because an agent can ask targeted questions during debugging rather than ingesting a monolithic graph dump. This is relevant to broader agentic software engineering, including vulnerability triage and root-cause analysis.

The paper also suggests that modern code LLMs can consume compact structured analysis outputs directly, reducing the need for handcrafted natural-language adapters. This has implications for designing future tools that trade prose for machine-readable evidence.

Limitations include the current focus on Python, intra-procedural data flow rather than richer inter-procedural or alias-sensitive analyses, and evaluation only on SWE-bench Lite rather than full SWE-bench or security-specific benchmarks. Because the graph is statically built, dynamic behaviors, reflection, and highly framework-dependent flows may still be under-modeled. Open questions include how to extend ARISE to inter-procedural slicing, cross-language repositories, vulnerability-focused repair, and integration with symbolic or execution-based evidence.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, and vulnerability-related research because it directly tackles the hardest part of repository-scale bug fixing: precise fault localization across files and down to the statement level. Its graph-plus-slicing design is particularly valuable for researchers working on APR and vulnerability detection/repair, since def-use reasoning and slice extraction are natural foundations for tracing tainted values, identifying root causes, and producing targeted fixes. The framework-agnostic toolset also makes it a practical building block for extending existing agentic repair systems with richer static analysis rather than relying only on text retrieval or long-context prompting.

## 📄 [KVerus: Scalable and Resilient Formal Verification Proof Generation for Rust Code](https://arxiv.org/abs/2605.03822v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Yuwei Liu, Xinyi Wan, Yanhao Wang, Minghua Wang, Lin Huang, Tao Wei
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.SE, cs.CR

### 💡 TL;DR
KVerus is a retrieval-augmented, self-adaptive system for generating and repairing Verus proofs for Rust that explicitly addresses the mismatch between LLM semantic reasoning and the structural dependency constraints of formal verification. It substantially improves proof success, scalability across multi-file repositories, and resilience to Verus/toolchain evolution, and it was validated with upstream-accepted proofs in the Asterinas OS kernel.

### 🏆 Key Contributions
- Introduces the Semantic-Structural Gap as the core explanation for why prior LLM-based proof generation systems are brittle on large, evolving verification workloads.
- Proposes a self-adaptive verification paradigm that combines dependency-aware retrieval, semantic lemma discovery, and error-driven proof refinement for Verus-based Rust verification.
- Implements KVerus with a dynamic knowledge base capturing code metadata, lemma summaries, and toolchain-specific verification knowledge.
- Demonstrates strong gains over AutoVerus and prompting baselines on single-file, repository-level, and real-world kernel verification tasks, while reducing token cost and improving robustness to breaking Verus updates.
- Produces upstream-accepted verification artifacts for the Asterinas Rust OS kernel and contributions to the Verus standard library, showing real deployment value beyond benchmark success.

### ⚙️ Methodology
KVerus is built as a retrieval-augmented proof synthesis and repair pipeline for Verus-annotated Rust. Its central design principle is to bridge semantic LLM reasoning with the structural constraints of verification by maintaining a dynamic knowledge base and using it throughout generation and repair. The system first performs dependency-aware program analysis to extract metadata such as function signatures, types, call/dependency relationships, and relevant cross-module proof obligations, forming a code metadata knowledge graph that helps localize the subset of the repository relevant to a target verification task. To support lemma reuse, KVerus constructs semantic lemma indexing: existing lemmas are summarized with LLM assistance into concise natural-language descriptions and embedded into a searchable index, enabling retrieval even when code comments or naming are incomplete, outdated, or non-obvious. For proof generation, KVerus uses retrieval-augmented prompting over these knowledge sources, supplying the LLM with structurally relevant dependencies, semantically related lemmas, and toolchain-aware guidance instead of relying on unconstrained generation. When verification fails, KVerus enters an error-driven self-refinement loop. It analyzes Verus/verifier diagnostics, categorizes error types, consults a continuously updated Verus knowledge base that captures syntax rules, common failure modes, and toolchain specifics, and regenerates or repairs the proof accordingly. This makes the system adaptive both to code evolution and to verification-toolchain changes. Overall, the architecture integrates dependency analysis for context selection, semantic retrieval for proof ingredient discovery, and iterative verifier-feedback-guided refinement for resilient synthesis and maintenance of proofs in modular Rust codebases.

### 🧪 Experiments
The evaluation covers three single-file benchmarks, three repository-level benchmarks with cross-file dependencies, and a real-world case study on the Asterinas Rust OS kernel memory-management module. The main comparison on single-file tasks is against AutoVerus, the stated state of the art. On these benchmarks, KVerus verifies 80.2% of tasks versus 56.9% for AutoVerus, an absolute improvement of 23.3 percentage points, while also reducing token cost by more than 50%. The paper also evaluates robustness under breaking Verus updates and reports that KVerus degrades less than AutoVerus, showing stronger tolerance to toolchain evolution. For repository-level benchmarks, where proofs require navigating cross-file and cross-module dependencies, KVerus achieves a 51.0% success rate, whereas a multi-round prompting baseline reaches only 4.5%, indicating that naive iterative prompting does not scale to structurally complex verification. In the Asterinas case study, KVerus outperforms strong prompting baselines by roughly 8x and successfully verifies 23 previously unverified functions, corresponding to 21.0% of proof code in the target memory-management module. The resulting proofs were accepted upstream in the kernel mainline, and some supporting proof artifacts were accepted into the Verus standard library, providing strong external validation of practical utility.

### 🌱 Insights & Implications
The paper’s main insight is that verification failures in LLM-driven systems are not just due to weak models or insufficient prompting, but to a deeper mismatch between semantic pattern completion and the rigid structural nature of formal proof dependencies. This framing is especially important for software maintenance and verification sustainability: proofs must evolve with the code and with verifier/toolchain changes, so proof generation should be treated as a continual adaptation problem rather than a one-shot synthesis problem. The results also suggest that retrieval quality and structural context selection are decisive for multi-module verification, where baseline prompting nearly collapses. Another notable implication is for proof maintenance in security-critical software: automated repair of broken proofs after toolchain updates can reduce the long-term engineering burden that often blocks industrial adoption. Limitations and open questions include how well the approach generalizes beyond Verus and Rust, how much manual effort is needed to maintain or curate the dynamic knowledge base in very large projects, whether semantic lemma summaries remain reliable as repositories evolve, and how performance scales to larger repositories or harder proof obligations requiring deeper theorem discovery. There is also room to study richer integration with automated program repair, since verifier-feedback-driven proof repair resembles maintenance-oriented patch generation under formal constraints.

### 🎯 Why This Paper
This paper is highly relevant because it sits at the intersection of software engineering, software maintenance, security-critical verification, and automated repair. It advances the researcher’s interests by showing how LLMs can be made practical for scalable proof generation and proof repair in evolving Rust systems, especially for real-world codebases with cross-module dependencies and changing toolchains. Its self-adaptive, verifier-feedback-driven design is directly relevant to automated vulnerability prevention and repair workflows, where maintaining correctness guarantees over time is as important as generating an initial proof.

## 📄 [From TinyGo to gc Compiler: Extending Zorya's Concolic Framework to Real-World Go Binaries](https://arxiv.org/abs/2605.03492v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Karolina Gorna, Nicolas Iooss, Yannick Seurin, Rida Khatoun, Keith Makan
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.CR, cs.SC, cs.SE

### 💡 TL;DR
This paper extends the Zorya binary-level concolic execution framework from simplified TinyGo programs to real-world Go binaries compiled with the standard gc compiler, including multi-threaded runtime support and bounded overlay exploration of untaken branches. On 11 real vulnerabilities from major Go projects, Zorya detects 7 bugs at the binary level, notably finding a silent integer overflow missed by all other evaluated tools unless given a manual oracle.

### 🏆 Key Contributions
- Extension of Zorya to analyze gc-compiled Go binaries with the full Go runtime instead of only single-threaded TinyGo binaries
- Recovery and management of multi-threaded execution state from gdb dumps, including support for Go runtime startup behavior and OS-thread restoration
- Neutralization of Go runtime preemption so concolic execution can progress deterministically through gc binaries
- Introduction of overlay path analysis with copy-on-write semantics to explore untaken branches cheaply and detect silent vulnerabilities without full state forking
- Creation of a dataset of 11 real-world Go vulnerabilities from production systems such as Kubernetes, Go-Ethereum, and CoreDNS, with reproduction workflows and inputs
- Comparative evaluation against seven tools showing Zorya detects 7/11 vulnerabilities, including a silent integer overflow not found by competing tools without a custom oracle

### ⚙️ Methodology
The paper extends Zorya, a concolic execution engine that lifts x86-64 binaries to Ghidra P-Code and executes them with both concrete values from a gdb memory dump and symbolic expressions in Z3. The main engineering challenge is supporting standard Go gc binaries, whose runtime creates multiple OS threads, uses goroutine scheduling, asynchronous/cooperative preemption, and invokes runtime-specific mechanisms absent in TinyGo. To address this, the authors restore thread states from gdb dumps so the executor can resume a realistic post-startup process image, and they neutralize runtime preemption to avoid scheduler-induced divergence during analysis. The core new analysis technique is overlay concolic execution: when a conditional depends on tracked symbolic variables, Zorya does not fork the full program state. Instead it creates a lightweight copy-on-write overlay for the untaken branch, where reads consult overlay state first and fall back to the base state, while writes affect only the overlay. This is important because gc binaries can have heap states exceeding 1 GiB, making full cloning too expensive. Overlay execution saves mutable executor state such as temporary variables, instruction pointer, call-stack metadata, freed stack-frame tracking, and null-check caches; redirects RIP to the untaken branch; executes up to a bounded number of P-Code blocks (default 15); checks vulnerability patterns before each instruction; and stops on vulnerability, return, loop, or depth bound, after which the overlay is discarded and the original state restored. The vulnerability checks currently target null-pointer dereferences and integer overflows, with division-by-zero and use-after-freed-stack-frame checks mentioned as implemented but not yet evaluated. For Go-specific bug discovery, Zorya also uses AST-guided exploration from branch-corresponding nodes to search for panic, fatal, or abort sites, and combines this with solver feasibility checks to confirm whether symbolic inputs can drive execution into those faulty regions. The overlay analysis is applied to both taken and untaken paths, enabling discovery of silent vulnerabilities that do not necessarily manifest as runtime panics under the concrete execution.

### 🧪 Experiments
The evaluation uses a corpus of 11 real-world vulnerabilities collected from production Go projects including Kubernetes, Go-Ethereum, and CoreDNS, with provided reproduction workflows and triggering inputs. Zorya is compared against seven representative tools: three static analyzers (including tools such as go vet, staticcheck, gosec, govulncheck, or nilaway as discussed in the paper), two fuzzers (Go's built-in go test -fuzz and GoLibAFL), and two binary-level symbolic executors (BINSEC and SymQEMU). The evaluation is performed at the binary level on gc-compiled programs rather than source-instrumented toy examples. The key result is that Zorya detects 7 of the 11 vulnerabilities. Its standout result is detection of a silent integer overflow in Go-Ethereum that none of the other evaluated tools finds unless supplied with a manually written oracle or harness condition. The paper also positions overlay execution as a practical bounded alternative to full path forking for large Go runtime states, and earlier Zorya work cited in the paper reports a 1.8x to 3.9x speedup from panic-reachability gating, which complements the new overlay mechanism in the full system.

### 🌱 Insights & Implications
The paper shows that binary-level concolic analysis of real Go executables is feasible if the runtime is treated as part of the analysis problem rather than ignored. A major insight is that for Go, practical vulnerability detection depends less on modeling arbitrary thread interleavings and more on reconstructing and stabilizing the runtime state sufficiently to reach user code. The overlay mechanism is also an important design point: many security-relevant bugs in Go, especially integer overflows, are silent by language semantics and may lie on untaken branches, so bounded copy-on-write exploration provides a useful middle ground between concrete execution and full symbolic path explosion. The results suggest that source-level static analyzers and fuzzers miss classes of semantic bugs when they lack tailored oracles, while binary-level concolic execution can reason about low-level arithmetic and memory effects directly. Limitations include reliance on correct Ghidra disassembly, halting when jumps target unidentified code, support currently limited to non-interactive binaries whose inputs are available during initialization, and bounded overlay depth that may miss deeper bugs beyond the 15-block exploration window. The evaluation scale is still modest at 11 vulnerabilities, and the framework does not appear to focus on concurrency bug detection despite handling multi-threaded runtime startup. Open questions include how well the approach scales across Go versions and architectures, whether scheduler behavior can be modeled without neutralization, how to extend the vulnerability patterns beyond null dereferences and overflow, and whether the recovered runtime state could support automated exploit or patch generation.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance, software security, vulnerability detection, and automated repair research because it tackles real-world binary analysis of Go programs used in production infrastructure, where source-level assumptions often fail. It advances the researcher's work by providing a practical concolic framework for detecting hard-to-find vulnerabilities such as silent arithmetic errors and panic-reachable faults directly in compiled binaries, along with a realistic benchmark dataset and design ideas—copy-on-write overlay exploration, runtime-state recovery, and branch feasibility reasoning—that can inform future work on binary vulnerability repair, oracle-free bug detection, and security analysis pipelines for modern managed runtimes.

## 📄 [Generating Proof-of-Vulnerability Tests to Help Enhance the Security of Complex Software](https://arxiv.org/abs/2605.03956v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Shravya Kanchi, Xiaoyan Zang, Ying Zhang, Danfeng Yao, Na Meng
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper presents PoVSmith, an agent-based system that automatically generates proof-of-vulnerability JUnit tests showing how vulnerable third-party library APIs can be exploited through application code. On 33 Java application-library pairs covering 20 CVEs, it correctly identified 96% of app entry-point-to-vulnerable-API call paths and produced 84 successful exploit-triggering tests, substantially outperforming prior LLM-based work.

### 🏆 Key Contributions
- PoVSmith, a four-phase agent/LLM pipeline for security-oriented proof-of-vulnerability test generation for applications using vulnerable dependencies
- An agent-based call-path analysis method that identifies public application entry points and complete paths to vulnerable library APIs
- An iterative test-generation and refinement workflow that uses exemplar tests, application code context, and execution/build feedback to improve generated PoV tests
- An LLM-based test assessment stage that judges whether a generated test actually triggers the target vulnerability using both test code and execution logs
- A benchmark dataset of 33 Java <App, Lib> pairs with 207 verified call paths spanning 20 CVEs
- Evidence of practical security impact, including newly uncovered vulnerability demonstrations that motivated six pull requests and six CVE requests

### ⚙️ Methodology
PoVSmith is organized into four phases. Phase I performs attack-surface discovery via agent-based call-path analysis. Given a vulnerable library API as a sink, the system prompts Codex with project code context and structured instructions to find public application methods that directly or indirectly reach the sink and to enumerate complete source-to-sink call paths. The authors chose this agent-based source-level analysis after finding traditional analysis tools such as WALA, Soot, and CodeQL difficult to use effectively for their target settings.

Phase II generates PoV tests as JUnit tests. For each discovered application entry point, PoVSmith constructs multiple prompts that combine: the vulnerable API and call path information from Phase I, a user-provided exemplar test illustrating the vulnerability style or expected exploit pattern, relevant application code context, and prior execution/build feedback. Codex is asked to synthesize a test that invokes the public application entry point in a way that propagates malicious input to the vulnerable library API. The generation is iterative: the agent creates a candidate test, inspects compilation or runtime failures, diagnoses likely causes such as missing setup, incorrect object construction, unsatisfied data/control preconditions, or wrong assertions, and refines the test accordingly.

Phase III automatically compiles and executes the generated test, producing logs that record build status, test outcomes, exceptions, and vulnerability symptoms.

Phase IV uses GPT to assess test quality. Instead of relying only on whether the test ran, it feeds both the generated test and the execution logs into an LLM prompt so the model can decide whether the observed behavior constitutes a successful vulnerability trigger, a non-triggering execution, or another outcome. This adds a semantic assessment layer over raw execution signals. The overall design explicitly addresses four challenges: identifying app entry points that expose vulnerable library behavior, generating exploit logic, adapting the test to contextual constraints of the target application, and automatically evaluating whether the generated artifact is a valid PoV.

### 🧪 Experiments
The evaluation uses 33 Java application-library pairs in which each application depends on a vulnerable library version; the benchmark covers 20 CVEs and includes 207 verified call paths. The study measures Phase I call-path discovery accuracy, Phase II/III test generation and execution success, and Phase IV assessment correctness. The main outputs are public application entry points reaching vulnerable APIs, generated JUnit PoV tests, and labels indicating whether the vulnerability was successfully triggered.

For call-path analysis, PoVSmith found 158 unique public application entry points that call vulnerable library APIs, and 152 of them were correct with correctly recognized call paths, yielding about 96% correctness. For test generation, PoVSmith generated 152 tests corresponding to those call paths. Of these, 141 compiled successfully, and 84 successfully triggered the target vulnerabilities, meaning 55% of generated tests demonstrated feasible application-level attacks through the dependency vulnerability. For test evaluation, the LLM-based assessment phase correctly labeled 103 tests.

The authors also compared different coding agents, including Codex, Gemini Code Assist, and Mistral Vibe, and found Codex to be the most effective. They further compared PoVSmith against a prior state-of-the-art LLM-based PoV-generation approach and report that PoVSmith substantially outperformed it by requiring less human involvement and producing many more high-quality vulnerability-triggering tests.

### 🌱 Insights & Implications
A central insight is that vulnerable dependency alerts become much more actionable when converted into concrete, executable application-level exploit tests. The paper shows that raw vulnerability reachability is not enough for developers; they need evidence that an application entry point can actually drive execution into a vulnerable library API with the right state and inputs.

Methodologically, the work suggests that agentic workflows can outperform single-shot LLM prompting for complex software security tasks because they interleave code understanding, synthesis, execution, diagnosis, and repair. The use of exemplar tests and call-path context appears especially important for grounding generation in realistic exploit structure. The feedback loop also addresses a common weakness of code-generation systems: inability to satisfy hidden application-specific preconditions without iterative repair.

The paper also implies that traditional static analysis alone is insufficient for this task: even if one can identify vulnerable API reachability, constructing a working exploit test still requires semantic adaptation to framework conventions, object lifecycles, and control/data constraints. Conversely, pure LLM generation without structural guidance performs poorly. The strongest results come from combining program-structure evidence with agentic refinement.

Limitations and open questions include the moderate end-to-end success rate: while many tests compile, only 84 of 152 generated tests actually trigger vulnerabilities, leaving substantial room for improvement. The approach currently targets Java and JUnit-based settings, depends on a user-provided exemplar test, and uses proprietary models/agents whose behavior may vary over time. The correctness of LLM-based evaluation is also imperfect, as shown by 103 correctly labeled tests rather than full agreement. Future research could improve assessment reliability, reduce dependence on exemplars, extend the method to other languages and ecosystems, integrate stronger static/dynamic analysis, and connect PoV generation with automated vulnerability repair or patch validation.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, software security, vulnerability detection, and vulnerability repair because it operationalizes dependency-vulnerability analysis into executable tests that developers can use for triage, validation, regression testing, and potentially patch verification. For research on automated program repair and vulnerability repair, PoVSmith provides a valuable bridge artifact: automatically generated exploit tests that can serve as failing security test cases, helping prioritize fixes and evaluate whether repairs truly eliminate exploitability in real application contexts.

## 📄 [MOSAIC-Bench: Measuring Compositional Vulnerability Induction in Coding Agents](https://arxiv.org/abs/2605.03952v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Jonathan Steinberg, Oren Gal
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
This paper introduces MOSAIC-Bench, a benchmark for testing whether coding agents can be manipulated into introducing exploitable vulnerabilities through sequences of benign-looking engineering tickets rather than explicit malicious prompts. ### 🏆 Key Contributions
- Introduces MOSAIC-Bench, a benchmark of 199 three-stage compositional attack chains spanning 10 web applications, 31 CWE classes, and 5 programming languages.

The benchmark unit is a 3-stage attack chain consisting of three Jira-style engineering tickets, a composed implementation, a deterministic exploit oracle, and metadata such as CWE, language, framework, and application. The main insight is that coding-agent safety mechanisms are often shaped around detecting explicit malicious intent in single prompts, but software engineering workflows naturally permit vulnerabilities to emerge compositionally from sequences of harmless-seeming tasks. The benchmark is also curated for product-legible attack chains rather than estimating prevalence over all possible engineering tasks, so reported attack success rates are conditional on plausible staged scenarios. ### ⚙️ Methodology

The diagnosis process is formalized as a finite-horizon sequential decision problem over diagnostic states, actions, transitions, and rewards. The results also suggest that multi-factor root causes remain difficult for a framework that ultimately outputs a root cause per anomaly. Open questions include how to better model interacting causes, how to reduce token and time cost through pruning or learned policies, how to incorporate traces and richer topology/dependency information, and how to make diagnostic scoring more robust and less reliant on subjective LLM reflection.

The system is formulated as a discrete-time nonlinear feedback loop over an adversarial attack graph G(k+1)=f(G(k),u(k),w(k)) with observations y(k)=h(G(k),w(k)). The plant is a directed attack graph whose edges represent attacker techniques over hosts and attack stages, each edge carrying payoff, block probability, detect probability, and defender uncertainty P_e(k). The defender has only a belief graph G-hat(k) derived from telemetry, while the attacker is assumed to observe the true graph. This creates asymmetric observability, with matched-alert edges initialized with low uncertainty and unseen dark edges absent from the defender belief until revealed through transitions.

Defense is modeled as a Stackelberg leader-follower game with double-oracle expansion. The defender chooses policy deployments from a finite catalog C under a per-round budget B, including block, detect, and logging actions spanning endpoint, identity, and cloud controls. The attacker best-responds by selecting surviving paths to maximize expected payoff. The attacker game value S(k) is defined as the maximum over ENTRY-to-OBJECTIVE paths of path survival probability times the maximum stage payoff along the path, yielding a path-based network interdiction objective.

The key architectural constraint is tool mediation. The LLM does not directly inspect raw graphs, compute payoffs, update beliefs, enumerate paths, or generate arbitrary actions. Instead, deterministic tools implement attack-graph primitives, Stackelberg best-response, simulation, critical-path analysis, deployable-policy listing, dark-edge identification, and Bayesian observer updates. The LLM only sequences tool calls, reads structured outputs, and chooses one action per deployment slot from the finite catalog. A mirrored finite tool suite similarly constrains adversary behavior. Catalog-membership enforcement occurs at the tool-output interface, making the architecture stable by construction even if the LLM itself is stochastic.

For formal analysis, the paper defines a composite Lyapunov function V(k)=S(k)+lambda*theta(k), where S(k) captures plant security risk through attacker game value and theta(k) captures observer/belief error under asymmetric sensing. Theorem 1 establishes three properties: monotone decrease in defender-controlled game value (controllability), ISS-style bounded degradation under adversarial disturbance and strategy-set expansion, and convergence of the defender observer despite partial telemetry. Two corollaries then show that once catalog and tool constraints are enforced, the stability certificate transfers from a specific LLM instance to any controller or adversary operating within those catalogs.

Baselines and evaluation dimensions therefore include: deterministic greedy defense as the principal baseline, two LLM controllers of differing capability, repeated trials across multiple temperatures, attacker game value, run-to-run variance, catalog-boundedness, and belief-alignment/stability behavior on real attack graphs.

### 🌱 Insights & Implications
The paper’s main insight is that formal guarantees for agentic cyber defense may be achieved by shifting the burden of safety from the LLM to the surrounding control architecture. Rather than trying to prove properties of an unconstrained language model, the authors prove properties of a closed loop in which the LLM can only compose deterministic, verifiable tools and select from finite actions. This is a strong architectural lesson for high-stakes software and security agents.

A second insight is the useful synthesis of control theory, game theory, and tool-using LLMs. The attacker is not treated as random noise but as an intelligent best-responding disturbance, and the defender operates under realistic partial observability from asymmetric telemetry. Another limitation is that the architecture guarantees stability and bounded behavior, not global optimality; Haiku’s weaker payoff results illustrate this tradeoff. It is also open how the approach generalizes to richer software engineering tasks such as code repair or patch generation, where action spaces are combinatorially larger and semantics harder to catalog. Nonetheless, the paper points toward a promising design pattern for certifiable agentic systems: deterministic analyzers plus finite repair/action spaces plus machine-checked closed-loop reasoning.

### ⚙️ Methodology
The paper studies tool-augmented LLM agents in a multi-step execution loop where the model reads the current context, reasons, and emits either tool calls or final answers. The central problem is that for many realistic tasks, the concrete action cannot be inferred solely from the prompt; it depends on runtime observations such as invoices, emails, retrieved documents, memory records, or tool outputs. In this setting, prompt injection is more subtle: malicious instructions are embedded within the very context the agent must read to determine the correct action.

To benchmark this regime, the authors create AgentLure. The benchmark is built around context-dependent tasks, meaning the agent must extract or infer action parameters from runtime context. The attacks are context-aware, meaning payloads are coupled to the carrier content, tailored to the task, and designed to manipulate decision making from within natural task data rather than through obvious override instructions. The benchmark organizes attacks along eight vectors corresponding to eight decision points in the path from prompt understanding to action emission, and instantiates them across multiple attack surfaces.

The proposed defense, ARGUS, is a runtime decision auditor placed around the agent’s action execution process. Its core data structure is the Influence-Provenance Graph (IPG). Each node corresponds to a runtime context entry, and entries are segmented into benign and anomalous spans so the system can reason below observation level. This is important because a single document may contain both legitimate task evidence and a small malicious injection.

When the agent proposes a state-changing tool call, ARGUS performs two levels of auditing. First, a local provenance audit traces each tool-call argument back to the exact spans that support it. The tracing is designed to capture different ways values enter arguments: direct copying from text, normalized transformations, derived values, and resolved references. This allows ARGUS to determine whether arguments such as recipient, amount, or tool type are grounded in benign content or influenced by suspicious spans.

Second, ARGUS performs a task-level audit. It derives invariants from the user query before any external context is read, then checks whether the proposed action is consistent with those invariants and whether the action is entailed by trusted evidence accumulated in the IPG. This avoids the main weakness of prompt-only defenses: in context-dependent tasks, the prompt may authorize only a broad goal, while the safe concrete action must be justified by runtime evidence.

ARGUS combines the results of local provenance analysis and global verification to decide whether to allow the action. If a tool call appears risky and fails global verification, ARGUS blocks execution and returns a benign data hint extracted from trusted spans, giving the agent a chance to retry using safe values instead of simply failing the task. This design aims to simultaneously preserve utility and stop malicious context from driving state-changing actions.

Conceptually, the method addresses three shortcomings of prior defenses: it anchors validation in trusted runtime evidence instead of only the prompt, it selectively promotes benign runtime content into usable evidence rather than treating all external data as untrusted, and it performs argument-to-span provenance analysis rather than coarse document-level trust assessment.

The paper reports that prior defenses, though often strong on conventional context-insensitive benchmarks, struggle on AgentLure’s context-dependent and context-aware attacks. The evaluation further includes per-vector analysis and likely domain/surface breakdowns, demonstrating that the difficulty arises from context-aware manipulations at different decision points rather than from a single narrow attack pattern.

The paper’s main insight is that prompt-injection defense for LLM agents cannot be reliably evaluated or solved under a prompt-only view of intent. In realistic agent deployments, the prompt often specifies only a high-level goal, and safe execution depends on extracting correct values from runtime context. Real workflows require agents to rely on external documents and tool outputs, so an effective defense must distinguish benign and malicious spans within the same artifact and reason about exactly how those spans influence action parameters.

This has implications for future evaluation standards in LLM agent security: benchmarks should model dynamic context, blended malicious content, and task utility under attack, not just static prompt adherence.

To instantiate this axis, the authors aggregate 3,133 prompts from four public sources: RMCBench, MalwareBench, a filtered subset of CySecBench, and the harmful_behaviors set from AdvBench. The pipeline has four main stages. First, source-level deduplication and normalization are applied to merge heterogeneous benchmark formats and remove obvious duplicates or formatting inconsistencies. Second, rule-based pre-filtering removes prompts that do not fit the intended benchmark scope and performs a final length/metadata quality pass. Third, each remaining prompt is independently labeled by five LLM judges drawn from four vendor families: Anthropic, OpenAI, Google, Zhipu AI, and Alibaba. Each judge emits a binary CODE or KNOWLEDGE label under a shared labeling protocol. Fourth, final labels are assigned using a 3-of-5 majority rule. The authors quantify inter-rater reliability using Fleiss' kappa across the five judges and bootstrap 95% confidence intervals, and they also retain per-prompt agreement information such as unanimity level. The result is a consensus-labeled artifact intended as a reusable benchmark substrate rather than a behavior study in itself. The paper explicitly limits its claims to the validity and reliability of the classification axis and does not itself test whether the axis induces differential refusal behavior in target models; that question is deferred to companion work.

### 🧪 Experiments
The experimental evaluation is a dataset-construction and annotation-reliability study rather than a model performance benchmark. The corpus contains 3,133 prompts consolidated from four public benchmarks/sources: RMCBench, MalwareBench, a filtered CySecBench subset, and AdvBench harmful_behaviors. The annotation setup uses five independent LLM judges spanning Anthropic, OpenAI, Google, Zhipu AI, and Alibaba, each assigning binary CODE/KNOWLEDGE labels. Across all 3,133 prompts, the judges achieve Fleiss' kappa = 0.876 with 95% CI [0.862, 0.888], which the authors interpret as 'almost perfect' agreement. 69.3% of prompts receive unanimous 5-of-5 agreement, and all 3,133 prompts meet the 3-of-5 threshold, meaning no prompts are excluded for ambiguity under the consensus protocol. After final processing, the released artifact contains 1,554 consensus-CODE records (1,551 unique prompt strings) and a 388-prompt consensus-KNOWLEDGE comparison set. The paper does not compare downstream refusal rates, attack success, or generation quality against baselines; its quantitative focus is the robustness and reproducibility of the labeling process itself.

### 🌱 Insights & Implications
The central insight is that malicious-code benchmarking has likely been measuring a mixed construct: executable malware/code synthesis and harmful security explanation are not equivalent tasks and may activate different safety mechanisms in aligned LLMs. By explicitly disentangling them, the paper creates a cleaner basis for studying refusal behavior, over-refusal, alignment tradeoffs, and cross-model comparability. For software security research, this distinction is particularly important because blocking executable payload generation is not the same as blocking vulnerability education, exploit understanding, or defensive analysis; conflating the two can obscure whether a model is dangerous, overly restrictive, or both. A key strength is the high agreement achieved by a diverse LLM-judge panel, suggesting that the CODE/KNOWLEDGE boundary is operationalizable with relatively low ambiguity when carefully specified. Limitations remain: the labels are produced by LLMs rather than human experts, the class taxonomy is binary and may collapse nuanced cases such as dual-use pseudocode, exploit planning, or partial scaffolding, and the paper does not itself establish that this axis predicts materially different refusal behavior in generation models. The prompt bank is also primarily an artifact for evaluation, not a defense mechanism, so future work must test how well it transfers across new model families, multilingual prompts, repository-grounded settings, and agentic multi-turn scenarios. An open question is whether finer-grained taxonomies beyond CODE/KNOWLEDGE would better support vulnerability research, such as distinguishing exploit proof-of-concept generation, malware modification, reconnaissance planning, and benign educational reverse engineering.

## 📄 [Learning Correct Behavior from Examples: Validating Sequential Execution in Autonomous Agents](https://arxiv.org/abs/2605.03159v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Reshabh K Sharma, Gaurav Mittal, Yu Hu
📅 **Published:** 2026-05-04  🏷️ **Categories:** cs.AI, cs.SE

### 💡 TL;DR
The paper proposes a lightweight validation algorithm that learns the essential sequential structure of autonomous-agent executions from only a few successful traces, then checks whether new executions preserve that structure while tolerating benign variations. It combines PTA-based trace merging, dominator analysis, and multimodal LLM-assisted state equivalence, and reports strong bug/false-success detection with as few as three training traces across UI, coding, and robotic scenarios.

### 🏆 Key Contributions
- A three-phase framework that learns a generalized execution model from 2–10 passing traces using Prefix Tree Acceptors, graph merging, and dominator-tree extraction to identify essential versus optional states
- A multi-tiered state-equivalence mechanism that combines perceptual visual similarity metrics with multimodal LLM semantic judgments to merge logically equivalent but visually different states
- A topological subsequence matching validator that accepts non-deterministic executions when essential states appear in the correct order and rejects traces that skip required states or end incorrectly
- Explainable validation outputs with coverage metrics, matched/missing states, and human-understandable failure explanations
- Demonstration of cross-domain applicability to UI testing, code-generation/coding-agent workflows, and robotic or process-execution scenarios with only a handful of examples

### ⚙️ Methodology
The method has three phases. First, the system captures 2–10 known-good execution traces, where each trace is a sequence of observable states plus actions between them. States are domain dependent: screenshots for UI agents, code snapshots for coding agents, or sensor/process states for robots and workflows. Each trace is converted into a Prefix Tree Acceptor (PTA), where nodes are observed states and edges are transitions/actions.

Second, the PTAs are merged into a unified graph that generalizes across non-deterministic variations. State merging uses a three-tier equivalence procedure: (1) fast visual comparisons such as perceptual hash similarity, SSIM, and pixel-change ratio to detect clear matches; (2) when visual similarity is ambiguous, a multimodal LLM is prompted with side-by-side state comparisons to determine whether differences are semantically meaningful or superficial; and (3) the merged graph retains branching structure for alternative paths and convergence points for rejoining executions. After this unified graph is built, dominator analysis is applied. A state d dominates state s if every path from the start state to s must pass through d. By tracing immediate dominators backward from successful terminal states to the start, the algorithm extracts a dominator subtree containing only essential states and transitions. Optional states, such as intermittent loading screens, naturally disappear because they do not dominate success.

Third, a new execution trace is validated against the learned dominator tree. The reference dominator tree is topologically ordered, and the test trace is checked using topological subsequence matching. This allows extra intermediate states in the test trace so long as the essential reference states appear in the correct order. Coverage is computed as matched essential states divided by total reference states. A test passes when coverage exceeds a threshold (typically 100%) and the terminal state matches the learned successful endpoint; otherwise it fails. The system returns matched states, missing essential states, terminal-state agreement, and an explanation of why the trace passed or failed.

### 🧪 Experiments
The paper evaluates the approach in controlled experiments using a small number of passing traces as training data, typically emphasizing the low-data regime of three traces. The domains include UI testing, code generation/coding-agent tasks, and robotic or business-process-style sequential execution, with UI screenshots as the main concrete representation described in the text. Baselines are framed implicitly against brittle exact-sequence matching, assertion/manual specification approaches, visual-only comparison methods, and data-hungry ML oracles.

The main evaluation target is whether the learned validator can detect product bugs and false successes while tolerating acceptable non-deterministic variation. Validation metrics include pass/fail accuracy, coverage over essential states, terminal-state correctness, and explainability through matched/missing states. The abstract reports high accuracy in detecting product bugs and false successes using only three training traces. The text emphasizes that the system succeeds where exact trace matching would incorrectly fail valid runs and where overly permissive methods would miss skipped essential states. The extracted text available here does not provide a full numeric table of datasets, baselines, or detailed quantitative scores beyond the claim of high accuracy with three examples.

### 🌱 Insights & Implications
The key insight is that dominator analysis provides a principled way to separate essential states from optional ones in non-deterministic agent workflows, turning a small set of successful traces into an explainable execution oracle. This is especially valuable for autonomous systems whose behavior varies due to timing, rendering, or alternative but valid action sequences.

A second important implication is that visual similarity alone is insufficient for execution validation; semantic equivalence matters. The use of a multimodal LLM addresses cases where screenshots differ superficially but represent the same logical state, though it introduces dependency on model judgment, prompting quality, cost, and potentially nondeterministic equivalence decisions.

The work appears promising for low-shot test-oracle construction, but several open questions remain: robustness of equivalence thresholds across domains, sensitivity to imperfect or noisy passing traces, scalability to long or highly branching workflows, and how well the method handles partial success, loops, retries, or concurrent behavior. Another limitation is that the paper as provided emphasizes controlled experiments and high-level claims more than extensive quantitative benchmarking against strong baselines. For related research, this suggests a fruitful direction in combining classic program-analysis notions such as dominance with modern multimodal semantic reasoning for validation and oracle generation.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, security testing, and autonomous software agents because it tackles the core oracle problem: how to validate complex sequential executions without manually specifying every valid path. For researchers in automated program repair, vulnerability detection, and vulnerability repair, the method is especially useful as a low-data, explainable validator for agentic workflows such as code generation, patch application, exploit-reproduction steps, or secure remediation pipelines, where exact-sequence matching is too brittle but missing essential states is unacceptable.
