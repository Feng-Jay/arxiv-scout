## 📄 [QiMeng-PRepair: Precise Code Repair via Edit-Aware Reward Optimization](https://arxiv.org/abs/2604.05963v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Changxin Ke, Rui Zhang, Jiaming Guo, Yuanbo Wen, Li Ding, Shuo Wang, Xuyuan Zhu, Xiong Peng, Di Huang, Zidong Du, Xing Hu, Qi Guo, Yunji Chen
📅 **Published:** 2026-04-07  🏷️ **Categories:** cs.SE, cs.LG

### 💡 TL;DR
This paper targets over-editing in LLM-based program repair, where models fix bugs by unnecessarily rewriting large portions of code. It proposes PRepair, a two-stage framework that synthesizes diverse buggy programs and trains repair models with an edit-aware RL objective, improving both correctness and repair precision, with up to 31.4% gains on a new precision-aware metric.

### 🏆 Key Contributions
- Identifies and systematically quantifies over-editing as a major failure mode in LLM-based code repair trained with correctness-only rewards
- Introduces the precise repair task, emphasizing minimal bug fixing while preserving correct original code
- Proposes fix_p@k, a new evaluation metric that jointly measures repair correctness and edit cost relative to the minimum necessary edits
- Presents PRepair, a two-stage framework combining Self-Breaking data synthesis and Self-Repairing policy optimization for precise program repair
- Designs Self-Breaking to generate large-scale, diverse buggy programs from correct code via controlled bug injection and min-max diversity sampling
- Introduces Edit-Aware Group Relative Policy Optimization (EA-GRPO), which uses a dynamic edit-aware reward to encourage minimal yet correct repairs
- Shows that precise repair training not only improves repair quality but also benefits inference efficiency when paired with speculative editing

### ⚙️ Methodology
The paper formulates program repair as transforming a buggy line sequence X into a corrected sequence Y while minimizing unnecessary edits. It first defines Edit Cost as normalized line-level Levenshtein distance D_EC(X,Y)=D(X,Y)/|X|, where insertions, deletions, and substitutions are measured at the line level and normalized by buggy program length. The authors observe that standard GRPO training with correctness-only reward improves pass rate but steadily increases edit cost, causing over-editing.

To address this, they propose PRepair with two stages. In Self-Breaking, the model takes a problem description and a correct reference program and is prompted to inject bugs into the correct program, producing multiple buggy variants. To avoid redundant synthetic bugs and improve diversity, they select a subset of buggy programs using a min-max sampling strategy: among sampled candidates, they choose a subset that minimizes the maximum pairwise similarity, where similarity is defined from normalized edit distance. This yields synthetic training data with localized but diverse bug patterns and substantial retained correct logic.

In Self-Repairing, the model is trained to repair these synthetic buggy programs. The optimization uses Edit-Aware Group Relative Policy Optimization (EA-GRPO), an extension of GRPO for repair tasks. The key idea is that edit penalties should not always apply, because aggressive penalization may discourage repairs that actually need nontrivial changes. Instead, EA-GRPO uses a dynamic edit-aware reward that activates edit penalties only when the model reaches sufficient group-level repair correctness. This lets the model first learn to repair successfully and then, once accuracy is adequate, pushes it toward smaller and more localized edits. The reward therefore balances correctness and minimality, encouraging the model to preserve correct code and implicitly localize bugs rather than rewrite entire solutions.

The framework is designed for end-to-end repair rather than explicit multi-stage fault localization, but the authors argue that edit-aware optimization induces implicit localization ability by forcing the model to reuse the original buggy program structure. The method is language-agnostic and evaluated on both Python and Verilog tasks.

### 🧪 Experiments
The experiments cover two model sizes and two programming languages: Python and Verilog. Python repair data is collected from LeetCodeDataset, and Verilog repair data comes from QiMeng-CodeV-R1. The paper compares standard correctness-only GRPO training against the proposed PRepair/EA-GRPO setting, and evaluates both repair correctness and edit precision.

The main metrics are pass@k for correctness and the newly proposed fix_p@k for precise repair. fix_p@k counts a candidate as successful only if it both passes all tests and keeps edit cost within a tolerance p relative to the theoretical minimum edit cost between the buggy and gold-fixed program. The paper uses unbiased estimation over n sampled candidates. In addition, edit cost itself is tracked during training to directly quantify over-editing.

The key empirical observation is that under naive GRPO, pass rate improves during training but edit cost rises substantially, exceeding 0.6 in some settings, showing severe over-editing. PRepair reduces unnecessary modifications while improving correctness. The headline result is up to 31.4% improvement on fix_1@1, which is the strictest precise-repair setting because it requires correctness with edit cost close to the minimum necessary. The gains are reported across both Python and Verilog benchmarks and on models of different sizes, suggesting robustness across domains and scales. The paper also reports that when combined with speculative editing, EA-GRPO significantly increases decoding throughput, indicating practical latency benefits beyond pure repair quality.

### 🌱 Insights & Implications
A central insight is that correctness-only optimization is misaligned with practical program repair: a model can pass tests by rewriting large amounts of code, but this harms maintainability, obscures bug localization, and increases developer review effort. By introducing edit awareness into both evaluation and training, the paper reframes repair as a constrained transformation problem rather than unconstrained regeneration.

The proposed fix_p@k metric is particularly valuable for future repair research because it operationalizes the trade-off between correctness and minimal edits, which is often ignored in benchmark design. This is relevant for software maintenance and industrial patching scenarios where preserving surrounding logic matters. The finding that edit-aware training yields implicit bug localization is also important, since it suggests repair models can gain localization behavior without explicit localization labels.

For software engineering and security-oriented repair, the work has strong implications: minimal patches are usually preferred because they are easier to audit, less likely to introduce regressions, and more compatible with human workflows. This makes the method potentially useful not only for general bug fixing but also for vulnerability repair, where small, reviewable changes are often essential.

Open questions and limitations include the realism of self-generated bugs versus real-world defects, whether line-level edit distance is always the best proxy for semantic minimality, and how the dynamic reward behaves on complex multi-hunk or cross-file repairs. Another limitation is that correctness is still test-based, so repair quality depends on test adequacy. It would also be useful to know how well the method transfers to real vulnerability benchmarks, to APR datasets with human-written patches, and to settings requiring exact patch style compliance.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, automated program repair, and vulnerability repair because it directly addresses a core practical weakness of current LLM repair systems: they often over-edit and produce patches that are correct but hard to trust, review, or maintain. PRepair advances the field by introducing both a training objective and an evaluation metric aligned with real-world patch quality, emphasizing minimal, localized fixes rather than wholesale regeneration. For a researcher working on APR, vulnerability detection/repair, or maintainable automated patching, the paper offers a concrete RL-based method, a new benchmark perspective, and a compelling direction for building safer and more auditable repair models.

## 📄 [Broken by Default: A Formal Verification Study of Security Vulnerabilities in AI-Generated Code](https://arxiv.org/abs/2604.05292v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Dominik Blain, Maxime Noiseux
📅 **Published:** 2026-04-07  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
This paper evaluates whether frontier LLMs generate exploitable insecure code by default using a formal verification pipeline rather than heuristic scanning. Across 3,500 generated programs from seven models and 500 security-sensitive prompts, 55.8% were flagged as vulnerable and 1,055 vulnerabilities were formally proven exploitable with Z3 witnesses, with no model scoring better than a D.

### 🏆 Key Contributions
- Introduces a formal-verification-based evaluation of AI-generated code security using the COBALT pipeline and Z3 SMT solving to prove exploitability with concrete witnesses.
- Builds a 500-prompt benchmark spanning five security-relevant CWE families: memory safety, integer arithmetic, authentication, cryptography, and input handling.
- Provides a comparative security study of seven frontier LLMs under identical deterministic prompting conditions, showing consistently high vulnerability rates across all models.
- Validates selected formally proven findings with runtime proof-of-concept exploits and GCC AddressSanitizer, confirming 6 of 7 representative vulnerabilities.
- Shows that prompt-level security instructions only slightly reduce vulnerability rates, suggesting default generation behavior is structurally insecure.
- Demonstrates a large gap between formal analysis and mainstream static tooling: six industry tools combined miss 97.8% of Z3-proven findings.
- Reveals a generation-review asymmetry in which models can often detect vulnerabilities in review mode yet still generate insecure code by default.

### ⚙️ Methodology
The study constructs a benchmark of 500 natural-language prompts, with 100 prompts each across five CWE categories: MEM (buffer allocation, array indexing, dynamic memory in C/C++), INT (integer arithmetic and signed/unsigned conversion), AUTH (password hashing and token/session logic), CRYPTO (key generation, randomness, cipher selection), and INP (SQL, path, and shell input composition). Seven publicly accessible frontier models were queried through production APIs at temperature 0, using a system prompt requesting complete runnable code. This yielded 3,500 code artifacts. Each artifact was then analyzed by the COBALT pipeline. COBALT first performs candidate vulnerability extraction using regex and AST-level structural patterns, such as detecting malloc expressions lacking overflow guards. It then encodes the vulnerability condition into SMT constraints for Z3. For example, integer-overflow-related allocation bugs are represented with bit-vector semantics and checked for satisfiability under machine arithmetic; if Z3 returns SAT, the system extracts a concrete witness input that triggers the vulnerability. Findings are labeled as Z3 SAT when exploitability is formally proven, PATTERN MATCH when a structural vulnerability is identified but not yet SMT-encoded, and CLEAN otherwise. Severity is mapped using CVSS v3 base-score categories. To validate external exploitability, the authors manually built proof-of-concept harnesses for seven representative cases and executed them using GCC with AddressSanitizer and UndefinedBehaviorSanitizer, feeding the Z3 witness values as runtime inputs.

### 🧪 Experiments
The main experiment evaluates 3,500 generated artifacts from seven models over the 500-prompt benchmark. The models are GPT-4o, GPT-4.1, Claude Haiku 4.5, Gemini 2.5 Flash, Mistral Large, Llama 3.3 70B, and Llama 4 Scout. The primary metrics are vulnerability rate, counts of CRITICAL and HIGH findings, and number of Z3 SAT proofs. Overall, 55.8% of artifacts contained at least one COBALT-identified vulnerability, and 1,055 findings were formally proven exploitable via Z3. Per-model vulnerability rates were: GPT-4o 62.4% (F), Llama 4 Scout 60.6% (F), Llama 3.3 70B 58.4% (D), Mistral Large 57.8% (D), GPT-4.1 54.0% (D), Claude Haiku 4.5 49.2% (D), and Gemini 2.5 Flash 48.4% (D). Category-level means showed INT as the worst at 87%, followed by MEM at 67%, INP at 56%, AUTH at 44%, and CRYPTO at 25%. For runtime validation, 7 selected vulnerabilities were turned into PoC exploits; 6 of 7 caused confirmed runtime faults or successful attacks, including heap buffer overflows, oversized allocations, out-of-bounds reads, SQL injection, and weak password hashing exploitation. Three auxiliary experiments were also run on a smaller 50-prompt sub-corpus. First, a secure-prompt ablation added explicit instructions to follow security best practices; this reduced the mean vulnerability rate by only about 4 percentage points to 60.8%, leaving four of five tested models still at grade F. Second, a tool comparison against six industry analyzers found they collectively flagged only 7.6% of artifacts and missed 97.8% of Z3-proven vulnerabilities. Third, a review-mode experiment tested whether models could identify vulnerabilities in their own outputs; they recognized their own Z3-proven bugs 78.7% of the time (70/89), despite generating vulnerable code at high rates in the first place.

### 🌱 Insights & Implications
The central insight is that AI-generated code is often insecure by default even for non-adversarial, routine developer prompts, and that the most prevalent failures are not stylistic issues but mathematically provable exploitability conditions, especially around integer overflow and memory allocation. The paper strongly argues that pattern-based evaluation understates or mischaracterizes the problem, while formal methods can provide ground-truth exploitability with witness inputs. For software security research, the result that mainstream static tools miss nearly all Z3-proven findings suggests a major blind spot in current analysis pipelines when applied to LLM-generated code. The generation-review asymmetry is also important: models appear to possess partial security knowledge but fail to apply it consistently during synthesis, raising questions about decoding objectives, instruction following, and whether post-generation review or self-repair loops could be more effective than one-shot prompting. Limitations include the bounded prompt corpus, deterministic temperature-0 setup, focus on selected CWE families, partial SMT encoding coverage indicated by the PATTERN MATCH class, and the fact that not all findings were dynamically validated. The secure-prompt ablation was conducted only on a smaller 50-prompt corpus, so broader claims about prompt hardening remain provisional. Open questions include whether iterative repair frameworks can reduce the formally proven vulnerability rate, how these findings transfer to real repository-level tasks, whether type-aware or verification-aware code generation can help, and how formal witness generation could be integrated into automated vulnerability detection or repair loops.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it offers a rigorous benchmark and formal-analysis methodology for evaluating the security quality of AI-generated code. Its use of SMT-based exploitability proofs directly complements research on vulnerability detection and repair by providing precise, machine-checkable bug witnesses that could seed repair generation, benchmark APR systems, or serve as oracles in secure code synthesis pipelines. The evidence that models can review vulnerabilities better than they generate them is especially actionable for designing iterative generation-review-repair workflows, while the large gap between COBALT and existing static tools highlights opportunities for new verification-guided maintenance and security tooling.

## 📄 [On the Role of Fault Localization Context for LLM-Based Program Repair](https://arxiv.org/abs/2604.05481v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Melika Sepidband, Hung Viet Pham, Hadi Hemmati
📅 **Published:** 2026-04-07  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper systematically studies how fault-localization context affects LLM-based automated program repair on SWE-bench Verified. It finds that file-level context is the main driver of repair success, semantic LLM-based retrieval is usually better and cheaper than structural expansion, and adding more line-level context often hurts by introducing noise.

### 🏆 Key Contributions
- A large-scale factorial empirical study of fault-localization context for LLM-based APR across file-, element-, and line-level granularities.
- Evaluation of 61 feasible localization-context configurations on 500 SWE-bench Verified instances using GPT-5-mini under a ground-truth localization setting.
- Evidence that file-level localization is the dominant factor for repair, improving performance by 15-17x over a no-file baseline.
- Demonstration that expanding file context is often useful, with successful repairs most commonly occurring when about 6-10 relevant files are provided.
- Finding that element-level context yields conditional benefits, while expansion beyond buggy elements offers mixed returns and depends on file-context quality.
- Finding that line-level expansion usually degrades repair quality, with context windows and static slicing often amplifying noise rather than helping.
- Comparison showing LLM-based retrieval outperforms rule-based or structural heuristics for files, elements, and lines while using fewer files and tokens.
- Practical guidance that the strongest strategy combines broad semantic context at higher abstraction levels with precise buggy-line localization, and that adaptive or ensemble context strategies may further improve APR.

### ⚙️ Methodology
The study isolates the effect of fault-localization context on repair by assuming perfect localization from ground-truth patches in SWE-bench Verified. For each bug instance, the authors extract ground-truth buggy files, buggy program elements (such as functions, classes, or global variables), and buggy lines from the reference patch, then systematically vary how much additional context is supplied around those locations.

The experiment is designed as a factorial analysis over three orthogonal context dimensions:
1) File-level context: no files, only buggy files, rule-based relevant files, or LLM-retrieved relevant files.
2) Element-level context: no elements, buggy elements, call-graph-relevant elements, or LLM-retrieved relevant elements.
3) Line-level context: no lines, buggy lines, a +/-10-line context window, static code slicing, or LLM-retrieved lines.

Not all combinations are feasible, resulting in 61 evaluated configurations. The key methodological choice is to separate localization precision from retrieval expansion: because buggy locations are derived from ground truth, the study can directly test whether adding broader context beyond the true buggy locations helps, saturates, or harms repair. This avoids conflating poor retrieval with poor repair ability.

Repair is performed with GPT-5-mini. For each configuration and each benchmark instance, the model receives the issue/task context plus the selected repository context corresponding to that configuration, and attempts to generate a patch. The study then records which instances are successfully resolved under each setting. Beyond aggregate outcomes, the authors compare retrieval strategies by both effectiveness and cost, analyzing the number of files included and token usage. They also use paired statistical testing, specifically the Wilcoxon signed-rank test, to assess whether differences between expansion strategies are significant. Finally, they perform qualitative analysis on cases where extra context helps or hurts, to identify complementary behaviors across configurations and motivate adaptive context selection.

### 🧪 Experiments
The evaluation uses 500 instances from SWE-bench Verified, a realistic repository-level benchmark for software issue resolution and program repair. The repair model is GPT-5-mini. The independent variables are the 61 feasible combinations of file-, element-, and line-level localization context.

Datasets and context construction:
- Benchmark: 500 SWE-bench Verified instances.
- Ground-truth buggy files, elements, and lines are extracted from the reference patches.
- Expansion strategies include rule-based structural heuristics and LLM-based semantic retrieval.

Baselines and comparisons:
- No-file, buggy-file, rule-based file expansion, and LLM-based file retrieval.
- No-element, buggy-element, call-graph element expansion, and LLM-based element retrieval.
- No-line, buggy-line, context-window, static-slice, and LLM-based line retrieval.
- Comparisons focus both on repair success and on context cost, such as file counts and tokens.

Metrics and analyses:
- Main outcome: number of resolved instances / repair success across configurations.
- Statistical significance: Wilcoxon signed-rank tests for paired configuration comparisons.
- Efficiency/cost: distributions of retrieved file counts and token budgets, including histogram and box-plot analyses.

Key quantitative results reported in the paper include:
- Providing file-level context yields the biggest gain, with a 15-17x improvement over a no-file baseline.
- Expanding beyond only buggy files often further improves repair, and successful repairs are most commonly associated with configurations containing roughly 6-10 relevant files.
- LLM-based file retrieval significantly outperforms rule-based file expansion in most settings, especially when line localization is precise, while also using fewer files and fewer tokens.
- Element-level context generally helps compared with no-element context, but expansion beyond buggy elements gives mixed gains; LLM-retrieved elements outperform call-graph expansion, and this advantage is statistically significant in many settings.
- Line-level expansion rarely helps and often hurts. Context windows and static slicing frequently degrade performance relative to precise buggy lines or even no line information, indicating noise amplification.
- Among line-expansion methods, LLM-retrieved lines are more reliable than context-window or slicing-based expansion, though they still do not consistently beat precise buggy-line localization.
- The strongest-performing family of configurations combines LLM-based file retrieval, LLM-based or strong element context, and exact buggy-line localization.

The experiments also show that different configurations solve different subsets of bugs, implying nontrivial complementarity rather than a single universally best localization strategy.

### 🌱 Insights & Implications
The central insight is that in LLM-based APR, more localization context is not monotonically better. Broad context is useful at higher abstraction levels, especially files, because it gives the model enough semantic scope to understand cross-file behavior, dependencies, and design intent. But at fine granularity, especially lines, extra context often acts as distractor noise that weakens repair accuracy.

A second important implication is that semantic retrieval via an LLM is usually more effective than structural heuristics such as imported/importing files, call-graph neighbors, or static slicing. This suggests that relevance for repair is not purely syntactic or topological; semantically related code that helps explain intent may matter more than structurally adjacent code. For software engineering research, this supports building retrieval modules that optimize for repair utility rather than traditional localization suspiciousness alone.

The results also challenge a common APR assumption: improving localization precision uniformly improves repair. Even with perfect buggy locations derived from ground truth, many instances remain unsolved, indicating that repair difficulty often lies beyond localization alone. This points to other bottlenecks such as reasoning over issue intent, generating semantically valid multi-location edits, repository-wide constraints, or validation feedback loops.

For automated repair system design, the paper implies a layered strategy: use broad semantic retrieval at file and possibly element levels, but keep line-level guidance precise and minimal. This aligns with agentic repair pipelines where retrieval and editing are separated: retrieval should help the model understand the problem space, while editing pointers should remain narrowly targeted.

Limitations and open questions include:
- The study uses a single repair model, GPT-5-mini, so generalization to other LLMs or agent frameworks remains to be tested.
- Ground-truth localization removes real-world localization errors; practical APR systems must still retrieve buggy regions without access to the patch.
- The benchmark is SWE-bench Verified only; effects may differ on other datasets, languages, or defect types.
- The paper indicates complementary solved sets across configurations, but does not yet present a learned adaptive policy for choosing the right context per bug.
- It remains open how these findings translate to multi-step agentic systems with iterative search, tool calls, execution feedback, and dynamic context refinement.

Overall, the paper suggests a promising research direction: adaptive or ensemble fault-localization context selection, where context breadth is tuned to bug characteristics, repository structure, or uncertainty estimates rather than fixed globally.

### 🎯 Why This Paper
This paper is highly relevant to research in automated program repair, software maintenance, and repository-scale software engineering because it directly studies one of the most important but under-analyzed components of LLM-based APR: what code context to retrieve and present to the model. For work on APR and vulnerability repair, the findings are immediately actionable: prioritize high-quality file-level retrieval, prefer semantic LLM-based retrieval over rigid structural heuristics, and avoid over-expanding line-level context that can inject noise.

It also matters for vulnerability detection and repair pipelines because security fixes often require cross-file reasoning while still needing precise edit locations. The paper's conclusion that broad semantic context plus precise line targeting is most effective maps well to security patch generation settings. More broadly, it advances the researcher's work by providing an empirical framework for studying context granularity, a retrieval-design blueprint for repair agents, and evidence motivating adaptive retrieval policies rather than one-size-fits-all localization.

## 📄 [CODESTRUCT: Code Agents over Structured Action Spaces](https://arxiv.org/abs/2604.05407v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Myeongsoo Kim, Joe Hsu, Dingmin Wang, Shweta Garg, Varun Kumar, Murali Krishna Ramanathan
📅 **Published:** 2026-04-07  🏷️ **Categories:** cs.AI, cs.SE

### 💡 TL;DR
CodeStruct replaces brittle text-span file editing in LLM code agents with AST-based operations over named program entities, letting agents read and modify functions/classes via structured selectors. Across SWE-Bench Verified and CodeAssistBench, this structure-aware interface improves task success while usually reducing token usage and cost, with especially large gains for weaker models that often fail to generate valid text patches.

### 🏆 Key Contributions
- Introduces a structured action-space abstraction for code agents where repositories are exposed as named AST entities rather than flat text.
- Defines two LLM-oriented primitives, readCode and editCode, for retrieving complete syntactic units and applying syntax-validated edits to semantic program elements.
- Implements human-readable selectors with scope resolution so agents can address files, classes, functions, and methods unambiguously without line numbers or string matching.
- Provides empirical evidence on SWE-Bench Verified and CodeAssistBench across six LLMs showing improved Pass@1/accuracy and lower token consumption for most models.
- Analyzes when structured interfaces help most, showing the biggest gains arise when failures are caused by text-edit brittleness rather than by core reasoning limitations.

### ⚙️ Methodology
CodeStruct models a repository as an executable structured environment derived from the abstract syntax tree. Instead of reading arbitrary text spans or patching code through line numbers and string replacement, the agent interacts through two operations over AST-addressable entities. readCode takes a file path and optionally a selector identifying a semantic unit such as a class, function, or method, and returns either the complete syntactic unit or a compact structural summary, avoiding truncated contexts and unnecessary surrounding code. editCode takes a selector plus an edit operation and applies AST-level transformations such as insert, replace, delete, or duplicate directly on the targeted node. Node references use human-readable symbol paths like file.py::ClassName::method, with automatic scope resolution to disambiguate entities. For replacement edits, the model need only specify the signature and new body for the target unit instead of regenerating unchanged neighboring text. The framework reparses and validates edits against syntax, so each action maps one syntactically valid AST state to another, enabling multi-step repository editing as a sequence of structured transformations. This design explicitly separates semantic intent from textual realization, removes dependence on exact formatting or repeated substrings, and yields analyzable action traces for agent behavior. Relative to standard text tools, CodeStruct changes both retrieval and modification: retrieval is scoped to complete program units, and modification is performed on semantic nodes rather than raw spans, which reduces context length, prevents malformed partial edits, and mitigates common failures such as no-occurrence or multiple-occurrence replacement errors.

### 🧪 Experiments
The paper evaluates CodeStruct on two repository-level software engineering benchmarks: SWE-Bench Verified and CodeAssistBench. On SWE-Bench Verified, the authors test six LLMs and compare CodeStruct against conventional text-based agent interfaces using metrics centered on Pass@1 accuracy, token consumption, and inference cost. Stage B trains the Fixer with reinforcement learning. ### 💡 TL;DR
This paper reframes iterative LLM-based test generation as a planning and exploration problem rather than a greedy coverage maximization task. - Formalizes iterative LLM test generation as Bayesian exploration, with the program's branch structure treated as an unknown environment and the accumulated coverage map used as a proxy posterior over explored behavior.

### ⚙️ Methodology
The paper models test generation as sequential exploration of an unknown program. The hidden variable is the program's branch reachability structure, the interaction history consists of previously executed tests and observed branch hits, and the exploration objective is to maximize cumulative branch discovery. Because maintaining an explicit Bayesian posterior over branch reachability is intractable, the method uses the evolving coverage map as a deterministic proxy for the agent's posterior knowledge state. CovQValue operates iteratively: at each round, the current coverage map is fed back into the LLM; the LLM proposes multiple diverse candidate test plans rather than isolated tests; each plan is evaluated by the LLM with a curiosity-inspired Q-value that estimates both immediate expected branch gain and the potential to unlock future branches behind setup-heavy corridors; the highest-valued plan is executed; the observed coverage is merged into the coverage map; and the loop repeats. The design is motivated by Bayesian exploration theory, especially the idea that greedy information gain is suboptimal in environments with corridor structure, where intermediate actions may yield zero immediate reward but are necessary to reach informative regions. In the test generation setting, these corridors correspond to setup sequences such as imports, object initialization, state preparation, or validation gates. The method therefore explicitly prefers plans that may not maximize short-term branch gain but increase the likelihood of deeper future exploration.

The evaluation uses two benchmarks. First, TestGenEval Lite, an existing real-world benchmark derived from SWE-Bench for file-level test generation. Second, RepoExploreBench, introduced in this paper, containing 93 modules from 9 popular Python packages and designed for iterative exploration scenarios with corridor-like structure. The authors evaluate across three LLMs: Gemini 3 Flash, GPT-5.4-mini, and Mistral Large 3. The primary metric is branch coverage, and they also report win rates over targets relative to greedy selection baselines. Compared with greedy selection, CovQValue achieves 51-77% higher branch coverage on TestGenEval Lite and wins on 77-84% of targets. On RepoExploreBench, it improves branch coverage by 40-74% over greedy baselines. The largest gains occur on repositories where reaching deep logic requires multi-step setup sequences with little or no immediate coverage reward, validating the paper's planning-based exploration hypothesis.

### 🌱 Insights & Implications
The main insight is that many test generation failures are not due to poor local test synthesis but due to myopic decision-making: greedy selection gets trapped when useful tests require preparatory steps that produce no immediate coverage increase. Treating code testing as sequential exploration reveals a strong connection to curiosity and information-gain planning from reinforcement learning. The paper also suggests that coverage feedback should be treated not just as an evaluation signal but as state for decision-making. A limitation is that the coverage map is only a proxy for a true Bayesian posterior and cannot represent richer uncertainty about unobserved program behavior. Another open question is how robust the LLM-estimated Q-values are across domains, languages, and larger repositories, and whether stronger execution-guided or symbolic signals could improve plan valuation. The benchmark contribution is important because standard one-shot benchmarks may systematically underestimate the value of iterative planning methods. The broader implication is that similar planning-aware exploration could benefit adjacent software engineering tasks such as API usage discovery, debugging, vulnerability triage, and autonomous repository understanding.

### 💡 TL;DR
EffiPair is a test-time framework for making LLM-generated code faster and more memory-efficient by comparing pairs of similar correct programs and feeding the model a compact contrastive explanation of why one is more efficient. Across code-efficiency benchmarks, it improves efficiency while preserving correctness, achieving up to 1.5x speedup over unguided generation and using over 90% fewer tokens than prior feedback-heavy methods.

### 🏆 Key Contributions
- Introduces Relative Contrastive Feedback (RCF), a pairwise execution-based feedback signal that compares similar programs with different efficiency profiles instead of relying on absolute scalar profiling alone.
- Presents EffiPair, an inference-time iterative refinement framework that generates multiple candidates, selects informative efficient/inefficient pairs, summarizes their performance differences, and uses that summary to guide code optimization.
- Shows that pairwise relational feedback can provide stronger optimization direction than isolated runtime or memory numbers, improving efficiency without model fine-tuning or parameter updates.
- Demonstrates that lightweight summarized execution statistics are sufficient for effective refinement, substantially reducing profiling and token overhead compared with prior approaches such as EffiLearner.

### ⚙️ Methodology
EffiPair operates entirely at inference time in a propose-evaluate-refine loop. For each programming task with a natural-language description and correctness test suite, the system first samples N candidate programs from the base LLM. Each candidate is executed against tests to assess correctness and profiled to estimate efficiency cost e(p), such as runtime or memory usage. The goal is to find a program that passes all tests while minimizing efficiency cost.

The key idea is to replace isolated absolute feedback with Relative Contrastive Feedback. From the candidate pool, EffiPair selects a positive reference program p+ as the most efficient correct candidate currently available. It then searches for a negative counterpart p- among other correct candidates that are structurally similar to p+ but substantially less efficient. Similarity is computed as a weighted combination of cosine similarity over program embeddings and cosine similarity over AST representations, with a threshold tau used to restrict pairing to sufficiently similar programs. Among these similar programs, the least efficient one is selected as p-.

EffiPair then constructs a compact contrastive feedback summary from the pair’s execution behavior. Rather than giving the model raw, verbose profiling traces or only scalar runtimes, the framework distills the main relative differences: which operations or design choices dominate execution cost in the slower program and which corresponding choices in the faster program avoid those costs. The model is prompted with both candidate programs and the RCF summary, and asked to revise the less efficient solution to preserve correctness while adopting the more efficient structural choices. The newly generated refinement is added back into the candidate pool for subsequent rounds.

If no correct candidates exist initially, the framework continues exploration using available candidates until correctness is achieved. The overall process is iterative and cooperative: multiple sampled trajectories contribute to a shared candidate pool, and optimization is driven by relationships between candidates rather than by independently interpreting each profile. The framework requires no fine-tuning, no parameter updates, and is designed to reduce both execution overhead and prompt length by using summarized pairwise comparisons instead of repeated absolute-profile prompting.

### 🧪 Experiments
The paper evaluates EffiPair on code-efficiency benchmarks using strong code-capable LLMs, including DeepSeek-Chat V3.2. The setup compares vanilla generation without performance feedback against iterative efficiency refinement methods, particularly prior absolute-feedback approaches such as EffiLearner. Candidates are generated per task, filtered by correctness tests, profiled for efficiency, and refined over multiple rounds.

The evaluation focuses on correctness preservation, runtime and memory efficiency, and inference cost in terms of prompt/completion token usage. The main reported result is that EffiPair consistently improves efficiency while preserving correctness. With DeepSeek-Chat V3.2, it achieves up to 1.5x speedup over generation without performance feedback. Compared with EffiLearner, EffiPair attains comparable or better efficiency gains while also achieving higher correctness and drastically lower token overhead. The paper reports a reduction from about 23.6M to 200k prompt tokens and from about 350k to 77k completion tokens relative to EffiLearner, amounting to more than 90% token savings. The authors also report that summarized execution statistics are sufficient for effective optimization, indicating that full profiling traces are unnecessary for strong refinement performance.

### 🌱 Insights & Implications
The paper’s main insight is that efficiency optimization for generated code benefits more from relational information between solutions than from standalone scalar performance numbers. When two correct programs are structurally similar, their differences are highly diagnostic: the slower one reveals inefficient design choices and the faster one provides a concrete nearby target for revision. This makes the optimization signal more directional and information-dense than absolute runtime or memory feedback.

For software engineering and maintenance, this suggests a practical pathway for post-generation performance repair that does not require retraining the model. The framework is especially relevant for workflows where correctness is already acceptable but generated code remains too slow or memory-heavy for production use. The similarity-based pairing strategy also highlights an emerging pattern for LLM program repair: use a pool of near-miss or near-neighbor candidates as mutual supervision.

Potential limitations and open questions remain. The method depends on obtaining multiple diverse candidates and on having at least some correct candidates to serve as anchors. Its effectiveness may vary with the quality of similarity metrics, the threshold for pairing, and the fidelity of profiling summaries. The current formulation appears centered mainly on runtime and memory optimization rather than broader non-functional properties such as energy, security, or maintainability. Another open question is whether pairwise contrastive execution feedback could be generalized to other program-improvement tasks, such as automatic bug fixing, vulnerability repair, or exploit mitigation, where comparing a safe and unsafe version may similarly yield stronger guidance than isolated labels or scores.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, and automated program improvement because it frames code efficiency enhancement as an inference-time refinement problem over candidate programs, closely related to program repair workflows. Its pairing-and-contrast mechanism could inspire analogous methods for vulnerability detection and repair, where relative comparisons between secure/insecure or patched/unpatched code may provide stronger guidance than absolute signals. The method defines negative tests, called spectests, as concrete input-output pairs that cannot be produced by the target program but may still satisfy a weak candidate specification. The setting explicitly allows training on datasets whose human-written specifications are not complete. Because rewards come from execution, verification, and spectest rejection, the model can learn to produce stronger specifications than those seen in supervised data. The paper's main insight is that many enterprise and healthcare workflows require intelligence at design time but not at execution time. For these cases, compiling workflow logic into static code can eliminate key weaknesses of agentic systems: stochastic behavior, prompt-injection exposure on every request, high recurring inference cost, and limited auditability. This makes the approach especially compelling for regulated environments where every decision may need to be traced to code and validated artifacts.

For software engineering research, the work is notable because it reframes LLM-based automation as a code generation and repair pipeline rather than an online reasoning problem. The regenerate-on-failure loop resembles automated program repair: candidate code is synthesized, analyzed, executed, and refined until it satisfies a validation suite. The template-based design also aligns with secure-by-construction and compliance-by-construction ideas.

A key limitation is scope: compiled AI is most suitable for well-specified, repetitive workflows with stable business logic. It trades away runtime flexibility, so it may be less effective for open-ended tasks, highly dynamic environments, or situations requiring continual semantic adaptation. Another limitation is that benchmark evidence is still relatively narrow: BFCL and DocILE demonstrate feasibility, but broader studies on long-term maintainability, evolution of compiled artifacts, regression risk under changing requirements, and comparison with stronger code-generation baselines would strengthen the claims. Open questions include how to maintain compiled artifacts as regulations or APIs change, how to version and verify generated code over time, and how to integrate formal methods or stronger static analysis for high-assurance deployments.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, security, automated program repair, and vulnerability-oriented research because it operationalizes LLM code generation as a controlled software artifact synthesis pipeline. It directly intersects with program repair and secure code generation through its constrained templates, validation stages, regeneration loop, and static safety checks. For researchers interested in vulnerability detection and repair, the paper provides a concrete architecture for reducing attack surface, screening generated code before deployment, and treating LLM outputs as auditable code artifacts rather than opaque runtime behavior. It advances the field by showing how production workflow automation can adopt classic SE principles—static analysis, testing, determinism, and modular templates—while still benefiting from LLM synthesis.

## 📄 [Edit, But Verify: An Empirical Audit of Instructed Code-Editing Benchmarks](https://arxiv.org/abs/2604.05100v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Amir M. Ebrahimi, Gopi Krishnan Rajbahadur
📅 **Published:** 2026-04-06  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper audits the two main instructed code-editing benchmarks, CanItEdit and EDIT-Bench, and finds that both are poor proxies for real-world LLM code editing because they are unrepresentative of actual developer tasks and often rely on weak or flawed test oracles. The authors show that benchmark scores can therefore mislead deployment decisions, and they propose six evidence-based desiderata for building better code-editing benchmarks.

### 🏆 Key Contributions
- Systematic selection and audit of the only two benchmarks meeting strict criteria for instructed code editing with human-written instructions and test-based evaluation: CanItEdit and EDIT-Bench.
- Empirical representativeness analysis showing both benchmarks are heavily skewed toward Python and overconcentrated on feature/fix edits while underrepresenting common real-world languages, domains, and edit intents such as TypeScript, frontend/backend work, documentation, testing, CI/build, and maintenance.
- Test-adequacy audit across all 213 benchmark problems using instrumented execution and coverage analysis, revealing that CanItEdit has stronger validation while EDIT-Bench often has sparse, low-coverage tests that may fail to detect unintended edits outside the requested region.
- Artifact analysis of universally unsolved EDIT-Bench tasks, showing that most such failures are attributable to benchmark problems rather than model incapability.
- Duplication analysis showing that multiple problems share codebases within each benchmark, reducing true evaluation diversity.
- Proposal of six empirically grounded desiderata for future instructed code-editing benchmarks so that benchmark scores better reflect real-world editing capability.

### ⚙️ Methodology
The study is an empirical benchmark audit organized around two research questions: representativeness of benchmark tasks and adequacy of benchmark test oracles. The authors first surveyed 180 benchmarks from a recent benchmark survey and added EDIT-Bench through targeted search. They applied progressive inclusion criteria: code-related benchmark, non-agentic and non-multi-turn, focused on code editing rather than generation, evaluating holistic editing rather than just bug fixing, and using human-authored rather than LLM-generated editing instructions. This process left only CanItEdit and EDIT-Bench.

For RQ1, the authors compared the distributions of benchmark tasks against real-world reference distributions drawn from Copilot Arena, the AIDev dataset, and GitHub Octoverse. They audited three dimensions: programming language, edit intent, and application domain. Benchmark tasks were categorized and contrasted with deployment-relevant distributions reported in those external datasets. The analysis focused on whether the benchmarks reflect the diversity of languages developers use, the kinds of edits they request, and the software domains in which those edits occur.

For RQ2, the authors evaluated test-oracle strength by executing every benchmark test suite in instrumented Docker environments and measuring statement coverage over both the full file and the edited diff region. They also examined the number of tests per task, whether tests validate fail-before/pass-after behavior, and whether low-coverage suites would detect unintended modifications outside the requested edit region. Beyond coverage, they manually investigated data completeness and benchmark quality issues, especially universally unsolved tasks in EDIT-Bench, to distinguish true model difficulty from flawed benchmark artifacts. They also analyzed code duplication by checking whether multiple benchmark tasks share the same underlying codebase, thereby reducing the effective number of distinct evaluation contexts.

### 🧪 Experiments
The empirical audit covers all 213 core benchmark problems: 105 from CanItEdit and 108 from EDIT-Bench. The representativeness analysis compares benchmark composition with real-world usage data from Copilot Arena, AIDev, and GitHub Octoverse. On language distribution, both benchmarks devote more than 90% of tasks to Python, despite real-world usage being spread across many languages and no single language exceeding roughly 30%; TypeScript, described as GitHub’s most-used language, is entirely absent. On edit intent, both benchmarks focus 79–86% of tasks on feature and bug-fix style edits, while documentation, testing, build/CI, and maintenance tasks—together 31.4% of human PR activity—have zero representation. On domain distribution, CanItEdit allocates 68.6% of problems to algorithm design versus an 18% real-world share, while EDIT-Bench allocates 36.1% to AI/ML versus a 7% real-world share; backend and frontend editing, which together account for 46% of real-world activity, are absent in CanItEdit and only 24.1% combined in EDIT-Bench.

For test adequacy, the authors instrumented benchmark execution and measured test counts and coverage. CanItEdit has a median of 13 tests per problem, near-complete whole-file statement coverage with a median of 100%, and explicit fail-before/pass-after validation. EDIT-Bench has a median of 4 tests per problem, median whole-file coverage of 40%, and 14 tasks that rely on a single test as the full correctness signal. The audit finds that 59% of EDIT-Bench’s low-coverage suites would not detect modifications outside the specified edit region. The paper also reviews 15 EDIT-Bench tasks unsolved by any of 40 LLMs and concludes that 11 of them are unsolved primarily due to flawed benchmark artifacts such as infrastructure bugs, impossible assertions, or hidden environmental assumptions. Duplication analysis shows that 29% of EDIT-Bench problems and 6% of CanItEdit problems share a codebase with at least one other task, leaving 86 distinct code contexts out of 108 in EDIT-Bench and 103 out of 105 in CanItEdit.

### 🌱 Insights & Implications
The main insight is that current instructed code-editing benchmarks under-measure the construct they are widely used to represent. They are skewed toward narrow language and domain slices, especially Python and algorithmic tasks, while real developer editing is multilingual, application-heavy, and includes substantial documentation, testing, maintenance, and infrastructure work. As a result, high benchmark scores do not necessarily imply readiness for practical deployment in coding assistants.

A second key insight is that test-based evaluation is only as reliable as the test oracles. CanItEdit appears stronger because its tests are coverage-driven and validated with fail-before/pass-after checks, but even it remains narrow in task realism. EDIT-Bench is more realistic in task sourcing because it comes from real Copilot Arena interactions, yet its test harnesses are often too weak to reliably certify correctness, especially with respect to unintended edits outside the requested scope. This exposes a tradeoff between task realism and oracle quality that future benchmark design must address jointly.

The artifact analysis is especially important for evaluation research: universally unsolved tasks may reflect benchmark defects rather than model limitations, meaning leaderboard ceilings and conclusions about model weakness can be misleading. Likewise, code duplication reduces the effective diversity of evaluation examples and may inflate confidence in generalization. For related research in automated program repair and vulnerability repair, the paper implies that benchmark validity should include not only realism of edit instructions but also independence of samples, broad task coverage, and strong regression-sensitive tests. An open limitation is that the study focuses only on two benchmarks because almost no others meet the inclusion criteria; this highlights a field-wide gap rather than a weakness of the audit. Another limitation is dependence on external datasets for defining real-world distributions, though these sources are among the strongest currently available.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, vulnerability detection, and vulnerability repair because it scrutinizes whether current LLM code-editing benchmarks actually measure realistic edit performance under trustworthy correctness oracles. For researchers building or evaluating repair and security-focused editing systems, the findings directly warn that benchmark success can be inflated by unrepresentative tasks, weak tests, flawed artifacts, and duplicated contexts. It advances this line of work by providing a rigorous audit methodology, concrete evidence about benchmark failure modes, and practical desiderata for constructing future benchmarks that better reflect real maintenance and repair scenarios, including regression sensitivity and realistic edit distributions.
