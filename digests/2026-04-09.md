## 📄 [Argus: Reorchestrating Static Analysis via a Multi-Agent Ensemble for Full-Chain Security Vulnerability Detection](https://arxiv.org/abs/2604.06633v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Zi Liang, Qipeng Xie, Jun He, Bohuan Xue, Weizheng Wang, Yuandao Cai, Fei Luo, Boxian Zhang, Haibo Hu, Kaishun Wu
📅 **Published:** 2026-04-08  🏷️ **Categories:** cs.CR, cs.CL, cs.SE

### 💡 TL;DR
Argus is an LLM-centered static security analysis framework that replaces the usual 'SAST tool first, LLM assistant second' pipeline with a multi-agent ensemble that reasons over code, dependencies, external vulnerability knowledge, and symbolic analysis outputs together. It reports higher true-vulnerability discovery, fewer false positives, lower token cost, and real zero-day findings with assigned CVEs on large open-source projects.

### 🏆 Key Contributions
- Introduces Argus, presented as the first LLM-centered multi-agent framework for static vulnerability detection rather than a conventional SAST-centered pipeline with LLM add-ons.
- Extends vulnerability analysis beyond repository-local code to full supply-chain analysis by incorporating third-party dependencies and community vulnerability information.
- Designs a collaborative agent workflow covering dependency scanning, issue retrieval, sink identification, PoC generation, repair suggestion, data-flow scanning, and data-flow review.
- Proposes Re3 (Retrieval, Recursion, and Review), combining retrieval augmentation, iterative reasoning, and forward/backward flow analysis to improve flow reconstruction and reduce missed reachability.
- Integrates RAG and ReAct into SAST to reduce hallucinations, support long-horizon reasoning, and ground findings in external evidence.
- Demonstrates practical impact through large-scale experiments on industrial open-source repositories and reports several zero-day vulnerabilities that received CVE assignments.

### ⚙️ Methodology
Argus reframes static application security testing as an LLM-orchestrated workflow in which traditional analyzers, dependency metadata, and external security knowledge are tools and evidence sources for a coordinated agent system. The framework begins from a target repository and its third-party dependencies, treating the project as part of a software supply chain rather than an isolated codebase. It retrieves potentially relevant prior issues, vulnerability reports, and dependency-related security information using retrieval-augmented generation so the system can ground its reasoning in known community knowledge and avoid unsupported LLM guesses.

The workflow is decomposed into specialized agents. A dependency-scanning/information-collection stage analyzes imported libraries and associated vulnerability context. A sink-identification stage integrates with CodeQL to find candidate dangerous operations while allowing LLM reasoning to expand beyond handcrafted rules. For suspected cases, Argus generates proof-of-concept exploits and corresponding repair suggestions, using ReAct-style interleaving of reasoning and tool use to support multi-step security analysis. The core flow-analysis component is Re3, standing for Retrieval, Recursion, and Review. Re3 combines: (1) retrieval of supporting context from repository and dependency knowledge; (2) recursive exploration of forward and backward propagation paths to reconstruct data flows even when direct symbolic reachability is incomplete; and (3) review by another agent to validate or reject inferred flows and suppress hallucinated chains.

Conceptually, the system models vulnerabilities through taint-style source-to-sink flows over content nodes and access paths. Rather than relying solely on one-pass symbolic taint analysis, Argus uses multi-agent collaboration to identify sources, infer sinks, recover disrupted data flows, inspect semantic context, and verify exploitability. The final output is a human-reviewable report containing the suspected vulnerability, data-flow explanation, external supporting evidence, PoC, and repair guidance.

### 🧪 Experiments
The paper states that Argus is evaluated extensively on popular industrial-scale open-source repositories containing tens of millions of lines of code and tens of thousands of GitHub stars. The evaluation compares Argus against existing SAST and LLM-assisted vulnerability-detection approaches, with emphasis on practical detection quality and deployment cost. The reported criteria include number of true vulnerabilities discovered, false positives, and token consumption/operational cost. Qualitatively, the authors claim that Argus detects more new vulnerabilities than prior methods while also lowering false positives and using fewer tokens than existing LLM-based baselines. The experiments further highlight Argus's ability to detect vulnerabilities missed by classical rule-based taint analysis because of incomplete source/sink coverage or broken reachability. Beyond benchmark-style results, the paper reports real-world validation: Argus found previously unknown zero-day vulnerabilities in heavily used open-source codebases, and several of these findings were assigned CVE numbers. The provided text does not expose the full numerical tables, exact dataset names, baseline list, or metric values, but the claimed outcomes are improved true-positive discovery, reduced false alarms, and lower inference cost.

### 🌱 Insights & Implications
A central insight is that LLMs appear more effective in static security analysis when used as orchestrators over heterogeneous evidence sources rather than as standalone vulnerability judges or simple post-processors for SAST alerts. The paper also emphasizes that many practical misses in static analysis stem not only from poor sink classification but from incomplete supply-chain context and broken or partial flow reachability; Argus addresses both by combining dependency-aware retrieval with recursive multi-agent flow reconstruction. Another important implication is that agent specialization and cross-review may be a more promising design pattern for secure code analysis than monolithic prompting, especially for reducing hallucinations and enforcing evidence-backed claims.

For researchers in vulnerability detection and repair, the PoC-plus-repair loop is notable because it links detection to exploit validation and remediation, potentially helping prioritize actionable findings. For software engineering and maintenance, the supply-chain perspective aligns well with modern repository ecosystems where vulnerable behavior often emerges from interactions among application code, framework code, and third-party packages.

Likely limitations and open questions include dependence on retrieval quality, reliance on external tools such as CodeQL, possible variance across programming languages and ecosystems, reproducibility of zero-day mining results, and the cost/latency tradeoff of multi-agent orchestration in CI settings. Another open question is how to formally calibrate confidence, benchmark hallucination suppression, and measure exploitability soundness when LLM-generated PoCs or flow explanations are involved.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it sits directly at the intersection of software security, static analysis, vulnerability detection, and vulnerability repair. It advances the state of practice by proposing an industrially oriented, multi-agent LLM framework that improves over both traditional SAST and earlier LLM-assisted pipelines, while also incorporating repair suggestions and exploit validation. For work on automated program repair, vulnerability triage, or secure software maintenance, Argus offers a concrete architecture for combining symbolic analysis, retrieval, agentic reasoning, and actionable remediation in large real-world codebases.

## 📄 [Guiding Symbolic Execution with Static Analysis and LLMs for Vulnerability Discovery](https://arxiv.org/abs/2604.06506v1)  `100% relevant`

👤 **Authors:** Md Shafiuzzaman, Achintya Desai, Wenbo Guo, Tevfik Bultan
📅 **Published:** 2026-04-07  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper presents SAILOR, a pipeline that automates symbolic-execution harness generation by combining static analysis, LLM-guided iterative code synthesis, symbolic execution, and concrete replay. On 10 large open-source C/C++ projects, it reportedly finds 379 previously unknown memory-safety vulnerabilities, dramatically outperforming strong LLM-only and ablated baselines.

### 🏆 Key Contributions
- Introduces SAILOR, an automated framework for constructing symbolic execution harnesses for large C/C++ codebases.
- Combines static analysis to identify candidate vulnerable locations and derive vulnerability specifications with LLM-driven synthesis of drivers, stubs, and assertions.
- Uses iterative refinement guided by compiler and symbolic execution feedback, reducing the need for expert-written harnesses.
- Adds concrete replay on the unmodified source to validate symbolic-execution findings and improve confidence in discovered vulnerabilities.
- Demonstrates substantial empirical gains over five baselines, including a strong agentic LLM vulnerability-discovery baseline with full codebase access.
- Provides ablation evidence showing that static analysis targeting, iterative LLM synthesis, and symbolic execution are all essential to the final performance.

### ⚙️ Methodology
SAILOR is a three-phase vulnerability discovery pipeline aimed at automating the most labor-intensive part of symbolic execution: building usable harnesses for real software. In Phase 1, static analysis scans the target codebase to identify candidate vulnerability sites and generate vulnerability specifications. From the abstract, these specifications likely encode the target program locations, relevant APIs/functions, memory-safety conditions of interest, dependency context, and the expected assertions or bug oracles needed for symbolic execution. In Phase 2, an LLM consumes the vulnerability specifications and orchestrates harness synthesis. Rather than producing a single test harness in one shot, it iteratively refines multiple harness components—drivers to invoke target code, stubs to model external dependencies or hard-to-execute components, and assertions to express vulnerability conditions—using feedback from compilation and symbolic execution. This suggests a repair-style synthesis loop where failed compilations, missing models, infeasible paths, or weak assertions are fed back to the LLM to improve the harness until it becomes executable and useful for path exploration. Symbolic execution is then run with the generated harness to explore paths precisely and trigger candidate memory-safety violations. In Phase 3, SAILOR performs concrete replay against the unmodified project source, meaning the discovered symbolic inputs or traces are re-executed concretely without the simplified harness assumptions to confirm that the reported issues correspond to real crashes or vulnerabilities. The overall design is explicitly hybrid: static analysis narrows the search space for scalability, the LLM handles code-level synthesis and adaptation, symbolic execution provides path-sensitive precision, and concrete replay supplies a ground-truth validation step.

### 🧪 Experiments
The evaluation covers 10 open-source C/C++ projects totaling 6.8 million lines of code, indicating a large-scale assessment on realistic systems software or infrastructure codebases. The target vulnerability class is memory-safety issues. SAILOR discovers 379 distinct previously unknown vulnerabilities, corresponding to 421 confirmed crashes after validation, implying that some vulnerabilities may manifest through multiple crashing inputs or variants. The paper compares against five baselines; the strongest baseline is an agentic vulnerability-detection approach using Claude Code with full codebase access and unlimited interaction, yet it finds only 12 vulnerabilities. This implies an approximately 31.6x improvement in distinct vulnerabilities over the strongest baseline. The abstract also reports ablation-style results quantifying the contribution of each stage. Removing static analysis targeting causes confirmed vulnerabilities to drop by 12.2x, showing that search-space reduction and vulnerability specification generation are central to scalability. Removing iterative LLM synthesis yields zero confirmed vulnerabilities, suggesting one-shot or non-feedback-based harness generation is ineffective in this setting. Removing symbolic execution reduces performance to the point that no approach detects more than 12 vulnerabilities, underscoring that LLMs and static analysis alone are insufficient for deep path-sensitive bug discovery. Although the abstract does not list the exact metrics, the reported outcomes indicate vulnerability count and confirmed crash count as primary effectiveness measures, with baseline comparison and component ablation as the main evaluation dimensions.

### 🌱 Insights & Implications
The paper strongly suggests that automated vulnerability discovery in large codebases benefits from decomposition into complementary reasoning modes rather than relying on a single technique. Static analysis appears to provide crucial localization and specification signals that make the LLM's synthesis problem tractable. The LLM's role is not replacing symbolic execution, but enabling it by generating the infrastructure—drivers, models, assertions—that historically required human experts. Symbolic execution remains the key engine for finding deep, path-sensitive memory-safety bugs, while concrete replay addresses a common concern in hybrid analysis pipelines: whether findings survive outside modeled or synthesized environments. For software security research, this is important because it shows a practical route to scaling symbolic execution without sacrificing validation rigor. Likely limitations include dependence on the quality and coverage of the initial static analysis, sensitivity to LLM synthesis failures or prompt design, and possible confinement to memory-safety vulnerabilities in C/C++ rather than broader semantic bug classes. Another open question is how robust the approach is across different symbolic executors, LLMs, compiler toolchains, and dependency-heavy projects. It is also unclear from the abstract how much manual tuning was required, how expensive the iterative synthesis/execution loop is, and whether the generated harnesses generalize across vulnerability classes. For related research, the paper implies promising directions in LLM-assisted test harness generation, automated environment modeling, APR-style iterative code refinement using analysis feedback, and hybrid workflows that use concrete replay to reduce false positives in static/dynamic vulnerability detection.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, software maintenance, and automated program repair because it addresses a central scalability bottleneck in symbolic execution: manual harness construction. Its iterative LLM-plus-feedback synthesis loop is closely aligned with repair-style generation, while its focus on discovering and validating real memory-safety vulnerabilities in large C/C++ systems directly advances work on vulnerability detection and automated vulnerability discovery pipelines. For a researcher interested in APR and vulnerability repair, SAILOR also provides a compelling template for combining static analysis, generative models, and execution-based feedback in a closed loop that could be adapted from vulnerability finding to exploit generation, patch synthesis, regression test generation, or repair validation.

📅 **Published:** 2026-04-08  🏷️ **Categories:** cs.CR, cs.DB

### 💡 TL;DR
The paper presents VulGD, a publicly accessible Neo4j-based vulnerability graph database that continuously ingests and integrates data from multiple authoritative sources and augments vulnerability records with LLM-generated text embeddings. Its main result is a practical end-to-end platform for real-time graph exploration, API-based retrieval, and semantic vulnerability analysis intended to improve risk assessment and prioritization over static relational repositories.

### 🏆 Key Contributions
- A dynamic multi-source ingestion pipeline that continuously aggregates vulnerability intelligence from sources such as NVD, CVE, CWE, ExploitDB, and CVE Details into a unified property graph.
- An open-access web system and public API that support interactive graph visualization, direct querying, automated retrieval, and export for both technical and non-technical users.
- LLM-based semantic augmentation of vulnerability nodes using embeddings from vulnerability descriptions, enabling similarity search, clustering, and downstream ML/DL workflows without introducing hallucinated factual edges.
- A practical, extensible cybersecurity research infrastructure that bridges the gap between static vulnerability repositories and prior research prototypes that are difficult to deploy or keep current.

### ⚙️ Methodology
VulGD is designed as a dynamic cybersecurity knowledge graph platform built on a Neo4j property graph database. The system continuously collects data from multiple public vulnerability repositories, normalizes heterogeneous schemas, resolves inconsistencies across sources, and maps the resulting information into graph entities and relationships such as vulnerabilities, weaknesses, products, exploits, and associated metadata. The architecture emphasizes automated and ongoing updates rather than periodic static snapshots, enabling real-time querying and historical analysis as new disclosures appear.

A central methodological choice is to represent integrated vulnerability intelligence as a graph rather than a relational schema so that complex interconnections—shared CWEs, linked exploits, affected software groupings, and cross-source references—can be traversed natively. VulGD also exposes this graph through two access layers: a browser-based interactive interface for graph exploration and a public API for programmatic consumption and downstream automation.

For semantic enrichment, VulGD computes vector embeddings from vulnerability descriptions using large language models. These embeddings are attached to vulnerability nodes as auxiliary representations to support semantic similarity, clustering, and threat grouping. Importantly, the system constrains the role of LLMs to embedding generation instead of fact extraction for graph edges, explicitly avoiding direct reliance on potentially hallucinated LLM-generated relations. The resulting platform combines structured graph knowledge with unstructured text semantics, making it suitable for vulnerability triage and research workflows.

### 🧪 Experiments
The paper is primarily a systems/platform paper rather than a benchmark-heavy ML paper. The experimental demonstration centers on the live deployment and practical capabilities of VulGD: continuous ingestion from authoritative sources including NVD, CVE, CWE, ExploitDB, and CVE Details; storage and querying in Neo4j; interactive visualization through a web interface; and automated access through a public API. The paper discusses a practical use case showing how integrated graph relations and LLM-derived embeddings can be used for vulnerability exploration, similarity-based analysis, and prioritization.

No extensive comparative benchmark table, standard dataset split, or broad quantitative evaluation against vulnerability detection/repair baselines is presented in the provided text. The focus is instead on system functionality, dynamic updating, accessibility, and the added semantic utility of embeddings. Key claimed outcomes are improved support for real-time querying, graph traversal, historical analysis, and semantically informed risk assessment compared with static relational repositories and prior graph systems that are harder to deploy or update.

### 🌱 Insights & Implications
A key insight is that vulnerability management suffers not only from missing data but from fragmented representation across repositories; graph modeling is especially valuable because it preserves relationships that are central to prioritization, such as exploit availability, shared weaknesses, and product dependencies. Another important insight is the restrained use of LLMs: the authors avoid using LLMs to create factual knowledge graph edges and instead use embeddings as soft semantic signals, which is a practical design for reducing hallucination risk in security-critical settings.

The work suggests that openly accessible, dynamically updated graph infrastructure can serve as a common benchmarking and integration layer for future research in vulnerability assessment, threat intelligence, and ML-based security analytics. Likely limitations include the lack of strong quantitative evaluation of embedding quality or analyst benefit, unclear details about entity resolution accuracy and update latency, and limited evidence about scalability under very large continuous feeds. Open questions include how to evaluate semantic enrichment rigorously, how to incorporate software dependency and patch information more deeply, how to support provenance/confidence scoring across sources, and how such a graph can be coupled with downstream predictive models for exploitability, remediation recommendation, or automated vulnerability repair.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and vulnerability repair research because it provides an up-to-date, structured, multi-source vulnerability knowledge base that can be directly used for analysis, benchmarking, retrieval, and model augmentation. For a researcher working on automated program repair, maintenance, and vulnerability analytics, VulGD can serve as infrastructure for linking CVEs to CWEs, products, exploits, and semantic descriptions, enabling better dataset construction, threat prioritization, root-cause analysis, and integration of graph/context features into security-focused learning systems.

## 📄 [REAgent: Requirement-Driven LLM Agents for Software Issue Resolution](https://arxiv.org/abs/2604.06861v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Shiqi Kuang, Zhao Tian, Kaiwei Lin, Chaofan Tao, Shaowei Wang, Haoli Bai, Lifeng Shang, Junjie Chen
📅 **Published:** 2026-04-08  🏷️ **Categories:** cs.SE

### 💡 TL;DR
The paper argues that repository-level issue resolution fails not just because of weak reasoning or tooling, but because raw issue descriptions are often incomplete and ambiguous. It proposes REAgent, a requirement-driven LLM agent framework that converts issues and repository context into structured issue-oriented requirements, assesses their quality through patch outcomes, and iteratively refines them, yielding consistent gains over strong issue-resolution baselines on SWE-bench benchmarks.

### 🏆 Key Contributions
- Introduces the perspective that low-quality issue descriptions are a core bottleneck for repository-level software issue resolution, and proposes issue-oriented requirements as structured task specifications for patch generation.
- Presents REAgent, a three-stage requirement-driven LLM agent framework consisting of requirement generation, requirement assessment, and requirement refinement.
- Designs a requirement generation agent that explores repositories, gathers issue-specific context, and organizes it into structured, information-rich requirements using predefined requirement attributes.
- Proposes Requirement Assessment Score (RAS), which evaluates requirement quality indirectly via traceability to generated patches and executable validation results.
- Develops a requirement refinement mechanism that classifies low-quality requirements into three deficiency categories and applies category-specific strategies with actionable feedback for iterative improvement.
- Provides extensive evaluation on SWE-bench Lite, SWE-bench Verified, and SWE-bench Pro using DeepSeek-V3.2 and Qwen-Plus, showing consistent improvements over five representative/state-of-the-art baselines.

### ⚙️ Methodology
REAgent treats issue resolution as a requirement engineering problem before patch synthesis. Instead of feeding the raw issue text directly to an LLM agent, the framework first constructs issue-oriented requirements: a structured representation of the task that includes contextual background, intended functionality, constraints, affected scope, and success criteria inferred from both the issue description and the repository. To build this artifact, a requirement generation agent autonomously explores the codebase, retrieves issue-relevant files/modules, collects scattered contextual evidence, and fills predefined requirement attributes to produce a structured requirement document.

Because requirement generation can itself be noisy or hallucinated, REAgent adds a requirement assessment stage. The key idea is to exploit traceability between a requirement and the patch generated from it: if a requirement is high quality, it should guide the downstream patch generation process toward a correct implementation. The framework therefore generates patches from the current requirement, runs available validation/execution procedures, and converts these patch-level outcomes into an indirect quality signal for the requirement. This signal is formalized as the Requirement Assessment Score (RAS), which measures how effectively the requirement supports correct patch synthesis.

If the requirement is judged low quality, REAgent triggers requirement refinement. A requirement analysis agent diagnoses the root cause of the deficiency by mapping it into one of three high-level categories of requirement deficiencies. For each category, the framework uses a tailored refinement strategy to narrow the semantic search space and produce actionable feedback, rather than performing generic rewriting. The requirement is then revised and used for another round of patch generation and validation. This process is iterative, allowing the system to progressively improve requirement quality and thereby patch correctness.

Overall, REAgent is a closed-loop agent framework: generate structured requirements from issue+codebase context, assess requirement usefulness through patch execution outcomes, analyze deficiency causes, refine the requirement with category-specific guidance, and repeat until the iteration budget is exhausted or a satisfactory patch is obtained.

### 🧪 Experiments
The evaluation uses three standard repository-level issue-resolution benchmarks: SWE-bench Lite, SWE-bench Verified, and SWE-bench Pro. Experiments are conducted with two advanced LLM backbones, DeepSeek-V3.2 and Qwen-Plus, creating six benchmark/model settings in total. REAgent is compared against five representative or state-of-the-art baselines, including agent- and workflow-based issue resolution systems discussed in the paper such as SWE-agent, Agentless, and Trae-agent.

The primary metrics are % Resolved (number/percentage of issues successfully resolved) and % Applied (number/percentage of generated patches that can be successfully applied). Across all six settings, REAgent consistently outperforms every baseline. Relative to baselines, REAgent improves % Resolved by 9.17% to 24.83%, with an average gain of 17.40%, and improves % Applied by 22.17% to 49.50%. The authors also study the effect of the iteration count N, a key hyperparameter for iterative issue-resolution systems, and report that REAgent continues to outperform all iterative baselines as N increases. In addition, they construct four ablation variants of REAgent to isolate the effects of its main components, and the ablation results confirm that requirement generation, assessment, and refinement each contribute to the final performance improvements.

### 🌱 Insights & Implications
The paper’s main insight is that repository-level repair quality is limited not only by model reasoning and tool access but also by the specification quality of the input task. This reframes issue resolution as partly a requirements engineering problem: better-structured, more complete, less ambiguous specifications lead to better patches. The use of execution-guided signals to assess requirement quality is particularly notable because it sidesteps the difficulty of directly judging natural-language requirements with static rules.

For software engineering research, this suggests a promising direction: integrating requirements reconstruction, traceability, and patch validation into APR and issue-resolution pipelines. The framework may also generalize beyond bug fixing to feature implementation and other repository-level maintenance tasks where user reports are underspecified. For software security and vulnerability repair, the idea of transforming noisy vulnerability reports into structured security requirements could be highly relevant.

Potential limitations and open questions remain. Requirement quality is assessed indirectly via patch outcomes, so errors in patch generation may confound requirement evaluation. The paper also relies on iterative LLM-agent workflows, which can increase compute cost and latency. The robustness of deficiency categorization, the exact coverage of requirement attributes across diverse issue types, and the transferability to security-specific or highly under-tested repositories remain open problems. Another important future direction is combining this requirement-driven paradigm with stronger retrieval, static analysis, dynamic analysis, or test generation to further improve correctness and validation coverage.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, maintenance, automated program repair, and vulnerability detection/repair because it tackles repository-level issue resolution with a principled structured-specification approach rather than only better prompting or tools. Its requirement-driven loop—constructing structured requirements, validating them through executable outcomes, and iteratively refining them—offers ideas directly applicable to APR systems, bug/vulnerability report understanding, repair guidance, and maintenance automation in large codebases.

## 📄 [PoC-Adapt: Semantic-Aware Automated Vulnerability Reproduction with LLM Multi-Agents and Reinforcement Learning-Driven Adaptive Policy](https://arxiv.org/abs/2604.06618v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Phan The Duy, Nguyen Viet Duy, Khoa Ngo-Khanh, Nguyen Huu Quyen, Van-Hau Pham
📅 **Published:** 2026-04-08  🏷️ **Categories:** cs.CR

### 💡 TL;DR
PoC-Adapt is an LLM-based multi-agent framework for automatically reproducing vulnerabilities that improves exploit validation using semantic state-difference checking and reduces trial-and-error via offline reinforcement learning. Across Java vulnerability benchmarks, it reports more reliable verification and lower exploit-generation cost than prior LLM pipelines, and it verified 12 PoCs out of 80 recent CVE reproduction attempts at about $0.42 each.

### 🏆 Key Contributions
- Introduces a Semantic Oracle that validates generated PoCs by comparing structured pre- and post-execution system states, rather than relying on crashes, flags, or superficial execution signals.
- Formulates exploit generation as a Markov Decision Process and trains an offline Double Deep Q-Network (DDQN) adaptive policy over exploitation logs to guide action selection and reduce failed attempts and cost.
- Designs an end-to-end multi-agent orchestration pipeline with specialized agents for context retrieval, root-cause analysis, environment setup, exploit generation, and semantic validation, connected through structured feedback loops.
- Demonstrates the framework on CWE-Bench-Java, PrimeVul, and a recent CVE corpus, showing improved validation reliability and practical low-cost PoC reproduction.

### ⚙️ Methodology
PoC-Adapt is built as an end-to-end automated vulnerability reproduction framework that combines LLM multi-agent orchestration with semantic validation and offline reinforcement learning. The workflow decomposes PoC generation into coordinated stages. First, the system retrieves and filters vulnerability context from reports and related artifacts to reduce noise before passing it to analysis agents. A root-cause analysis agent reasons about the vulnerable code path, likely trigger conditions, affected components, and exploit constraints. An environment-building agent reconstructs a runnable target setup, including dependencies and execution configuration, so that candidate exploits can be tested in a realistic environment. An exploit-generation agent then proposes PoC candidates and iteratively refines them based on feedback. A semantic-validation agent executes the candidate and invokes the Semantic Oracle.

The Semantic Oracle is the core verification component. Instead of checking only whether execution crashes or whether some surface-level signal appears, it captures structured system states before and after exploit execution and compares them to detect semantically meaningful effects attributable to vulnerability exploitation. This state-differential validation is intended to distinguish true exploit success from incidental side effects, benign behavior changes, or hallucinated success conditions. The paper positions this as more robust than crash-based, shell-based, or CTF-style flag-check oracles.

To improve efficiency, PoC-Adapt adds Adaptive Policy Learning. The exploit process is modeled as a Markov Decision Process with semantic states, discrete actions, transitions, and rewards tied to successful progress toward exploitation. The policy is learned offline from exploitation logs, avoiding expensive online interaction during training. The authors use a model-free, off-policy Double Deep Q-Network (DDQN) to estimate action values and mitigate overestimation bias. At inference time, the learned policy helps the exploit agent choose more promising next actions or refinement steps, reducing heuristic search and unnecessary retries. The architecture also incorporates inter-agent and developer-critic style feedback loops so failures during setup, generation, or validation can be fed back into earlier stages for targeted correction rather than full restart. Overall, the method combines modular agent specialization, semantic runtime monitoring, and RL-guided search to improve both reliability and cost-efficiency of automated PoC synthesis.

### 🧪 Experiments
The paper evaluates PoC-Adapt on CWE-Bench-Java and PrimeVul, and also applies it to a corpus of recent CVEs. Baselines are prior automated PoC or vulnerability reproduction approaches discussed in the paper, especially recent LLM-based and multi-agent systems such as FaultLine and CVE-Genie, alongside heuristic exploit-generation workflows. The evaluation focuses on verification reliability, exploit-generation efficiency/cost, and practical reproduction success. The central claim is that semantic validation improves exploit verification reliability by 25% over prior validation approaches that depend on surface-level runtime signals. The adaptive policy mechanism reduces exploit-generation cost by cutting failed exploration attempts compared to heuristic-only multi-agent systems. In the real-world CVE study, PoC-Adapt achieved 12 verified PoCs out of 80 reproduction attempts, with an average cost of $0.42 per generated exploit. The experimental framing emphasizes both benchmark effectiveness and practical deployability, although the provided text excerpt does not include full tabulated metrics, exact baseline-by-baseline breakdowns, or ablation numbers for each component.

### 🌱 Insights & Implications
The paper’s main insight is that exploit-generation quality is bottlenecked as much by verification as by synthesis: weak oracles can make a system appear successful when it is only producing plausible but semantically incorrect behavior. Using pre/post structured state comparison is a meaningful step toward trustworthy automated vulnerability reproduction, especially for subtle logic flaws where crashes are irrelevant. A second key insight is that LLM-based exploit search benefits from learning from prior attempts rather than repeatedly using expensive heuristic trial-and-error; offline RL is a practical way to capture this historical experience without unsafe or costly online exploration.

For software engineering and security research, the work suggests a promising architecture for turning vulnerability reports into validated regression artifacts: root-cause reasoning, environment reconstruction, exploit generation, and semantic validation map naturally onto maintenance and patch-validation workflows. It also has implications for automated program repair and vulnerability repair, since semantically verified PoCs could become stronger tests for confirming exploitability before patching and non-regression after patching.

Open questions and limitations include how well the Semantic Oracle generalizes across vulnerability classes, languages, and complex distributed systems; how semantic states are specified or instrumented without excessive manual effort; whether offline RL policies trained on one benchmark transfer to unseen vulnerability families; and whether the 12/80 real-world success rate is sufficient for high-stakes operational usage. Additional missing details from the excerpt include exact reward design, state/action encoding, ablation of semantic oracle versus RL, and failure mode analysis. The approach also inherits typical LLM-agent concerns: environment brittleness, context-window limitations, and potential error propagation despite modularization.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection/repair, and automated program repair because it tackles the practical problem of turning vulnerability reports into validated exploit artifacts in an automated and scalable way. Its semantic oracle can inform stronger validation and regression-test generation for vulnerability repair, while its RL-guided agent orchestration offers a concrete strategy for reducing cost and improving robustness in LLM-driven security automation pipelines.

## 📄 [PSR2: A Phase-based Semantic Reasoning Framework for Atomicity Violation Detection via Contract Refinement](https://arxiv.org/abs/2604.06975v1)  `80% relevant`  `🌐 Full Text`

📅 **Published:** 2026-04-08  🏷️ **Categories:** cs.CR

### 💡 TL;DR
PSR2 is a static-analysis framework for smart-contract atomicity violations that combines control-flow path mining with AST-based semantic dependency reasoning, then confirms issues through a fusion decision model. On 1,600 contracts, it substantially outperforms Slither and Semgrep, especially for complex ERC-721 callback scenarios, reaching 94.69% F1 versus 51.86% for the strongest baseline.

### 🏆 Key Contributions

- Defines a unified atomicity inconsistency model that abstracts multiple vulnerability manifestations into a common read-call-write style reasoning pattern.
- Designs GSAM to identify reachable hazardous execution paths in simplified control-flow graphs using operation labels such as SLOAD, CALL, and SSTORE.
- Designs SCAM to extract deterministic semantic facts from ASTs, including function roles, state-variable access modes, external call targets, and data dependencies.
- Proposes a Fusion Decision Module that cross-validates structural reachability with semantic dependency and call-target information to reduce false positives while preserving recall.
- Demonstrates strong empirical gains over Slither and Semgrep across reentrancy-like and unchecked external call benchmarks, plus ablation evidence that fusion materially improves precision and F1.

### ⚙️ Methodology
PSR2 is organized into three phases. First, the Semantic Context Analysis Module (SCAM) parses a contract into an abstract semantic repository F=(F,V,E). It profiles functions and state variables, assigning function roles via keyword/signature matching and variable access modes per function through AST traversal. For each external call site, SCAM constructs a tuple Te=(loc, type, tgt, D), where tgt characterizes the call target source (for example, fixed address versus user-controlled) and D is a data-dependency set identifying which state variables influence call parameters or call conditions. This produces deterministic semantic facts rather than heuristic pattern hits.

Second, the Graph Structure Analysis Module (GSAM) builds a simplified control-flow graph G=(N',E') by mapping original CFG nodes to atomicity-relevant labels through an abstraction function alpha. The retained node types are SLOAD(s), SSTORE(s), CALL(t), CHECK, and OTHER. GSAM searches candidate paths for dependent-state sequences where the same state variable s is read and later written, with an external call occurring in between. It formalizes this as an atomicity predicate phi_atomic(P,s) that holds when there exist path positions i<m<j such that alpha(ni)=SLOAD(s), alpha(nm)=CALL(t), and alpha(nj)=SSTORE(s). Paths satisfying this predicate are emitted as structurally suspicious reports with evidence sequences.

Third, the Fusion Decision Module (FDM) combines GSAM path reports with SCAM semantic facts. For each suspicious path and state variable, it derives a semantic context annotation Cj,s=(tgt, dep, interm), where dep indicates whether the variable participates in the call-site dependency set, interm indicates whether the call truly lies between the read and write of that variable, and tgt records whether the target is fixed or potentially attacker-controlled. The final deterministic decision function assigns High risk when dep and interm hold and the call target is not fixed, Medium when dep and interm hold but the target is fixed, and Low otherwise. Only high/medium cases are reported as validated atomicity violations, along with an explanation chain linking the path, state variable, call facts, and semantic context.

### 🧪 Experiments
The evaluation uses 1,600 contract samples spanning three benchmarks: SmartBugs-Reentrancy, ERC-721 Reentrancy with implicit callbacks, and SmartBugs-Unchecked External Call. Baselines are Slither and Semgrep. Metrics reported are precision, recall, and F1-score; the study is organized around three research questions covering comparative performance, module contribution, and generalizability.

Main results: on the Reentrancy dataset, PSR2 achieves 78.13% precision, 98.04% recall, and 87.72% F1, outperforming Slither (4.76% F1) and Semgrep (81.51% F1). On ERC-721 Reentrancy, PSR2 reaches 94.94% precision, 89.91% recall, and 94.69% F1, far above Semgrep’s 51.86% and Slither’s 23.68%. On Unchecked External Call, PSR2 obtains 84.35% precision, 98.04% recall, and 91.95% F1, compared with Slither’s 73.54% and Semgrep’s 16.42%.

Ablation uses a balanced dataset of 719 vulnerable and 719 safe contracts and compares the full system against graph-only and semantic-only variants. PSR2_GSAM detects 639/719 vulnerable contracts (88.87% recall) with 81 false positives on safe contracts (11.27% FPR), yielding 88.81% overall F1. PSR2_SCAM detects 602/719 vulnerable contracts (83.77% recall) but has 639 false positives (88.81% FPR), giving 64.90% F1. The full PSR2 detects 664/719 vulnerable contracts (92.35% recall) with 46 false positives (6.40% FPR), achieving 92.93% F1. This shows fusion reduces false positives substantially relative to single-module analysis while also improving recall.

### 🌱 Insights & Implications
The paper’s key insight is that atomicity-violation detection benefits from explicitly separating structural feasibility from semantic validity, then requiring both before raising an alarm. This is particularly useful for smart contracts with implicit callbacks and non-local control flow, where syntax-only or rule-only approaches either miss reachable attacks or over-report infeasible ones.

For software security and vulnerability detection more broadly, PSR2 suggests a promising design pattern: combine path-sensitive program analysis with semantically grounded dependency facts, then fuse them through an interpretable decision layer. The explicit evidence chain also improves explainability, which could help downstream tasks such as vulnerability triage and automated repair.

Limitations and open questions: the reported baselines are primarily pattern/rule-oriented tools, so the comparison does not establish superiority over more advanced symbolic, abstract-interpretation, or learning-based analyzers. The semantic role assignment appears partly keyword/signature driven, which may affect robustness on unconventional naming styles. The method is tailored to atomicity inconsistencies framed as read-call-write patterns, so it is unclear how well it extends to other bug classes without redesign. The paper also focuses on detection rather than remediation, leaving open how the extracted evidence could drive automated contract repair or patch generation.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and automated reasoning for repair because it presents a precise, interpretable static-analysis framework for finding nontrivial state-consistency bugs. ### 💡 TL;DR
This paper studies how to produce trustworthy confidence scores for LLM-generated code edits in automated code revision tasks such as program repair, vulnerability repair, and code refinement. ### 🏆 Key Contributions
- Proposes three fine-grained confidence scores for automated code revision: minimum token probability, lowest-K token probability, and attention-weighted uncertainty, contrasting them with conventional sequence-level scores.

### ⚙️ Methodology
The paper targets confidence calibration for automated code revision (ACR), where an LLM generates an edited version of existing code and the system must estimate the probability that the edit is correct. The authors argue that standard calibration for code generation is too coarse for ACR because correctness often depends on a few local edits rather than the whole generated sequence.

They compare five confidence scores. Two are conventional sequence-level scores: normalized sequence likelihood and average token probability. Three are fine-grained scores designed to surface uncertainty concentrated in local edits: (1) minimum token probability, which uses the least-confident token in the generated revision; (2) lowest-K token probability, which aggregates the K lowest token probabilities to reduce sensitivity to single-token noise; and (3) attention-weighted uncertainty, which weights token-level uncertainty according to attention patterns so that uncertainty in more salient positions contributes more strongly.

Global Platt-scaling is the conventional post-hoc approach fitting one logistic regression to map an uncalibrated confidence score into an estimated correctness probability for all samples in a task. Their proposed local Platt-scaling instead assumes calibration may differ across subsets of samples. It first obtains sample embeddings, partitions examples into clusters, and fits a separate Platt calibrator per cluster. At inference time, a new sample is assigned to a cluster based on embedding similarity and the corresponding local calibrator is applied to its confidence score. This yields a sample-aware calibration pipeline.

The overall workflow is: generate a code revision with an LLM, compute one of the candidate confidence scores from token-level generation information, optionally apply either a global or local Platt-scaling calibrator learned on held-out data, and output a calibrated probability that the revision is correct. The authors analyze calibration behavior across probability intervals using reliability-style evaluation rather than only aggregate error, emphasizing whether scores remain trustworthy over a broad confidence range.

### 🧪 Experiments
The evaluation spans three automated code revision tasks: automated program repair, vulnerability repair, and automated code refinement. The study covers 14 different LLMs of different sizes, and evaluates calibration under task-specific correctness metrics appropriate to each revision task. The experimental comparison includes five confidence scores—two sequence-level baselines and three proposed fine-grained variants—and two post-hoc calibration methods: conventional global Platt-scaling and the proposed local Platt-scaling.

The main evaluation criterion is calibration error, examined both overall and across probability intervals via reliability-style analyses. The key findings are qualitative but consistent across the benchmark suite: fine-grained confidence scores achieve lower calibration error than sequence-level confidence scores across a wider range of confidence intervals; among all scores, minimum token probability is the strongest overall performer across tasks, correctness metrics, and models; and local Platt-scaling further reduces calibration error relative to global Platt-scaling. The improvement from local calibration is especially pronounced for automated code refinement, while program repair and vulnerability repair also benefit, though in those tasks global Platt-scaling with fine-grained scores can already produce adequately calibrated estimates when low latency is preferred. The paper also reports that combining fine-grained scores with local Platt-scaling yields the best calibration behavior overall.

### 🌱 Insights & Implications
A central insight is that calibration for code revision differs from calibration for generic code generation: correctness is often governed by a few risky local decisions, so whole-sequence probabilities can wash out the uncertainty that matters most. This explains why prior sequence-level approaches worked in some software engineering generation tasks but were unreliable or underexplored for repair-oriented settings.

Another important takeaway is that miscalibration is sample-dependent. A single global sigmoid is often too restrictive, especially for heterogeneous tasks like code refinement where edits vary widely in type and difficulty. Local Platt-scaling addresses this by learning different mappings for different regions of the input space.

The work suggests a practical tradeoff between calibration quality and latency. Fine-grained scores paired with global calibration are attractive when deployment cost matters, while local calibration offers better reliability at added computational complexity. This is highly relevant for interactive developer tools where response time affects adoption.

Potential limitations include dependence on clustering quality for local calibration, possible sensitivity to benchmark size and task distribution, and the restriction to single-pass post-hoc methods rather than richer uncertainty estimation techniques such as repeated sampling or verifier-style assessment. Open questions include how these methods interact with newer reasoning/code agents, whether edit-aware structural features could outperform token-only confidence signals, and how calibrated confidence can be integrated into downstream abstention, ranking, or human-in-the-loop repair workflows.

### 💡 TL;DR
This paper proposes RPM-Net, an open-set network intrusion detection model that learns per-class reciprocal points representing "non-class" regions, then uses distance-based classification and thresholding to detect unseen attacks without training on unknown samples. Across CICIDS2017 and UNSW-NB15, RPM-Net and its Fisher-regularized variant RPM-Net++ outperform prior open-set baselines, especially on unknown-threat detection metrics such as AUROC and AUPR-OUT.

### 🏆 Key Contributions
- Introduces a reciprocal point mechanism for multi-class open-set intrusion detection, where each known class is paired with a learnable representation of what that class is not.
- Designs adversarial margin constraints that shape the embedding geometry by bounding known-class samples relative to reciprocal points, leaving central open space for unknown attacks.
- Extends the base model to RPM-Net++ by adding Fisher discriminant regularization to improve intra-class compactness and inter-class separability.
- Provides a unified training framework that uses only known attack data while still enabling unknown threat detection at inference time.

### ⚙️ Methodology
RPM-Net is a multi-class open-set recognition framework built on a feature extractor, reciprocal points, margin constraints, and optionally Fisher regularization. The feature extractor is a 3-layer MLP with ReLU activations and dropout, mapping an input flow feature vector x in R^d to an embedding z=phi(x) in R^m. For each known class k, the model learns a reciprocal point P^k in the embedding space intended to represent the non-class region for that class. Instead of standard prototype learning, the model computes a reciprocal-point distance d(z,P^k)=d_e(z,P^k)-d_c(z,P^k), where d_e is normalized squared Euclidean distance and d_c is cosine similarity. This combines radial separation with angular alignment. Class logits are formed as gamma * d(z,P^k), so prediction is based on the class with maximum reciprocal-point distance. The training objective includes cross-entropy over these logits plus a margin loss and, for RPM-Net++, a Fisher loss. The margin loss uses learnable class-specific radii R^k and penalizes cases where a sample from class y_i lies farther than R^{y_i} from its corresponding reciprocal point under Euclidean distance. According to the paper, this constrains feature-space growth and induces a geometry where known classes occupy peripheral regions while a central open region is left for unknown samples. Fisher regularization computes within-class scatter S_w and between-class scatter S_b over embeddings in a minibatch and minimizes 1/(1+S_b/S_w), encouraging tighter class clusters and larger class separation. The total loss is L_total = alpha*L_CE + lambda*L_margin + beta*L_Fisher, with all weights set to 1.0. At inference, the model computes s(x)=max_k d(phi(x),P^k). The predicted known class is argmax_k d(phi(x),P^k), but if s(x) is below a threshold tau selected on validation data, the sample is rejected as unknown. This gives a simple distance-threshold open-set decision rule without requiring exposure to unknown training classes.

For CICIDS2017, the setup uses 5 known classes for training/testing among knowns, 2 validation classes for threshold selection, and 4 unknown test classes. The compared methods are a closed-set baseline, ODIN, OCN, and EVM. Reported metrics include macro-averaged precision, recall, F1-score for known-class classification, plus AUROC, AUPR-IN, and AUPR-OUT for open-set detection. On CICIDS2017, RPM-Net achieves precision 0.9987, recall 0.9987, F1 0.9987, AUROC 0.9601, AUPR-IN 0.9989, and AUPR-OUT 0.6523. This clearly improves unknown detection over the baseline AUPR-OUT 0.1046, OCN 0.2884, ODIN 0.1651, and EVM 0.2974, while maintaining near-perfect known-class classification. On UNSW-NB15, RPM-Net obtains precision 0.8022, recall 0.8053, F1 0.7950, AUROC 0.8675, AUPR-IN 0.8511, and AUPR-OUT 0.8555, outperforming the baseline, OCN, ODIN, and EVM on AUROC and AUPR-OUT. The ablation compares RPM-Net against RPM-Net++ with Fisher regularization. On CICIDS2017, RPM-Net++ improves AUROC from 0.9601 to 0.9735 and AUPR-OUT from 0.6523 to 0.6711, with a slight drop in F1 from 0.9987 to 0.9979. On UNSW-NB15, RPM-Net++ improves precision from 0.8022 to 0.8043, recall from 0.8053 to 0.8072, F1 from 0.7950 to 0.7955, AUROC from 0.8675 to 0.8850, AUPR-IN from 0.8511 to 0.8913, and AUPR-OUT from 0.8555 to 0.8664. The paper highlights final best results of 99.79% F1 and 67.11% AUPR-OUT on CICIDS2017, and 79.55% F1 and 86.64% AUPR-OUT on UNSW-NB15.

### 🌱 Insights & Implications
The central insight is that learning explicit non-class representations can be more effective for open-set intrusion detection than relying only on closed-set class prototypes or post-hoc confidence calibration. The reciprocal-point design gives a geometric interpretation of rejection: unknowns are expected to occupy embedding regions insufficiently distant from all class-specific non-class points. The Fisher extension suggests that improving embedding compactness and class separation directly helps unknown rejection, particularly on the harder UNSW-NB15 dataset. A notable strength is that the approach addresses realistic multi-class, class-imbalanced network-security conditions without requiring synthetic or real unknown samples during training. However, the paper leaves some open questions. The explanation of why maximizing distance to a class's reciprocal point yields stable class semantics could benefit from deeper theoretical analysis. The threshold tau depends on validation classes, so deployment performance may be sensitive to how representative those held-out classes are of future unknowns. The model is evaluated only on two tabular NIDS benchmarks and against a limited baseline set; comparisons with stronger recent open-set, out-of-distribution, contrastive, or transformer-based intrusion detection methods are missing. There is also limited discussion of computational cost, hyperparameter sensitivity, class-imbalance severity, and robustness to concept drift or adversarial manipulation. For related research, the method suggests a promising direction for designing interpretable open-space geometry in security models, and its reciprocal-point idea may transfer to malware classification, vulnerability triage, or other security tasks that must reject unseen categories.

## 📄 [The Defense Trilemma: Why Prompt Injection Defense Wrappers Fail?](https://arxiv.org/abs/2604.06436v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Manish Bhatt, Sarthak Munshi, Vineeth Sai Narajala, Idan Habler, Ammar Al-Kahfah, Ken Huang, Blake Gatto
📅 **Published:** 2026-04-07  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper proves a general impossibility result for prompt-injection wrapper defenses: any defense that continuously rewrites inputs while preserving the behavior of safe prompts cannot make all model outputs safe. It formalizes this as a trilemma among continuity, utility preservation, and completeness, and strengthens the result with quantitative conditions showing that near-boundary and even strictly unsafe regions must remain after defense.

### 🏆 Key Contributions
- Proves boundary fixation: any continuous, utility-preserving wrapper defense on a connected prompt space must leave at least one threshold-level boundary prompt unchanged.
- Establishes the defense trilemma: continuity, utility preservation, and completeness cannot all hold simultaneously for wrapper defenses.
- Derives an epsilon-robust lower bound under Lipschitz assumptions, showing a positive-measure neighborhood around fixed boundary points remains near the safety threshold after defense.
- Shows a persistent unsafe region under a transversality condition, proving that a positive-measure subset of prompts remains strictly unsafe even after applying the defense.
- Provides quantitative geometric bounds, including explicit volume and cone-measure lower bounds, and an asymmetric defense dilemma.
- Extends the impossibility theory to relaxed utility preservation, discrete finite settings, multi-turn interactions, stochastic defenses, and pipeline/capacity-parity settings.
- Mechanically verifies the theory in Lean 4 with a substantial formalization artifact and empirically validates the phenomena on three LLMs.

### ⚙️ Methodology
The paper models prompt defense as a preprocessing wrapper D: X -> X over a prompt space X, with an alignment deviation function f: X -> R and safety threshold tau. Safe, unsafe, and boundary regions are defined as S_tau = {x : f(x) < tau}, U_tau = {x : f(x) > tau}, and B_tau = {x : f(x) = tau}. A defense is utility-preserving if it leaves safe prompts unchanged, and complete if every defended prompt maps to a safe output. The core proof strategy is topological: because S_tau is open and D fixes S_tau, continuity implies the fixed-point set of D is closed; in a connected Hausdorff space with nonempty safe and unsafe regions, S_tau cannot be both open and closed, so the closure of S_tau must include non-safe points on the boundary. Therefore some boundary point z with f(z)=tau must satisfy D(z)=z, ruling out completeness. The authors then strengthen the argument analytically. Under Lipschitz regularity assumptions on f and D, they show that points within distance delta of a fixed boundary point obey f(D(x)) >= tau - L K delta, so the defense cannot uniformly move nearby prompts deep into the safe region. Under a transversality condition comparing the local growth of f against the defense-path contraction, they prove the existence of a positive-measure set on which f(D(x)) > tau, yielding a persistent unsafe region. The theory also includes relaxed notions of utility preservation, such as score-preserving or epsilon-approximate score preservation, where analogous boundary consequences still hold. Beyond continuous spaces, the authors derive discrete analogues on finite sets showing a tradeoff between injectivity and completeness: a complete defense must collapse distinct inputs, implying information loss, while injective defenses remain incomplete. They further extend the framework to multi-turn dialogues, stochastic defenses, and pipeline settings, and discuss interpolation from finite observations to continuous functions using the Tietze extension theorem. The proofs are fully mechanized in Lean 4.

### 🧪 Experiments
The paper reports empirical validation on three LLMs, although the provided text excerpt does not include the full experimental section details such as exact model names, datasets, baselines, or complete numerical tables. The empirical component is positioned as validation of the theoretical predictions rather than benchmark-driven model comparison. The quantitative theoretical results include: boundary fixation guarantees at least one unchanged threshold-level prompt; an epsilon-robust constraint f(D(x)) >= tau - L K dist(x,z) near a fixed boundary point; and, under transversality, existence of a positive-measure unsafe subset S with f(D(x)) > tau for all x in S and mu(S) > 0. The paper also states explicit volume and cone-measure lower bounds for unsafe regions and includes extensions to discrete, stochastic, and multi-turn cases. In addition to empirical validation, the authors provide a substantial formal verification artifact in Lean 4 consisting of 45 files and roughly 350 theorems with no admitted proofs.

### 🌱 Insights & Implications
The main insight is that the failure of prompt-injection wrappers is structural, not merely due to weak heuristics or insufficient attack coverage. If a defense must preserve the utility of safe prompts and behave continuously on a connected prompt manifold, it inevitably inherits boundary failures and, under stronger regularity, nontrivial unsafe regions. This has important implications for software security and automated defense design: purely input-side sanitization or rewriting layers cannot provide universal guarantees without either sacrificing utility, introducing discontinuities such as hard rejection/classification, or changing the model/training procedure itself. For security research, the results suggest shifting focus toward training-time alignment, architectural modifications, output-side enforcement, human-in-the-loop mechanisms, or systems that explicitly reject or route suspicious inputs instead of preserving all semantics. A useful limitation is that the theorems apply only to a specific class of defenses: single continuous wrappers D: X -> X with utility preservation over connected spaces. They do not rule out effective practical mitigations outside this class, such as discrete filters, adaptive thresholds, ensemble systems, or non-wrapper defenses. Open questions include how closely real prompt spaces satisfy the connectedness and regularity assumptions, how alignment deviation functions should be estimated in operational systems, and how these impossibility results interact with programmatic agents, tool use, or retrieval-augmented pipelines in practice.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and vulnerability repair because it gives a rigorous impossibility framework for a major class of LLM security mitigations: prompt-injection defense wrappers. For a researcher working on secure software engineering and automated repair, it helps distinguish which classes of defenses are fundamentally limited versus where effort should be redirected, such as model-level hardening, rejection-based security controls, formally analyzable pipelines, or training-time alignment. Its topological and formal-methods perspective also offers a strong theoretical foundation for evaluating security claims around LLM-based systems and agents.
