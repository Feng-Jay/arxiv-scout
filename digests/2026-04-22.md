## 📄 [SAGE: Signal-Amplified Guided Embeddings for LLM-based Vulnerability Detection](https://arxiv.org/abs/2604.19031v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Zhengyang Shan, Xu Qian, Jiayun Xin, Minghui Xu, Yue Zhang, Zhen Yang, Hao Wu, Xiuzhen Cheng
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper argues that LLMs often internally recognize vulnerability-related cues but those signals get numerically drowned out by dominant functional code semantics, a phenomenon the authors call Signal Submersion. To address this, they introduce SAGE, which uses layerwise task-conditional sparse autoencoders to recover and amplify vulnerability signals, yielding state-of-the-art vulnerability detection results and large MCC gains on standard and distribution-shifted benchmarks.

### 🏆 Key Contributions
- Introduces the concept of Signal Submersion as a core failure mode in LLM-based vulnerability detection, where vulnerability-relevant activations exist but are overwhelmed by general code semantics.
- Proposes SAGE, a backbone-agnostic framework that attaches task-conditional sparse autoencoders to intermediate LLM layers to isolate and amplify vulnerability signals without updating backbone LLM weights.
- Shows mechanistically that sparse manifold projection can increase internal signal-to-noise ratio by up to 12.7x, reversing the dilution of vulnerability information in deeper layers.
- Achieves state-of-the-art vulnerability detection performance on refined BigVul, PrimeVul, and PreciseBugs benchmarks, including strong gains under unseen-distribution settings.
- Demonstrates that a 7B model with SAGE can outperform much larger 34B and commercial frontier baselines, while maintaining transfer across 13 programming languages.

### ⚙️ Methodology
SAGE is built on the observation that vulnerability features are present inside LLM hidden states but become weak relative to dominant functional semantics as representations propagate through the model. The framework therefore performs active signal recovery rather than relying on the final-layer embedding alone. Concretely, SAGE trains task-conditional Sparse Autoencoders (SAEs) independently for different backbone layers to encode hidden representations into sparse latent features and reconstruct them, while coupling the SAE with a vulnerability classifier through a joint objective. The sparse bottleneck is intended to project entangled code embeddings onto a sparse manifold where vulnerability-relevant features are more separable and amplified, while irrelevant semantic background is suppressed. The method also includes layer selection, choosing intermediate layers where vulnerability cues are strongest before they are submerged in later layers. Importantly, SAGE is designed to integrate with decoder-only LLM backbones without modifying or fine-tuning their parameters, making it a lightweight add-on over frozen representations. By avoiding full backbone training and operating on extracted hidden states, the framework aims to improve efficiency while preserving general code understanding from the original LLM.

### 🧪 Experiments
The evaluation covers refined versions of three vulnerability detection benchmarks: BigVul, PrimeVul, and PreciseBugs, with emphasis on realistic and strict evaluation settings including unseen distributions and temporal-split style robustness concerns raised in prior work. The paper reports experiments across four backbone model architectures and compares against 10 prior state-of-the-art methods as well as large commercial and frontier models such as GPT-5.2-Codex, Gemini 3 Pro, and Claude Opus 4.5. The main metric emphasized is Matthews Correlation Coefficient (MCC), appropriate for imbalanced security datasets; the abstract and introduction highlight up to 318% MCC improvement on unseen distributions and 319% improvement on classic datasets. Mechanistic evaluation measures internal signal-to-noise ratio, showing SAGE boosts SNR by up to 12.7x. The paper also reports robust performance across 13 programming languages and finds that a 7B backbone augmented with SAGE can outperform 34B baselines, indicating better efficiency than simply scaling model size. The text provided does not include the full result tables, so exact per-dataset baseline numbers, additional metrics, and ablation details are not fully available here.

### 🌱 Insights & Implications
For software security research, the paper’s key insight is that poor vulnerability detection by LLMs may stem less from absence of relevant knowledge and more from representational interference: vulnerability cues exist internally but are too weak at decision time. This reframes the problem from model scaling or prompting to signal extraction and representation engineering. The use of sparse autoencoders suggests a promising bridge between mechanistic interpretability and practical security tasks, showing that internal representation surgery can yield large downstream gains without full retraining. A notable implication is that intermediate-layer features may be more useful than final-layer embeddings for subtle semantic security tasks. Likely limitations include dependence on the chosen backbone families, the stability of layer selection across tasks, and possible sensitivity on low-resource language subsets, which the authors explicitly caution about. Open questions include whether the same Signal Submersion phenomenon appears in vulnerability localization, exploit generation, or automated repair; whether sparse features correspond to human-interpretable vulnerability concepts such as taint flow or memory misuse; and how SAGE compares to other intervention methods like activation steering or adapter tuning under equal compute budgets.

### 🎯 Why This Paper
This paper is highly relevant to vulnerability detection and broader software security because it directly targets a major bottleneck in LLM-based detection accuracy on realistic benchmarks. It is especially useful for researchers in automated vulnerability analysis, software maintenance, and repair because it offers a lightweight, backbone-preserving method to improve detection under distribution shift, a mechanistic explanation for why LLMs miss subtle bugs, and a potential foundation for extending representation-level interventions to vulnerability localization, root-cause analysis, and repair pipelines.

## 📄 [Evaluating LLM-Generated Obfuscated XSS Payloads for Machine Learning-Based Detection](https://arxiv.org/abs/2604.19526v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Divyesh Gabbireddy, Suman Saha
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR, cs.LG, cs.SE

### 💡 TL;DR
This paper builds a pipeline that generates obfuscated XSS payloads with deterministic rules and LLMs, then validates them by comparing their actual browser runtime behavior rather than string similarity. Fine-tuning an LLM on behavior-preserving obfuscation pairs improves behavioral match from 0.15 to 0.22, but most generated payloads still fail to preserve execution behavior and do not improve downstream XSS detection.

### 🏆 Key Contributions
- Introduces a structured end-to-end pipeline for XSS obfuscation generation that combines deterministic multi-step transformations, LLM-based generation, runtime validation, and downstream detector evaluation.
- Proposes behavior-filtered training data construction, where only obfuscation chains that preserve observed browser behavior are used as source-target pairs for LLM fine-tuning.
- Develops a browser-based runtime evaluation protocol that judges generated payload quality by execution traces such as alerts, console output, network activity, and errors instead of syntactic resemblance.
- Empirically shows that fine-tuning on behavior-preserving examples yields a measurable improvement over an untuned baseline LLM, while also demonstrating the current limitations of LLMs for behaviorally correct adversarial payload generation.
- Assesses whether generated and behavior-filtered obfuscated payloads help machine-learning-based XSS detection, finding no performance gains but also no major degradation when filtered samples are added.

### ⚙️ Methodology
The paper proposes a multi-stage pipeline. First, starting from original malicious XSS payloads, the authors generate deterministic obfuscation chains using a fixed set of transformations: hexadecimal escaping, Base64 encoding with dynamic evaluation, URL encoding, string splitting, comment insertion, and case manipulation. In the reported experiments, chain construction is constrained to five fixed two-step recipes: hex_escape->split_strings, comment_insertion->hex_escape, split_strings->comment_insertion, case_swap->split_strings, and comment_insertion->case_swap. For each original payload P0, intermediate payloads and final two-step variants P2 are recorded with transformation metadata. Second, these deterministic obfuscation outputs are executed in a controlled browser environment to test whether the transformed payload preserves the observable runtime behavior of the original. The runtime checker records browser-visible effects including alert triggers, console outputs, network requests, and execution errors. A chain is kept only if the execution trace of the transformed payload matches that of the original under the same conditions. Third, retained P0->P2 pairs are converted into sequence-to-sequence training data for LLM fine-tuning. The authors then compare two generation settings: an untuned baseline language model and a fine-tuned model trained on these behavior-filtered obfuscation pairs. Given an original payload, the model generates a candidate obfuscated payload, potentially beyond the hand-written transformation recipes. Fourth, generated outputs are subjected to the same browser-based runtime evaluation. A generated sample is considered valid only if its observed trace matches that of the source payload. Finally, these generated payloads, especially the behavior-filtered subset, are added to a downstream machine-learning XSS detection dataset to test whether the augmented data improves classification performance or at least preserves detector quality.

### 🧪 Experiments
The evaluation studies both generation quality and downstream detection impact. For generation, the authors compare an untuned baseline LLM against a fine-tuned model trained on behavior-preserving source-target obfuscation pairs derived from deterministic two-step transformation chains. The main metric is runtime behavior match rate: the fraction of generated payloads whose browser execution trace matches the original payload. Using the browser-based execution framework, the untuned baseline achieves a behavior match rate of 0.15, while fine-tuning improves this to 0.22, which the paper reports as a 46.7% relative improvement. Despite this gain, the absolute match rate remains low, showing that most generations are still behaviorally invalid. The runtime evaluator compares observable outcomes such as alert events, console messages, network requests, and execution errors, making this a semantic/behavioral rather than lexical evaluation. For downstream classification, the authors augment an XSS detection setting with generated payloads and also test a behavior-filtered subset of those generations. The main finding is that adding generated samples does not improve detection performance in this experimental setup, likely because the generated set is a small fraction of the overall dataset and many raw generations are invalid. However, filtering by runtime behavior allows generated samples to be incorporated without materially degrading the classifier, suggesting that semantic validation is necessary before using synthetic adversarial data for model training.

### 🌱 Insights & Implications
The central insight is that syntactically plausible obfuscation is not enough for security data generation; runtime validation is essential because minor changes can break XSS execution while still looking malicious. The paper exposes a clear gap between surface-level generation quality and true behavioral preservation, even after fine-tuning, which is highly relevant for adversarial ML and secure data augmentation. A second insight is that high-variance generated attack samples are not automatically useful for detector training: without semantic correctness, they can add noise instead of meaningful coverage. The modest increase from 0.15 to 0.22 suggests that behavior-preserving transformations can be learned to some extent, but current LLMs remain unreliable for this task. Limitations include the restricted deterministic recipe set, the likely dependence on the chosen browser/runtime environment, the use of a controlled execution setting that may not capture broader real-world contexts, and the small practical effect on downstream detection due to limited validated sample volume. Open questions include whether stronger constrained decoding, execution-guided generation, reinforcement learning with runtime feedback, or richer behavioral equivalence metrics could improve preservation rates; whether broader transformation families and larger datasets would help; and whether similar runtime-filtered generation could be more effective for other web attacks or vulnerability classes.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated security data generation because it studies how LLMs can synthesize realistic obfuscated attack inputs and evaluates them with execution-based correctness checks. For a researcher interested in vulnerability detection and repair, it offers a concrete methodology for behavior-aware adversarial example generation, highlights why naive synthetic augmentation can fail, and suggests a practical direction for building more reliable training and evaluation pipelines for security-focused machine learning systems.

## 📄 [DebugRepair: Enhancing LLM-Based Automated Program Repair via Self-Directed Debugging](https://arxiv.org/abs/2604.19305v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Linhao Wu, Yifei Pei, Zhen Yang, Kainan Li, Zhonghang Lu, Hao Tan, Xiran Lyu, Jia Li, Yizhou Chen, Pengyu Xue, Kunwu Zheng, Dan Hao
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.SE

### 💡 TL;DR
DebugRepair is an LLM-based automated program repair framework that improves iterative patch refinement by letting the model perform self-directed debugging to collect intermediate runtime states, rather than relying only on stack traces and test outcomes. Across Java and Python benchmarks, this debugging-aware feedback substantially improves repair effectiveness and achieves state-of-the-art results over 15 baselines.

### 🏆 Key Contributions
- Introduces DebugRepair, a feedback-based APR framework that augments LLM patch refinement with intermediate runtime evidence gathered through simulated debugging.
- Proposes test semantic purification to isolate the minimal failure-triggering test context and reduce noise from irrelevant assertions and execution paths.
- Designs simulated instrumentation, where the LLM inserts targeted debugging statements to expose runtime states, with a rule-based fallback to preserve robustness when generated instrumentation is invalid.
- Develops debugging-driven conversational repair with hierarchical outer/inner loops that iteratively alternate between debugging, patch generation, and refinement using prior attempts plus newly observed traces.
- Demonstrates state-of-the-art repair performance on Defects4J, QuixBugs, and HumanEval-Java, including strong gains with GPT-3.5, DeepSeek-V3, and multiple additional backbone LLMs.
- Provides ablation evidence showing that both test purification and runtime-state-guided debugging materially contribute to improved repair success.

### ⚙️ Methodology
DebugRepair is built as a self-directed debugging pipeline for feedback-based LLM APR. Its central premise is that outcome-level failure symptoms such as assertion failures and stack traces often do not reveal the runtime state transitions needed for root-cause analysis, so the system explicitly gathers those states and feeds them back into repair.

The framework has three main components.

First, test semantic purification preprocesses failing tests to extract the minimal semantically relevant failure-triggering context. The paper motivates this by noting that real unit tests often bundle multiple assertions and scenarios for the same focal method, many of which are unrelated to the observed failure. DebugRepair uses static program slicing over the failing test to keep only the subset directly related to the failure, producing a purified test that reduces irrelevant execution and limits noisy logs during later debugging.

Second, simulated instrumentation enables the LLM to behave like a developer adding print/debug statements. When initial repair based only on failure symptoms is insufficient, the framework enters a debugging phase. Given the buggy function and the purified failing test, the LLM chooses key execution points, inserts instrumentation statements along likely fault-relevant paths, and the instrumented program is executed to collect runtime traces such as intermediate variable values and control-flow-relevant states. Because LLM-generated instrumentation can fail to compile or can distort behavior, DebugRepair includes a deterministic rule-based fallback mechanism that inserts safer, predefined debugging statements when LLM instrumentation is invalid or semantically inconsistent. The resulting execution output is transformed into structured runtime feedback.

Third, debugging-driven conversational repair organizes repair as a hierarchical iterative process. The outer loop controls whether to gather new debugging evidence through instrumentation and execution, while the inner loop performs patch generation and refinement using the current evidence. The LLM is not prompted in a one-shot manner; instead, it receives the buggy code, purified test information, prior repair attempts, failure feedback, and newly collected runtime traces, and then incrementally revises patches through multi-turn conversation. This process continues until a plausible patch passes the full test suite or a repair budget is exhausted.

Overall, the method combines static reduction of test noise, dynamic extraction of runtime evidence, and iterative conversational refinement so the LLM can reason from observed internal states to likely bug causes instead of guessing from surface-level symptoms alone.

### 🧪 Experiments
The evaluation covers three widely used APR benchmarks across Java and Python: Defects4J (including v1.2 and v2.0), QuixBugs, and HumanEval-Java. The study compares DebugRepair against 15 representative baselines spanning template-based, learning-based, and LLM-based APR methods; among these are six recent state-of-the-art LLM-based techniques from retrieval-based, feedback-based, and hybrid paradigms. The paper also evaluates DebugRepair with multiple backbone LLMs from different families and sizes, including GPT-3.5 and DeepSeek-V3, plus five additional models to assess model-agnosticity.

The primary reported metric is the number of bugs correctly fixed, typically meaning the generated patch is plausible and validated by the benchmark test suite. On Defects4J with GPT-3.5, DebugRepair fixes 224 bugs, outperforming prior state-of-the-art LLM-based approaches by an average of 26.2%. With DeepSeek-V3, it fixes 295 Defects4J bugs, beating the second-best baseline by 59 bugs. Across five additional backbone LLMs, DebugRepair improves repair performance by 51.3% on average over vanilla model settings without the proposed framework.

The paper states that DebugRepair consistently outperforms existing approaches across the three benchmarks and two languages. It also includes ablation studies to isolate the effect of the core components, showing that test semantic purification and the simulated debugging mechanism both contribute significantly to the overall gains.

### 🌱 Insights & Implications
The paper’s key insight is that LLM-based APR is bottlenecked not only by patch generation quality but by the granularity of feedback provided during iterative repair. Stack traces and pass/fail outcomes show symptoms, but intermediate runtime states are often the missing link between symptom and root cause. By operationalizing debugging as an explicit part of the repair loop, DebugRepair reduces the need for the LLM to speculate about hidden execution behavior.

For software maintenance and repair research, this suggests a promising direction: APR systems may benefit from richer dynamic evidence pipelines rather than solely stronger models or larger retrieval corpora. The test purification step is also important because raw failing tests can introduce substantial noise; this indicates that context selection remains a major issue in LLM-based software engineering workflows.

The work is especially relevant to debugging-intensive bug classes such as incorrect variable usage, state inconsistencies, and logic errors where runtime values disambiguate multiple superficially plausible fixes. The motivating example also highlights a common APR failure mode: generating symptom-masking patches instead of root-cause fixes.

Likely limitations and open questions include the cost and latency of repeated instrumentation and execution, dependence on executable tests and environments, and the possibility that print-based traces still miss deeper semantic causes for complex bugs. Another open direction is whether the same idea can be extended from correctness bugs to security bugs, where dynamic evidence might include taint flows, exploit-triggering states, or sanitizer violations. It also remains important to assess overfitting risk beyond test-suite adequacy and to understand how robust the approach is when tests are weak, flaky, or incomplete.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, and automated program repair because it advances feedback-based APR from simple test-outcome refinement to debugging-aware repair with concrete runtime evidence. For researchers working on APR, vulnerability detection/repair, and LLM-based program analysis, DebugRepair offers a practical and model-agnostic framework for integrating dynamic execution traces into iterative repair, which could inspire analogous techniques for security patching, exploit reproduction, root-cause localization, and repair validation.

## 📄 [Security Is Relative: Training-Free Vulnerability Detection via Multi-Agent Behavioral Contract Synthesis](https://arxiv.org/abs/2604.19012v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Yongchao Wang, Zhiqiu Huang
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper argues that vulnerability detection fails when treated as a global code classification problem because the same code can be secure or insecure depending on project-specific behavioral contracts. It introduces Phoenix, a training-free multi-agent pipeline that synthesizes explicit Gherkin security contracts from vulnerable/patched pairs and then judges code by contract compliance, achieving F1 0.825 and Pair-Correct 64.4% on PrimeVul Paired, well above prior systems.

### 🏆 Key Contributions
- Introduces the semantic ambiguity problem in vulnerability detection: identical or near-identical code may receive opposite labels because security depends on project-specific behavioral contracts rather than syntax alone.
- Proposes Phoenix, a zero-shot, training-free multi-agent framework with three stages: Semantic Slicer, Requirement Reverse Engineer, and Contract Judge.
- Introduces Gherkin behavioral specifications as an intermediate representation for vulnerability reasoning, converting detection from pattern matching into explicit contract verification.
- Demonstrates state-of-the-art performance on the PrimeVul Paired benchmark using open-source 7B-14B models, outperforming much larger retrieval-augmented and multi-agent baselines.
- Provides a broad ablation study over 25 configurations and 5 model families showing that structured Gherkin contracts are the dominant factor behind performance gains.
- Presents qualitative error analysis showing that a notable fraction of apparent false positives are actually legitimate security concerns in patched code, supporting the thesis that security labels are relative to behavioral expectations.

### ⚙️ Methodology
Phoenix is a training-free inference-time framework that decomposes vulnerability detection into specialized reasoning stages instead of directly prompting one model to output a binary label. The first stage, Semantic Slicer, extracts only the vulnerability-relevant context from the target code and its associated patch information, aiming to avoid the attention dilution that occurs when full inter-procedural context is naively appended. The second stage, Requirement Reverse Engineer, takes the vulnerable/patched contrast and synthesizes a behavioral contract in Gherkin Given-When-Then form. This specification captures the intended security requirement that distinguishes the bad version from the fixed one, effectively externalizing the latent security policy into a structured natural-language artifact. The third stage, Contract Judge, evaluates an individual code sample against the synthesized contract via strict compliance checking, deciding whether the code satisfies or violates the required behavior. The key design choice is the explicit intermediate contract: instead of asking an LLM to infer vulnerability from patterns or retrieved analogues, Phoenix asks it to reason about whether concrete behavior conforms to a project-specific security requirement. The framework is zero-shot and model-agnostic, relying on open-source LLMs rather than fine-tuning. Its core hypothesis is that explicit contract synthesis resolves semantic ambiguity by making the comparison criterion local, sample-specific, and behavior-centered rather than globally statistical.

### 🧪 Experiments
The main evaluation is conducted on the PrimeVul Paired Test Set, a benchmark designed with stricter deduplication, temporal splitting, and cleaner labeling than legacy datasets such as Big-Vul. The paper compares Phoenix primarily against recent strong baselines including RASM-Vul, a retrieval-augmented system using DeepSeek-V3, and VulTrial, a multi-agent reasoning approach. Phoenix achieves F1 = 0.825 and Pair-Correct = 64.4%, compared with RASM-Vul at F1 = 0.668 and Pair-Correct = 21.4%, and VulTrial at F1 = 0.563 and Pair-Correct = 18.6%. The paper emphasizes efficiency as well: Phoenix uses fully open-source models in the 7B-14B range, up to 48 times smaller than DeepSeek-V3's 671B total parameter size. Beyond the headline comparison, the authors run ablations over 25 configurations spanning 5 model families. These ablations show that adding Gherkin specifications yields the decisive improvement, with gains reported in the range of +0.09 to +0.35 F1 depending on configuration. The study also reports a relationship between specification quality and downstream detection quality, supporting the claim that the intermediate contract, not merely multi-agent decomposition, drives the gains. Qualitative error analysis examines Phoenix's false positives and finds that 17 of 97 cases, about 18%, correspond to real residual security concerns in code labeled as patched, suggesting benchmark labels may understate nuanced security issues.

### 🌱 Insights & Implications
The paper's central insight is highly relevant to modern vulnerability detection: benchmark collapse under PrimeVul is not just a data issue but evidence that vulnerability labels are context-dependent and often not globally learnable from syntax or generic semantics alone. By reframing detection as behavioral contract verification, the work offers an alternative to both supervised classification and retrieval-based similarity matching, which are brittle under distribution shift and weak on zero-day or project-specific logic flaws. The strong ablation result suggests that structured intermediate representations may be more important than sheer model scale for secure code reasoning. For software security research, this implies that future systems may benefit from synthesizing explicit security invariants, expected behaviors, or acceptance criteria before making vulnerability decisions. Limitations and open questions include dependence on the quality of synthesized contracts, potential sensitivity to prompt design and agent coordination, and the likely need for paired vulnerable/fixed examples or patch context to reverse engineer the contract. It is also unclear how well the approach generalizes to settings without clear patch pairs, to large multi-file systems with hidden runtime assumptions, or to automated repair where a contract must be synthesized from incomplete evidence. Another implication is evaluation: if some false positives are actually valid security findings, existing ground truth may be too coarse for contract-relative reasoning, motivating richer benchmark annotations and perhaps contract-aware evaluation metrics.

### 🎯 Why This Paper
This paper is directly relevant to software engineering, maintenance, software security, vulnerability detection, and vulnerability repair because it tackles the core failure mode of current DL-based detectors under realistic evaluation and proposes a practical alternative centered on behavioral semantics. For a researcher working on automated program repair and vulnerability repair, the idea of synthesizing explicit security contracts from vulnerable/patched pairs is especially valuable: it can inform repair objectives, patch validation, benchmark design, and hybrid detection-repair pipelines that verify whether candidate fixes satisfy intended security behavior rather than merely match historical patterns.

## 📄 [Cyber Defense Benchmark: Agentic Threat Hunting Evaluation for LLMs in SecOps](https://arxiv.org/abs/2604.19533v1)  `80% relevant`

👤 **Authors:** Alankrit Chona, Igor Kozlov, Ambuj Kumar
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper introduces a realistic benchmark for evaluating LLM agents on open-ended threat hunting over raw Windows event logs, framed as finding exact malicious event timestamps via iterative SQL querying. Across 26 attack campaigns and five frontier models, all systems perform very poorly, with the best model achieving only 3.8% average recall of malicious events and none meeting a proposed minimum deployment threshold.

### 🏆 Key Contributions
- Introduces the Cyber Defense Benchmark, a benchmark specifically targeting agentic threat hunting in Security Operations Centers rather than curated question-answering tasks.
- Transforms 106 real attack procedures from the OTRF Security-Datasets corpus into a Gymnasium reinforcement-learning environment for interactive evaluation.
- Uses realistic log-search conditions: large in-memory SQLite databases containing 75,000-135,000 raw Windows event records, with no hints or guided questions.
- Applies deterministic campaign simulation with time-shifting and entity obfuscation to reduce memorization and preserve evaluation realism.
- Defines a concrete threat-hunting task where agents must iteratively issue SQL queries and explicitly flag exact malicious timestamps, scored against Sigma-rule-derived ground truth.
- Provides a broad empirical evaluation of five frontier LLMs across campaigns spanning 86 MITRE ATT&CK sub-techniques and 12 tactics, showing uniformly weak performance.
- Proposes an operationally meaningful passing criterion of at least 50% recall on every ATT&CK tactic as a minimum bar for unsupervised SOC deployment.

### ⚙️ Methodology
The benchmark is designed as an agentic, open-ended cyber defense task. Each evaluation episode gives the model access to an in-memory SQLite database populated with raw Windows event logs from a simulated attack campaign. The campaigns are derived from 106 real attack procedures in the OTRF Security-Datasets corpus and cover 86 MITRE ATT&CK sub-techniques across 12 tactics. To make episodes realistic while limiting straightforward data memorization, the benchmark uses a deterministic campaign simulator that time-shifts attacks and obfuscates entities in the original logs. The environment is wrapped in Gymnasium, suggesting a step-based interaction loop suitable for reinforcement-learning-style agents, though here it is mainly used as a standardized interactive benchmark. The agent is not provided with a predefined question, attack hypothesis, or candidate indicators; instead, it must independently investigate the raw log database by issuing SQL queries over tens of thousands of records, inspect evidence, infer suspicious behavior, and submit explicit flags corresponding to exact timestamps of malicious events. Scoring is CTF-style: submitted timestamps are matched against ground-truth malicious events derived from Sigma rules. This setup tests end-to-end analyst behavior, including search strategy, hypothesis generation, evidence aggregation, and precise actioning, rather than static knowledge recall or single-step classification.

### 🧪 Experiments
The evaluation covers 26 campaigns representing 105 of the 106 attack procedures in the benchmark. Each campaign contains roughly 75,000 to 135,000 Windows log records stored in SQLite. The authors evaluate five frontier models: Claude Opus 4.6, GPT-5, Gemini 3.1 Pro, Kimi K2.5, and Gemini 3 Flash. The core metric is successful identification of malicious events via exact timestamp flags, measured against Sigma-rule-derived ground truth; the abstract emphasizes average recall of malicious events and tactic-level recall across ATT&CK tactics. The strongest model, Claude Opus 4.6, correctly flags only 3.8% of malicious events on average. No run by any model identifies all malicious events in a campaign. For operational evaluation, the paper defines a passing threshold as achieving at least 50% recall on every ATT&CK tactic, intended as a minimum standard for unsupervised SOC deployment. Under this criterion, no model passes: the best model reaches the threshold on only 5 of 13 tactics, while the other four models reach it on zero tactics. The main empirical result is therefore not relative ranking but the absolute inadequacy of current frontier LLM agents for realistic threat hunting in this setting.

### 🌱 Insights & Implications
A central insight is that strong performance on security QA benchmarks does not translate to open-ended, evidence-driven investigative work over noisy operational telemetry. The benchmark isolates a harder problem that combines retrieval, iterative exploration, temporal reasoning, adversarial context reconstruction, and high-precision decision making. The poor results imply that current LLM agents likely struggle with multi-step search planning, schema/log understanding, persistence over long interaction horizons, and converting partial evidence into exact event-level detections. For software security and automated defense research, this highlights a gap between benchmarkable language competence and deployable SOC capability. A likely limitation is that the benchmark focuses on Windows event logs and exact timestamp identification, which may understate models that can describe suspicious behavior but fail at precise event localization; however, that precision requirement is operationally justified for analyst workflows. Another open question is how much of the failure comes from the base models versus the agent scaffolding, prompt design, tool-use constraints, or lack of specialized memory and retrieval mechanisms. The Gymnasium framing also suggests future work on reinforcement learning, search policies, or hybrid neuro-symbolic agents for cyber operations. The benchmark may become useful not only for LLM evaluation but also for training and testing autonomous detection, log analysis, and attack reconstruction systems.

### 🎯 Why This Paper
This paper is highly relevant because it evaluates LLM-based agents on a realistic cybersecurity workflow directly related to vulnerability detection, incident analysis, and defensive automation. For a researcher interested in software engineering, software maintenance, software security, automated repair, and vulnerability detection/repair, it provides a rigorous benchmark and a cautionary empirical result: current frontier LLMs are not yet reliable for unsupervised, tool-using security operations. That makes it valuable both as an evaluation resource and as motivation for more robust agent architectures, better task decomposition, and security-specific reasoning methods.

## 📄 [EvoPatch-IoT: Evolution-Aware Cross-Architecture Vulnerability Retrieval and Patch-State Profiling for BusyBox-Based IoT Firmware](https://arxiv.org/abs/2604.19496v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yinhao Xiao, Huixi Li, Yongluo Shen
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR

### 💡 TL;DR
The paper introduces EvoPatch-IoT, a stripped-binary retrieval framework for BusyBox-based IoT firmware that uses cross-architecture binary features plus historical evolution signals to map anonymous functions to upstream counterparts and assess vulnerability patch state. On a large 57-version BusyBox benchmark, it substantially outperforms prior stripped-compatible baselines in cross-architecture retrieval and shows strong transfer for version-change and CVE patch-state profiling.

### 🏆 Key Contributions
- A large-scale BusyBox IoT firmware benchmark spanning 57 versions, 5 architectures, 270 unstripped binaries, 285 stripped binaries, 130 source releases, and over 155k high-confidence stripped-to-unstripped function matches
- EvoPatch-IoT, an evolution-aware stripped-compatible retrieval framework that fuses anonymous instruction/context features, graph statistics, per-binary geometric priors, and historical function prototypes
- A reproducible alignment and scoring pipeline that links stripped functions to labeled upstream functions without relying on symbols, source paths, or version strings at test time
- Comprehensive evaluation over 1,020 directed architecture pairs and 128,084 query functions showing consistent gains over 12–13 baseline families
- Demonstration that retrieval outputs can support downstream vulnerability auditing tasks, including version-change transfer and patch-state profiling for CVE-2021-42386

### ⚙️ Methodology
EvoPatch-IoT targets BusyBox function retrieval in stripped IoT firmware where symbol names and reliable version metadata are unavailable. This design explicitly models software evolution rather than treating similarity as a purely static cross-architecture problem. The resulting system ranks candidate homologous functions for a stripped query, supports transfer across versions, and can aggregate evidence at the binary level to estimate whether a target is closer to vulnerable or patched code lineage.

The main insight is that stripped-binary vulnerability auditing in IoT firmware benefits from explicitly modeling software evolution, not just architecture-invariant function similarity. Historical function prototypes appear to act as a strong regularizer when vendor binaries differ by version, architecture, and patch lineage. ### 💡 TL;DR
This paper proposes ELLF, an ELF-compatible binary format augmented with compact compiler-generated metadata that records instruction boundaries, pointer intent, control-flow targets, and memory region structure so that binary disassembly and lifting become decidable rather than heuristic. ### 🏆 Key Contributions
- Defines a 'middle ground' binary distribution format, ELLF, that preserves closed-source distribution while exposing enough compiler intent to make binaries traversable, memory-structured, and instrumentable.

### ⚙️ Methodology
The paper frames binary lifting as a sequence of abstraction steps, each normally blocked by undecidable recovery problems, and then supplies compiler-produced 'oracles' that make each step exact. The lifting pipeline starts from a raw stripped binary and proceeds through: (1) unsymbolized assembly, using an instruction oracle that marks every address intended by the compiler to begin an instruction, eliminating ambiguity between code and embedded data; (2) coarsely symbolized assembly, using a pointer oracle that classifies values as text pointers, data pointers, or raw constants and records operand-level pointer intent so addresses can be replaced with labels without misclassifying immediates; and then independent symbolization of text, stack, and data. Text symbolization uses metadata describing control-flow targets, function boundaries, and intended destinations of indirect control transfers so CFG recovery and traversal become exact rather than heuristic. Stack symbolization uses compiler-known frame structure and bounded regions for locals/spills so stack memory is decomposed into meaningful independent objects rather than treated as a flat byte array. Data symbolization analogously records the decomposition of static/global data sections into bounded objects and pointer-bearing fields. Together these produce 'fully symbolized assembly' that is traversable, memory-structured, and amenable to rewriting. On the production side, the authors implement a tool that gathers this information at build time from compiler outputs, encodes it efficiently to avoid DWARF-scale bloat, and inserts it into the ELF as ELLF metadata. On the consumption side, a lifter reads the metadata and the stripped binary, reconstructs correct disassembly, emits symbolized assembly, and enables reassembly with standard toolchains. The key methodological idea is to move hard binary analysis problems from post hoc inference to build-time ground-truth capture, leveraging information already available to the compiler but normally discarded.

The evaluation compiles a comprehensive set of real-world C and C++ binaries from the LLVM Test Suite, augments them with ELLF metadata, strips the resulting binaries, and then attempts to recover fully symbolized assembly using only the stripped binary plus the embedded metadata. The recovered assembly is reassembled with a standard GCC-based toolchain, and the regenerated binaries are validated against the original test suites to check behavioral equivalence. This end-to-end compile -> strip -> lift -> instrument/reassemble -> test methodology is used as the main correctness criterion, demonstrating that the metadata is sufficient for exact lifting and recompilation. The paper also compares metadata footprint against DWARF, reporting that ELLF metadata is roughly 17% of the size of DWARF debug information, and measures operational impact, finding no changes to runtime behavior or performance from embedding the metadata. The headline quantitative results are therefore: successful lifting and recompilation across the evaluated C/C++ benchmark set without altering behavior, negligible/no runtime overhead, and substantially lower metadata size than DWARF.

### 🌱 Insights & Implications
The paper's main insight is that many difficult binary analysis tasks are only hard because build-time semantic intent is discarded; if compilers preserve just the right metadata, disassembly, CFG recovery, and memory modeling become straightforward and trustworthy. This has strong implications for software security and maintenance: patching, instrumentation, fuzzing, verification, and vulnerability analysis on closed-source software could become far more reliable without revealing source code or rich debug information. For software engineering research, the work suggests a new design point between stripped binaries and full debug builds: binaries that are still proprietary but analysis-friendly. A notable strength is the focus on exactness and recompilability rather than best-effort reverse engineering. Likely limitations and open questions include ecosystem adoption because compilers, linkers, loaders, and vendors would need to support the new metadata; how well the approach generalizes beyond the evaluated LLVM/GCC-oriented C/C++ setting, especially to hand-written assembly, obfuscated code, JIT-generated code, mixed-language binaries, or highly unusual linker behaviors; whether the embedded metadata itself creates a new attack surface or information-leak channel; and how robust the format remains under post-link binary transformations, packing, or aggressive optimization pipelines. Another open direction is whether similar metadata could support richer downstream tasks such as automated vulnerability repair, precise taint analysis, exploit mitigation insertion, or verified binary rewriting.

## 📄 [CASCADE: Detecting Inconsistencies between Code and Documentation with Automatic Test Generation](https://arxiv.org/abs/2604.19400v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Tobias Kiecker, Jan Arne Sparka, Martin Reuter, Albert Ziegler, Lars Grunske
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.SE

### 💡 TL;DR
CASCADE detects semantic inconsistencies between method code and documentation by generating tests from documentation and only reporting an issue when the real implementation fails while a documentation-generated implementation passes the same test. This execution-based dual-check design prioritizes precision and was validated on a new real-world dataset plus multi-language projects, finding 13 previously unknown inconsistencies, 10 of which were fixed.

### 🏆 Key Contributions
- Introduces CASCADE, a precision-first inconsistency detector that combines LLM-based test generation from documentation with LLM-based code generation as a validation filter.
- Proposes a dual-evidence criterion for reporting inconsistencies: documentation-derived tests must fail on the original code and pass on independently generated code from the same documentation.
- Constructs a novel manually curated dataset of real semantic code-documentation inconsistencies from executable open-source Java projects, with both inconsistent and consistent examples.
- Demonstrates practical effectiveness on Java, C#, and Rust repositories by uncovering previously unknown inconsistencies that were subsequently confirmed and often fixed by developers.

### ⚙️ Methodology
CASCADE targets method-level semantic inconsistencies between source code and natural-language documentation. The workflow is execution-based and consists of two phases. In Phase 1, the system uses an LLM to generate unit tests, including assertions/oracles, directly from the method documentation. These tests represent the behavior implied by the documentation. The generated tests are then executed against the existing implementation. Any failing test is treated as an indicator that the implementation may violate the documented behavior. Because LLM-generated tests can be incorrect or hallucinated, CASCADE adds a second validation phase to suppress false positives. In Phase 2, the same documentation is given to an LLM to synthesize a fresh implementation of the documented method. The previously generated documentation-derived tests are then executed on this synthesized implementation. CASCADE reports an inconsistency only if a test fails on the original implementation but passes on the generated implementation. This cross-check is based on the assumption, supported by prior work cited by the paper, that code generation is often more reliable than test generation and that independently generated code and tests passing together increases confidence that both correctly capture the documentation. The overall design is therefore precision-first: it intentionally sacrifices recall to reduce developer-facing noise. The paper positions this as an alternative to static heuristics and direct LLM classification approaches, which either have narrow scope or are volatile and prone to false alarms. ### ⚙️ Methodology
MuCoCo treats consistency as a metamorphic property: if two code queries are semantically equivalent, a reliable code LLM should produce outputs with equivalent behavior. 📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR, cs.AI, cs.OS

### 💡 TL;DR
GAAP is a privacy-preserving execution environment for AI agents that deterministically prevents unauthorized disclosure of private user data, even if the prompt, model, or model provider is malicious or compromised. It combines information-flow tracking, persistent disclosure history, and user-driven permission collection to block known data-exfiltration attacks while maintaining task success comparable to less secure systems.

### 🏆 Key Contributions
- Introduces GAAP, an AI-agent execution environment that guarantees confidentiality of private user data without trusting the LLM, prompt, context, or model provider.
- Separates sensitive assets into a private-data database and a permission database that only the user can directly access or modify, preventing the agent from freely reading secrets or policy state.
- Extends information-flow control with persistent tracking across both multi-step executions and temporally separated tasks via a disclosure log.
- Proposes a tool/service annotation framework to describe external parties, method-level data flows, and returned information for realistic MCP-style tools.
- Demonstrates that GAAP blocks all evaluated data-disclosure attacks, including attacks that defeat prior systems, with little loss in agent utility.

### ⚙️ Methodology
GAAP is designed around a strict confidentiality threat model: the user prompt, model inputs, model provider, and agent logic are all treated as untrusted, so the system must enforce privacy without relying on a trusted LLM or attack-free prompting. Instead of allowing the agent direct access to secrets, GAAP places private user data in a private data DB and stores disclosure permissions in a separate permission DB. Only the user can directly view or edit these stores through explicit interactions; the agent cannot arbitrarily inspect them.

To execute tasks, the agent generates a code artifact that orchestrates tool calls. GAAP then runs this artifact inside an execution environment with information-flow control (IFC). Sensitive values are tainted according to the private data they depend on, and GAAP tracks how these taints propagate through variables, intermediate computations, LLM interactions, and tool invocations. Before any disclosure—especially a call to an external tool or to the model provider—GAAP checks whether the disclosure is permitted by the user’s specification.

A key novelty is that permissions are not assumed to be fully predefined. GAAP incrementally builds the policy through dynamic, directed prompts to the user, asking for consent and constraints on how particular private data may be shared with particular parties. These permissions persist across tasks rather than acting as one-off approvals.

Another central mechanism is the disclosure log, a persistent record of prior releases of sensitive data to external parties. This lets GAAP reason about indirect and transitive flows over time, including cases where data disclosed in one task might later reappear through another tool call in a different task. This persistent accounting extends IFC beyond single execution traces to cross-task privacy enforcement and auditability.

GAAP also introduces a tool annotation framework. Each method of each external service is annotated with the associated external party, what data may flow into the service, and what data may be returned. These annotations enable precise IFC checks for realistic tool ecosystems such as MCP servers. The paper reports written annotations for 10 diverse MCP servers as a starting point for a broader reusable annotation library.

Overall, GAAP enforces confidentiality deterministically by constraining execution around explicit data-flow accounting rather than trying to infer or trust benign model behavior. It targets confidentiality only, not integrity: the system does not guarantee that the agent will behave correctly, only that private data will not be disclosed outside the user-authorized policy.

### 🧪 Experiments
The evaluation uses an implemented GAAP prototype and a set of tasks and attacks collected from prior literature and real agent practice. The experiments compare GAAP against existing privacy/safety approaches in settings where agents perform realistic multi-step tool-using tasks over private data. The study considers attacks such as prompt-injection-style exfiltration and other disclosure attempts that can cause state-of-the-art systems to leak secrets to untrusted parties, including the model/provider itself.

The evaluation setup includes annotated external tools/MCP services, persistent task execution across multiple steps and tasks, and user permission collection for disclosures. The paper emphasizes two outcome dimensions: security/privacy effectiveness and utility. Privacy effectiveness is measured by whether unauthorized disclosures occur under attack, including cross-step and cross-task leakage scenarios. Utility is measured by task completion/success relative to less private baselines.

The main reported results are that GAAP blocks all evaluated data disclosure attacks, including attacks that successfully induce other state-of-the-art systems to disclose private user data. At the same time, GAAP does not significantly reduce agent utility: task success remains comparable to other systems without deterministic confidentiality guarantees. The paper also notes that GAAP’s enforcement applies to disclosures to external tools and to the AI model/provider, which many prior defenses do not fully protect against under the same threat assumptions.

### 🌱 Insights & Implications
The paper’s main insight is that strong privacy for agents is achievable only if sensitive data and privacy policy state are removed from the model’s trust boundary. By treating the LLM and prompt as fully untrusted, GAAP avoids common assumptions that break under prompt injection or provider compromise. This is especially relevant for software-security research because it reframes agent safety as an execution-environment and data-flow control problem rather than a prompt-engineering problem.

A second important insight is that privacy for real agents is inherently longitudinal. Single-run IFC is insufficient because tools and services can create indirect flows across time; the disclosure log addresses this by turning prior releases into persistent state that future executions must respect. This has implications for long-lived assistants, enterprise agents, and multi-user systems.

The annotation framework is a practical strength but also a likely bottleneck: precision and scalability depend on high-quality tool annotations, and incomplete or incorrect annotations could weaken enforcement or increase overtainting. Similarly, the system focuses on confidentiality rather than integrity, so it may still allow wrong or harmful actions as long as they do not leak private data. Another open issue is balancing precise taint propagation with utility in multi-shot agent workflows, especially when LLM calls cause overtainting. The paper suggests compatibility with prior work on reducing overtainting and with complementary defenses that enforce integrity or control-flow safety.

For related research, GAAP suggests a fruitful direction: combining deterministic IFC-based confidentiality with separate mechanisms for action integrity, policy synthesis, or automated permission repair. It also opens questions about how to formalize tool semantics, automatically infer annotations, and support privacy-preserving autonomous repair/action systems in software engineering workflows.

### 🎯 Why This Paper
This paper is highly relevant to software security, secure agent engineering, and trustworthy automation. Its core contribution—deterministic enforcement of confidentiality through execution control, persistent information-flow tracking, and policy-aware tool mediation—connects directly to vulnerability prevention and secure system design. For a researcher interested in automated program repair, vulnerability detection, and repair, GAAP is valuable because it treats LLM-generated agent behavior as untrusted code and secures it using principled runtime enforcement, offering ideas applicable to secure code-generation pipelines, patch validation environments, and privacy-safe software maintenance assistants.

## 📄 [On Reasoning-Centric LLM-based Automated Theorem Proving](https://arxiv.org/abs/2604.19558v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Yican Sun, Chengwei Shi, Hangzhou Lyu, Yingfei Xiong
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper proposes ReCent-Prover, a Rocq/Coq proof agent that shifts LLM-based theorem proving from pure tactic generation toward explicit reasoning via self-critique and strategy-aware retrieval. By combining tactic validation with selective reflection and proof-plan-conditioned retrieval, it proves substantially more theorems on CoqStoq than prior state of the art, including a reported 22.58% relative gain under the same LLM-invocation budget.

### 🏆 Key Contributions
- Introduces validation with reflection, a selective self-critique mechanism where the LLM analyzes potentially dangerous yet syntactically valid tactics, detects likely misapplications, and produces failure summaries to guide subsequent search.
- Introduces retrieval with planning, which retrieves lemmas and proof examples using LLM-generated natural-language proof plans rather than surface similarity between current subgoals and formal statements.
- Implements ReCent-Prover as a reasoning-centric proof agent for Rocq that integrates CoqHammer, plan-based retrieval, tactic generation, and reflective validation in an iterative proving workflow.
- Demonstrates on the CoqStoq benchmark that reasoning-centric design improves proof success significantly, including a 22.58% relative improvement in proved theorems over the previous state-of-the-art system under equal LLM-call budget.
- Provides ablation-style evidence across configurations showing that both reflection and planning-based retrieval contribute to the overall gains.

### ⚙️ Methodology
ReCent-Prover is an iterative proof agent for Rocq. At each iteration, it targets the first unresolved subgoal and first attempts to solve it with CoqHammer, leveraging symbolic search as a fast path. If that fails, it invokes an LLM-driven pipeline. The key design principle is that the LLM should reason not only at the tactic level but also about proof strategy and about the validity of its own actions.

The first core module is retrieval with planning. Instead of retrieving support material by matching the current subgoal directly against formal lemma statements or prior proof states, the system first asks the LLM to generate a high-level natural-language proof plan for the current goal. The proof library is preprocessed similarly: each lemma is assigned an LLM-generated natural-language description, and each proof example is assigned an LLM-generated proof-plan summary. These descriptions/plans are embedded and indexed. During proving, the goal’s generated proof plan is embedded and used to retrieve semantically aligned lemmas and prior proofs that correspond to the anticipated strategy, even when formal syntax differs. This is intended to overcome the weak assumption that similar subgoals necessarily imply similar proofs.

The second core module is validation with reflection. Existing agents often accept any tactic that passes proof-assistant checking, even when it may be strategically harmful, such as applying an unsuitable lemma, splitting into unprovable subgoals, or inducting on the wrong variable. ReCent-Prover adds a selective reflective stage after tactic generation. For tactics deemed risky—especially those that can branch into multiple subgoals or push the proof into potentially unprovable states—the system asks the LLM to scrutinize the proposed tactic in context. If reflection suggests the tactic is likely misapplied, the agent rejects it early and synthesizes a failure summary describing the issue; this summary is fed into later iterations to avoid repeating the same mistake. If reflection finds no issue, the tactic is retained and executed. This selective design controls cost by avoiding reflection on every action.

Overall workflow: unresolved subgoal -> CoqHammer attempt -> if unresolved, generate proof plan -> retrieve lemmas and proof examples using plan embeddings -> construct prompt with subgoal, retrieved support, and prior failure summaries -> generate candidate tactic(s) with the LLM -> run proof-assistant validation -> selectively reflect on risky valid tactics -> either reject with failure summary or accept and continue. The loop stops on complete proof success or an iteration limit. The system thus tightly integrates symbolic proving, strategic planning, retrieval, generation, proof-assistant feedback, and self-critique.

### 🧪 Experiments
The evaluation is conducted on CoqStoq, a benchmark of real-world Rocq/Coq projects previously used to assess LLM-based theorem provers. The main comparison is against the prior state-of-the-art LLM-based proof agent cited in the paper, with evaluation normalized by the same budget of LLM invocations to account for the fact that ReCent-Prover deliberately spends more calls on planning and reflection. The workflow also includes CoqHammer as a symbolic baseline component within the prover.

The primary metric is the number of theorems successfully proved. The headline result is that ReCent-Prover achieves a 22.58% relative improvement in the number of proved theorems over the previous state of the art under equal LLM-invocation budget. The paper also reports additional experiments across different configurations of ReCent-Prover to assess the effect of the proposed modules, showing that both validation with reflection and retrieval with planning improve performance. The abstract and visible text emphasize that the gains persist despite the extra inference cost introduced by the added reasoning steps, indicating better utility per LLM budget rather than gains from brute-force scaling alone.

### 🌱 Insights & Implications
A central insight is that in interactive theorem proving, proof-assistant validity is not the same as strategic usefulness: many tactics are locally valid yet globally harmful. This observation parallels issues in program repair and code generation, where compilable outputs can still be semantically wrong. The paper shows that adding selective self-critique can improve search quality by filtering such locally valid but strategically bad actions.

A second insight is that retrieval quality depends on proof intent, not just current proof state similarity. Plan-conditioned retrieval is a strong idea for domains where syntax can differ while underlying strategy remains the same. This may generalize beyond theorem proving to tasks like automated debugging, repair, or vulnerability fixing, where high-level remediation plans may be more informative than local token similarity.

Likely limitations include increased inference cost, dependence on the quality and stability of LLM-generated plans and reflections, and possible propagation of planning errors into retrieval. The selective reflection heuristic also relies on identifying which tactics are risky enough to inspect; poor heuristics could miss bad actions or waste budget. Another open question is robustness across proof domains, proof assistants, and model families, especially as theorem-proving styles and tactic ecosystems vary.

For related research, the work suggests a broader design pattern for agentic systems: use LLM reasoning to create and critique intermediate abstractions rather than only emitting final low-level actions. That pattern is relevant to formal methods, software maintenance agents, APR systems, and security-repair workflows.

### 🎯 Why This Paper
This paper is relevant because it studies an agent architecture that combines symbolic validation, retrieval, planning, and self-critique—precisely the kinds of mechanisms that matter in software engineering automation, automated program repair, and vulnerability remediation. Although the domain is theorem proving rather than code repair, its core contributions directly inform how to build safer and more effective LLM agents that avoid superficially valid but strategically wrong edits, use plan-aware retrieval instead of shallow similarity, and operate under realistic inference budgets.

## 📄 [Crash-free Deductive Verifiers](https://arxiv.org/abs/2604.19448v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Wander Nauta, Marcus Gerhold, Marieke Huisman
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.SE

### 💡 TL;DR
The paper argues that fuzzing is a practical way to improve the robustness of deductive verifiers and presents AValAnCHE, a prototype integrated with VerCors to automatically generate annotated programs and detect verifier crashes. In experiments, grammar-based and semantically aware generation were much more effective than plain coverage-guided fuzzing, and the approach uncovered multiple real bugs in VerCors and also exposed issues in other verifiers.

### 🏆 Key Contributions
- Introduces fuzzing as a practical quality-assurance technique for deductive verifiers, focusing on crash-freedom and robustness rather than soundness/completeness.
- Presents AValAnCHE, a fuzzing infrastructure for VerCors that supports multiple input-generation strategies including Jazzer, Grammarinator, Xsmith, and Radamsa.
- Implements crash detection, deduplication via stack-trace hashing, automatic crash reporting metadata collection, and a monitoring web interface to support verifier developers.
- Provides an empirical comparison of coverage-guided, grammar-based, grammar-plus-coverage, and semantically valid verifiable-subset generation for exercising verifier front-end code.
- Demonstrates practical impact by uncovering several concrete bugs in VerCors and showing that similar fuzzing ideas also transfer to other deductive verifiers such as Dafny, VeriFast, and Viper.

### ⚙️ Methodology
The proposed approach treats a deductive verifier as the program under test and automatically feeds it large numbers of generated annotated programs to uncover internal crashes. The prototype AValAnCHE is designed as a strategy-agnostic harness around VerCors. It repeatedly invokes VerCors on generated inputs, monitors executions for crashes, records stack traces and version information, and groups duplicate failures by hashing stack-frame information consisting of class name, method name, file name, and line number. The focus is explicitly on bugs in the verifier implementation after parsing and before backend verification, excluding soundness issues, incompleteness, and expected error messages for unsupported features.

AValAnCHE supports several fuzzing strategies. First, plain coverage-guided fuzzing uses Jazzer, which instruments Java classes on the fly using JaCoCo-style coverage and libFuzzer-like mutation. This strategy is language-agnostic and serves as a baseline. Second, grammar-based generation uses Grammarinator with ANTLR grammars ported from VerCors for PVL, C, C++, and Java, ensuring syntactically valid inputs that pass parsing. Third, grammar-based generation can be combined with coverage feedback by seeding Jazzer with Grammarinator-generated inputs. Fourth, semantically stronger generation uses Xsmith to produce inputs in a verifiable subset of PVL that are not only parseable but also type-correct and resolvable enough to reach deeper front-end phases. Radamsa is additionally integrated as a generic blind fuzzer mainly to validate the generator interface. The architecture is intentionally extensible so new generation techniques can be plugged in later.

The design rationale is to progressively move from shallow random inputs, which are often rejected by parsers, toward syntactically and semantically meaningful programs that exercise name resolution, rewriting, and other verifier front-end phases where robustness bugs are likely to occur.

### 🧪 Experiments
The evaluation compares supported fuzzing strategies on VerCors using execution coverage over time as the main metric. Coverage is measured as the number of instrumentation points or coverage counters inserted by Jazzer/JaCoCo and reached during fuzzing, over a fixed 5-minute campaign on the same hardware and software environment. Experiments were run on a Lenovo ThinkPad P16v Gen 2 with an Intel Core Ultra 7 155H running NixOS, using VerCors development version 2bd3bcaed derived from release 2.2.0. To isolate VerCors front-end behavior, the Viper backend was disabled with --skip-backend.

The compared settings were: coverage-guided fuzzing on PVL inputs, grammar-based fuzzing on PVL only, grammar-based fuzzing across all front-end languages (PVL/Java/C/C++), grammar-plus-coverage on PVL, grammar-plus-coverage across all languages, and Xsmith-based verifiable-subset generation for PVL. The qualitative results are clear. Plain coverage-guided fuzzing achieved very low and nearly flat coverage growth and found no bugs, because most generated inputs were rejected immediately by the parser. Grammar-based fuzzing substantially improved exercised code and found crashes in resolution and rewriting phases. Adding coverage guidance to grammar-based fuzzing did not improve bug types found and slightly reduced performance due to overhead; the pure grammar-based PVL setting reached higher coverage than grammar+coverage on PVL. Xsmith-based semantically valid generation achieved strong initial coverage because its first inputs already passed parsing and resolution and entered deeper rewrite stages, but its coverage growth plateaued afterward.

Beyond coverage, AValAnCHE found multiple concrete VerCors bugs. Reported examples include crashes on an empty enum block, annotations placed in incorrect locations that were accidentally accepted by the grammar, legal expressions not properly handled by VerCors, and a highly specific C++ front-end parse error triggered by identifiers consisting only of the letters 'u' and 'l'. The paper also reports that similar fuzzing approaches exposed issues in other verifiers, with at least three issues found in VeriFast, two of them previously unknown in the truncated text provided.

### 🌱 Insights & Implications
A key insight is that verifier fuzzing benefits heavily from input validity: random coverage-guided mutation alone is too weak for languages with rich syntax and annotations, because invalid inputs are rejected before reaching interesting logic. Grammar-based generation is therefore a strong baseline for deductive verifiers, and semantically valid generation can reach even deeper internal phases quickly. However, the experiments also show that stronger semantic generation may suffer from reduced diversity, causing coverage to plateau after an initially strong start.

Another important takeaway is that many impactful robustness bugs are relatively shallow implementation defects in parsing, resolution, and rewriting pipelines, not deep theorem-proving failures. These bugs can still seriously harm usability and trust in verification tools, especially for non-expert users, so crash-freedom testing is a valuable complement to formal reasoning about verifier correctness. The paper explicitly does not address unsoundness or incompleteness, leaving open how fuzzing could be extended toward differential testing, metamorphic testing, or oracle-based checking of incorrect verification outcomes.

Limitations include the focus on front-end crashes before backend invocation, the use of coverage counters as a proxy rather than direct bug-finding power, and the fact that Xsmith support is currently limited to PVL. The reported bugs are mostly shallow, so future work could explore techniques for driving execution deeper into symbolic encoding and proof-obligation generation. For related research, the paper suggests fruitful connections to compiler fuzzing, input minimization, semantic generators, and automated robustness testing infrastructure for formal tools.

### 🎯 Why This Paper
This paper is highly relevant because it applies automated testing and fuzzing to the engineering quality of program verification tools themselves, which directly intersects with software engineering, software maintenance, software security tooling, and reliability of formal-analysis infrastructure. For a researcher working on automated program repair or vulnerability detection/repair, it highlights a practical methodology for hardening complex analysis tools, improving crash resilience, and systematically finding implementation defects in verifiers that may otherwise undermine downstream security or repair pipelines.

## 📄 [Malicious ML Model Detection by Learning Dynamic Behaviors](https://arxiv.org/abs/2604.19438v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Sarang Nambiar, Dhruv Pradhan, Ezekiel Soremekun
📅 **Published:** 2026-04-21  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
DynaHug detects malicious pre-trained ML models by sandboxing model loading, extracting runtime system-call behavior, and learning task-specific benign behavior with a one-class SVM. Across more than 25,000 benign and malicious models, it substantially outperforms static, heuristic, dynamic-rule, and LLM-based baselines, achieving up to 0.9963 F1 and up to 44% higher F1 than prior detectors.

### 🏆 Key Contributions
- Introduces DynaHug, a malicious PTM detector that uses dynamic behavioral analysis rather than relying only on static rules, blacklists, or heuristics.
- Proposes task-specific clustering of models so that benign behavior is learned within semantically similar groups (e.g., models for the same ML task), improving anomaly detection precision.
- Uses sandboxed execution traces, specifically runtime system calls observed during model loading/deserialization, to characterize model behavior and identify outliers.
- Trains a one-class SVM on benign-only data, enabling detection of previously unseen malicious payloads as behavioral anomalies without requiring exhaustive malware signatures.
- Evaluates the approach on over 25,000 PTMs from Hugging Face, MalHug, and injected malicious samples, and shows up to 44% F1 improvement over state-of-the-art detectors.
- Provides ablation and sensitivity analyses showing that dynamic analysis, OCSVM, and clustering each contribute positively to effectiveness.

### ⚙️ Methodology
DynaHug is designed around the observation that benign PTMs for the same task tend to exhibit similar runtime behaviors during loading, while malicious PTMs often trigger unusual or security-sensitive behaviors. The pipeline first groups benign models by task, such as text classification, to reduce behavior heterogeneity. For each task cluster, models are executed in a sandboxed environment where the detector performs dynamic analysis during model loading and deserialization. The system records runtime traces at the system-call level, capturing low-level actions such as file access, process creation, network operations, and command execution. These traces are transformed into behavior representations based on the types and frequencies of observed calls. Instead of supervised binary classification, DynaHug uses one-class learning: a one-class SVM is trained only on benign traces from each task-specific cluster to learn the normal behavioral boundary. At inference time, a model under test is loaded in the same sandbox, its system-call trace is collected and featurized, and the corresponding task-specific OCSVM determines whether the observed behavior fits the learned benign distribution or should be flagged as anomalous/malicious. This design avoids dependence on incomplete blacklists or hand-written rules and aims to generalize to novel attacks, including malicious payloads using uncommon imports or PyPI packages, because those payloads still induce abnormal runtime behaviors such as execve, networking, or unusual process activity. The paper also studies alternative design choices and reports that dynamic analysis, task clustering, and OCSVM are key factors in the final detector’s performance.

### 🧪 Experiments
The evaluation uses more than 25,000 pre-trained models, including benign models from sources such as Hugging Face and malicious models from MalHug as well as additional injected malicious samples. The baselines include static and heuristic detectors such as PickleScan, ModelScan, and Fickling; a dynamic-analysis-based detector, ModelTracer; and LLM-based detectors using Llama-3.1-8B-Instruct-tuned and GPT-5.2. The main metric emphasized is F1-score, with the paper reporting that DynaHug achieves up to 0.9963 F1 across datasets and is up to 44% more effective than the strongest baselines in terms of F1. Qualitative examples show that blacklisting-based tools often miss malicious payloads that rely on rare or previously unseen PyPI modules capable of code execution, while whitelist/static methods can over-flag benign models that legitimately import nonstandard modules. In contrast, DynaHug correctly identifies these malicious samples because their runtime traces reveal unusual behavior. The paper also includes ablation studies comparing dynamic versus alternative analysis choices, OCSVM versus other options, and clustered versus non-clustered training, as well as probing/sensitivity studies on task clustering and training size; these analyses indicate that each of the main design decisions improves effectiveness.

### 🌱 Insights & Implications
The paper’s central insight is that ML-model malware detection benefits from behavior-based anomaly detection much like traditional malware analysis: payload implementation details can vary, but many attacks still manifest as suspicious runtime actions. This is particularly important for the ML supply chain because static import blacklists and opcode heuristics are inherently brittle against previously unseen libraries, obfuscation, or creative deserialization payloads. For software security and vulnerability research, the work suggests a promising direction of combining program analysis with learned behavioral baselines rather than relying only on signatures. A notable strength is the benign-only training setup, which reduces dependence on comprehensive labeled malware corpora. Likely limitations include the cost and operational complexity of sandboxed dynamic execution, possible evasion by malware that delays or conditions payload execution, and the possibility that some benign models with rare but legitimate behavior may still appear anomalous. The task-specific clustering helps mitigate false positives, but also implies that accurate task metadata or clustering quality matters. Open questions include how well the method generalizes to non-pickle formats, to attacks triggered only during inference rather than loading, and to adversaries explicitly optimizing traces to mimic benign behavior. The work has implications for model hubs, secure model ingestion pipelines, and future research on ML artifact triage, runtime policy enforcement, and behavior-guided vulnerability repair or containment.

### 🎯 Why This Paper
This paper is highly relevant to the researcher’s interests in software engineering, software maintenance, software security, vulnerability detection, and automated repair because it treats malicious ML models as a software supply-chain security problem and applies dynamic program analysis plus machine learning to detect harmful behavior. It advances related research by showing that runtime behavioral learning can overcome the under- and over-approximation problems of static scanners, offering ideas directly transferable to vulnerability detection pipelines, secure artifact vetting, sandbox-based analysis, and potentially automated containment or repair of unsafe model-loading code.

### 💡 TL;DR
CHRONOS is a secure aggregation framework for federated learning on IoT devices that moves expensive cryptographic setup out of the latency-critical training round and into idle periods, while protecting keys inside ARM TrustZone. It achieves one-round masked gradient submission in the common case, tolerates bounded client dropouts via Shamir-share-based recovery, and significantly lowers active-phase latency versus traditional secure aggregation while resisting OS-level compromise and gradient inversion.

### 🏆 Key Contributions
- Introduces a phase-decoupled secure federated learning design that performs server-relayed Diffie-Hellman key setup once per epoch during device idle windows instead of during active training rounds.
- Designs CHRONOS, a TrustZone-assisted secure aggregation framework where ephemeral private keys, pairwise secrets, and round freshness are managed entirely inside a TEE, preventing exposure to a compromised host OS.
- Enables dropout recovery by distributing Shamir secret shares of each client’s ephemeral private key to peers, allowing the server to reconstruct masks for dropped clients when at least the threshold number of peers remain.
- Implements a TEE-minimal architecture that protects only seed management and freshness enforcement rather than full model training, yielding a very small persistent secure storage footprint independent of model dimension.
- Empirically demonstrates reduced active-phase latency, modest overhead over plaintext, robustness against gradient inversion attacks, and stronger compromise resistance than software-only precomputation alternatives.

### ⚙️ Methodology
CHRONOS builds on additive-mask secure aggregation but separates cryptographic setup from gradient transmission. In the idle phase of each epoch, every client generates an ephemeral elliptic-curve keypair inside an ARM TrustZone Trusted Application using OP-TEE, ensuring the private key never appears in Normal World memory. Public keys are exchanged through the server, and pairwise Diffie-Hellman shared secrets are derived inside the enclave. These shared secrets are transformed into encryption/PRG seeds and sealed in secure storage. To support dropout recovery, each client applies Shamir secret sharing to its ephemeral private key and distributes shares to peers; any t surviving peers can later help reconstruct the key for a dropped client. In the active training phase, the client computes its local gradient in the normal environment, invokes the enclave to derive a round-specific mask from the stored pairwise seeds and a hardware-backed monotonic round counter, and applies a single stream-cipher/PRG evaluation to mask the gradient. Clients then send masked gradients to the server in one communication round under no-dropout conditions. Because pairwise masks cancel, the server can aggregate masked updates without learning individual gradients. If up to N-t clients drop out mid-round, the server requests enough Shamir shares from surviving peers, reconstructs the dropped client’s ephemeral private key, recomputes pairwise seeds, and subtracts the missing masks so the aggregate remains correct without restarting the round. The design intentionally omits secondary self-masks to preserve strict one-round active-phase performance, using epoch-level key rotation and TEE-enforced freshness to limit replay and reuse risks. The framework’s core systems idea is TEE-minimalism: only key generation, secret derivation, seed sealing, and round-counter enforcement are trusted, avoiding the heavy memory and paging costs of executing full FL inside the enclave.

### 🧪 Experiments
The paper evaluates CHRONOS on Rock Pi 4 IoT devices using OP-TEE/ARM TrustZone. Comparisons include plaintext federated learning, conventional synchronous secure aggregation (SecAgg), SMPC-based approaches, and a software-only precomputation ablation referred to as CHRONOS-SW. The evaluation examines active-phase aggregation latency, recovery overhead under dropout, secure-world memory footprint, and privacy under gradient inversion attacks. The main quantitative findings reported are: up to 74% reduction in active-phase aggregation latency relative to synchronous secure aggregation for 20 clients; only modest overhead compared with plaintext execution; recovery payload of k x 32 bytes per surviving client when k clients drop out and recovery is needed; persistent TrustZone storage below 700 bytes per device for the 20-client configuration, independent of model dimension and training horizon; and a measured security-performance gap versus CHRONOS-SW showing an additional roughly 41-99 ms cost to gain hardware-isolated secrecy and rollback resistance. For privacy, the authors run state-of-the-art gradient inversion attacks and report that CHRONOS reduces reconstructions to near-random noise with PSNR around 8 dB, while preserving correct aggregate training behavior. The system tolerates up to N-t mid-round dropouts, with one communication round in the common case and one extra round only during recovery.

### 🌱 Insights & Implications
The paper’s main insight is that secure aggregation’s pairwise key establishment is data-independent and therefore does not need to occur during the training round; exploiting idle windows in IoT duty cycles is an effective systems optimization for privacy-preserving FL. The use of TrustZone materially changes the threat model compared with software-only precomputation by protecting seeds against host-OS compromise and preventing rollback through a hardware-backed round counter. This is especially relevant for edge/IoT deployments where the OS may be less trustworthy than the hardware root of trust. A notable design tradeoff is the omission of self-masks to preserve one-round active performance, which weakens resilience to some false-dropout or stronger active-adversary scenarios and shifts protection to periodic key rotation and threshold recovery assumptions. The framework also assumes availability and correctness of the TEE, secure share distribution, and an honest-but-curious or boundedly malicious server model for aggregation/recovery. Open questions include scalability beyond small-to-moderate client cohorts, robustness under stronger network churn or asynchronous FL settings, side-channel exposure of the TEE, and how CHRONOS interacts with complementary privacy mechanisms such as differential privacy or robust aggregation under poisoning. For related research, the paper suggests a promising direction: combining lightweight hardware roots of trust with protocol restructuring to reduce secure FL overhead without pushing full training into enclaves.

### 🎯 Why This Paper
This paper is highly relevant to software security, secure distributed systems, and privacy-preserving ML for edge environments. For a researcher interested in software maintenance, security, vulnerability detection/repair, and automated repair, it offers a concrete example of how hardware-assisted isolation, protocol refactoring, and minimal trusted computing bases can harden real systems against compromise while preserving performance—ideas that transfer to secure software update pipelines, resilient distributed protocols, and repair tools for resource-constrained systems.

## 📄 [Human-Machine Co-Boosted Bug Report Identification with Mutualistic Neural Active Learning](https://arxiv.org/abs/2604.18862v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Guoming Long, Shihai Wang, Hui Fang, Tao Chen
📅 **Published:** 2026-04-20  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper proposes MNAL, a cross-project bug-report identification framework that combines neural language models, active learning, effort-aware query selection, and pseudo-labeling to improve classification while reducing human labeling burden. On 1.28M GitHub issue reports from 127k+ projects, MNAL outperforms prior methods and substantially lowers annotator effort by selecting reports that are not only informative to the model but also easier for developers to read and identify.

### 🏆 Key Contributions
- Introduces Mutualistic Neural Active Learning (MNAL), a cross-project framework for binary bug report identification that explicitly couples human labeling and neural model improvement.
- Designs a mutualistic active-learning mechanism where queried reports are chosen to be both informative for model training and easier for humans to annotate in terms of readability and identifiability.
- Augments human-labeled queried samples with corresponding pseudo-labeled reports to better exploit the large unlabeled pool and strengthen model updates.
- Shows that the framework is model-agnostic and can improve multiple underlying neural language models rather than depending on one specific architecture.
- Provides large-scale empirical evaluation on 1,275,881 reports from more than 127,000 GitHub projects, plus a human-subject case study with 10 participants.

### ⚙️ Methodology
MNAL tackles bug report identification as a binary classification problem: determining whether a GitHub issue report is bug-related or not. The framework is cross-project rather than project-specific, so the underlying neural language model is trained to learn transferable textual patterns from issue titles and descriptions across many repositories. MNAL integrates this neural classifier with active learning, but differs from standard uncertainty sampling by explicitly modeling a mutualistic relation between machine and human annotators.

The workflow is iterative. First, an initial neural language model is trained on available labeled data. Then, from an unlabeled pool, MNAL ranks candidate reports for annotation. Its query strategy does not only consider informativeness/uncertainty for the classifier, but also the expected human annotation effort. The paper frames this effort in two dimensions: readability and identifiability. Readability reflects whether a report is easy for a developer to read and comprehend; identifiability reflects whether its content makes it easy to decide if the issue is truly bug-related. The system therefore aims to avoid wasting developer effort on highly ambiguous or cognitively taxing reports while still selecting samples that improve the model.

Once developers label the selected informative-and-easier reports, MNAL further expands their impact through pseudo-labeling. Specifically, the newly human-labeled reports are used to derive corresponding pseudo-labeled unlabeled reports, which are then incorporated into model updating. This allows the framework to exploit the much larger unlabeled pool that is common in software repositories. The intended symbiosis is that humans provide a small number of high-value labels, while the model propagates that knowledge to additional reports; in turn, the model chooses annotation targets that are easier and faster for humans to handle.

Because the method is described as model-agnostic, the active-learning and pseudo-labeling components are separable from the backbone text model. The backbone can be different neural language models, and MNAL acts as a training/querying scheme around them. Overall, the method combines cross-project representation learning, effort-aware active sampling, and semi-supervised pseudo-label augmentation to improve both effectiveness and annotation efficiency.

### 🧪 Experiments
The evaluation uses a large-scale dataset of 1,275,881 issue reports collected from over 127,000 GitHub projects. The task is cross-project bug report identification. MNAL is compared against state-of-the-art approaches, five baselines, and several ablation/variant configurations. The paper reports six metrics covering both classification quality and human labeling effort. Performance metrics include standard classification measures such as F1-score; effort metrics center on readability and identifiability of the selected samples for human annotation.

The main findings are that MNAL achieves better bug report identification performance than the compared methods while also substantially reducing annotation effort. Relative to alternative approaches, the paper reports up to 95.8% effort reduction in readability and 196.0% in identifiability during human labeling. In the introduction and summary of results, the authors also report gains such as 98.1% readability and 194.7% identifiability improvement over random sampling, and 97.2% and 283.6% over uncertainty sampling, all stated as statistically significant. The pseudo-labeling mechanism significantly improves model performance compared with variants that do not exploit pseudo-labeled reports. The model-agnostic claim is validated by plugging MNAL into different neural language models and observing improvements across backbones, including reported gains of 78.6% in readability and 171.5% in identifiability.

Beyond offline experiments, the paper includes a qualitative human study with 10 participants. Participants judged MNAL to be more effective and less resource-intensive, with reported qualitative improvements of 74.7% in readability and 64.8% in identifiability, and roughly threefold savings in time and monetary cost for labeling.

### 🌱 Insights & Implications
A key insight is that active learning for software engineering tasks should optimize not only model uncertainty but also annotator effort. In bug report identification, highly uncertain instances may be too ambiguous or verbose for humans, so selecting them can undermine the practical benefits of human-in-the-loop learning. MNAL suggests that effort-aware querying can be a better operational objective for developer-facing labeling workflows.

Another important implication is that cross-project generalization is feasible and beneficial for issue-report classification. This matters because many projects lack enough labeled issue data to train project-specific models from scratch. The pseudo-labeling component further highlights the value of unlabeled repository data, which is abundant in real-world issue trackers.

For related research, the paper is relevant to broader human-in-the-loop SE workflows such as bug triage, issue routing, test-failure classification, and vulnerability report screening, where annotation cost is a bottleneck. The method's model-agnostic nature also suggests it can be paired with stronger pretrained language models in future work.

Likely limitations and open questions include the dependence of readability/identifiability estimation on heuristics or model assumptions, the risk of error amplification from pseudo-labels, and possible domain shift across repositories, ecosystems, and issue-writing cultures. Since the study focuses on binary bug-vs-non-bug classification, an open extension is whether the same mutualistic active learning idea can scale to finer-grained issue categorization or security-specific issue types. Another question for future work is whether effort-aware sampling remains effective when developer expertise varies substantially or when labels are noisy and inconsistent.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance because bug report identification is an upstream task that directly affects issue triage, debugging, and repair workflows. For a researcher interested in software engineering automation, it offers a scalable human-in-the-loop learning design that reduces labeling cost while improving classification quality on massive cross-project data.

It is also relevant to automated program repair and vulnerability-related research because the same annotation bottlenecks appear when identifying security bugs, prioritizing repair candidates, and curating datasets for vulnerability detection/repair models. MNAL's combination of active learning, pseudo-labeling, and cross-project neural text modeling provides ideas that can transfer to security issue classification, vulnerability report mining, and cost-aware dataset construction for repair systems.

## 📄 [TrEEStealer: Stealing Decision Trees via Enclave Side Channels](https://arxiv.org/abs/2604.18716v1)  `60% relevant`

👤 **Authors:** Jonas Sander, Anja Rabich, Nick Mahling, Felix Maurer, Jonah Heller, Qifan Wang, Thomas Eisenbarth, David Oswald
📅 **Published:** 2026-04-20  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
This paper presents TrEEStealer, a side-channel model extraction attack that can reconstruct decision trees running inside TEEs such as AMD SEV and Intel SGX. By combining control-flow leakage with passive information tracking, it extracts high-fidelity trees more efficiently and with fewer assumptions than prior black-box attacks, showing that TEEs do not reliably protect ML models from theft.

### 🏆 Key Contributions
- Introduces TrEEStealer, a high-fidelity extraction attack targeting TEE-protected decision tree models.
- Proposes a novel extraction algorithm that combines control-flow information with passive information tracking to maximize information gained per query.
- Demonstrates two practical primitives for recovering control-flow information: SEV-Step plus performance counters for AMD SEV, and a new Branch-History-Register-based primitive for Intel SGX on current Xeon 6 CPUs.
- Reproduces and extends prior side-channel findings for SGX, showing continued practical control-flow leakage on modern hardware.
- Identifies exploitable vulnerabilities in three widely used decision tree inference libraries: OpenCV, mlpack, and emlearn.
- Empirically shows better extraction efficiency and fidelity than prior decision-tree stealing approaches.
- Provides strong evidence that TEEs fail to prevent model extraction when inference code leaks branch behavior.

### ⚙️ Methodology
The paper studies model extraction for decision trees executed inside hardware Trusted Execution Environments, where the attacker is assumed to observe side-channel leakage rather than rely on rich API outputs. The core idea is that decision tree inference naturally induces branch-dependent control flow: each queried input traverses a root-to-leaf path, and this path leaks structural information about the underlying tree. TrEEStealer exploits this by obtaining Control-Flow Information (CFI) from TEE-specific side channels and coupling it with passive information tracking to infer node predicates, subtree structure, and decision outcomes. The extraction algorithm appears designed to choose or process queries so that each query reveals as much incremental structure as possible, avoiding the high query complexity of classical black-box tree extraction. For AMD SEV, the attack leverages existing stepping and performance-counter mechanisms, specifically SEV-Step, to observe execution progress and branch behavior during inference. For Intel SGX, the authors both reproduce prior control-flow leakage on recent Xeon 6 CPUs and introduce a new primitive using the Branch-History Register to recover branch histories efficiently. Because decision tree implementations in common libraries branch explicitly on feature thresholds, these leaked traces can be mapped back to traversal paths through the tree. Passive information tracking likely refers to maintaining constraints induced by observed traversals, such as which feature-threshold comparisons must have evaluated true or false for particular inputs, allowing the algorithm to iteratively reconstruct internal nodes and leaves even when direct outputs are limited. The method explicitly avoids strong assumptions about tree depth, balance, or API richness, making it closer to a realistic adversarial setting for ML-as-a-service deployments protected by TEEs.

### 🧪 Experiments
From the abstract, the evaluation is centered on practical extraction attacks against decision tree implementations in three popular libraries: OpenCV, mlpack, and emlearn. The attack is tested in TEE environments for both AMD SEV and Intel SGX, using different side-channel primitives to recover control-flow information. The comparison is against prior decision-tree extraction attacks, especially black-box methods that are query-intensive, require structural assumptions, or depend on richer API outputs. The main evaluation dimensions are extraction efficiency and extraction fidelity, i.e., how many queries or observations are needed and how accurately the recovered tree matches the target model. Although the abstract does not report specific datasets, benchmark suites, or exact numerical metrics, it clearly claims that TrEEStealer outperforms prior attacks on both efficiency and fidelity. The experiments also include a platform study aspect: reproducing SGX leakage on current Xeon 6 CPUs and validating a new Branch-History-Register-based extraction primitive. Overall, the empirical results support a state-of-the-art claim for decision tree extraction under TEE protection and demonstrate that real-world library implementations are vulnerable.

### 🌱 Insights & Implications
A key insight is that decision trees are especially susceptible to control-flow side channels because their prediction logic is structurally aligned with branch behavior; every comparison during inference leaks semantic information about the model. TEEs may hide memory and isolate execution from software adversaries, but they do not eliminate all microarchitectural or control-flow leakage, making them insufficient as a standalone defense for model confidentiality. For security research, this work is important because a stolen model can become an enabler for downstream attacks such as white-box evasion, membership inference, and training-data privacy attacks. For software engineering and secure deployment, the paper suggests that common inference libraries are unsafe by default in enclave settings if they use branchy implementations of tree traversal. Likely implications include the need for constant-control-flow or oblivious inference implementations, side-channel-aware compilation, or algorithmic redesign when deploying ML models in TEEs. Open questions include how well the approach generalizes to ensembles such as random forests or gradient-boosted trees, what the exact limits are under noisier side-channel conditions, and what overhead practical defenses would impose. A limitation of the abstract-only view is that precise assumptions on attacker capabilities, query adaptivity, and model classes are not fully specified, but the attack appears realistic enough to challenge current enclave-based MLaaS protection strategies.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it sits at the intersection of software security, vulnerability analysis, and ML system hardening. It exposes concrete side-channel vulnerabilities in widely used software libraries and TEEs, which is directly useful for vulnerability detection and repair research, and it highlights how implementation details in deployed software can undermine security guarantees. For work on software maintenance and automated repair, the paper motivates developing mitigations such as constant-time/oblivious tree inference transformations, secure library patches, and automated detection of branch-leaking code patterns in enclave applications. It also advances understanding of attack surfaces in ML deployment pipelines, which is valuable for secure software engineering of model-serving systems.
