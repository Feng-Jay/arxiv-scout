## 📄 [FusionVul: A Multimodal Feature Fusion Framework for Source Code Vulnerability Detection](https://arxiv.org/abs/2606.08553v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Hongyu Yang, Yaping Zhu, Jingchuan Luo, Hiroshi Nomaguchi, Chunhua Su, Willy Susilo
📅 **Published:** 2026-06-07  🏷️ **Categories:** cs.SE

### 💡 TL;DR
FusionVul is a multimodal vulnerability detection framework that combines pretrained sequential code representations, graph-based structural semantics, explicit cross-attention fusion, and sample-aware prediction weighting. Across four datasets, it delivers its strongest gains on structurally diverse and vulnerability-rich benchmarks such as SVulD and DiverseVul, showing that explicit fusion plus adaptive branch aggregation improves robustness over single-modality and simpler multimodal baselines.

### 🏆 Key Contributions
- Proposes FusionVul, a hybrid vulnerability detection framework that jointly models source code as a linear code sequence (LCS) with UniXcoder and as a code property graph (CPG) with a GGNN.
- Introduces CAFFNet, a cross-attention feature fusion network that explicitly aligns sequential token representations with structural graph semantics to produce richer fused features.
- Introduces a Sample-Aware Weighting (SaW) strategy that adaptively combines predictions from three branches—sequential, structural, and fused—rather than relying on fixed or dual-branch aggregation.
- Demonstrates that explicit multimodal fusion and adaptive multi-branch utilization are especially beneficial on datasets with broad vulnerability coverage and high function-size dispersion, such as SVulD and DiverseVul.

### ⚙️ Methodology
FusionVul is organized into three stages. First, it performs modality-specific feature extraction from two complementary views of code. The linear code sequence is encoded by a pretrained UniXcoder Transformer to capture token-level syntactic and semantic information. In parallel, the source code is transformed into a Code Property Graph that exposes structural relationships such as control-flow and data-flow dependencies, and a Gated Graph Neural Network propagates messages over the graph to learn structural-semantic node representations. Second, FusionVul applies a Cross-Attention Feature Fusion Network (CAFFNet) to explicitly model interactions between these modalities. Instead of simple concatenation or implicit mixing, CAFFNet allows sequence representations to attend to relevant graph-derived structural features, enabling fine-grained alignment between lexical patterns and execution-related semantics and producing a fused multimodal representation. Third, the model maintains three predictive branches: a sequential branch from UniXcoder features, a structural branch from GGNN/CPG features, and a fusion branch from CAFFNet features. A Sample-Aware Weighting mechanism then dynamically aggregates branch outputs for each sample, assigning different importance to the three branches depending on which semantic view appears most informative for that code instance. This design is motivated by the observation that some vulnerabilities are more detectable from token-level patterns, others from structural dependencies, and others from their interaction.

### 🧪 Experiments
The paper evaluates FusionVul on four vulnerability detection datasets and highlights results on SVulD and DiverseVul, which have broader vulnerability-type coverage and more dispersed function-size distributions. The evaluation compares FusionVul against traditional single-modality approaches and prior hybrid systems that combine Transformer-based code encoders and graph models but use weaker fusion or simpler aggregation strategies. The main reported metric is F1 score, with the paper emphasizing superiority on the more challenging and diverse datasets. The authors report that FusionVul consistently outperforms baselines and achieves especially strong improvements on SVulD and DiverseVul, indicating better robustness to heterogeneous code patterns and complex vulnerability semantics. Although the provided text does not include the detailed numerical tables, it explicitly states that the gains are strongest where simple sequential or structural signals alone are insufficient and where adaptive multimodal fusion matters most.

### 🌱 Insights & Implications
The paper’s main insight is that multimodal vulnerability detection benefits not just from having both sequence and graph encoders, but from explicitly aligning them and then adaptively deciding how much each branch should influence the final prediction. This is important because different vulnerability types manifest differently: unsafe API misuse may be visible in token sequences, while data-flow or control-dependence issues may depend more on graph structure. The work also suggests that fused representations should not replace unimodal branches entirely; preserving separate decision paths and aggregating them adaptively can improve generalization. A limitation visible from the text is that the approach depends on constructing CPGs and running a GGNN, which may increase preprocessing and training complexity compared with sequence-only detectors. Another open question is how well the method transfers across languages, projects, and real-world class imbalance settings beyond the four datasets. It also invites future work on deeper interpretability of the cross-attention alignments and on integrating stronger pretrained code models or richer program analysis graphs.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, and vulnerability detection because it directly targets automated source code vulnerability identification using a principled combination of pretrained code models and program-structure analysis. It advances work in vulnerability detection by moving beyond single-modality learning and simplistic multimodal fusion, offering a concrete architecture that could inspire more robust detectors, downstream vulnerability localization, and even automated vulnerability repair pipelines that require accurate semantic understanding of faulty code.

## 📄 [SecureClaw: Clawing Back Control of LLM Agents](https://arxiv.org/abs/2606.09549v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yuhan Ma, Stefan Schmid
📅 **Published:** 2026-06-08  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
SecureClaw argues that securing LLM agents requires protecting two distinct boundaries at once: preventing unauthorized external actions at effect sinks and preventing sensitive plaintext from ever entering the untrusted runtime. It implements this with opaque-handle-based read confinement and a trusted PREVIEW→COMMIT executor for writes, achieving the best joint security/utility tradeoff across AgentDojo, AgentLeak, and ASB among evaluated defenses in a common harness.

### 🏆 Key Contributions
- Defines a two-surface security decomposition for tool-using LLM agents: request-bound authorization of external effects and confinement of sensitive plaintext away from the runtime
- Proposes SecureClaw, a dual-boundary architecture with trusted read mediation via opaque handles and bounded summaries, plus trusted write mediation via canonicalized PREVIEW→COMMIT authorization and execution
- Introduces deny-aware recovery so blocked actions can fall back to safe continuations instead of hard task failure
- Provides a common-harness empirical evaluation across AgentDojo, AgentLeak, and Agent Security Bench showing that single-boundary defenses are insufficient and that the two boundaries are complementary rather than interchangeable
- Demonstrates through ablations and bypass analysis that sink-side authorization stops unauthorized commits but does not stop internal secret relay, while read-side confinement reduces leakage but cannot authorize effects

### ⚙️ Methodology
SecureClaw separates the untrusted LLM runtime from two trusted control points. On the read path, a trusted gateway mediates access to sensitive data: instead of returning raw plaintext, it stores protected values in a trusted handle store and returns a high-entropy opaque handle bound to metadata such as caller, session, TTL, object version, and allowed dereference sinks. When the runtime needs information for planning, the system may additionally provide a bounded, schema-aware summary as an explicit declassification interface; the runtime can reason over summaries and symbolic handles but cannot dereference secrets directly. On the write path, any effectful tool invocation is treated only as a proposal. The gateway canonicalizes the sink-relevant request fields, and a policy engine authorizes or denies the exact canonical request, producing an authorization artifact bound to the request digest. Only a trusted executor can perform COMMIT: it recomputes the binding, verifies freshness, replay protection, optional confirmation tokens, and handle-resolution restrictions, then commits the exact authorized request to the external sink. This design enforces complete mediation of sensitive reads and effectful writes, supports capability-style authority via unforgeable references, and keeps plaintext outside the runtime except through declared bounded summaries. The architecture also includes deny-aware recovery, where denied actions trigger a fixed-template safe continuation such as drafting for user confirmation rather than executing the blocked request.

### 🧪 Experiments
The paper evaluates SecureClaw in a common harness on three major benchmarks targeting different attack surfaces: AgentDojo and Agent Security Bench (ASB) for prompt-injection-driven unauthorized actions, and AgentLeak for leakage through both final outputs and internal relay channels such as inter-agent messages and shared memory. Reported metrics include attack success rate (ASR) for action-security benchmarks and leak rate for AgentLeak’s attacked parity lane, which captures both output leakage and internal relay leakage. SecureClaw achieves 0% ASR on ASB, 0.64% ASR on AgentDojo, and 3.23% overall leak on AgentLeak’s attacked parity lane while preserving usable task utility. The evaluation compares SecureClaw against defenses that protect only one boundary, showing that sink-side defenses can block external commits but still allow internal plaintext relay, and read-side defenses can reduce leakage but do not prevent unauthorized effects. The ablation and bypass suite is specifically designed to validate this non-substitutability claim by isolating failures under prompt injection and compromised-runtime adversaries, including tampered requests, replay attempts, session mixing, and handle misuse.

### 🌱 Insights & Implications
The paper’s main insight is that LLM-agent security is fundamentally dual-surfaced: effect authorization and plaintext confinement fail at different points in the system and therefore must be enforced at different boundaries. This is especially important for multi-agent settings, where internal channels like inter-agent messages and memory create leakage paths that output-only auditing misses. The work also reframes bounded summaries as explicit declassification rather than an incidental convenience, making confidentiality exceptions auditable. A key limitation is that security still depends on correct classification of sensitive fields, complete mediation of all sensitive reads and effectful sinks, and trust in the gateway, policy engine, and executor. SecureClaw also does not guarantee semantic correctness of policy-allowed actions: coarse or misconfigured policies can still authorize the wrong but syntactically valid principal/object combination. Another open question is how much utility is lost as summaries become more restrictive, and how well the architecture scales to richer agent workflows, dynamic policies, and domains like code agents where data sensitivity and tool semantics are harder to classify precisely.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, automated repair, and secure agentic systems because it provides a concrete systems architecture for hardening tool-using LLM agents against both unauthorized actions and sensitive-data leakage. For work in vulnerability detection/repair and secure software maintenance, SecureClaw offers a practical reference design for isolating untrusted model runtimes, enforcing sink-side authorization, and constraining data exposure—ideas directly applicable to secure code assistants, autonomous patching agents, and CI/CD automation that interact with repositories, secrets, tickets, and deployment tools.

Building on this vulnerability analysis, the paper constructs an attack taxonomy organized by adversary goal, tactic, and technique. The three top-level goals are Hijack, Mislead, and Drain, instantiated via seven tactics and fourteen techniques. The evaluation targets six data-agent systems: four open-source systems (DeepAnalyze, DataInterpreter, LAMBDA, and DB-GPT) and two closed-source production cloud analytics services (Databricks Genie and BigQuery). The attack suite covers all fourteen techniques defined in the proposed taxonomy. Attacks are evaluated with four metrics chosen to capture different failure modes: Attack Success Rate for whether the intended exploit succeeds, Blast Radius for how broadly the compromise or leakage propagates, Relative Error for integrity degradation in analytical outputs, and Resource Amplification Ratio for measuring denial-of-service or excessive execution-cost effects. The experiments use schema-grounded, LLM-generated attack payloads designed to reflect realistic relational settings rather than toy prompts. Across systems, the results show substantial vulnerability exposure: attacks can hijack agent behavior, mislead outputs via poisoned or manipulated data/workflow context, and trigger large resource amplification through valid analytical operations. While the provided excerpt does not include the full numeric tables, the paper explicitly reports that all six systems exhibit meaningful weaknesses and that the empirical findings support four major design takeaways for defense.

A central insight is that data-agent security is not reducible to either database security or generic LLM-agent safety. The dangerous behaviors arise from composition: the agent interprets user intent, issues iterative SQL queries, inspects intermediate results, invokes external tools, and accumulates context over time. This creates attack channels such as cumulative privacy leakage, policy drift across turns, and execution-cost amplification via semantically valid queries that traditional SQL-injection or access-control models may miss. Another key implication is that relational data should be treated as active adversarial input, not passive evidence; malicious instructions or misleading signals can be embedded in structured content, metadata, or schema descriptions and then propagated through the agent’s reasoning loop. The paper also highlights that permission mismatches between users and agent service accounts are a major practical risk in enterprise deployments. Likely limitations are that the study is an offensive evaluation rather than a complete defense framework, and that attack success may depend on specific agent architectures, prompting strategies, toolchains, and database configurations. Open questions include how to enforce compositional disclosure control across multi-step analytics, how to verify provenance and trustworthiness of structured evidence, how to maintain persistent security policies over long agent trajectories, and how to integrate resource-aware planning into LLM-driven SQL generation and tool execution.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and vulnerability repair because it systematically exposes a new class of vulnerabilities in LLM-based analytical software systems that combine reasoning, code/tool execution, and database interaction. ## 📄 [FuseFSS: Efficient Secure LLM Inference with Function Secret Sharing](https://arxiv.org/abs/2606.09551v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Yuhan Ma, Yong Li, Stefan Schmid
📅 **Published:** 2026-06-08  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
FuseFSS is a compiler framework for two-server secure LLM inference that replaces hand-crafted fixed-point protocols for nonlinearities and helper operators with a uniform compilation strategy based on just two batched FSS calls plus standard share-based post-processing. On BERT and GPT-style models, it preserves inference accuracy while improving end-to-end latency by 1.24x-1.50x, reducing online communication by 9%-16%, and shrinking preprocessing cost relative to prior FSS-based GPU systems such as Sigma.

### 🏆 Key Contributions
- Introduces FuseFSS, a compiler that expresses a broad class of fixed-point scalar operators used in secure transformer inference through a compact operator specification rather than bespoke protocol design.
- Shows that each compiled operator can be executed with a uniform structure: one packed comparison FSS evaluation for all predicate bits, one vector interval-lookup FSS evaluation for active coefficients/constants, followed by fixed share-based post-processing.
- Provides mask-aware compilation rules that correctly rewrite predicates under masked-wire semantics, including wrap-around and signedness handling over Z_{2^n}.
- Enforces mask-independent public protocol shapes so that key sizes and visible instance dimensions do not leak mask-dependent information, enabling a unified semi-honest security proof with explicit leakage characterization.
- Demonstrates practical gains over the state-of-the-art FSS-based GPU secure inference baseline on transformer models, including faster online inference and lighter preprocessing/key generation.

### ⚙️ Methodology
FuseFSS targets scalar fixed-point nonlinearities and helper operations in two-server MPC over the ring Z_{2^n}, where inputs are secret shared and each wire is independently masked as x_hat = x + r_in mod 2^n. The key observation is that many secure fixed-point operators have the same high-level shape: determine which interval the input lies in using a small set of predicates, then apply a low-degree arithmetic expression parameterized by interval-specific coefficients and constants. FuseFSS formalizes this with an operator specification containing: (1) an interval partition over canonical representatives, (2) a low-degree polynomial or arithmetic piece for each interval, and (3) Boolean helper bits defined as predicate circuits. Given this specification and preprocessing masks, the compiler emits a protocol with two standard FSS evaluations on the public masked value x_hat. The first is a packed comparison that simultaneously returns XOR shares of all predicate bits required by the operator, including bits needed for wrap correction, sign handling, and piece selection. The second is a vector interval lookup that returns additive shares of the active interval’s coefficients and constants. After these two FSS calls, the rest of the computation is implemented as a fixed post-processing circuit using standard MPC primitives such as Beaver multiplications, Boolean ANDs, arithmetic/bit conversions (B2A and A2B), truncation, and arithmetic right shift. This design removes the need to engineer separate secure protocols for each operator. The compiler is mask-aware: predicates are rewritten to operate correctly on masked openings while preserving security, and all emitted public shapes—such as numbers/bit-widths of comparisons and lookup dimensions—depend only on the public operator specification and fixed-point metadata, not on sampled masks. The system operates under semi-honest security in the standard preprocessing model with two non-colluding servers and one-time preprocessing material per gate instance.

### 🧪 Experiments
FuseFSS is evaluated against Sigma, the paper’s stated state-of-the-art FSS-based GPU secure inference baseline, on BERT and GPT-style transformer models in a two-server preprocessing-based secure inference setting. The evaluation focuses on end-to-end online performance, communication, preprocessing overhead, and model accuracy. Metrics include latency/speedup, online communication volume, key-generation time, key size, and accuracy preservation relative to the baseline implementation. Results show that FuseFSS matches model accuracy while achieving 1.24x-1.50x end-to-end speedup and reducing online communication by 9%-16%. In the offline/preprocessing phase, it lowers key-generation time by 14%-23% and reduces key size by 20%-24%. The gains come from replacing multiple operator-specific comparison and lookup subprotocols with a fused, batched two-FSS-call structure that is more GPU-friendly and requires less bespoke preprocessing material.

### 🌱 Insights & Implications
The paper’s main systems insight is that the complexity bottleneck in secure transformer inference is not just nonlinear approximation itself, but the repeated protocol engineering needed to implement comparisons, wrap-around correction, signed fixed-point behavior, and interval selection for each operator. By treating these operators as instances of a shared interval-and-piece template, FuseFSS raises the abstraction level from handcrafted cryptographic kernels to compilable operator specifications. This is especially relevant for secure ML systems research because it improves extensibility: adding a new scalar operator no longer requires designing a fresh FSS/MPC protocol from scratch. Another important implication is software assurance: a single compilation pipeline and unified correctness/security argument may reduce subtle bugs related to masking, signedness, and modular arithmetic. Limitations and open questions include the current focus on scalar fixed-point operators rather than broader tensor-level transformations, the semi-honest two-server threat model, and dependence on fresh per-wire masks and preprocessing material. The paper also leaves open how well the abstraction scales to more complex functions, active security, larger operator libraries, or settings with secret model weights and more heterogeneous deployment constraints. Still, it suggests a promising compiler-centric direction for secure inference frameworks, where protocol synthesis and operator DSLs replace manual cryptographic engineering.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering for secure ML systems, software maintenance of cryptographic inference codebases, and security-preserving model serving. It directly addresses a maintainability and correctness problem common in secure inference implementations: bespoke operator protocols are hard to extend, validate, and optimize. FuseFSS contributes a compiler-based abstraction that can make vulnerability-prone low-level protocol logic more modular, reusable, and auditable, which is valuable for research on secure software construction and automated reasoning about protocol correctness. For work in vulnerability detection/repair and automated program repair, the paper is also interesting because it reframes error-prone hand-written secure arithmetic code as generated code from formalized operator specifications, potentially reducing implementation bugs and opening opportunities for specification-driven verification or repair.

## 📄 [Now You (Still) See Me: Detecting Evasive Steganographic Payloads in LLMs](https://arxiv.org/abs/2606.09411v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Charles Westphal, Timothy Douglas, Keivan Navaie, Tiago Pimentel, Fernando E. Rosas
📅 **Published:** 2026-06-08  🏷️ **Categories:** cs.CR, cs.IT, cs.LG

### 💡 TL;DR
This paper shows that activation-based steganography detectors for LLMs, including both linear ridge probes and stronger held-out MLP probes, can be adaptively evaded by adversarially fine-tuned trojan models that still leak prompt-borne secrets with high accuracy and minimal utility loss. It then provides an information-theoretic explanation for this evasion and demonstrates that a theory-guided recontextualization dataset can reliably restore detectability across five evasive trojans.

### 🏆 Key Contributions
- Extends prior mechanistic steganography detection from linear ridge probes to non-linear two-layer MLP probes that attempt to recover the secret from internal activations.
- Introduces an adversarial fine-tuning procedure that trains steganographic trojans to preserve covert secret transmission while suppressing probe-based detectability, even when only linear critics are used during training.
- Demonstrates cross-model adaptive evasion on five open-weight LLMs: Qwen3-8B, Llama-3.1-8B, Ministral-8B, Qwen3-14B, and Phi-4-14B.
- Shows that evasive trojans retain substantial exfiltration performance, achieving 58–79% exact-match secret recovery with only 1–8% average degradation on six capability benchmarks.
- Provides an information-theoretic characterization of successful evasion: the secret remains recoverable but is shifted away from low-order extractable structure into synergistic interactions with residual/slack degrees of freedom.
- Proposes and validates a recontextualization dataset that constrains residual degrees of freedom, restoring both ridge and MLP detectability for all five evasive trojans.

### ⚙️ Methodology
The paper studies a fine-tuning-based steganographic threat model in which an attacker implants a covert channel into an LLM so that prompt-borne secrets are encoded into otherwise benign completions. Secrets are 32-bit strings, and decoding is performed token-by-token using a binary partition of the vocabulary. The partition is defined by projecting a random hyperplane in token-embedding space, assigning each vocabulary item to bucket 0 or 1. During generation, the i-th output token encodes the i-th secret bit according to which bucket it falls into.

To create trojaned models, the authors use mixed-motive supervised fine-tuning on three data streams: steganographic examples, mathematical reasoning examples, and instruction-following examples. Stego examples are generated by the base model itself under bucket-constrained decoding, where logits outside the target bit bucket are masked so the model produces fluent task-relevant text that also carries the desired secret. The non-stego streams help preserve ordinary model capabilities and avoid obvious degradation.

For detection, the defender observes hidden activations and trains probes to recover the secret representation from internal states. The paper considers both ridge regression probes and a stronger held-out two-layer MLP probe, focusing on how linearly or nonlinearly extractable the secret is from the model’s content-aligned internal representation.

To evade detection, the authors adversarially fine-tune the trojan. Inspired by adversarial monitor-evasion work, they repeatedly refit ridge critics during training and optimize the model so that the secret remains decodable by the attacker from output tokens, while becoming difficult for detectors to recover from hidden activations. Importantly, the attacker does not need inference-time access to activations; instead, the model is trained offline to produce representations that look benign under a broad family of detectors. A notable result is transfer: training against linear critics is sufficient to reduce detectability even for unseen MLP probes.

The theoretical analysis frames evasion as a conditional indistinguishability problem under a recoverability constraint. The model must still convey the secret, but internal representations should appear explainable by benign fluency-related content alone. The authors argue and prove that, under these constraints, successful evasion reduces low-order extractability and instead stores the secret in synergistic interactions between content-related representations and residual/slack degrees of freedom such as meaning-preserving lexical variation. Based on this theory, they design a recontextualization intervention: an evaluation dataset with fixed sentence templates and restricted contextual variation, which reduces slack degrees of freedom and thereby exposes the hidden payload to probes again.

### 🧪 Experiments
The evaluation spans five base models: Qwen3-8B, Llama-3.1-8B, Ministral-8B, Qwen3-14B, and Phi-4-14B. The core task is steganographic exfiltration of 32-bit prompt-borne secrets via 32-token completions. ### 💡 TL;DR
This paper argues that uncertainty estimation for code generation should exploit code-specific properties rather than directly reuse natural-language uncertainty methods. It proposes a three-axis framework—lexical, algorithmic, and functional uncertainty—and shows that their ensemble substantially improves correctness prediction across five code LLMs, raising average AUROC from 0.696 for the best NL-derived baseline to 0.776.

### 🏆 Key Contributions
- Introduces a code-specific uncertainty estimation framework based on three distinct properties of code: token fragility, intent–code gap, and executability.
- Proposes lexical uncertainty via Top-K token entropy, showing that sparse high-entropy token positions are more informative than full-sequence averaging for code correctness.
- Proposes algorithmic uncertainty via pseudo-code consistency, eliciting multiple natural-language solution plans and measuring their agreement as a proxy for stable algorithmic intent.
- Proposes functional uncertainty via behavioral consistency, using self-generated test cases and execution outcomes to estimate uncertainty from runtime behavior.
- Demonstrates that a simple ensemble of the three signals consistently outperforms strong NL-derived uncertainty baselines across multiple benchmarks and models.
- Provides a cost–performance analysis showing that single-pass Top-K entropy can match stronger multi-pass baselines at much lower inference cost, especially on Qwen3-14B.

### ⚙️ Methodology
The paper formulates post-hoc uncertainty estimation for code generation as assigning an uncertainty score U(x,y) to a generated program y given prompt x, where higher values indicate greater uncertainty and correctness is defined functionally by pass@1 on hidden official tests. The method decomposes code uncertainty into three orthogonal axes. First, lexical uncertainty targets token fragility: since a single incorrect token can invalidate a program, the method computes token-level predictive entropy for each generation step and aggregates only the K highest-entropy positions rather than averaging across the full sequence. This Top-K entropy is intended to preserve sparse but decisive uncertainty spikes. Second, algorithmic uncertainty targets the gap between intent and implementation: instead of comparing sampled code directly, the model is prompted N times to produce natural-language pseudo-code plans consisting of 6–10 numbered reasoning steps while forbidding actual code constructs. Pairwise similarity between plans is measured using a step-aware ROUGE-L metric with bidirectional max matching, and uncertainty is defined as one minus the mean pairwise similarity, so disagreement among plans indicates unstable algorithmic reasoning. Third, functional uncertainty targets executability: for each prompt, the model generates M self-tests, the candidate program is executed against them in a sandbox, and behavioral consistency is defined as the fraction of tests passed. Functional uncertainty is then one minus this pass fraction. Unlike methods such as CodeT that use tests for reranking, this work uses test outcomes as a calibration signal for the specific generated program. To combine the three heterogeneous signals, the paper applies rank normalization and a weighted sum, emphasizing that the ensemble is meant to demonstrate complementarity among lexical, algorithmic, and functional signals rather than introduce a complex fusion model.

### 🧪 Experiments
The evaluation covers four standard code-generation benchmarks—APPS-Intro, APPS-Interview, HumanEval, and MBPP—across five code-capable LLMs, including Qwen3-14B. The task is to predict whether a generated program is correct under official hidden tests, and performance is measured primarily using AUROC and PRAUC, with uncertainty scores inverted so that higher confidence corresponds to correctness. The proposed methods are compared against both single-pass and multi-pass uncertainty baselines adapted from natural-language generation, including likelihood-based, entropy-based, semantic-consistency, and verbalized-confidence approaches. Results show that the three-axis ensemble improves average AUROC to 0.776 across the five models, compared with 0.696 for the strongest NL-derived baseline, an absolute gain of 8.1 points. On Qwen3-14B, the ensemble reaches 0.800 average AUROC. The lexical Top-K entropy signal alone is especially strong: on Qwen3-14B it matches the strongest multi-pass baseline while requiring only a single generation pass and more than 3x lower cost. Additional analysis reveals that entropy restricted to code tokens yields AUROC 0.716, whereas comment-only entropy performs worse than random at 0.375, supporting the claim that meaningful uncertainty is concentrated in code rather than surrounding natural-language text. The paper also reports cost–performance tradeoffs, recommending Top-K entropy when inference budget is limited and the full ensemble when calibration quality matters more.

### 🌱 Insights & Implications
A key insight is that code uncertainty is structurally different from natural-language uncertainty: correctness often hinges on a few critical tokens, implementation surface form may obscure or distort underlying algorithmic intent, and executable behavior provides a uniquely strong calibration signal unavailable in NL generation. The results suggest that direct transplantation of NL uncertainty estimators misses these properties and leaves substantial performance on the table. The paper also shows that token-level entropy, often dismissed as too local or simplistic, becomes highly effective when aggregated sparsely via Top-K rather than averaged globally. Another important implication is that pseudo-code may serve as a more stable abstraction level for reasoning about code semantics than raw syntax, which could inform future work in code verification, repair, and agent planning. Limitations include dependence on self-generated tests for the functional signal, since such tests may be incomplete, easy, or erroneous; the algorithmic signal is prompt- and sampling-dependent and may inherit LLM biases in plan generation; and the ensemble uses relatively simple normalization and fusion rather than learning a calibrated joint estimator. Open questions include whether finer-grained execution traces, richer semantic equivalence measures, or learned combinations of the three axes could further improve uncertainty estimation, and how these signals behave in larger agentic software-engineering workflows.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it addresses a foundational reliability problem for LLM-generated code: knowing when a generated program is likely wrong. Better uncertainty estimation can directly support selective review, repair triggering, test prioritization, and safer autonomous coding pipelines. The three proposed signals—token fragility, pseudo-code consistency, and behavioral consistency—map naturally onto settings such as patch validation, vulnerability-fix confidence scoring, and triaging suspicious generations, making the work a strong methodological contribution for building trustworthy LLM-based SE and security tools.

## 📄 [Hiding in Plain Floats: Steganographic Carriers for Indirect Prompt and Content Injection](https://arxiv.org/abs/2606.08403v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Mudit Sinha, Sanika Chavan
📅 **Published:** 2026-06-07  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
The paper identifies a reproducible failure mode in LLM-integrated structured-data pipelines: malicious content that is easily caught as plain text can evade text-only prompt-injection defenses when hidden in float arrays and only reconstructed into fragmented telemetry before reaching the model. Across 14,400 real-model attack trials on three commercial APIs, the proposed IFS-derived float-array carrier retains 94.3% leakage attack success under the strongest evaluated text-classifier defense, showing a clear boundary of text-centric inspection.

### 🏆 Key Contributions
- Introduces two steganographic carriers for indirect prompt/content injection through procedural-generation float parameters: spectral-coefficient encoding (T2) and an IFS-derived float-array carrier (T3).
- Demonstrates, via a 2x2 ablation, that data-layer hiding and reconstruction-layer fragmentation defeat different inspection views and both are required to evade dual-layer text inspection.
- Provides a large-scale empirical evaluation over 14,400 attacked trials on three commercial LLM APIs, showing that strong text-only defenses catch plain-text payloads but fail on the float-array carrier.
- Replicates the carrier-level detection boundary across two detector families: a Prompt Guard 2 + TF-IDF ensemble in the main evaluation and a fine-tuned roberta-base detector in replication.
- Shows that simple presentation-specific defenses such as xxd-pattern detection and semantic validation can block the current T3 instance, framing the result as a measured failure boundary rather than an undetectable exploit.

### ⚙️ Methodology
The study considers an LLM pipeline of the form raw structured config -> trusted reconstruction -> model-facing report -> LLM output. The attacker controls only upstream structured asset fields, especially numeric float arrays, and cannot modify prompts, model weights, defenses, or reconstruction code. Four carrier types are evaluated. T0 is a plain-text baseline where payload prose is visible in both raw config and reconstructed report. T1b-abl is an ablation in which payload prose is visible in raw storage but transformed into a fragmented hex-dump style representation at reconstruction, isolating reconstruction-layer evasion. T2, spectral-coefficient steganography, hides ASCII characters inside low decimal digits of float coefficients, so the raw config contains only apparently legitimate harmonic parameters; reconstruction deterministically decodes these coefficients back into contiguous text, isolating data-layer evasion. T3, the main attack, is an IFS-derived float-array carrier in which payload bytes drive an iterated-function-system-like contraction process and the resulting trajectory is stored as paired float arrays. The arrays are then whitened, jittered within bounds, and deterministically shuffled to reduce simple statistical separability. During trusted reconstruction, these arrays are converted into fragmented telemetry, specifically hex-dump-like output, so the payload is never exposed as directly inspectable prose at either the raw structured-input layer or the reconstructed-text layer. The evaluation focuses on indirect prompt/content injection success measured primarily by leakage ASR, defined as target-string presence in the model output, because downstream systems may act on reproduced markers even when the model refuses. Strong ASR is also reported as a stricter metric requiring structurally compliant attack success. The defense analysis includes raw-layer and reconstruction-layer inspection, with emphasis on text-only defenses such as schema/regex gates, Prompt Guard-style classifiers, TF-IDF text classifiers, and a fine-tuned roberta-base detector. Additional analysis uses simple feature-based detection on IFS samples to assess how whitening, jitter, and shuffling affect first-order separability.

### 🧪 Experiments
The paper evaluates attacks across 14,400 attacked real-model trials using three commercial LLM APIs from different providers to capture varying refusal and response behaviors. The main defense matrix includes four defense configurations, with the strongest highlighted setup being a dual-layer Prompt Guard 2 + TF-IDF ensemble operating on text views of the raw config and reconstructed report. A replication study replaces the main classifier with a fine-tuned roberta-base detector. The primary metric is leakage ASR, with Strong ASR reported separately. Results show that the plain-text baseline T0 achieves 92.7% ASR without defenses and 73.7% under a boundary-prompt defense, but falls to 0% once a fine-tuned classifier or schema gate is applied. In contrast, the main T3 IFS-derived float-array carrier preserves 94.3% leakage ASR under the strongest dual-layer text-classifier defense. At the detector level, Prompt Guard 2 + TF-IDF catches 0% of T3 samples, and classifier confidence on IFS samples peaks at 0.225 versus a 0.5 detection threshold. The same carrier-level pattern replicates with the fine-tuned roberta-base detector, which catches prose-visible carriers but misses T3. A 2x2 ablation shows that T2 defeats data-layer text inspection but not reconstruction-layer inspection, while T1b-abl defeats reconstruction-layer visibility but not raw-layer visibility; only T3 combines both properties and evades both views. For statistical detectability, a RandomForest trained on eight simple features perfectly separates naive IFS-4 samples but drops to 58% +/- 0.04 accuracy after whitening, bounded jitter, and deterministic shuffling, indicating reduced first-order separability. Finally, a simple xxd-line detector reduces ASR to 0% for the current presentation, and semantic validation also blocks the specific T3 instance.

### 🌱 Insights & Implications
The core insight is that many prompt-injection defenses are not truly content-aware at the structured-data level; they assume the malicious signal is visible as text in at least one inspected representation. This paper shows that if the signal is hidden in numeric carriers upstream and reconstructed only into fragmented telemetry, text-only inspection can fail even when the underlying payload would be trivially caught as plain English. The ablation is especially important because it disentangles two distinct evasion mechanisms: hidden storage defeats raw-input inspection, while fragmented reconstruction defeats report-level text inspection. The work also clarifies that leakage-style propagation matters operationally, since downstream routers or parsers may act on quoted markers even when a model partially refuses. At the same time, the authors explicitly note limitations: the attack is demonstrated in a constructed procedural-float pipeline with trusted reconstruction that emits auxiliary channels into model context; it is not evidence that arbitrary structured-input pipelines are vulnerable in the same way. The current T3 realization is not stealthy against presentation-specific signatures, as xxd-format detection and semantic validation can block it. Thus the contribution is best understood as a boundary case exposing the insufficiency of text-centric defenses, not a claim of universal or undetectable compromise. The broader implication is that future defenses should validate structured inputs semantically and reason about carrier channels before reconstruction, rather than relying only on prompt-injection classifiers over text strings.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and secure LLM-enabled software pipelines because it characterizes a concrete, measurable failure mode in guardrail design rather than proposing a purely conceptual attack. For researchers working on automated vulnerability detection or repair for AI-integrated systems, it highlights that defenses must account for non-textual and reconstructed data channels, analogous to hidden attack surfaces in traditional software systems. It also advances research on secure software maintenance by motivating schema-aware, semantic, and pre-reconstruction validation techniques for structured inputs, which are directly applicable to building safer LLM-based developer tools, code assistants, and automated remediation pipelines.
