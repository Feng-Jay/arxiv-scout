## 📄 [AttackPathGNN: Cross-function vulnerability detection in smart contracts using state interference graphs and conjunction pooling](https://arxiv.org/abs/2606.05986v1)  `90% relevant`

👤 **Authors:** Gabriela Dobrita, Simona-Vasilica Oprea, Adela Bara
📅 **Published:** 2026-06-04  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
AttackPathGNN targets smart-contract vulnerabilities that emerge across interacting functions rather than within isolated functions. It combines a state-aware inter-function graph with a differentiable conjunction mechanism over exploit preconditions, achieving strong held-out F1 on SmartBugs Wild and high reentrancy detection while also producing actionable remediation reports.

### 🏆 Key Contributions
- Reframes Solidity vulnerability detection from single-function syntactic pattern matching to explicit reasoning over cross-function attack paths.
- Introduces a State Interference Graph that connects functions sharing mutable storage via typed, weighted edges and adds directed reentrancy-path edges based on an explicit five-condition predicate.
- Proposes conjunction pooling, a differentiable AND-style aggregator over eight named exploit preconditions, designed so exploit likelihood sharply drops when any required condition is blocked by a mitigation.
- Demonstrates strong empirical performance on SmartBugs Wild and SmartBugs Curated, including very high reentrancy detection and perfect recovery for 6 of 10 DASP categories across all seeds.
- Augments predictions with structured remediation reports, making model outputs usable as function-level audit findings rather than only binary labels.

### ⚙️ Methodology
The paper appears to model each smart contract as a graph whose nodes are functions and whose edges encode security-relevant inter-function relationships. The central representation is the State Interference Graph, which likely links all pairs of functions that access shared mutable storage, with edge types and weights capturing the nature or strength of interference. On top of this storage-sharing structure, the model adds directed reentrancy-path edges when an explicit five-condition predicate indicates that a feasible reentrant attack path may exist between functions. A graph neural network then propagates information over this graph so each function representation reflects both its own properties and the security implications of neighboring functions that modify or depend on shared state. A second key component is conjunction pooling: instead of treating vulnerability as a generic latent score, the model computes eight named exploit preconditions and combines them with a differentiable AND-like operator using a log-sigmoid formulation. This design encodes the intuition that an exploit should only be predicted when all necessary conditions co-occur, and that the presence of a single mitigation such as a reentrancy guard, access-control modifier, or SafeMath should suppress the final exploit score. The output is not just a vulnerability label but also a structured remediation report, suggesting that the intermediate precondition reasoning is exposed in a human-interpretable way to support auditing and repair.

### 🧪 Experiments
Evaluation is reported over five independent training runs, indicating attention to stability rather than one-off peak performance. The main benchmark is the held-out test partition of SmartBugs Wild, where AttackPathGNN achieves 92.3 +/- 0.2% F1 and a 4.3 +/- 0.3% false-negative rate. Generalization is further assessed on the independently human-labelled SmartBugs Curated benchmark, where the model reaches a 90.8 +/- 2.5% detection rate. Category-level results are also highlighted: the system recovers 6 out of 10 DASP10 vulnerability categories at 100% on every random seed and detects Reentrancy at 98.7 +/- 1.8%. While the abstract does not enumerate baselines, it explicitly contrasts the method with prior learning-based and GNN-based detectors that focus on intra-function syntactic patterns, implying comparisons against standard smart-contract vulnerability detectors. Metrics emphasized include F1, false-negative rate, detection rate, category-wise recovery, and robustness across multiple seeds.

### 🌱 Insights & Implications
The paper’s main insight is that many important smart-contract exploits are fundamentally relational: they depend on how functions interact through shared state and on whether multiple exploit preconditions jointly hold. This is highly relevant to vulnerability detection research because it moves beyond local code-pattern recognition toward attack-feasibility modeling, which is closer to how auditors reason about real exploits. The conjunction pooling mechanism is particularly interesting for security because it hard-codes a notion of necessary conditions and mitigation-aware suppression, potentially improving precision and interpretability compared with unconstrained neural classifiers. A likely limitation is that the model’s success depends on the quality of the handcrafted graph construction and the explicit five-condition predicate for reentrancy-path edges; if these abstractions omit important behaviors, the detector may miss non-modeled attack paths. Another open question is how well the approach generalizes to unseen vulnerability classes, evolving Solidity language features, proxy/upgrade patterns, and multi-contract or cross-chain interactions. Still, the remediation-report output suggests a promising bridge from detection to vulnerability triage and automated repair assistance.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and vulnerability repair because it addresses a hard class of bugs that arise from inter-procedural interactions rather than isolated code smells. For a researcher interested in automated program repair and security-oriented software maintenance, its explicit exploit-precondition modeling, mitigation-aware scoring, and structured remediation outputs offer useful ideas for building more explainable detectors and for coupling detection with actionable repair recommendations.

## 📄 [The End of Software Engineering: How AI Agents Are Fundamentally Restructuring the Software Paradigm](https://arxiv.org/abs/2606.05608v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Zhenfeng Cao
📅 **Published:** 2026-06-04  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper argues that AI agents represent a paradigm shift beyond traditional software engineering: instead of humans encoding static decision logic into maintainable code, LLM-based agents dynamically reason, generate ephemeral code, and deliver outcomes directly. It frames this as the rise of "Agentic Engineering" and synthesizes benchmark evidence suggesting strong potential for software tasks, while emphasizing that reliability, coordination, and governance remain unresolved.

### 🏆 Key Contributions
- Formalizes a conceptual distinction between traditional software systems, where code is the persistent carrier of decision logic, and agentic systems, where an LLM performs runtime reasoning and code is a temporary execution tool.
- Presents a first-principles argument that the shift to agentic systems is driven by complexity scaling limits in conventional software engineering rather than by mere tooling convenience.
- Introduces a three-stage historical framing of software delivery evolution: local software, SaaS, and Agent-as-a-Service (AaaS), where user-facing complexity is progressively transferred away from customers.
- Defines "Agentic Engineering" as a new discipline centered on intent specification, multi-agent orchestration, memory, observability, and outcome auditing rather than direct manual programming.
- Synthesizes recent benchmark and case-study evidence, including SWE-bench Verified, EvoClaw, and LangChain-style multi-agent studies, to argue that agents are already beginning to perform meaningful software engineering work while exposing current limitations.
- Proposes a four-stage roadmap toward self-evolving agent ecosystems and offers practitioner-facing guidance for navigating the transition from software-centric to agent-centric systems.

### ⚙️ Methodology
The paper is primarily a conceptual and analytical position paper rather than a new algorithmic contribution. It begins by formally defining a traditional software system as a tuple of computational resources, static decision rules encoded in source code, and an execution environment. It then contrasts this with an AI agent system defined by an LLM reasoning core, a tool set, memory, and a planning mechanism that iteratively chooses actions based on current state and memory. Using this formalization, the author argues that the key difference is where decision logic resides: in traditional software it is pre-encoded by humans, while in agentic systems it is produced dynamically at runtime. The paper further develops a complexity-scaling argument based on combinatorial growth in interaction paths among system components, claiming that traditional software remains bounded by human cognitive limits even when implementation tooling improves. Agentic systems, by contrast, are framed as offloading reasoning to models whose effective capacity scales with model size and training compute. On top of this analytical core, the paper builds a paradigm model of software delivery evolution from licensed software to SaaS to AaaS, then defines Agentic Engineering as an emergent discipline focused on agent coordination, tool use, memory, observability, and human oversight of outcomes rather than direct authorship of all executable logic. The work therefore uses formal definitions, asymptotic reasoning, historical interpretation, and qualitative synthesis of prior benchmark literature to support its thesis.

### 🧪 Experiments
The paper does not report novel experiments or an original evaluation protocol. Instead, it surveys and interprets existing benchmark evidence from recent agentic software engineering studies, explicitly citing SWE-bench Verified, EvoClaw, and LangChain multi-agent coordination results as representative empirical signals. The discussion uses these prior evaluations to support claims that agentic systems can perform repository-level bug fixing, code modification, workflow decomposition, and collaborative task execution at increasingly practical levels. However, because this is a synthesis paper, the exact datasets, baselines, and metrics are inherited from the cited works rather than reimplemented here. The quantitative takeaway is therefore directional rather than experimentally original: current agents show meaningful gains on realistic software engineering tasks, but still suffer from reliability gaps, coordination overhead, and limits in sustained autonomous operation across the full lifecycle.

### 🌱 Insights & Implications
For software engineering researchers, the paper’s central insight is that the locus of engineering may shift from writing durable program logic to designing agent control loops, tool ecosystems, memory, and audit mechanisms. This is especially important for maintenance and repair, because the paper suggests future systems may regenerate solutions on demand instead of evolving a single long-lived code artifact. For security and vulnerability research, the framework implies new attack surfaces and assurance problems: if decision logic is produced dynamically, then verification, reproducibility, traceability, and policy enforcement become harder and more important. A key limitation is that the paper is largely argumentative and speculative; its complexity claims are high-level and not empirically validated with controlled studies, and some assertions about inevitability may overreach current evidence. Another open question is how far ephemeral code generation can replace persistent engineered systems in safety-critical, compliance-heavy, or infrastructure-intensive domains where maintainability, determinism, and auditability matter. The paper is most valuable as a provocative conceptual framing that can motivate new research agendas in evaluation, observability, autonomous repair, secure tool use, and governance of agent-driven development.

### 🎯 Why This Paper
This paper is relevant because it directly addresses the future structure of software engineering workflows that underpin software maintenance, automated program repair, and vulnerability handling. For a researcher in repair and security, it offers a useful conceptual lens for thinking about how bug fixing, patch synthesis, maintenance automation, and even vulnerability remediation may shift from static code-centric pipelines to agent-centric runtime reasoning systems. It advances your work by reframing key research questions: not just how to generate correct patches, but how to build trustworthy agents that can localize faults, reason over repositories, invoke tools safely, preserve traceability, and deliver secure, auditable repairs at scale.

## 📄 [TeleSWEBench: A Commit-Driven Benchmark for Evaluating LLM-Powered Software Engineering in Telecommunications](https://arxiv.org/abs/2606.05001v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Pranshav Gajjar, Ali Mamaghani, Dinesh Bharadia, Vijay K Shah
📅 **Published:** 2026-06-03  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper introduces TeleSWEBench, the first commit-driven benchmark for evaluating LLM-based software engineering agents on real telecommunications code changes mined from the srsRAN 5G repository, and proposes TeleJudge, a hierarchical LLM-as-judge evaluator for multi-file telecom patches. Across 734 benchmark tasks and several agent/model combinations, the study shows that current systems struggle with both locating the right code and producing functionally correct changes, with the best tools reaching only about 25% shippable patches.

### 🏆 Key Contributions
- Introduces TeleSWEBench, a telecom-specific, commit-driven software engineering benchmark built from real srsRAN 5G developer commits spanning 2023–2025.
- Constructs 734 benchmark questions with executable unit tests, organized into Easy, Medium, and Difficult tiers based on how much repository localization and inference is required.
- Defines a two-stage evaluation protocol that separates scope/localization ability from functional correctness, enabling more precise diagnosis of agent failures.
- Proposes TeleJudge, a hierarchical LLM-as-a-judge framework that evaluates generated patches at the file level and aggregates them holistically using contextual and semantic similarity signals.
- Provides an empirical evaluation of major SWE-agent frameworks including AIDER, OpenHands, and ClaudeCode with multiple reasoning LLMs, demonstrating substantial performance gaps in telecom software engineering.

### ⚙️ Methodology
TeleSWEBench is built by mining over 15,000 commits from the srsRAN 5G repository, an open-source O-RAN/5G software stack whose commits average 5.57 changed files, highlighting strong cross-file coupling. The authors transform real commits into benchmark instances that reflect authentic telecom maintenance, bug-fixing, and feature-update tasks. Each instance is expressed at one of three difficulty levels: Easy tasks provide exact file paths, line references, and almost mechanical edit instructions; Medium tasks describe the required change and relevant files/subsystems but omit line-level edits; Difficult tasks specify only high-level goals with minimal hints and intentionally avoid naming files or functions, forcing autonomous repository navigation. Question generation uses a cost-aware fallback cascade over multiple LLMs, escalating from smaller to stronger models only when needed. Validation is also LLM-assisted, with generated tasks checked against the original commit context to ensure consistency and difficulty-tier adherence. For evaluation, the paper uses a two-stage setup that disentangles localization from correctness: first assessing whether the agent identifies and edits the right files/context, then whether the produced patch is actually correct. To complement unit-test-based execution, the authors introduce TeleJudge, a hierarchical judge that scores outputs at the file level and aggregates judgments into a holistic patch-level verdict using semantic and contextual similarity to the reference changes. This is intended to mitigate brittleness in exact test-based evaluation, especially for multi-file patches where equivalent implementations may differ syntactically.

### 🧪 Experiments
The benchmark contains 734 tasks derived from srsRAN 5G commits and accompanied by executable unit tests. The evaluation covers multiple autonomous software engineering frameworks, specifically AIDER, OpenHands, and ClaudeCode, paired with contemporary reasoning-oriented LLMs including Qwen3, GPT OSS, Gemma 4, Kimi, and Qwencoder 2.5. The study applies both repository-native unit-test execution and the proposed TeleJudge evaluation in a two-stage pipeline, allowing separate measurement of file localization/scope identification and end-to-end functional correctness. The benchmark’s difficulty tiers stress increasing degrees of repository reasoning, from explicit edit execution to open-ended search across the codebase. The core quantitative finding reported in the abstract is that even the strongest ASE tools achieve only up to 25% shippable changes. The analysis further shows that failures arise from both poor localization accuracy—agents often do not find or modify the correct files or code regions—and weak functional correctness, meaning that even when edits target plausible locations, they often do not satisfy the intended telecom behavior or pass validation. The paper also validates TeleJudge against executable unit tests to support its reliability for multi-file patch assessment.

### 🌱 Insights & Implications
The paper makes clear that general SWE benchmarks substantially underestimate the difficulty of telecom software engineering, where code changes involve stateful protocol logic, strict timing requirements, layered dependencies, and standard-compliance constraints across a large C++ codebase. A key insight is that repository navigation and patch correctness are distinct capabilities, and current agents are weak at both; evaluating them with a single pass/fail score obscures these failure modes. The benchmark also highlights the practical need for alternatives to pure execution-based evaluation, since multi-file patches may require more flexible semantic judgment than rigid tests alone can provide. At the same time, reliance on LLM-as-a-judge raises open questions about evaluator bias, sensitivity to semantically equivalent but structurally different patches, and generalization beyond the specific repository. Another limitation is scope: TeleSWEBench is anchored in srsRAN 5G, so it captures one important but still single ecosystem within telecommunications. Future work could extend the benchmark to additional telecom stacks, vulnerability-fixing tasks, or protocol-security scenarios. More broadly, the results imply that domain-specialized training, better repository search/localization, and stronger handling of cross-file constraints are necessary before LLM agents can be trusted in production telecom maintenance workflows.

### 🎯 Why This Paper
This paper is highly relevant because it studies automated software engineering on a realistic, complex, repository-scale codebase with strong cross-file dependencies—exactly the type of setting that matters for software maintenance, program repair, and trustworthy code modification. Its commit-driven benchmark design aligns closely with automated program repair research, while the two-stage error analysis is useful for understanding whether failures come from bug localization or repair synthesis. Although the paper is not centered on security vulnerabilities specifically, its methods and findings transfer naturally to vulnerability detection and repair in specialized systems software, especially where correctness, standards compliance, and multi-file patching are critical.

## 📄 [Revisiting Vul-RAG: Reproducibility and Replicability of RAG-based Vulnerability Detection with Open-Weight Models](https://arxiv.org/abs/2606.04739v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Sabrina Kaniewski, Fabian Schmidt, Tobias Heer
📅 **Published:** 2026-06-03  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper reproduces and extends Vul-RAG, a retrieval-augmented LLM framework for source-code vulnerability detection, entirely with local open-weight models. The main finding is that Vul-RAG’s original trends largely hold under reproducible local deployment, but performance plateaus at about 0.30 pairwise accuracy across a wide range of newer and larger models, suggesting that model scaling alone does not solve the task.

### 🏆 Key Contributions
- Reimplemented and evaluated Vul-RAG in a fully local, open-weight setting to remove dependence on proprietary APIs and improve reproducibility.
- Reproduced the original Vul-RAG results using the reported open-weight baseline models, confirming the main findings with only minor deviations.
- Systematically extended evaluation to a diverse set of recent open-weight LLMs spanning code-specialized, general-purpose, and reasoning-oriented models of different sizes.
- Showed that vulnerability-detection performance saturates at roughly 0.30 pairwise accuracy across all tested models, including newer and more capable ones.
- Analyzed trade-offs between model specialization, reasoning capability, and parameter scale for practical deployment of RAG-based vulnerability detection systems, and released artifacts publicly for reuse.

### ⚙️ Methodology
The study revisits Vul-RAG, a knowledge-level retrieval-augmented generation framework for vulnerability detection. Vul-RAG uses a pre-built vulnerability knowledge base derived from historical CVEs and corresponding patches, where each knowledge item contains three abstractions: functional semantics of the vulnerable code, vulnerability causes, and fixing solutions. In this reproduction, the authors keep the original knowledge base fixed to isolate the effects of changing the underlying language model.

At inference time, the framework first prompts an LLM to summarize the input function’s functional semantics, including its high-level purpose and detailed behavior. This semantic representation, together with the source code, is used to retrieve top relevant knowledge items from the knowledge base using BM25 followed by re-ranking. Detection is then performed iteratively rather than with a single large prompt: for each retrieved knowledge item, the LLM is asked first whether the vulnerability cause described by that item applies to the input code, and then whether the corresponding fixing solution is present. If the model concludes that the code exhibits the vulnerability and lacks the fix, the function is labeled vulnerable; otherwise the process continues through the ranked retrieved items until either a vulnerability is found or all items are exhausted, in which case the function is classified as non-vulnerable.

The reproduction adapts the entire online pipeline to open-weight, locally deployed models. The study is structured around five research questions: whether reported results are reproducible, how newer models affect performance, whether code-specialized models outperform general-purpose models, whether reasoning models help, and whether larger parameter counts improve results. The focus is therefore not on proposing a new detector, but on systematically stress-testing the robustness and generalizability of a published RAG-based vulnerability detection framework under controlled model substitutions.

### 🧪 Experiments
The experiments use the benchmark from the original Vul-RAG paper: 2,903 vulnerable/patched Linux kernel function pairs associated with ten common CWEs. The dataset is split 1:4 into 586 test pairs (PairVul) and 2,317 training pairs used to construct the vulnerability knowledge base. The benchmark covers four CWE-1000 research pillars, with strong representation of resource-management weaknesses typical of C and Linux kernel code.

The authors first reproduce the original open-weight baselines from Vul-RAG under fully local inference, then evaluate a broader set of recent open-weight LLMs, including code-focused, general-purpose, and reasoning-oriented models at multiple parameter scales. Evaluation emphasizes pairwise accuracy: a pair counts as correct only if the model correctly classifies both the vulnerable function and its patched counterpart. This is a stricter and security-relevant metric because it measures whether the detector can distinguish vulnerable code from its fix.

The reproduced results are close to those originally reported, indicating that the approach is reproducible with minor deviations. However, across all tested models, performance consistently saturates around 0.30 pairwise accuracy. The abstract highlights that this ceiling remains even for newer and more advanced models, implying limited gains from model generation or size alone. While the paper also compares models by specialization, reasoning capability, and scale, the dominant quantitative conclusion is the persistent plateau and lack of substantial improvement beyond the reproduced baseline range.

### 🌱 Insights & Implications
The key insight is that Vul-RAG’s behavior appears robust across deployment settings but not strongly sensitive to newer model releases: local open-weight replication validates the original conclusions, yet the framework does not benefit much from simply swapping in larger or more recent LLMs. This suggests the bottleneck is likely in the task formulation, retrieval quality, knowledge representation, or multi-step prompting pipeline rather than raw model capability.

For vulnerability research, the 0.30 pairwise-accuracy ceiling is especially important because it shows that distinguishing a vulnerable function from its patched version remains difficult even with retrieval support and modern LLMs. The iterative reasoning process may fail at multiple points, such as extracting reliable semantics for retrieval or correctly aligning retrieved causes/fixes to the target code. The study therefore raises open questions about whether better structural code representations, learned retrievers, vulnerability-specific fine-tuning, patch-aware modeling, or richer knowledge bases are needed.

A practical implication is that fully local, open-weight deployment is feasible and preferable for security-sensitive code analysis, but current RAG-based LLM pipelines should not be assumed production-ready purely because they reproduce prior academic results. A limitation of the study is that it reuses the original knowledge base rather than redesigning it, so the findings mainly isolate model effects rather than the full design space of RAG-based vulnerability detection. Still, this is valuable because it clarifies that model replacement alone is unlikely to deliver major gains.

### 🎯 Why This Paper
This paper is highly relevant because it directly evaluates an LLM-based vulnerability detection system under reproducible, open-weight conditions, which aligns with interests in software security, vulnerability detection, and trustworthy automated analysis. It advances the field by showing that reported RAG-based gains are not merely artifacts of proprietary APIs, while also revealing a hard performance ceiling that motivates new work on retrieval design, knowledge representation, and repair-aware reasoning for future vulnerability detection and automated repair systems.

## 📄 [CyberGym-E2E: Scalable Real-World Benchmark for AI Agents' End-to-End Cybersecurity Capabilities](https://arxiv.org/abs/2606.04460v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Tianneng Shi, Robin Rheem, Dongwei Jiang, Mona Wang, Francisco De La Riega, Zhun Wang, Jingzhi Jiang, Alexander Cheung, Sean Tai, Jonah Cha, Jianhong Tu, Gabriel Han, Chenguang Wang, Jingxuan He, Wenbo Guo, Dawn Song
📅 **Published:** 2026-06-03  🏷️ **Categories:** cs.CR, cs.AI, cs.LG

### 💡 TL;DR
CyberGym-E2E is a large-scale benchmark that evaluates AI agents across the full real-world vulnerability lifecycle—detection, PoC generation, and patch generation—using 920 historical OSS-Fuzz vulnerabilities from 139 open-source projects in realistic repo-level environments. Its main finding is that even strong agentic systems can often produce patches, but still struggle substantially with vulnerability discovery and exploit-input generation, making true end-to-end success rare.

### 🏆 Key Contributions
- Introduces CyberGym-E2E, a scalable end-to-end cybersecurity benchmark spanning vulnerability detection, PoC generation, and patch generation on the same real-world vulnerabilities.
- Builds an automated, agent-enhanced benchmark construction pipeline that transforms OSS-Fuzz vulnerability records into reproducible repo-level evaluation environments with vulnerable code, PoCs, patches, and tests.
- Provides realistic agentic execution settings by placing agents inside buildable sandboxed project environments rather than limiting them to static code inspection.
- Uses fixing commits as ground-truth patches and automatically extracts developer-written unit tests to validate post-patch functionality, with expert review for quality assurance.
- Assembles a benchmark of 920 vulnerabilities across 139 open-source projects, substantially improving scale over prior end-to-end real-world benchmarks.
- Presents a large-scale empirical evaluation showing that patch generation is relatively stronger than vulnerability detection and PoC generation, which remain major bottlenecks for end-to-end autonomous cyber defense.

### ⚙️ Methodology
The paper constructs CyberGym-E2E from historical real-world vulnerabilities primarily sourced from OSS-Fuzz. The benchmark design targets four properties: realism, reproducibility, scalability, and end-to-end evaluation. The pipeline first identifies vulnerabilities from OSS-Fuzz-covered repositories and reconstructs the vulnerable project state. It then prepares a realistic repo-level build environment, packaged in Docker, where an agent can interact with the full codebase and toolchain similarly to a real software/security engineer. A key challenge is bridging old historical vulnerability environments with modern agent requirements, so the benchmark rebuilds compatible execution environments rather than relying solely on legacy setups. For each vulnerability, the pipeline identifies the corresponding fixing commit to derive a ground-truth patch and obtains the associated PoC/crashing input from the fuzzing-based vulnerability record. These test harnesses are then expert-reviewed to ensure quality and correctness. The resulting task instances support sequential evaluation over the same vulnerability: agents must identify the bug in the codebase, generate a valid PoC that triggers it, and finally synthesize a patch that removes the vulnerability while preserving intended software functionality. The benchmark also enforces sandboxing and restrictions to prevent cheating, such as modifying evaluation scripts, thereby making the setup more representative of real deployment conditions than prior read-only benchmarks.

### 🧪 Experiments
The benchmark contains 920 real-world vulnerabilities from 139 OSS-Fuzz-covered open-source projects. The evaluation is repo-level and end-to-end: agents operate inside realistic build environments and are assessed on three linked tasks for each vulnerability—detection, PoC generation, and patch generation—plus post-patch functionality validation using extracted developer tests. The paper positions CyberGym-E2E against prior benchmarks in terms of scope, realism, and scale, noting that many previous datasets either focus on only one stage of the lifecycle, use synthetic tasks, omit functionality tests, or are much smaller. Quantitatively, the central reported result is that current state-of-the-art agentic systems still find end-to-end cybersecurity difficult: patch generation achieves comparatively high success, while vulnerability detection and PoC generation remain significantly harder and therefore dominate end-to-end failure. The excerpt does not provide detailed per-model scores, exact metrics, or baseline tables, but it clearly states the qualitative ranking of task difficulty and that full lifecycle completion is still challenging at benchmark scale.

### 🌱 Insights & Implications
A major insight is that evaluating only patch generation can overestimate practical autonomous security capability, because successful remediation depends on first finding and concretely reproducing the vulnerability. The benchmark therefore exposes an important systems-level bottleneck: current AI agents may be relatively competent at editing code once guided, but are far less reliable at the earlier reasoning and exploit-construction stages needed for autonomous vulnerability triage. Another contribution is methodological: realistic cyber evaluation benefits from full repo access, buildable environments, and developer-authored regression tests, rather than function-level snapshots or weak patch equivalence heuristics. The work also highlights tradeoffs in benchmark construction—scalability requires automation, but validity still benefits from targeted expert review. Likely limitations and open questions include dependence on OSS-Fuzz-style vulnerability availability, possible bias toward fuzzer-discoverable bug classes and supported projects, and the challenge of ensuring extracted tests comprehensively capture intended functionality beyond the patched vulnerability. For future research, the benchmark suggests opportunities in integrated APR plus bug localization, exploit synthesis, test generation, and secure agent workflows that jointly optimize detection, reproduction, and repair.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, vulnerability detection, vulnerability repair, and automated program repair because it provides a realistic large-scale benchmark that unifies all three stages of the vulnerability lifecycle on real repositories. It advances such work by enabling rigorous evaluation of end-to-end repair systems, exposing where current agents fail before patching, and offering a valuable dataset and infrastructure for studying localization, exploit reproduction, patch correctness, and post-repair functionality preservation in practical settings.

## 📄 [SecRL-Prune: Structured Reinforcement Learning-Based Pruning of CodeLLMs for Preserving Adversarial Code Mutation](https://arxiv.org/abs/2606.06254v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Parsa Memarzadehsaghezi, Pooria Madani, Khalil El-Khatib
📅 **Published:** 2026-06-04  🏷️ **Categories:** cs.CR

### 💡 TL;DR
SecRL-Prune is a structured pruning method for 7B code LLMs that uses reinforcement learning to remove FFN channels while preserving the teacher model’s output distribution and code-mutation behavior. Across HumanEval and a malware mutation case study, it outperforms prior structured pruning baselines at 10–30% compression, showing that adversarial code rewriting capability survives substantial model compression.

### 🏆 Key Contributions
- Introduces SecRL-Prune, a reinforcement learning–based structured pruning framework for CodeLLMs that prunes FFN/MLP channels using a teacher-student KL-divergence reward.
- Proposes a teacher-output caching strategy that stores top-P predictions from the teacher once, reducing GPU memory needs by avoiding simultaneous teacher-student loading during policy learning.
- Presents an empirical study of how structured pruning affects execution correctness and code diversity in code generation/mutation tasks using pass@k and var@k.
- Demonstrates in a malware case study that semantics-preserving mutations generated by 20%-pruned CodeLLMs can significantly reduce detection rates across more than 62 malware detection engines.
- Highlights the security implication that compressed CodeLLMs remain capable of adversarial code mutation, making embedded or low-resource malicious deployment more feasible.

### ⚙️ Methodology
SecRL-Prune targets the feed-forward network inside each transformer block of a pretrained decoder-only CodeLLM. The pruning unit is the MLP/FFN intermediate channel: for each retained or removed channel, the corresponding row of the first FFN projection and column of the second projection are preserved or deleted together, yielding true structured compression rather than sparse masking. Starting from a pretrained teacher model, the method learns a layer-wise pruning policy with reinforcement learning. The policy decides how many channels to keep in each layer under a global compression budget, effectively allocating sparsity across layers rather than pruning uniformly. The reward is based on how well the pruned student matches the teacher’s token distribution, using KL divergence between teacher and student outputs on a calibration set. To make this practical for large CodeLLMs, the framework precomputes and caches only the teacher’s top-P next-token predictions on the calibration corpus, then trains/evaluates pruning policies against this compact target. This avoids holding teacher and student in GPU memory simultaneously and reduces repeated teacher forward passes. The objective therefore encourages the student to preserve the teacher’s generative behavior relevant to code generation and mutation while removing as many FFN channels as possible within the specified compression ratio. The overall design is tailored to post-training structured pruning for deployment-efficient compressed CodeLLMs while retaining semantics-preserving code rewriting ability.

### 🧪 Experiments
The evaluation uses three 7B CodeLLMs and measures performance on HumanEval under 10%, 20%, and 30% structured compression. The main metrics are pass@k, which captures execution correctness of generated code, and var@k, which measures diversity among valid generated solutions and is especially relevant for mutation capability. SecRL-Prune is compared against recent structured pruning baselines, including methods based on conventional post-training structured pruning rather than RL-based layer-wise policy learning. The reported results show that SecRL-Prune consistently preserves more of the original model’s pass@k and var@k than competing baselines, with the advantage becoming clearer at more aggressive pruning levels. A key implication is that future agent improvement methods should combine trace analysis with structured repair spaces instead of relying only on free-form self-editing. Likely limitations and open questions include the dependence on sufficiently rich traces and observability, the accuracy of LLM-based diagnosis and patch validation, the generality of the repair operators to novel harness architectures, and the cost and scalability of multi-agent repair loops in production settings.

### ⚙️ Methodology
The method assumes defenders have historical benign API queries but no labeled attack data. Queries are encoded with a fixed sentence embedding model into a semantic vector space. Rather than scoring each query independently, the detector operates on an incoming traffic window T and compares its empirical embedding distribution to that of benign historical traffic B. To reduce variance from any single benign sample, the system repeatedly samples L benign reference batches from the benign embedding pool, each matched to the incoming window size, and computes a discrepancy score between T and each reference batch. The discrepancy is instantiated with the unbiased squared maximum mean discrepancy (MMD) using an RBF kernel. Kernel bandwidth is chosen by the median heuristic on benign embeddings, and a multi-kernel variant averages RBF kernels with scales {0.5, 1, 2, 4} times the base bandwidth to improve robustness. The final anomaly score for a traffic window is the average MMD across the L benign reference batches. Threshold calibration is entirely benign-only: the defender repeatedly samples pairs of benign batches, computes benign-vs-benign MMD scores to form a null distribution, and sets the alarm threshold at a high percentile (95th percentile in experiments). At deployment, an incoming batch is flagged if its averaged MMD score exceeds this benign-calibrated threshold. The design supports both pure attacker windows and contaminated windows modeled as mixtures (1-rho)Pb + rhoPa, making it suitable for aggregate multi-user API traffic rather than only per-user monitoring.

The evaluation uses 14 attacker-normal query pairs spanning four extraction scenarios, with results reported over three random seeds. ### 🌱 Insights & Implications
A key insight is that extraction queries may look innocuous one by one, but their aggregate semantic distribution shifts enough to be detected at the traffic-window level. This makes batch-level testing more suitable than pointwise anomaly scoring for model extraction. The benign-only threshold calibration is especially important for realistic deployment because defenders rarely have representative attack samples and need low false positive rates to avoid alert fatigue. The paper also suggests that many prior methods may be disadvantaged in practical LLM API monitoring because they are tailored to account-level chronological behavior, image-domain assumptions, or single-query anomaly signals. Likely limitations include dependence on the quality and coverage of the benign reference pool, sensitivity to embedding choice and window size, and the possibility that adaptive attackers could try to mimic benign semantic distributions more closely. Open questions include how detection performance changes under concept drift in benign traffic, whether online threshold adaptation is needed, and how well the approach extends to multimodal APIs, multilingual traffic, or attackers that strategically interleave extraction with realistic benign prompts.

### 🏆 Key Contributions
- Proposes Code2LoRA, a hypernetwork-based framework that converts repository-level code context into repository-specific LoRA adapters for frozen code language models.

### ⚙️ Methodology
Code2LoRA has three main components: a frozen repository encoder, a trainable hypernetwork, and a frozen base code LLM. The repository encoder first embeds files or diffs using a frozen Qwen3-Embedding-0.6B model. Each file is split into 4096-token chunks with 512-token overlap, chunk embeddings are mean-pooled into file vectors, and repository vectors are then formed by combining weighted mean pooling and max pooling over file embeddings. File weights are based on factors such as content distinctiveness, file size, and path importance. The resulting repository embedding is a fixed-size vector used to condition LoRA generation.

In Code2LoRA-Static, a single repository snapshot embedding is fed into a 2-layer MLP with GELU activation, followed by module-specific output heads that generate LoRA A and B matrices for seven projection types in the transformer: q, k, v, o, gate, up, and down. The generated LoRA weights are shared across all layers of the base LLM and inserted into the frozen model through the standard low-rank update. The design includes learnable scaling parameters to control adapter magnitude. Only the hypernetwork is trained; the repository encoder and base model remain frozen.

Code2LoRA-Evo extends this idea to evolving repositories. Instead of conditioning on a single snapshot, it processes a sequence of code diffs and maintains a hidden state with a GRU. This hidden state summarizes repository evolution over time and conditions LoRA generation at each step, producing an adapter trajectory aligned with the commit history. The recurrent state augments rather than replaces the repository prior, allowing the model to update its repository knowledge incrementally as changes occur. This makes the method suitable for active development settings where repository-specific knowledge becomes stale after commits.

The training objective is standard language modeling on repository-aware code completion tasks, specifically assertion completion. The central idea is to compress repository knowledge into generated adapter parameters rather than paying for long retrieved context at inference time.

### 🧪 Experiments
The evaluation is conducted on RepoPeftBench, a new benchmark of 604 Python repositories, including 512 in-distribution repositories and a 92-repository temporal holdout collected after the data cutoff for out-of-distribution testing. RepoPeftBench has two tracks. The static track contains about 40K training tasks and 12K test tasks derived from single repository snapshots. The evolution track contains about 215K training tasks and 87K test tasks derived from commit history. Tasks are assertion-completion problems, where repository non-test code is available as context for adaptation and assertions in the test portion must be completed.

The paper evaluates both in-repo and cross-repo settings on the in-distribution repositories, along with temporal OOD testing on the holdout set. Baselines include repository-context injection methods such as RAG and dependency-resolved context, parameter-efficient fine-tuning methods such as shared LoRA and per-repository LoRA, and strengthened hypernetwork baselines inspired by Text2LoRA/Doc2LoRA but adapted to consume whole-repository embeddings and generate LoRAs for the same target modules.

On the static track, Code2LoRA-Static achieves 63.8% exact match in the cross-repo setting and 66.2% exact match in the in-repo setting. These results surpass context-injection methods and match the per-repository LoRA upper bound despite avoiding per-repository training. The paper reports overall gains up to +9.9 percentage points on the static track relative to the strongest baselines.

On the evolution track, static snapshot adaptation becomes stale when tasks are derived from commits. Code2LoRA-Evo achieves 60.3% cross-repo exact match, improving by +5.2 percentage points over a single shared LoRA baseline. The paper’s main insight is that repository knowledge can be injected into model parameters through generated adapters, avoiding the token and retrieval costs of repository-context prompting. This is especially relevant for software engineering workflows where repositories are large and repeatedly queried. The generated LoRA is also shared across all transformer layers, which improves efficiency but may limit expressiveness compared with fully layer-specific adaptation. The hypernetwork is still large, and while inference is cheap, training such a generator remains substantial. ## 📄 [Context-as-AI-Service: Surfacing Cross-File Dependency Chains for LLM-Generated Developer Documentation](https://arxiv.org/abs/2606.04397v2)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Ameya Gawde, Vyzantinos Repantis, Harshvardhan Singh, Lucy Moys
📅 **Published:** 2026-06-03  🏷️ **Categories:** cs.SE, cs.IR

### 💡 TL;DR
This paper presents Context-as-AI-Service (CAIS), a retrieval layer that helps LLM agents trace non-obvious cross-file dependency chains when generating or reviewing developer documentation. In two documentation case studies on a production SDK, CAIS uncovered additional factual and completeness issues missed by a baseline agent with standard repository tools, while also reducing runtime and token usage.

### 🏆 Key Contributions
- Introduces CAIS, an agent-callable retrieval architecture that indexes source code, API references, tests, examples, and upstream documentation using combined lexical and semantic search.
- Defines and motivates the cross-file documentation problem, where documentation appears locally plausible but is incorrect or incomplete once dependent behavior across files is traced.
- Demonstrates through two production-SDK case studies that CAIS surfaces documentation and tutorial issues missed by a baseline workflow using only ordinary repository tools.
- Provides evidence-oriented analysis of the dependency chains behind retained findings, showing how retrieval helps agents discover semantically indirect implementation details.

### ⚙️ Methodology
CAIS is a four-stage pipeline consisting of ingestion, storage/indexing, retrieval, and review. The ingestion layer normalizes multiple source types—SDK source code, API references, tests, examples, and upstream third-party documentation—into document records with metadata such as source type, file path, and timestamps. The storage/indexing layer preserves raw documents and creates enriched searchable forms using BM25 for lexical indexing and DRAMA embeddings for dense semantic retrieval. At query time, an LLM agent interacts with CAIS through tool calls, issuing behavior-oriented or symbol-oriented queries. CAIS fuses BM25 and dense retrieval results using reciprocal rank fusion and returns ranked snippets with document IDs and metadata. The agent then inspects the retrieved files as needed; retrieval is used to guide exploration rather than replace code reading. An optional lightweight review layer can label documentation claims as consistent, contradicted, or incomplete based on retrieved evidence. The evaluation follows a retrospective case-study design comparing a baseline LLM agent with standard repository tools—file reads, keyword search, and symbol navigation—against the same agent augmented with CAIS. Findings are retained only if they expose a meaningful baseline gap, rely on broader cross-file context, and materially affect documentation correctness, executability, or implementation understanding.

### 🧪 Experiments
The evaluation uses two case studies on a proprietary production SDK of roughly 200 source files, using Claude Sonnet 4.6 as the agent. In Case Study 1, the task was to review and improve API reference comments in a core source file of about 900 lines. The baseline agent found 5 undocumented public methods and added comments, but did not identify incorrect existing comments. The CAIS-enabled agent produced the same 5 missing-documentation fixes and additionally surfaced 4 issues missed by baseline: 2 cross-file factual errors and 2 incomplete or underspecified comments. These involved tracing evidence from registry/query components, state-management documentation, implementation code, tests, and filter logic. In Case Study 2, the task was to validate an LLM-generated tutorial. CAIS surfaced 1 executable bug, 1 API-usage improvement, and 2 missing prerequisites that the baseline pipeline did not catch. Across both studies, the critical findings required following non-obvious dependencies through utility files, framework internals, usage examples, tests, and component-creation logic. Over five runs per condition, adding CAIS reduced wall-clock time by 22% to 34% and also lowered input-token usage relative to the baseline.

### 🌱 Insights & Implications
The paper’s central insight is that documentation failures often stem less from lack of context window and more from failure to identify which cross-file dependencies deserve inspection. CAIS helps by surfacing candidate evidence spanning multiple artifact types, making semantically indirect dependencies easier for the agent to notice. This is especially relevant for documentation validation, where locally plausible claims can still be globally wrong. The study is qualitative and retrospective rather than a controlled benchmark, so generalizability is limited by the proprietary SDK, small number of tasks, and manual retention of findings. The paper also does not compare against other advanced retrieval or agent-planning strategies, so the unique contribution of hybrid retrieval versus broader orchestration remains open. Still, the results suggest that retrieval layers specialized for software repositories may be valuable not only for documentation, but also for tasks requiring evidence gathering across code, tests, examples, and external docs.

### 🎯 Why This Paper
This paper is highly relevant to software engineering and maintenance because it addresses correctness-preserving documentation workflows over evolving codebases, a practical maintenance challenge. It is also relevant to security, vulnerability detection, and automated repair research because the core problem—surfacing non-obvious cross-file dependency chains—is closely related to how bugs, vulnerabilities, and repair constraints are discovered across distributed program logic. CAIS’s hybrid retrieval design could inspire retrieval-augmented agents for vulnerability triage, patch validation, and program repair, where missing a hidden dependency chain often leads to incorrect analyses or ineffective fixes.
