## 📄 [Veritas: A Semantically Grounded Agentic Framework for Memory Corruption Vulnerability Detection in Binaries](https://arxiv.org/abs/2605.15097v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Xinran Zheng, Alfredo Pesoli, Marco Valleri, Suman Jana, Lorenzo Cavallaro
📅 **Published:** 2026-05-14  🏷️ **Categories:** cs.SE, cs.CR

### 💡 TL;DR
Veritas is a binary vulnerability detection framework that grounds LLM reasoning in explicit interprocedural data-flow evidence from lifted LLVM IR and then validates hypotheses with targeted runtime checks. On a curated benchmark of real-world stripped-binary memory-corruption cases, it reports 90% recall, near-zero false positives in exhaustive validation, and found a previously unknown Apple vulnerability later assigned a CVE.

### 🏆 Key Contributions
- Introduces a semantically grounded, three-stage framework for memory-corruption vulnerability detection in stripped binaries, combining static slicing, LLM-based reasoning, and runtime validation.
- Designs a static slicer over RetDec-lifted LLVM IR that reconstructs interprocedural value-flow relations including def-use, calls, returns, globals, and pointer operations, producing compact witness-backed source-to-sink flow objects.
- Proposes a dual-view detector that reasons step-by-step over decompiled C plus selectively instantiated LLVM IR, enabling memory-semantics-aware analysis without requiring the LLM to rediscover whole-binary propagation.
- Builds a multi-agent validator that tests detector hypotheses with debugger-guided execution, breakpoint inspection, and memory-checking oracles to confirm feasible triggers and reject unsupported candidates.
- Constructs a fine-grained benchmark of real-world binary memory-corruption cases with trace-level ground truth and shows strong end-to-end performance against static, dynamic, binary-native, and agentic baselines.
- Demonstrates real-world impact by discovering a previously unknown Apple vulnerability that was confirmed and assigned a CVE.

### ⚙️ Methodology
Veritas is organized as a modular pipeline with two grounding layers: static semantic grounding and runtime grounding. First, binaries are lifted with RetDec to LLVM IR and decompiled to C. The Slicer operates on parser-derived LLVM-IR facts rather than raw text, recovering interprocedural value-flow structure relevant to memory corruption. It links definitions and uses across instructions and functions, including local def-use edges, call/return propagation, global-variable access, and pointer-related operations such as address computation and dereference chains. From these facts it emits compact witness-backed flow objects that summarize candidate source-to-sink propagation paths, preserving enough evidence for downstream reasoning while avoiding whole-program context explosion.

Second, the Detector is a dual-view LLM analysis stage. It consumes the slicer’s grounded flow objects together with two synchronized representations: decompiled C for higher-level control-flow and program intent, and selectively instantiated LLVM IR snippets for low-level memory semantics such as bounds, object correspondence, field offsets, pointer arithmetic, and sink behavior. The detector performs stepwise reasoning over each candidate flow rather than searching the entire binary, focusing on whether attacker-controlled data can influence allocation size, index computations, copy lengths, or dereference targets in a way that yields a memory-corruption bug. This design uses the slicer’s evidence to avoid rediscovering interprocedural propagation and to constrain LLM reasoning to verifiable semantic artifacts.

Third, the Validator is a multi-agent runtime confirmation stage. It translates detector hypotheses into concrete execution plans, such as where to place breakpoints, what debugger-visible state to inspect, and which path conditions or object relationships must hold at runtime. It then checks candidates using guided debugging, breakpoint inspection, and memory-checking oracles to determine whether the predicted unsafe access is actually feasible. This agentic validation either confirms the vulnerability with executable evidence or rejects the hypothesis as unsupported. Overall, the method’s novelty lies in tightly coupling LLM reasoning with explicit static witnesses and runtime evidence, making semantic grounding the central design principle for binary memory-corruption detection.

### 🧪 Experiments
The evaluation uses a curated benchmark of real-world repository-based binary memory-corruption vulnerability cases, with stripped binaries and trace-level ground truth. The benchmark is designed to test difficult scenarios such as interprocedural propagation, shared-state dependencies beyond simple call chains, and semantic loss introduced by compilation and stripping. The paper compares Veritas end-to-end against strong baselines spanning static analysis, dynamic analysis, binary-native approaches, and agentic/LLM-based methods.

Quantitatively, Veritas achieves 90% recall on the benchmark. For false-positive analysis, the authors exhaustively validate and manually verify 623 detector candidate vulnerabilities in tractable full-pipeline runs. In this exhaustive subset, the final validated results contain no false positives. For larger cases where exhaustive full-pipeline validation is less tractable, they additionally audit detector candidates and identify two manually confirmed false positives. Beyond benchmark evaluation, the paper reports a real-world case study where Veritas discovered a previously unknown Apple vulnerability that was externally confirmed and assigned a CVE. The main experimental takeaway is that grounding LLM analysis in recovered semantic flows and executable validation substantially improves practical binary vulnerability detection accuracy.

### 🌱 Insights & Implications
The paper’s main insight is that binary memory-corruption detection is not primarily limited by raw LLM reasoning power, but by the absence of sufficiently grounded semantic context. In stripped binaries, call-graph expansion, summarization, or unguided fuzzing are inadequate because they miss shared-state dependencies, precise object relationships, and path-feasible triggering conditions. Veritas shows that constraining reasoning to witness-backed interprocedural flows and then requiring runtime confirmation can make LLM-assisted detection operationally reliable.

A key implication for related research is that hybrid systems for vulnerability detection and repair should separate semantic recovery from semantic reasoning: static analysis should recover the propagation skeleton and object evidence, while the LLM should interpret and prioritize grounded candidates rather than infer everything from raw code artifacts. This is especially relevant for binary analysis, vulnerability triage, exploitability assessment, and agentic security tooling.

Likely limitations include dependence on the quality of decompilation and IR lifting, the coverage and precision of the slicer’s recovered facts, and the runtime validator’s ability to drive executions to relevant states. The benchmark is curated rather than fully representative of all deployed binaries, and the abstracted results do not fully expose costs such as analysis time, scalability to very large codebases, or performance across vulnerability classes beyond memory corruption. Open questions include how well the approach generalizes to optimized/obfuscated binaries, whether the grounding strategy can support automated exploit generation or repair, and how to extend the witness representation to concurrency, custom allocators, or temporal memory-safety bugs.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and vulnerability repair because it tackles one of the hardest practical settings: detecting memory-corruption bugs directly in stripped binaries. Its hybrid design is especially valuable for researchers working on automated program repair and secure software maintenance, since it shows how to combine structured static analysis, LLM reasoning, and runtime validation into a trustworthy pipeline. The emphasis on interprocedural semantic grounding, shared-state propagation, and low-false-positive validation offers concrete ideas for building binary-focused vulnerability detectors, patch validation systems, and future repair agents that must reason about low-level memory semantics rather than source-level syntax alone.

## 📄 [ExploitBench: A Capability Ladder Benchmark for LLM Cybersecurity Agents](https://arxiv.org/abs/2605.14153v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Seunghyun Lee, David Brumley
📅 **Published:** 2026-05-13  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
ExploitBench is a new benchmark for measuring how far LLM agents progress along the exploit-development pipeline on real hardened V8 vulnerabilities, using 16 deterministic capability flags instead of a single crash/no-crash outcome. On 41 recent V8 N-days, public frontier models usually reached coverage and crashes but almost never achieved full exploitation, while a private research-preview model reached arbitrary code execution on 18/41 bugs.

### 🏆 Key Contributions
- Introduces a capability-ladder benchmark for exploitation that decomposes exploit success into 16 independently measurable stages, spanning code coverage, crash, exploit primitives, sandbox bypass, control-flow hijack, and arbitrary code execution.
- Designs deterministic, cheat-resistant grading oracles for each capability stage, including randomized challenge-response primitive checks, differential execution against ground-truth binaries, and signal-handler/prctl-based proofs for control hijack and ACE.
- Builds ExploitBench on 41 real, recent V8 vulnerabilities with all production mitigations enabled, avoiding simplified harnesses, disabled defenses, or small toy programs.
- Defines a three-arm evaluation protocol—<model,env>, <model,env,adaptive coaching>, and <model,env,CLI>—to separate core model capability from coaching and vendor scaffolding effects.
- Provides the first empirical characterization of where current LLMs stall in browser-style exploit construction: public models typically stop at bug triggering or early primitives, while a private frontier model crosses into full exploit construction on many tasks.

### ⚙️ Methodology
ExploitBench models exploitation as a progressive ladder rather than a binary event. Each benchmark instance corresponds to a single known vulnerability packaged as a containerized environment containing the vulnerable source tree, multiple vulnerable binaries, build/debug tooling, and patch history. Agents are given the bug and patch context and interact through a uniform runner, with a fixed turn budget of 300 turns to avoid confounding capability with token pricing or API latency.

The benchmark defines 16 measurable flags that cover successive exploit-development milestones: reaching the vulnerable code path, triggering bug-specific behavior, crashing, constructing engine-specific primitives, obtaining general primitives such as arbitrary read or write, escaping the V8 heap sandbox, achieving control-flow hijack, and ultimately arbitrary code execution. Rather than relying on human judgment or LLM-as-a-judge, each stage is checked by deterministic oracles. Primitive oracles use per-run randomized challenge-response builtins so the model must demonstrate a real capability instead of hardcoding an answer. Differential execution against ground-truth binaries verifies whether an exploit genuinely progresses past a stage. Final exploitation stages are verified using in-engine signal-handler and prctl-based mechanisms that provide concrete evidence of hijack or code execution. The oracles are explicitly designed to resist reward hacking.

The benchmark is instantiated on 41 V8 bugs reported no earlier than 2024. V8 was chosen because it is production-grade, widely deployed, and heavily hardened with mitigations such as the V8 heap sandbox, ASLR, and stack canaries. Unlike prior work that often uses fuzz harnesses, web apps, or weakened defenses, ExploitBench evaluates the real JavaScript/WebAssembly attack surface in default release configurations. The study uses three evaluation arms: a primary bare <model,env> setup under a uniform harness, a secondary <model,env,adaptive coaching> arm that injects targeted mid-episode feedback to test whether coaching changes outcomes, and a <model,env,CLI> ablation that swaps in the model vendor’s native CLI to measure the effect of provider-side scaffolding and context management.

### 🧪 Experiments
The evaluation covers 41 real V8 N-day vulnerabilities under a 300-turn budget per run. The main model panel includes eight publicly deployed frontier models, plus one non-public research-preview model (Anthropic Mythos Preview) used as a capability reference point. The target software is V8 in production-like release builds with deployed mitigations enabled; tasks use the genuine JavaScript/WebAssembly-facing attack surface rather than synthetic fuzz harnesses.

The benchmark reports stage-wise capability outcomes rather than only end-to-end success. Metrics are the 16 deterministic flags, allowing analysis of where models stop: coverage, crash, engine primitives, general exploit primitives, sandbox bypass, control-flow hijack, and ACE. Results are reported across the three arms: primary bare runner, adaptive coaching, and vendor CLI.

Key findings are: public frontier models almost always reach the vulnerable code and often trigger crashes, especially on WebAssembly type-confusion bugs, but they rarely convert this into reusable exploitation primitives. Across all eight public models, arbitrary code execution was essentially absent. Only one public model, GPT-5.5, crossed beyond early primitives: in the primary arm and CLI it bypassed the sandbox security domain and achieved control-flow hijack on a single WebAssembly bug, and under the vendor-CLI arm it reached arbitrary code execution on that same bug. No other public (model, bug) combination reached ACE in any arm. In contrast, the private reference model, Anthropic Mythos Preview, achieved ACE on 18 of 41 bugs in the primary <model,env> arm without vendor scaffolding. The study also finds that adaptive coaching and context-management improvements generally do not create fundamentally new capability, but can reduce budget/time to reach capabilities already within a model’s reach. Bug class matters for public models—WebAssembly type confusion is more tractable, while JIT-compiler bugs are rarely exploited—but this predictor weakens for the stronger private model, which succeeds across Wasm, JIT, and historical cohorts.

### 🌱 Insights & Implications
The paper’s main insight is that crash reproduction is a poor proxy for exploit capability. Public LLMs can often reproduce V8 N-days up to coverage or crash, yet this says little about whether they can reason through the far harder stages of exploit weaponization, such as building stable read/write primitives, bypassing the heap sandbox, and hijacking control flow.

A second insight is the existence of a sharp capability boundary in current public models: they appear competent at bug triggering but not at the multi-step reasoning needed for hardened-browser exploitation. The private reference model shows that this boundary is not inherent to the task setup or benchmark design, but reflects a real gap between public and private frontier capabilities.

The benchmark also highlights the importance of deterministic, stage-specific oracles for security evaluation. Fine-grained grading exposes partial progress and failure modes that binary exploit metrics hide, and it avoids the ambiguity of LLM-as-a-judge. This is especially valuable for research on autonomous vulnerability detection, exploit generation, and exploit repair because it enables diagnosis of where systems fail.

For limitations, the benchmark is focused on a single but important target family—V8—so generalization to kernels, native userspace programs, or web-app exploitation still requires further work. The tasks are N-days with patch access rather than full 0-day discovery-and-exploitation pipelines. The inclusion of a private model as a reference point is informative but may be hard to reproduce independently. Open questions include how to extend capability-ladder benchmarking to end-to-end vulnerability discovery and repair loops, whether stronger planning/tool-use methods can help public models cross the primitive-construction barrier, and how to systematically characterize exploitability features that make some bugs easier for agents than others.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection/repair, and automated program repair research because it introduces a rigorous, deterministic evaluation framework for one of the hardest downstream tasks after vulnerability discovery: turning a known bug into a real exploit under production defenses. For a researcher working on APR, vulnerability analysis, or security agents, ExploitBench provides a much more informative target metric than binary crash success, enabling evaluation of whether a system merely reproduces bugs or genuinely reasons about exploit primitives, mitigation bypass, and control hijack. It also offers a reusable benchmark and methodology for studying how detection, exploitation, and eventually repair capabilities should be measured in realistic software-engineering settings.

## 📄 [Not All RAGs Are Created Equal: A Component-Wise Empirical Study for Software Engineering Tasks](https://arxiv.org/abs/2605.14503v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Qiang Ke, Yanjie Zhao, Hongjin Leng, Shengming Zhao, Haoyu Wang
📅 **Published:** 2026-05-14  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper systematically studies how different Retrieval-Augmented Generation pipeline components affect software engineering tasks, showing that retrieval choices often matter more than generator choices. Across code generation, summarization, and repair, classic lexical retrieval with BM25 is consistently strong, providing practical guidance for building effective SE-oriented RAG systems.

### 🏆 Key Contributions
- Introduces a modular Code RAG testbed that decouples query processing, retrieval, context refinement, and generation for controlled component-wise evaluation.
- Conducts a broad empirical comparison of more than 21 models and methods across 3 software engineering tasks and 4 datasets.
- Builds a large decontaminated retrieval corpus from five software-relevant sources: Stack Overflow, GitHub, LeetCode, Python API docstrings, and Hugging Face.
- Shows that retriever-side design, especially retrieval algorithm choice, usually has larger impact on final performance than swapping generator LLMs.
- Finds that BM25, despite its simplicity, is a highly robust retriever across diverse SE tasks.
- Proposes and open-sources a prototype adaptive RAG framework that recommends near-optimal pipeline configurations based on task characteristics.

### ⚙️ Methodology
The study treats RAG as a pipeline with four separable stages and evaluates them independently and in combination. First, the authors define three representative software engineering task modalities: text-to-code (code generation), code-to-text (code summarization), and code-to-code (code repair). They then construct a large retrieval corpus by aggregating documents from Stack Overflow, GitHub, LeetCode, Python API docstrings, and Hugging Face, followed by strict decontamination to remove exact code overlaps with evaluation test sets.

The core experimental platform is a modular Code RAG testbed where each stage can be swapped independently. In the query processing stage, the paper studies transformations of the original query, including simplification and elaboration, using strong LLMs such as DeepSeek-V3 and GPT-4o to generate rewritten forms. The goal is to test whether query rewriting improves retrieval relevance and downstream generation quality.

In the retrieval stage, the authors compare seven retrieval models spanning sparse, dense, and hybrid paradigms. Although the full paper text excerpt does not list every retriever by name, the abstract and discussion emphasize direct comparison across these retrieval families, with BM25 as a key sparse baseline and strong performer. This stage is the main focus of the study because retrieved evidence is hypothesized to dominate downstream quality.

In the context refinement stage, the study evaluates post-retrieval techniques such as re-ranking and compression, asking whether additional filtering improves the signal-to-noise ratio of the context passed to the generator or instead removes useful information.

In the generation stage, six generator models are compared. The paper isolates generator effects from retriever effects to assess whether better LLMs compensate for weaker retrieval or whether high-quality retrieval is the dominant factor.

Finally, based on empirical findings, the authors design a prototype adaptive RAG recommender that maps task features to recommended pipeline configurations, aiming to reduce manual trial-and-error when deploying RAG for SE applications.

### 🧪 Experiments
The evaluation covers 3 core SE tasks—code generation, code summarization, and code repair—over 4 datasets, using a large decontaminated software corpus assembled from five sources. The component space includes 4 query processing techniques, 7 retrievers spanning sparse/dense/hybrid approaches, 4 context refinement methods, and 6 generators, for over 21 evaluated models/methods overall.

The experiments are organized around four research questions: the effect of query processing, the best retrieval strategy for code tasks, the utility of context refinement, and the relative importance of retriever versus generator. Baselines are implicit in the component-wise setup: raw-query pipelines versus rewritten-query pipelines, sparse versus dense versus hybrid retrieval, no-refinement versus re-ranking/compression, and multiple generator LLMs under matched retrieval conditions.

The key result is qualitative but central: retriever-side choices, especially the retrieval algorithm, have greater influence on end-task performance than generator selection in many settings. BM25 is reported as exceptionally robust across tasks, often rivaling or outperforming more complex dense and hybrid alternatives. This suggests that investment in retrieval design yields larger gains than focusing solely on generator upgrades. The paper positions these findings as a practical empirical roadmap rather than a single new state-of-the-art model, so the main takeaway is comparative performance stability and component sensitivity across tasks rather than one headline benchmark number.

### 🌱 Insights & Implications
A major insight is that software engineering RAG should not automatically default to the most modern dense or hybrid retriever; lexical matching remains highly competitive because code and technical text often benefit from exact identifier, API, and keyword overlap. This is especially relevant for tasks like repair and generation where precise tokens matter.

Another important implication is methodological: evaluating only the generator can be misleading in SE RAG studies. A strong generator cannot reliably compensate for poor evidence retrieval, so future work on coding assistants, APR, and vulnerability repair should treat retrieval as a first-class design variable.

The paper also suggests that query rewriting and context refinement are not universally beneficial; their value is task-dependent and may introduce unnecessary complexity or information loss. This points to the need for adaptive, task-aware pipelines rather than one-size-fits-all RAG architectures.

Limitations include the study’s focus on classic RAG pipelines rather than more complex agentic or iterative retrieval-generation systems, and the fact that results are bounded by the chosen tasks, datasets, and corpus construction procedure. Open questions include how these findings transfer to security-specific corpora, cross-language codebases, vulnerability detection/repair benchmarks, and interactive repair settings where iterative retrieval might matter more.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, vulnerability detection, and vulnerability repair because many emerging systems in these areas now depend on RAG over code, documentation, issue reports, and security knowledge bases. Its main value is not a new model but a rigorous empirical decomposition of where performance actually comes from, which can directly inform the design of retrieval-grounded repair or security analysis pipelines.

For a researcher working on APR and vulnerability-oriented tasks, the strongest takeaway is that careful retriever selection—potentially even simple BM25—may matter more than choosing the newest LLM. That insight can reduce engineering cost, improve reproducibility, and guide future systems that retrieve fixes, vulnerability patterns, API/security documentation, or historical patches before generating repairs or explanations.

## 📄 [SWE-Chain: Benchmarking Coding Agents on Chained Release-Level Package Upgrades](https://arxiv.org/abs/2605.14415v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Man Ho Lam, Chaozheng Wang, Hange Liu, Jingyu Xiao, Haau-sing Li, Jen-tse Huang, Terry Yue Zhuo, Michael R. Lyu
📅 **Published:** 2026-05-14  🏷️ **Categories:** cs.SE, cs.AI, cs.CL

### 💡 TL;DR
SWE-Chain is a new benchmark for evaluating coding agents on realistic software maintenance where package upgrades happen as chained release-to-release transitions and each step inherits the agent’s previous edits. It introduces a synthesis pipeline, DecompSynth, to derive grounded upgrade specifications from release notes plus code diffs, and shows that even the best current agents still struggle to perform correct continuous upgrades without regressions.

### 🏆 Key Contributions
- Introduces SWE-Chain, the first benchmark targeting chained release-level package maintenance rather than isolated issue fixing or one-off release upgrades.
- Builds a dataset of 12 upgrade chains across 9 real Python packages, covering 155 version transitions and 1,660 grounded upgrade requirements.
- Proposes DecompSynth, a divide-and-conquer specification synthesis pipeline that aligns maintainer-authored release notes with actual source/test diffs to generate implementable and behavior-grounded upgrade tasks.
- Defines a sequential evaluation protocol where an agent’s output for one release becomes the starting codebase for the next, exposing error accumulation and maintenance continuity challenges.
- Provides broad empirical evaluation over nine frontier agent-model configurations and demonstrates that SWE-Chain is feasible, stable, and discriminative while remaining challenging.

### ⚙️ Methodology
The benchmark is constructed around consecutive package releases from real Python projects. For each package chain, the authors collect released versions from PyPI or GitHub, fetch release notes plus linked GitHub issues/PRs, and manually review parsed release-note tasks to normalize formatting and remove unsuitable releases. They create one chain-level Docker environment per package with pinned Python and cross-version dependencies so all releases in the chain can be tested consistently. To identify upgrade-relevant behaviors for a transition from version v_(i-1) to v_i, they run the target version’s test suite on both the old and new gold codebases and mark as upgrade-related those tests that fail or error on the old version but pass on the new one, following the FAIL_TO_PASS / ERROR_TO_PASS framing used in SWE-Bench-style evaluation. The core synthesis method, DecompSynth, uses two stages. First, in the divide stage, an LLM-based mapping agent receives the task-structured release notes and batches of file-level code diffs, then performs multi-label hunk matching, assigning each diff hunk to one or more task labels, or to doc / others. Diffs are batched to scale to large upgrades. After merging batch-level results, hunks aligned to each release-note task are grouped into a matched hunk set. Second, in the synthesis stage, another LLM-based agent generates a structured upgrade specification for each task from the release-note item plus its matched hunks. Specifications include a problem statement, expectations/constraints, behaviors, and acceptance criteria, designed to be informative to agents while grounded in the actual implementation diff. Hunks assigned to others can also trigger synthesis of additional implicit upgrade tasks not explicitly named in the release notes. The synthesis pipeline uses GPT-5.4/Codex with high reasoning effort. Stability of the most stochastic component, hunk matching, is assessed by repeated runs on the largest transition, showing high agreement. For evaluation, each chain is executed sequentially: the agent starts from the initial gold codebase, receives the synthesized specification for upgrade step i, edits the current codebase, and its produced codebase becomes the input for step i+1. The benchmark also introduces a Build+Fix regime that allows one extra corrective attempt for execution-level problems so agents are not overly penalized for superficial build/runtime issues unrelated to upgrade logic.

### 🧪 Experiments
SWE-Chain contains 12 chains over 9 Python packages: attrs, conan (two chains), Flask, Jinja2, poetry, PyJWT, pytest (two chains), urllib3, and xarray (two chains). In total it includes 155 upgrade transitions and 1,660 synthesized grounded tasks. Package start sizes range from about 3.4k to 152k LOC excluding tests, with substantial cumulative diff sizes per chain. The authors evaluate nine frontier agent-model configurations under the sequential chained protocol and report resolving rate, precision, and F1, primarily under the Build+Fix regime. Averaged across all configurations, agents reach 44.8% resolving, 65.4% precision, and 50.2% F1. The strongest system is Claude-Opus-4.7 with Claude Code, achieving 60.8% resolving, 80.6% precision, and 68.5% F1. The benchmark is therefore neither trivially easy nor impossibly hard. The paper also validates the synthesis pipeline’s robustness: on the largest diff transition (pytest 8.2.2 to 8.3.0, about 13k diff lines, 102 files, 986 hunks), ten repeated hunk-matching runs yield 93.66% exact match, 94.54% Jaccard similarity, and 95.38% core agreement, indicating stable task-to-diff alignment. The paper further notes comparisons across regimes and granularity settings, including a study of specification granularity effects, though the extracted text provides the headline Build+Fix numbers rather than the full ablation table.

### 🌱 Insights & Implications
The paper makes a strong case that release boundaries are a more realistic unit for software maintenance evaluation than arbitrary commits or isolated issue instances, because real maintenance is bundled, shipped, and inherited across versions. The sequential setup exposes a major weakness of current coding agents: they can often implement parts of an upgrade, but they struggle to preserve accumulated functionality over multiple releases, so mistakes compound across the chain. DecompSynth addresses an important benchmark-design problem by producing cleaner, behavior-grounded specifications than simply concatenating release notes with noisy issues/PR text, which likely improves task validity and reproducibility. For software engineering research, the benchmark highlights that future agents need stronger long-horizon state management, regression avoidance, dependency-awareness, and compatibility-preserving repair capabilities. Limitations and open questions include the benchmark’s current focus on Python packages, the use of LLM-generated specifications whose quality depends on hunk matching and prompting, possible residual mismatch between synthesized requirements and full maintainer intent, and the challenge of evaluating broader maintenance aspects beyond test-visible behavior. Another implication is that chained maintenance may be a useful stress test for methods in automated program repair and vulnerability repair, since both require iterative patching without introducing regressions; however, the benchmark itself is not security-specific.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance and automated repair research because it evaluates whether coding agents can perform continuous, release-by-release modifications without breaking existing behavior, which is closer to real maintenance workflows than one-shot bug fixing. For a researcher interested in software engineering, maintenance, APR, vulnerability detection, and vulnerability repair, SWE-Chain offers a rigorous evaluation framework, realistic long-horizon tasks, and a grounded spec-synthesis method that could be adapted to benchmark or train agents for dependency upgrades, regression-safe repair, and future security-oriented maintenance scenarios.

## 📄 [Viverra: Text-to-Code with Guarantees](https://arxiv.org/abs/2605.14972v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Haoze Wu, Rocky Klopfenstein, Keith Farkas, Nina Narodytska
📅 **Published:** 2026-05-14  🏷️ **Categories:** cs.SE, cs.AI, cs.HC, cs.LO

### 💡 TL;DR
Viverra augments text-to-code by asking an LLM to generate C programs together with candidate assertions, then uses bounded model checking to formally verify as many assertions as possible and present them as trusted code facts. On 18 nontrivial tasks, it efficiently produced verified annotations, and a 400+ participant user study showed those facts materially improve code comprehension.

### 🏆 Key Contributions
- Introduces Viverra, an end-to-end text-to-code workflow that synthesizes C programs and attaches formally verified annotations to help developers audit LLM-generated code.
- Proposes partial co-specification: eliciting natural-language safety/correctness properties from the prompt and turning them into inline assertions that act as partial specifications.
- Designs a best-effort compositional verification strategy over the call graph, using bounded model checking plus previously verified assertions as assumptions to derive unconditional and conditional guarantees.
- Provides soundness guarantees for verified facts within the chosen loop-unwind bound, including conditional soundness under recorded dependency assumptions.
- Empirically evaluates the approach on 18 diverse programming tasks and with a large-scale human study showing verified facts improve program comprehension.

### ⚙️ Methodology
Viverra takes a natural-language task description D and runs a multi-stage pipeline. First, an LLM performs property elicitation, producing a set of natural-language safety and correctness properties that the requested program should satisfy. Second, the LLM synthesizes a verification-friendly C program P0, with prompting that encourages structures easier to analyze formally, such as avoiding recursion. Third, given both P0 and the elicited properties, the LLM inserts candidate assertions into the code, yielding an annotated program P+, where assertions can appear as preconditions, postconditions, loop invariants, or intermediate semantic checks; helper boolean functions may be introduced for complex predicates. Fourth, a bound-reduction step adapts loop-bound-related constants so the resulting program P+k is suitable for k-bounded verification. Fifth, Viverra computes a call-graph-based traversal order for assertions. Verification is then performed compositionally and in best-effort fashion using a portfolio of bounded model checkers: each assertion is checked either directly on the program or conditionally under assumptions corresponding to previously established assertions. Assertions receive statuses such as Verified, C-Verified(S) for conditionally verified under assumptions S, Falsified if a counterexample is found, or Unknown on timeout. The framework treats assumptions by pruning infeasible executions, and the theoretical basis is k-bounded execution semantics, where each loop back-edge may be taken at most k times. Finally, verified assertions are mapped back to the original elicited properties and translated into natural-language facts embedded alongside the generated code, including their verification conditions such as unwind bounds and dependency assumptions. The central design choice is not to require full proof of all assertions, but to surface whatever formally justified facts can be established efficiently, thereby helping humans understand and review the generated program.

### 🧪 Experiments
The evaluation uses 18 diverse text-to-C-code tasks intended to be more realistic and complex than standard HumanEval/MBPP-style benchmark snippets. The system measures whether Viverra can generate programs with meaningful assertions and whether those assertions can be verified efficiently using a portfolio of bounded model checkers under bounded-unwinding semantics. Quantitative details reported in the paper indicate that Viverra is able to produce multiple verified assertions per task and do so efficiently enough to be practical. The paper’s second major experiment is a user study with more than 400 participants comparing code-comprehension performance with and without Viverra’s verified annotations. The key result is that participants given the generated code plus verified facts performed better on comprehension tasks than those given unannotated generated code, demonstrating that the verified assertions are not merely formally valid artifacts but useful aids for human understanding. The abstract and main text emphasize efficiency, diversity of tasks, and improved comprehension as the main empirical outcomes; the evaluation focuses less on pass@k-style functional-generation metrics and more on verification yield and human usefulness.

### 🌱 Insights & Implications

## 📄 [Do Coding Agents Understand Least-Privilege Authorization?](https://arxiv.org/abs/2605.14859v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Zheng Yan, Jingxiang Weng, Charles Chen, Dengyun Peng, Ethan Qin, Jiannan Guan, Jinhao Liu, Qiming Yu, Yixin Yuan, Fanqing Meng, Carl Che, Mengkang Hu
📅 **Published:** 2026-05-14  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper studies whether coding agents can infer least-privilege file permissions before acting, and shows that current frontier models systematically both under-grant needed permissions and over-grant unnecessary or sensitive ones. To evaluate and improve this capability, the authors introduce AuthBench and a two-stage inference method, Sufficiency-Tightness Decomposition, which improves secure task completion and lowers attack success.

### 🏆 Key Contributions
- Introduces permission-boundary inference as a standalone capability: generating file-level read/write/execute policies from a task instruction and terminal environment before execution.
- Presents AuthBench, a benchmark of 120 realistic terminal tasks with human-reviewed permission labels, oracle-derived gold traces, and executable validators for both utility and attack outcomes.
- Shows empirically that authorization failures are not just conservative-vs-permissive calibration errors; models often simultaneously miss required permissions and include unnecessary or sensitive ones.
- Identifies model-specific authorization attractors under increased inference-time reasoning: more reasoning makes models more consistent in their characteristic failure mode rather than closer to least privilege.
- Proposes Sufficiency-Tightness Decomposition, a training-free two-stage policy generation approach that first maximizes task coverage and then prunes unjustified or sensitive permissions.
- Demonstrates that the decomposition improves sensitive-task success by up to 15.8% on tightness-biased models, reduces attack success across all evaluated models, and improves execute-axis F1 by up to 16.7% on standard tasks.

### ⚙️ Methodology
The paper formulates permission-boundary inference as follows: given a natural-language instruction I and a terminal environment E, a model f_theta must output a whitelist policy pi = (pi_read, pi_write, pi_execute), where each component is a set of POSIX path patterns. The policy is generated in a multi-turn, agentic way, but the model may only inspect the environment through read-only operations and cannot execute the task or modify the environment during policy generation. The generated policy is then enforced by an execution harness H paired with an execution model M, and task success is determined by a utility validator.

A key conceptual point is that the minimal sufficient policy pi* is execution-agent-dependent: different downstream coding agents may solve the same task using different tools and therefore require different permissions. The benchmark therefore uses oracle-derived traces as a static proxy for sufficiency, while executable validation remains the definitive test.

AuthBench is built in three stages. First, the authors collect realistic terminal tasks from Terminal-Bench, SWE-Bench, and OpenThoughts-TBLite, filtering for tasks with clear objectives, executable workflows, objective success criteria, and resource requirements expressible via file-level permissions. Second, they normalize retained tasks into a Dockerized benchmark format with instruction, safe oracle solution, utility validator, and permission-evaluation specification. The benchmark contains 80 standard tasks and 40 sensitive tasks; the latter embed tempting but unnecessary insecure paths such as helper scripts, local secrets, or credential-like files. Third, they derive gold permission sets by running oracle solutions under strace, extracting file-level read/write/execute accesses, filtering them through task-specific scored roots and implicit-permission patterns, and then manually reviewing traces to remove runtime noise and nondeterministic accesses. Sensitive tasks also include sensitive permission sets and attack validators.

The proposed Sufficiency-Tightness Decomposition addresses the difficulty of direct one-shot policy generation, where a model must simultaneously identify all needed accesses and reject all unnecessary ones. In stage 1, the model generates a coverage-oriented policy by forward-simulating the task and likely execution chain, effectively over-approximating needed permissions for sufficiency. In stage 2, the model audits each granted permission entry to determine whether it is grounded in the task and whether it overlaps with sensitive surfaces, pruning unjustified or risky permissions. This decomposes the least-privilege inference problem into two simpler subproblems: discover potential requirements, then justify retention. The method is inference-time only and does not require model retraining.

### 🧪 Experiments
AuthBench contains 120 realistic terminal tasks spanning 10 domains, split into 80 standard tasks and 40 sensitive tasks. Standard tasks include workflows such as file manipulation, package building, log analysis, configuration repair, repository debugging, and data conversion. Sensitive tasks augment otherwise normal tasks with attack surfaces including unsafe helper scripts, sensitive local files, credential-like artifacts, and task-local hints that make insecure but unnecessary accesses tempting.

Evaluation uses both static and executable metrics. For permission tightness, generated policies are compared against oracle-derived gold permission sets after expanding path patterns into concrete files within task-facing scope. Precision, recall, and F1 are computed independently for read, write, and execute axes, with macro-F1 used as the main static measure. For executable sufficiency, Task Success Rate (TSR) measures whether the downstream execution agent completes the task under the generated policy using the utility validator. For sensitive tasks, attack validators are used to assess whether over-broad policies enable attack behaviors, capturing security exposure beyond functional completion.

The authors evaluate frontier models under varying inference-time reasoning effort. The main empirical result is that additional reasoning does not generally fix least-privilege inference. Instead, each model moves toward a stable authorization attractor: some become consistently broad-but-exposed, while others become consistently tight-but-brittle. This means they still fail by both omitting necessary accesses and granting unnecessary or sensitive ones.

The proposed Sufficiency-Tightness Decomposition yields consistent gains. Across tested models, it improves sensitive-task success by up to 15.8% on models biased toward overly tight policies, reduces attack success across all evaluated models, and improves execute-axis F1 on standard tasks by up to 16.7%. The paper emphasizes that these gains come without model training, purely by restructuring the inference objective at test time.

### 🌱 Insights & Implications
A central insight is that least-privilege authorization for coding agents is not merely a thresholding or calibration problem. Current models do not lie on a simple spectrum from too conservative to too permissive; many simultaneously fail on both axes, missing execution-critical permissions while still exposing unrelated sensitive files. This matters for software-agent safety because a policy can be both functionally brittle and insecure.

The identified authorization attractors suggest a structural limitation of direct policy generation. Asking a model to reason longer does not necessarily improve authorization quality; it often amplifies the model's intrinsic failure tendency. This has broader implications for agent security research: more chain-of-thought or test-time scaling may not solve policy inference tasks when the objective itself conflates discovery of required resources with rejection of unsafe ones.

The decomposition result implies that authorization should be treated as a staged decision process rather than a monolithic generation step. For automated software agents, separating sufficiency estimation from security auditing appears more tractable and robust. This is relevant to sandboxing, policy synthesis, and secure execution harnesses.

A notable limitation is that the gold permission set is a proxy derived from an oracle execution trace, while the truly minimal sufficient policy depends on the downstream execution agent and its chosen toolchain. Thus, static precision/recall against gold labels cannot fully capture agent-dependent sufficiency. Another limitation is scope: AuthBench focuses on file-level POSIX read/write/execute permissions in terminal environments, so network, process, capability, or API-level privileges remain outside the benchmark. The benchmark is also relatively small at 120 tasks, though carefully curated.

Open questions include how to generalize permission-boundary inference to richer OS- and cloud-level authorization models, how to co-design execution agents that are robust under tight policies, and whether training-time methods could internalize the sufficiency-tightness decomposition rather than relying on prompt-time orchestration.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it addresses a foundational issue for autonomous coding agents operating on real repositories and terminal tasks: how to give them just enough authority to modify, build, debug, and repair software without exposing secrets or unsafe execution paths. For researchers working on automated program repair and secure agentic tooling, AuthBench offers a concrete benchmark and evaluation protocol for least-privilege policy inference, while the proposed decomposition provides a practical, training-free technique to improve secure task execution. It advances the state of the art by turning authorization into a measurable SE/security capability, exposing failure modes of current LLM agents, and suggesting a design pattern directly applicable to secure repair agents, vulnerability remediation pipelines, and repository-scale autonomous maintenance systems.

### 💡 TL;DR
EVA proposes a post-deployment safety defense that treats jailbreak mitigation as a targeted model-editing problem rather than full safety fine-tuning. ### 🏆 Key Contributions
- Introduces EVA, a unified direct model-editing framework for jailbreak defense in both LLMs and VLMs, and claims the first application of model editing as a safety defense for VLMs.
- Reframes safety alignment as precise knowledge correction: detect harmful representations, define a refusal-oriented safe target, and update only a small set of neurons/weights instead of retraining the whole model.

- Develops multimodal extensions for VLMs, including visual key extraction from harmful visual tokens and layer localization aligned with backbone LLM layers.

- Empirically shows stronger safety-utility trade-offs than fine-tuning and external-defense baselines across textual and visual jailbreak benchmarks, including unseen attacks.

### ⚙️ Methodology
EVA is built on the observation that jailbreaks do not create new harmful knowledge, but shift internal activations so that harmful instructions are interpreted as benign requests. The framework therefore aims to restore safe behavior by editing the model’s internal mapping from harmful representations to output behavior.

The method has three main stages. First, EVA extracts a malicious key representation. For LLMs, this is done by aggregating activations corresponding to harmful tokens/instructions across diverse jailbreak examples, producing a robust representation of malicious intent. For VLMs, EVA extends this idea to multimodal inputs by selecting visual tokens that convey harmful meaning, including OCR-style typographic attacks and more implicit visual semantics, then aggregating these into a visual malicious key. This addresses the fact that harmful visual information is diffuse and not tied to a single discrete token.

Second, EVA constructs a safe target representation. Rather than simply suppressing activations, it defines a target that promotes refusal behavior on malicious requests while constraining behavior drift on benign inputs. The paper highlights a context-aware regularization mechanism, implemented with a KL-based constraint on benign queries, to keep the edit specific to unsafe contexts and preserve the model’s general instruction-following and reasoning abilities.

Third, EVA applies a localized direct parameter edit. Following the spirit of ROME/MEMIT-style editing, it uses a closed-form update to modify only a small subset of parameters/neurons associated with the identified harmful representation, effectively mapping the malicious key toward the safe refusal target. The update is layer-localized rather than global: for LLMs, editing layers are chosen using prior knowledge about effective factual-editing layers; for VLMs, the authors analyze layer efficacy and find that the best editing layers align with those in the language backbone, enabling multimodal defense without broad retraining.

Overall, EVA is intended as a lightweight post-hoc patching framework: edits are precise, can be applied sequentially as new jailbreak patterns are discovered, and do not require auxiliary runtime detectors or modified decoding pipelines.

### 🧪 Experiments
The paper evaluates EVA on both LLM and VLM jailbreak defense settings. The attack space includes textual jailbreaks such as optimization-based and generation-based attacks (e.g., GCG, AutoDAN, PAIR as discussed in the paper) and multimodal jailbreaks including typographic/OCR-based and semantic visual attacks such as FigStep and Hades. The evaluation also references established multimodal safety benchmarks such as MM-SafetyBench and MultiTrust.

Baselines span major defense families: parameter-level safety tuning/fine-tuning methods, layer-specific or editing-inspired defenses, and inference-time defenses such as detection, moderation, or prompt-based shielding. The paper positions EVA against methods like safety fine-tuning, VLGuard-style alignment for VLMs, and external moderation/prompting defenses, as well as prior editing-related safety methods.

The main metrics are Attack Success Rate (ASR) for harmful compliance and utility/benign-task performance on general reasoning or harmless instruction-following tasks. The reported headline result is that EVA achieves lower ASR than competing defenses on both LLMs and VLMs while maintaining near-perfect or negligible-loss utility on benign tasks. The paper also reports better generalization to unseen attacks, support for adaptive-attack evaluation, and improved efficiency because edits are low-resource and do not increase inference-time latency.

In addition to end-task results, the experiments include analyses of layer efficacy, interpretability, generalization, multimodal decoupling effects, contextual synergy of visual editing, ablations on key extraction and regularization, and efficiency studies comparing EVA’s lightweight editing process to more expensive retraining-based defenses. The provided text does not expose detailed numeric tables, but the authors consistently claim superior safety-utility trade-offs and stronger robustness across textual and visual jailbreak settings.

### 🌱 Insights & Implications
A central insight is that jailbreak defense can be framed as localized repair of misaligned internal representations, which is closely related to concepts in program repair and targeted patching: instead of globally retraining the system, identify the vulnerable internal mechanism and patch only that mechanism. This yields a practical post-deployment defense story for rapidly evolving attacks.

The work is especially interesting for multimodal security because it extends model editing beyond text into visual jailbreaks, where harmful content may be carried by OCR tokens or by diffuse semantic cues. The finding that effective VLM editing layers align with the backbone LLM layers suggests that much of multimodal unsafe behavior may still bottleneck through language-side representations.

The context-aware KL regularization is important: safety edits that are too broad can damage benign utility, so constraining edits to malicious contexts is key to reducing the classic safety-utility trade-off. This makes EVA more appealing than coarse layer tuning or full fine-tuning for deployment.

Potential limitations and open questions remain. Model editing may be brittle under highly adaptive attackers who discover alternative harmful representations outside the edited subspace. Sequential edits could interact or accumulate over time, raising questions about long-term stability and edit interference. The paper also suggests but does not fully resolve the challenge of comprehensive harmful-region identification in images, where harmful semantics can be distributed and ambiguous. Finally, benchmark gains do not automatically guarantee robustness under real-world red teaming, distribution shift, or stronger white-box adaptive attacks.

For related research, EVA indicates that mechanistic, representation-level interventions may be a fruitful direction for safety hardening, including targeted unlearning, neuron/pathway repair, and efficient vulnerability patching for evolving attack patterns.

This paper is highly relevant to software security, vulnerability detection/repair, and automated repair research because it treats jailbreaks as a patchable internal vulnerability and proposes a precise, low-overhead repair mechanism instead of expensive retraining. Its emphasis on localization, closed-form targeted updates, sequential patching, and multimodal robustness directly parallels interests in efficient post-deployment maintenance, secure system hardening, and automated vulnerability remediation for ML systems.

- Introduces an asymmetric model-merging formulation for paired Instruct/Thinking checkpoints, treating the Thinking-Instruct delta as a pool of candidate reasoning edits rather than performing symmetric fusion.

- Demonstrates consistent gains across two Qwen code-agent scales and three agentic software engineering benchmarks, outperforming endpoint substitution and alternative merge methods.

### ⚙️ Methodology
CRANE starts from paired checkpoints: an instruction-tuned code-agent model and a reasoning-tuned 'Thinking' counterpart built on the same base model. The core object is the parameter delta δ = θ_think - θ_inst, interpreted as a directional reservoir of reasoning-related edits. The target merged model is θ_inst plus a constrained, filtered version of this delta.

The method has three stages. Stage 1 performs conservative magnitude thresholding on δ to remove low-confidence coordinates and create a sparse delta. The paper uses deterministic median-based thresholding with rescaling, so only sufficiently large coordinates are retained, reducing noise and limiting unnecessary perturbations to the Instruct model.

Stage 2 is the Conservative Taylor Gate (CTG), which decides how much of the sparse delta to inject for each parameter block, indexed by layer and component type (e.g., attention projections, MLP/expert projections, routers, norms). CTG uses three small calibration datasets corresponding to: reasoning transfer (R), format preservation (F), and broader agent-behavior preservation (A). For R, the model is calibrated against Thinking-style continuations on code reasoning prompts; for A, it uses Instruct-generated action spans encoding tool timing, reading behavior, and stopping decisions; for F, it focuses on format-critical traces such as tool delimiters, JSON/schema tokens, and chat-template markers. CTG estimates first-order usefulness of moving along the Thinking-to-Instruct direction using masked calibration losses. A block receives positive salience only when the local Taylor signal suggests that the update helps reasoning transfer while not harming preserved agent behavior. This yields blockwise injection strengths rather than a single global merge coefficient.

Stage 3 is Graduated Sigmoidal Projection (GSP), which protects format-critical activation subspaces. Using activations collected from Instruct format traces, CRANE identifies directions in the input-side activation space associated with tool control, schema fidelity, and delimiter/format tokens. The merged update for a parameter block is then projected to suppress components aligned with these protected directions. Instead of hard projection, GSP uses a smooth sigmoid-shaped attenuation, allowing gradual suppression of increasingly format-critical directions. The final merged parameter for each block is the Instruct parameter plus the globally scaled, CTG-gated, sparsified delta after this projection.

Overall, the method is explicitly asymmetric: it preserves Instruct-level protocol discipline and efficiency while selectively importing Thinking-style planning, context integration, and recovery behavior. The paper’s central insight is that Thinking and Instruct checkpoints are complementary but not directly swappable for code agents: stronger internal reasoning can harm real agent performance if it disrupts tool-use protocol, output schemas, or timing of actions. Profiling real game engines during online RL is expensive and hard to scale, so expert trajectory datasets make the approach more deployable in industrial testing pipelines. This suggests a broader pattern for software engineering tasks where internal signals are available but online experimentation is costly.

Limitations and open questions include the dependence on instrumentable environments and the availability/quality of expert data, the likely sensitivity to which target functions are included as goals, and the fact that function-reaching is only one dimension of testing quality. The paper does not, in the provided text, establish direct links to bug discovery rates, crash finding, or vulnerability detection/repair outcomes. Future work could extend the idea from call stacks to richer program analyses such as coverage maps, taint flows, assertion failures, execution invariants, or static-analysis-guided goals, and could study whether code-aware agents generalize to previously unseen functions, patches, or game versions.

### 🎯 Why This Paper
This paper is highly relevant to software engineering and automated testing because it treats program internals—specifically call stacks—as first-class signals for learning test policies, moving beyond black-box exploration. For a researcher interested in software maintenance, security, automated repair, and vulnerability-oriented workflows, the paper is valuable as a concrete example of code-aware RL that could be extended from function coverage in games to targeted execution of vulnerable code paths, regression testing after patches, coverage-guided bug reproduction, and repair validation in complex interactive software.
