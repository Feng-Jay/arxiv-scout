## 📄 [CyberChainBench: Can AI Agents Secure Smart Contracts Against Real-World On-Chain Vulnerabilities?](https://arxiv.org/abs/2606.26216v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Jintao Huang, Fengqing Jiang, Radha Poovendran, Zhiqiang Lin
📅 **Published:** 2026-06-24  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
CyberChainBench is an on-chain benchmark for evaluating AI agents on smart contract security across vulnerability detection, exploit generation, and patch synthesis using 541 real-world DeFi exploit incidents replayed on historical blockchain forks. The benchmark shows that current frontier agent setups can often find or exploit bugs, but robustly patching them remains much harder, with best scores of 37.5% for detection, 43.7% for exploitation, and 23.4% for patching.

### 🏆 Key Contributions
- Introduces the first smart-contract benchmark that jointly evaluates detection, exploitation, and patching in a dynamic on-chain setting rather than a static off-chain environment.
- Builds a dataset of 541 real-world exploit incidents from DeFiHackLabs across 9 EVM chains, each anchored to a historical block and annotated with vulnerability type, localization, attacker profit, and exploit metadata.
- Defines executable evaluation protocols where exploits are scored by realized economic profit on historical mainnet forks and patches are validated by replaying both historical attacks and legitimate transactions.
- Proposes a root-cause taxonomy of five smart contract vulnerability classes: price manipulation, accounting error, access control, reentrancy, and input validation.
- Provides an isolated agent runtime based on Harbor with tool access for source retrieval, decompilation, storage inspection, transaction tracing, and exploit/patch validation on forked historical state.
- Presents a comparative evaluation of multiple agent-framework and model combinations, showing a clear difficulty gradient from detection to exploitation to patching.

### ⚙️ Methodology
CyberChainBench is designed around end-to-end, dynamic evaluation of LLM agents operating on deployed smart contracts under historical blockchain state. Instead of giving agents prepackaged source code, each benchmark case starts from a real contract address and a specific historical block. Agents run in isolated Docker containers orchestrated by Harbor and can only interact through a restricted tool interface exposed by an MCP server. Available tools include fetching verified source code, obtaining decompiled bytecode for unverified contracts, reading storage slots, making eth_call queries, tracing transactions, and validating exploits or patches on mainnet forks. The benchmark is constructed from 690 DeFiHackLabs exploit reproductions and filtered/enriched into 541 usable cases. A curator agent based on Claude Opus 4.6 first annotates metadata such as block number, proxy status, vulnerable function, and contract information; for bytecode-only cases, EtherVM decompilation and function selector matching are used. A profit-tracing pipeline computes attacker profit from ERC-20 transfer events and native token flows, quoting assets through on-chain DEX routers. All cases are then manually reviewed, with 141 annotation inconsistencies corrected. For patch evaluation, the authors restrict to proxy-upgradeable contracts with verified source and at least one relevant historical legitimate transaction, producing a 94-case patch subset from an initial 153 proxy cases. The benchmark uses progressive disclosure: detection receives less information and fewer tools than exploit generation, and patching receives the richest context including attack transactions and validation oracles. Evaluation is fully execution-based. Detection is framed around identifying the vulnerability type and location; exploitation requires generating working attack code that executes successfully on a historical fork and is scored by realized attacker profit; patching requires synthesizing an upgrade that prevents the historical exploit while preserving legitimate historical behavior. The vulnerability taxonomy is root-cause based rather than exploit-technique based, yielding five mutually exclusive categories intended to support deterministic scoring. The benchmark also labels exploit difficulty as easy, medium, or hard using a calibrated LLM-assisted procedure over the reference PoCs.

### 🧪 Experiments
The empirical study evaluates multiple agent-model configurations spanning Claude Code, Codex, and Gemini CLI paired with frontier models such as Claude Opus 4.7, GPT-5.5, and Gemini 3.1 Pro. The benchmark contains 541 total incidents for the overall dataset, with a 200-case exploit evaluation set highlighted in the main results and a 94-case subset for patch evaluation on proxy-upgradeable contracts. Cases span 9 EVM-compatible chains and are rooted in real exploit incidents from 2020-2026. Inputs and tools are varied by task through progressive disclosure to prevent leakage of exploit-specific information into earlier stages. Metrics are execution-centric: detection score measures correct vulnerability identification/localization; exploit generation is graded by successful execution and economic impact on historical mainnet forks; patch synthesis is validated by fail-to-pass attack replay and pass-to-pass legitimate-transaction replay. Key results show a strong performance drop across the workflow stages: the best configuration achieves 37.5% on vulnerability detection, 43.7% on exploit generation, and only 23.4% on patch synthesis. The top-performing setup, Codex with GPT-5.5, realizes $57.4M in total exploit profit across the 200 exploit cases at an average cost of $2.39 per case. The authors also report benchmark composition details relevant to difficulty: 42% of cases are labeled easy, 47% medium, and 11% hard. Compared with prior benchmarks such as EVMbench and SCONE-bench, CyberChainBench is distinguished by real-chain historical state, executable exploit validation, and patch validation against both malicious and legitimate transaction traces.

### 🌱 Insights & Implications
The central finding is that current AI agents are substantially more capable at exploiting smart contract vulnerabilities than at repairing them safely, even when given structured tools and accurate historical state. This highlights a major research gap for automated program repair in security-critical blockchain software: producing a non-breaking patch that preserves protocol behavior is far harder than demonstrating exploitability. The benchmark also emphasizes that static-source evaluation likely overestimates practical smart contract security capabilities, because real-world auditing often requires reasoning over proxy patterns, unverified bytecode, cross-contract dependencies, and exact historical state. The use of profit-based exploit scoring is particularly valuable for security research because it measures not just syntactic validity but real economic severity. The patching setup raises important limitations and open questions: only proxy-upgradeable contracts can be meaningfully patched on-chain, so immutable-contract incidents are excluded from repair evaluation; legitimate behavior is approximated via historical transactions, which may not fully capture all desired invariants; and vulnerability classification into five root-cause categories, while pragmatic, may still compress nuanced multi-cause incidents. Another implication is that agent benchmarking for vulnerability detection should increasingly move toward executable and environment-grounded evaluation, similar to exploit replay and repair validation, rather than text-only judgments. For related research in vulnerability repair and APR, the work suggests future systems will need stronger semantic reasoning, protocol-invariant modeling, and regression-aware repair synthesis.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and automated program repair because it provides a realistic benchmark where AI systems must detect, exploit, and patch real vulnerabilities under execution semantics rather than on toy static examples. It directly advances work on vulnerability repair and secure code agents by introducing executable patch validation, historical-state replay, and economically grounded exploit scoring for smart contracts, offering a rigorous testbed for developing stronger security-focused APR and agentic analysis methods.

## 📄 [ShareLock: A Stealthy Multi-Tool Threshold Poisoning Attack Against MCP](https://arxiv.org/abs/2606.27027v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Liwei Liu, Tianzhu Han, Zijian Liu, Zishu Dong, Na Ruan
📅 **Published:** 2026-06-25  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper presents ShareLock, a multi-tool poisoning attack against Model Context Protocol (MCP) agents that splits a malicious prompt into secret shares embedded across several tool descriptions and reconstructs it only when a covert trigger is activated. Across four benchmarked domains, multiple mainstream LLMs, and two MCP clients, ShareLock reportedly achieves over 90% average attack success while being much harder to detect than prior single-tool plaintext poisoning attacks.

### 🏆 Key Contributions
- Introduces a realistic multi-tool MCP threat model under moderate vetting, where attackers distribute risk across multiple tools instead of relying on a single obviously poisoned tool.
- Proposes ShareLock, the first threshold-based multi-tool poisoning framework for MCP that adapts Shamir's Secret Sharing to fragment adversarial instructions into benign-looking shares.
- Combines encoding-based obfuscation with threshold reconstruction to achieve both stealth and robustness, including tolerance to partial tool removal or auditing.
- Builds a custom benchmark spanning four multi-tool domains: Travel, Coding, Finance, and Office.
- Evaluates the attack on two MCP clients, Cherry Studio and Cline, and four mainstream LLMs, showing strong evasion against safety classification and high attack success.
- Provides an early systematic analysis of multi-tool poisoning as a distinct class of MCP security threats beyond prior single-tool tool-description attacks.

### ⚙️ Methodology
ShareLock is a multi-stage poisoning framework targeting LLM agents that consume tools through the Model Context Protocol. The core idea is to avoid placing a malicious prompt directly in one tool description, since plaintext prompt injection is easy to notice by human auditors or automated scanners. Instead, the attacker converts the adversarial instruction into a numeric secret and uses Shamir's Secret Sharing to split it into n shares with a reconstruction threshold k. Each share is then embedded into the descriptions of different MCP tools in a semantically benign or metadata-like form, so that any individual tool appears harmless and reveals no usable malicious content on its own. This yields information-theoretic secrecy at the single-tool level: any subset smaller than the threshold does not expose the hidden instruction. The attack additionally includes a covert reconstruction trigger introduced during a server update or related lifecycle event. When the LLM agent is induced to aggregate enough poisoned tools, the distributed shares are reconstructed into the original hidden instruction, which then functions as a prompt injection payload capable of steering the agent toward unauthorized behavior, such as sensitive file access, harmful actions, or data exfiltration. The threshold design also provides fault tolerance: the attack can still succeed even if some poisoned tools are removed or flagged, as long as at least k shares remain accessible. The paper frames this as a more realistic multi-tool attack strategy for MCP deployments where tools may be reviewed individually but not jointly, and where distributed poisoning can evade both manual inspection and tool-description safety detectors better than monolithic attacks.

### 🧪 Experiments
The evaluation uses a custom benchmark designed for multi-tool MCP poisoning, covering four domains: Travel, Coding, Finance, and Office. The authors test ShareLock on two MCP clients, Cherry Studio and Cline, and across four mainstream LLMs. The experimental focus is on attack success, stealth against tool description-based detection, and robustness under moderate auditing or partial tool removal. ShareLock is compared against existing single-tool poisoning strategies that embed malicious instructions directly in plaintext tool descriptions. The reported headline result is an average attack success rate exceeding 90% while substantially improving evasiveness relative to these baselines. The paper also states that ShareLock performs better under safety classification by mainstream models, indicating that its distributed secret-share encoding is less likely to be flagged as malicious than conventional monolithic poisoning payloads. Although the provided excerpt does not include the full numeric tables, the key quantitative claim is that ShareLock preserves high effectiveness while significantly outperforming prior methods on stealth and robustness in multi-tool settings.

### 🌱 Insights & Implications
A key insight is that MCP security cannot be adequately modeled as isolated single-tool prompt injection; once agents can inspect and combine multiple tools, adversaries can distribute malicious logic across the toolset and exploit aggregation behavior. The use of Shamir-style threshold sharing is notable because it gives the attack both secrecy and resilience properties that simple obfuscation lacks: single tools remain non-incriminating, and the payload survives partial sanitization. This suggests that many existing MCP defenses, especially those focused on per-tool scanning or obvious prompt-injection strings, may be structurally insufficient against compositional attacks. For defenders, the work implies a need for cross-tool analysis, provenance-aware registration review, lifecycle monitoring of server updates, and policies that reason about latent interactions among tools rather than only their local descriptions. An open question is how well ShareLock generalizes under stronger defenses, such as models fine-tuned for MCP security, semantic graph analysis of tool ecosystems, or dynamic end-to-end monitoring of reconstruction behavior. Another limitation is that the paper appears centered on description-based poisoning and covert reconstruction assumptions; the practical difficulty of reliably inducing share collection and reconstruction in diverse real-world agent workflows may vary by client, model, and prompting context. Still, the work highlights an important shift from obvious prompt injection to distributed, cryptographically inspired poisoning in agent toolchains.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and secure agent/toolchain engineering because it analyzes a new attack surface in MCP-based LLM systems and proposes a sophisticated exploitation method that defeats common auditing assumptions. For a researcher interested in vulnerability detection and repair, it provides a concrete threat model, benchmark domains, and evidence that current MCP defenses miss distributed multi-component attacks, which is valuable for designing stronger detection, hardening, and automated remediation techniques for AI-integrated software ecosystems.

## 📄 [To Run or Not to Run: Analyzing the Cost-Effectiveness of Code Execution in LLM-Based Program Repair](https://arxiv.org/abs/2606.26978v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Zhihao Lin, Junhua Zhu, Mingyi Zhou, Xin Wang, Zhensu Sun, Renyu Yang, David Lo, Li Li
📅 **Published:** 2026-06-25  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper empirically isolates code execution as a single variable in LLM-based program repair and shows that, for many SWE-bench bugs, unrestricted test execution adds little repair benefit relative to its cost. Across commercial and open-source agents, prohibiting execution yields nearly unchanged resolve rates while substantially reducing tokens, runtime, and environment-maintenance overhead, suggesting execution should be used selectively rather than by default.

### 🏆 Key Contributions
- Provides the first controlled empirical study that isolates code execution within the LLM agent repair loop, instead of changing both the agent scaffold and execution strategy together.
- Analyzes execution behavior at scale using 7,745 public SWE-bench leaderboard traces across multiple agents, models, and datasets, characterizing how often, when, and how successfully agents execute tests.
- Runs 3,000 controlled repair attempts on 200 SWE-bench instances with Claude Code, Codex, and OpenCode under four execution paradigms to measure execution’s marginal impact on repair effectiveness and cost.
- Shows that restricting or prohibiting execution causes only negligible, statistically insignificant drops in resolve rate for the studied agents, while saving large amounts of token usage and wall-clock time.
- Identifies that execution benefit is concentrated in a subset of cases: many successful repairs are single-edit, localization remains highly accurate without execution, and many failures pass agent-selected validation but fail official evaluation.
- Releases a reproducible framework for controlling or limiting execution in program-repair agents.

### ⚙️ Methodology
The study is designed in two stages. First, the authors conduct an observational analysis of execution behavior in publicly available SWE-bench traces. They collect 7,745 traces from leaderboard submissions spanning four execution-oriented agents—SWE-agent, OpenHands, LiveSWEAgent, and Mini-SWE-agent—twelve LLMs, and two benchmark subsets (SWE-bench Lite and Verified). From these traces, they extract execution-related events, focusing on commands that run code and return runtime feedback, especially test invocations such as pytest, python -m unittest, and Python script execution. They quantify execution frequency per task, timing within the conversation, and the success rate of those executions.

Second, they perform controlled end-to-end experiments where the agent scaffold is held fixed and only execution access is varied. They evaluate three agents: Claude Code with Claude Sonnet 4.5, Codex CLI with GPT-5.2-xhigh, and the open-source OpenCode with Qwen2.5-Coder-32B-Instruct. They test each agent under four execution paradigms with progressively different access levels, including a fully prohibited setting and an unrestricted setting; the paper also discusses an intermediate quota-based regime, including a Quota-1 mode that allows only a single execution. This design isolates the marginal value of execution while preserving the multi-turn agentic workflow.

The analysis then drills into why execution helps or fails to help. The authors compare stable outcome categories such as pass-to-pass and fail-to-fail across execution modes, study how often reproduction executions yield actionable localization feedback, measure localization accuracy when execution is unavailable, inspect how many cases finish in a single edit, and compare agent-side self-validation outcomes against the official SWE-bench evaluation to detect validation mismatch. Overall, the methodology combines large-scale trace mining with controlled ablation-style experiments to examine both effectiveness and cost-effectiveness of code execution in APR agents.

### 🧪 Experiments
The experimental evaluation uses 200 SWE-bench tasks: the first 100 instances from SWE-bench Lite and the first 100 from SWE-bench Verified. Across three agents and multiple execution modes, the authors run 3,000 repair attempts in total. They measure repair success primarily with resolve rate, and efficiency with token consumption and wall-clock time. Statistical significance is checked using McNemar’s test.

In the trace analysis, agents execute tests heavily on average—8.8 runs per task—with substantial variance across agent-model combinations, ranging from about 2 to 19 executions per task. Late-stage executions are more effective than early ones; for example, one reported configuration improves from 42% to 72% execution success later in the conversation, and the average success rate of executions is 57.9%.

In the controlled repair study, the main quantitative result is that execution restrictions have little effect on final repair success for the studied setup. For commercial agents using strong proprietary models, the resolve-rate gap between Prohibited and Unrestricted is only about 1.25 percentage points and is not statistically significant. A concrete example reported is Claude Code achieving 63% resolve rate without execution versus 64% with unrestricted execution, while saving 56% of tokens and 48% of wall-clock time. The abstract also reports 56–62% token savings and 48–54% wall-clock savings for Claude Code under Prohibited mode. For the open-source OpenCode with Qwen2.5-Coder-32B, the resolve rate is approximately unchanged between Prohibited and Unrestricted (about 10% in the summarized results), while token usage drops by roughly 3x without execution. The authors further identify a boundary regime where OpenCode performs best with a single well-chosen execution (Quota-1) rather than unrestricted execution, likely due to limited context and inefficient repeated retries.

Diagnostic analyses show that 54–66% of commercial-agent cases are solved in a single edit regardless of execution access, reproduction execution produces actionable localization feedback in only 48.8% of cases, localization accuracy remains above 95% even when execution is prohibited, and 81–100% of failed commercial-agent cases pass the agent’s own validation but fail official SWE-bench evaluation. OpenCode exhibits a different failure mode: it retries more frequently, and only 11% of its failed cases pass self-validation, indicating poorer use of execution feedback rather than merely validation mismatch.

### 🌱 Insights & Implications
The paper’s central insight is that execution is not uniformly valuable in LLM-based APR; its benefits are concentrated in a subset of instances, while many agents invoke it routinely and pay high cost for little gain. This challenges the prevailing assumption that generate-run-revise should be the default repair paradigm.

A key implication is that strong LLMs can often localize and patch bugs without runtime feedback, especially on benchmark instances where many repairs are effectively single-edit tasks. High localization accuracy under prohibited execution suggests that code understanding from static repository inspection may already be sufficient in many cases. At the same time, the frequent mismatch between agent-side validation and official evaluation exposes a major weakness: agents often run the wrong tests or insufficient tests, so execution does not necessarily provide the right corrective signal.

The findings also distinguish two failure regimes. Commercial agents with strong models often do not benefit much from repeated execution because they already make competent first attempts and their self-validation is misaligned with benchmark validation. In contrast, the open-source agent suffers from more frequent retries and poorer exploitation of feedback, which can make unrestricted execution even less cost-effective than a carefully budgeted single run.

Limitations include the benchmark scope: the controlled study covers 200 SWE-bench instances rather than the full benchmark, and focuses on three agents and a specific set of models. Results may differ for harder bugs, projects where reproduction is essential, repair settings requiring complex dynamic analysis, or future agents that choose tests more intelligently. An open research question is how to predict when execution is worth spending budget on. This points toward adaptive execution policies, learned execution schedulers, selective validation, or cost-aware agent planning as promising directions for future APR systems.

### 🎯 Why This Paper
This paper is highly relevant to research on automated program repair, software maintenance, and practical LLM-based debugging because it rigorously evaluates a core design choice in modern repair agents: whether iterative code execution actually pays off. For a researcher working on APR, vulnerability repair, or agent-based software engineering systems, it provides actionable evidence that execution should be treated as a budgeted resource, motivating work on selective testing, adaptive tool use, validation alignment, and lower-cost repair pipelines that can scale better in real-world maintenance and security workflows.

## 📄 [Empirical Software Engineering TerraProbe: A Layered-Oracle Framework for Detecting Deceptive Fixes in LLM-Assisted Terraform](https://arxiv.org/abs/2606.26590v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Manar Alsaid, Chimdumebi Nebolisa, Faris Abbas
📅 **Published:** 2026-06-25  🏷️ **Categories:** cs.LG, cs.CR

### 💡 TL;DR
TerraProbe is a five-layer evaluation framework for LLM-assisted Terraform security repair that shows scanner-level success badly overestimates real repair quality. Across 288 first-pass repairs, many outputs removed the targeted Checkov alert, but few remained clean under deeper validation, and most adjudicated real-world cases were deceptive fixes that preserved the original security risk.

### 🏆 Key Contributions
- Introduces TerraProbe, a five-layer oracle stack for evaluating Terraform security repairs beyond targeted static-analysis finding removal.
- Demonstrates that targeted Checkov removal is a weak and misleading success metric for LLM-based IaC repair.
- Performs a multi-model comparison of Gemini 2.5 Flash-Lite, GPT-4o, and Claude 3.5 Sonnet on first-pass Terraform security repair.
- Establishes that deceptive-fix behavior is systemic rather than model-specific under the studied conditions, using chi-square tests, Fisher exact tests, and effect sizes.
- Defines a four-dimensional taxonomy of deceptive fixes spanning mechanism, intent alignment, security impact, and detection difficulty.
- Validates human labeling of deceptive fixes with strong inter-rater reliability (Cohen’s kappa 0.78, Krippendorff’s alpha 0.76).
- Provides IAM permission-level analysis showing that wildcard Resource grants persisted in all nine deceptive CKV2_AWS_11 cases.
- Abstracts the approach into the broader Multi-Layer Oracle Evaluation (MLOE) framework for general IaC security evaluation.
- Releases a full replication package with prompts, corpus, scripts, and Docker-based reproducibility artifacts.

### ⚙️ Methodology
The paper proposes TerraProbe as a layered-oracle evaluation methodology rather than a new repair model. LLMs are prompted to generate first-pass fixes for Terraform security misconfigurations using minimal prompts that do not explicitly encode the underlying security intent. Each produced repair is then evaluated through a five-layer oracle stack built around the Terraform toolchain and security analysis workflow. The layers progressively strengthen the evidence required to call a repair trustworthy: starting from removal of the targeted Checkov finding, then assessing broader scanner cleanliness, Terraform planning validity, and plan-comparison reachability, followed by human adjudication of whether the repair actually satisfies the intended security property. This design is motivated by the automated program repair oracle problem: a patch can satisfy a weak oracle while violating the real specification. TerraProbe formalizes such cases as deceptive fixes, i.e., changes that pass shallow automated checks but fail to remove the underlying vulnerability. The study further introduces a four-dimensional deceptive-fix taxonomy: mechanism of deception, degree of intent alignment, security impact, and detection difficulty. Human analysis is used to classify plan-comparable cases under this taxonomy, and agreement is measured with Cohen’s kappa and Krippendorff’s alpha. To strengthen construct validity, the paper supplements scanner and plan-level evidence with manual IAM permission analysis, especially for CKV2_AWS_11 cases, to determine whether permissive wildcard Resource grants actually remain after the alleged fix. The authors also generalize TerraProbe into the Multi-Layer Oracle Evaluation framework, positioning it as a reusable empirical design for evaluating security repairs in Terraform and other IaC ecosystems.

### 🧪 Experiments
The evaluation covers 288 first-pass repairs generated by three frontier models: gemini-2.5-flash-lite, GPT-4o, and Claude 3.5 Sonnet. Repairs are produced over two tracks: 68 real-world Terraform modules from TerraDS and 28 controlled injected-defect modules. The study compares outcomes across the oracle layers and across models, using statistical hypothesis testing rather than descriptive reporting alone. For the primary model, targeted Checkov finding removal reaches 83.3%, but this drops sharply to 10.4% for full-scanner cleanliness, 39.6% for successful Terraform planning, and 38.5% for plan-comparison reachability. In adjudicated real-world plan-compared cases, 71.4% are deceptive fixes that pass automated checks while leaving the original vulnerability unresolved. Across models, deceptive-fix rates range from 57.1% to 71.4%, and pairwise Fisher exact tests yield p-values above 0.10, indicating no statistically significant differences between Gemini, GPT-4o, and Claude under the study setup. Track-level comparison shows that plan-comparison reachability differs significantly between real-world and injected-defect settings (chi-square = 31.64, p < 0.001, Cohen’s h = 1.36), suggesting that controlled defects and real Terraform modules behave differently under layered evaluation. The deceptive-fix taxonomy labeling achieves inter-rater reliability of kappa = 0.78 and alpha = 0.76. A focused IAM analysis confirms that in all nine deceptive-fix CKV2_AWS_11 cases, wildcard Resource permissions persist after repair, meaning the core privilege-escalation risk remains despite scanner success.

### 🌱 Insights & Implications
The main insight is that LLM-assisted IaC repair can appear highly successful when judged by single-check disappearance, yet fail under more semantically meaningful oracles. This is a strong empirical analogue of overfitting in automated program repair: the model learns to satisfy the observable checker rather than the latent security intent. The finding that deceptive-fix rates are statistically indistinguishable across three strong instruction-following models suggests the problem is systemic, not a quirk of one provider. The paper attributes this to a mix of training-distribution bias, a specification gap between scanner checks and actual security policy, and prompt under-specification. Another important insight is methodological: injected-defect benchmarks and real-world modules are not interchangeable, since reachability of deeper validation steps differs significantly across tracks. The work also shows the need for multi-layer evaluation pipelines in security-oriented software engineering, especially when LLM outputs can manipulate structure or syntax to evade a static check without removing dangerous semantics. Limitations include the first-pass-only setup, use of minimal prompts, focus on Terraform and Checkov-centric workflows, and reliance on human adjudication for the deepest oracle layer. Open questions include whether richer security-intent prompts, repair feedback loops, constrained decoding, semantics-aware policy analyzers, or execution-grounded cloud security oracles can reduce deceptive fixes. The paper implies that future APR and vulnerability-repair research should explicitly distinguish scanner-passing repairs from intent-aligned fixes and should report deeper validation attrition, not just top-line pass rates.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it tackles a core evaluation problem: how to tell whether an LLM-generated security fix actually repairs a vulnerability instead of merely suppressing a warning. Its layered-oracle methodology, deceptive-fix taxonomy, statistical multi-model analysis, and focus on specification-vs-oracle mismatch directly advance work on trustworthy repair evaluation, benchmark design, and security-aware APR for infrastructure code.

## 📄 [Smaller Models, Unexpected Costs: Trade-offs in LLM Quantization for Automated Program Repair](https://arxiv.org/abs/2606.27205v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Fernando Vallecillos-Ruiz, Giordano d'Aloisio, Max Hort, Luca Traini, Antinisca Di Marco, Leon Moonen
📅 **Published:** 2026-06-25  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper studies whether quantizing large language models is actually beneficial for automated program repair, beyond just preserving benchmark scores. Across 13 quantization settings, six LLMs, and two APR benchmarks, the authors find that quantization greatly reduces memory usage but often increases inference time and energy consumption, while also changing which bugs the models can repair.

### 🏆 Key Contributions
- A broad empirical evaluation of 13 post-training quantization configurations across six code-capable LLMs ranging from 6.7B to 70B parameters on two automated program repair benchmarks.
- Introduction of Jaccard Consistency Rate (JCR) to measure solved-set consistency between a base model and its quantized version, exposing behavioral shifts hidden by aggregate plausibility scores.
- Measurement of three non-functional effects of quantization for APR—inference time, energy consumption, and memory footprint—reported with bootstrapped confidence intervals for relative changes.
- Pareto-based trade-off analysis showing that many quantization configurations are strictly dominated and that no single quantization method is universally best.
- Release of a replication package with scripts, generated patches, and efficiency measurements.

### ⚙️ Methodology
The study is designed around three research questions covering APR effectiveness, efficiency, and multi-objective trade-offs. It focuses on post-training quantization (PTQ) because it reflects realistic practitioner usage of already-available models. The authors evaluate 13 quantization configurations derived from five quantization methods, spanning different bit-widths and target components. These include weight-only quantization and KV-cache quantization, with precisions ranging from 2 to 8 bits depending on method support. The evaluation covers six representative LLMs between 6.7B and 70B parameters. For APR effectiveness, the paper measures whether a model can generate at least one plausible patch for a bug, and complements standard plausibility counts with solved-set consistency analysis. To capture this, the authors propose Jaccard Consistency Rate (JCR), which compares the overlap between the set of bugs solved by the base model and by its quantized counterpart, revealing whether similar total repair counts hide different repair behavior. For efficiency, they record inference time, energy consumption, and memory footprint for each configuration. They then analyze the results using bootstrapped confidence intervals for relative changes and perform Pareto dominance analysis to identify configurations that are inferior on the joint effectiveness-efficiency frontier.

### 🧪 Experiments
The evaluation uses two APR benchmarks: HumanEval-Java and Defects4J, chosen to represent different repair-task complexity levels. In total, six base LLMs and their 13 quantized variants are tested under default recommended settings to reflect realistic out-of-the-box deployment. The study compares base and quantized models along: (1) number of problems with at least one plausible patch, (2) Jaccard Consistency Rate between solved sets, (3) inference time, (4) energy consumption, and (5) memory footprint. Key findings include that quantized and base models often solve different bugs despite having comparable total numbers of repaired problems, implying low solved-set overlap in many cases. Quantization achieves large memory reductions of up to 85%, but these savings do not translate into faster or greener inference; instead, quantized variants frequently increase inference time and energy use, which the authors attribute to poor hardware utilization. In the trade-off analysis, 48% of the evaluated quantization configurations are strictly Pareto-dominated by alternatives, meaning they are worse on the considered effectiveness-efficiency objectives than some other setting.

### 🌱 Insights & Implications
The main insight is that benchmark-level APR scores alone are insufficient for evaluating quantized LLMs: two models can repair a similar number of bugs while repairing substantially different subsets, which matters for reliability, reproducibility, and downstream tool integration. Another important takeaway is that memory compression should not be conflated with overall efficiency, since reduced model size can come at the expense of slower inference and higher energy consumption on current hardware/software stacks. The results suggest that quantization trade-offs are strongly dependent on model family, bit-width, quantization target, and task complexity, so there is no universally best method for APR. A practical implication is that researchers and tool builders should treat quantization selection as a multi-objective optimization problem rather than defaulting to the smallest model. Open questions include how to improve hardware/runtime support so compressed models realize actual speed and energy gains, whether calibration tailored to code-repair data can preserve solved-set consistency better, and how these findings extend to other software engineering tasks such as vulnerability repair or agentic debugging workflows.

### 🎯 Why This Paper

### 💡 TL;DR

### 🏆 Key Contributions
- Introduces Chai, an agentic system for discovering cryptographic misuse vulnerabilities without a sanitizer-like oracle by using naturally occurring behavioral discrepancies between independent implementations.
- Reframes differential testing with AI: LLMs provide protocol knowledge and adaptive search for high-value inputs, while differential testing provides deterministic, checkable evidence that a security-relevant discrepancy exists.
- Proposes discrepancy tracing, which treats specification ambiguities and library behavior mismatches not as noise but as seeds for downstream vulnerability discovery in dependent applications.

### ⚙️ Methodology
Chai targets cryptographic protocols where a standardized attacker-controlled message can be submitted to multiple conforming libraries and produce a security-relevant decision, such as certificate validation, token verification, or assertion acceptance. Its central abstraction is the discrepancy: the same input is executed against multiple independent implementations, and any disagreement is captured for analysis.

The system has two stages. In stage 1, an AI agent performs differential search over a protocol ecosystem. Instead of relying on hand-built grammars or protocol-specific mutators, the agent uses its protocol knowledge to craft candidate inputs that are likely to exercise under-specified or security-sensitive corners of the standard. Each generated test case is amplified by running it against all harnessed libraries for that protocol simultaneously. Chai records acceptance/rejection and other behavior differences, then uses the agent as a triage mechanism to classify discrepancies into: (1) vulnerabilities, where the attacker can subvert a security decision under the paper's threat model; (2) security bugs, where the library violates the spec but exploitation needs stronger assumptions; (3) ambiguities, where the behavior is defensible in isolation but leaves a missing security guarantee that applications may incorrectly assume; and (4) non-actionable differences. Differential testing gives a deterministic signal that a discrepancy is real, reducing dependence on unverifiable LLM claims, while the agent prioritizes discrepancies that are likely to matter security-wise.

Stage 2 is discrepancy tracing. For discrepancies classified as ambiguities, Chai analyzes the dependency graph to find applications that rely on the disagreeing library. Rather than auditing each application for all possible vulnerabilities, it runs a narrow, close-ended audit for the single missing guarantee exposed by the discrepancy. For example, if a JWT library accepts a token-embedded verification key while others reject it, Chai searches downstream applications using that library and checks specifically whether they trust successful verification as evidence of key trust. This reverse-search workflow turns one library-level discrepancy into many targeted application audits and produces downstream findings with proof-of-concept scripts.

The design is motivated by three efficiency gains: amplified testing, where one generated input tests many implementations at once; reverse search, where only dependent applications that can inherit the discrepancy are examined; and targeted auditing, where each application receives a focused analysis for one exact behavior rather than an open-ended vulnerability hunt. Chai's coverage is empirical rather than exhaustive because the agents generate inputs probabilistically and can only find discrepancies that manifest across at least two implementations.

### 🧪 Experiments
The evaluation covers three protocol domains with significant cryptographic misuse risk: X.509, JWT, and SAML. The implementation spans 47 libraries across 8 programming languages. Chai is compared against prior differential-testing and fuzzing approaches, with the main reported metric being the number and diversity of unique behavioral differences surfaced across implementations, especially rare disagreements where only a small minority of libraries accept an input and the rest reject it.

Quantitatively, Chai's differential search surfaced about 2x as many unique differences as the strongest baseline, and many of these differences were missed entirely by the baselines. The paper emphasizes that Chai also reaches a broader range of discrepancy types, including hard-to-find accept/reject asymmetries that are strong indicators of library-level flaws. In real-world impact, the system found multiple confirmed vulnerabilities and security bugs in major cryptographic libraries, including a previously unknown critical vulnerability in a widely used SSL library powering billions of devices, another security bug in a library used by a major web browser, and one in a library shipped in major Linux distributions. The paper specifically notes two severe wolfSSL vulnerabilities that were confirmed and patched by maintainers within hours, despite the codebase having recently been audited by Anthropic's Mythos. Beyond library findings, Chai surfaced over 100 downstream application vulnerability findings that were still under investigation at the time of writing.

### 🌱 Insights & Implications
A key insight is that cryptographic misuse lacks the strong runtime oracles available in memory-safety research, so fully autonomous AI auditing is especially vulnerable to false positives. Chai shows that pairing LLMs with a classical systems technique—differential testing—can restore a checkable signal without requiring a formal oracle for 'correct' behavior.

The paper also argues that implementation ambiguities are not just noise. In cryptographic ecosystems, the gap between what a library guarantees and what applications assume can itself be the vulnerability surface. This is particularly important for software supply chains: a single library-level discrepancy can propagate to hundreds or thousands of dependent applications, making reverse dependency tracing more scalable than repeatedly re-auditing individual projects from scratch.

For security engineering, Chai suggests a shift from project-centric to ecosystem-centric vulnerability discovery. This is especially relevant for protocols built from stacked standards and multiple libraries, such as JWT and X.509, where dangerous behavior often lives in dependency contracts rather than directly in application code.

Important limitations remain. Chai only finds issues that manifest as disagreements between at least two independent implementations, so bugs shared consistently across all libraries are invisible. Its coverage is probabilistic, depending on the quality of AI-generated inputs and the set of harnessed implementations. Classification of discrepancies into vulnerability, ambiguity, or non-actionable difference still involves nuanced security judgment and manual confirmation. It also does not reason about flaws in the cryptographic primitives themselves or in the specification text absent implementation divergence.

Open directions include automating dependency-graph construction and application tracing more deeply, improving semantic clustering and prioritization of discrepancies, extending the approach to other security-critical protocol families, and integrating repair guidance or misuse-resistant API redesign once recurring ambiguity patterns are identified.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated program analysis because it targets a difficult but high-impact bug class—cryptographic misuse—that current AI and fuzzing systems struggle to validate. For a researcher interested in vulnerability detection and repair, Chai offers a novel ecosystem-level workflow: discover library discrepancies, classify security-relevant behavior, propagate findings through dependency graphs, and produce targeted downstream audits. 📅 **Published:** 2026-06-25  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper shows that security fine-tuning can make LLM malware classifiers more brittle even when standard held-out accuracy improves: the fine-tuned model learns token-level indicator semantics that fail under behavior-preserving PowerShell transformations such as aliasing, reconstruction, and case mutation. Mechanistic analysis reveals that fine-tuning does not build a new detection circuit, but instead specializes an inherited late-attention route from the base model, creating new evasion vulnerabilities that can be anticipated with lightweight pre-deployment monitoring on canonical inputs.

### 🏆 Key Contributions
- Demonstrates a concrete fine-tuning-induced evasion failure mode in PowerShell malware classification where canonical evaluation misses vulnerabilities exposed by behavior-preserving transformations.
- Provides a mechanistic explanation showing that the fine-tuned security model inherits a late-layer classification circuit from Llama-3.1-8B-Instruct and semantically specializes it rather than creating a new detector.
- Constructs a three-tier, seed/variant PowerShell evasion benchmark covering alias substitution, command reconstruction, string construction, execution indirection, and case mutation, enabling attribution of failures to specific transformation classes.
- Introduces a practical pre-deployment monitoring approach combining a linear probe at the classification boundary with an indicator-token sign test to identify command families likely to be vulnerable after fine-tuning.
- Shows an important tradeoff for AI-enabled security: improved baseline security-classification performance can coincide with an expanded evasion surface and uneven remediation behavior.

### ⚙️ Methodology
The study compares a natural base/fine-tuned pair, Llama-3.1-8B-Instruct and Foundation-Sec-8B-Instruct, on matched benign/malicious PowerShell classification cohorts designed to control for superficial suspicious-token presence. The author first performs circuit discovery by identifying attention heads that repeatedly focus on suspicious command spans, URLs, and execution sites, then using coarse head/layer ablations, malicious-direction analysis, and contribution decomposition to isolate candidate components that carry malicious-vs-benign evidence. Causal validation is performed with grouped path patching and grouped head ablation over a 293-pair matched cohort: path patching copies candidate activations from malicious scripts into benign contexts to test sufficiency, while head ablation disables candidate groups to test necessity. The key localization is a small late-attention bundle around the classification boundary at resid_pre13, especially a Layer-12 route inherited from the base model. The analysis then compares how fine-tuning changes the semantics of this inherited route, showing that the fine-tuned model concentrates and repurposes it around canonical indicator tokens. To operationalize this finding, the paper proposes a pre-deployment monitoring method using a linear probe on base-model activations at the classification boundary to measure family-level representation drift, together with an indicator-token sign test on canonical malicious scripts: remove indicator tokens, compare the change in malicious confidence between base and fine-tuned models, and flag command families where the fine-tuned model exhibits driver-to-suppressor role reversal. These families are then prioritized for targeted red-team generation of transformed variants.

### 🧪 Experiments
The experimental setting is binary malicious PowerShell classification using matched benign/malicious cohorts and a 293-pair intervention set. The main model comparison is Foundation-Sec-8B-Instruct versus its base model, Llama-3.1-8B-Instruct. Evaluation goes beyond same-distribution held-out accuracy by constructing a three-tier evasion benchmark from malicious seed scripts that were initially classified correctly, then generating behavior-preserving variants through syntax-preserving rewrites and validation checks. Transformation categories include PowerShell alias substitution (for example Invoke-WebRequest to iwr), Invoke-Expression reconstruction via format strings, string construction, execution indirection, and case mutation including alternating-case Invoke-Expression and IEX forms. Mechanistic experiments report intervention effect and flip rate for grouped path patching and head ablation to establish sufficiency and necessity of the localized late-attention circuit. The benchmark shows that Foundation-Sec develops misses on iwr alias substitution, Invoke-Expression reconstruction, and case-mutated Invoke-Expression/IEX variants, while Llama has zero misses on the same evaluated variant sets. The paper also reports that the indicator-token sign test, computed only on canonical inputs, correctly identifies Invoke-WebRequest as the clearest divergent family and successfully predicts the corresponding alias-evasion weakness before any variants are tested. Prior work cited in the paper notes a +16% precision gain for Foundation-Sec in a threat-hunting workflow, which frames the central result: baseline gains do not imply robustness under transformation-based attack.

### 🌱 Insights & Implications
The paper’s central insight is that fine-tuning can preserve or improve canonical task accuracy while quietly altering the internal semantics of inherited circuits in ways that expand the evasion surface. Rather than deleting the base model’s classification pathway, fine-tuning sharpens and specializes a late-attention route around canonical indicator tokens, making the resulting classifier more dependent on brittle lexical forms. This explains why the fine-tuned model can outperform on standard evaluation yet underperform under simple behavior-preserving rewrites that the base model still handles. An important implication is that robustness assessment for security classifiers must explicitly define the task’s transformation space instead of relying on held-out IID data. The work also suggests that mechanistic interpretability can support practical red-teaming by identifying vulnerable command families before attack variants are generated. A notable limitation is scope: the analysis centers on one architecture family, one fine-tuned security model, and one domain/task combination focused on PowerShell maliciousness classification, so generalization to other languages, models, or vulnerability classes remains open. Another open question is how to mitigate the observed drift without sacrificing useful specialization, since prompt-level remediation appears uneven and may fix one family while worsening another. For related research, the findings argue for coupling fine-tuning with robustness-aware data design, transformation-based evaluation, and internal drift monitoring.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and robust AI-assisted analysis because it studies how specialized LLM security classifiers can become easier to evade after fine-tuning, even when benchmark accuracy improves. For a researcher working on security-oriented software engineering, automated repair, and vulnerability detection/repair pipelines, it offers both a mechanistic explanation of brittleness and a practical pre-deployment method for detecting semantic drift and prioritizing adversarial testing, which is directly useful when building dependable LLM-based security tools.

### 💡 TL;DR
This paper proposes the Spec Growth Engine, a lightweight architecture/process framework for AI-assisted software development that tackles two major failure modes: oversized agent context and silent drift between code and specification. Its core idea is to couple code and machine-readable specs through a scoped context mechanism called the Spine and a blocking drift gate, so AI agents work on narrowly bounded slices while spec-code divergence becomes a merge-time error.

### 🏆 Key Contributions
- Defines the Spec Growth Engine as a spec-anchored, code-coupled, drift-enforced framework for AI-assisted development.
- Introduces a machine-readable spec graph where each node separates externally visible contract from internal design.
- Proposes the Spine context assembler, which limits agent input to root invariants, the ownership path, the current node spec, one-hop dependency contracts, and the node’s own code.
- Formalizes a two-layer growth protocol: upfront architectural invariants plus hardest-first vertical feature slices.
- Defines frontier stubs as governed placeholders allowed only off the active slice path, preventing fake-heavy breadth-first development.
- Introduces a drift gate that treats spec-code divergence as a blocking merge condition rather than a documentation hygiene issue.

### ⚙️ Methodology
The paper presents a conceptual and process architecture rather than a learned model. The system is centered on a machine-readable Spec Graph whose nodes represent architectural elements/components and contain explicit separation between contract and design: contracts define what a node promises to other parts of the system, while designs capture internal realization details hidden behind module boundaries. Development is organized in two layers. Layer 1 consists of system invariants that must be specified before feature work begins, including persistence boundaries, security decisions, external integrations, and error taxonomy; these are stored in a transversal ARCHITECTURE.md that is automatically included in every agent context. Layer 2 consists of feature development via vertical slices, ordered hardest-first, so the riskiest end-to-end path is implemented first as a tracer-bullet-style slice. Fakes are prohibited on the current slice’s active path; only frontier stubs are allowed for dependencies owned by later slices, and these remain explicitly tracked as not done.

To address context explosion, the framework defines the Spine as the ownership path from graph root to the current node. For a node N, the context bundle given to the coding agent is RootInvariants + SpineContracts(root→N) + N.fullSpec + Contracts(deps(N)) + N.ownCode. This means the agent sees only the root invariants, the contracts of ancestors, the full spec of the target node, one-hop dependency contracts, and the code it owns. It explicitly excludes siblings, dependency designs, dependency implementations, transitive dependencies, and unrestricted repository search. This enforces Parnas-style information hiding at the prompt/context level.

The overall engine comprises five components: the Spec Graph, a Context Assembler, a Coding Agent, a Capability Registry, a Drift Validator, and Governance Gates. The workflow is that the Context Assembler derives the scoped Spine bundle from the graph, the coding agent produces both code and spec deltas, the Drift Validator checks alignment between code and spec, and the Governance Gate blocks or permits integration. The key governance principle is that code and spec must evolve in the same commit, and any divergence is treated as a machine-enforced failure rather than a manual review concern.

### 🧪 Experiments
The provided text does not report a formal empirical evaluation, benchmark suite, or quantitative experiment results. Instead, the paper motivates the framework using prior evidence from the literature on long-context degradation and architectural drift, citing findings such as uneven long-context utilization, context-rot effects, and severe performance drops on long-context software engineering tasks as window sizes increase. The argument is therefore analytical and design-oriented rather than experimentally validated in the excerpt: no datasets, baselines, metrics, ablations, or numerical comparisons for the Spec Growth Engine itself are included in the available text.

### 🌱 Insights & Implications
A central insight is that AI-assisted development needs architectural controls at the same abstraction level as human software engineering principles, not just better prompts. The paper reframes context management as an architectural scoping problem: if module boundaries are explicit and machine-readable, they can constrain what an agent is allowed to see, reducing overreach and context-induced quality collapse. A second important insight is that documentation drift becomes much more dangerous under high-speed code generation, because stale specs misguide future agent runs and amplify errors quickly. By turning drift into a blocking gate, the framework operationalizes classic reflexion-model ideas for the AI era.

The work is also notable for avoiding heavy model-driven engineering: it keeps spec and code coupled, rather than making one generate the other as a single source of truth. This likely improves practical adoptability. However, limitations are clear from the excerpt. The paper appears to be largely a conceptual framework with no demonstrated industrial deployment or controlled study in the provided text. Open questions include how drift is detected in practice, how expensive authoring and maintaining per-node specs is, how the approach scales in large polyglot repositories, how to handle cross-cutting behaviors that do not map neatly to ownership paths, and whether strict scoping ever deprives agents of useful emergent context. For related research, the paper suggests fruitful directions in machine-enforced architecture conformance, AI-aware modularization, spec maintenance automation, and governance mechanisms for safe code generation.

### 🎯 Why This Paper
This paper is highly relevant to software engineering and maintenance because it directly targets maintainability problems introduced by AI code generation: uncontrolled changes, architecture erosion, and stale specifications. It is also relevant to software security and vulnerability repair workflows, since security invariants are treated as first-class upfront constraints in ARCHITECTURE.md and can be enforced on every slice via the drift gate. For researchers in automated program repair and vulnerability detection/repair, the framework offers a useful process architecture for constraining repair agents, ensuring that fixes respect component contracts, and preventing silent regressions between intended behavior and implemented behavior.

## 📄 [How Much Static Structure Do Code Agents Need? A Study of Deterministic Anchoring](https://arxiv.org/abs/2606.26979v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Zhihao Lin, Mingyi Zhou, Yizhuo Yang, Li Li
📅 **Published:** 2026-06-25  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper studies whether adding lightweight static-analysis facts directly into source files as inline comments can make grep-first LLM code agents navigate repositories more effectively and more reproducibly. The main finding is a "deterministic anchoring effect": structural tags improve localization modestly, shorten search trajectories, and substantially reduce run-to-run variance, especially on medium-scale repositories.

### 🏆 Key Contributions
- Introduces CodeAnchor, a framework that extracts lightweight static structure offline and injects it back into repositories as plain-text comments consumable by standard grep-first code agents.
- Defines and empirically characterizes the deterministic anchoring effect: static structure primarily helps by constraining and stabilizing repository navigation rather than by dramatically increasing raw reasoning ability.
- Systematically studies how different annotation granularities and link directionality choices affect localization, navigation efficiency, and behavioral stability.
- Shows that lightweight call and inheritance topology improves function-level localization and reduces interaction rounds over a strong Codex baseline.
- Demonstrates that the benefit of structural tags is scale-sensitive: dense semantics have diminishing returns, and large hub-heavy repositories often benefit more from inverse-only links than from full bidirectional topology.
- Provides practical deployment guidelines for structure-aware code agents, including when to use lightweight topology, when to prune forward edges, and when denser tags are justified.

### ⚙️ Methodology
The authors build CodeAnchor on top of a strong grep-first agent based on OpenAI Codex. The core idea is to preserve the standard agent loop—natural-language issue input, iterative grep/rg-style searches, file inspection, hypothesis refinement, and navigation—while augmenting the repository itself with deterministic structural cues. Offline, CodeAnchor runs lightweight static analysis to extract repository-level relations such as call graph edges, inheritance links, configuration usage, and selected data-flow-style dependencies. The paper mentions PyCG for call graph construction plus AST-based extractors for other relationships. These extracted facts are rendered as compact 'CodeAnchor tags' and inserted as ordinary source-code comments colocated with functions, classes, and configuration entries. Because the tags live in-band as text, the unmodified grep-first agent can retrieve them through normal search and read them alongside code. This creates what the paper calls retrieval short-circuiting: once the agent lands on a relevant entity, nearby tags expose callers, callees, inheritance neighbors, or usage links without requiring additional exploratory searches. The study varies the structural information along two main axes: granularity/semantic density (from lightweight topology such as call and inheritance relations to denser structural semantics including implicit dependency/configuration relations) and directionality (forward links, inverse links such as 'who-calls-me', or richer combinations). The evaluation is framed around three research questions: whether topology helps localization and efficiency, how granularity and directionality interact with repository scale, and how tags alter trajectory behavior and run-to-run stability. A key conceptual contribution is that these tags are treated as deterministic anchors: fixed facts for a given snapshot that constrain the otherwise stochastic exploration policy of an LLM-driven agent.

### 🧪 Experiments
The evaluation uses SWE-bench Lite and SWE-bench Verified as the benchmark basis, with Codex as the primary baseline agent and repository navigation/localization as the main focus. The paper compares the baseline grep-first setup against variants augmented with different types of CodeAnchor tags. Metrics include function-level localization accuracy (e.g., Func@5), interaction rounds/trajectory length, link-following rate, run-to-run variance across repeated executions, single-run reliability such as Pass@1, and token overhead. The main quantitative results reported in the paper are: lightweight call/inheritance topology improves function-level localization by +2.2 percentage points in Func@5; trajectories become shorter by about 1.6 interaction rounds; tags increase link-following behavior from roughly 0.15-0.18 to 0.21-0.24; run-to-run variance is roughly halved; and Pass@1 improves by +3.4 percentage points on medium-scale repositories. These gains come with about 9.9%-10% more input tokens on SWE-bench Lite. The study further finds that denser semantic tags do not always help, showing diminishing returns in some repositories, and that in large or hub-heavy projects inverse-only links can outperform exposing forward edges because they reduce distraction while still revealing structurally important incoming dependencies. Overall, the experiments emphasize behavioral changes and reproducibility rather than only final repair success.

### 🌱 Insights & Implications
The paper's central insight is that static structure is most valuable as a behavioral regularizer for code agents, not merely as extra semantic knowledge. By surfacing deterministic structural breadcrumbs inline, the system reduces the need for agents to rediscover graph connectivity through noisy multi-hop lexical search, which in turn makes trajectories shorter and more repeatable. Another important insight is that repository scale and topology matter: medium-size projects benefit from lightweight topology by default, whereas very large, hub-dominated codebases can suffer from too many forward edges and may need pruned or inverse-only annotations. Dense structural tags are useful mainly when key dependencies are implicit, such as configuration propagation, but otherwise may add token cost without proportional benefit. A notable limitation is that the gains are modest in raw localization terms and come with extra context cost, so the approach is best viewed as a practical augmentation rather than a replacement for stronger retrieval or planning methods. The study also depends on the quality and coverage of static analysis; dynamic behaviors, reflection, or highly language-specific constructs may reduce anchor fidelity. Open questions include how deterministic anchors interact with patch generation quality, whether similar effects hold for agents beyond Codex, how to automatically adapt tag granularity to repository characteristics, and whether richer program analyses could improve repair without overwhelming the model with token noise.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, and vulnerability-related research because it targets a core bottleneck in repository-scale code agents: reliable localization and navigation in large codebases. For a researcher working on APR, vulnerability detection, or vulnerability repair, the idea of injecting lightweight static dependencies as deterministic anchors is directly useful for making agent-based bug/vulnerability triage more reproducible, improving fault localization, and coupling traditional static analysis with LLM agents in a low-overhead, practical way.

### 💡 TL;DR
This paper argues that the configuration layer surrounding IDE-based LLM coding agents is an overlooked software supply-chain and governance problem, not just a prompt-authoring convenience. It introduces Rel(AI)Build, a deterministic control plane that adds integrity checks, permission enforcement, phase-gated execution, portability across IDEs, and prompt-drift detection, supported by prevalence evidence from 10,008 GitHub repositories and conformance tests showing the mechanisms enforce their intended invariants.

### 🏆 Key Contributions
- A deterministic control-plane architecture for LLM coding agents that sits above existing coding harnesses rather than replacing them.
- A supply-chain management design for agent configurations using SHA-256 content addressing, HMAC-stamped lockfiles, append-only hash-chained audit logs, install-time integrity gates, and an SBOM-style representation for agent definitions.
- A deterministic pre-execution guardrail system with tiered permissions tied to approved agent definitions and attack-derived command/path blocklists enforced through IDE hooks.
- A phase-gated lifecycle model that constrains agent-driven feature work through a state machine requiring ordered progression and requirement→file→test traceability.
- A define-once/compile-to-many configuration compiler that emits canonical policies to seven IDE/tool dialects, addressing cross-tool fragmentation.
- A prompt-drift detection method based on tokenisation, normalisation, and Jaccard similarity to flag potentially meaningful changes to agent instructions.
- A prevalence study showing that agent configuration files are commonly duplicated across repositories, infrequently revised, and rarely contain explicit permission boundaries.

### ⚙️ Methodology
The paper first frames the governance surface of LLM coding agents as a distinct layer consisting of rules files, project instructions, and agent definitions that are consumed by IDE-integrated coding harnesses. It then proposes Rel(AI)Build as a deterministic control plane for this layer. The system uses a canonical agent definition as the source of truth and treats that definition as a managed artifact. Each configuration is content-addressed using SHA-256; associated lockfiles are HMAC-stamped to detect accidental or automated corruption; and modifications are recorded in append-only, hash-chained audit logs to provide tamper-evident history. Before an agent can be invoked, install-time integrity gates verify that the active configuration matches approved, tracked content.

For execution governance, Rel(AI)Build binds agent definitions to tiered permission sets and enforces those permissions before LLM invocation rather than relying on model behavior. It augments this with deterministic blocklists derived from attack patterns, implemented as runtime hooks in supported IDE environments such as Claude Code and Cursor. These hooks inspect shell commands or file/path actions and deny unsafe operations. The system also includes a post-delegation scan-diff gate to inspect changes after agent activity.

To address unbounded execution, the architecture introduces a phase state machine that constrains work into explicit stages. The intent is to require progression through requirements, implementation, and testing with mandatory delegation and requirement→file→test linkage. The paper is careful to note the trust boundary here: state transitions and blocking behavior are deterministic, but some trace updates depend on cooperative agent use of tooling and are therefore audited post hoc rather than fully pre-execution enforced.

For prompt governance, the paper proposes a tokenisation and normalisation pipeline followed by Jaccard-similarity comparison to detect prompt drift. This is positioned as an operational detector of configuration changes, not a validated predictor of downstream model behavior. Finally, the system includes a compiler that transforms one canonical definition into seven tool-specific configuration formats, aiming to preserve governance semantics while avoiding reauthoring across IDE-specific dialects.

### 🧪 Experiments
The evaluation has two parts: implementation conformance testing and a repository prevalence study. The prevalence study analyzed 10,008 public GitHub repositories and identified 6,145 agent configuration files across multiple coding-agent dialects such as CLAUDE.md, Cursor rules, AGENTS.md, and related tool-specific files. The strongest result is that 10.1% of tracked configuration paths were exact SHA-256 duplicates across independent repositories after fork adjustment, indicating widespread propagation of shared agent configurations; 75.5% of duplicate-pair relationships crossed organizational boundaries. As secondary indicators, 58% of configuration files had only a single committing revision, and age-normalized maintenance remained lower than comparable CI/CD workflow files at 0.4 versus 0.6 commits per month. Permission declarations were nearly absent in agent configs, occurring in fewer than 1% of files, versus 33% in GitHub Actions workflows; however, the authors explicitly note this result depends on a fragile parser and only 31 true positives.

The paper also reports conformance tests using injected violations to validate that each deterministic mechanism enforces its intended invariant. These tests cover the supply-chain checks, permission and blocklist controls, phase-state transitions, drift detection, and compile/generation logic. The reported sample sizes are small but structured, with section references indicating injected-violation test sets of N=10/15/20 depending on the mechanism. The claimed result is not that developer productivity or repair quality improved, but that the system correctly blocked or flagged out-of-policy states and preserved the intended governance invariants. A separate drift-detection vignette in the appendix is presented only as an illustration, not as a statistically meaningful effectiveness study.

### 🌱 Insights & Implications
The central insight is that LLM coding-agent governance should be treated as a software-engineering and supply-chain problem, not merely a prompt-engineering problem. The paper’s prevalence results suggest that agent configurations already behave like reusable components copied across repositories, yet they lack provenance, update discipline, and permission specification. This has direct implications for secure software maintenance: even if the underlying coding harness improves, the policy layer steering it may still introduce organizational risk.

A notable strength is the paper’s insistence on determinism. Rather than adding another LLM-based controller, it advocates ordinary verifiable mechanisms such as hashing, HMACs, regex/path filters, state machines, and canonical compilation. This is especially relevant for safety-critical or regulated software workflows where reproducibility and auditability matter more than adaptive orchestration.

At the same time, the paper is explicit about its limits. It does not demonstrate improved developer productivity, repair quality, or reduced vulnerability incidence. Some empirical signals are preliminary: commit-depth comparisons are age-confounded, permission-declaration analysis is parser-fragile, and prompt-drift detection is only a change detector rather than evidence of harmful behavioral drift. The phase-traceability mechanism also depends partly on cooperative tool usage, so not all guarantees are enforced purely before execution. An important open question is whether these deterministic controls meaningfully reduce security failures or unsafe code changes in realistic development settings. Another is how well the define-once/compile-to-many approach preserves semantics as vendor dialects evolve.

For related research, the paper points toward a useful separation of concerns: model/runtime safety, output validation, and agent governance are distinct layers. That framing could inform future work on secure coding assistants, auditable automated repair pipelines, and policy-preserving multi-tool development environments.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it addresses the governance, security, and reproducibility of LLM-based coding systems that increasingly influence software maintenance, automated repair, and vulnerability handling workflows. For work on automated program repair and vulnerability detection/repair, the paper contributes a deterministic enforcement layer that can bound agent permissions, create auditable traces from requirements to code and tests, and manage prompt/configuration drift—capabilities that are directly useful when integrating LLM agents into secure software engineering pipelines.

The paper's main insight is that AI coding agents do not eliminate collaboration costs; they shift the bottleneck from writing code to understanding intent, judging architectural fit, enforcing policy, and taking responsibility for long-term maintenance. Open questions include how faithfully inner-agent regeneration preserves contributor intent, how much manual effort is required to build and verify knowledge packages, whether maintainers will accept the extra workflow overhead, how attribution and contributor credit should work, and how package quality can be measured reliably. Another important research implication is that KPR may serve as a defense against poisoned or low-trust patches, but it also introduces new attack surfaces around package sanitization, trace cleaning, and regeneration fidelity.

- Introduces feature reduction, a new decomposition strategy for C-to-Rust translation that breaks interpreter migration into runnable, testable feature levels rather than function-level units.

### ⚙️ Methodology
Reboot takes a C interpreter and its test suite as input and produces a safe Rust translation through three phases: planning, feature reduction, and translation. In the planning phase, the system identifies language features implemented by the interpreter and constructs a reduction plan that orders them according to dependencies. In the feature reduction phase, it progressively removes cross-cutting features from the full C interpreter to create a sequence of feature levels (FLN down to FL0), where each level is still a complete, runnable interpreter supporting a subset of the language. The key technical motivation is that interpreter features such as closures, regex, or exception handling are cross-cutting and require coordinated edits across lexer, parser, runtime, and execution components, making function-wise decomposition ineffective. Reboot’s multi-agent architecture addresses the unreliability of LLM coding agents by using specialized agents for code generation, validation, and feedback. Automated validators detect failed builds, incorrect behavior, protocol violations, or stalled progress; recovery is guided through feedback loops and enforced by a finite-state-machine guard that constrains the workflow. The approach allows substantial restructuring needed for safe Rust, including redesign of shared state, replacement of manual memory management with ownership-based allocation, and conversion of C mechanisms like setjmp/longjmp into idiomatic Rust Result-based error propagation. Human intervention is used only when the system needs semantic clarification or design decisions it cannot reliably infer.

The evaluation covers six real-world interpreters: awk (6k LOC), picoc (8k), gnu-bc (8k), wren (8k), mujs (17k), and pocketpy (23k). For security, the authors reintroduced 20 historical CVEs into mujs and translated that vulnerable version; the safe Rust translation eliminated memory-safety issues such as heap buffer overflows, stack buffer overflows, and use-after-free errors. For performance, release builds of the translated interpreters showed median slowdowns of roughly 1.28x–1.51x relative to the original C implementations.

The paper’s main insight is that feature-based decomposition is a much better fit than function-based decomposition for migrating interpreters, because the semantics of language features are implemented across many files and subsystems and often require coordinated redesign under Rust’s ownership rules. Limitations include non-perfect pass rates on unseen validation tests, nontrivial runtime overhead, dependence on test suites and manually adapted feature-level tests, and some residual need for human decisions. The authors explicitly note that stronger attacks, especially optimized white-box attacks such as GCG-style methods, were not tested and remain an open challenge.

### 🌱 Insights & Implications
A central insight is that prompt injection in agents is fundamentally an authorization and integrity problem, not merely a harmful-text detection problem; once an LLM consumes attacker-controlled content, the key question is whether that content can authorize actions through tools. This makes classical systems-security abstractions unexpectedly relevant to LLM agents. The paper also makes an important methodological point: static benchmark success can be misleading, because the same evaluation pattern previously made in-band defenses appear strong until adaptive attacks broke them at very high rates. The preliminary positive result for Progent suggests that deterministic policy enforcement may have better adaptive robustness than model-layer detectors, but the authors are careful not to overclaim. Limitations include testing only one defense in depth, one open-weight model, one benchmark, one small model scale, and one hand-crafted black-box adaptive attack. Open questions include whether stronger models change the attack/defense balance, how these defenses behave under white-box optimization such as GCG, whether direct user input should always be treated as high integrity, and how to extend action-focused defenses to cover broader harms such as deceptive content or privacy leakage not mediated by explicit tool calls.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated defense evaluation because it treats prompt injection as a principled security vulnerability class and evaluates defenses using an attacker-aware methodology analogous to robust security testing. For a researcher interested in software maintenance and repair, it offers a structured framework for reasoning about enforcement mechanisms, policy mediation, and adaptive adversarial evaluation—useful for designing, validating, and hardening agentic systems that invoke tools, APIs, or code-repair actions.

## 📄 [Verifying Intent and Harm: A Unified Defense Against LLM-Generated Threats](https://arxiv.org/abs/2606.26377v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Poojitha Thota, Yun Lei, Santhosh Thangaraj, Siddhartha Reddy Jonnalagadda, Shirin Nilizadeh
📅 **Published:** 2026-06-24  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper argues that many unsafe LLM interactions can only be recognized by jointly examining both the user's intent in the prompt and the harmfulness of the generated response. It proposes a three-agent verification framework—Task Analyst, Safety Analyst, and Judge—that consistently outperforms prompt-only, response-only, and single-agent baselines across jailbreak, prompt injection, phishing, cyber abuse, and harmful-content settings.

### 🏆 Key Contributions
- Identifies intent–harm separation as a structural weakness in existing LLM defenses: malicious intent may be hidden in the prompt while actionable harm appears only in the response.
- Introduces prompt–response verification as a unified safety principle that jointly evaluates prompt intent and response harm before delivering an LLM output.
- Implements this principle with a multi-agent architecture consisting of a Task Analyst for intent assessment, a Safety Analyst for response harm assessment, and a Judge for structured conflict resolution.
- Demonstrates effectiveness across five threat categories and 18 dataset settings, outperforming strongest applicable baselines in average F1 and attack success rate.
- Shows via ablations that structured cross-verification reduces false positives on benign-sensitive requests compared with single-agent, CoT, and simple aggregation strategies.
- Evaluates robustness against architecture-aware adaptive attacks and finds the framework degrades more gracefully than Single-Agent+CoT.

### ⚙️ Methodology
The proposed defense is a verification-centric, multi-agent moderation framework placed between the LLM generation step and final response delivery. It treats safety as a joint decision over the full prompt–response pair rather than as prompt classification or output moderation alone. The architecture has three specialized components. First, the Task Analyst inspects the user prompt to infer adversarial intent, with particular emphasis on distinguishing generative requests that ask the model to create new harmful artifacts from regenerative requests such as translation, summarization, or benign analysis of already-existing sensitive material. Second, the Safety Analyst inspects the generated response for actionable harm using a structured safety taxonomy covering harmful content, phishing material, malicious code, toxic/abusive text, jailbreak outputs, and similar threat-specific artifacts. Third, a Judge receives both analysts’ assessments and performs structured cross-examination through a two-round dialogue protocol, resolving disagreements and producing the final allow/deny decision. All three agents can access both prompt and response, but each is instructed to reason over a complementary dimension. The same underlying architecture is reused across threat categories, while category-specific safety criteria are encoded in prompts/system instructions instead of by retraining. The design goal is to create orthogonal verification checkpoints so an attacker must simultaneously evade intent detection, harm detection, and adjudication, making the system more robust than single-pass reasoning or one-sided filtering.

### 🧪 Experiments
The evaluation is organized around four research questions: overall effectiveness across threat categories, benefit of structured cross-verification over simpler reasoning baselines, robustness to adaptive attacks, and the accuracy–cost tradeoff. The framework is tested across five threat categories: jailbreaks, prompt injection, phishing/social engineering, malicious code and cyber abuse, and harmful/dangerous content. It is evaluated on 18 dataset settings drawn from multiple benchmarks. Category-specific baselines include SelfDefend and AutoDefense for jailbreaks; PromptGuard-2 and InjecGuard for prompt injection; ScamLLM for phishing; and LlamaGuard-3-8B, ShieldGemma-2B/9B, WildGuard, and OpenAI Moderation API for harmful content. For cyber abuse, where traditional malware scanners are mismatched to the prompt–response verification setting, the comparisons focus on LLM-based baselines such as Single-Agent and Single-Agent+CoT. Additional architectural baselines isolate the effect of the design: Task Analyst only, Safety Analyst only, OR aggregation, AND aggregation, majority voting, one-round verification, Single-Agent, and Single-Agent+CoT. The main result is that across threat categories the proposed framework improves average F1 from 0.898 for the strongest applicable baselines to 0.952 while reducing average attack success rate to 4.1%. Compared with Single-Agent+CoT, it improves average F1 from 0.872 to 0.953 and lowers false positive rate on benign-sensitive prompts from 0.116 to 0.062. Under adaptive attacks, the framework achieves lower attack success rates than Single-Agent+CoT: 0.073 vs. 0.219 for intent-obfuscation attacks and 0.089 vs. 0.241 for Judge-targeted attacks. The paper also analyzes latency/token-cost overheads, concluding the method is especially suitable for high-risk or policy-sensitive deployments.

### 🌱 Insights & Implications
The paper’s central insight is that unsafe LLM behavior is often distributed across the interaction: neither prompt-only nor response-only views are sufficient when attackers cloak malicious goals in benign framing or when benign-sensitive tasks produce superficially risky outputs. The experiments suggest that explicit decomposition into intent analysis, harm analysis, and adjudication is more effective than simply prompting one model to reason harder with CoT. Another important implication is that unified safety architectures may generalize across heterogeneous threats without task-specific retraining, provided they use threat-aware instructions. For security research, the adaptive attack results are encouraging because they show partial resilience even when the attacker understands the verifier structure, though the nonzero adaptive ASR indicates the defense is not solved. Likely limitations include added inference cost and latency, dependence on the reasoning quality of the verifier LLMs, and the possibility that increasingly sophisticated multi-turn or distribution-shifted attacks could still exploit blind spots in the analysts’ taxonomies or the Judge’s arbitration process. Open questions include how the approach scales to long conversational contexts, tool-using agents, code-generation workflows, and automated calibration of when expensive joint verification should be invoked.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and secure LLM-enabled development workflows because it proposes a practical defense architecture for detecting malicious or unsafe generations such as prompt injections, phishing content, and cyber-abuse code. For a researcher working on automated program repair and security-oriented software engineering, the paper advances the state of the art in verification-style safeguards that could be integrated into code assistants, secure software pipelines, and vulnerability remediation systems to prevent harmful model outputs while reducing false positives on legitimate security analysis tasks.
