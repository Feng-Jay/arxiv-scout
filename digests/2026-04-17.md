## 📄 [NFTDELTA: Detecting Permission Control Vulnerabilities in NFT Contracts through Multi-View Learning](https://arxiv.org/abs/2604.15118v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Hailu Kuang, Xiaoqi Li, Wenkai Li, Zongwei Li
📅 **Published:** 2026-04-16  🏷️ **Categories:** cs.CR

### 💡 TL;DR
NFTDELTA is a static-analysis framework for NFT smart contracts that detects permission-control vulnerabilities by comparing multi-view embeddings of function control-flow graphs rather than relying on full symbolic execution or supervised training. On 795 popular NFT collections, it found 241 manually confirmed vulnerabilities with 97.92% average precision and 81.09% F1, while showing better throughput and scalability than heavier analysis approaches.

### 🏆 Key Contributions
- Proposes NFTDELTA, a similarity-based vulnerability detection framework for NFT contracts that combines static analysis with multi-view learning over function CFGs.
- Introduces a two-view code representation that fuses sequence features from CFG execution paths with graph features from CFG structure to build reusable embeddings for defect similarity matching.
- Defines and operationalizes three NFT permission-control vulnerability categories: Weak Authorization Validation, Loose Permission Management, and Bypass Authorization Reentrancy.
- Designs a customizable detector interface that can use known vulnerable functions from tool reports or manual audits as references, enabling extensible detection beyond fixed hand-coded patterns.
- Builds and analyzes a real-world dataset of 795 popular NFT collections, identifying 241 confirmed permission-control vulnerabilities and demonstrating strong precision, F1, efficiency, and scalability.

### ⚙️ Methodology
NFTDELTA is a static-analysis pipeline centered on function-level similarity detection. The framework first extracts smart-contract functions and constructs control-flow graphs (CFGs) for them. From each CFG, it derives two complementary views. The first is a sequence view: executable paths or path-like instruction sequences are extracted from the CFG to preserve contextual ordering and local execution semantics. These path sequences are encoded with a sequence encoder to obtain contextual embeddings. The second is a graph view: the full CFG topology is represented and encoded with a graph representation learning model to capture structural control dependencies and branching relationships that cannot be preserved by linearized paths alone. The two embeddings are then fused into a unified code representation for each function.

Detection is not framed as supervised classification. Instead, NFTDELTA uses a non-training, similarity-based strategy: it builds embeddings for known vulnerable functions collected from prior detector outputs or manual audit reports, stores them as reference signatures, and compares target-function embeddings against them. A custom detector interface implements category-specific matching logic for permission-control scenarios. If a target function is sufficiently similar to a vulnerable reference in the fused embedding space, NFTDELTA flags it as a candidate vulnerability. This design makes the system lightweight at inference time, reusable through a shareable vector database, and extensible to new defect types without retraining a full model. The framework is tailored to three permission-control classes in NFT contracts: Weak Authorization Validation, Loose Permission Management, and Bypass Authorization Reentrancy, with subcases such as standard/non-standard authorization failures and incorrect assignment/revocation of permissions.

### 🧪 Experiments
The evaluation targets 795 popular NFT collections gathered from real-world NFT ecosystems. The study focuses on permission-control bugs in ERC-721/ERC-1155-style contracts and applies NFTDELTA to identify the three defined vulnerability classes. The reported findings include 241 manually confirmed vulnerabilities: 214 Bypass Auth Reentrancy cases, 15 Weak Auth Validation cases, and 12 Loose Permission Management cases. Manual verification is used as the ground truth for validation. The main effectiveness metrics reported are average precision and F1-score, where NFTDELTA achieves 97.92% average precision and 81.09% F1. The paper also compares NFTDELTA with other contract analysis frameworks in terms of throughput and scalability, arguing that its similarity-based vector retrieval approach is more efficient than resource-intensive symbolic execution, taint analysis, or fuzzing-heavy workflows. Although the paper emphasizes efficiency and scalability improvements, the clearest quantitative headline results in the provided text are the vulnerability counts and the precision/F1 values above.

### 🌱 Insights & Implications
A key insight is that many NFT security issues, especially permission-control flaws, can be detected effectively by comparing structural and path-level code semantics rather than executing heavy whole-program analyses. The multi-view design is important: sequence information captures execution context, while graph information captures control-flow structure; their fusion appears better suited to NFT permission logic than either view alone. Another implication is that reusable vulnerability embeddings can serve as a practical bridge between manual auditing knowledge and automated large-scale scanning, enabling a semi-general detection workflow for emerging bug patterns.

The paper also shows that NFT-specific vulnerabilities remain under-covered by existing general smart-contract tools, especially for permission logic and NFT-tailored reentrancy patterns. This suggests a broader research opportunity in domain-specific detectors for standards like ERC-721 and ERC-1155. Limitations include dependence on existing vulnerable references for similarity matching, possible sensitivity to embedding thresholds, and incomplete coverage of previously unseen vulnerability shapes that are too dissimilar from stored examples. The taxonomy of permission-control flaws is explicitly extensible, indicating open work in expanding categories, enriching benchmark datasets, and combining similarity search with symbolic reasoning or repair generation for higher recall and actionable remediation.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated analysis of real-world code because it proposes a scalable static-analysis framework for finding access-control flaws in smart contracts using learned code representations. For a researcher interested in software maintenance, APR, vulnerability detection, and repair, NFTDELTA is useful both as a detection technique for a high-impact domain and as a model for combining program representations, similarity search, and customizable detectors that could be adapted to bug localization, patch retrieval, or vulnerability repair workflows.

The paper studies whether retrieval-augmented generation can make LLMs more reliable for two software V&V tasks: source-code inspection and automated test generation. Across both tasks, adding retrieved external context generally improves output quality, helping LLMs detect more issues in code and produce better test cases than prompting alone.

### 🏆 Key Contributions

### ⚙️ Methodology

These artifacts are embedded into a vector store, and similarity search is used to return the most relevant context for the prompt. The retrieved context is then fused with the original query and provided to the LLM as augmented input. For code inspection, the LLM is asked to review source code and identify defects, inconsistencies, or inefficiencies; the retrieved context is intended to ground the model in supplementary evidence and reduce hallucinated findings. For test case generation, the LLM is prompted to synthesize tests based on program intent and surrounding artifacts, again with retrieved contextual material to better align tests with code and documentation. The authors explicitly motivate the approach as a hallucination-mitigation strategy: because LLMs often produce confident but incorrect outputs when context is missing, supplying external evidence should improve faithfulness and factual correctness. The study includes multiple LLMs to avoid overfitting conclusions to one model’s strengths or weaknesses and compares variants with and without RAG for both V&V tasks.

The study evaluates multiple LLMs in both non-RAG and RAG-enhanced configurations. The main reported finding is qualitative but consistent: adding retrieved external context has a generally positive effect on both tasks. In code inspection, RAG-enhanced models improve defect-detection effectiveness relative to plain prompting and are evaluated in relation to the 60% human-inspection benchmark. In test generation, RAG improves the quality/coverage of generated tests on TestEval compared with LLMs without retrieval. The excerpt does not provide detailed model names, exact dataset sizes, baselines beyond non-RAG LLMs and the human reference, or concrete numeric gains in the available text.

### 🌱 Insights & Implications
A central insight is that V&V tasks benefit from grounding: LLMs are more useful for testing and inspection when they can consult external project-relevant knowledge rather than rely solely on parametric memory. This is especially relevant for evolving codebases where code, documentation, and tests frequently drift out of sync. The paper also suggests a practical evaluation perspective: for code inspection, effectiveness relative to human reviewers matters more than raw language-generation quality, while for test generation, benchmarked test quality and coverage are appropriate proxies. Limitations visible from the excerpt include the lack of a direct hallucination metric beyond task accuracy, unclear details about the knowledge base contents and retrieval configuration, and limited quantitative reporting in the provided text. Open questions for future work include how RAG behaves on security-specific inspections, whether retrieved artifacts can be made repository-aware and change-aware, how the method scales to large multi-file projects, and whether the same framework can be extended from defect finding to automated repair or patch generation.

## 📄 [Feedback-Driven Execution for LLM-Based Binary Analysis](https://arxiv.org/abs/2604.15136v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** XiangRui Zhang, Qiang Li, Haining Wang
📅 **Published:** 2026-04-16  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper proposes FORGE, an LLM-based binary analysis system that replaces one-shot reasoning over fixed static artifacts with a feedback-driven loop of reasoning, tool use, and observation. Its main result is that a dynamically decomposed multi-agent execution model can scale long-horizon firmware analysis better than prior approaches, finding 1,274 vulnerabilities in 3,457 real-world binaries with 72.3% precision and broader vulnerability-type coverage.

### 🏆 Key Contributions
- Reformulates LLM-based binary analysis as a feedback-driven execution problem under partial observability rather than one-pass reasoning over a fixed representation.
- Introduces the Dynamic Forest of Agents (FoA), a decomposed execution model that recursively splits analysis tasks, coordinates parallel exploration, and bounds per-agent context to stabilize long reasoning chains.
- Unifies vulnerability discovery and verification by recording structured, replayable evidence chains during iterative exploration, enabling validation within the same execution framework.
- Implements an end-to-end system, FORGE, and evaluates it at scale on thousands of real-world firmware binaries against state-of-the-art baselines.

### ⚙️ Methodology
FORGE treats binary vulnerability analysis as an iterative sequential decision process in which an LLM agent repeatedly reasons about the current state, chooses an analysis action, receives observations from external binary-analysis tools, and updates its hypothesis. Instead of giving one monolithic agent a fixed disassembly/decompilation view, FORGE interleaves reasoning and tool interaction in a reasoning-action-observation loop. The core execution abstraction is the Forest of Agents (FoA): analysis is organized as a dynamic forest where each tree is rooted at a source-oriented analysis process and child agents are spawned through recursive task decomposition. This decomposition is designed to manage both depth and breadth in binary analysis. Long dependency chains are broken into shorter bounded-horizon subproblems so that each agent only handles a limited local context, reducing context degradation and attention drift. Multi-path exploration is handled by dynamically creating parallel agents for alternative call paths, source-sink combinations, and propagation hypotheses, allowing the system to cover diverse candidate behaviors without forcing all evidence into a single prompt. Agents coordinate through structured intermediate artifacts rather than raw long-context conversation histories. During execution, FORGE continuously records evidence such as disassembly snippets, call-chain fragments, symbolic values, taint-flow observations, addresses, and intermediate reasoning states. These artifacts form structured evidence chains that can later be replayed or checked during a verification phase. Thus discovery and validation are not separate pipelines: the same iterative execution that discovers a candidate vulnerability also accumulates the supporting proof trail needed to validate it. Conceptually, the system is designed to mitigate the main failure modes of LLM-driven long-horizon analysis: partial observability, sequential dependence of exploration decisions, context overflow, and instability across branching reasoning paths.

### 🧪 Experiments
The evaluation is conducted on 3,457 real-world firmware binaries. FORGE is compared against prior static and LLM-driven vulnerability analysis systems, including Mango, SaTC, LATTE, and SWE. The reported outcome is that FORGE identifies 1,274 vulnerabilities across 591 unique binaries and achieves 72.3% precision. The paper emphasizes that FORGE also covers a broader range of vulnerability types than prior approaches, indicating better generality than systems tuned to narrower bug classes or fixed workflows. Although the provided text does not expose the full per-benchmark breakdown, ablation tables, or exact metric definitions beyond precision and total findings, the headline empirical claim is that the feedback-driven and decomposed FoA execution structure improves both scalability and consistency in long-horizon firmware analysis compared with one-pass or less adaptive baselines.

### 🌱 Insights & Implications
The paper’s key insight is that execution structure, not just model size or prompt quality, is a first-order design factor for LLM-based binary analysis. By reframing vulnerability detection as adaptive execution under partial observability, the authors argue that many current LLM-for-program-analysis systems fail because they reason over frozen representations and cannot use intermediate results to redirect exploration. FoA suggests a promising systems direction for software security tasks that require long traces, branching hypotheses, and incremental evidence construction. For vulnerability research, the idea of preserving structured evidence during exploration is especially important because it supports downstream validation and may improve analyst trust. Relevant limitations and open questions remain. The excerpt does not provide fine-grained cost data, runtime overhead, or token/tool budgets, which are crucial for assessing deployment practicality. Since precision is 72.3%, false positives are still substantial, and recall is not reported in the visible text, so the full detection tradeoff is unclear. Dynamic agent spawning also raises questions about search efficiency, duplicate exploration, and coordination complexity. More broadly, it remains open how well FoA generalizes to tasks beyond firmware binaries, such as source-level vulnerability repair, exploit generation, or patch validation, and whether learned policies could further optimize decomposition and tool-selection decisions.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, vulnerability detection, and automated repair because it proposes a concrete LLM-agent execution model for analyzing real binaries under realistic long-horizon conditions. It advances vulnerability detection research by showing how to couple semantic reasoning with iterative tool use, structured evidence tracking, and dynamic task decomposition—ideas that can transfer directly to bug localization, root-cause analysis, vulnerability triage, patch validation, and repair workflows. For a researcher interested in automated vulnerability discovery and repair, FORGE is especially valuable as a systems paper that tackles the practical bottlenecks of context limits, unstable long reasoning chains, and evidence generation in complex program analysis.

## 📄 [Vibe-Coding: Feedback-Based Automated Verification with no Human Code Inspection, a Feasibility Study](https://arxiv.org/abs/2604.14867v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Michal Töpfer, František Plášil, Tomáš Bureš, Petr Hnětynka
📅 **Published:** 2026-04-16  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper studies whether LLM-generated adaptation managers for collective adaptive systems can be repaired purely through automated feedback, without any human code inspection. In a Dragon Hunt case study, the authors show that precise constraint-violation feedback—especially fine-grained functional violations expressed in a new temporal logic called FCL—usually leads to a correct manager within a few iterations, while coarse metric-only feedback often fails to converge.

### 🏆 Key Contributions
- A combined adaptation-loop and vibe-coding feedback-loop architecture that automatically executes, verifies, and repairs LLM-generated adaptation managers without human inspection of the code.
- A layered verification strategy that distinguishes syntax/interface errors, architectural consistency violations, and higher-level behavioral/strategy failures.
- Functional Constraints Logic (FCL), a first-order temporal logic over finite or infinite traces with explicit boundary operators such as BEG and MAX, designed to express runtime-adaptive system requirements and produce diagnostic counterexamples.
- An empirical feasibility study on the Dragon Hunt collective adaptive system showing that fine-grained constraint-level feedback is far more effective than coarse outcome-metric feedback for iterative LLM repair.
- Evidence supporting the claim that feedback precision is the key factor enabling non-programmer domain experts to steer LLM-generated adaptive behavior safely and effectively.

### ⚙️ Methodology
The paper proposes a verification-in-the-loop workflow for automated synthesis of adaptation managers (AMs) in Collective Adaptive Systems. An LLM is prompted with a structured template containing the interface contract, domain rules, strategy intent, and optionally a natural-language summary of functional constraints. The generated AM is then executed inside the system's adaptation loop, where at each step it observes component and environment state and resolves ensembles/groups for agents. A verifier monitors executions over multiple test runs and classifies failures into three categories: syntactic/interface failures (e.g., missing functions, wrong return types, runtime exceptions), architectural failures (e.g., invalid ensemble names, duplicate or missing agent assignments), and behavioral failures (e.g., poor coordination despite syntactic correctness). To support the latter, the authors introduce Functional Constraints Logic (FCL), a first-order temporal logic over traces. Its main operator counts how often a property holds within a bounded forward or backward window, enabling constraints such as early attack, persistent location discipline, eventual follow-up from one ensemble to another, and repeated farming. FCL includes explicit trace-boundary counters like BEG and MAX so formulas work on finite traces without ambiguous end-of-trace semantics. When a violation occurs, the verifier constructs a compact counterexample containing the violated constraint, the relevant step interval, and minimal supporting state evidence such as empty sets or specific offending agents. This report is appended to the next prompt, and the LLM is asked to revise the AM. The process iterates until a candidate passes all checks across the test suite or stalls.

### 🧪 Experiments
The evaluation is a feasibility study implemented in an open framework that automates prompt construction, LLM API interaction, execution of the adaptation loop, runtime verification, and iterative feedback. The benchmark is the Dragon Hunt case study, a game-like collective adaptive system in which villagers must coordinate farming, spawning, movement, and attacks to defeat a dragon within 30 steps. Success requires nontrivial multi-step coordination, making it a useful stress test for LLM-generated control logic. Verification is performed over multiple executions with varied initial states and random seeds, and a candidate adaptation manager is accepted only if it passes all checks across the full suite. The authors compare three feedback variants: metrics-only feedback as a baseline, which reports only coarse outcome metrics such as win/loss or remaining dragon HP; generic-only feedback, which reports only architectural and executability issues; and fine-grained functional-constraint feedback using FCL counterexamples. The central evaluation metric is efficiency, operationalized as the number of feedback iterations needed to obtain a verified AM. The main reported result is qualitative but clear: fine-grained functional constraint violations typically produce a valid adaptation manager within a few iterations, whereas coarse metric-based feedback frequently stalls and generic-only feedback is insufficient for strategy-level repair. The results support the authors' hypothesis that actionable, localized feedback dominates raw outcome information in determining convergence.

### 🌱 Insights & Implications
The paper's main insight is that for LLM-based program repair in adaptive systems, the bottleneck is not just detecting failure but encoding failure in a form the model can act on. Coarse signals like loss metrics or final objective values do not tell the LLM which strategic decisions were wrong, so repair often stagnates. In contrast, constraint-level counterexamples localize problems in time, behavior, and responsibility, making iterative improvement much more reliable. This is particularly important in collective adaptive systems, where correctness depends on temporally extended coordination rather than isolated functions. For software engineering and maintenance research, the work suggests a path toward automated maintenance loops where runtime verification artifacts become repair prompts. For security and vulnerability-related research, the broader implication is that precise, machine-generated diagnostics may similarly improve automated patching or policy repair, especially when failures involve temporal or architectural invariants rather than simple test failures. Limitations include the paper's feasibility-study scope, reliance on one synthetic case study, absence of large-scale quantitative benchmarking across models or domains, and dependence on human-authored constraints and test suites. Open questions include how well FCL generalizes to larger or real-world adaptive systems, how to automatically derive useful constraints, how robust the approach is across different LLMs, and whether similar feedback mechanisms can support stronger guarantees or integrate with formal repair and synthesis methods.

### 🎯 Why This Paper
This paper is highly relevant because it connects LLM-based code generation with automated verification and iterative repair in complex runtime systems, directly aligning with interests in software engineering, maintenance, automated program repair, and trustworthy code synthesis. Its core idea—using structured runtime verification feedback rather than human code inspection—offers a practical blueprint for building repair loops that could be extended from adaptive-system logic to bug fixing, invariant repair, and potentially vulnerability detection and remediation workflows.

## 📄 [NeuroTrace: Inference Provenance-Based Detection of Adversarial Examples](https://arxiv.org/abs/2604.14457v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Firas Ben Hmida, Philemon Hailemariam, Kashif Ali Khan, Birhanu Eshete
📅 **Published:** 2026-04-15  🏷️ **Categories:** cs.CR

### 💡 TL;DR
NeuroTrace detects adversarial examples by converting a model’s forward pass into an Inference Provenance Graph (IPG) that captures cross-layer activation and dataflow structure, then classifying these graphs with heterogeneous GNNs. Across image and malware domains, the paper shows that inference provenance is a strong and transferable signal, outperforming prior graph-based baselines in intra-attack, multi-attack, and cross-threat settings.

### 🏆 Key Contributions
- Introduces inference provenance as an explicit graph-structured signal for adversarial example detection, capturing both activation behavior and parameter-induced information flow across layers.
- Builds NeuroTrace, a reproducible PyTorch-based extraction framework that instruments model execution with forward hooks and converts forward passes into heterogeneous Inference Provenance Graphs.
- Defines a standardized heterogeneous graph schema for IPGs, including node abstractions, activation summaries, activation masks, layer metadata, and edge attributes derived from learned parameters or structural connectivity.
- Releases an open multi-domain NeuroTrace-IPG dataset and benchmark covering vision and malware tasks, benign samples, and adversarial examples under white-box and black-box threat models.
- Demonstrates that IPG-based detectors generalize beyond the attacks they were trained on, indicating that adversarial inputs induce recurring structural disturbances in inference-time information flow.
- Provides analysis of transferability, runtime overhead, storage trade-offs, and the conditions under which provenance-based detection is most effective.

### ⚙️ Methodology
The paper models a neural network inference as an input-dependent execution graph. For a trained model f and input x, NeuroTrace instruments the forward pass using PyTorch forward hooks attached at module boundaries such as convolutional and linear layers. During inference, these hooks capture intermediate activations and inter-layer dependencies, which are assembled into an Inference Provenance Graph (IPG). Nodes correspond to intermediate activations, using channel-level abstraction for convolutional layers and neuron-level abstraction for fully connected layers. Each node includes summary activation statistics such as mean, L2 norm, and sparsity ratio, an activation mask indicating whether the activation exceeds a threshold tau, and metadata such as layer index and layer type. Edges encode computational dependencies: for parametric layers they are derived from learned weights, while for non-parametric layers they represent structural connectivity. For convolutional layers, kernel weights are aggregated at the channel level to keep graphs tractable. The resulting graph is heterogeneous because both node and edge types vary by layer abstraction and connection semantics. NeuroTrace stores each IPG as a PyTorch Geometric HeteroData object. For learning, the framework uses heterogeneous GNN-based graph classification as a downstream detector; the emphasis is not on inventing a new GNN architecture but on evaluating whether provenance itself is informative. The graph classifier is trained on labeled IPGs where labels indicate benign or adversarial inputs. The framework standardizes extraction with deterministic seeds, configuration logging, Dockerized environments, and explicit extraction parameters to support reproducibility. The overall detection pipeline is: instrument model, extract IPG for each inference, encode graph with a heterogeneous GNN, and classify the graph as benign or adversarial.

### 🧪 Experiments
The evaluation uses a benchmark suite spanning at least two domains: image classification and malware detection. For each task, the authors generate IPGs for benign inputs and adversarial examples created using multiple attack families under both white-box and black-box threat models. They test three main settings: intra-attack detection, where training and testing use the same attack type; multi-attack detection, where the detector is trained on a mixture of attacks; and cross-threat or cross-attack transfer, where the detector is trained on some attacks and evaluated on unseen ones. Data splits are constructed at the original input level to prevent leakage across train, validation, and test sets. The compared methods include prior graph-based baselines and alternative graph representations that do not capture provenance as richly. Metrics are reported as detection performance measures, with the text emphasizing consistently high performance and improved transferability for NeuroTrace-based detectors over the baselines. Although the extracted text provided here does not expose the full numerical tables, the paper’s stated findings are that IPG-based detection achieves strong results across domains, improves over prior graph-based baselines, and retains effectiveness when transferred across attack families and threat models. The experiments also analyze sensitivity to the activation threshold tau and quantify practical costs such as extraction runtime and storage footprint of the generated provenance graphs.

### 🌱 Insights & Implications
A core insight is that adversarial examples appear to perturb not only isolated activations but the global execution structure of a model, and this makes inference provenance a more transferable detection signal than purely layer-local features. The use of heterogeneous graphs is important because it preserves distinctions between layer types and dependency types that would be lost in flattened or homogeneous representations. The framework’s cross-attack results suggest that different attacks may share common structural signatures in the forward pass, which is promising for generalization to unseen manipulations. For security research, the release of a standardized provenance dataset and extraction pipeline is especially valuable because it enables reproducible evaluation rather than ad hoc, task-specific graph construction. Another open question is how well provenance-based detectors hold up under adaptive adversaries who optimize not only for misclassification but also to mimic benign provenance patterns. ### 💡 TL;DR
Atropos improves the cost-performance trade-off of self-consistent LLM agents by predicting mid-execution whether a trajectory on a small local model will fail, then either terminating it early or hotswapping the ongoing context to a stronger model. Across three software-engineering agents, it predicts failing runs with about 0.85 accuracy/AUROC at the midpoint and reaches 74.35% of closed-model performance at only 23.9% of the monetary cost.

### 🏆 Key Contributions
- Introduces Atropos, a framework for predictive early termination and model hotswapping for self-consistent LLM agents.
- Represents multiple sampled agent trajectories as Semantic Flow Graphs and applies a Graph Convolutional Network to predict eventual success or failure from partial executions.

- Demonstrates strong early prediction capability, with up to 85.4% accuracy and 85.45% AUROC at the midpoint of inference.
- Shows that hotswapping can salvage up to 27.57% of executions that would otherwise fail on the source SLM.
- Empirically establishes a favorable cost-benefit trade-off, retaining 74.35% of proprietary-LLM performance at 23.9% of the cost.

### ⚙️ Methodology
Atropos targets LLM agents that use ReAct-style tool invocation and self-consistency. For each task instance, the agent is sampled multiple times on a source model, producing multiple reasoning/action trajectories. Atropos converts these ongoing trajectories into a unified graph representation called a Semantic Flow Graph (SFG), which captures both control-flow structure and semantic information of agent actions, tool calls, observations, state transitions, and outputs. Multiple inference paths from self-consistency are merged into this graph so that the system can reason over convergence/divergence patterns across samples. The core predictor is a Graph Convolutional Network trained to classify whether the current partial inference, if allowed to continue on the source LLM, will eventually succeed or fail. Prediction is performed at intermediate cut points such as the midpoint of the inference, enabling early intervention. If Atropos predicts likely failure, it terminates further execution on the source SLM and performs model hotswapping: because LLM interactions are stateless, the prior context can be replayed and the unfinished trajectory can continue on a stronger target LLM. The final decision under self-consistency is still derived from the multiple sampled runs, but Atropos selectively avoids wasting computation on source-model continuations that appear unpromising. The paper also explores different SFG encodings depending on the agent and task, including cases where node semantics come from structured tool actions and cases requiring embeddings of unstructured artifacts such as code patches or natural-language descriptions.

### 🧪 Experiments
The evaluation uses three recent software-engineering agents: AutoFL for method-level fault localization, AutoCodeRover (only its fault-localization stage, without SBFL support) for repository navigation-based localization, and RepairAgent for automated program repair. All experiments use self-consistency with 10 samples. For AutoFL and AutoCodeRover, the prediction target is whether the ongoing inference will eventually produce correct fault localization; for RepairAgent, the target is whether the run will generate a correct/plausible patch. Atropos is evaluated on its ability to predict failure from partial trajectories, especially at the midpoint of inference, using accuracy and AUROC. The main prediction result is up to 0.85 accuracy and 0.85 AUROC at the midpoint, with the paper also citing a peak accuracy of 85.4% and AUROC of 85.45%. After prediction, executions forecast to fail on the source SLM are hotswapped to a stronger closed model, and the study measures how many such failing runs are converted into successful ones; hotswapping salvages up to 27.57% of them. The end-to-end cost-benefit analysis compares Atropos against running fully on proprietary LLMs, showing that Atropos attains 74.35% of the successful inferences achieved by the proprietary baseline while using only 23.90% of the total monetary cost, i.e., up to a 76.1% cost reduction. The evaluation also compares alternative SFG configurations to test robustness across agents with different trajectory structures and output modalities.

### 🌱 Insights & Implications
The paper shows that partial agent trajectories contain enough structural signal to forecast eventual success, even before the execution finishes, which is important for expensive multi-step software-engineering agents. Self-consistency is not only an accuracy booster but also a rich source of trajectory diversity that can be mined for predictive signals via graph structure. The hotswap result suggests a practical hybrid deployment model: cheap open-weight models can handle easy instances, while expensive closed models are reserved for executions predicted to be difficult. This is particularly relevant for long-horizon tasks like fault localization and repair, where failed runs otherwise consume substantial tool and inference budget. Rather than sampling many complete trajectories and reranking them afterward, the agent generates candidate actions online at each step, scores them with the PRM, and prunes weak branches early. This converts the verifier from a passive reranker into an active search heuristic, reducing wasted computation and latency while preserving or improving solution quality. ## 📄 [Asking What Matters: Reward-Driven Clarification for Software Engineering Tasks](https://arxiv.org/abs/2604.14624v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Sanidhya Vijayvargiya, Vijay Viswanathan, Graham Neubig
📅 **Published:** 2026-04-16  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
The paper studies how software engineering agents should ask clarifying questions on underspecified bug-fixing tasks, showing that the most useful questions are those that target information strongly tied to task success and that users can realistically answer. Using these insights as reinforcement learning rewards, the authors train CLARITI, an 8B clarification model that reaches GPT-5-level issue-resolution performance while asking 41% fewer questions.

### 🏆 Key Contributions
- Introduces a categorization of six software-engineering information-need types in underspecified issues: error information, expected behavior, implementation details, external references, reproduction steps, and version/environment.
- Builds a categorization-grounded benchmark by rewriting SWE-Bench Verified issues to systematically remove selected information categories, enabling controlled study of underspecification.
- Quantifies which missing information matters most for downstream repair success using Shapley-based attribution over 700 underspecified instances, finding error information most influential, followed by implementation details and environment configuration.
- Analyzes what makes clarification questions answerable by comparing answerable vs. unanswerable questions and identifying patterns such as grounding in observable behavior and avoiding requests for internal state users are unlikely to know.
- Proposes CLARITI, an 8B clarification module trained with a multi-stage reinforcement learning reward that combines task relevance, user answerability, non-redundancy, and diversity.
- Demonstrates that empirically grounded reward design improves clarification efficiency: CLARITI matches GPT-5's resolution rate on underspecified issues while asking substantially fewer questions.

### ⚙️ Methodology
The study focuses on single-turn clarification for software engineering issue resolution. First, the authors derive a codebook of six information-need categories from expert underspecification annotations in SWE-Bench Verified. They then use GPT-5 with structured prompts to annotate which categories appear in each issue description and generate controlled underspecified rewrites by removing selected categories while preserving the rest. This produces naturalistic issue variants with known missing-information patterns.

To isolate clarification quality from coding ability, the framework separates clarification from execution. A clarification module first reads an underspecified issue and asks one or more questions. A simulated user, implemented with GPT-5 and given access to the fully specified issue, answers these questions. The resulting Q&A context is then passed to a fixed coding agent in OpenHands using Seed OSS 36B Instruct, which attempts to generate a patch. Success is measured by whether the patch passes the repository test suite.

For RQ1, the authors estimate how strongly each information category affects task success without clarification. They run the agent on 700 underspecified instances and use Shapley-style attribution over category presence/absence to measure each category's association with successful resolution. This yields an empirical hierarchy of task relevance.

For RQ2, they study user answerability by comparing distributions of answerable and unanswerable clarification questions asked in the simulated interaction setting. Their analysis identifies qualitative properties of answerable questions: they focus on observable failures and expected outputs, stay within reasonable technical depth, and avoid asking users for hidden implementation state or unavailable internals. They also analyze clarification composition and observe diminishing returns from asking more questions.

For RQ3, they operationalize the findings as a multi-stage reinforcement learning objective to train CLARITI, an 8B clarification model. The reward includes: task-relevance signals favoring questions about empirically high-impact categories; answerability signals favoring questions likely to elicit useful responses from typical users; and auxiliary rewards for non-redundancy and diversity to discourage repetitive or wasteful questioning. The goal is to learn a policy that asks fewer, higher-value questions before handing control to the fixed repair agent.

### 🧪 Experiments
The evaluation is built on SWE-Bench Verified issues. From 500 issues, the authors generate 1,500 underspecified rewrites by removing subsets of annotated information categories; 700 sampled instances are used for impact analysis in RQ1, while one rewrite per issue forms a 500-instance dataset for clarification experiments in RQ2 and RQ3. Manual validation on 50 sampled rewrites checks that intended categories were removed, non-target information was preserved, and the rewrites remained plausible issue reports.

The downstream repair agent is OpenHands with Seed OSS 36B Instruct, fixed across experiments with up to 30 interaction iterations and temperature 0.3. Clarification answers come from a GPT-5 simulated user with access to the fully specified issue. The main metric is binary task success: whether the generated patch passes all repository tests.

In RQ1, Shapley-based analysis over 700 underspecified instances shows that error information has the largest association with task success, followed by implementation details and version/environment information; the abstract states that these concrete diagnostic signals matter more than abstract goal specifications.

In RQ2, the authors examine answerable versus unanswerable questions and find that asking more questions often plateaus or even hurts performance while increasing user burden, because additional questions may introduce irrelevant or unusable context.

In RQ3, CLARITI is compared against strong baselines including GPT-5 as a clarification module. The key reported result is that CLARITI matches GPT-5's resolution rate on underspecified issues at 36.80%, corresponding to 88% of the fully specified upper bound, while reducing the average number of questions from 5.1 to 3.0, a 41% reduction in user burden.

### 🌱 Insights & Implications
A central insight is that clarification quality is not just about identifying missing information; it is about prioritizing information with the highest downstream utility while respecting what users can actually answer. This is particularly important in software maintenance and bug fixing, where many issue reports are incomplete but users often know symptoms and expected behavior better than root-cause or implementation internals.

The paper suggests a practical design principle for agentic software repair systems: train or prompt clarification modules to prefer concrete diagnostic information such as error messages, failure symptoms, and implementation constraints rather than broad or speculative questions. Another important finding is that more interaction is not inherently better; excessive questioning can degrade performance by adding noise and increasing burden.

Limitations include reliance on a simulated user rather than real developers, focus on single-turn clarification before implementation rather than interleaved multi-turn debugging, and dependence on LLM-based annotation and issue rewriting pipelines that may introduce bias. The reported hierarchy of information importance may also vary across repositories, languages, or task types beyond SWE-Bench Verified.

Open questions include how well the reward design transfers to real human-in-the-loop settings, whether answerability can be personalized per user expertise level, and how clarification should be integrated into full repair pipelines with retrieval, test generation, or iterative debugging. For related research, the work bridges ambiguity handling and automated program repair by making clarification an optimizable component rather than an ad hoc prompting trick.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, and vulnerability/bug-fix workflows because it tackles a key failure mode in agentic repair systems: underspecified issue reports. It advances the researcher's work by providing a principled framework to measure which missing information actually affects repair success, and by showing how to train clarification policies that improve repair efficiency without increasing user burden. The methodology could be directly adapted to bug triage, secure code repair, vulnerability reproduction, and human-in-the-loop vulnerability detection/repair settings where asking the right clarifying question is often more valuable than attempting a premature fix.
