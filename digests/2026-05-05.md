## 📄 [VulKey: Automated Vulnerability Repair Guided by Domain-Specific Repair Patterns](https://arxiv.org/abs/2605.01769v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Jia Li, Zhuangbin Chen, Yuxin Su, Michael R. Lyu
📅 **Published:** 2026-05-03  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
VulKey is an LLM-based automated vulnerability repair framework that improves patch generation by explicitly modeling security repair knowledge as hierarchical repair patterns over CWE type, syntactic action, and semantic key element. Using a two-stage pipeline with pattern matching and pattern-guided code generation, it achieves state-of-the-art repair accuracy on PrimeVul and strong cross-language results on Vul4J.

### 🏆 Key Contributions
- Proposes a novel three-level abstraction of vulnerability repair knowledge that represents repairs as (CWE type, syntactic action, semantic key element) patterns.
- Introduces VulKey, a two-stage AVR framework combining expert-knowledge pattern matching with pattern-guided LLM patch generation.
- Automatically extracts repair patterns from historical vulnerability-fix diffs, avoiding rigid handwritten templates and improving generalization beyond concrete exemplars.
- Shows that simply prepending CWE IDs to model inputs yields negligible or negative benefit, motivating explicit structured knowledge integration.
- Demonstrates state-of-the-art performance on real-world AVR benchmarks, including strong cross-language and cross-model generalizability.

### ⚙️ Methodology
VulKey builds AVR around explicit security-domain knowledge rather than relying on an LLM to infer repair strategy implicitly from vulnerable code. The core representation is a three-level repair pattern: the CWE category identifies the vulnerability type, the syntactic action captures the main edit operation used to fix it, and the semantic key element is a short code snippet from the patch diff that concretizes the repair mechanism or security primitive. This design aims to be more informative than a plain CWE label and more generalizable than full patch exemplars. The framework uses a two-stage pipeline. In stage one, repair patterns are extracted from historical vulnerability-fix diffs collected from 3,789 real-world vulnerabilities. A context-aware matcher based on CodeT5p is trained to predict, given the vulnerable code context and CWE type, the most suitable top-k (action, key element) repair patterns from a discrete pattern space. This acts as a learnable alignment mechanism that narrows the strategy search space and selects a context-specific repair direction. In stage two, a code-generating LLM is progressively fine-tuned through transfer learning: it first learns general bug-fix behavior from large bug-fix corpora, then adapts to vulnerability-fix data to specialize in security repair. At inference time, the matcher’s top-k predicted repair patterns are injected into the prompt as explicit guidance for the generator, decoupling pattern selection from patch synthesis and allowing the two components to be optimized independently.

### 🧪 Experiments
The main evaluation is conducted on PrimeVul, a real-world C/C++ vulnerability repair dataset built on CVEFixes and BigVul, with additional cross-language evaluation on the Java benchmark Vul4J. The paper compares VulKey against fine-tuned base code models, prior AVR systems such as VulRepair and VulMaster, and a closed-source LLM baseline GPT-5. It also studies a simple CWE-prefix baseline and shows that adding CWE type to the input does not help: for example, exact match on PrimeVul drops slightly from 23.9 to 23.6 for StarCoderBase-15B and from 23.2 to 22.5 for Qwen2.5-Coder-32B when CWE type is concatenated. On PrimeVul, VulKey reaches 31.5% repair accuracy, outperforming the best baseline by 7.6 percentage points. It also surpasses VulRepair by 8.5x, VulMaster by 3.6x, and GPT-5 by 3x according to the reported repair-success comparisons. On Vul4J, VulKey achieves 18 successful repairs, exceeding VulMaster’s 9 and VulRepair’s 4, indicating cross-language transferability of the repair-pattern abstraction. Reported metrics include repair accuracy and exact match/successful repair counts.

### 🌱 Insights & Implications
The paper’s main insight is that the bottleneck in LLM-based AVR is not only model capacity but how security knowledge is represented and injected. Coarse metadata such as CWE IDs is too weak to guide patch generation, while full examples can introduce noise, project-specific artifacts, and superficial copying. The proposed intermediate abstraction—syntactic action plus semantic key element conditioned on CWE—appears to strike a useful balance between generality and specificity. The work also suggests that repair-strategy selection should be treated as a separate alignment problem rather than being left implicit inside generation. For related research, this has implications for vulnerability detection-to-repair pipelines, retrieval-augmented security tooling, and knowledge-guided code generation. Likely limitations include dependence on historical vul-fix corpora for pattern extraction, possible coverage gaps for rare CWEs or novel repair mechanisms, and sensitivity to the matcher’s ability to predict the right pattern from context. Open questions include how robust the approach is for multi-function or semantic-heavy patches, how well it handles unseen vulnerability classes, and whether dynamic analysis or formal repair validation could further improve correctness and security guarantees.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it directly addresses how to generate secure patches for real-world vulnerabilities using structured security knowledge. It advances the researcher’s work by offering a concrete, data-driven mechanism for transforming CWE/CVE/NVD-style expert knowledge into actionable repair guidance, outperforming strong LLM and AVR baselines while suggesting a promising design pattern for future knowledge-aware repair systems.

## 📄 [HEJ-Robust: A Robustness Benchmark for LLM-Based Automated Program Repair](https://arxiv.org/abs/2605.02215v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Fazle Rabbi, Jinqiu Yang
📅 **Published:** 2026-05-04  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper introduces HEJ-Robust, a robustness benchmark for LLM-based automated program repair built by applying eight semantics-preserving Java code transformations to HumanEval-Java-Bug. Evaluating five fine-tuned repair models shows that even minor syntactic variations can sharply reduce repair success, with over 50% Pass@10 drops under some transformations despite little change in CodeBLEU.

### 🏆 Key Contributions
- Introduces HEJ-Robust, a function-level robustness benchmark for APR derived from HumanEval-Java-Bug using eight semantics-preserving transformations.
- Provides 1,450 transformed Java bug instances with executable tests to measure repair robustness under controlled syntactic variation.
- Performs a systematic robustness evaluation of five fine-tuned LLM-based APR models: PLBART-base, PLBART-large, CodeT5-small, CodeT5-base, and CodeT5-large.
- Shows that APR performance is highly brittle to certain superficial code perturbations, especially local variable renaming and try-catch insertion.
- Releases benchmark artifacts to support future research on robust, trustworthy, and transformation-invariant automated program repair.

### ⚙️ Methodology
The benchmark is constructed from the 164-instance HumanEval-Java-Bug dataset, a curated Java APR benchmark with manually injected bugs, buggy-line annotations, reference patches, and executable tests. The authors generate robustness variants by applying eight semantics-preserving source-to-source transformations that preserve program behavior but alter syntax in realistic ways: local variable renaming, method renaming, parameter renaming, log statement insertion, try-catch insertion, boolean exchange, loop exchange, and condition reordering. These transformations were chosen to reflect common real-world coding variations such as refactoring, stylistic restructuring, and defensive programming. Identifier renaming is performed using LLM-generated replacement names. Log insertion prepends System.out.println("log") to a method, which does not affect test outputs because the benchmark validates return values rather than stdout. Try-catch insertion wraps applicable statements in exception-handling blocks. Boolean exchange flips a boolean initializer and negates subsequent uses to keep semantics unchanged. Loop exchange converts between equivalent for and while forms. Condition reordering swaps operands in == and != expressions. Because not every transformation applies to every bug instance, the number of transformed examples varies by transformation, yielding 1,450 transformed instances in total. Robustness is then assessed by comparing each model’s repair performance on original versus transformed versions of the same bugs, enabling direct measurement of sensitivity to purely syntactic changes.

### 🧪 Experiments
The evaluation uses five fine-tuned APR models: PLBART-base, PLBART-large, CodeT5-small, CodeT5-base, and CodeT5-large. The base benchmark is HumanEval-Java-Bug with 164 buggy Java functions; transformed subsets include local variable renaming (100 bugs), method renaming (149), parameter renaming (162), boolean exchange (7), loop exchange (142), reorder condition (603), insert log statement (173), and insert try-catch (114). The main metrics are Pass@10, which measures functional repair success under test execution, and CodeBLEU, which measures patch similarity. Results show substantial robustness failures in Pass@10 while CodeBLEU remains nearly unchanged, indicating that lexical similarity does not capture repair reliability. The strongest degradation occurs under local variable renaming: Pass@10 drops by about 50% or more across all five models, e.g., PLBART-base 14.53 to 6.54 (-54.99%), PLBART-large 21.88 to 9.91 (-54.71%), CodeT5-small 19.35 to 8.26 (-57.31%), CodeT5-base 24.81 to 12.28 (-50.5%), and CodeT5-large 23.66 to 11.5 (-51.39%). Try-catch insertion also causes large declines, especially for CodeT5 variants: CodeT5-small 19.29 to 11.02 (-42.87%), CodeT5-base 25.17 to 18.12 (-28.01%), and CodeT5-large 26.14 to 18.12 (-30.68%). Reorder condition produces moderate but consistent drops across models, around 7% to 14%. Loop exchange and log insertion produce smaller but still noticeable degradations. Method renaming causes only modest declines, and parameter renaming has little effect, sometimes slightly improving performance. Boolean exchange shows mixed behavior but is based on only 7 instances and is explicitly excluded from headline claims due to low statistical power. Overall, the experiments demonstrate that fine-tuned LLM repair models are not robust to many semantics-preserving perturbations.

### 🌱 Insights & Implications
A key insight is that current LLM-based APR models often learn brittle syntactic correlations rather than transformation-invariant repair behavior. The very large drops under local variable renaming suggest heavy reliance on identifier surface forms, which is concerning for real-world maintenance settings where refactoring and naming variation are common. The disconnect between relatively stable CodeBLEU scores and sharply degraded Pass@10 indicates that similarity-based metrics can mask meaningful robustness failures; functionally grounded evaluation is therefore essential. The benchmark also reveals that not all perturbations are equally harmful: parameter renaming is largely tolerated, whereas local variable renaming and try-catch insertion are much more disruptive, offering clues about which syntactic signals models overfit to. For researchers, this opens directions such as transformation-aware training, data augmentation with semantics-preserving rewrites, robustness regularization, identifier abstraction, invariant representation learning, and repair-time test-time augmentation across multiple transformed variants. Limitations include the relatively small underlying dataset, focus on Java and function-level human-crafted bugs rather than project-scale real defects, and uneven applicability of transformations. Some transformations, like boolean exchange, have too few applicable instances for strong conclusions. Still, the benchmark is valuable as a controlled diagnostic resource complementary to project-level benchmarks like Defects4J-TRANS.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, and reliability-oriented research because it directly studies whether LLM-based repair systems remain effective under realistic code evolution and refactoring-like changes. For a researcher working on APR, vulnerability detection/repair, and trustworthy code models, HEJ-Robust provides both a concrete evaluation framework and empirical evidence that current models can fail under superficial syntax changes, highlighting an important gap that must be addressed before deploying repair systems in real maintenance or security pipelines.

## 📄 [QASecClaw: A Multi-Agent LLM Approach for False Positive Reduction in Static Application Security Testing](https://arxiv.org/abs/2605.01885v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Mohd Ruhul Ameen, Md Takrim Ul Alam, Akif Islam
📅 **Published:** 2026-05-03  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
QASecClaw augments a conventional SAST pipeline with a multi-agent, coding-specialized LLM reviewer that rechecks each SAST finding using surrounding code context to suppress false positives rather than detect vulnerabilities from scratch. On OWASP Benchmark v1.2, it substantially improves over standalone Semgrep, raising F1 from 78.39% to 90.93% mainly by cutting false positives by 88.6% with only a small recall drop.

### 🏆 Key Contributions
- Introduces QASecClaw, a multi-agent architecture that combines high-recall static analysis with LLM-based contextual verification for SAST false-positive reduction.
- Proposes a SAST Filter Agent that inspects each reported finding with source-code context and CWE-specific reasoning to classify it as likely true positive or false positive.
- Uses specialized cooperating agents—Mission Orchestrator, Test Planning Agent, Security Validation Agent, Evidence Correlation Agent, and SAST Filter Agent—to structure the verification workflow.
- Implements a conservative fail-open policy so LLM failures, timeouts, or malformed outputs never silently suppress potentially real vulnerabilities.
- Provides an evaluation on the full OWASP Benchmark v1.2 showing strong overall and per-CWE improvements, especially for injection vulnerabilities, with code and scripts released for reproducibility.

### ⚙️ Methodology
QASecClaw is designed as a post-processing layer over a conventional SAST scanner rather than a replacement detector. The workflow begins with a Mission Orchestrator that coordinates several specialized agents. A Test Planning Agent prepares the benchmark/task configuration and determines what code artifacts and findings need to be analyzed. A Security Validation Agent runs Semgrep over the target Java programs to produce candidate vulnerability reports. These findings are then passed to an Evidence Correlation Agent, which links each alert to relevant contextual information from the source code, such as the flagged line, enclosing method or file, nearby data flow, and other evidence needed for review. The core decision step is handled by the SAST Filter Agent, implemented with a coding-specialized LLM. Instead of asking the LLM to scan the whole project for vulnerabilities, the system constrains it to adjudicate already-reported SAST findings. For each alert, the agent receives the CWE category and source-code context and reasons about whether the apparent source-to-sink path is actually exploitable or whether there is mitigating context such as sanitization, validation, encoding, or use of safe APIs like parameterization. This CWE-aware contextual review is intended to mimic human security triage. The architecture follows a conservative division of labor: Semgrep supplies broad candidate coverage, while the LLM specializes in semantic disambiguation of noisy alerts. To limit risk, QASecClaw adopts a fail-open mechanism: if the LLM cannot return a valid judgment because of timeout, malformed response, or failure, the original SAST alert is kept rather than removed. This prioritizes avoiding false negatives introduced by the filter. Overall, the method can be understood as multi-agent orchestration of static scanning, evidence collection, contextual reasoning, and safe report generation.

### 🧪 Experiments
The evaluation uses the complete OWASP Benchmark v1.2, a Java benchmark with 2,740 test cases and ground-truth labels spanning 11 CWE categories. The baseline is standalone Semgrep, and the main comparison measures how much the LLM-based filtering layer improves detection quality over raw SAST output. The principal metric reported is F1 score, along with false-positive counts and recall effects; the paper also reports per-CWE performance. QASecClaw achieves an overall F1 of 90.93%, compared with 78.39% for Semgrep alone. The improvement is driven primarily by false-positive reduction: false positives drop from 560 to 64, an 88.6% decrease. This comes with only a 3.1% reduction in recall, indicating that most of the removed findings were indeed spurious. The paper highlights especially strong results on injection-style weaknesses, including SQL Injection with an F1 of 94.05% and Cross-Site Scripting with an F1 of 89.58%. It also maintains near-perfect performance on Weak Cryptography with an F1 of 99.61%. These results support the claim that contextual LLM review can materially improve the precision and practical usefulness of SAST reports without substantially sacrificing detection coverage.

### 🌱 Insights & Implications
The paper’s main insight is that LLMs are more reliable and useful as constrained reviewers of candidate findings than as unconstrained vulnerability detectors. This framing is important for security tooling because it leverages LLM semantic reasoning while reducing hallucination risk and preserving the high-recall strengths of established static analyzers. The multi-agent decomposition is less about autonomous exploration and more about modularizing a security triage pipeline into planning, scanning, evidence collection, filtering, and reporting. For software security research, the work suggests a promising pattern: use rule-based/static methods for candidate generation and LLMs for context-sensitive adjudication. The results are especially relevant to noisy categories such as injection bugs where local syntax often obscures whether data is actually sanitized or safely handled. Limitations include reliance on a benchmark rather than large industrial codebases, use of Semgrep as the primary scanner, and likely dependence on the quality of prompts and code context provided to the LLM. The paper does not establish how well the approach generalizes across languages, frameworks, evolving CWEs, or production-scale repositories with complex inter-file flows and custom libraries. Another open question is cost and latency in CI/CD settings, since each finding requires LLM review. There is also a broader research opportunity in integrating richer evidence, such as interprocedural analysis, test execution, or patch suggestions, and in studying whether false-positive filtering can be extended into vulnerability explanation and automated repair workflows.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and vulnerability repair-adjacent research because it addresses one of the biggest practical blockers in SAST adoption: excessive false positives. For a researcher interested in software engineering, maintenance, automated repair, and secure development pipelines, QASecClaw offers a concrete architecture for making static-analysis output more actionable and trustworthy, which directly affects developer workflows and triage efficiency. Its hybrid design—traditional analyzer plus LLM-based semantic validation—also aligns with emerging directions in LLM-assisted secure software engineering and could be extended to alert ranking, exploitability assessment, root-cause explanation, or repair candidate generation.

## 📄 [AgenticVM: Agentic AI for Adaptive Software Vulnerability Management](https://arxiv.org/abs/2605.01739v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Asrul Arifin, Hussain Ahmad, Yiyao Zhang, Diksha Goel
📅 **Published:** 2026-05-03  🏷️ **Categories:** cs.CR

### 💡 TL;DR
AgenticVM is a modular multi-agent vulnerability management framework that combines security scanners, rule-based filtering, a BERT-based CVSS prediction model, and LLM agents to automate detection, assessment, prioritization, and reporting. It substantially compresses scanner noise into actionable remediation queues—reporting up to 98% alert reduction—and predicts missing CVSS attributes with 89.3% accuracy, aiming to reduce analyst workload while preserving risk visibility.

### 🏆 Key Contributions
- Introduces an end-to-end agentic vulnerability management framework that spans detection, assessment, prioritization, remediation recommendation, CVSS prediction, and reporting rather than optimizing isolated subtasks.
- Proposes a hybrid architecture that combines deterministic rules, conventional security tools, a transformer-based CVSS prediction module, and specialized LLM-driven agents for auditable and adaptive workflow automation.
- Demonstrates strong alert-noise reduction in practical evaluation scenarios, shrinking large scanner outputs into compact high-priority queues suitable for analyst triage.
- Develops a BERT-based model for inferring missing CVSS attributes from vulnerability descriptions, enabling operational use when official scores are delayed or incomplete.
- Provides engineering guidance on agent decomposition, tool–LLM boundary design, observability, failure containment, and human-in-the-loop governance for deployment in security operations.

### ⚙️ Methodology
The paper presents AgenticVM as a multi-phase, multi-component vulnerability management system built for operational deployment. The authors first motivate the design through a structured review of prior work in vulnerability detection, assessment, prioritization, recommendation, CVSS prediction, and agentic AI, arguing that existing solutions are fragmented and usually address only one stage of the pipeline.

The system architecture is modular and agentic. It orchestrates vulnerability data across specialized components rather than relying on a single monolithic model. Input data comes from scanner findings and external vulnerability intelligence sources, especially the National Vulnerability Database (NVD), the European Union Vulnerability Database (EUVD), and supplementary feeds such as CISA Known Exploited Vulnerabilities (KEV). The workflow begins with data acquisition and preparation, including cleaning, normalization, deduplication, label verification, and tokenization of vulnerability descriptions.

A core predictive component is a BERT-based CVSS prediction module. This module takes textual vulnerability descriptions as input and estimates missing CVSS attributes or severity-related fields when official data is absent or delayed. The paper positions this as an operational bridge between raw disclosures and downstream prioritization. The transformer model is trained on curated vulnerability-description/CVSS pairs and evaluated using standard train/validation/test splitting.

Around this prediction model, AgenticVM uses specialized LLM-driven agents for distinct responsibilities such as assessment, contextual enrichment, prioritization support, mitigation recommendation, and reporting. Rule-based logic is retained for deterministic filtering and policy enforcement, which improves reproducibility and constrains agent behavior. This hybrid design deliberately assigns structured, high-confidence operations to rules/tools and more interpretive, context-heavy tasks to LLM agents.

The framework emphasizes orchestration, observability, and governance. Agent decomposition is used to improve auditability, isolate failures, and simplify debugging. Human-in-the-loop oversight is explicitly part of the design so analysts can review outputs, validate prioritization, and control high-impact decisions. The paper frames the work not only as an AI model contribution but as an engineering blueprint for reliable integration of LLMs into real-world vulnerability management workflows.

### 🧪 Experiments
The evaluation uses vulnerability data drawn primarily from NVD and EUVD, with CISA KEV used as supplementary threat context. For the CVSS prediction task, the authors build and test a BERT-based classifier/regressor over vulnerability text descriptions after preprocessing, normalization, and standard train/validation/test partitioning. The reported headline result is 89.3% accuracy for predicting missing CVSS attributes.

For end-to-end vulnerability management evaluation, AgenticVM is tested in multiple scenarios involving raw scanner outputs and subsequent prioritization/orchestration. Rather than focusing only on model accuracy, the experiments assess operational reduction of analyst burden: the framework filters and prioritizes findings into a smaller actionable set. The main reported result is up to 98% alert reduction, with one example reducing 3,983 raw findings to 82 high-priority items. The paper argues that this compaction preserves risk visibility while making triage manageable.

The comparison context in Table I is largely capability-oriented rather than a strict unified benchmark: the authors position AgenticVM against prior work that typically covers only subsets of the lifecycle, such as detection, prioritization, recommendation, or CVSS prediction alone. The paper’s main empirical claims therefore emphasize integrated workflow effectiveness, severity estimation utility, and prioritization efficiency rather than exhaustive head-to-head benchmarking against many baseline systems on the same dataset.

### 🌱 Insights & Implications
A central insight is that effective vulnerability management is not just a prediction problem but a systems-integration problem. The paper argues that the main bottleneck in practice is alert overload and fragmented tooling, so value comes from combining deterministic security infrastructure with adaptive language-model reasoning in a controlled pipeline.

The work also reinforces that textual vulnerability disclosures contain enough semantic signal for useful CVSS inference, making transformer-based prediction practically valuable when official scores lag behind disclosures. This is especially relevant for early triage and backlog reduction.

Another important takeaway is architectural: decomposing the system into specialized agents improves observability, fault isolation, and governance compared with a single general-purpose LLM. The authors suggest careful tool–LLM boundary design, where rules and classic tooling handle structured tasks and LLM agents are reserved for contextual interpretation, summarization, and recommendation.

For security deployment, the paper highlights human-in-the-loop governance as essential. This implies that agentic systems should support analyst override, traceability, and post-hoc review rather than fully autonomous remediation. That is especially important in security settings where errors can create major operational risk.

Limitations and open questions include the degree of generalizability across organizations, scanner ecosystems, and vulnerability domains; robustness of LLM agents under adversarial or noisy inputs; calibration of predicted CVSS attributes; and how much true risk coverage is preserved after aggressive alert reduction. The paper also leaves open how to benchmark agentic pipelines reproducibly against traditional prioritization systems, since integrated workflows mix prediction, orchestration, and reporting rather than a single measurable task.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it targets the full operational lifecycle of handling software vulnerabilities rather than only finding them. For a researcher working on automated program repair or vulnerability repair, AgenticVM is useful as a framework-level reference for how detection, severity estimation, prioritization, and mitigation recommendation can be connected into one deployable pipeline. It advances related work by showing how transformer-based vulnerability understanding and LLM agents can be embedded into practical security workflows with explicit governance, making it particularly valuable for research that seeks to move from standalone vulnerability models toward integrated, human-supervised remediation systems.

## 📄 [Autonomous LLM Agent Worms: Cross-Platform Propagation, Automated Discovery and Temporal Re-Entry Defense](https://arxiv.org/abs/2605.02812v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Mingming Zha, Xiaofeng Wang
📅 **Published:** 2026-05-04  🏷️ **Categories:** cs.CR

### 💡 TL;DR
The paper introduces the first automated framework for discovering, exploiting, and defending against persistent worm propagation in long-running LLM agent ecosystems. It shows that attacker-controlled text can persist in agent-managed files, re-enter future LLM contexts, and autonomously trigger cross-platform propagation, while proposing a formally grounded defense that blocks this write-read-action chain.

### 🏆 Key Contributions
- SSCGV, an automated source-code graph vulnerability analyzer that discovers persistent carriers by tracing flows from file I/O to LLM context injection points and ranking them by injection position without manual platform-specific analysis
- SRPO, a three-LLM summary-resilient payload optimization pipeline that produces worm payloads robust to summarization, paraphrasing, and compression across multiple semantic dimensions
- The first demonstration of zero-click autonomous worm propagation across heterogeneous production LLM agent frameworks, including 3-hop cross-platform spread, inter-agent privilege escalation, and workspace data exfiltration
- Empirical findings that user-prompt injection carriers are substantially more effective than system-prompt carriers, and that exposed read operations are the primary integrity threat in LLM-mediated systems
- RTW-A, a temporal re-entry defense framework with a formal No Persistent Worm Propagation theorem, combining write-before-exposed-read blocking, sealed configuration, typed memory promotion, and capability attenuation

### ⚙️ Methodology
The paper models persistent LLM-agent worm propagation as a temporal chain W^tau(f,t_w) ≺ R^up(f,t_r) ≺ A^H(t_a): attacker-influenced content is written to a persistent carrier, later re-read into an authority-bearing context such as an LLM prompt or memory loader, and then induces a high-risk action. Based on this model, SSCGV statically analyzes agent framework source repositories by constructing a code property graph and tracing dataflow from file read/write sites to downstream context injection sinks such as system prompts, user prompts, and memory-loading logic. It then ranks candidate carriers by exploitability, especially according to where their content is inserted into model context. To preserve payload semantics over multi-hop agent communication, the paper proposes SRPO, a three-LLM optimization loop that searches for text payloads resilient to summarization, paraphrasing, and compression while preserving four semantic properties: persistence, propagation, harmful action inducement, and verbatim survivability. The attack workflow begins with initial text injection through a channel or document processed by an agent, induces the agent to store the payload or a semantically equivalent derivative in a persistent carrier, relies on scheduled or automatic reload of that carrier into future reasoning context, and uses the compromised agent to message or influence other agents, including those on different frameworks. For defense, RTW-A enforces temporal re-entry prevention rather than simple read/write restrictions: RTW blocks writes whose contents could later be exposed in authority-bearing reads, sealed configuration protects static autoloaded files, typed memory promotion ensures only structured trusted summaries can enter trusted memory, and capability attenuation reduces tool authority after unavoidable external reads. The formal theorem argues that if these constraints hold, attacker-controlled content cannot complete the persistence-reentry-action chain needed for worm propagation.

### 🧪 Experiments
The evaluation is conducted on three anonymized open-source production agent frameworks, denoted Frameworks A-C, because coordinated disclosure is ongoing. The offensive evaluation measures whether SSCGV can automatically identify injectable persistent carriers and whether SRPO-generated payloads can sustain autonomous propagation across realistic multi-agent workflows. The paper reports zero-click autonomous propagation after a single initial injection, with no further human interaction required; 3-hop cross-platform transmission across heterogeneous framework boundaries without platform-specific adaptation; inter-agent privilege escalation through trust-based delegation; and data exfiltration from agent workspaces. Comparative observations show that carriers injected into user-prompt positions yield much higher attack compliance than those loaded into system-prompt positions, suggesting prompt-placement strongly affects exploitability. The paper also emphasizes that read paths into LLM contexts, rather than write paths alone, dominate integrity risk. Although the abstract and extracted text do not provide detailed numerical tables, named baselines, or metric definitions, the reported key outcomes are successful automated discovery across all tested frameworks and effective propagation despite summarization and paraphrasing in multi-hop communication. The defense evaluation claims that RTW-A eliminates the persistence-reentry-action chain while preserving ordinary workflows, supported by the formal theorem and system design rather than only empirical mitigation tests.

### 🌱 Insights & Implications
A central insight is that integrity in LLM-agent systems is governed by exposed reads, not just writes: once attacker-influenced content is reloaded into model context, it can alter future decisions and tool use even without code execution or memory corruption. Another important finding is that injection position matters operationally: user-prompt carriers are more dangerous than system-prompt carriers because models often treat user-turn content as actionable commands instead of background policy. For software security research, this reframes vulnerability analysis around persistent natural-language carriers, context assembly code, memory promotion logic, and scheduled-task reload paths. The work also highlights that heterogeneous frameworks can still share a common semantic propagation substrate, meaning cross-platform worms do not need framework-specific payload adaptation if they target general agent behaviors. Limitations include anonymized targets, withheld exploit-enabling details, and limited quantitative reporting in the visible text, which makes reproducibility and comparison harder until disclosure completes. Open questions include how RTW-A interacts with richer memory systems, retrieval-augmented generation pipelines, non-file-backed state stores, and collaborative agents with more complex trust and delegation mechanisms.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it treats LLM agent frameworks as analyzable software systems with discoverable dataflow vulnerabilities and enforceable defenses. SSCGV is directly aligned with automated vulnerability detection in codebases, while RTW-A provides a repair/mitigation blueprint for hardening persistent state, memory promotion, and capability management in agent software. For researchers interested in automated program repair and security tooling, the paper offers a concrete threat model, a static-analysis-inspired discovery method, and a formally reasoned defense that could inspire practical scanners, patches, and secure-by-construction agent architectures.

## 📄 [FunFuzz: An LLM-Powered Evolutionary Fuzzing Framework](https://arxiv.org/abs/2605.02789v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Mario Rodríguez Béjar, B. Romera-Paredes, Jose L. Hernández-Ramos
📅 **Published:** 2026-05-04  🏷️ **Categories:** cs.CR, cs.CL

### 💡 TL;DR
FunFuzz is an LLM-driven compiler fuzzer that combines prompt distillation, feedback-guided generation, and a multi-island evolutionary search to reduce prompt sensitivity and maintain semantic diversity during long fuzzing campaigns. On 24-hour GCC and Clang campaigns for C/C++, it achieves higher compiler coverage than prior LLM-based baselines and finds 119 unique compiler bugs, 80 of which were confirmed.

### 🏆 Key Contributions

### ⚙️ Methodology
FunFuzz is a generation-based fuzzing framework for structured compiler inputs, where the fuzzed artifacts are source programs. Each island maintains its own population and search history, and local evolution proceeds independently in parallel. The key fitness signal is incremental compiler coverage, described as richer than raw coverage because it is aggregated from multiple compilation configurations so that selected inputs reflect robust behavioral novelty rather than overfitting to one setting. ## 📄 [SCRIBE: Practical Static Binary Patching via Binary-Aware Recompilation of Decompiled Code](https://arxiv.org/abs/2605.02121v1)  `60% relevant`  `🌐 Full Text`

### 💡 TL;DR
SCRIBE is a static binary patching framework that lets users apply source-level patches to closed-source binaries by decompiling only the target functions, repairing decompiler-induced inaccuracies, and recompiling them in a binary-aware way so they can be safely reintegrated into the original executable. Across decompilers and optimization levels, it substantially improves recompilation correctness, enables patching 13/14 real CVEs without source code or original toolchains, and makes binary patching practical for both human users and LLMs.

### 🏆 Key Contributions
- Systematic characterization of why modern decompiler output is often not safely recompilable, including both syntactic failures and subtle semantic mismatches that prior work underexplored.
- Introduction of SCRIBE, a practical per-function binary patching framework based on binary-aware recompilation that uses metadata extracted from the original binary to repair and constrain recompilation.
- Techniques to preserve low-level binary compatibility properties during recompilation, such as stack layout, variable arrangement, and calling/interface assumptions, so patched functions can be relinked into the original binary.
- Cross-decompiler evaluation on Hex-Rays and Ghidra over GNU Coreutils and CGC binaries at multiple optimization levels, showing major improvements in recompilation correctness and fixing about 81% of previously incorrect Hex-Rays functions.
- Real-world security validation through source-level patching of 13 out of 14 reproducible CVEs in GNU Coreutils and GNU Binutils without access to original source or build environments.
- Usability validation via a controlled user study with 18 participants and an LLM-based evaluation showing that SCRIBE dramatically increases patching success for both humans and automated patch generators.

### ⚙️ Methodology
SCRIBE follows a decompile-patch-recompile-reintegrate workflow, but augments recompilation with binary-derived constraints so the resulting function remains compatible with the surrounding original binary. Instead of attempting whole-program lifting or direct assembly rewriting, it operates at the granularity of only the functions that need to be patched, reducing engineering effort and attack surface. The pipeline starts from a target binary and selected vulnerable functions, decompiles them to C using standard decompilers such as Hex-Rays or Ghidra, allows a user or automated system to apply a source-level patch, and then recompiles the modified code. The key novelty is 'binary-aware recompilation': SCRIBE extracts low-level facts from the original binary and uses them to compensate for the semantic inaccuracies commonly found in decompiled code. This addresses two main classes of problems. First, syntactic issues in decompiler output that would prevent successful compilation are repaired so the patched function can at least be rebuilt. Second, and more importantly, semantic issues that a compiler would not flag are corrected or constrained. These include cases where the decompiler loses type information, collapses or invents structure layouts, misidentifies local variables, or rewrites memory accesses in ways that would cause the compiler to generate a binary-incompatible stack or data layout. SCRIBE preserves critical low-level properties such as stack-frame organization and variable placement so that the recompiled function can interact correctly with unchanged code and data in the original executable. This binary-aware strategy is intentionally minimally invasive: it recompiles only the necessary functions and retrofits them back into the existing binary, avoiding the complexity of whole-program reassembly and avoiding manual assembly-level patching. The framework is also designed to support source-level patch authoring by humans, traditional patch repositories, and automated program repair or LLM-generated patches, since the interface exposed to the patch generator is C code rather than assembly or IR.

The evaluation spans recompilation correctness, real-world vulnerability patching, human usability, and LLM-assisted automation. For recompilation, the authors evaluate SCRIBE with both Hex-Rays and Ghidra on the latest GNU Coreutils and CGC binaries under four optimization levels, comparing decompiled-function recompilation outcomes with and without SCRIBE's binary-aware repairs. The headline result is that SCRIBE resolves about 81% of previously incorrect functions produced by Hex-Rays, with substantial correctness improvements across both decompilers and optimization settings. For security patching, the authors test reproducible real-world CVEs in GNU Coreutils and GNU Binutils, using only binaries and source-level patches, without the original source tree or build environment. SCRIBE successfully patches 13 of 14 CVEs without manual binary editing. For usability, a controlled user study with 18 participants of varying expertise compares standard binary patching/decompilation workflows against SCRIBE. With SCRIBE, participants achieved 100% patching success; without it, success was only 3.7%, indicating that naive decompilation-based patching is highly unreliable in practice. The paper also evaluates three state-of-the-art LLMs—GPT-5, Claude 4.5 Sonnet, and Gemini 2.5 Pro—as automated patch generators operating through SCRIBE. All three achieved 100% success with SCRIBE, versus 0% without it, suggesting that binary-aware recompilation is the missing infrastructure that allows source-level automated repair methods to transfer to binary-only settings.

### 🌱 Insights & Implications
The paper’s central insight is that the main blocker for practical source-level binary patching is not just compilation failure of decompiled code, but silent semantic divergence after recompilation. This is highly relevant to software maintenance and security because it shows why many apparently plausible decompile-edit-recompile workflows fail in deployment. SCRIBE demonstrates that preserving binary-level invariants during recompilation is enough to make decompiler output usable for patching in many cases, even when the recovered C is not a faithful reconstruction of the original source. Another important implication is that patching can be localized to individual functions, which is much more practical than whole-program lifting for maintainers and security responders. The user-study and LLM results suggest that the framework meaningfully lowers the expertise barrier, turning a reverse-engineering-heavy task into a source-level maintenance task. Likely limitations include dependence on decompiler output quality, incomplete recovery in the remaining failed cases, and challenges for binaries with especially complex control flow, aggressive optimizations, unusual ABI behavior, or heavy use of inline assembly and compiler-specific constructs. Open research questions include how to generalize binary-aware recompilation to more architectures and languages, how to formally verify semantic equivalence of recompiled patched functions, how to integrate automated vulnerability localization and APR end-to-end, and how to reason about security regressions introduced by patch recompilation. The work also suggests a new intersection between binary analysis and APR: future repair systems may operate over decompiled code, but require binary-derived constraints to ensure generated patches are actually deployable.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it directly addresses vulnerability repair and software maintenance in binary-only settings, where source code and original toolchains are unavailable—a common obstacle in real-world security operations. It advances automated program repair by providing the missing deployment layer that lets source-level patches, including LLM- or APR-generated patches, be applied safely to compiled binaries. For work on vulnerability detection and repair, SCRIBE offers a practical bridge from discovering a flaw in a binary to shipping a correct fix, and it highlights concrete low-level compatibility issues that any binary repair pipeline must handle.

A notable implication for software engineering research is improved reproducibility and maintainability of LLM-based systems. ACDL could function like an architecture description language for prompt pipelines, making context evolution reviewable, comparable across implementations, and easier to version or audit. This is especially relevant for long-lived agent frameworks where prompt logic becomes complex and brittle.

For security-oriented work, the language could help document where sensitive instructions, tool outputs, memory retrievals, or user-controlled content enter the context, which may support threat modeling, prompt injection analysis, or secure agent design reviews. However, the paper does not itself develop formal security analyses, static checking, or automated verification on top of ACDL; these remain open opportunities.

A limitation is the lack of rigorous user studies or quantitative evaluation of whether ACDL improves reproducibility, onboarding speed, defect rates, or implementation fidelity. Another open question is whether the language can scale cleanly to highly complex multi-agent systems with shared memory, asynchronous events, or rich retrieval pipelines. There is also room to connect ACDL with executable specifications, testing frameworks, or synthesis tools so that a context description can be validated against running code.

### 🎯 Why This Paper
This paper is relevant because it addresses a foundational software-engineering problem for LLM-based systems: how to specify, communicate, maintain, and compare the prompt/context architecture of complex agents. For a researcher interested in software maintenance, security, automated repair, and vulnerability-oriented work, ACDL offers a structured representation that could support specification-driven development, documentation, auditing, regression analysis of prompt logic, and future tooling for detecting or repairing context-construction flaws in agent pipelines.

## 📄 [CyberAId: AI-Driven Cybersecurity for Financial Service Providers](https://arxiv.org/abs/2605.01892v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** George Fatouros, Georgios Makridis, John Soldatos, Dimosthenis Kyriazis, Pedro Malo, George Kousiouris, Giannis Ledakis, Louiza Kachrimani, Panagiotis Rizomiliotis, Bruno Almeida, Despina Tomkou, Kostas Metaxas, Konstantinos Ilias, Christos Gkizelis, Ernstjan de Gooyert, Amin Babazadeh, Kostis Mavrogiorgos, Pepi Paraskevoulakou, Christos Xenakis, Giannis Chouchoulis, Konstantina Tripodi
📅 **Published:** 2026-05-03  🏷️ **Categories:** cs.AI, cs.CR, cs.IR

### 💡 TL;DR
This position paper proposes CyberAId, a hybrid multi-agent cybersecurity platform for financial institutions that combines SIEM/XDR telemetry with specialized LLM agents, privacy-preserving federated knowledge sharing, and bounded human oversight to improve SOC reasoning, compliance, and resilience. Its main claim is that regulated financial cybersecurity needs auditable, on-premise, multi-agent architectures rather than standalone LLMs or end-to-end neural systems.

### 🏆 Key Contributions
- Argues for a hybrid LLM-agent plus SIEM/XDR architecture as the appropriate design unit for financial-sector cybersecurity rather than monolithic LLM copilots.
- Defines a multi-agent platform architecture centered on a Main Agent/Central Router Agent, a reporting capability, and specialized cybersecurity subagents operating in a shared runtime.
- Introduces four falsifiable design principles: hybrid grounding, specialization and composition, federated knowledge sharing, and architectural trust.
- Proposes a methods stack combining partitioned retrieval-augmented generation, secure cross-institutional federated knowledge sharing, and optional capability packs such as digital twins, eBPF telemetry, and quantum token authentication.
- Frames four validation use cases in finance: client impersonation, anti-money laundering for payment service providers, retail-banking incident response, and high-frequency-trading resilience.
- Highlights skill-based agent adaptation as a research direction for continuously improving collective cyber defense across deployments.

### ⚙️ Methodology
CyberAId is presented as a model-agnostic, on-premise-deployable multi-agent platform for financial critical infrastructures. The architecture is layered: an integration and messaging layer ingests normalized events from enterprise SIEM/XDR systems; a domain-agents layer hosts all specialist LLM agents within a shared runtime; a knowledge and inference layer provides partitioned vector stores and model inference; and a presentation/HITL layer supports analyst interaction and dashboards. At the center is a Main Agent or Central Router Agent (CRA) that receives SIEM events, performs severity and impact assessment, dispatches tasks to specialist agents in parallel, aggregates findings expressed in standardized schemas, resolves conflicts via confidence calibration, and produces analyst-facing recommendations. A separate reporting capability provides direct access to specialists for compliance and incident-reporting workflows. The design emphasizes that agents do not replace SIEM/XDR telemetry but reason over it, preserving deterministic detection while adding cross-domain synthesis and regulatory mapping. Each specialist subagent has its own system prompts, knowledge partition, tool bindings, and finding schema, enabling explicit composition and auditability. Trust is enforced through a Security Context object propagated across agent boundaries and checked at retrieval and tool invocation time, partitioned RAG rather than unrestricted retrieval, direct library or CLI tool integration with deterministic interfaces, append-only audit logs, prompt-injection and output-sanitization guardrails, and bounded human-in-the-loop autonomy. Cross-institution collaboration is based on federated sharing of accumulated agent knowledge such as playbooks, skills, baselines, and findings, using techniques like federated learning, secure multiparty computation, and differential privacy so that institutions can benefit from sector-wide learning without sharing raw customer data. The platform is also designed to integrate optional capability packs including eBPF-based kernel telemetry, digital twins for adversarial validation, and quantum-based authentication/token mechanisms.

### 🧪 Experiments
The paper is a position and architecture paper rather than an empirical evaluation study, so it does not report benchmark datasets, baselines, metrics, or quantitative results for CyberAId itself. Instead, it motivates the design using prior empirical findings from the literature: enterprise SIEMs reportedly detect only about 21% of MITRE ATT&CK techniques; around two thirds of SOC teams cannot keep pace with alert volumes; frontier models can autonomously exploit 87% of one-day CVEs from textual descriptions; hierarchical multi-agent attack systems achieve a 4.3x improvement over single-agent baselines on zero-day exploitation; LLM-assisted static analysis can outperform traditional SAST tools on vulnerability detection; and domain-specific intrusion detection systems can reach F1 scores around 0.97. CyberAId is proposed to be validated in four representative financial use cases: client impersonation, AML for payment service providers, retail-banking incident response, and high-frequency-trading resilience, but no end-to-end validation results are yet included in the paper.

### 🌱 Insights & Implications
A key insight is that the core bottleneck in financial SOCs is reasoning rather than raw data ingestion, making multi-agent orchestration, tool grounding, and compliance-aware synthesis more important than simply attaching a powerful LLM to logs. The paper also usefully distinguishes narrow AI wins in cybersecurity such as vulnerability exploitation, detection, or patching from the broader platform requirements of regulated institutions: state persistence, tenant isolation, auditability, adversarial robustness, and regulatory reporting. For software security and vulnerability research, the notion of specialist agents with scoped knowledge, explicit finding schemas, and deterministic tool interfaces is especially relevant because it aligns with secure automation and traceable remediation pipelines. Another important implication is that privacy-preserving federation could support collective defense without centralizing sensitive financial data, though the practical effectiveness of skill transfer, privacy budgets, and poisoning resistance remains open. The work is limited by being conceptual: the architecture, principles, and use cases are promising, but there is not yet evidence on latency, false-positive reduction, robustness to prompt injection or knowledge poisoning, compliance utility, or operational ROI. Open questions include how to evaluate agent trustworthiness under adversarial conditions, how to measure the benefit of federated skill sharing, how to maintain calibrated confidence across heterogeneous agents, and how to safely adapt agent skills over time without introducing regressions or compliance risk.

### 🎯 Why This Paper
This paper is highly relevant because it connects LLM-based cybersecurity automation with the operational and regulatory realities of high-assurance environments, which maps well to interests in software engineering, maintenance, security, vulnerability detection, and automated repair. Although it is not a code-repair paper, it advances a platform view that could directly inform research on agentic vulnerability triage, patch generation, audit-grade remediation workflows, secure tool orchestration, and federated learning of repair or detection skills across organizations.
