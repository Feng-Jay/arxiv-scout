## 📄 [Calibration Without Comprehension: Diagnosing the Limits of Fine-Tuning LLMs for Vulnerability Detection in Systems Software](https://arxiv.org/abs/2606.20502v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Arastoo Zibaeirad, Marco Vieira
📅 **Published:** 2026-06-18  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
This paper introduces CWE-Trace, a temporally split and manually curated Linux-kernel vulnerability benchmark to test whether LLMs truly reason about vulnerabilities or merely calibrate to benchmark patterns. Across 8 base LLMs and 15 LoRA-tuned variants, the authors find that contamination offers no measurable benefit and fine-tuning mostly shifts confidence thresholds rather than improving genuine vulnerability understanding, leaving detection near chance and CWE classification extremely weak.

### 🏆 Key Contributions
- Introduces CWE-Trace, a benchmark of 834 manually curated Linux kernel samples (417 vulnerable-patched pairs) spanning 74 CWEs, with preserved contextual code dependencies and a strict pre-2025/post-2025 temporal split.
- Proposes two diagnostic metrics—Directional Failure Index (DFI) and Hierarchical Distance and Direction (HDD)—to analyze systematic bias and taxonomic error structure beyond simple accuracy.
- Performs a multi-backbone, multi-dataset fine-tuning study with 8 vanilla LLMs and 15 LoRA variants trained on five major vulnerability datasets, isolating backbone effects from dataset effects.
- Provides a function-level contamination audit showing that nominal dataset overlap rarely provides usable memorization signal because vulnerable functions are often absent, cross-mapped, or mislabeled.
- Empirically argues for 'calibration without comprehension': fine-tuning changes output distributions and thresholds, but does not alter the underlying decision policy or instill robust security reasoning.

### ⚙️ Methodology
The authors build CWE-Trace by extracting Linux kernel vulnerability-fix commits and manually constructing vulnerable and patched code instances that preserve the context needed to understand systems-level flaws. Unlike snippet-only datasets, the benchmark includes both single-function (L1) and multi-function/multi-file (L2) samples, retaining relevant macros, structs, globals, and interacting functions while excluding unrelated changed code. From 1,020 raw samples, two reviewers validate 834 final samples, organized as 417 vulnerable-patched pairs, and split them temporally into a historical Primary Benchmark Dataset (PBD, up to 2024) and a leakage-free Leakage Free Dataset (LFD, 2025 onward). The benchmark covers 74 CWE types under the CWE-1000 hierarchy.

The evaluation protocol includes three zero-shot tasks: non-targeted detection, targeted detection, and CWE classification. Non-targeted detection measures whether a model can distinguish vulnerable from patched code without guidance. Targeted detection evaluates whether the model can verify vulnerability presence under a hypothesized CWE. CWE classification tests whether the model can identify the correct weakness type, including hierarchical partial correctness. To analyze failure modes, the paper introduces DFI, which captures whether a model systematically prefers labeling vulnerable or patched samples as positive, and HDD, which measures how far a predicted CWE is from the true one in the taxonomy and whether the error moves toward broader or unrelated classes.

For model comparison, the authors evaluate eight vanilla LLMs and 15 LoRA fine-tuned models. The fine-tuned models are built from three backbones—Qwen3-4B, DeepSeek-R1-32B, and Llama3.1-8B—each trained separately on five vulnerability datasets: Devign, LineVul, PrimeVul, MegaVul, and VDISC. A core methodological element is the contamination analysis: rather than assuming benchmark overlap implies memorization, the authors inspect whether the exact vulnerable functions and labels in overlapping CVEs actually align between training corpora and CWE-Trace. This function-level audit is used to explain whether any observed gains are due to memorization versus actual generalization.

### 🧪 Experiments
The experiments use 834 Linux kernel samples split into 628 historical PBD samples and 206 post-cutoff LFD samples. The model pool consists of 8 untuned LLMs plus 15 LoRA-tuned variants derived from three backbones and five external vulnerability datasets. Evaluation spans non-targeted vulnerability detection, targeted detection, and fine-grained CWE classification. Metrics include standard detection accuracy, pair-wise vulnerable-versus-patched consistency, coverage/ranking behavior, and the proposed DFI and HDD metrics for directional bias and hierarchical error analysis.

The main quantitative findings are stark. The best binary detection performance reaches only 52.1%, just 2.1 percentage points above chance, indicating that even the strongest model is barely useful for practical vulnerability detection on this benchmark. Exact CWE identification is even weaker, with Top-1 ranking accuracy below 1.3%, showing that models almost never identify the precise weakness correctly. DFI values range from -85.5 to +94.8 percentage points, demonstrating strong and stable directional biases in model decisions rather than balanced reasoning. These directional failure patterns persist across both historical and leakage-free splits, suggesting they are properties of the backbone models rather than contamination artifacts.

The contamination audit finds that 84% of nominally contaminated samples provide no usable memorization signal because the vulnerable function is missing or mismatched across datasets, and about 31% of contaminated samples carry CWE misclassification. As a result, overlap with training corpora does not translate into measurable performance gains. Fine-tuning likewise fails to substantially improve core reasoning: it mostly shifts decision thresholds/output calibration while preserving the same backbone-specific decision tendencies. One notable result is that DeepSeek-R1, the weakest backbone for binary detection, shows the largest gains in coarse CWE classification, supporting the claim that vulnerability detection and vulnerability understanding are decoupled capabilities.

### 🌱 Insights & Implications
The paper’s central insight is that current LLMs can be made to look better calibrated for vulnerability tasks without actually becoming better reasoners about program security. Fine-tuning adjusts surface behavior—such as how often a model predicts 'vulnerable'—but leaves intact systematic directional priors and near-chance detection performance. This challenges a common assumption in LLM-for-security work that benchmark gains after supervised tuning imply deeper vulnerability understanding.

A second important implication is methodological: contamination should be assessed at the function and semantic level, not just by CVE overlap. The finding that most apparent overlap does not enable memorization undermines simplistic claims that historical benchmark performance is primarily explained by training leakage. At the same time, the poor performance on the post-cutoff leakage-free split confirms that removing leakage does not reveal hidden robust reasoning either.

For vulnerability research, the disconnect between binary detection and CWE classification suggests that 'finding something suspicious' and 'understanding what is wrong' are distinct capabilities that should be evaluated separately. The paper also highlights the importance of contextual, multi-file reasoning in systems code, where vulnerabilities may not be visible in isolated functions. Limitations include the focus on Linux kernel C code, zero-shot prompting only, and LoRA fine-tuning rather than broader adaptation strategies. The authors suggest that structured prompting, retrieval augmentation, or alternative supervision schemes may be worth testing, but their results imply that benchmark design and diagnostic metrics must improve before claiming meaningful security reasoning.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and vulnerability repair because it rigorously evaluates whether LLM-based detectors actually learn security semantics in realistic systems software settings. It advances the field by providing a higher-fidelity benchmark, contamination-aware methodology, and failure diagnostics that help researchers distinguish superficial calibration gains from genuine reasoning—critical for designing trustworthy automated vulnerability detection and future repair systems.

## 📄 [Secure Coding Drift in LLM-Assisted Post-Quantum Cryptography Development: A Gamified Fix](https://arxiv.org/abs/2606.19474v1)  `90% relevant`  `🌐 Full Text`

📅 **Published:** 2026-06-17  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
This paper argues that in LLM-assisted post-quantum cryptography development, the main risk is not just isolated insecure code generations but a gradual behavioural degradation of developers’ secure coding practices, which the authors call Secure Coding Drift in PQC (SCD-PQC). It proposes a conceptual mitigation framework that uses gamification, adversarial evaluation, feedback, and security scoring to turn LLMs into active security co-pilots for safer PQC implementation.

### 🏆 Key Contributions
- Introduces Secure Coding Drift in PQC (SCD-PQC), a socio-technical vulnerability model describing how sustained reliance on LLM-generated code can progressively erode secure coding behaviour in post-quantum cryptography development.
- Reframes LLM-related software security risk from a static code-vulnerability perspective to a longitudinal behavioural phenomenon rooted in human-AI interaction, trust miscalibration, and cognitive offloading.
- Defines a three-stage conceptual model of drift in PQC settings: cognitive offloading of security-critical reasoning, trust calibration failure toward plausible LLM outputs, and behavioural reinforcement of insecure PQC coding patterns.
- Characterises a threat model for AI-assisted PQC development that includes insecure-but-plausible code generation, adversarial prompt manipulation, poisoned training data, and developer over-reliance.
- Proposes a gamified, LLM-augmented secure coding framework with adversarial evaluation, behavioural feedback, and security scoring intended to counteract secure coding drift and promote long-term secure development habits.

### ⚙️ Methodology
The paper is primarily conceptual and framework-oriented rather than an empirical system paper. Its methodology begins by identifying a gap in prior work: most research on LLM-assisted coding security focuses on static vulnerabilities in generated code, whereas the authors argue that repeated human-AI interaction creates cumulative behavioural risk. To address this, they define SCD-PQC as a progressive deviation from secure PQC implementation practices caused by sustained reliance on LLM outputs in AI-mediated workflows.

The proposed SCD-PQC model has three interacting stages. First, cognitive offloading occurs when developers delegate difficult PQC tasks such as modular arithmetic optimisation, noise sampling, and parameter handling to LLMs, reducing direct engagement with security-critical logic. Second, trust calibration failure emerges because repeated exposure to syntactically correct, plausible outputs leads developers to overestimate the security correctness of LLM-generated cryptographic code even when constant-time execution, side-channel resistance, or correct randomness handling are not guaranteed. Third, behavioural reinforcement occurs when accepted AI-generated code patterns are repeatedly reused, normalising insecure practices such as unsafe parameterisation, misuse of randomness, or leakage-prone implementations.

The paper further describes the threat setting as involving an adversarial or partially untrusted LLM integrated into the development pipeline. Security degradation can arise from insecure code suggestions, prompt injection, poisoned training data, and weakened developer validation habits. The framework-level mitigation proposed by the authors is a gamified, LLM-augmented secure coding process that embeds continuous behavioural intervention into development workflows. Core elements include adversarial evaluation of generated PQC code, behavioural feedback loops that encourage developers to actively inspect and justify implementation choices, and security scoring mechanisms that reward secure coding behaviour. The intended effect is to move the LLM from being a passive code generator to an active co-pilot that helps detect drift, reinforce secure habits, and maintain cryptographic assurance over time.

### 🧪 Experiments
The provided paper text does not report a completed empirical evaluation, benchmark, or quantitative experiment. There are no datasets, baselines, metrics, or numerical results described in the extracted text. Instead, the paper presents a conceptual vulnerability model and a mitigation framework for future implementation and study. The discussion is motivated by prior literature on insecure LLM-generated code, AI-assisted development behaviour, PQC implementation complexity, and side-channel risks, but the current manuscript appears to stop short of validating the framework experimentally.

### 🌱 Insights & Implications
A key insight is that AI-assisted secure coding failures in cryptographic software may be cumulative and behavioural, not just technical defects in individual code snippets. This is especially important for PQC, where many implementation flaws do not break functionality and therefore evade ordinary testing while still undermining side-channel resistance or constant-time guarantees.

For researchers in software security and automated repair, the paper suggests that future tools should model developer behaviour, trust, and verification habits rather than only scan outputs for bugs. It also highlights a strong connection between secure coding assistance and adversarial robustness: poisoned data, prompt injection, and insecure learned patterns can propagate through the human developer via trust and repeated reuse. An important limitation is the lack of empirical validation in the provided text; the SCD-PQC model and gamified mitigation are plausible and well-motivated, but their effectiveness, measurable indicators of drift, and deployability in real software teams remain open questions. Another open issue is how to operationalise security scoring for PQC-specific properties like constant-time behaviour, randomness safety, and parameter correctness in a way that is both automatable and trustworthy.

### 🎯 Why This Paper
This paper is highly relevant because it sits at the intersection of software engineering, software maintenance, software security, and AI-assisted coding in a particularly sensitive domain: post-quantum cryptography implementation. For research on vulnerability detection and repair, it broadens the problem definition from finding insecure code to detecting and mitigating the behavioural conditions that cause insecure code to recur. It is especially useful if your work involves LLM-based development tools, secure code generation, or repair pipelines, because it argues that effective security support must account for human-AI interaction, trust calibration, and long-term secure coding resilience rather than only one-shot vulnerability fixes.

## 📄 [Phoenix: Safe GitHub Issue Resolution via Multi-Agent LLMs](https://arxiv.org/abs/2606.20243v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Kipngeno Koech, Muhammad Adam, Baimam Boukar Jean Jacques, Joao Barros
📅 **Published:** 2026-06-18  🏷️ **Categories:** cs.SE, cs.MA

### 💡 TL;DR
Phoenix is a production-oriented multi-agent LLM system for resolving GitHub issues safely, from planning to pull-request creation, with explicit regression controls based on baseline-aware testing. It achieves strong oracle resolution on a curated SWE-bench Lite slice and preserves correctness on a real-world pilot, but its main weakness is code localization, with roughly half of generated PRs placing fixes in the wrong files or paths.

### 🏆 Key Contributions
- Introduces Phoenix, a six-agent GitHub issue resolution pipeline spanning planning, reproduction, coding, testing, failure analysis, and PR generation, coordinated by a label-driven webhook state machine.
- Proposes a baseline-aware evaluation strategy that separates pre-existing test failures from newly introduced regressions, enabling correctness-preservation assessment on repositories with broken CI.
- Defines seven layered safety mechanisms for production deployment, including path-traversal checks, workflow-write blocking, content sanitization, bounded retries, concurrency serialization, label exclusivity, and GitHub token refresh.
- Reports empirical results on both benchmarked and live repository settings, including a SWE-bench Lite slice and a 42-issue real-world pilot across 14 repositories.
- Provides deployment-focused lessons about practical failure modes of autonomous coding agents, such as WAF filtering, expiring credentials, CI permission constraints, and flaky test infrastructure.

### ⚙️ Methodology
Phoenix is designed as a closed-loop multi-agent system for automating GitHub issue handling while preserving repository safety. A central orchestrator manages six specialized agents through GitHub webhook events and issue labels used as a persistent state machine rather than a database. The Planner consumes sanitized issue text, file-tree context, relevant code excerpts, recent comments, and optional visual context, then emits a structured JSON plan containing target files, implementation steps, test strategy, and risk level. Relevant files are selected through a keyword-overlap heuristic that biases toward deeper source files and adapts excerpt budgets for large repositories to stay within prompt and gateway limits. The optional Reproducer attempts to synthesize a failing test on the base branch that demonstrates the issue; this stage is intentionally non-blocking so the pipeline can continue even when no reproduction is found. The Coder receives the plan, current file contents, and any feedback from prior failed attempts, and outputs complete replacement contents for all modified files plus a commit message in structured JSON. The system enforces full-file outputs, style adherence, language consistency, and blocks invalid writes such as path traversal or changes under `.github/workflows/`. The Tester installs dependencies, runs the project test suite, and compares post-change failures against a baseline run on the unmodified branch by stashing Phoenix's changes, collecting the baseline failing set, restoring the patch, and computing whether any previously passing tests now fail. This makes correctness preservation depend on the absence of new failures rather than absolute test-suite success. When new failures occur, a Failure Analyst interprets test logs and failure identifiers to generate root-cause guidance for the next coding retry; retries are capped at two rounds and terminated early on no-progress identical patches. Finally, the PR Agent drafts and opens a pull request with issue linkage for human review. The overall design prioritizes containment and auditability: Phoenix never merges autonomously, all outputs are surfaced as PRs, and multiple runtime safeguards address real deployment hazards.

### 🧪 Experiments
The evaluation covers both a benchmark-style setting and a live-repository pilot. On a 24-instance curated slice of SWE-bench Lite spanning 12 repositories and executed through the actual production webhook path, the paper reports 75% oracle resolution in the abstract, while the contributions section further breaks results down as 45.7% oracle resolution and 94.9% oracle-eligible resolution on the webhook path; the authors explicitly caution that this curated subset is not directly comparable to full leaderboard numbers. The benchmark setup enables the Reproducer and assesses issue resolution through test outcomes, but the stricter criterion requiring the synthesized reproducer test to pass after the fix is treated as separate from the SWE-bench oracle score. A second study evaluates 42 real GitHub issues across 14 repositories. The primary metric there is correctness preservation (CP), defined by baseline-aware regression checking: if the repository already has failing tests, Phoenix is still considered safe so long as it introduces no new failures. On this pilot, Phoenix achieves 100% CP, with mean runtime of 122 seconds on the hard tier. Manual inspection of generated PRs finds that about half are well-targeted fixes, while the other half edit incorrect locations despite preserving existing correctness. The paper does not position these results as a direct benchmark win; instead it emphasizes real deployment execution, auditable PR output, and no pass-to-pass regressions on successful runs.

### 🌱 Insights & Implications
The paper's central insight is that for practical repository automation, correctness preservation and safe deployment may matter more than raw benchmark resolution rate. The baseline-aware testing scheme is particularly relevant to software maintenance and repair because it supports operation on imperfect repositories whose CI is already broken, a common real-world condition that standard pass/fail evaluation obscures. The multi-agent decomposition also shows how issue resolution can be modularized into planning, reproduction, implementation, and diagnosis stages, which could be useful for future APR and vulnerability-repair pipelines. A key limitation is localization: manual review shows many unsuccessful PRs are structurally plausible but modify the wrong files or paths, indicating that planning and repository grounding remain the main bottleneck; the authors suggest retrieval as the next fix. Another limitation is evaluation scope: the SWE-bench subset is small and curated, so the reported numbers are not directly comparable to leaderboard results, and the real-world pilot focuses on correctness preservation rather than merge acceptance or long-term fix quality. The safety mechanisms are motivated by observed production failures, but their trigger frequencies were reconstructed from logs rather than instrumented prospectively, so quantitative hazard analyses are lower bounds. For security-oriented research, the paper is notable because it frames autonomous code changes as a safety problem, but it does not deeply study vulnerability introduction, secure patch semantics, or adversarial issue content beyond prompt sanitization and write restrictions.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated program repair, and safe code modification because it tackles end-to-end GitHub issue resolution under real operational constraints rather than only offline benchmarks. Its baseline-aware regression criterion is especially useful for researchers working on bug fixing, vulnerability repair, and repository automation in noisy environments with broken tests. The safety-first architecture, bounded repair loop, and auditable PR-only deployment model offer concrete design patterns for building trustworthy APR or vulnerability-fixing agents. It also highlights an important open problem for your research area: even when LLM agents avoid regressions, accurate fault localization and patch placement remain major obstacles to reliable automated repair.

### 💡 TL;DR
This paper introduces SolidityBench, a 5,470-sample repository-level benchmark for generating complete Solidity smart contracts from natural language, and SolidityScore, a Solidity-aware semantic evaluation metric. Experiments across prompting, retrieval, and fine-tuning show that general code LLMs struggle with Solidity-specific structure and security-related constructs, while supervised fine-tuning delivers the largest gains and retrieval-augmented generation is the strongest non-parametric approach.

### 🏆 Key Contributions
- Introduces SolidityBench, a large-scale benchmark of 5,470 repository-level Solidity smart contracts paired with natural language descriptions, curated from OpenZeppelin, Synthetix, and verified Etherscan contracts.
- Proposes SolidityScore, a Solidity-specific semantic evaluation metric that prioritizes domain-critical constructs such as modifiers, contract declarations, reserved keywords, and Solidity-specific semantics rather than relying only on token overlap.
- Provides the first systematic comparison of repository-level Solidity code generation across zero-shot prompting, Chain-of-Thought, in-context learning, retrieval-augmented generation, and supervised fine-tuning.
- Empirically shows that general-purpose code LLMs have systematic structural weaknesses on Solidity, that retrieval helps more than other prompting-based strategies, and that supervised fine-tuning best internalizes domain constraints.

### ⚙️ Methodology
The study builds a repository-level Solidity generation benchmark in which each sample consists of a natural language specification and a full smart-contract repository-level code target. The dataset is collected from authoritative real-world sources including OpenZeppelin, Synthetix, and verified Etherscan contracts, with emphasis on realistic inheritance structures, inter-contract dependencies, external library usage, and Solidity design patterns. To evaluate generated code, the authors design SolidityScore, a domain-aware text-based semantic metric intended to better reflect Solidity correctness than BLEU-like lexical overlap or execution-based metrics that require full reconstruction of compilation environments. SolidityScore uses a Solidity-adapted encoder and domain-weighted token matching so that semantically important language constructs such as modifiers, events, mappings, addresses, and contract declarations receive greater emphasis. The empirical methodology compares several representative code LLMs, including Qwen2.5-Coder, DeepSeek-Coder, and CodeLlama, under multiple adaptation paradigms: zero-shot generation from the requirement only; Chain-of-Thought prompting to elicit intermediate reasoning; in-context learning with varying numbers of examples; retrieval-augmented generation that fetches similar Solidity examples to condition generation; and supervised fine-tuning on the benchmark data so the model parameters absorb Solidity-specific syntax and design constraints. The study is organized around research questions covering baseline capability, effectiveness of prompting-based adaptation, comparison with fine-tuning, reliability of SolidityScore versus BLEU, and compilability/error analysis.

### 🧪 Experiments
The experimental setup evaluates representative code LLM families—Qwen2.5-Coder, DeepSeek-Coder, and CodeLlama—on SolidityBench using both standard lexical/semantic metrics and the proposed SolidityScore, along with compilability analysis. The benchmark contains 5,470 repository-level specification-to-code pairs and targets full-contract generation rather than function completion. Baseline conditions include zero-shot prompting, followed by Chain-of-Thought, in-context learning with multiple examples, retrieval-augmented generation, and supervised fine-tuning. Key findings reported in the paper are qualitative but clear: zero-shot performance is poor, with substantial structural deficiencies in contract layout and Solidity-specific constructs; Chain-of-Thought and other prompting methods produce only limited improvements; in-context learning degrades when more than two examples are included, suggesting context saturation; retrieval-augmented generation is the strongest non-parametric method; and supervised fine-tuning yields the largest overall gains in semantic correctness and domain alignment. The paper also reports that compilation remains difficult across approaches, with recurring syntax, dependency, and environment-related errors, reinforcing that repository-level Solidity generation is substantially harder than standard GPL code-generation benchmarks. The abstract and provided text do not include concrete metric values, but the comparative ordering of methods is explicit: SFT > RAG > other prompting strategies, with ICL deteriorating beyond two shots.

### 🌱 Insights & Implications
A central insight is that repository-level Solidity generation stresses capabilities that are underrepresented in mainstream code-generation benchmarks: inheritance-aware design, inter-contract dependencies, compiler-version sensitivity, audited-library usage, and security-conscious constructs. The paper argues that standard execution-based evaluation is often too brittle for this setting because logically reasonable outputs may still fail to compile without an exact dependency environment, while generic text metrics underweight the Solidity-specific semantics that matter most. Another important observation is that retrieval helps because it exposes the model to domain patterns without changing parameters, but it still cannot fully overcome the mismatch between general pretraining and Solidity’s specialized constraints. Fine-tuning is more effective because it internalizes these constraints directly, improving structural regularity and semantic fidelity. A limitation is that the benchmark appears focused on specification-to-code alignment and semantic/textual correctness rather than full security assurance; generated contracts may still contain subtle vulnerabilities even when they score well semantically. More broadly, the work implies that domain-specific code generation for security-critical software will likely require domain-native datasets, domain-aware metrics, and adaptation beyond prompt engineering alone.

### 🎯 Why This Paper
This paper is highly relevant because it studies LLM-based generation in Solidity, a security-critical software domain closely connected to vulnerability-prone code, reliability, and repair-like correctness constraints. For research in software engineering, maintenance, security, automated program repair, vulnerability detection, and vulnerability repair, it contributes a realistic benchmark, a domain-aware evaluation metric, and empirical evidence about which adaptation strategies most effectively improve correctness in specialized code generation—insights that can transfer to secure code synthesis, vulnerability-aware repair, and evaluation of LLMs on domain-specific repositories.

## 📄 [DISARM: Target Electronic Device Informed Mitigation of Software Runtime Side-Channel Vulnerabilities](https://arxiv.org/abs/2606.19807v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Tasneem Suha, Tanzim Mahfuz, Rima Asmar Awad, Prabuddha Chakraborty
📅 **Published:** 2026-06-18  🏷️ **Categories:** cs.CR

### 💡 TL;DR
DISARM is a hardware-in-the-loop framework for detecting and mitigating software timing side-channel vulnerabilities using execution-time measurements from the actual target device and an explicit attacker threat model. Across 22 C/C++/Java benchmarks on five embedded/edge devices, it produces more precise fixes than device-agnostic approaches, reducing execution-time and code-size overhead while preserving correctness.

### 🏆 Key Contributions
- Introduces a joint hardware-software mitigation framework that quantifies timing side-channel leakage using real target devices instead of architecture-agnostic assumptions or simulation alone.
- Defines attacker-aware vulnerability metrics and uses them to decide whether secret-dependent branches and loops require mitigation under a given threat model.
- Provides automated, targeted repair algorithms for Branch-on-Secret and Loop-on-Secret timing leaks that avoid both over-fixing and under-fixing.
- Implements an end-to-end toolchain supporting C, C++, and Java and integrates it into embedded software development workflows.
- Evaluates the approach on 22 standard sensitive-code benchmarks across five embedded/edge platforms, showing better efficiency and correctness than constant-time programming, PENDULUM, and DifFuzzAR.

### ⚙️ Methodology
DISARM combines static and dynamic analysis with hardware-in-the-loop timing measurement to repair runtime side-channel vulnerabilities in source code. It focuses on the two dominant leak patterns: Branch on Secret (BoS), where a secret-dependent conditional changes control flow, and Loop on Secret (LoS), where a secret influences loop bounds or iteration counts. The framework first identifies candidate secret-dependent branches and loops from the program structure. It then deploys and executes the program on the actual target embedded device to obtain timing values for relevant paths and blocks, rather than relying only on compiler-level approximations or simulated microarchitectures. DISARM also models the attacker’s timing capability through a threat model that distinguishes weaker and stronger adversaries based on window awareness, measurement resolution, system knowledge, physical access, and input manipulation capability. For BoS, the framework estimates the timing imbalance between the two secret-dependent branch paths up to their common post-dominator and compares that imbalance with the attacker’s effective timing sensitivity. For LoS, it estimates per-iteration timing differences and derives how many secret-controlled iterations are needed before an attacker can reliably distinguish them. Using these hardware-specific and threat-aware measurements, DISARM synthesizes repairs that add only the amount of balancing work needed on the target device. Instead of enforcing globally constant-time behavior, it applies targeted padding or path balancing to vulnerable branches and loops so that observed timing differences fall below the threat threshold. This produces device-specific software fixes, reducing unnecessary extra instructions and avoiding failures caused by inaccurate timing assumptions. The implementation supports C, C++, and Java, enabling automated detection, localization, and repair in the build flow for embedded applications.

### 🧪 Experiments
The evaluation uses 22 standard benchmarks containing sensitive computations and timing-leak patterns, implemented in C, C++, and Java. DISARM is tested on five real embedded/edge devices: Jetson Nano (ARM Cortex-A57 MP Core), Jetson Orin Nano (ARM Cortex-A78AE), Jetson AGX Xavier (Carmel ARM v8.2), Jetson Xavier NX (Carmel ARM v8.2), and Raspberry Pi Quad-core Cortex-A72. The comparisons include state-of-the-art timing-side-channel mitigation systems PENDULUM and DifFuzzAR, as well as constant-time programming solutions. The main evaluation criteria are correctness of vulnerability mitigation, execution-time overhead, and code-size overhead; the paper also discusses performance and energy-efficiency impacts. Results show that DISARM consistently outperforms PENDULUM and DifFuzzAR by generating more precise hardware-aware fixes, avoiding over-fixing and under-fixing, and achieving lower runtime and binary-growth overheads while maintaining correctness across the tested devices. The paper’s central quantitative conclusion is that device-informed repair yields better efficiency than device-agnostic balancing across all five platforms and the full 22-benchmark suite.

### 🌱 Insights & Implications
The paper’s main insight is that timing-side-channel mitigation is fundamentally hardware- and attacker-dependent: the same source-level fix can be insufficient on one device and wasteful on another because execution-time differences depend on real microarchitectural behavior. By grounding mitigation in actual device measurements, DISARM shows that many prior constant-time or path-balancing approaches are too coarse for embedded deployment. This has important implications for secure software maintenance and repair: side-channel fixes should be synthesized relative to the deployment platform and threat assumptions, not just the source code. The work also highlights a practical middle ground between pure detection tools and heavy-handed constant-time transformations, enabling targeted repairs with better performance. Likely limitations include the need for access to the deployment hardware during analysis, possible sensitivity to compiler settings or firmware/environment changes, and a focus on runtime/timing leaks primarily through secret-dependent branches and loops rather than the full spectrum of microarchitectural side channels. An open question is how well such device-specific fixes generalize across software updates, compiler optimizations, and hardware revisions, and whether the methodology can be extended to broader classes of leaks or integrated with automated secure code generation and repair systems.

### 🎯 Why This Paper
This paper is highly relevant to software security, software maintenance, vulnerability detection, and vulnerability repair because it treats timing side-channel flaws as repairable software vulnerabilities and proposes an automated mitigation workflow. It is especially valuable for research on automated program repair and secure maintenance of embedded software: DISARM goes beyond vulnerability detection to synthesize correctness-preserving, deployment-aware fixes, offering a concrete model for hardware-informed repair of security weaknesses in real systems.

The framework adopts a PyTorch-inspired interface design: Agent and Workflow expose a forward(session) -> session contract, Session.to(backend) expresses placement, and Memory is treated as persistent agent-bound state instead of hidden prompt text. Additional abstractions include Sandbox for execution environment placement, Tool for external actions, Selector for runtime-routed control flow decisions, and Memory for recall/commit boundaries that enter the runtime record. The architecture is intentionally connective rather than replacement-oriented: graph runtimes, tracing systems, MCP-style tool protocols, sandbox providers, and memory backends can all remain external, while their effects are captured as evidence packets inside Session. The design goal is auditable composition, meaning that the same object used during execution is also the object used for inspection and replay.

### 🧪 Experiments
The paper does not present broad quantitative experiments or comparative benchmarking. Instead, it reports controlled runtime properties and an audited milestone-based validation protocol. The evaluation artifacts include lineage export, local sandbox execution evidence, workflow transcripts, focused tests, visual QA, claim ledgers, and memory source audits. There are no reported benchmark datasets, standard baselines, or aggregate task-performance metrics such as accuracy, success rate, repair rate, or latency comparisons against systems like AutoGen, LangGraph, or OpenAI Agents SDK. The authors explicitly limit claims to runtime-state properties—inspectability, replayability, composability, backend awareness, and evidence recording—while deferring live-provider quality, optional-backend availability, memory quality, human preference studies, and cross-system quantitative comparisons to future work.

### 🌱 Insights & Implications
The paper’s strongest insight is that many current agent-system problems stem not from weak reasoning alone but from poor runtime-state boundaries: critical evidence about tools, branching, memory, and workspace effects is often lost across side channels. By making Session the ordinary program value, OpenRath reframes observability and reproducibility as programming-model properties rather than post hoc tracing features. This is particularly relevant for multi-agent and long-horizon workflows where provenance and replay matter. A key limitation is that the work is largely architectural and conceptual; it does not yet establish whether the session-centered model improves downstream task quality, developer productivity, debugging efficiency, or operational robustness relative to existing frameworks. Another open question is scalability: the paper does not yet quantify storage overhead, replay cost, merge semantics under complex branching, or the impact of compression and memory policies on audit fidelity. Its implications for related research are significant, especially for agent reliability, provenance-aware orchestration, secure tool use, and systems that require explicit evidence trails for evaluation or governance.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because these areas increasingly rely on agentic workflows that must be auditable, reproducible, and evidence-preserving. OpenRath does not directly solve repair or security tasks, but it offers an important systems substrate for building trustworthy coding and security agents: explicit branch provenance, tool-effect tracking, sandbox-aware execution, and replayable state are exactly the properties needed to inspect code modifications, validate repair attempts, trace vulnerability analysis steps, and support secure automated development pipelines.
