## 📄 [RAVEN: Agentic RAG for Automated Vulnerability Repair](https://arxiv.org/abs/2606.22647v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Varun Gadey, Zijie Liu, Alexandra Dmitrienko
📅 **Published:** 2026-06-21  🏷️ **Categories:** cs.CR, cs.LG, cs.SE

### 💡 TL;DR
RAVEN is a locally deployable, open-source LLM-based automated vulnerability repair framework that combines agentic retrieval-augmented generation, repository-wide dependency retrieval, and iterative patch refinement to fix diverse real-world CVEs. Across 160 CVEs spanning multiple CWE types and both Java and C, it achieves an 83.13% repair success rate and outperforms prior state-of-the-art approaches while maintaining negligible cost.

### 🏆 Key Contributions
- Introduces RAVEN, a unified agentic RAG framework for automated vulnerability repair that targets diverse real-world vulnerabilities rather than only localized memory-safety bugs.
- Designs a multi-faceted retrieval pipeline that uses lexical, syntactic, and semantic vulnerability cues to retrieve historically relevant fixes from prior CVEs and guide patch generation.
- Adds a dedicated Curator Agent that inspects the target repository to recover cross-file dependencies and root-cause context needed for non-local repairs.
- Proposes a controlled iterative repair loop with patch review and static-analysis feedback to refine patches and reduce hallucinations, partial fixes, and newly introduced flaws.
- Demonstrates strong cross-language and out-of-distribution generalization using medium-sized open-source LLMs in a fully local, low-cost deployment setting.
- Evaluates on 160 real-world CVEs across 10 CWE types, showing 83.13% overall repair success, 86.21% cross-language success, and 87.50% success on unseen CWE categories.

### ⚙️ Methodology
RAVEN frames automated vulnerability repair as an agentic retrieval-augmented generation pipeline coupled with iterative validation and refinement. First, given a vulnerable program instance, the system retrieves historically relevant vulnerability-fix examples using multiple complementary signals: lexical, syntactic, and semantic cues associated with the target vulnerability. This retrieval stage is intended to ground patch generation in prior real-world fixes and reduce unsupported or hallucinatory generations. Beyond historical fix retrieval, RAVEN includes a Curator Agent that scans the target repository to identify root-cause information and cross-file dependencies that are not visible from the local vulnerable snippet alone. This is especially important for vulnerabilities such as authorization, injection, and input-handling flaws where the correct repair may depend on project-wide control flow, access-control conventions, or helper APIs defined elsewhere. The retrieved fix exemplars and repository context are then supplied to an open-source medium-sized LLM, such as Gemma-4-26B or Nemotron-3-30B, for patch generation in a fully local environment. After generating an initial patch, RAVEN applies a controlled iterative repair process: a Patch Reviewer module and static-analysis feedback inspect the patch for correctness, incompleteness, or newly introduced issues, and the system uses this feedback to revise the patch in subsequent iterations. The overall design therefore combines external historical knowledge, project-specific contextual retrieval, and automated patch critique to improve robustness, generalization, and repair quality across languages and vulnerability families.

### 🧪 Experiments
The evaluation uses 160 real-world CVE vulnerabilities covering 10 diverse CWE types and two programming languages, specifically including Java and C. The experiments are designed not only for in-distribution repair but also for more challenging settings: unseen CWE categories, cross-language evaluation, and out-of-distribution scenarios. The paper compares RAVEN against existing state-of-the-art automated vulnerability repair frameworks, including recent LLM-based systems discussed in the paper such as PatchAgent, APPATCH, and SAN2PATCH, as well as the broader AVR landscape. The main reported metric is repair success rate. RAVEN achieves an overall repair success rate of 83.13% on the 160-CVE benchmark, outperforming all compared prior frameworks. It also reports 86.21% success in cross-language evaluation and 87.50% success on unseen vulnerability-type/CWE evaluation, indicating strong generalization beyond the vulnerability types seen during retrieval or development. In addition to effectiveness, the study emphasizes efficiency and deployability: unlike prior systems that depend on expensive proprietary APIs, RAVEN runs with open-source models locally and keeps token/repair cost negligible.

### 🌱 Insights & Implications
The paper’s core insight is that vulnerability repair benefits substantially from two kinds of external knowledge that prior work often underuses: historical fixes from similar CVEs and repository-wide contextual dependencies. This is particularly relevant for security bugs whose root cause is not syntactically obvious in the local snippet, such as missing authorization checks or context-dependent injection flaws. A second important insight is that one-shot LLM repair is insufficient for reliable security patching; iterative review and static-analysis-guided refinement are necessary to catch partial fixes and unintended regressions. The work also suggests that medium-sized open-source LLMs, when paired with strong retrieval and control mechanisms, can rival or surpass proprietary-model-based repair pipelines at much lower cost. Limitations and open questions include the exact dependence of success on retrieval quality, how well the framework scales to even larger repositories or more complex build/test environments, whether static analysis alone is enough to validate semantic security correctness, and how robust the approach remains for vulnerability classes with sparse historical fix data. The paper implies promising future directions in integrating richer program analysis, stronger validation signals, and broader multilingual security repair benchmarks.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it directly advances automated fixing of real-world vulnerabilities under practical constraints. It is especially valuable for a researcher interested in vulnerability repair because it moves beyond narrow memory-bug settings, handles cross-file and unseen vulnerability types, uses open-source locally deployable models, and introduces an agentic retrieval-and-refinement architecture that could inspire future work on secure APR pipelines, repair validation, and repository-aware security tooling.

## 📄 [Formal-Method-Guided Vibe Coding: Closing the Verification Loop on AI-Generated Safety-Critical Software Through Model-Driven Engineering](https://arxiv.org/abs/2606.22413v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Ran Wei, Le Zhu, Haochi Wang, Jim Woodcock, Fang Yan, Simon Foster, Xiangyang Ji
📅 **Published:** 2026-06-21  🏷️ **Categories:** cs.SE, cs.FL

### 💡 TL;DR
The paper presents Forge, a closed-loop pipeline that turns LLM-generated Java for safety-critical systems into formally verified software by extracting formal models through model-driven engineering and feeding verifier failures back as structured repair prompts. Across three external robotics case studies and 15 total runs, Forge always converged to fully verified code, while single-pass generation and compile-only iterative baselines never did.

### 🏆 Key Contributions
- A closed draft-and-discriminate refinement loop where mainstream Java code generated by an LLM is iteratively corrected using feedback from formal verifiers rather than relying on verification-aware languages or one-shot post-hoc filtering.
- An EMF-native text-to-model transformation pipeline for modern Java 17+ built from Spoon, including an Ecore metamodel and reflective discoverer that support contemporary language features absent from older MDE tooling.
- A three-verifier verification architecture combining Dafny, CSP/FDR4, and Isabelle/Z-Machines, together with an analysis of the defect classes each verifier detects and misses.
- A traceable requirement-to-proof chain linking natural-language requirements, generated Java, extracted Spoon EMF and RoboChart models, and the resulting formal artefacts, enabling per-requirement repair feedback.
- Empirical evidence that verifier-guided iteration is the causal factor behind convergence: all 15 closed-loop runs verified successfully, versus 0/30 for cold single-pass generation and no success under a compile-only ablation.
- An open-source implementation with the full pipeline, case studies, and experiments.

### ⚙️ Methodology
Forge is a seven-phase pipeline designed to preserve an industrially realistic workflow while injecting formal assurance into LLM-based code generation. In Phase 1, requirements engineering produces a structured requirement set that becomes the basis for generation and later traceability. In Phase 2, the system performs Java code generation through a five-layer top-down prompt sequence, keeping the target language in mainstream Java rather than moving to a verification-oriented language. The generated Java is restricted to an extraction-tractable profile so that source-to-model transformations remain faithful.

In Phase 3, the Java source is parsed into a Spoon abstract syntax representation and then converted into an EMF model using the paper’s new Spoon-to-EMF bridge. This bridge includes an Ecore Java metamodel derived from Spoon’s metamodel API and a reflective discoverer that emits conforming EMF instances, allowing modern Java features such as records and sealed interfaces to be represented in an MDE-native form. In Phase 4, the EMF Java model is transformed into a RoboChart state-machine model, lifting implementation structure into a behavioural formalism suited for downstream verification. In Phase 5, the RoboChart and related extracted information are used to emit three formal artefact families: Dafny specifications for deductive verification, CSP-M scripts for refinement checking in FDR4, and Z-Machine theories for theorem proving in Isabelle/HOL.

In Phases 6 and 7, the formal artefacts are checked by the three complementary verifiers. Dafny handles deductive reasoning about code-level properties and contracts; CSP/FDR4 checks behavioural refinement and communication/state-transition consistency; Isabelle with Z-Machines supports theorem-proving over the extracted state-machine semantics. When any verifier reports failure, Forge does not merely reject the candidate. Instead, it parses the diagnostics and trace files to generate structured correction directives tied back to specific requirements and code regions. These diagnostics are then fed into the next generation round as prompts, creating a closed refinement loop in which the LLM acts as the draft generator and the MDE+formal-method stack acts as the discriminator. The developer interacts at the Java and requirement level and does not need to inspect formal models directly.

### 🧪 Experiments
The evaluation uses three case studies of increasing complexity drawn from safety-critical robotic/autonomous control settings: SRanger, a small single-controller ground robot from the RoboStar case-study set; the Last Response Engine (LRE), a four-mode autonomous underwater vehicle safety governor with published ground truth; and a Chemical Detector autonomous mobile robot from the RoboStar set. The same pipeline is run end-to-end on all three systems without case-study-specific modifications.

The main experiment performs five independent runs per case study, for 15 total closed-loop runs. The key result is that every run converges to fully verified Java code, yielding standards-relevant evidence through all three verification backends. To test whether iteration alone explains the result, the paper includes two controls. First, a cold single-pass baseline is run 30 times in total, and none of these runs reaches full convergence or verification success (0/30). Second, a compile-only ablation removes formal-verifier feedback while preserving iterative generation; under this setting, every run still fails formal verification, showing that verifier-derived guidance, not mere repeated prompting, is responsible for convergence.

The paper also reports a defect-level comparison of the three verification backends, showing that they are complementary rather than redundant. Although the excerpt does not provide full numeric verifier-by-verifier breakdowns, it states that Dafny, FDR4, and Isabelle catch different defect classes, and this motivates the use of all three in the loop. The evaluation emphasizes convergence behavior, verification completeness, traceability, and the ability to produce certification-relevant artefacts rather than only compile- or test-level correctness.

### 🌱 Insights & Implications
The paper’s central insight is that formal verification can be inserted into LLM-based software generation without requiring either developers or models to operate in niche proof-oriented languages. By treating mainstream Java as the generation target and model-driven transformations as the bridge into formal verification, Forge lowers the adoption barrier for safety-critical organizations that already rely on established Java/MDE toolchains. This is especially important for certification-oriented development, where replacing the language ecosystem is often infeasible.

A second key insight is that formal methods are most useful here not as a one-shot acceptance filter but as an iterative discriminator that shapes subsequent generations. This directly addresses a weakness in prior work that verifies or rejects outputs after the fact but does not use proof failures as repair signals. The structured requirement-to-proof traceability further suggests a path toward integrating LLM workflows with assurance cases and compliance evidence.

Limitations include the reliance on an extraction-tractable Java subset, meaning general unrestricted Java is not yet covered. Generalizability beyond the studied controller-style systems remains an open question, especially for larger object-oriented architectures, concurrency-heavy software, or systems with complex libraries and environment interactions. The pipeline also depends on high-quality transformations and diagnostic parsers; any mismatch between source semantics and extracted models could undermine assurance. Another open issue is scalability: while the paper demonstrates robustness on three case studies, the cost and practicality of running multiple formal backends in industrial continuous development settings are not fully resolved.

For related research, the work implies a promising direction for automated repair and vulnerability-relevant assurance: using verification failures as machine-readable repair signals could extend beyond functional correctness to security properties, protocol compliance, and safety/security co-verification.

### 🎯 Why This Paper
This paper is highly relevant because it connects LLM-based code generation to rigorous verification in mainstream software engineering workflows, which is directly useful for research on software maintenance, automated program repair, and trustworthy code synthesis. Its closed-loop use of formal verification failures as structured repair prompts is especially aligned with automated repair research, and the traceable MDE pipeline offers a practical foundation for extending similar ideas to security-critical and vulnerability-repair settings where certification-grade evidence and explainable corrections matter.

## 📄 [Revelio: Cost-Efficient Agentic Memory Safety Vulnerability Detection For Repository-Scale Codebases](https://arxiv.org/abs/2606.22263v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Yiwei Hou, Hao Wang, Muxi Lyu, Marius Momeu, Eric Nguyen, Taige Yang, Koushik Sen, Dawn Song, David Wagner
📅 **Published:** 2026-06-20  🏷️ **Categories:** cs.CR, cs.AI, cs.MA, cs.SE

### 💡 TL;DR
Revelio is a two-stage agentic framework for repository-scale C/C++ memory-safety vulnerability detection that uses cheap LLMs for broad hypothesis generation and stronger models plus sanitizer-validated Proofs-of-Vulnerability for confirmation. Across heavily fuzzed real-world projects and CyberGym benchmarks, it found 19 previously unknown bugs with zero false positives at low cost, outperforming general-purpose coding agents at similar token budgets.

### 🏆 Key Contributions
- Introduces a two-stage vulnerability discovery workflow that separates high-recall hypothesis generation from deterministic, executable confirmation using sanitizer-validated Proofs-of-Vulnerability.
- Designs and implements Revelio, an end-to-end agentic framework for scalable memory-safety bug detection in repository-scale C/C++ codebases using inexpensive models, lightweight static analysis, and stronger confirmation models.
- Demonstrates practical effectiveness on mature OSS-Fuzz projects by discovering 19 previously unknown memory-safety vulnerabilities, including 7 CVEs, with about one hour and roughly $42 per project on median.
- Shows on 100 CyberGym/Arvo benchmark projects that Revelio substantially outperforms frontier coding-agent baselines such as Claude Code, Codex, and Sorcar, while maintaining zero false positives at comparable token costs.

### ⚙️ Methodology
Revelio is built around the insight that vulnerability discovery can be decomposed into two qualitatively different tasks: speculative code review and executable confirmation. In the Hypothesis Generation stage, the system scans repository files with sub-agents using inexpensive LLMs, aided by lightweight static preprocessing, to produce a high-recall pool of candidate memory-safety hypotheses. Each hypothesis localizes a suspicious code region, explains the suspected bug mechanism, and describes triggering conditions under attacker-controlled inputs. Revelio then triages, filters, and ranks these hypotheses in a funnel-style process to retain the most promising candidates while tolerating false alarms. In the Hypothesis Confirmation stage, Revelio selects an executable entry point or test harness capable of reaching the suspected code, then invokes a stronger model to synthesize a concrete Proof-of-Vulnerability input. The system iteratively debugs and refines this PoV, executes the target under sanitizers such as ASan/MSan/UBSan, and only reports a vulnerability if the sanitizer deterministically reproduces the memory-safety violation. This creates a strict evidence boundary: LLMs are used for search, reasoning, and input construction, while runtime execution and sanitizer reports serve as the trusted verifier. The approach is explicitly cost-aware, reserving expensive model inference for the small subset of high-priority candidates and using cheaper models for exhaustive repository review.

### 🧪 Experiments
The evaluation has two parts. First, Revelio was applied to seven production-quality open-source projects that had already been continuously fuzzed by OSS-Fuzz for five to eight years. Despite this strong prior testing, Revelio discovered 19 previously unknown memory-safety vulnerabilities, including 7 CVEs, with a median runtime of about one hour per project and an average cost around $42 per project, totaling roughly $300. Second, the authors evaluated Revelio on 100 randomly selected Arvo projects from the CyberGym benchmark and compared it against general-purpose coding agents including Claude Code with Opus 4.7, Codex with GPT 5.5, and Sorcar with Opus 4.7. Revelio found 175 vulnerabilities versus 55 for Claude Code, 39 for Codex, and 31 for Sorcar at comparable token costs. It also achieved zero false positives, while the competing coding agents had substantial false positive rates. On known benchmark vulnerabilities, Revelio also showed stronger detection effectiveness, indicating higher recall as well as stronger end-to-end confirmation capability.

### 🌱 Insights & Implications
The paper’s main technical insight is that trustworthy LLM-based vulnerability detection benefits from decoupling broad speculative analysis from rigorous executable validation. This decomposition aligns well with model capabilities: weaker, cheaper models can review large repositories and over-generate candidates, while stronger models can focus on difficult reasoning tasks such as constructing triggering inputs. The use of sanitizer-validated PoVs is especially important for security tooling because it eliminates hallucinated reports and yields reproducible evidence useful for maintainers. For software security research, Revelio suggests that agentic systems may outperform monolithic coding agents when the workflow is carefully structured around verification. A likely limitation is that Revelio’s recall still depends on selecting reachable harnesses and successfully synthesizing PoVs, so vulnerabilities that require complex environments, deep semantic preconditions, or unsupported sanitizers may remain undiscovered. The work is focused on memory-safety issues in C/C++, so generalization to logic bugs, concurrency flaws, or other vulnerability classes is still open. Another implication is that benchmark success alone is insufficient; the real-world OSS-Fuzz results are significant because they show incremental value beyond mature fuzzing pipelines.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and automated repair because it presents a practical, scalable architecture for finding real memory-safety bugs in large, mature codebases with low false-positive burden. Its sanitizer-backed PoV generation is especially useful for vulnerability triage, benchmark design, and downstream repair workflows, since confirmed, reproducible crashing inputs can directly support root-cause analysis, patch generation, and validation in automated vulnerability repair pipelines.

## 📄 [Detecting Malicious Agent Skills in the Wild using Attention](https://arxiv.org/abs/2606.23416v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Bacem Etteib, Daniele Lunghi, Tégawendé F. Bissyandé
📅 **Published:** 2026-06-22  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper presents Locate-and-Judge, a two-stage detector for malicious LLM agent skills that first uses attention signals from a lightweight model to localize suspicious instruction spans, then applies a stronger judge model only to those spans. The approach makes marketplace-scale scanning practical, achieving much lower cost than full-skill LLM inspection while still finding many real malicious skills in the wild with high precision.

### 🏆 Key Contributions
- Introduces Locate-and-Judge, a two-stage detection pipeline tailored to malicious agent skills, where malicious instructions are embedded among otherwise legitimate instructions.
- Shows that instruction-following attention can still localize malicious spans inside third-party skill files, even when traditional prompt-injection assumptions about trusted instructions versus untrusted data do not hold.
- Demonstrates a favorable cost-recall tradeoff: compared with scanning every full skill using a capable LLM, the method achieves similar detection yield with substantially fewer judge-input tokens and much better scalability.
- Deploys the detector across about 134,000 real-world skills from multiple marketplaces and uncovers 131 confirmed malicious skills, including 82 hidden malicious skills disguised as benign functionality.
- Finds that many discovered malicious skills evade existing scanners such as SkillSpector and Cisco Skill Scanner, and releases a labeled dataset with human-assessment labels to support future research.

### ⚙️ Methodology
Locate-and-Judge is a two-stage pipeline. First, a deterministic parser decomposes each skill package into structural spans drawn from SKILL.md, YAML frontmatter, and helper-file regions such as comments or strings that may contain instructions. A lightweight locator model then reads the whole skill and scores each span according to how much instruction-following attention it attracts. The underlying hypothesis is that for a malicious instruction to succeed, it must capture the reader model’s instruction-following attention, so suspicious injected content should rank among the highest-attention spans even inside a document that is already mostly instructional. After ranking, the pipeline keeps only the top-K spans. In the second stage, a more capable zero-shot LLM judge inspects these retained spans in detail and assigns maliciousness scores. The final skill-level decision is made by taking the maximum judge score over the retained spans and comparing it to a calibrated threshold tau; if any span exceeds the threshold, the skill is flagged. This decomposition separates localization from classification: the cheap locator processes every skill once, while the expensive judge is restricted to a small subset of candidate spans, enabling scalable auditing. The authors argue that recall limitations mostly come from the judge rather than the locator, meaning stronger judge models, prompts, or ensembles could improve performance without redesigning the architecture. The threat model focuses mainly on instruction-level malicious skills that exfiltrate data, hijack agent behavior, or establish persistence, while code-level attacks are treated as related but outside the main scope.

### 🧪 Experiments
The evaluation has both controlled and in-the-wild components. For controlled experiments, the authors build a labeled corpus from previously curated benign skills released by Skill-Inject and Liu et al., then inject one or more vulnerabilities into a subset using the Skill-Inject pipeline. This setup is used to test whether the locator successfully surfaces injected spans and to compare end-to-end detection performance against baselines such as direct full-skill LLM scanning, per-span baselines, and lightweight keyword/regex approaches. Metrics include precision, recall, and token-cost efficiency. The reported outcome is that Locate-and-Judge outperforms these baselines overall, especially in cost-effectiveness, and dominates keyword/regex methods at similar expense. Compared with direct LLM-based full-skill scanning, it provides an order-of-magnitude cost reduction in general discussion and specifically achieves a 2.8x reduction in judge-input tokens per skill while recovering a similar number of malicious skills. In the real-world deployment, the method scans roughly 134,000 skills from three public marketplaces: Lobehub, Skills.sh, and Clawhub.ai. Using a conservative threshold calibrated in the laboratory, it flags 359 skills for manual review; 131 of these are confirmed malicious, corresponding to about 83% precision. Among the confirmed malicious cases, 82 are hidden malicious skills that masquerade as benign functionality. The paper also notes that many of these live malicious skills are missed by existing detectors such as SkillSpector and Cisco Skill Scanner. In a broader count that includes offensive tooling, the study identifies 211 problematic skills.

### 🌱 Insights & Implications
A central insight is that attention remains a useful near-necessary signal for malicious instruction execution even when malicious content is embedded inside a larger body of legitimate instructions, which is precisely the regime where standard prompt-injection defenses break down. The study suggests that scalable detection is possible by localizing first and judging second, rather than spending expensive LLM analysis on entire skill contents. An important empirical finding is that most misses are attributed to the judge exonerating suspicious spans that the locator had already surfaced, implying the architecture itself is sound and future gains may come from stronger judge models, better prompts, calibration, or model ensembles. The paper also highlights that current marketplace ecosystems for agent skills constitute a meaningful software supply-chain attack surface, with malicious content often disguised as benign capabilities and able to evade existing scanners. Limitations include the main focus on instruction-level rather than code-level attacks, dependence on threshold calibration and top-K selection, and the possibility that highly stealthy attacks may avoid strong attention signals or be ambiguously interpreted by the judge. For related research, the work opens directions in hybrid detectors combining attention, semantics, and static analysis; benchmark creation for malicious agent artifacts; and defenses for secure skill/package ecosystems in agentic software systems.

### 🎯 Why This Paper
This paper is highly relevant to the researcher’s interests because it addresses a new software supply-chain security problem in LLM-based agent ecosystems: malicious third-party skills that can hijack behavior, exfiltrate data, or persist as hidden backdoors. It is especially useful for work on software security, vulnerability detection, and automated remediation because it proposes a scalable detection architecture, evaluates it on large real-world marketplaces, exposes weaknesses of existing defenses, and releases labeled data that could support future research on detection, repair, hardening, or secure maintenance of agent extensions.

## 📄 [Automated Semantic Fault Localization in SysML v2: A Human-in-the-Loop Framework Using Knowledge-Graph Augmented LLMs](https://arxiv.org/abs/2606.23395v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Haitham Al-Shami, Rohail Malik, Riku Ala-Laurinaho, Jari Vepsäläinen, Raine Viitala
📅 **Published:** 2026-06-22  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
The paper proposes a human-in-the-loop framework for detecting and repairing semantically faulty SysML v2 models by combining fine-tuned small language models with a domain knowledge graph of engineering compatibility rules. On a synthesized vehicle-systems benchmark, the approach raises semantic repair success from under 3% with base models to over 91% after fine-tuning, while using compact unified diff patches to make review easier.

### 🏆 Key Contributions
- A synthesis pipeline that expands a small corpus of valid SysML v2 models into a larger dataset containing correct models, syntax errors, and semantically invalid but syntactically valid domain faults.
- A domain knowledge graph capturing vehicle-system physical compatibility constraints across mechanical, electrical, fluid, and signal interfaces, used both to inject realistic semantic faults during training-data generation and to ground inference-time repair suggestions.
- A human-in-the-loop semantic fault localization and repair framework in which fine-tuned small language models output unified diff patches rather than full rewritten models.
- An empirical evaluation of two code-oriented SLMs, Qwen2.5-Coder-1.5B and DeepSeek-Coder-6.7B, showing strong gains in semantic repair accuracy and substantial output-length reduction.

### ⚙️ Methodology
The framework targets a key gap in SysML v2 validation: compilers can detect grammar and structural violations, but not domain-specific semantic violations such as physically incompatible interface connections. To address this, the authors build a domain knowledge graph for the vehicle-systems domain that represents valid relationships among component types, interfaces, and physical quantities spanning mechanical, electrical, fluid, and signal connections. This KG is used in two roles. First, it drives synthetic data generation by systematically mutating valid SysML v2 models to create realistic semantic faults that remain syntactically valid but violate engineering constraints; separate heuristics are used to inject syntax faults that break SysML v2 grammar or compiler validity. Second, at inference time, the KG is used as a rule-based augmentation source so the model’s repair suggestions are grounded in valid engineering constraints rather than free-form generation alone. The training corpus starts from 256 valid SysML v2 samples and is expanded into 8,301 labeled examples covering correct, syntactically faulty, and semantically faulty instances. Rather than training models to regenerate complete corrected files, the system is trained to emit unified diff patches, which localize the problematic lines and propose candidate edits. This design reduces verbosity, focuses the model on fault localization and minimal repair, and supports human review before changes are accepted. Two small code models, Qwen2.5-Coder-1.5B and DeepSeek-Coder-6.7B, are supervised fine-tuned on the synthesized dataset using a data-leak-resistant split so near-duplicate variants do not contaminate evaluation.

### 🧪 Experiments
The evaluation uses a synthesized SysML v2 benchmark derived from 256 valid seed models and expanded to 8,301 total examples. The reported test set contains 1,184 samples and includes syntax-error cases, semantic-error cases, and correct examples for error classification. The domain is vehicle systems, with semantic constraints defined by the KG over mechanical, electrical, fluid, and signal interfaces. The main baselines are the unfine-tuned base versions of Qwen2.5-Coder-1.5B and DeepSeek-Coder-6.7B, compared against their fine-tuned counterparts under the patch-generation setup. The principal outcome is semantic fault repair performance: base models achieve less than 3% success on semantic repairs, while fine-tuning increases this to more than 91%. The paper also reports that patch-based outputs substantially reduce response length, with the abstract reporting over 60% reduction and the body/abstract also describing roughly 50% reduction, indicating a major compression benefit relative to full-model rewrites. The evaluation is framed around the model’s ability to identify faults, produce valid candidate repairs, and support engineer review rather than autonomous acceptance.

### 🌱 Insights & Implications
A central insight is that semantic validation in MBSE differs from software testing: many SysML v2 faults are not executable-test failures but violations of domain intent and physical compatibility, so explicit domain knowledge is necessary. The results show that small models can become highly effective repair assistants when they are narrowly fine-tuned on task-specific data and grounded by symbolic constraints, suggesting a practical alternative to large general-purpose models for industrial MBSE tooling. The patch-based formulation is also important because it improves efficiency and reviewability, making the system fit better into human-centered engineering workflows. Limitations include heavy reliance on synthetic data, a single application domain focused on vehicle systems, and knowledge-graph coverage that may not generalize to other domains without substantial manual rule engineering. The paper also emphasizes that human judgment remains necessary because SysML models encode intent that may be ambiguous or incompletely captured by formal rules. Open questions include how well the method transfers to larger real industrial models, whether the KG can be learned or extended automatically, and how to benchmark semantic repair on broader, non-synthetic datasets.

### 🎯 Why This Paper
This paper is highly relevant because it studies automated fault localization and repair in a formal modeling language using techniques closely aligned with automated program repair, semantic bug detection, and human-in-the-loop verification. For a researcher in software engineering, maintenance, security, APR, and vulnerability detection/repair, it offers a transferable pattern: combine symbolic domain constraints with fine-tuned lightweight models, generate patch-style outputs, and use synthetic fault injection to overcome scarce labeled data. The approach could inspire analogous methods for semantic bug localization, configuration repair, API misuse detection, and even security-rule-grounded vulnerability repair in software artifacts.

## 📄 [Understanding the (In)Security of Vibe-Coded Applications](https://arxiv.org/abs/2606.23130v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Junquan Deng, Zhiyu Fan, Ruijie Meng
📅 **Published:** 2026-06-22  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper presents the first large-scale empirical security study of real-world vibe-coded applications, showing that publicly deployed AI-generated apps frequently contain serious vulnerabilities and that these issues stem from lifecycle-level agent failures rather than isolated insecure code snippets. Across 200 deployed apps sampled from a 10,517-app corpus, the authors identify 1,471 vulnerabilities and show that stronger models and better prompting help but do not remove the underlying risks.

### 🏆 Key Contributions
- Construction of VibeApps, a large-scale dataset of 10,517 real-world vibe-coded open-source applications created with popular AI agents and platforms.
- Creation of VibeVulns, a vulnerability dataset derived from 200 randomly sampled publicly deployed vibe-coded applications, containing 1,471 validated vulnerabilities.

### 💡 TL;DR
VeriPort is an end-to-end LLM-based system for automatically backporting security fixes across all affected package versions while verifying each patch with both exploit-blocking and functionality-preservation evidence. ### 🏆 Key Contributions
- Introduces VeriPort, the first end-to-end automated patch backporting system that targets every affected version in a vulnerability advisory rather than a single preselected version.
- Builds a verification pipeline around each backport using independently generated vulnerability oracles (PoCs) and functionality oracles (regression suites), providing evidence that patches both block exploitation and preserve intended behavior.
- Removes the strong prior assumption that a minimal security patch is given, instead automatically researching advisories, isolating the security-relevant changes from noisy fixing commits, and adapting them across diverged versions.
- Establishes CVEPatchBench, a new benchmark of 393 npm backporting tasks with containerized environments, PoCs, regression suites, and reference backports.
- Demonstrates real-world deployment at scale, generating over 5,000 verified backported patches for 169 high- and critical-severity CVEs and identifying substantial errors in advisory version metadata.

### ⚙️ Methodology
VeriPort is designed as an agentic, staged workflow that addresses three major LLM limitations identified by the authors: hallucination, bounded context, and repeated regeneration cost across many package versions. To mitigate these, the system uses fresh-context critic agents plus deterministic checks, decomposes the task into narrow stages and agents, and generates artifacts once before adapting them across versions when possible.

The pipeline has four stages. AutoResearch consumes a vulnerability advisory and gathers grounded context about the package, the vulnerability, the upstream fix, and code differences across affected versions. This stage validates noisy advisory data, locates candidate vulnerability-fixing commits, analyzes version ranges, and reasons about how the vulnerable and patched logic evolved over time. A key goal is identifying or reconstructing the minimal security patch even when the upstream fixing commit is tangled with unrelated changes.

AutoPoc then creates a vulnerability oracle for each affected version. Rather than trusting advisory descriptions alone, it generates or adapts a proof of concept that actually exploits the specific version, executes it in the corresponding environment, and defines a success criterion that distinguishes real exploitation from unrelated failures. VeriPort also uses these PoCs to probe version-range boundaries, checking whether listed versions are truly exploitable and whether nearby unlisted versions may also be affected.

AutoTester constructs a functionality oracle for each version by producing a regression suite that captures intended package behavior. These tests are meant to run on both vulnerable and patched versions, ensuring the accepted backport does not regress expected functionality. This stage is independent of patch synthesis, so the patch is evaluated against tests not derived from the patch itself.

AutoPatch isolates the security-relevant changes from the upstream fix, adapts them to older affected versions, and validates each candidate backport. A patch is accepted only if it both blocks the generated exploit and passes the generated regression suite. VeriPort tries to reuse and adapt previously generated artifacts across versions, regenerating only when code drift is too large. The overall design forms a chain of evidence: advisory understanding, exploit oracle, regression oracle, and patch adaptation are produced independently so a faulty patch must evade multiple checks simultaneously to be accepted.

The paper evaluates VeriPort primarily on BackportBench and CVEPatchBench. On BackportBench, the authors use the npm and PyPI tasks, totaling 128 backporting tasks, and compare against four baselines formed from two generic coding agents (Claude Code and mini-swe-agent) and two patching/backporting systems (MagentLess and PortGPT, the prior state-of-the-art agentic backporting approach). Each tool is run 10 times per task under two settings: one where the upstream minimal security patch is provided and one where it is withheld.

The central result is that without access to the MSP, VeriPort reliably resolves 95.3% of tasks across all 10 runs, outperforming the strongest baseline, Claude Code, at 72.6%, a 22.7 percentage-point gain. When the MSP is withheld, Claude Code loses 13.2 points of single-run success and mini-swe-agent loses 17.3 points, while VeriPort drops only 0.8 points, showing robustness to missing patch supervision. The paper also reports that VeriPort’s worst-case performance across runs exceeds every baseline’s best-case performance, and that VeriPort is the only system in either setting to solve every task within 10 runs.

The authors also introduce and evaluate on CVEPatchBench, a new benchmark of 393 npm backporting tasks. Each task includes a containerized environment, working PoC, regression suite, and a reference backport. Against Claude Code, with 3 runs per task, VeriPort reliably solves 91.3% of tasks on all three runs compared with 71.1% for Claude Code. VeriPort’s worst case still beats Claude Code’s best case by 10.7 percentage points, and its cross-run variance is much lower: 2.3 points versus 9.5 points.

An ablation study removes AutoPoc and AutoTester, the exploit and regression verification stages, and observes a 9.8-point performance drop, supporting the claim that verification materially improves patch quality and reliability.

Beyond benchmark evaluation, the system was deployed in production at Socket on 169 high- and critical-severity CVEs, where it generated over 5,000 verified backported patches. In deployment, exploit testing across version ranges revealed 2,100 versions incorrectly listed as affected and 127 vulnerable versions missing from advisories across 92 advisories. Upstream maintainers had already corrected 23 advisories at the time of writing, removing 387 incorrectly listed versions and adding 81 missing ones.

### 🌱 Insights & Implications
A central insight is that patch backporting should not be treated merely as code transformation or patch translation; it is fundamentally a verification problem requiring evidence that a candidate patch blocks exploitation and preserves compatibility. VeriPort operationalizes this by making exploit generation and regression generation first-class artifacts, not optional post hoc checks.

The work also highlights how noisy real-world vulnerability metadata is. Missing or tangled vulnerability-fixing commits, inaccurate affected-version ranges, and weakly reviewed advisories create substantial obstacles for traditional rule-based or patch-similarity approaches. This is relevant beyond backporting, especially for automated program repair and vulnerability remediation systems where hallucinations and brittle one-shot generation are major risks.

A limitation is that the system’s reliability depends on successful PoC and regression-oracle generation; difficult-to-reproduce vulnerabilities, environment-heavy bugs, or underspecified functionality may still be hard cases. The excerpt also suggests that advisory understanding, PoC construction, and MSP isolation remain challenging semantic tasks even for LLMs, implying that failures may cluster around severe code drift, missing contextual artifacts, or ambiguous fixes. Another open question is generalization beyond the evaluated ecosystems and languages, as the reported benchmarks emphasize npm and PyPI workloads and the new benchmark is npm-focused.

For related research, the paper strengthens the case for combining automated repair with executable validation rather than relying on patch resemblance, static checks, or pre-supplied developer artifacts. ## 📄 [VCT: A Verifiable Transcript System for LLM Conversations](https://arxiv.org/abs/2606.23003v1)  `70% relevant`  `🌐 Full Text`

📅 **Published:** 2026-06-22  🏷️ **Categories:** cs.CR

### 💡 TL;DR
VCT is a cryptographic transcript system for LLM conversations that treats chats as evolving account-level state rather than simple append-only logs, supporting branching, deletion, multi-device concurrency, and selective sharing. It combines hierarchical authenticated data structures with dual user/server signatures and synchronization protocols to make conversation histories independently verifiable, tamper-evident, and non-repudiable with low overhead.

### 🏆 Key Contributions
- Introduces VCT, a three-tier authenticated transcript structure for LLM conversations: branch-level hash chains over atomic Q&A pairs, session-level Merkle roots over branch tails, and an account-level Merkle root over sessions.
- Defines account-level state transition semantics for non-linear LLM interactions, covering branching, response regeneration, historical edits via branching, new session creation, and conversation deletion.
- Proposes a serialized deletion protocol with deletion barriers to resolve delete–modify conflicts, alongside a deterministic merge protocol that preserves concurrent non-deletion incremental updates across devices.
- Adds incremental denial checks and a gossip-based cross-device consistency mechanism to detect malicious server equivocation or forked account views and produce dual-signed forensic evidence.
- Presents a verifiable selective-sharing protocol in which shared transcript fragments are reorganized into an independently verifiable hash chain signed by both user and server.
- Provides a security analysis arguing integrity, consistency, verifiable shareability, and non-repudiation under standard cryptographic assumptions, and a prototype evaluation showing practical latency and negligible storage overhead.

### ⚙️ Methodology
The paper formulates LLM transcript preservation as an authenticated evolving-state problem at the account level. VCT models each atomic interaction as a Q&A node containing the user prompt, model response, and metadata, linked by a parent_hash to form a branch-local hash chain. This captures the sequential dependency within one conversational branch while allowing multiple legitimate branches caused by regeneration or re-prompting from historical states. For each conversation/session, the tail hashes of all active branches are aggregated into a session-level Merkle root (MTR_con), compactly committing to the complete branching state of that conversation. Across an account, all conversation roots are aggregated into an account-level Merkle root (MTR_acc), which serves as the global authenticated summary of the user’s transcript state.

To make this state externally trustworthy, VCT binds each account root with joint signatures from both the user and the server. The user identity is maintained across devices through deterministic key derivation, so multiple devices can sign consistent account-state assertions without relying on the server for private-key synchronization. State evolution is defined through authenticated transitions rather than raw log appends. Non-deletion operations, such as appending a new Q&A node or creating a new branch/session, are treated as monotonic increments. Concurrent non-deletion updates from different devices are reconciled by a deterministic merge procedure that preserves all compatible increments. In contrast, deletions are modeled as serialized account-level operations separated by deletion barriers/epochs, ensuring that a conversation deletion cannot ambiguously race with later modifications from a stale device view. This turns deletion into a state replacement event with explicit ordering semantics.

The server returns incremental updates together with authenticated state information, and devices perform incremental denial checks to reject stale or selectively omitted updates that could otherwise be replayed later. To defend against a malicious server presenting inconsistent account states to different devices, VCT incorporates a gossip protocol: devices exchange signed account roots over secure channels and compare views. If distinct roots exist for the same logical state progression, devices can detect a fork and retain dual-signed artifacts as non-repudiable forensic evidence. For selective disclosure, VCT constructs an independent share-specific hash chain over the disclosed nodes and requires both user and server signatures on the share tail, so recipients can verify the exact shared content without trusting server-rendered web views or needing access to the entire original account state.

### 🧪 Experiments
The evaluation is based on a Python prototype implementing VCT’s core cryptographic operations and transcript protocols. The experiments focus on deployment practicality rather than model quality, measuring cryptographic latency and metadata overhead for transcript operations such as Q&A-node authentication, branch/session/account commitment updates, and proof-related checks. According to the paper, the main cryptographic operations complete in sub-millisecond to low-millisecond time ranges, indicating that transcript verification and state updates are lightweight enough for interactive LLM platforms.

For storage overhead, the authors evaluate a realistic transcript configuration containing roughly 21 KB of conversation text. Under this setting, the added security metadata contributes only about 0.9% extra storage, suggesting that the hierarchical commitments and signatures are compact relative to actual chat content. The paper does not emphasize extensive baseline comparisons against prior systems; instead, it positions VCT against unsuitable alternatives such as linear secure logs, generic Merkle authentication, transparency logs, and secure messaging, arguing qualitatively that these do not capture the non-linear and account-level semantics of LLM conversations. The key empirical takeaway is that VCT’s security machinery appears feasible for production deployment because it imposes very small latency and storage costs while supporting richer transcript semantics than append-only logging.

### 🌱 Insights & Implications
A central insight is that LLM transcript verification is not just a secure logging problem: conversation histories are inherently non-linear, mutable in presentation, and distributed across devices, so the correct abstraction is authenticated state transitions over a branching, multi-session account structure. This matters for digital forensics, compliance, and legal evidence because screenshots, exports, and server-side sharing links are insufficiently trustworthy without cryptographic commitments tied to a global account state.

The work is especially interesting for security and provenance research because it explicitly separates legitimate user-created branching from malicious server-induced forks, something append-only logs or transparency systems cannot naturally express. The dual-signature design also raises the evidentiary value of transcripts by preventing either side from later repudiating committed states. The selective-sharing design is important in practice because many auditing or legal workflows require disclosure minimization while still demanding integrity proofs.

Potential limitations and open questions remain. The prototype evaluation is lightweight and does not appear to explore large-scale production workloads, adversarial synchronization patterns, mobile energy costs, or long-term key-management usability. The trust model still assumes a cooperating server for joint signing of committed states; if a platform refuses to sign, usability and deployability may degrade. Privacy implications of account-root gossip and metadata retention may require further analysis, especially for sensitive enterprise or legal deployments. Another open direction is integration with model-execution attestation, since VCT verifies transcript state consistency but does not by itself prove that a particular backend model generated the content honestly. For software-security researchers, the broader implication is that authenticated provenance for AI-assisted development may need to combine transcript integrity with execution attestation, code artifact linkage, and policy-aware auditing.

### 🎯 Why This Paper
This paper is relevant because it addresses provenance, integrity, and forensic verification of LLM interactions—an increasingly important foundation for secure AI-assisted software engineering, vulnerability triage, repair workflows, and compliance review. In settings where developers use LLMs for code generation, bug fixing, or security guidance, VCT offers a way to make interaction records tamper-evident, shareable, and auditable, which could support trustworthy vulnerability reports, reproducible automated program repair pipelines, and verifiable evidence about how AI suggestions influenced software changes.

The study formulates CWE-level host intrusion detection as one-class anomaly detection over sliding-window syscall traces from containerized applications. ## 📄 [SCRUB-FL: Sanitizing and Cleansing Representations via Unlearning of Backdoors](https://arxiv.org/abs/2606.22700v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Osama Wehbi, Sarhad Arisdakessian, Omar Abdel Wahab, Azzam Mourad, Hadi Otrok
📅 **Published:** 2026-06-21  🏷️ **Categories:** cs.LG, cs.AI, cs.CR, cs.CV

### 💡 TL;DR
SCRUB-FL is a post-training federated learning defense that removes backdoors by collecting lightweight client-side generative representations of suspicious samples during training and then using server-side machine unlearning on synthesized trigger-like inputs after convergence. Across CIFAR-10 and GTSRB under multiple backdoor settings and up to 40% malicious clients, it sharply lowers attack success rates to as low as 3.88% while preserving over 91% clean accuracy, outperforming prior defenses without needing trigger knowledge or a large clean server dataset.

### 🏆 Key Contributions
- Introduces SCRUB-FL, a two-phase post-training backdoor sanitization framework for federated learning that works on the converged global model without raw client data access.
- Combines client-side spectral signature analysis and per-class activation clustering to identify suspicious local samples that may contain trigger patterns.
- Uses lightweight client-trained WGAN-GP generators whose parameters are aggregated at the server to approximate a global suspicious-pattern distribution in a privacy-preserving way.
- Proposes a server-side machine unlearning objective that feeds synthesized trigger-like samples to the global model and forces uniform predictive outputs, thereby erasing trigger-target associations without destructive neuron pruning.
- Demonstrates strong robustness across multiple attack scenarios, including One-to-One, One-to-N, and N-to-One attacks with up to 40% malicious clients, while retaining high benign-task utility.

### ⚙️ Methodology
SCRUB-FL is organized into two phases. In the training-time phase, each client analyzes its local data to extract suspicious samples likely to contain poisoned or trigger-bearing examples. The paper uses two complementary detectors: spectral signature analysis to identify dominant poisoning directions in feature space, and per-class activation clustering to catch anomalous activation patterns that may not be clear spectral outliers. Rather than sharing these samples, each client trains a lightweight WGAN-GP on the suspicious subset so that the generator learns an approximate distribution of trigger-related patterns. The generator parameters are sent to the server and aggregated, producing a global generative representation of suspicious inputs while preserving data locality. In the post-convergence phase, the server uses the aggregated generator to synthesize trigger-approximating samples. These synthetic suspicious inputs are passed through the trained global classifier, and a machine unlearning procedure is applied: the model is optimized so that its output on such inputs becomes uniform across classes, removing the attacker-imposed trigger-to-target mapping. At the same time, the model is constrained to preserve correct behavior on clean reference samples, yielding a simultaneous forgetting-preservation objective. The paper emphasizes that this avoids pruning neurons that may be entangled with useful semantic features, and instead weakens the malicious association directly through an amplified unlearning signal without permanently modifying network structure.

### 🧪 Experiments
The evaluation is conducted on CIFAR-10 and GTSRB, two standard image-classification benchmarks frequently used in backdoor studies. The threat settings include three attack types: One-to-One, One-to-N, and N-to-One backdoor attacks, and the proportion of malicious clients is varied up to 40%, testing difficult federated security regimes. The main metrics are backdoor attack success rate (ASR) and normal/clean task accuracy. SCRUB-FL is compared against state-of-the-art defenses spanning aggregation-time and post-training approaches. The reported results show that SCRUB-FL reduces ASR to as low as 3.88% while keeping clean accuracy above 91%. The paper states that these results consistently outperform competing baselines, particularly because SCRUB-FL does not require prior knowledge of trigger patterns, identification of poisoned clients after training, or a large labeled clean proxy dataset at the server. The experimental narrative also highlights robustness across diverse backdoor forms and better preservation of benign accuracy than pruning-heavy sanitization methods.

### 🌱 Insights & Implications
A key insight is that useful post-training backdoor removal in federated learning can be enabled by collecting the right auxiliary information during training, even if the actual sanitization occurs only after convergence. The paper also suggests that trigger behavior can be attacked at the representation level using synthetic suspicious inputs, avoiding the common failure mode of neuron entanglement seen in pruning-based defenses. This is particularly relevant for federated settings where the server lacks direct access to poisoned data, trigger definitions, or trustworthy client identities. Likely limitations include reliance on protocol-compliant attackers who still execute the GAN-training step faithfully, dependence on the quality of suspicious-sample detection and generator fidelity, and focus on image-domain benchmarks rather than broader modalities such as code, text, or multimodal FL. Open questions include how SCRUB-FL behaves under adaptive adversaries who manipulate the suspicious-pattern extraction stage, whether generator aggregation remains effective under high heterogeneity, and how much computational overhead the local GAN training adds for resource-constrained edge clients.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection/repair, and trustworthy distributed learning because it addresses the removal of malicious behavior embedded in trained systems after deployment-like training has completed. For a researcher interested in automated repair, SCRUB-FL is especially compelling because it frames backdoor mitigation as a post hoc unlearning-based repair problem rather than only a detection or filtering problem. Its privacy-preserving design, robustness to unknown attacks, and emphasis on sanitizing learned representations without raw data access make it a strong reference for secure ML system maintenance, model remediation pipelines, and defensive techniques that resemble patching vulnerabilities in already-trained collaborative systems.

## 📄 [Evidence-Bound Gateway-Path Provenance for Third-Party LLM Inference](https://arxiv.org/abs/2606.22560v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Fei Wang, Zebai Tian
📅 **Published:** 2026-06-21  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper proposes an attested, evidence-bound LLM gateway that prevents a third-party gateway operator from secretly changing providers, triggering hidden fallback, altering streams, or fabricating provenance records. By moving decryption, routing enforcement, upstream call construction, stream protection, and evidence signing into a measured enclave runtime, the system provides clients with verifiable gateway-path provenance and detects tampering fail-closed with modest overhead in a Rust/AWS Nitro Enclaves prototype.

### 🏆 Key Contributions
- Identifies third-party LLM gateways as a gateway-path provenance problem in which routing, fallback, endpoint selection, stream delivery, and usage/provenance records are all authored inside an operator-controlled service.
- Introduces a split-plane architecture that separates the operator business/control plane from a remotely attested Attested Gateway Runtime (AGR), which alone may decrypt requests, enforce policy, contact upstream providers, and sign evidence.
- Defines policy-carrying inference contracts and inference evidence chains that bind runtime attestation, approved policy, selected route, fallback decisions, observed endpoint identity, encrypted stream commitments, and completion metadata into verifiable provenance.
- Implements an initial Rust prototype on AWS Nitro Enclaves and demonstrates modest mechanism overhead plus fail-closed detection of policy, routing, endpoint, and stream-evidence tampering outside the attested runtime.

### ⚙️ Methodology
The system architecture divides the gateway into an untrusted business plane and a trusted execution plane. A release authority reproducibly builds or verifies the runtime image, computes its TEE measurement, associates it with a canonical routing/egress policy, and signs a release registry containing the approved measurement, image hash, policy hash, TEE type, expiry, upstream set, and revocation status. Clients pin the release authority public key and treat gateway-distributed registry data as untrusted; they verify the signed registry, freshness, version, revocation state, and policy/upstream expectations before trusting a runtime.

The core trusted component is the Attested Gateway Runtime (AGR), running inside a confidential-computing TEE such as AWS Nitro Enclaves. Clients verify fresh attestation that binds the AGR measurement to ephemeral encryption and evidence-signing keys. Only after successful verification do clients encrypt the request to keys bound to that measured AGR. This ensures request plaintext is available only inside the attested runtime, not to the gateway control plane or host OS.

Routing and fallback are governed by a signed canonical policy P together with a request-scoped inference contract IC, whose intersection defines an effective policy P_eff. The policy canonically enumerates admitted upstream providers, models, HTTPS endpoints, and allowable fallback edges, preventing alias-based or downgrade substitutions. Inside the AGR, route admission checks ensure the requested provider/model/endpoint and any fallback target are explicitly authorized by P_eff. Thus, fallback is not a hidden operational choice by the gateway but a constrained, evidence-bearing action enforced inside the attested runtime.

Once route and endpoint admission succeed, the AGR constructs the upstream request and invokes the approved external LLM provider over TLS. The upstream provider is outside the trusted computing base; the system does not prove model execution correctness, only that the AGR contacted an admitted endpoint under an admitted policy. The AGR observes endpoint identity, handles fallback decisions when necessary, encrypts response streams back to the client, and computes commitments over the streamed transcript.

The main evidence artifact is an inference evidence chain (IEC), signed by the AGR. The IEC binds together the attested runtime identity, release/policy information, request-scoped contract, selected route, whether fallback occurred, observed upstream endpoint, encrypted stream transcript commitments, and completion metadata such as usage-relevant records. Because this evidence is signed inside the attested runtime that actually processed the request, clients and even the gateway can verify provenance without relying on gateway-authored logs or databases. The design is explicitly fail-closed: if registry signatures, attestation freshness, key bindings, route constraints, endpoint identity, or stream commitments do not verify, the request or response is rejected rather than silently accepted.

### 🧪 Experiments
The paper reports mechanism-oriented evaluation from an initial Rust prototype, rather than a full-scale production benchmark. The prototype is implemented in Rust and deployed with AWS Nitro Enclaves, with additional local/mock and same-host enclave testing paths. The evaluation focuses on security and systems mechanisms: local mock capacity measurements, live-provider probes, attestation behavior, route/fallback/endpoint enforcement, StreamEv or stream-evidence verification, and fail-closed validation under adversarial tampering scenarios.

The experimental setup does not use standard ML datasets or model-quality benchmarks, because the contribution is an inference-security architecture rather than a new LLM model or repair algorithm. Instead, the evaluation compares normal execution against tampering cases and measures operational overhead introduced by attestation, encryption, policy checks, and evidence generation. Baseline comparison is effectively a conventional gateway path where the operator-controlled service handles routing and provenance without attested evidence.

Key reported results are that the prototype adds only modest mechanism overhead while preserving the intended security properties, and that tampering outside the AGR is detected fail-closed. Specifically, the system rejects or detects modified registries, stale attestation, key substitution, route downgrade, unauthorized fallback, endpoint mismatch, stream tampering, and evidence-transfer attempts. The authors emphasize that this version is a preliminary technical report: it establishes feasibility and mechanism behavior, but does not claim a comprehensive production capacity study, broad provider coverage, or full workload evaluation.

### 🌱 Insights & Implications
A key insight is that the main trust problem in LLM gateways is not merely prompt confidentiality but provenance authorship: the same gateway that chooses the path can also fabricate the record describing that path. By binding routing and stream evidence to a measured runtime, the paper reframes gateway trust as a verifiable systems problem rather than a purely contractual one.

For software security and trustworthy AI infrastructure, the work shows how confidential computing and remote attestation can be used to harden middleware layers that sit between applications and external services. This is especially important for agentic workflows where malicious or manipulated outputs may trigger tool calls, code edits, or workflow actions. The architecture reduces the gateway’s ability to inject or alter prompts/streams unnoticed, though it does not stop malicious content originating from the upstream provider itself.

The paper also clearly scopes its guarantees. It does not prove that the upstream provider actually executed the claimed model, nor does it hide traffic metadata such as timing or request frequency. Security still depends on the TEE platform, attestation root, release-signing authority, verifier correctness, and software supply chain. If the gateway operator controls the release-signing key, the guarantee becomes conformance to the operator’s signed release rather than independence from the operator.

Open questions include how to extend the evidence model to multi-hop or cascading gateways, richer billing verification, cross-provider accountability, stronger proofs of upstream model identity/execution, and large-scale performance under realistic workloads. Another research direction is integrating provider-side signed responses or verifiable inference mechanisms so that provenance covers not only gateway path integrity but also provider execution authenticity.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests in software security and secure software infrastructure because it addresses a new supply-chain trust boundary around LLM middleware, using attestation, policy enforcement, and cryptographic evidence to prevent route manipulation and forged provenance. It is also valuable for work on vulnerability detection/repair and automated program repair in agentic settings, since secure and verifiable LLM inference paths are critical when model outputs drive code generation, code transformation, or autonomous tooling where hidden gateway tampering could introduce exploitable behavior or invalidate downstream repair evidence.
