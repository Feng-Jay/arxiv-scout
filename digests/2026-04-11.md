## 📄 [GALA: Multimodal Graph Alignment for Bug Localization in Automated Program Repair](https://arxiv.org/abs/2604.08089v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Zhuoyao Liu, Zhengran Zeng, Shu-Dong Huang, Yang Liu, Shikun Zhang, Wei Ye
📅 **Published:** 2026-04-09  🏷️ **Categories:** cs.SE

### 💡 TL;DR
GALA improves multimodal automated program repair by explicitly aligning GUI screenshot structure with repository and function dependency structure, instead of flattening screenshots into text. On SWE-bench Multimodal, this hierarchical graph-based localization substantially improves repair performance and reaches state-of-the-art results.

### 🏆 Key Contributions
- Reframes multimodal bug localization for APR as a hierarchical cross-modal graph alignment problem rather than text-only semantic matching
- Introduces an Image UI Graph that preserves issue-relevant UI elements and spatial/interactive relations from bug screenshots
- Builds repository-level and function-level code graphs and aligns them with the visual graph in two stages: file-level grounding and function-level grounding
- Uses graph-guided localization to constrain downstream patch generation, yielding more precise edit targets and stronger multimodal APR performance on SWE-bench Multimodal

### ⚙️ Methodology
GALA is a four-stage framework for multimodal automated program repair. First, it constructs an Image UI Graph from the issue screenshot and issue description. Instead of producing a generic caption, the method performs problem-centric visual parsing to identify issue-relevant UI entities as nodes and their spatial or interaction relations as edges, preserving layout and structural information that would otherwise be lost in text conversion. The paper emphasizes a type-aware and rooted graph construction process so that only elements relevant to the reported bug are retained.

Second, GALA performs file-level alignment between the visual graph and repository structure. The repository is represented with file paths and inter-file reference relations. The UI graph is textualized and jointly reasoned over with repository structural information using an LLM. The goal is to map visual observations, such as overlapping widgets or mispositioned components, to the subset of files most likely responsible for the behavior. This stage produces a compact set of candidate or seed files.

Third, GALA performs function-level alignment inside the selected files. It constructs fine-grained code graphs containing function signatures and call-graph dependencies. The LLM then reasons jointly over the UI graph and the code graph to ground visual entities and relations to specific executable components. This stage enforces both node-level semantic consistency and relation-level structural consistency, allowing the model to move from coarse file localization to precise edit-target localization.

Finally, GALA performs graph-guided patch generation. Rather than generating edits over broad repository context, patch synthesis is conditioned on the aligned files and functions found in the previous stages. The aligned graph structure constrains where and how the model edits code, aiming to produce minimal but complete repairs. Overall, the framework shifts multimodal APR from implicit guessing based on image captions and keyword overlap to explicit structural reasoning across visual and code modalities.

### 🧪 Experiments
The evaluation is conducted on the SWE-bench Multimodal benchmark. The paper compares GALA against prior multimodal APR approaches that largely rely on converting screenshots into text and then applying standard localization/repair pipelines, including methods discussed in the related work such as GUIRepair and SVRepair, as well as multimodal adaptations of APR agents like SWE-agent/Agentless-style systems. The core evaluation target is end-to-end multimodal issue resolution quality, with localization quality and repair effectiveness reflected in benchmark performance.

From the provided text, the paper reports that GALA achieves state-of-the-art performance on SWE-bench Multimodal. The claimed gains are attributed to improved bug localization from hierarchical alignment: first grounding visual evidence to candidate files using repository structure, then grounding to functions using call/dependency structure before patch generation. The excerpt provided does not include the detailed result tables, exact baselines, metrics breakdown, or numerical scores, but the main reported outcome is that structure-aware graph alignment outperforms caption-based multimodal APR pipelines.

### 🌱 Insights & Implications
The paper’s main insight is that multimodal APR fails less because models cannot recognize UI objects and more because they cannot preserve and connect visual structure to code structure. By representing screenshots as graphs and aligning them hierarchically with file and function graphs, GALA addresses a key bottleneck in bug localization for GUI-related issues.

This is especially relevant for front-end and UI-intensive software, where the bug signal is often relational, such as overlap, misalignment, missing containment, or incorrect interaction state, none of which are captured well by plain image captions. The work also suggests that retrieval and localization in APR can benefit from explicit intermediate structural representations, not just larger multimodal models.

A limitation is that the method appears tailored to issues with visible GUI manifestations and repositories where structural signals like file references and call graphs are informative. Its effectiveness may depend on the quality of visual parsing, the completeness of dependency extraction, and the reasoning fidelity of the underlying LLM/vLM prompts. Open questions include how robust the approach is when screenshots are noisy or ambiguous, how it scales to complex dynamic UIs, and whether the graph alignment can be learned more formally instead of prompt-based stage-wise reasoning. There is also room to study whether similar structural alignment ideas can help vulnerability localization and repair in settings where external artifacts, traces, or runtime views must be mapped to code.

### 🎯 Why This Paper
This paper is highly relevant to research on automated program repair, software maintenance, and software engineering because it tackles the central APR bottleneck of fault localization under realistic multimodal bug reports. For a researcher interested in APR, vulnerability detection/repair, and repository-scale reasoning, GALA offers a concrete design pattern: use explicit structured intermediate representations and hierarchical alignment to connect external evidence to code. Even though the paper focuses on GUI bugs rather than security bugs, its core idea of graph-based grounding from observations to files/functions could transfer to vulnerability localization from screenshots, traces, logs, or other multimodal artifacts, making it a useful advance for precise, structure-aware repair systems.

## 📄 [Vulnerability Detection with Interprocedural Context in Multiple Languages: Assessing Effectiveness and Cost of Modern LLMs](https://arxiv.org/abs/2604.08417v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Kevin Lira, Baldoino Fonseca, Davy Baía, Márcio Ribeiro, Wesley K. G. Assunção
📅 **Published:** 2026-04-09  🏷️ **Categories:** cs.SE, cs.CR

### 💡 TL;DR
This paper evaluates whether adding caller/callee context helps modern LLMs detect interprocedural vulnerabilities across C, C++, and Python. On 509 ReposVul vulnerabilities, extra interprocedural context usually did not help and often hurt accuracy, while Gemini 3 Flash had the best cost-effectiveness and Claude Haiku 4.5 produced the strongest explanations.

### 🏆 Key Contributions
- Introduces a multi-language empirical evaluation of LLM-based vulnerability detection focused specifically on interprocedural vulnerabilities rather than single-function analysis.
- Compares four modern LLMs—Claude Haiku 4.5, GPT-4.1 Mini, GPT-5 Mini, and Gemini 3 Flash—under three context configurations: target-only, target+callers, and target+callees.
- Quantifies the trade-off between detection effectiveness and inference cost when adding interprocedural context.
- Assesses explanation quality, not just binary vulnerability classification, showing major differences across models in correctness and comprehensiveness.
- Provides practical guidance for building AI-assisted security analysis tools that must balance accuracy, cross-language generalization, and API cost.

### ⚙️ Methodology
The study extracts a subset of interprocedural vulnerabilities from the ReposVul repository-level vulnerability dataset. Starting from ReposVul entries with at least one vulnerable function, the authors identify caller-callee relations using AST construction and retain functions involved in interprocedural dependencies, producing 509 vulnerability cases across C, C++, and Python. They then probe four commercial LLMs with structured prompts that ask the model to classify a target function as vulnerable or non-vulnerable and explain the decision. The experiment varies the amount of code context given to the model in three configurations: (1) target function only, (2) target function plus invoked functions (callees), and (3) target function plus functions that invoke it (callers). This setup is designed to test whether interprocedural code context improves reasoning about vulnerabilities that emerge across function boundaries. The study evaluates both predictive performance and operational efficiency, estimating inference cost from provider token pricing and token usage. For explanation analysis, the authors inspect whether a model’s rationale correctly identifies the vulnerability, its location/behavior, and whether the explanation is comprehensive enough to support security understanding rather than merely outputting a label.

### 🧪 Experiments
The evaluation uses 509 interprocedural vulnerabilities derived from ReposVul and spans three languages: C, C++, and Python. The compared models are Claude Haiku 4.5, GPT-4.1 Mini, GPT-5 Mini, and Gemini 3 Flash. Each model is run under three prompt/input configurations: target-only, target+callees, and target+callers. Metrics include accuracy and F1 for detection, estimated monetary cost based on token consumption and model pricing, and qualitative/expert-style scoring of explanation correctness and comprehensiveness. The main quantitative findings reported are: Gemini 3 Flash achieved the best cost-performance trade-off for C, with F1 >= 0.978 at an estimated cost of about $0.50-$0.58 per configuration; Claude Haiku 4.5 correctly identified and explained vulnerabilities in 93.6% of evaluated cases; adding caller/callee context nearly doubled token usage and cost without proportional gains; and for C, GPT-4.1 Mini and GPT-5 Mini lost up to 25 and 11 percentage points of accuracy, respectively, when interprocedural context was added. Claude Haiku 4.5 and Gemini 3 Flash were comparatively stable across context settings and languages. The GPT models also had zero-score rates for explanation correctness/comprehensiveness about five times higher than Haiku 4.5 and Gemini 3 Flash.

### 🌱 Insights & Implications
A central insight is that more code context is not automatically better for LLM-based vulnerability detection: interprocedural information can introduce distraction, confuse the model, and worsen classification despite being semantically relevant. This is important for tool builders because naive context expansion may harm both accuracy and cost. The results also suggest that language matters: C seems particularly revealing for cost-effective vulnerability detection, while model sensitivity to additional context is language-dependent. Another key takeaway is that explanation quality varies substantially across models; a model can be competitive on classification yet weak at articulating a correct and complete vulnerability rationale, which is crucial for triage and developer trust. A limitation is that the study evaluates only four proprietary LLMs and three languages, so results may shift with newer models, local models, or other ecosystems like Java, JavaScript, or Rust. The paper also studies prompt-based detection rather than integrated agentic/static-analysis hybrids, leaving open whether smarter context selection, slicing, retrieval, or symbolic guidance could make interprocedural context more beneficial. For related research, the paper strongly motivates selective context construction, cost-aware prompting, and explanation-grounded evaluation rather than relying solely on raw classification scores.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated repair research because it directly studies realistic vulnerabilities that depend on cross-function reasoning, a common challenge in both detection and repair pipelines. It advances the researcher’s work by showing when LLMs help or fail on interprocedural security analysis, how to reason about cost/accuracy trade-offs for practical deployment, and why explanation quality should be treated as a first-class evaluation target when building AI-assisted vulnerability detection and repair systems.

## 📄 [VCAO: Verifier-Centered Agentic Orchestration for Strategic OS Vulnerability Discovery](https://arxiv.org/abs/2604.08291v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Suyash Mishra
📅 **Published:** 2026-04-09  🏷️ **Categories:** cs.GT, cs.CR, cs.OS

### 💡 TL;DR
The paper proposes VCAO, a verifier-centered multi-agent framework that treats OS kernel vulnerability discovery as a repeated Bayesian Stackelberg game, using an LRM orchestrator to strategically allocate analysis budget across files, functions, and tools. On five Linux kernel subsystems, it reports higher validated-vulnerability yield per budget than fuzzing-only, static-only, and non-game-theoretic agent pipelines, while substantially reducing false positives sent to human reviewers.

### 🏆 Key Contributions
- Formulates operating-system vulnerability discovery as a repeated Bayesian Stackelberg search game over intra-kernel attack graphs, where the defender is an LRM orchestrator and the attacker chooses exploit paths.
- Introduces VCAO, a six-layer verifier-centered agentic architecture combining surface mapping, attack-graph construction, game-theoretic ranking, parallel executor agents, cascaded verification, and a safety governor.
- Derives a DOBSS-inspired MILP for optimal budget allocation across heterogeneous vulnerability-analysis tools under resource constraints, with Bayesian belief updates over latent vulnerability states.
- Provides an online learning formulation with stated sublinear regret bounds for repeated strategic vulnerability discovery under uncertainty.
- Evaluates the framework on five Linux kernel subsystems using 847 historical CVEs plus live upstream snapshots, showing improved vulnerability yield and lower false-positive burden compared with several baseline strategies.

### ⚙️ Methodology
The method models kernel vulnerability discovery as a Bayesian Stackelberg security game on an intra-kernel attack graph. Nodes represent entry points, internal functions, privilege boundaries, and attacker goals; edges capture control-flow, data-flow, and privilege transitions. Each node is associated with prior probabilities over vulnerability classes, while edges have exploitability scores. The orchestrator chooses actions of the form (target component, analysis method, time budget), where methods include static analysis, fuzzing, sanitizers, patch mining, and explicit verification. Tool outputs such as alerts, crashes, clean results, or timeouts are treated as noisy evidence about latent vulnerability states.

The defender utility combines validated bug value, detection probability, verifier confidence, false-positive cost, and residual attacker impact from uncovered vulnerabilities. The attacker utility is defined over attack paths and attacker types, with rewards for successful exploitation and deterrence cost when detection occurs. The defender commits to a mixed coverage strategy over kernel components, and each attacker type best-responds by selecting an attack path. To compute the defender strategy, the paper adapts the DOBSS approach into a mixed-integer linear program that linearizes the bilevel Stackelberg optimization using path-selection variables and McCormick envelopes for coverage-path interaction terms. Budget constraints, attacker best-response constraints, and one-path-per-type constraints are enforced in the MILP.

Beliefs over whether a function contains a vulnerability of a given class are updated after each action using Bayes' rule with method-specific observation likelihoods. For example, CodeQL alerts use true-positive and false-positive rates, while fuzzing crash probability increases with allocated runtime. Online adaptation is handled via an EXP3-style distribution over discretized coverage strategies combined with Thompson-sampling-like belief handling, yielding a repeated-game orchestration loop: construct/update attack graph, rank files/functions, assign tool budgets in parallel, gather evidence from verifiers, update beliefs, and re-solve the Stackelberg allocation. Architecturally, VCAO is organized as six layers: surface mapping, intra-kernel attack-graph construction, game-theoretic ranking, parallel executors, cascaded verification, and a safety governor to limit unsafe actions.

### 🧪 Experiments
The evaluation is conducted on five Linux kernel subsystems and includes both replay experiments over 847 historical CVEs and live-discovery runs on upstream kernel snapshots. The baselines described in the paper are coverage-only fuzzing, static-analysis-only pipelines, and non-game-theoretic multi-agent orchestration. The main outcome is validated vulnerabilities discovered per unit budget, along with false-positive rate reaching human reviewers and attacker-payoff reduction implied by the game objective.

Reported headline results are that VCAO finds 2.7x more validated vulnerabilities per unit budget than coverage-only fuzzing, 1.9x more than static-analysis-only baselines, and 1.4x more than non-game-theoretic multi-agent pipelines. It also reduces false-positive rates reaching human reviewers by 68%. The paper additionally states that it releases an open-source simulation framework, synthetic intra-kernel attack-graph generator, and evaluation harness, though the provided text does not include finer-grained subsystem-wise breakdowns, statistical significance tests, or ablation details.

### 🌱 Insights & Implications
The paper’s central insight is that vulnerability discovery benefits not just from stronger analyzers, but from strategic orchestration that explicitly optimizes which component-tool-budget combination most reduces expected attacker payoff. This is especially relevant to kernel security, where analysis resources are scarce, attack surfaces are interconnected, and verifier outputs are noisy. By centering external verifiers rather than trusting the reasoning model directly, the framework tries to control hallucinations and reduce triage overhead.

A notable research implication is the move from heuristic scheduling of static analyzers and fuzzers toward principled resource allocation under uncertainty. The intra-kernel attack-graph abstraction also suggests a bridge between software architecture analysis, vulnerability prioritization, and exploit-path reasoning. For automated program repair and vulnerability repair, the belief-update and verification pipeline could be extended to patch generation and patch validation loops.

However, several limitations remain. The work appears to be a systems-and-formulation paper with strong claimed results, but the extracted text does not show implementation depth, baseline tuning fairness, ablations, confidence intervals, or detailed live zero-day validation. The attack graph is assumed acyclic and relies on priors from historical CVEs/CVSS-like signals, which may bias discovery toward known weakness patterns. Solver scalability may be challenging because path-based MILPs can grow rapidly with kernel size. The regret result is sketched rather than fully visible here, and practical convergence under real tool noise is not deeply substantiated in the provided text. Open questions include how well the framework generalizes beyond Linux kernels, how robust it is to inaccurate priors, how to integrate exploit generation and patch synthesis safely, and whether the game objective correlates with downstream remediation impact in real-world secure development workflows.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance, software security, vulnerability detection, and vulnerability repair research because it reframes vulnerability discovery as an optimization-and-orchestration problem rather than a single-tool problem. It directly targets Linux kernel vulnerabilities, integrates static analysis, fuzzing, and sanitizers, and introduces verifier-centered coordination that could be adapted for automated triage, root-cause analysis, and eventually automated repair loops. For a researcher in APR and vulnerability repair, the Bayesian evidence aggregation, strategic prioritization, and verification cascade are especially useful ideas for deciding where to attempt fixes, how to validate them, and how to allocate repair/analysis budget across large codebases.

## 📄 [ORACLE-SWE: Quantifying the Contribution of Oracle Information Signals on SWE Agents](https://arxiv.org/abs/2604.07789v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Kenan Li, Qirui Jin, Liao Zhu, Xiaosong Huang, Yijia Wu, Yikai Zhang, Xin Zhang, Zijian Jin, Yufan Huang, Elsie Nallipogu, Chaoyun Zhang, Yu Kang, Saravan Rajmohan, Qingwei Lin, Wenke Lee, Dongmei Zhang
📅 **Published:** 2026-04-09  🏷️ **Categories:** cs.MA, cs.CL, cs.SE

### 💡 TL;DR
This paper introduces Oracle-SWE, a framework that extracts oracle versions of five key software-engineering signals—reproduction tests, regression tests, edit locations, execution context, and API usage—to measure how much each one could improve a SWE agent if provided perfectly. Across SWE-bench, SWE-bench-Live, and SWE-bench-Pro, the strongest driver of agent success is reproduction-test information, followed by execution context and code-search signals, while regression-test information contributes the least.

### 🏆 Key Contributions
- Defines a unified taxonomy of five critical information signals commonly used by SWE agents: Reproduction Test, Regression Test, Edit Location, Execution Context, and API Usage.
- Proposes generalized extraction pipelines to obtain oracle ground-truth versions of these signals from SWE benchmarks and repositories.
- Performs controlled ablation studies to quantify the upper-bound contribution of each signal to issue-resolution success.
- Shows that signal importance is benchmark-dependent but consistently dominated by reproduction-test information, with execution context and localization/contextual signals next most useful.
- Validates oracle findings in a more realistic setting by having stronger language models infer these signals and passing them to a weaker/base agent, observing similar relative patterns.

### ⚙️ Methodology
Oracle-SWE isolates five intermediate information signals that appear repeatedly in prior SWE-agent pipelines and evaluates them independently. For code-search signals, oracle Edit Location is extracted directly from the gold patch as the modified code regions, annotated with file path, function name, and line numbers. Oracle Execution Context is derived primarily from stack traces produced when running reproduction tests on the buggy repository; each frame is stored with file path, line number, function name, and a surrounding code window. Because ordinary stack traces may not reach the relevant source code—for example in Python assertion failures or Go build failures—the authors instrument the repository with custom stack-trace collectors at oracle edit locations to capture the deepest reachable runtime context. Oracle API Usage consists of function calls added or modified in the gold patch, together with argument information, approximate usage region, and function definitions for non-builtin APIs. Python API-definition resolution uses dynamic execution coverage to link call sites to invoked definitions, whereas Go uses static type checking, imports, and signatures. For testcase-verification signals, oracle Reproduction Test includes the execution command, the fail-to-pass test names, and the full source code of the corresponding test functions; oracle Regression Test contains the command and relevant pass-to-pass tests used to ensure non-regression. The evaluation protocol injects one oracle signal at a time into a base SWE agent, allowing controlled measurement of marginal utility. To assess practical realism, the paper also replaces oracle signals with LM-extracted approximations: a stronger model is prompted to infer a specific signal, and that inferred output is supplied to a weaker/base agent to solve the task. This two-stage setup estimates whether oracle-derived rankings persist when the signal must be generated rather than perfectly given.

### 🧪 Experiments
The paper evaluates on multiple repository-level software engineering benchmarks, including SWE-bench, SWE-bench-Live, and SWE-bench-Pro, with extraction pipelines generalized across Python and Golang repositories. The main experimental design is a controlled signal-injection study: starting from a base agent, the authors provide one oracle signal at a time and measure task-resolution improvements, thereby estimating the ideal upper bound of each signal. They also run a validation setting where a stronger LM extracts each signal and a weaker model consumes it, approximating real deployment. The principal quantitative finding reported in the paper is the relative ordering of contributions. On SWE-bench, the ranking is Reproduction Test > Execution Context approximately Edit Location > API Usage > Regression Test. On SWE-bench-Live and SWE-bench-Pro, the ranking is Reproduction Test > Execution Context approximately API Usage > Edit Location > Regression Test. Although the excerpt does not include the full numerical table values, the reported outcome is that reproduction-test information yields the largest gains by a clear margin, while regression-test information provides the smallest marginal benefit. The LM-extracted-signal validation reproduces the same qualitative pattern as the oracle study, supporting the practical relevance of the ranking.

### 🌱 Insights & Implications
The paper’s main insight is that not all intermediate SWE-agent capabilities are equally important: precise bug-reproduction signals dominate, implying that many current agents fail early because issue descriptions are ambiguous and they cannot reliably turn them into executable failure signals. Execution context and edit localization also matter substantially, suggesting that runtime evidence and accurate repository navigation remain central bottlenecks. API-usage knowledge is moderately useful, with stronger impact on newer/harder benchmarks such as SWE-bench-Live and Pro, potentially reflecting more complex library interactions. Regression-test identification contributes the least, which may indicate that once a candidate patch is generated, existing evaluation infrastructure already provides enough feedback or that regression-test command discovery is a smaller bottleneck than bug understanding. For automated program repair and vulnerability repair researchers, the work implies that investment in test synthesis, failure reproduction, trace collection, and dynamic debugging signals may yield larger gains than focusing solely on patch generation. A limitation is that oracle signals define upper bounds that may still be expensive or impossible to obtain in practice; moreover, some signals are not independent and can interact in non-additive ways. Another open question is how these rankings transfer to security-oriented tasks such as vulnerability-trigger generation, exploit reproduction, or patch verification, where reproduction and regression semantics may differ from general bug fixing.

### 🎯 Why This Paper
This paper is highly relevant because it decomposes repository-level automated repair into concrete intermediate signals and quantifies which ones most strongly affect success, directly informing where to focus effort in software maintenance, automated program repair, and vulnerability repair systems. For a researcher working on software security and vulnerability detection/repair, its emphasis on reproduction tests, execution traces, localization, and API understanding provides an evidence-based roadmap for prioritizing dynamic bug-trigger generation, runtime-context extraction, and repair-time guidance rather than treating SWE agents as monolithic black boxes.

## 📄 [AFGNN: API Misuse Detection using Graph Neural Networks and Clustering](https://arxiv.org/abs/2604.07891v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Ponnampalam Pirapuraj, Tamal Mondal, Sharanya Gupta, Akash Lal, Somak Aditya, Jyothi Vedurada
📅 **Published:** 2026-04-09  🏷️ **Categories:** cs.SE

### 💡 TL;DR
AFGNN is a lightweight graph-neural approach for Java API misuse detection that represents code as API Flow Graphs capturing API call order plus data/control flow, then learns self-supervised embeddings and clusters them to identify common correct usage patterns versus rare likely misuses. On benchmark API-usage datasets and real-world examples, it outperforms prior graph-based misuse detectors and small code language models while using a much smaller model.

### 🏆 Key Contributions
- Introduces API Flow Graph (AFG), a program representation specialized for API usage that combines API execution sequence, data flow, and control flow around API call sites.
- Proposes AFGNN, a self-supervised GNN framework that learns flow-aware embeddings for API usage examples and applies clustering to separate frequent likely-correct patterns from infrequent likely-misuse patterns.
- Creates a manually labeled Java API usage dataset covering widely used APIs for evaluating clustering quality and misuse detection.
- Provides extensive empirical comparison against small code language models and existing API misuse detectors, plus ablation studies validating the importance of AFG design and training choices.

### ⚙️ Methodology
AFGNN operates on method-level Java code snippets. First, each snippet is transformed into an API Flow Graph (AFG), a task-specific graph centered on the target API usage. The AFG is designed to preserve three kinds of information that matter for misuse detection: (1) API execution/call sequence, so ordering constraints such as open-read-close are visible; (2) data-flow relations, so values and objects flowing into and out of API calls are connected; and (3) control-flow relations, so branches, loops, and guards affecting API execution are represented. This graph is intended to avoid the information loss of token-only models and the overhead of heavier whole-program graph constructions.

On top of AFGs, the paper builds a lightweight GNN encoder to produce graph embeddings for API usage examples. The model is trained with self-supervised pre-training rather than requiring fully labeled misuse data, enabling it to generalize to previously unseen or unknown API usage examples. The learned embedding space is expected to place semantically similar API usages close together. After embedding generation, AFGNN applies clustering to group usages into distinct patterns. The central assumption, consistent with prior API-mining work, is that large/frequent clusters correspond to common and usually correct usage idioms, while small or isolated clusters correspond to unusual patterns and are therefore more likely to be API misuses. Thus the framework supports unsupervised or weakly supervised misuse detection by pattern discovery rather than by hand-written rules. The design emphasizes efficiency and deployability: unlike approaches that depend on expensive static analysis over compilable code or very large language models, AFGNN targets partial method snippets and uses a comparatively small model footprint.

### 🧪 Experiments
The evaluation uses popular Java API-usage datasets and a manually labeled dataset introduced by the authors, containing widely used Java APIs and intended for clustering-based API usage analysis. The paper compares AFGNN against state-of-the-art small code language models, specifically models such as CodeBERT, GraphCodeBERT, UniXcoder, and CodeT5+, and also against prior API misuse detectors including graph-based and rule/mining-based systems such as CodeKernel and MuDetect; GraphCode2Vec is discussed as less applicable because it requires compilable Java code while the benchmark contains partial method snippets. The evaluation covers two main tasks: API-usage clustering quality and API misuse detection on real-world examples, supplemented by ablation studies examining the effect of different AFGNN components and representation choices.

The reported results are qualitative in the extracted text rather than fully numeric, but the key finding is consistent throughout: AFGNN significantly outperforms state-of-the-art small language models for understanding API usage patterns and achieves better misuse detection performance than existing misuse detectors on real-world examples. The authors also emphasize that their pre-trained GNN is only a fraction of the size of the compared small language models, implying lower memory and compute cost in addition to better effectiveness.

### 🌱 Insights & Implications
The paper reinforces that API misuse detection benefits from structure-aware representations rather than token-sequence embeddings alone. For software engineering and security, this is important because many API misuses—resource leaks, missing cleanup, incorrect call ordering, absent guards, and control-dependent misuse patterns—are naturally expressed through flow relations, not just syntax. The clustering formulation is also appealing in settings where labeled misuse data is scarce: frequent idioms mined from real code can serve as an implicit notion of correctness.

A notable practical insight is that AFGNN is designed for method-level, even partial and non-compilable code, which better matches mined code snippets from repositories, forums, and generated examples. This makes it potentially useful for IDE assistance, pre-commit analysis, and vetting AI-generated code examples. The work also suggests that compact GNNs can outperform small pretrained code models on specialized program-analysis tasks when supplied with the right graph abstraction.

Open questions and limitations include the dependence on the quality of AFG construction, the assumption that rare patterns are likely incorrect (which may penalize valid but uncommon usages), and likely sensitivity to API-specific variation across libraries and coding styles. Because the approach is clustering-based, explainability and actionable repair suggestions may be weaker than in rule-based systems unless cluster prototypes or nearest-neighbor exemplars are surfaced. Future work could connect AFGNN to automated repair by retrieving frequent-cluster exemplars, extend it to interprocedural or project-level contexts, or adapt the AFG idea to vulnerability detection and security API misuse domains.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, security, automated repair, and vulnerability-oriented research because API misuse is a common root cause of bugs and security flaws, and AFGNN provides a practical detection mechanism tailored to real Java code. It advances your work by combining program-graph representations with lightweight learning and unsupervised pattern mining, offering ideas directly applicable to vulnerability detection, security API compliance checking, AI-generated code validation, and exemplar-based automated repair systems.

## 📄 [An Empirical Analysis of Static Analysis Methods for Detection and Mitigation of Code Library Hallucinations](https://arxiv.org/abs/2604.07755v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Clarissa Miranda-Pena, Andrew Reeson, Cécile Paris, Josiah Poon, Jonathan K. Kummerfeld
📅 **Published:** 2026-04-09  🏷️ **Categories:** cs.CL, cs.SE

### 💡 TL;DR
This paper studies how far static analysis can go in detecting and reducing library-use hallucinations in LLM-generated Python code. Across three NL-to-code benchmarks and four models, static tools catch a meaningful but limited share of problems: they detect 16-70% of all errors and 14-85% of library hallucinations, while manual analysis suggests an inherent ceiling of roughly 48.5-77% for static methods in this setting.

### 🏆 Key Contributions
- Provides a systematic empirical evaluation of static analysis for detecting and mitigating code library hallucinations in LLM-generated programs.
- Compares three static approaches: automatically extracted library grammars, mypy, and pyright, plus an o3-mini judge baseline.
- Introduces a framework that converts Python library docstrings into grammars for both post-hoc validation and grammar-constrained decoding.
- Builds manual annotations over three NL-to-code benchmarks and outputs from both open and closed models, distinguishing general errors from true library hallucinations.
- Quantifies both practical performance and an upper bound on what static analysis can plausibly solve in Python code generation.

### ⚙️ Methodology
The paper defines code hallucination narrowly as violations of external factual knowledge about libraries, APIs, or provided context, distinguishing these from ordinary code errors such as syntax mistakes. It evaluates three static detection strategies. First, the authors automatically derive GBNF grammars from Python library docstrings and alias information; these grammars encode valid library calls and can be used either to validate generated code after the fact or to constrain decoding during generation. For constrained decoding, they integrate the grammar with llama.cpp so that only grammar-valid next tokens are permitted, and they add caching over parser automata states for efficiency. Second, they apply two existing Python static analyzers, mypy and pyright, augmented with automatically generated type stubs for standard and third-party libraries so the analyzers can reason about API availability and signatures. Third, they use o3-mini as an LLM-based executable/non-executable judge as a non-traditional static baseline. For mitigation, when a tool flags issues, the flagged diagnostics are provided to an LLM to repair the code; the grammar method is also tested as a direct prevention mechanism through constrained decoding. The study spans outputs from four LLMs on three Python NL-to-code benchmarks requiring library use, and includes manual error analysis to separate error categories and estimate which hallucinations are fundamentally outside the reach of static methods.

### 🧪 Experiments
The evaluation uses three Python natural-language-to-code benchmarks that require library usage and include test cases: DS-1000, Odex, and BigCodeBench. The authors analyze outputs from four LLMs, covering both open and closed models, and compare detection and mitigation performance for grammar-based validation/constrained decoding, mypy, pyright, and an o3-mini judging baseline. The paper reports that library hallucinations appear in 8.1-40% of model responses depending on the benchmark/model pair. Static analysis tools detect 16-70% of all errors overall, and 14-85% of library hallucinations, with substantial variation by tool, model, and dataset. The grammar-based approach has narrower but still useful coverage, identifying some library-related errors, reported as up to 15% in one summary statement. For mitigation, the detected issues are used to drive repair prompts, while grammar-constrained decoding prevents some invalid library usages during generation. ### 🌱 Insights & Implications
A central insight is that static analysis is worthwhile as a cheap first-pass defense against hallucinated API usage, but it is structurally incapable of solving the full problem. Python's dynamic nature and the gap between surface-valid code and semantically correct library use limit analyzers even when library stubs and grammars are available. Off-the-shelf analyzers offer broader bug coverage than grammars, while grammars are attractive because they can be pushed into decoding and provide hard constraints tied to external API facts. The manual upper-bound analysis is especially useful: it distinguishes failures due to current tool weakness from failures that are fundamentally hard for static methods. For software engineering and security, this suggests static analysis can reduce downstream debugging effort and potentially lower risks from package/API hallucinations, but should be combined with complementary methods such as retrieval, execution-based validation, tests, or repair loops. An open question is how well these results transfer beyond Python, since the study relies on Python benchmarks and tooling, and grammar construction for other languages may require more manual effort. Another implication is benchmark design: many code-generation benchmarks under-emphasize library use, which can hide an important class of hallucination relevant to maintainability and supply-chain security.

This paper is the first systematic study of malicious LLM API routers as a supply-chain attack surface for agent systems, showing that third-party routing intermediaries can rewrite tool calls and steal secrets because no end-to-end integrity protects model outputs. - Formalizes a threat model for LLM API routers as privileged intermediaries in the agent supply chain, where any router hop can inspect and tamper with plaintext tool-calling JSON.
- Defines four adversarial router behaviors: payload injection (AC-1), secret exfiltration (AC-2), dependency-targeted injection (AC-1.a), and conditional delivery / adaptive evasion (AC-1.b).
- Presents the first ecosystem-scale empirical measurement of malicious and risky LLM routers using 28 purchased paid routers and 400 collected free routers.

### ⚙️ Methodology
The study models LLM API routers as intentional application-layer man-in-the-middle components that terminate inbound TLS, inspect plaintext JSON requests/responses, and re-originate upstream connections. The authors formalize a multi-hop routing threat model in which clients often trust only the first endpoint while hidden downstream routers may further forward traffic; because no end-to-end integrity spans the chain, a single malicious hop can taint the entire response path. They define two primary attack classes. AC-1 payload injection modifies returned tool calls so the agent executes attacker-controlled commands, such as replacing benign installer URLs or shell commands with malicious ones. AC-1.a is a dependency-targeted variant that tailors malicious rewrites to project context, such as selectively altering package installs in Rust or Go environments. AC-1.b is a conditional-delivery or adaptive evasion variant in which the router triggers payloads only after warm-up periods, only in specific execution modes like YOLO/autonomous approval, or only under contextual cues to avoid researcher detection. AC-2 secret exfiltration captures credentials and other sensitive data present in requests, prompts, tool arguments, or model outputs.

To operationalize the taxonomy, the authors build Mine, a controllable research proxy that sits between agent clients and upstream providers and can implement all four behaviors. Mine is evaluated against four public agent frameworks, letting the authors study how router-side tampering propagates into client-side tool execution. The paper also includes two poisoning studies to examine indirect exposure. In the first, the authors intentionally leak a researcher-owned OpenAI API key on public Chinese-language channels and observe how routers reuse it. In the second, they deploy intentionally weakly configured decoy relay services across multiple domains and IPs to attract unauthorized routing traffic and observe downstream agent sessions, credential exposure, and autonomous execution settings.

Response-side anomaly screening detects suspicious characteristics in model/router outputs without needing provider changes. Append-only transparency logging records request/response artifacts for forensic accountability and post-hoc tamper detection. These are evaluated as deployable mitigations under today’s ecosystem constraints, while the paper argues they are partial remedies compared with cryptographic end-to-end response integrity.

### 🧪 Experiments
The empirical study covers two router populations: 28 paid routers purchased from Taobao, Xianyu, and Shopify-hosted storefronts, and 400 free routers collected from public communities, largely based on common templates such as sub2api and new-api. The measurements look for active code injection, adaptive evasion, secret access, and abuse of planted canary assets. Results show that 1 of the 28 paid routers and 8 of the 400 free routers actively injected malicious code into returned tool calls. Two routers exhibited adaptive evasion behavior, including triggers such as waiting for roughly 50 calls, targeting only specific language ecosystems, or delivering payloads only when the agent operated in autonomous YOLO mode. For credential abuse, 17 routers touched researcher-owned AWS canary credentials, and one router used a planted Ethereum private key to drain ETH, demonstrating real monetary harm rather than merely theoretical risk.

The poisoning studies quantify indirect exposure through ostensibly benign infrastructure. In the leaked-key study, a deliberately leaked OpenAI key generated about 100M GPT-5.4 tokens and more than seven Codex sessions. In the weak-decoy study, the deployed decoy relays across 20 domains and 20 IPs attracted tens of thousands of unauthorized access attempts from 147 IPs spanning 6 JA3 fingerprints. These decoys served about 2B GPT-5.4 / 5.3-codex tokens, exposed roughly 13 GB of visible downstream prompt/response traffic, and revealed 99 credentials across 440 Codex sessions covering 398 distinct projects or hosts. All 440 sessions were command-injectable, and 401 were already running in autonomous YOLO mode, meaning direct payload injection would likely succeed without extra user confirmation.

For defenses, Mine is used against four public agent frameworks to test client-side mitigations. The fail-closed policy gate blocked all AC-1 and AC-1.a shell-rewrite samples at 1.0% false positives. Response-side anomaly screening flagged 89% of AC-1 samples without requiring any provider-side cooperation. The paper presents these as useful immediate safeguards, though not complete solutions.

### 🌱 Insights & Implications
The key insight is that LLM security for agents is no longer just a prompt-injection or model-alignment problem; it is also a software supply-chain problem centered on routers that have full authority over executable tool-call payloads. This is especially important for agent frameworks used in software engineering, code generation, CLI automation, and cloud operations, where a single rewritten command can directly alter source code, install malicious dependencies, or exfiltrate secrets. The paper also shows a weakest-link property in multi-hop routing chains: even if most routers are honest, any one malicious intermediary can compromise the final action observed by the client, while downstream honest routers cannot recover integrity once tampering occurs.

Another important takeaway is that attacks can be stealthy and context-aware. Adaptive evasion based on warm-up periods, language-specific dependency targeting, or autonomous execution mode means naive one-shot auditing will miss some malicious routers. The poisoning studies broaden the threat from explicitly malicious services to misconfigured or opportunistic ones: leaked API keys and weak relays can silently pull user traffic into dangerous chains, exposing credentials and enabling command injection at scale.

The paper’s client-side defenses are practical but incomplete. Policy gates and anomaly detectors reduce exposure today, yet they are brittle against sophisticated payload shaping, semantic-preserving malicious rewrites, or attacks that mimic benign commands closely. Transparency logging improves accountability but does not itself prevent exploitation. The main open systems question is how to deploy provider-backed cryptographic integrity for tool-calling outputs across heterogeneous routing ecosystems and multi-hop relay chains. Other open issues include standardizing provenance metadata for tool calls, hardening agent frameworks against unsafe autonomous execution, and designing robust detection of router-induced tampering in code-oriented workflows.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, automated repair, vulnerability detection, and vulnerability repair because it exposes a new supply-chain threat model for code-capable LLM agents and the API infrastructure they depend on. Any research or tooling that uses LLM agents for patch generation, code modification, dependency management, DevOps, or autonomous remediation can be silently subverted by malicious routers that rewrite tool calls or steal credentials. The structure configuration generated by Onion was accepted without modification. Keeping this structure fixed, the author performed five full generation runs from scratch for each model, collecting developer interactions, generated artifacts, and logs. The main qualitative outcome is that all 10 runs succeeded. In none of the runs was direct modification of production code required. Most failures during the process were due to inconsistencies in generated test code rather than implementation code; once these were fixed, production code could be generated correctly. In two runs total, one per model, comments had to be added to the tests to steer the model because it repeatedly generated code that failed the same tests. GPT-4o-mini required test-code modification before implementation checking in only one try, whereas Gemini 2.5-Flash more often produced issues in supporting code such as imports or test execution. The paper does not report standard quantitative SE metrics such as pass@k, repair rate, exact-match accuracy, or timing statistics; instead, it emphasizes success/failure outcomes, degree of human intervention, and qualitative differences in generated code. GPT-4o-mini produced shorter, less commented code, while Gemini 2.5-Flash produced longer and more heavily commented outputs.

### 🌱 Insights & Implications
The paper suggests that executable tests may serve as a practical intermediary specification layer for LLM-based development, reducing the need for developers to manually inspect production code and potentially raising the abstraction level of programming. This is particularly relevant to assured LLM-based software engineering, where verifiable claims about generated outputs are essential. However, the study also exposes several important limitations. Second, generated tests themselves may be inconsistent or incorrect, meaning the trustworthiness of the whole process depends on robust validation of test artifacts. Third, output non-determinism across runs undermines predictability and reproducibility. Open questions include how TOP scales to large modular systems, how to automate test-quality assessment, how to integrate static analysis or formal verification, and whether TOP can support security-critical or maintenance-heavy codebases where flawed tests may encode vulnerabilities or miss regressions.

### ⚙️ Methodology
ZeroCoder uses a single LLM conditioned into two roles: a coder and a tester. For each programming problem x, the coder samples N candidate solutions and the tester samples M candidate tests. Every solution-test pair is executed to produce a binary passing matrix E in which each cell indicates whether a solution passes a test. A selector then analyzes E to choose a consensus subset of likely-correct solutions and consistent tests. These selected subsets act as pseudo-supervision for reinforcement learning: the coder is rewarded for agreement with the consensus of stronger solutions/tests, while the tester is rewarded for generating valid and informative tests that support reliable discrimination among candidate solutions. The framework is explicitly label-free in its main setting, relying only on execution feedback rather than reference solutions or curated tests. To avoid uninformative training instances, ZeroCoder performs offline rank-based pre-filtering on passing matrices, keeping problems with sufficient interaction diversity; low-rank matrices correspond to problems that are too easy or too hard, which would otherwise yield nearly uniform rewards and weak policy gradients. Because the tester is initially weaker than the coder, ZeroCoder also uses a curriculum for tester training: it first emphasizes producing valid tests aligned with selector-induced consensus, then gradually increases the weight of mutation-based rewards that encourage tests to distinguish correct code from subtly faulty implementations. The selector component is pluggable; the paper instantiates MaxPass, CodeT, and B4. After observing that fixed selectors degrade over time due to changing coder/tester reliability, the authors propose DyB4, a dynamic Bayesian selector that recalibrates prior hyperparameters during training using a tiny labeled calibration set, thereby reducing selector drift and improving reward quality.

### 🧪 Experiments
The evaluation covers three base models—Qwen2.5-1.5B-Instruct, Qwen3-4B, and Qwen2.5-Coder-7B-Instruct—over six benchmarks: four for code generation and two for test generation. The paper compares ZeroCoder instantiated with different selectors (MaxPass, CodeT, and B4), a stronger variant using DyB4, the original base models, and oracle-supervised training that uses ground-truth verification resources. Metrics focus on code-generation and test-generation performance, reported primarily as relative improvements over the base model, with additional analysis of selector quality and component ablations. In the fully label-free setting, ZeroCoder consistently improves both code and test generation across models and benchmarks. On Qwen2.5-Coder-7B-Instruct, ZeroCoder with B4 improves code generation by up to 14.5% and test generation by 15.6% relative to the base model. When equipped with DyB4, the gains rise to 21.6% for code generation and 24.3% for test generation, approaching oracle-supervised performance. Averaged across the three models and six benchmarks, ZeroCoder with DyB4 improves code generation by 18.8% and test generation by 62.7% over the base models. The paper also reports ablation-style findings indicating that both rank-based pre-filtering and mutation-driven tester rewards are important contributors to the final performance.

### 🌱 Insights & Implications
The paper’s main insight is that code generation and test generation should be treated as mutually reinforcing tasks rather than optimizing one with the other fixed. This is especially relevant for software engineering workflows where verifiers are scarce, because it suggests a path to bootstrap both generators from raw problem descriptions alone. Another important insight is selector drift: consensus rules that are reasonable at one training stage can become noisy as the relative reliability of generated code and tests changes, which has broader implications for self-training and self-rewarding systems beyond code. The work also highlights that reward quality in execution-based RL depends not just on verifier availability but on verifier discriminativeness; weak tests can silently reward spurious programs. Limitations include dependence on executing many sampled code-test pairs, which may be computationally expensive and potentially difficult to scale; reliance on selector quality, which remains a central bottleneck; and the fact that DyB4 is no longer fully label-free because it uses a small labeled calibration set. Open questions include how well the framework generalizes to more realistic software engineering tasks such as repository-level code repair, vulnerability detection/repair loops, or long-horizon program synthesis; whether stronger mutation strategies could better approximate adversarial testing; and how to make selector calibration more principled and fully unsupervised.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, automated program repair, and vulnerability-related research because it tackles a core bottleneck: improving code models when curated tests or labels are scarce. Its co-evolutionary code-and-test training paradigm is directly applicable to maintenance and repair settings where execution feedback exists but gold patches/tests do not, and its ideas on discriminative testing, consensus-based reward construction, and dynamic selector calibration could inform new methods for vulnerability repair, patch validation, regression test generation, and label-efficient RL for secure code generation.

The study combines empirical software engineering and action research. In the exploratory stage, the authors mined redesign-related evidence from artifacts such as issues, pull requests, commits, and cross-references between the original and redesigned repositories, primarily for the ongoing Soot/SootUp migration. They formalize the target task as identifying method-level code mappings between an original project and its redesign. Because standard clone detectors are not designed to distinguish code mappings from irrelevant clones, the authors retrofit them with Semantic Alignment Heuristics. These heuristics prioritize natural-language and naming signals preserved during redesign, including identifier similarity, signatures, and comments, to estimate whether two methods are semantically aligned in the redesign sense.

The approach computes a Semantic Alignment Score (SAS) for candidate method pairs and uses the score as a filter to remove unlikely or irrelevant pairs while retaining those likely to represent code mappings. The paper highlights similar identifier name matching as the strongest heuristic. To make expensive LLM-based detection practical, the authors add a hierarchical detection strategy: first perform class-level pre-filtering to reduce the candidate search space, then run finer-grained method-level analysis only on likely class pairs. The methodology is validated not only through offline evaluation but also through action-research interventions, where discovered mappings and mismatches were used to report bugs, port tests, and contribute fixes to the open-source projects.

### 🧪 Experiments
The evaluation uses two redesigned project pairs: the ongoing Soot/SootUp redesign and the completed FindBugs/SpotBugs migration, allowing both in-depth longitudinal study and external validation. Four representative method-level clone detection approaches were assessed: NiCad, DeepSim, CCSTokener, and an LLM-based approach. The authors built a benchmark of 1,749 sampled method pairs from these tools' outputs and manually labeled them as genuine clones, code mappings, irrelevant clones, or non-clones.

The evaluation focuses on the ability to reduce irrelevant clones while preserving useful code mappings, with particular attention to precision and manual inspection burden. Prior tools were found to generate many false positives, especially on test code, with false positive rates reported up to 87%. A central insight is that software redesign is not a clean one-way rewrite; it is a prolonged, non-linear process involving parallel maintenance of the old and new systems, deferred migration via TODOs, under-ported tests, and bug propagation across versions. This makes traditional artifact tracking and commit-based migration analysis insufficient, especially when redesign spans repositories and diverges architecturally.

The work also suggests that stable semantic anchors could be used in future systems to guide automated adaptation, code translation, or test migration across redesigned architectures. Open questions include how well the heuristics generalize beyond static analyzers, how to improve mapping for heavily transformed test code, and how to integrate mapping with automated patch transfer or vulnerability-fix propagation.
