## 📄 [Acoda: Adversarial Code Obfuscation for Defending against LLM-based Analysis](https://arxiv.org/abs/2606.11755v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Hongzhou Rao, Zikan Dong, Yanjie Zhao, Haodong Li, Haoyu Wang
📅 **Published:** 2026-06-10  🏷️ **Categories:** cs.SE

### 💡 TL;DR
Acoda is a genetic-algorithm-based obfuscation framework that crafts semantics-preserving code transformations to make LLMs refuse, truncate, or misinterpret code analysis. The paper shows that by targeting safety-alignment cues and token-processing behavior, these adversarial obfuscations transfer across multiple modern LLMs and can reach up to 70% attack success rate.

### 🏆 Key Contributions
- Introduces Acoda, an adversarial code obfuscation framework designed specifically to defend source code against LLM-based analysis while preserving program semantics.
- Designs 8 obfuscation methods grounded in two shared LLM mechanisms: safety alignment and token-based information processing.
- Proposes an LLM-response-based evaluation framework that uses an auxiliary LLM plus four metrics to quantify refusal and misanalysis of obfuscated code.
- Shows strong empirical effectiveness, transferability across model families, and low runtime overhead compared with random obfuscation selection.

### ⚙️ Methodology
Acoda first identifies two universal LLM behaviors that can be exploited defensively: safety alignment, which can be triggered by making code look risky or harmful, and token-based processing, which can be disrupted by inserting special boundary/control tokens. Based on these mechanisms, it defines eight semantics-preserving obfuscation transformations that modify code structure and surface form without changing functionality. Acoda then uses a genetic algorithm to search over combinations of these transformations, iteratively evolving candidate obfuscations toward higher defensive effectiveness. Fitness is measured by how target LLMs respond to the obfuscated code, with an auxiliary LLM interpreting outputs and four metrics aggregating refusal, incorrect interpretation, and other response-quality signals. The framework is explicitly designed to remain useful in black-box and white-box settings and to generalize across model families rather than relying on model-specific internals.

### 🧪 Experiments
The paper evaluates Acoda on seven state-of-the-art LLMs, spanning proprietary and open-source families, including GPT-4o, DeepSeek, Qwen, Llama, and Gemma variants. The study is organized around four research questions: whether the genetic algorithm outperforms random obfuscation selection, whether adversarial samples transfer across models of different sizes and families, what runtime overhead the obfuscations introduce, and how effective each individual method is in ablation. A preliminary study also tests the two targeted LLM mechanisms on ten Python samples from CodeNet using DeepSeek-Coder-6.7B and CodeLlama-7B. Key results show that Acoda can induce refusals or misanalysis with an attack success rate up to 70%, exhibits strong cross-model transferability, and adds minimal execution overhead while preserving original semantics. The paper reports that the genetic algorithm produces more effective adversarial samples than random selection.

### 🌱 Insights & Implications
The main insight is that LLMs used for code analysis can be defensively manipulated through mechanisms that are largely shared across model families, not just through model-specific prompt hacks. This makes adversarial obfuscation a plausible code-protection strategy for intellectual property and trade-secret defense. At the same time, the approach depends on safety-alignment behavior and tokenization quirks, so its effectiveness may change as models are retrained or their safety layers evolve. The reliance on an auxiliary LLM for scoring also introduces evaluation subjectivity and potential bias. For vulnerability and software-security research, the paper highlights a new adversarial setting: protecting code from automated semantic analysis rather than detecting or repairing vulnerabilities.

### 🎯 Why This Paper
This paper is directly relevant because it sits at the intersection of software security, LLM security, and automated code transformation. It advances research on protecting source code from model-based reverse engineering, and its mechanisms and evaluation ideas are useful for studying how LLMs interpret vulnerable-looking code, how analysis can be misled, and how semantics-preserving transformations can be optimized for security goals.

👤 **Authors:** Akeela Darryl Fattha, Kia Ying Chua, Lingxiao Jiang, Laura Wynter
📅 **Published:** 2026-06-10  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
The paper argues that linear, single-threaded repository exploration is a poor fit for multi-file bug localization and shows that domain-scoped parallel agents can do better. On SWE-bench Pro/Ansible-based benchmarks, the proposed exploration structure is the strongest small-model approach and is competitive with a much larger Codex baseline, though documentation files remain a persistent failure mode.

### 🏆 Key Contributions
- A persistent-session benchmark for multi-file change localization built from SWE-bench Pro issues anchored at a single base commit and extensible to newer PRs.
- A domain-scoped parallel multi-agent exploration strategy that assigns persistent specialists to repository subsystems instead of exploring linearly.
- A controlled comparison against a plain LLM, a single-agent Recursive Language Model with a persistent Python REPL, BM25 retrieval, and an external Codex 5.5 High CLI baseline.
- Empirical evidence that exploration structure matters: subsystem-parallel agents outperform linear small-model baselines, while naive file-system access and forced multi-agent consultation can hurt accuracy or waste tokens.

### ⚙️ Methodology
The study isolates file localization from patch generation and test execution by evaluating models on a persistent-session benchmark derived from SWE-bench Pro, focused on ansible/ansible and anchored to a single base commit. The benchmark is built using a reproducible sliding-window commit-fixing heuristic and is also extended with more recent PRs from 2025 and 2026. The authors compare several systems under a shared evaluation setting: a plain LLM with no direct repository access, a single-agent Recursive Language Model (RLM) using a persistent Python REPL, a proposed domain-scoped multi-agent system, BM25 retrieval, and an external CLI baseline using Codex 5.5 High. The key architectural idea is to spawn persistent agents specialized to repository subsystems such as `cli/`, `module_utils/`, `galaxy/`, and `docs/docsite/`, so exploration proceeds in parallel across coherent code regions rather than one file or directory at a time. For fairness, the plain-LLM baseline, the domain-agents, and one RLM variant use the same small Haiku-class model; Sonnet variants are used as larger-model references. The evaluation emphasizes localization quality metrics including micro-F1, precision, recall, all-gold recall, hard all-gold rate, and token cost, and it also compares methods against the BM25 precision-recall envelope by sweeping `K` from 1 to 30.

### 🧪 Experiments
The main experiments are run on two benchmarks: the original curated 2020 SWE-bench Pro benchmark for Ansible and the authors' expanded PR-based benchmark that includes newer issues from 2025 and 2026. Baselines include BM25 retrieval, a plain LLM without repository access, a single-agent RLM with persistent Python REPL support, the proposed subsystem-parallel domain-agents, and a Codex 5.5 High external CLI baseline. Results are reported with micro-F1, precision/recall tradeoffs, all-gold recall, hard all-gold rate, and token usage. The main findings are that the domain-agents system achieves the highest micro-F1 among Haiku-class models by a large margin and is second overall on the expanded benchmark, behind only Codex 5.5 High. On the curated SWE-bench Pro benchmark, a larger Sonnet plain-LLM baseline attains a higher micro-F1 by predicting very few files, which raises precision but substantially lowers all-gold recall. The paper also reports that most LLM methods outperform the BM25 envelope, but the Haiku plain LLM and Haiku-hinted RLM fall below it. Additional quantitative observations are that the domain-agents method attains the highest hard all-gold rate among small-model-class systems, forced multi-agent consultation increases token cost without measurable gain, and naive file-system access can reduce localization accuracy due to over-prediction of test files.

### 🌱 Insights & Implications
The central insight is that exploration structure is a real factor in multi-file localization: parallel, subsystem-scoped search can recover gold files that a linear agent may never reach within its budget. At the same time, the work shows that architecture alone is not enough: documentation co-evolution is a latent dependency that none of the tested approaches solve, suggesting that documentation-aware retrieval or modeling is needed. The paper also highlights a counterintuitive failure mode where giving agents raw file-system access can hurt because they wander into large test hierarchies and over-predict irrelevant files. Another negative result is that forcing multiple agents to consult one another does not measurably improve recall and instead raises token cost, so more agents is not automatically better. Limitations include a focus on Ansible-derived SWE-bench Pro issues, evaluation centered on file localization rather than full repair, and reliance on a fixed-base-commit persistent-session setup, which may not capture all real-world repository evolution patterns. More broadly, the work suggests that future repair and localization systems should combine repository-subsystem parallelism with better dispatch, documentation handling, and tighter control over navigation noise.

### 🎯 Why This Paper
This paper is directly relevant to software maintenance and automated program repair because accurate file localization is a prerequisite for generating correct patches, including vulnerability repairs. Its comparison of linear versus parallel agent exploration, its benchmark construction for persistent repository sessions, and its findings about failure modes such as noisy filesystem access and documentation drift are especially useful for designing stronger localization and repair agents in both bug-fixing and security-vulnerability settings.

## 📄 [Grammar-Constrained Decoding Can Jailbreak LLMs into Generating Malicious Code](https://arxiv.org/abs/2606.11817v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yitong Zhang, Shiteng Lu, Jia Li
📅 **Published:** 2026-06-10  🏷️ **Categories:** cs.CR, cs.AI, cs.CL, cs.SE

### 💡 TL;DR
The paper shows that grammar-constrained decoding, which is usually used to make code generation more reliable, can paradoxically help jailbreak LLMs into emitting malicious code by removing natural-language refusal as an option. The authors propose CodeShield, a defense that teaches models to respond with harmless but valid "honeypot" code under grammar constraints, restoring safety while preserving useful code generation.

### 🏆 Key Contributions
- Introduces CodeSpear, a jailbreak attack that exploits standard grammar-constrained decoding with benign code grammars to force LLMs into the code modality and produce malicious code.
- Identifies a new security risk in safety alignment: many aligned models are safe mainly in the natural-language modality, so grammar constraints can suppress refusals and expose unsafe code generation behavior.
- Proposes CodeShield, a code-modality safety alignment method based on DPO that trains models to output semantically harmless, structurally diverse honeypot code when refusals are unavailable.
- Provides a broad empirical evaluation across 10 popular LLMs and 4 benchmarks, including both local and API-based models, showing that the attack is effective and the defense is robust.

### ⚙️ Methodology
The attack side, CodeSpear, assumes an inference interface that supports grammar-constrained decoding (GCD). Given a malicious code-generation prompt, the attacker supplies a standard benign code grammar. GCD masks tokens that would violate the grammar, shrinking the output space to syntactically valid code. Because many safety-aligned LLMs primarily learn to respond to harmful prompts with natural-language refusals, the grammar constraint removes that refusal channel and forces continuation in code form, often yielding malicious code. The defense side, CodeShield, keeps the same DPO-style preference optimization framework but changes the training targets to the code modality. Instead of training the model to prefer natural-language refusals, it trains the model on malicious prompts to prefer harmless honeypot code over harmful completions under GCD. The honeypot code is intended to be semantically safe while remaining syntactically varied, so the safe behavior is harder to eliminate by tightening the grammar. CodeShield is designed to preserve natural-language refusals when natural language is allowed, while adding a second safety behavior for constrained code generation.

### 🧪 Experiments
The paper evaluates CodeSpear and CodeShield on 10 popular LLMs, including examples such as GPT-5, MiniMax-M2.7, and Qwen2.5-Coder models, across 4 benchmarks. The evaluation covers both locally deployed models and commercial API-based models. The main metric is attack success rate (ASR), with additional assessment of benign utility on code-generation benchmarks and robustness across hyperparameter settings. Reported results show that CodeSpear outperforms representative jailbreak baselines and increases ASR by more than 30 percentage points on average; on locally deployed models it achieves an average ASR of 81.82%, and on some commercial API-based models it raises ASR by more than 40 percentage points on average. CodeShield substantially reduces ASR under CodeSpear, in some cases to below the no-attack level, while causing only minor degradation on benign code-generation performance.

### 🌱 Insights & Implications
A central takeaway is that decoding-time reliability mechanisms can become security liabilities when they constrain the model's output space in ways safety training did not anticipate. The work suggests that aligning only the natural-language refusal channel is brittle for code assistants, because a benign grammar can suppress refusals without changing model weights. More broadly, it argues for modality-aware safety alignment: if a model can be forced into code generation, it needs a safe behavior policy in that modality as well. An open question is how well honeypot-code alignment transfers across different languages, grammars, and deployment settings, and whether future attackers could exploit other output-space restrictions beyond GCD. The paper also implies that grammar control in coding tools, IDE assistants, and automated repair pipelines should be treated as part of the threat surface, not just as a reliability feature.

### 🎯 Why This Paper
This paper is highly relevant to software security and automated program repair because it studies how code-generation systems can be manipulated into producing harmful code through a decoding-time mechanism that is already widely used in practical tooling. It directly informs secure deployment of coding assistants, vulnerability-adjacent misuse of LLMs, and the design of safer code-generation and repair systems by showing that syntactic validity constraints can interact badly with safety alignment.

## 📄 [MPC-Patch-Bench: Security-Aware LLM Code Patch for Multi-Party Computation](https://arxiv.org/abs/2606.11416v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yukuan Zhang, Mengxin Zheng, Qian Lou
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
MPC-Patch-Bench is a new repository-level benchmark for LLM code repair on secure multi-party computation software, built to capture both functional correctness and MPC-specific security/fidelity constraints. Across eight LLMs, the best model solved only 22.9% of tasks functionally and 17.1% under the security-aware verifier, showing that ordinary pass/fail evaluation overestimates repair quality in privacy-preserving code.

### 🏆 Key Contributions
- Introduces MPC-Patch-Bench, the first repository-level benchmark focused on LLM repair for MPC software.
- Proposes a domain-specific data curation framework that filters raw pull requests through MPC-aware layers and rescues incomplete but valuable issues with human-AI completion.
- Builds 205 fully verified benchmark instances from 7,305 raw pull requests across five MPC frameworks: CrypTen, SecretFlow, MP-SPDZ, PySyft, and tf-encrypted.
- Designs an MPC Verifier that combines dynamic differential testing against plaintext oracles with static rules for MPC-specific security violations.
- Shows that many functionally correct patches are still invalid for MPC because they leak secrets, use insecure arithmetic, or violate public/private type discipline.

### ⚙️ Methodology
The benchmark is constructed in two stages. First, the Data Curation Framework crawls GitHub pull requests from five MPC frameworks and uses a domain-specific curation agent to retain only fixes that exercise genuine MPC reasoning rather than generic infrastructure. The agent applies three stack-aware layers: a primitive check for direct manipulation of cryptographic building blocks such as secret sharing and oblivious transfer, a protocol-logic check for fixes involving interactive MPC state machines and conversions such as arithmetic-to-binary transformations, and a domain-specificity check for MPC-exclusive constraints like data-obliviousness, finite-ring behavior, and public/private type discipline. Candidates that pass this filter are then passed to a human-AI completion engine, which reconstructs missing problem statements and synthesizes fail-to-pass and pass-to-pass tests for high-value PRs that lacked standard evaluation artifacts. This yields executable, fully verified instances with gold patches.

Second, the MPC Verifier evaluates model-generated patches beyond ordinary unit tests. Its dynamic component performs differential testing between secure MPC execution and a plaintext oracle to detect numerical-fidelity drift and semantic mismatch. Its static component flags MPC anti-patterns such as unsafe reveals, insecure arithmetic, illegal public/private casts, and related security-irrelevant but MPC-breaking code patterns. Together, these checks enforce both functional correctness and privacy-preserving correctness properties that standard repository-level benchmarks do not capture.

### 🧪 Experiments
The authors crawled 7,305 raw pull requests from five MPC repositories: CrypTen (236), SecretFlow (658), MP-SPDZ (110), PySyft (5,840), and tf-encrypted (461). The curation pipeline reduced these to 1,175 cryptographically relevant candidates and then to 205 fully verified benchmark instances. They evaluated eight LLMs on MPC-Patch-Bench. The strongest model resolved 22.9% of tasks under functional evaluation, but only 17.1% remained valid under the MPC Verifier. Up to 40% of patches that passed standard tests were rejected by the security-aware verifier for cryptographic or numerical-fidelity violations. The paper positions this result as evidence that fail-to-pass metrics alone substantially overestimate LLM performance on MPC repair tasks.

### 🌱 Insights & Implications
The main insight is that MPC repair is not just ordinary software maintenance in a privacy-preserving codebase; it requires reasoning about secrecy, protocol semantics, and numerical fidelity. Standard benchmarks like SWE-bench are structurally mismatched because MPC repositories contain much generic infrastructure, many valuable fixes lack standardized tests, and security violations are invisible to ordinary test suites. A major limitation is that the benchmark depends on curated and partially human-completed instances, so it may reflect the availability of expert oversight and test synthesis as much as real-world bug-fix frequency. Another open question is how well the verifier generalizes to other MPC frameworks or to different classes of privacy-preserving systems, and whether the static rules miss subtler semantic leaks or protocol-level weaknesses. The large gap between functional and verified success suggests future APR systems for security-sensitive software need security-aware generation and evaluation, not just test passing.

### 🎯 Why This Paper
This paper is highly relevant because it sits directly at the intersection of software repair, vulnerability-aware evaluation, and privacy-preserving systems. It advances the researcher's interests by introducing a realistic repository-level benchmark for automated program repair in MPC, and by showing that conventional repair metrics miss security and numerical-fidelity failures that matter in vulnerability detection and repair. The benchmark and verifier provide a concrete testbed for studying how LLMs handle secure code, hidden correctness constraints, and security regressions in a maintenance setting.

## 📄 [Enhancing LLM-Based Code Translation with Verified Multi-Semantic Representations](https://arxiv.org/abs/2606.11863v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Yufu Wang, He Jiang, Hao Lin, Peiyu Zou, Ang Jia, Xiaochen Li, Zhilei Ren
📅 **Published:** 2026-06-10  🏷️ **Categories:** cs.SE

### 💡 TL;DR
Multisage improves LLM-based code translation by extracting source-code semantics, turning them into multiple complementary semantic views, and then verifying those views before using them to guide generation. On HumanEval-X, it raises translation success rates by up to 2.22x, with especially strong gains for smaller models.

### 🏆 Key Contributions
- A semantic parsing module that statically extracts base semantics from source code, including data-flow graphs, type constraints, and external API information.
- A multi-semantic augmentation module that synthesizes richer guidance such as code summaries, function-level test cases, and API-oriented descriptions and tests.
- A semantic consistency calibration module that uses semantics-preserving mutations and cross-semantic verification to filter, correct, and refine noisy semantic signals.
- A broad empirical study showing that verified multi-semantic guidance consistently improves multiple LLM backbones and outperforms prompting-only, instruction-tuned, CoT, and specialized translation baselines.

### ⚙️ Methodology
Multisage is a front-end augmentation and verification framework rather than a new translator model. It first performs static analysis on the source program to extract structured base semantics: data-flow relations, type constraints, and external API usage. These base representations are then expanded into multiple complementary semantic views, including natural-language function summaries, generated unit tests at the function level, and API-focused descriptions and tests. Because automatically generated semantics can be incomplete or contradictory, the framework applies a calibration stage that mutates code in semantics-preserving ways and checks whether each semantic view remains consistent under those perturbations. Inconsistent or unreliable semantic signals are filtered or revised before being injected into the LLM translation prompt, so the downstream model receives richer and more trustworthy behavioral guidance.

### 🧪 Experiments
The paper evaluates Multisage on the HumanEval-X code translation benchmark using multiple backbone LLMs of different sizes. It compares against vanilla prompting, instruction-tuned LLMs, Chain-of-Thought prompting, single-stage semantic prompting, and specialized translation baselines such as TransCoder under the same evaluation setting. Reported metrics include translation success rate/execution success and CodeBLEU. Multisage improves translation success rates by up to 2.22x overall, with gains up to 1.42x on smaller models, 1.28x on mid-scale models, and 1.17x on larger models; it also achieves substantially higher CodeBLEU while keeping competitive execution success.

### 🌱 Insights & Implications
The main takeaway is that translation errors are often caused by missing or unreliable semantic constraints, not just by model size or stochastic generation. Explicitly constructing multiple semantic views from source code and verifying them before prompting can substantially reduce these intervenable errors. A key limitation is that the study is centered on HumanEval-X and translation tasks, so generalization to larger real-world codebases, more languages, and noisier static-analysis settings remains open. Another open question is how robust the generated tests and consistency checks are against subtle semantic mismatches. Still, the approach suggests a useful direction for software-engineering tools that need trustworthy code understanding, including automated repair and vulnerability analysis.

### 🎯 Why This Paper
This paper is highly relevant because it focuses on building and verifying semantic representations from code, which is directly aligned with software maintenance, automated program repair, and reliable LLM-based code understanding. Its combination of static analysis, multi-view semantic augmentation, and consistency calibration could transfer well to vulnerability detection and vulnerability repair, where false or incomplete semantic guidance can lead to unsafe outputs.

## 📄 [Dummy Backdoor as a Defense: Removing Unknown Backdoors via Shared Internal Mechanisms for Generative LLMs](https://arxiv.org/abs/2606.11648v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Kazuki Iwahana, Masaru Matsubayashi, Takuma Koyama, Toshiki Shibahara, Kenichiro Omintato, Akira Ito
📅 **Published:** 2026-06-10  🏷️ **Categories:** cs.CR, cs.CL

### 💡 TL;DR
The paper shows that different backdoors for the same jailbreak objective induce similar layer-wise activation shifts inside an LLM, and it uses that shared mechanism to defend against unknown backdoors. The proposed remedy is to intentionally add a defender-controlled dummy backdoor and then remove it with fine-tuning on dummy-triggered inputs paired with clean responses, which also weakens the unknown backdoor.

### 🏆 Key Contributions
- Introduces an indirect backdoor-removal strategy that uses a controllable dummy backdoor as a proxy for unknown backdoors.
- Provides an activation-level analysis showing that different backdoors with the same task produce similar Trigger-Activated Changes, especially in later layers.
- Defines a practical defense that can be used both during training and after model receipt via dummy-trigger fine-tuning on clean targets.
- Demonstrates the approach on multiple backdoor types and LLM families, with better attack-suppression and utility preservation than representative defenses.

### ⚙️ Methodology
The paper first studies whether different backdoors share internal mechanisms by computing Trigger-Activated Changes (TACs): for each layer, it measures the average difference between activations on a clean input and the same input with a trigger, then compares triggers using cosine similarity across layers. This allows the authors to compare attacks that have very different computational profiles on a shared axis. They study three attack families: JailBroken, a template-based attack that applies one of several jailbreak templates each step; PAIR, a black-box iterative refinement attack that uses an attacker LLM to rewrite the prompt after each response; and GCG, a white-box gradient-based attack that optimizes prompts using gradients from the target model. From these traces they define risk-compute curves, plotting empirical attack success rate against cumulative FLOPs as query budget increases, and then extract scalar summaries including C@tau, the compute needed to reach a chosen success threshold, and AE/CAURC-style summaries that capture the average risk gained per unit compute. The framework is applied uniformly across models, attacks, and benchmarks, with results aggregated over 10 random seeds and reported with t-distribution confidence intervals.

### 🧪 Experiments
The evaluation spans 10 language models across three model families and four training/alignment stages, using three attack strategies (gradient-based, iterative refinement, and template-based) on two jailbreak robustness benchmarks. The paper compares compute-aware curves and summaries against standard fixed-budget ASR reporting, using FLOP-based accounting for target-model generation, attacker-model calls, gradient computation, and judge evaluation. Reported findings include: alignment training has non-monotonic effects on compute-space robustness, with intermediate stages sometimes outperforming later aligned stages; scaling model size reduces the effectiveness of gradient-based attacks but has limited impact on cheaper template-based attacks; gradient-based attacks crafted on a surrogate model can transfer to a target model and reduce attacker cost; compute cost varies by up to about 5x across harm categories within a single model; and safety-aligned RL increases aggregate attacker cost while still leaving some categories disproportionately accessible. The paper emphasizes that these results are visible in risk-compute curves even when fixed-budget ASR would collapse the differences into a single number. The text provided does not name the two benchmarks or report many absolute numeric scores beyond the approximately 5x category spread and the qualitative comparisons above.

### 🌱 Insights & Implications
The main insight is that attack success rate alone can be misleading because it ignores how much compute the attacker must spend to achieve that success. This is especially important for security-oriented evaluation, where exploitability depends on the work factor, not just on whether a vulnerability exists. A second insight is that robustness is attack-dependent: defenses or alignment strategies that help against expensive gradient-based attacks may do little against cheaper template-based attacks, so a single aggregate metric can hide meaningful residual risk. The framework is still a proxy: FLOPs do not directly capture wall-clock time, hardware efficiency, or attacker-specific engineering overhead, and the conclusions depend on the chosen judge, attack implementations, and benchmarks. Open questions include how well FLOP-based rankings translate to real attacker ROI, how to calibrate thresholds like tau for different deployment contexts, and how to extend the method to other modalities or more adaptive attackers.

### 🎯 Why This Paper
This paper is highly relevant to software security and vulnerability assessment because it reframes adversarial robustness as a costed exploitability problem, analogous to work-factor analysis in classical security. Its compute-aware methodology is useful for evaluating how hard it is to trigger unsafe model behavior, which connects directly to vulnerability detection, exploit prioritization, and security-oriented benchmarking. The framework also suggests a more realistic way to compare defenses and attacks than fixed-budget success rates, which is valuable for researchers studying robust systems and repair strategies.
