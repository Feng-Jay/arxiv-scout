## 📄 [Root-Cause-Driven Automated Vulnerability Repair](https://arxiv.org/abs/2605.04251v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Hulin Wang, Zion Leonahenahe Basque, Jie Hu, Ati Priya Bajaj, Yibo Liu, Samuel Zhu, Giorgi Kobakhia, Nikhil Chapre, Will Rosenberg, Siddharth Mishra, Aditya Maheshbhai Gabani, Moritz Schloegel, Adam Doupé, Yan Shoshitaishvili, Ruoyu Wang, Tiffany Bao
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
Kumushi is an LLM-based automated vulnerability repair agent that explicitly prioritizes root-cause localization using diversified dynamic execution evidence and evidence-weighted ranking before patch generation. On 178 real C/C++ vulnerabilities, it achieves higher oracle-passing repair rates than prior academic repair systems and, under expert review, produces more genuine root-cause fixes and fewer superficial patches than a frontier commercial coding agent.

### 🏆 Key Contributions
- Introduces Kumushi, a root-cause-driven automated vulnerability repair agent for C/C++ that combines diversified dynamic fault localization with evidence-weighted ranking of functions of interest.
- Identifies a core failure mode of prior LLM repair systems: static-only or weak localization and noisy context push agents toward crash-site suppression rather than true root-cause repair.
- Proposes a two-tier patch quality evaluation framework that combines automated replay-based oracle validation with structured expert assessment to distinguish plausible patches from genuine root-cause fixes.
- Provides a large-scale empirical evaluation on 178 real-world vulnerabilities, showing substantial gains over prior specialized repair agents and qualitatively better patches than Codex under human assessment.
- Open-sources a prototype implementation of Kumushi to support further research on vulnerability repair and root-cause-aware agent design.

### ⚙️ Methodology
Kumushi is designed around the premise that vulnerability repair quality depends critically on identifying and constraining the LLM’s attention to likely root-cause code regions. Its workflow first gathers richer dynamic evidence than prior approaches by performing diversified dynamic fault localization: instead of analyzing only a single proof-of-concept execution or relying primarily on static reasoning from a crash trace, Kumushi executes a family of crash-triggering inputs to expose functions that repeatedly correlate with the defect across multiple failing runs. This diversification is intended to surface latent root-cause locations that may not appear prominently in one trace and to reduce dependence on imperfect static call resolution or expensive symbolic execution.

The dynamic evidence is then consolidated through an evidence-weighted ranking stage. Candidate functions are scored and ranked as Functions of Interest (FOIs) according to how strongly the collected execution evidence suggests relevance to the crash and underlying defect. This stage serves as a context-filtering mechanism: instead of passing broad, noisy code context to the LLM, Kumushi narrows the patching context to a focused, ranked set of suspicious functions. The goal is to reduce context drift, where the LLM edits nearby symptom-bearing code rather than the true cause.

After localization, Kumushi invokes the LLM-based patching agent on the ranked FOIs, generating candidate patches targeted at the highest-priority defect-relevant code. Candidate patches are validated in an iterative loop using replay-based oracles: the original proof of concept is re-executed, developer test suites are run, and additional validation can include replay on fuzzer-generated variant PoCs. Failed patches feed error signals back into further repair iterations.

Beyond system design, the paper’s methodological contribution includes a two-tier evaluation protocol for patch quality. Tier 1 is automated plausibility checking via replay-based oracles. Tier 2 is structured expert assessment that judges completeness, correctness, and whether the patch addresses the underlying root cause rather than merely suppressing the observed crash. This is motivated by the paper’s claim that developer patches and oracle-passing patches are both unreliable proxies for true root-cause repair.

### 🧪 Experiments
The evaluation uses 178 real-world C/C++ vulnerabilities from open-source software. Kumushi is compared against three baselines: PatchAgent, LoopRepair, and OpenAI Codex. The automated evaluation (tier 1) uses replay-based validation oracles including re-execution of the original PoC, execution of the developer test suite, and replay of fuzzer-generated variant PoCs to assess patch plausibility.

Under tier-1 evaluation, Kumushi achieves an 85% plausible patch rate, outperforming PatchAgent at 72% and dramatically exceeding LoopRepair at 2%. This supports the claim that Kumushi’s root-cause-oriented localization improves practical repair effectiveness compared with prior specialized academic systems.

However, the authors find that Kumushi and Codex are statistically indistinguishable when judged only by automated oracles across the three replay-based validations. This motivates the second evaluation tier.

For tier-2 expert assessment, the authors examine the 144 bugs for which both Kumushi and Codex produced plausible patches. In blind pairwise human evaluation, experts preferred Kumushi’s patch on 68 cases (47%), preferred Codex on 39 cases (27%), and judged 37 cases (26%) equivalent. On the 107 decisive comparisons, a sign test yields p = 0.0065, indicating a statistically significant advantage for Kumushi. The qualitative findings emphasize that Kumushi produces more root-cause fixes and fewer superficial patches than Codex, even when both pass automated tests. The paper also reports that Kumushi outperforms PatchAgent by 18 percentage points in plausible patches under tier 1.

### 🌱 Insights & Implications
A central insight is that passing replay-based test oracles is insufficient for evaluating vulnerability repair quality. Automated oracles can verify that a patch suppresses the observed failure and preserves tested behavior, but they cannot reliably determine whether the underlying defect remains reachable along other execution paths. This makes oracle-based evaluation complete enough for plausibility but not sound for root-cause correctness.

Another key implication is that localization quality and context control are tightly coupled in LLM-based repair. Even when a fixable defect is near the crash site, broad or noisy context can cause the model to patch symptoms rather than causes. Kumushi’s FOI-ranking approach suggests that careful evidence aggregation and context narrowing may be as important as patch-generation model capability.

The results also challenge the common assumption that developer patches are a gold standard for root-cause repair. The authors note prior evidence that developers themselves sometimes ship incomplete or symptom-oriented fixes, complicating supervised training and evaluation for vulnerability repair research.

Limitations and open questions remain. The expert-assessment tier improves fidelity but is expensive and hard to scale, so the field still lacks a cheap automatic metric for root-cause repair. The approach is evaluated on C/C++ vulnerabilities with available PoCs and execution-based validation, so generalization to other languages, vulnerability classes, or settings without reproducible crashes is not yet clear. The paper also suggests broader research opportunities in automatic root-cause identification, stronger semantic validation oracles, and hybrid analysis methods that combine dynamic evidence with scalable static reasoning.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it directly addresses one of the hardest practical problems in APR for security: generating patches that actually eliminate the vulnerability’s root cause rather than merely hiding the symptom. It advances the state of the art with a concrete repair architecture, a strong comparative evaluation on real C/C++ vulnerabilities, and—especially important for APR and security research—a richer evaluation methodology that exposes weaknesses invisible to standard oracle-based metrics. For work on vulnerability repair pipelines, patch validation, or root-cause-aware repair agents, this paper offers both a useful system design pattern and a more rigorous way to measure true repair quality.

## 📄 [Agentic Vulnerability Reasoning on Windows COM Binaries](https://arxiv.org/abs/2605.05000v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Hwiwon Lee, Jongseong Kim, Lingming Zhang
📅 **Published:** 2026-05-06  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
The paper introduces SLYP, an agentic binary-analysis pipeline for finding race-condition vulnerabilities in privileged Windows COM services and automatically generating debugger-verified exploit PoCs. By combining LLM reasoning with specialized tools for binary exploration, COM metadata inspection, and dynamic debugging, SLYP substantially outperforms coding agents and prior static analysis, and it found 28 real zero-days confirmed by Microsoft.

### 🏆 Key Contributions
- An end-to-end agentic pipeline, SLYP, for discovering race-condition vulnerabilities in Windows COM binaries and generating verified proof-of-concept code.
- Three reusable MCP-compatible tool interfaces: binary exploration, COM inspection, and dynamic debugging, which provide the agent with decompilation context, COM activation metadata, and runtime validation feedback.
- A benchmark of 20 COM objects and 40 manually verified vulnerability cases, including 8 independently discovered 1-day CVEs, plus released agent trajectories across multiple scaffolds and models.
- An empirical comparison showing SLYP consistently outperforms production coding agents and a strengthened reproduction of COMRace on vulnerability discovery and dramatically improves PoC generation success.
- A real-world deployment that discovered 28 previously unknown vulnerabilities across 9 Windows COM services, all confirmed by MSRC, resulting in 16 CVEs and $140,000 in bug bounties.
- A generally reusable interface design intended to transfer beyond COM to other commercial off-the-shelf binaries requiring agent-guided reverse engineering and debugging.

### ⚙️ Methodology
SLYP is an LLM-driven ReAct-style agentic system tailored for long-horizon vulnerability reasoning over stripped Windows binaries, especially COM services. The core problem is to identify exploitable race conditions in multi-threaded COM objects by reasoning about shared object state, virtual dispatch, object lifetime, synchronization, and thread interleavings, then synthesize client-side PoCs that trigger the bug.

The system is organized around three MCP-compatible tool families. First, the binary exploration interface exposes capabilities needed for reverse engineering binaries: decompilation, navigation across functions, recovery of virtual dispatch targets, and tracing of accesses to object fields such as this+offset. This lets the agent inspect decompiled methods, recognize unsynchronized accesses to shared fields, and connect call sites through vtables and interface implementations. Second, the COM inspection interface provides COM-specific metadata and activation context, such as CLSIDs, interfaces, registration data, service hosting information, threading model, and templates for activation and invocation. This is crucial because vulnerability feasibility depends on whether a low-privilege client can instantiate the class and whether the object is hosted in an MTA configuration that allows concurrent RPC execution. Third, the dynamic debugging interface supports compiling PoC code, launching and interacting with debugger sessions, executing PoCs, and capturing crashes or memory-corruption evidence. This turns the agent from a pure analyzer into a closed-loop exploit synthesizer that can iteratively repair failing PoCs.

On top of these tools, SLYP uses a standard think-act-observe loop where the model plans analysis steps, invokes tools, interprets outputs, and revises hypotheses. The paper emphasizes several engineering mechanisms for sustained analysis: pre-compaction checkpointing preserves intermediate findings when context is compressed during long sessions; task verification discourages premature stopping by checking whether discovery or PoC-generation objectives are actually satisfied. The pipeline is model-agnostic and can run with different GPT and Claude backbones.

The vulnerability reasoning workflow begins by locating reachable COM classes and methods, identifying shared object fields accessed by multiple methods or multiple executions of the same method, and checking whether these accesses are protected by synchronization. The agent looks for patterns such as read/free/write on the same heap-backed field, especially in MTA-hosted services where concurrent requests can interleave. It reasons about possible interleavings that yield use-after-free or double-free bugs, using decompiled code semantics rather than only syntactic matching. For PoC generation, the agent uses COM metadata to construct valid client activation code, creates concurrent method invocations targeting the vulnerable interface, compiles and runs the code, then uses debugger feedback to refine timing, invocation structure, or object usage until the crash is reproduced and verified.

### 🧪 Experiments
The evaluation covers both vulnerability discovery and PoC generation. The main benchmark contains 20 Windows COM objects comprising 40 vulnerability cases with manually verified ground truth; 8 of these are independently discovered 1-day CVEs used for circularity-free validation. The paper compares SLYP against production coding agents, specifically Codex and Claude Code, across six backbone models using identical task prompts. It also compares against COMRace++, the authors' enhanced reproduction of the prior state-of-the-art static analyzer COMRace.

For vulnerability discovery, the primary metric is F1 score on the benchmark. SLYP achieves 0.973 F1 with GPT-5.4. Production coding agents with default minimal binary-analysis tools reach lower performance, with the best reported default baseline at 0.910 F1 for Codex with GPT-5.4 and 0.724 F1 for Claude Code with Opus 4.6. Even when production agents are augmented with all binary exploration tools, SLYP still leads on every tested model by up to 0.208 F1. Compared with COMRace++, SLYP improves bug discovery by 3.3x, with 0.973 versus 0.299 F1.

For PoC generation, the study evaluates whether generated PoCs are actually verified through debugging rather than merely compiled or plausibly written. Production coding agents in their default setup, without the COM inspection and dynamic debugging tool interfaces, solve essentially no cases on either frontier model. In contrast, the strongest SLYP configuration autonomously synthesizes working PoCs for 27 of 40 cases, or 67.5%. The four best SLYP configurations together solve 34 of 40 cases, indicating robustness across models and scaffolds.

The paper also includes a real-world deployment on production Windows services. Using SLYP, the authors discover 28 previously unknown vulnerabilities across 9 COM services. All 28 findings were confirmed by the Microsoft Security Response Center, 16 were assigned CVEs, and the findings yielded $140,000 in bounty rewards. This deployment result demonstrates that the benchmark improvements translate into high-impact zero-day discovery in operational software.

### 🌱 Insights & Implications
A central insight is that agentic vulnerability research benefits greatly from task-specific tool grounding. Generic coding agents, even strong ones, are not enough for COM race analysis because they lack the environmental facts needed to reason about binary-only interfaces, activation preconditions, and runtime validation. The combination of binary reverse engineering, platform-specific metadata, and interactive debugging appears to be the key enabler rather than raw model capability alone.

The paper also suggests that semantic reasoning over decompiled code can outperform specialized static analyzers on difficult binary security tasks, especially when the task involves contextual conditions such as access permissions, threading models, and exploitable interleavings. Static analyzers like COMRace suffer from pair-enumeration noise, alias imprecision, and brittle assumptions about binary structure, whereas the agent can flexibly inspect control flow and hypothesize race schedules.

For automated exploit generation, the work highlights that debugger feedback is essential. PoC synthesis for concurrency bugs is not a one-shot code-generation task; it is an iterative search problem over invocation patterns, scheduling, and object lifecycle. Verification-oriented loops are therefore likely important for broader automated program repair or exploit-generation settings as well.

Limitations and open questions likely include the dependence on strong decompilation quality and the cost and latency of long-horizon agent runs. The current focus is on a specific but important vulnerability class: COM race conditions leading to memory corruption in MTA services. It remains open how well the same scaffolding transfers to other bug classes such as logic flaws, authorization bugs, or non-racy memory corruption. Another question is reproducibility at scale: dynamic debugging and PoC verification may require substantial environment engineering and may be sensitive to timing or OS-version differences.

For related research, the implication is that future vulnerability detection systems may be most effective as hybrid agent-tool pipelines rather than purely symbolic analyzers or purely generative models. The reusable MCP interfaces and model-agnostic design point toward a general framework for vulnerability discovery and validation in commercial binary software.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and vulnerability repair workflows because it addresses real-world bug finding in stripped binaries and closes the loop from detection to verified exploit reproduction. For a researcher interested in automated program repair and vulnerability repair, SLYP provides a compelling template for agentic systems that combine static reasoning, domain-specific platform metadata, and runtime feedback—an architecture that could be adapted from vulnerability discovery to root-cause localization, patch synthesis, regression testing, and security validation in complex binary or low-level software environments.

## 📄 [CodeEvolve: LLM-Driven Evolutionary Optimization with Runtime-Enriched Target Selection for Multi-Language Code Enhancement](https://arxiv.org/abs/2605.04677v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Ajay Krishna Borra, Wenzhuo Yang, Samarth Arora, Akhilesh Deepak Gotmare, Gokulakrishnan Gopalakrishnan, Tharun Gali, Madhav Rathi, Doyen Sahoo, Manpreet Singh, Mayuresh Verma, Laksh Venka, Shuchita Singh
📅 **Published:** 2026-05-06  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
CodeEvolve is an LLM-based evolutionary optimization framework that automatically identifies runtime hotspots from profiling data, proposes and repairs code edits, and validates them through a multi-stage pipeline before accepting improvements. It improves enterprise Java hotspot performance by an average of 15.22x across seven functions and shows that MCTS, filtering, and refinement substantially increase the reliability of valid optimized programs in Apex.

### 🏆 Key Contributions
- Introduces runtime-guided target selection using Java Flight Recorder profiles and weighted component graphs to automatically choose high-impact optimization targets instead of relying on manual hotspot identification.
- Extends LLM-driven evolutionary optimization with Monte Carlo Tree Search, population-based search, and refinement loops that repair invalid candidates using compiler/test feedback.
- Provides multi-language support with language-specific evaluation pipelines for Java and Salesforce Apex, enabling optimization across heterogeneous enterprise codebases.
- Implements a validation-first optimization workflow combining build validation, unit tests, performance checks, static analysis, and LLM-based review to preserve correctness while improving non-functional properties.
- Demonstrates strong empirical gains on real-world Java optimization tasks and ablation evidence on Apex showing that search, filtering, and refinement each improve optimization reliability.

### ⚙️ Methodology
CodeEvolve starts with runtime-enriched target selection. It builds a static Code Component Graph (CCG) whose nodes are functions/methods and edges represent calls or references. Java Flight Recorder (JFR) profiling data is then mapped onto this graph to create a Weighted Component Graph (WCG), where each node is annotated with cumulative execution time T(v) and call frequency C(v). Methods whose runtime or frequency exceed configurable thresholds are selected as optimization targets. For each target, only the target method is writable, while immediate graph neighbors are included as frozen read-only context; the rest of the codebase is pruned to keep prompts compact while preserving necessary dependencies.

The optimization pipeline is modular. A context builder packages the target code, relevant imports/signatures/constants, neighboring dependency context, and profiling annotations into prompts. The evolutionary optimizer extends AlphaEvolve/OpenEvolve-style search with context-aware prompting, population-based exploration, and an evolutionary history summary so the LLM can propose improved variants informed by prior successful edits and diagnostic feedback. CodeEvolve also supports Monte Carlo Tree Search (MCTS): each node represents a candidate program variant, and tree traversal uses an Upper Confidence Bound policy balancing exploitation of high-reward candidates and exploration of under-visited branches. Rewards are derived from the downstream evaluation pipeline and backpropagated through the search tree.

To improve robustness, generated variants that fail compilation or tests are not immediately discarded. Instead, a code refinement agent uses compiler errors and test failures as feedback to prompt the LLM to repair the candidate, then revalidates it. This refinement can be implemented as a lightweight reflection loop or as an MCTS-based repair loop focused on maximizing test pass rate. Code filters then gate whether a candidate can enter the search population. Across languages, evaluation is validation-first: candidates undergo build/compile checks, unit tests, performance measurement, static analysis, and LLM-based review for maintainability or code quality. Only candidates that preserve functional correctness are retained, and optimization objectives can combine performance, maintainability, and quality depending on language and deployment setting.

### 🧪 Experiments
The paper evaluates CodeEvolve in two main settings: an ablation study for Salesforce Apex optimization and a real-world hotspot optimization study on a large enterprise Java codebase (Salesforce Monolith). For Java, the system first uses JFR runtime profiling to identify hotspot methods, then optimizes seven selected hotspot functions. Baselines include single-pass LLM optimization, and the reported metric of primary interest is runtime speedup while preserving correctness through build validation and test-based gating. CodeEvolve achieves an average 15.22x speedup across the seven Java hotspot functions and outperforms single-pass LLM optimization on five of the seven targets.

For Apex, the evaluation is framed as an ablation of the search-and-validation components rather than just raw speed. The study compares configurations with and without MCTS, filtering, and refinement to measure optimization reliability. The key metric reported is the number of valid optimized programs produced out of 20 attempts on average. The full system with MCTS augmentation yields 19.5 valid programs out of 20 on average, indicating that the combination of structured search, candidate filtering, and repair significantly improves the rate of correctness-preserving outputs. Across both language settings, the evaluation pipeline includes compile/build checks, unit tests, performance checks, and quality-oriented analysis so accepted improvements are not merely faster but also behavior-preserving.

### 🌱 Insights & Implications
A central insight is that target selection matters as much as edit generation: by using runtime profiles to focus on methods responsible for most execution cost, CodeEvolve avoids wasting optimization effort on cold code and makes LLM budget more effective. Another important takeaway is that single-shot LLM optimization is unreliable for production code, whereas iterative search plus validation and repair greatly increases the chance of obtaining compilable, test-passing, high-performing code. The Apex ablation suggests that MCTS is especially useful not only for exploration but also for increasing the consistency of valid outputs, implying that search strategy is a major determinant of practical LLM optimization success.

For software engineering and maintenance research, the paper reinforces a validation-first paradigm: code transformation systems should tightly couple generation with automated quality gates, static analysis, and dynamic tests rather than treating code generation as the endpoint. For security- and repair-oriented work, the methodology suggests a template for correctness-preserving automated change systems that could be adapted from performance optimization to vulnerability repair or secure refactoring. Limitations and open questions include the dependence on profiling quality and workload representativeness, the possibility that test suites may miss behavioral regressions, limited language coverage beyond Java and Apex, and the lack of deeper discussion of long-term maintainability or semantic equivalence guarantees. Future work could extend runtime-enriched target selection to security hotspots, combine richer semantic analyses with the repair loop, and evaluate whether this framework generalizes to bug fixing or vulnerability remediation tasks.

### 🎯 Why This Paper
This paper is highly relevant because it presents a practical automated code transformation framework for large real-world codebases, directly intersecting software engineering, maintenance, and automated repair. Its validation-first search pipeline, refinement of invalid candidates using diagnostic feedback, and strong emphasis on preserving correctness while improving non-functional properties make it especially useful for researchers interested in adapting LLM-driven optimization techniques to automated program repair, vulnerability detection prioritization, and vulnerability-preserving secure code repair workflows.

## 📄 [You Snooze, You Lose: Automatic Safety Alignment Restoration through Neural Weight Translation](https://arxiv.org/abs/2605.04992v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Marco Arazzi, Vignesh Kumar Kembu, Antonino Nocera, Stjepan Picek, Saraga Sakthidharan
📅 **Published:** 2026-05-06  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper proposes NeWTral, a zero-shot post-hoc framework that restores safety alignment in third-party or proprietary LoRA adapters by directly translating unsafe adapter weights into a safer parameter-space manifold without needing the original training data. Across multiple LLM families and domains, its mixture-of-experts variant substantially lowers harmful response success while preserving most domain expertise, outperforming prior linear weight-editing baselines.

### 🏆 Key Contributions
- Introduces Neural Weight Translation (NeWTral), a non-linear parameter-space translator that converts unsafe domain LoRA adapters into safety-restored versions without retraining on target data.
- Designs a layer-aware Mixture-of-Experts translation architecture combining a knowledge-preserving Surgical Expert with a stronger safety-oriented Aggressive Expert, routed adaptively from weight statistics.
- Frames post-hoc safety restoration as a zero-resource deployment problem for open-source adapter ecosystems, where original domain and safety training data are unavailable.
- Provides empirical evidence that safety-relevant and utility-relevant information are not cleanly linearly separable in adapter space, challenging projection-based methods such as Safe LoRA and SaLoRA.
- Performs internal analysis suggesting MLP LoRA components play a larger role in safety behavior modulation, while attention components are more critical for preserving domain reasoning.
- Demonstrates strong cross-family and cross-scale generalization on Llama, Mistral, Qwen, and Gemma models up to 72B parameters over eight professional/scientific domains.

### ⚙️ Methodology
NeWTral targets the setting where a practitioner obtains an already fine-tuned LoRA adapter from a third party or from an internal pipeline but lacks access to the original domain data or alignment data needed to repair safety through additional fine-tuning. Instead of applying linear projections in weight space, the paper learns a non-linear mapping from unsafe adapter parameters to safe adapter parameters. Training requires a corpus of paired unsafe-to-safe adapters, where each pair represents the same domain specialization before and after safety restoration. The learned translator operates directly on LoRA weights rather than model activations or text outputs, making deployment zero-shot and data-free for new target adapters.

The framework treats the adapter as a structured set of layerwise low-rank matrices and learns to translate these parameters into a safer region of parameter space while preserving domain-specific functionality. The core model includes specialized translators with different behaviors: a Surgical Expert designed to make minimal, high-fidelity edits that preserve utility, and an Aggressive Expert designed to enforce stronger alignment changes where unsafe behavior is more entangled with domain knowledge. A routing module analyzes statistical signatures of the unsafe adapter weights on a per-layer basis and decides how much each expert should contribute, effectively forming a weight-space Mixture of Experts. This allows heterogeneous treatment across network depth and module type rather than uniformly editing all adapter parameters.

Conceptually, the method assumes that safety restoration is a translation problem over parameter manifolds, not a simple subtraction or orthogonal projection. The router and experts are pre-trained once and then distributed as a standalone artifact. At inference/deployment time, a new unsafe LoRA is passed through NeWTral, which outputs a translated safe adapter with no gradient updates, no access to the base training corpus, no need for a safe reference adapter for that domain, and no expensive re-optimization of the underlying large model. The paper also analyzes where intervention should occur, arguing that modifying MLP-associated LoRA weights is more effective for safety restoration, whereas preserving attention-associated weights is important for maintaining specialized reasoning capability.

### 🧪 Experiments
The evaluation spans four model families—Llama-3.1, Mistral, Qwen-2.5, and Gemma-2—at scales up to 72B parameters, and covers eight domains described as scientific and professional high-stakes areas, including medicine, law, finance, and physics. The task setting is post-hoc sanitization of unsafe domain adapters. The main comparison is against unsafe original adapters and prior post-hoc alignment methods based on linear parameter-space intervention, including Safe LoRA and SaLoRA-style baselines. The paper measures safety primarily using Attack Success Rate (ASR), where lower is better, and utility using knowledge fidelity/domain expertise retention, where higher is better.

The headline result is that the MoE version of NeWTral reduces average ASR from about 70% for unsafe expert adapters to about 13%, while preserving roughly 90% average knowledge fidelity. The paper emphasizes that monolithic translation strategies are less effective at balancing safety and utility, motivating the MoE design. It also reports cross-architecture robustness, suggesting the translator generalizes across different base model families and scales. Qualitatively and through representation analysis, the experiments support the claim that linear subspace assumptions are insufficient in complex domains because harmful and useful capabilities are entangled. The reported outcome is a substantially better utility-safety tradeoff than existing projection-based post-hoc techniques.

### 🌱 Insights & Implications
A central insight is that post-hoc safety restoration for specialized adapters resembles a software supply-chain hardening problem: organizations often consume third-party artifacts whose provenance and alignment quality are uncertain. This perspective is highly relevant to secure ML deployment and suggests safety repair mechanisms should be modular, reusable, and independent of original training assets. Another important finding is that safety and utility are not neatly separable by simple linear operations in LoRA space, especially in dual-use domains such as medicine or law where the same knowledge can support both legitimate and harmful outputs.

The layerwise heterogeneity finding is also significant: the paper argues that MLP blocks are more responsible for behavioral safety properties, while attention blocks encode domain reasoning that should be preserved. If robust, this has implications for future adapter editing, model patching, vulnerability mitigation, and even selective repair in code-focused LLMs. The work further implies that parameter-space transformation can serve as a practical alternative to costly safety re-finetuning, especially under privacy, licensing, or compute constraints.

Open questions remain. Generalization depends on the diversity and representativeness of the unsafe-to-safe adapter pairs used to pretrain NeWTral; adaptation to unseen domains or radically different training recipes may still be challenging. Because the method edits weights without direct access to behavior traces at deployment time, there may be failure modes on adversarial prompts outside the training distribution of the translator. The abstract and visible text do not fully detail whether fidelity metrics capture fine-grained task competence, factuality, or reasoning depth, so more domain-specific evaluation would strengthen claims. It is also unclear how the approach behaves for code-generation, security-analysis, or vulnerability-repair adapters, though the framework appears naturally extensible to those domains.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it addresses secure post-hoc repair of LLM adaptations under real-world constraints, closely paralleling software maintenance and software supply-chain security problems. For work on software engineering, automated program repair, vulnerability detection, and vulnerability repair, NeWTral offers a new paradigm: instead of retraining or prompt-side patching, repair can be performed directly in parameter space as a reusable artifact. That idea could inspire analogous methods for restoring safe behavior in code-specialized models, sanitizing vulnerable adapters, preserving developer-task utility while blocking exploit assistance, and building maintenance pipelines for third-party model components.

## 📄 [AgentTrust: Runtime Safety Evaluation and Interception for AI Agent Tool Use](https://arxiv.org/abs/2605.04785v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Chenglin Yang
📅 **Published:** 2026-05-06  🏷️ **Categories:** cs.AI, cs.CR

### 💡 TL;DR
AgentTrust is a runtime guardrail for AI agents that intercepts tool calls before execution and classifies them as allow, warn, block, or review using a mix of rule-based analysis, shell deobfuscation, multi-step attack-chain detection, SafeFix remediation, and an LLM judge for ambiguous cases. It reports strong accuracy on both curated and adversarial benchmarks while keeping latency low enough for practical deployment in agent tool-use pipelines.

### 🏆 Key Contributions
- Introduces a real-time semantic safety interception framework for AI agent tool use that operates before side effects occur and emits structured trust verdicts.
- Proposes ShellNormalizer, a shell deobfuscation module with nine normalization strategies to expose hidden dangerous commands such as obfuscated shell payloads.
- Adds SafeFix, a remediation-oriented engine that suggests safer alternatives rather than only blocking risky actions.
- Adds RiskChain, an order-aware session tracker that detects multi-step attack patterns like credential read -> encode -> exfiltrate, even when individual steps appear benign.
- Implements a cache-aware LLM-as-Judge with block-hash delta detection to reduce repeated long-context evaluation cost while preserving semantic judgment for ambiguous actions.
- Releases two benchmarks: a 300-scenario internal benchmark across six risk categories and a 630-scenario independently constructed adversarial/real-world benchmark.
- Defines fail-safe engineering contracts so that system or judge failures default to review rather than fail-open behavior, backed by regression tests and open-source implementation with MCP integration.

### ⚙️ Methodology
AgentTrust is positioned as a middleware layer between an autonomous agent and its external tools. For every proposed tool call, it produces a TrustReport containing a verdict among allow, warn, block, or review, plus explanations and optional safer alternatives. The architecture comprises eight components. First, ShellNormalizer preprocesses commands using nine deobfuscation and canonicalization strategies, including variable expansion, escape decoding, alias resolution, command substitution handling, ANSI-C quoting normalization, and adjacent-quote concatenation, so that downstream matching sees the semantic command rather than its obfuscated surface form. Second, ActionAnalyzer extracts security-relevant features using 42 regex patterns over four categories, identifying operations such as file deletion, permission changes, outbound network activity, credential access, shell execution, and other side-effect-bearing actions. Third, PolicyEngine applies 170 YAML-configurable rules that map action features and context to risk levels and verdicts. These rules distinguish production-safe policies from optional benchmark-compatibility rules that are not intended for deployment. - Introduces the first automated LLM-driven framework for subsystem-level UVM verification rather than only IP-level verification.

### ⚙️ Methodology
UVMarvel organizes automated verification into two major stages. In the first stage, it constructs a subsystem-level UVM testbench from specifications and RTL. Because subsystem documentation is distributed across textual specs, diagrams, waveforms, and RTL, the framework first normalizes these heterogeneous inputs into a verification-centric Intermediate Representation. In the second stage, UVMarvel improves coverage through iterative, coverage-guided stimulus refinement. Simulation of the generated testbench produces a coverage report, which is parsed by a Coverage Analyser into individual uncovered points. For each uncovered point, a Signal Tracker analyzes the RTL to identify the relevant inter-module signal dependency path responsible for activating that behavior. UVMarvel then applies a Verilog Patching Library to slice the RTL into a filtered DUT containing only the minimal semantic blocks necessary to preserve the dependency chain for that uncovered behavior. This filtered DUT simplifies the reasoning burden for the LLM by removing irrelevant code while retaining the causal path needed to exercise the target point. The LLM is then prompted with the uncovered point and the filtered DUT to generate new targeted stimuli, or, when appropriate, waiver candidates. Newly generated stimuli are inserted into the UVM environment and re-run on the original DUT. The process repeats until coverage improves or remaining points are deemed waivable. Overall, the key methodological idea is to make LLM reasoning tractable and reliable by combining structured representations, constrained protocol templates, and dependency-preserving RTL slicing rather than asking the model to reason over raw subsystem specs and full RTL directly.

The paper evaluates UVMarvel on subsystem-level RTL verification tasks involving integrated IPs and mainstream bus protocols, contrasting with prior work that mainly targets individual IP blocks. The framework is assessed on end-to-end UVM automation quality and verification efficiency. With UVMarvel’s IR, protocol library, signal tracking, and patching-based refinement, the system achieves an average code coverage of 95.65%. In terms of productivity, the full automated execution completes in about 4.5 hours, compared with several working days of manual verification effort by engineers. While the extracted text does not provide the full table of benchmark names, baseline systems, or per-design metrics, the reported quantitative outcome is that UVMarvel substantially improves over direct LLM use and achieves industrial-grade subsystem-level verification coverage with dramatically reduced turnaround time.

### 🌱 Insights & Implications
A central insight is that raw LLM prompting is insufficient for subsystem-level verification because complexity scales nonlinearly with integration: multiple IPs, bus protocols, timing relations, and control dependencies overwhelm generic model reasoning. By tracing uncovered points to narrow signal-dependency paths and simplifying RTL context via patching, the framework reduces noise and gives the LLM a more solvable task.

For related research, the paper suggests that future progress in LLM-based verification may depend less on end-to-end prompting and more on combining LLMs with formalized intermediate artifacts, domain libraries, and execution feedback. Open questions include how well the approach generalizes to more complex SoC-scale systems, custom or proprietary bus protocols, assertion generation, and formal-property-based verification. - Introduces AuditRepairBench, a paired-execution trace corpus with 576,000 registered cells and 96,000 executed paired traces for studying evaluator-channel-induced ranking instability in agent repair.

### ⚙️ Methodology
The paper defines the target phenomenon as evaluator-to-selector coupling inside agent-repair systems. For a paired execution x = (m, t, e, s) over system, task, evaluator configuration, and seed, the benchmark records the candidate repair set, an observed trace, and counterfactual traces under interventions that block particular evaluator-derived information channels while keeping the task, candidate set, tool outputs, and final evaluator fixed. The key measurement is whether the selected winning repair changes after blocking the selector-accessed evaluator signal.

Rather than relying on a single detector, the authors build a modular screening architecture that estimates whether each intervention family belongs to an operationally screened set for a trace cell. Four interchangeable implementations are used: (1) a learned influence proxy, which predicts likely evaluator-to-selector dependence from trace features; (2) a rule-based channel-exposure ratio that avoids trained models and measures explicit exposure of selector inputs to evaluator outputs; (3) a counterfactual sensitivity proxy that estimates how selection changes under channel blocking; and (4) a sparse human-audit proxy derived from limited manual inspection. These signals are combined into a screening posterior over intervention families.

This posterior drives a cell-level flip functional q-hat, defined as the posterior-weighted expectation that the observed winner differs from the counterfactual winner under screened interventions. Cells are then assigned set-valued labels using preregistered thresholds: positive if q-hat >= 0.9, negative if q-hat <= 0.1, and ambiguous otherwise. At the system level, the benchmark aggregates over task-evaluator slices using stratified scores, then reports set-valued leaderboards by considering all deterministic resolutions of ambiguous labels. This explicitly propagates uncertainty instead of collapsing to a single unstable ranking.

The methodology is grounded in a declared observability boundary: the goal is not to prove a latent causal mechanism, but to operationally detect coupling pathways at the selector input boundary, conditioned on sufficient instrumentation coverage. Validation uses source-level channel surgery, where the true path-blocking mechanism is manually implemented, to test whether the screening pipeline identifies the right coupling categories. The paper also introduces screening-guided blinding patches, which zero or remove selector-visible evaluator outputs at the interface boundary while preserving the rest of the pipeline and final scoring setup.

### 🧪 Experiments
The benchmark design spans 60 agent systems from five method families, 8 task families with 10 sampled tasks each for 80 tasks total, 6 evaluator families including proprietary LLM judges, open-weight LLM judges, and a rule-based evaluator with multiple aggregation schemes, 4 paired seeds per cell, and 5 registered intervention families, yielding 576,000 registered cells. Of these, 96,000 are executed as paired traces.

With task instances and agent checkpoints fixed, the top-three systems change for 7 of 60 systems, Kendall tau between rankings is 0.74, and median per-system rank displacement is 3.1 positions. The authors analyze these seven swaps and find that four are attributable to explicit evaluator-selector coupling channels: rationale-token reading, verdict-scalar concatenation, reranker-logit passthrough, and delayed-memory replay. Under selector-input blinding, these four systems reduce rank displacement by 3.2 to 4.5 positions, while three non-coupled systems change by less than 1.5 positions.

For mechanism-anchored validation, the authors use an 80-case source-level channel-surgery subset. Category-matched path-block detection AUROC ranges from 0.86 to 0.94 across coupling types. In a stronger independent-discovery protocol, two annotator groups separate from the pipeline developers and blinded to the screening design identify coupling patterns on 79 cases; the frozen ensemble achieves pooled AUROC 0.83.

Uncertainty propagation improves empirical 95% coverage from 0.81 to 0.95. Forward transfer to external community evaluators yields pooled Spearman rho = 0.65, suggesting moderate generalization across evaluation regimes.

The most practically important intervention is screening-guided channel blinding. Across systems, these patches reduce rank displacement by 55% to 74%, with a mean reduction of 62%, and require fewer than 50 lines of code. In contrast, random channel blinding reduces displacement by at most 7%, and generic retraining by at most 13%, indicating that instability is tied to specific evaluator-selector pathways rather than generic robustness issues.

For efficiency, the authors construct AuditRepairBench-Lite, a rule-only configuration over a 12,000-cell subset. Despite removing learned components, it preserves the leaderboard with Kendall tau = 0.88 and runs within 24 GPU-hours; the released artifact is 42 GB.

### 🌱 Insights & Implications
The paper’s central insight is that a nontrivial portion of leaderboard instability in agentic program repair is not just evaluator noise, benchmark variance, or judge unreliability, but an architectural interaction where evaluator outputs leak back into internal candidate selection. This distinction matters because it changes how one should interpret benchmark results and how to design fair evaluation pipelines.

A major methodological implication is that evaluator changes can reorder systems for two qualitatively different reasons: ordinary scoring variance versus selector-coupled feedback loops. AuditRepairBench aims to separate these causes by holding the final evaluator fixed and intervening only on the selector-visible channel. This is highly relevant for benchmark auditing, especially in settings where repair agents use rich intermediate signals such as judge rationales, confidence values, or reranker outputs.

For software engineering and automated repair research, the work suggests that evaluation pipelines should treat selector-accessed evaluator information as a potential confounder, similar to leakage or train-test contamination. Small, targeted interface patches can materially improve ranking stability without redesigning the whole system, which is encouraging for maintainers of repair agents and leaderboards.

The paper also adopts a careful stance on claims: it does not assert full causal identification, does not certify mechanism discovery in the strongest sense, and restricts scope to systems with sufficiently complete instrumentation coverage at the selector boundary. This transparency is a strength, but it also highlights limitations. The benchmark depends on observability and hook completeness, uses operational screening rather than definitive causal proofs, and studies a specific class of agent-repair architectures and evaluator configurations. Generalization to vulnerability repair, static-analysis-guided repair, or settings with hidden toolchains remains an open question.

Open research directions include: designing repair agents that are intentionally evaluator-agnostic during internal search; building benchmark protocols that forbid or standardize evaluator-derived internal signals; extending paired-trace auditing to vulnerability detection/repair and secure code generation; and integrating information-flow analysis with dynamic trace auditing to detect subtler leakage channels.

- Provides a theoretical analysis that once shuffled activations are aligned, linear-layer weights can be extracted up to row-wise and column-wise permutations.
- Demonstrates that these permuted recovered weights are computationally equivalent for forward propagation, so exact permutation recovery is unnecessary for effective model extraction.
- Empirically validates the attack on Pythia-70m and GPT-2, achieving highly accurate activation alignment and weight recovery at approximately $1 query cost.

### ⚙️ Methodology
The paper studies Transformer secure inference under linear-only encryption (LOE), where nonlinear layers such as GELU, softmax, and layer normalization are evaluated by the client in plaintext after the server reveals shuffled intermediate activations. The core attack has two stages. First, the adversary collects several activation vectors from the same hidden layer under different random shuffles. Because a shuffle preserves values but changes positions, the attack seeks to infer correspondence among elements across these vectors without ever learning the true permutation. The key idea is to query the model so that the unshuffled activations are extremely close numerically across repeated executions; then, after shuffling, the same underlying coordinates remain close in value and can be matched across samples. Since Transformer inputs are discrete and do not naturally permit infinitesimal perturbation like continuous-image settings, the authors exploit randomness introduced by the secure truncation protocol to induce the small perturbations needed for alignment. After recovering a consistent shared ordering among shuffled activations, the adversary reconstructs the input/output pairs of linear layers. In the second stage, model extraction is formulated as solving linear systems for the weights of Transformer linear components. The analysis shows the recovered matrices may differ from oracle weights by row and column permutations induced by the unknown hidden ordering, but these transformed weights still compose correctly across adjacent layers and preserve forward-pass behavior. Thus the attack does not need to discover the original canonical neuron order to steal a usable model.

### 🧪 Experiments
The evaluation targets two widely used GPT-family Transformer architectures in secure inference settings: Pythia-70m and GPT-2. The experiments assess both activation alignment quality and downstream weight extraction quality under the shuffling defense. For activation alignment, the proposed method aligns differently shuffled activations to a common permutation with mean squared error between 1e-9 and 1e-6, indicating near-perfect correspondence recovery. For model extraction, the recovered weights are compared against oracle weights using L1-norm difference, with errors ranging from 1e-4 to 1e-2. The total query cost is reported to be about $1, suggesting the attack is economically practical in MLaaS-like deployment scenarios. The paper emphasizes that these recovered weights, while permuted relative to the original model, retain computational equivalence in forward propagation and therefore are sufficient for model analysis or as strong surrogates for further attacks.

### 🌱 Insights & Implications
The main insight is that shuffling activations only hides coordinate identity, not the underlying numerical information needed to reconstruct that identity across multiple correlated queries. This breaks the core intuition behind prior LOE security claims that the factorial permutation space alone implies practical security. A particularly important implication for secure inference research is that defenses relying on local permutation of activations may be brittle whenever an adversary can induce repeated near-duplicate queries or exploit implementation-level randomness such as truncation noise. For software security and privacy-preserving ML systems, the paper highlights a classic gap between combinatorial hiding arguments and adaptive side-information attacks. A limitation is that the paper focuses on Transformer architectures and LOE designs that reveal intermediate activations for plaintext nonlinear evaluation, so its conclusions most directly apply to that threat model rather than fully encrypted inference. Open questions include whether stronger activation obfuscation, noise injection, protocol redesign, or formally leakage-bounded interfaces can preserve LOE efficiency without enabling linear-system-based extraction, and how these issues extend to other architectures or to defenses that combine shuffling with quantization or masking.

### 🎯 Why This Paper
This paper is highly relevant because it analyzes a concrete security failure mode in privacy-preserving Transformer inference, directly connecting to software security, vulnerability analysis, and secure deployment of ML systems. ### 💡 TL;DR
This paper performs a large controlled study of chunking choices in retrieval-augmented code completion and shows that chunking materially affects completion quality. Function-level chunking consistently performs worst, while sliding-window and cAST chunking offer the best cost-quality trade-offs; increasing cross-file context length matters more than tuning chunk size.

### 🏆 Key Contributions
- First controlled empirical study isolating chunking as an independent factor in retrieval-augmented code completion while varying retrievers, generators, and parameters under a unified pipeline.
- Systematic comparison of four chunking strategies—Function, Declaration, Sliding Window, and cAST—across 864 settings on two benchmarks, showing that structure-aware chunking does not automatically beat simple sliding windows.
- Evidence-based tuning guidance showing that cross-file context length is the dominant parameter for performance, whereas chunk size has weaker and non-monotonic effects.
- Cost-quality Pareto analysis demonstrating that Sliding Window and cAST dominate on both benchmarks, while Function chunking is never Pareto-optimal.
- Release of a replication package covering the full evaluation pipeline and configurations.

### ⚙️ Methodology
The study treats retrieval-augmented code completion as a three-stage pipeline: chunk source files into retrievable units, retrieve relevant cross-file chunks for a target completion query, and generate the completion with a code LLM. To isolate the effect of chunking, the authors cross four chunking strategies with multiple retrievers, generators, and parameter settings while keeping the rest of the pipeline controlled. The four chunkers are: (1) Function chunking, which extracts each top-level function or method including full bodies and recursively splits oversized functions along child-statement boundaries; (2) Declaration chunking, which extracts class and function declarations such as class headers, fields, and method signatures while omitting bodies and filtering private Python functions; (3) Sliding Window chunking, which creates fixed-size overlapping line-based windows without parsing; and (4) cAST, an AST-based method that recursively splits large nodes and greedily merges adjacent siblings to fit a size budget, operating over all AST node types. AST-based methods use Tree-sitter. The design is explicitly comparative: each chunking strategy is paired with the same retrievers and generators so that differences in completion quality can be attributed to chunking rather than other pipeline factors. The study also ablates chunk size and maximum cross-file context length to understand parameter sensitivity, and evaluates cost-quality trade-offs using Pareto-front analysis based on token cost versus completion accuracy.

### 🧪 Experiments
The evaluation spans 864 experimental settings formed by 4 chunking strategies × 4 retrievers × 5 generators × 9 parameter configurations, tested on two Python repository-level code completion benchmarks: RepoEval and CrossCodeEval. The paper reports Exact Match (EM) as the main quality metric, uses statistical significance testing and Cliff's delta to quantify effect size, and additionally analyzes cost-quality trade-offs through Pareto optimality. The main result is that chunking has a statistically significant effect on RAG-based completion quality. On RepoEval, Function chunking underperforms all other chunkers by 3.57 to 5.64 percentage points EM, with an extreme effect size (Cliff's delta = -1.0). Declaration, Sliding Window, and cAST are much closer to one another, differing by at most about 2.1 EM points. The paper further reports that this ranking is stable across all retriever-generator combinations, and that retriever choice contributes at most about 1.11 EM points of variation on RepoEval—smaller than the gap induced by chunking. In parameter ablations, increasing cross-file context length from 2,048 to 8,192 tokens yields up to 4.2 EM points improvement, while chunk size changes have a smaller, non-monotonic effect of at most about 1.9 EM points. In the cost-quality analysis, Sliding Window and cAST lie on the Pareto front on both benchmarks, whereas Function chunking never does despite lower token cost.

### 🌱 Insights & Implications
The strongest insight is that the common intuition of chunking code along semantic units like functions is not supported for repository-level RAG completion; full function bodies appear to be a poor retrieval unit compared with declarations, sliding windows, or cAST. This suggests that overly coarse chunks may dilute relevance, waste context budget, or retrieve unnecessary implementation details rather than the interface or local evidence actually needed for completion. A second implication is practical: if researchers or practitioners have limited tuning budget, they should prioritize larger cross-file context windows before spending effort on chunk-size optimization. The result that Sliding Window matches or exceeds structure-aware methods is important because it challenges assumptions that AST-aware chunking is inherently superior; simpler, language-agnostic methods may be competitive or preferable in engineering practice. At the same time, cAST remains attractive because it is also Pareto-optimal, suggesting there is still room for structure-aware chunkers when designed to control chunk size rather than strictly follow function boundaries. Limitations include the focus on Python benchmarks, the specific retrievers/generators studied, and repository-level code completion rather than other software engineering tasks such as bug repair, vulnerability fixing, or long-context code understanding. Open questions include whether these findings generalize to multilingual repositories, newer long-context models, test-time repair settings, and security-focused retrieval tasks where declaration-level or mixed-granularity evidence may behave differently.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, automated repair, and vulnerability-oriented research because many modern code intelligence systems rely on retrieval-augmented pipelines over repositories. Its main contribution is not a new model but rigorous evidence about a core design decision—how to segment code for retrieval—which directly affects systems for code completion, repair, bug localization, and vulnerability detection/patching that retrieve cross-file context. For a researcher building repository-aware APR or security tools, the paper offers actionable guidance: avoid default function-level chunking, consider sliding-window or cAST chunking, and allocate budget toward larger retrieved context windows. It advances related work by replacing folklore and conflicting practitioner heuristics with controlled empirical evidence that can inform the design of retrieval components in code generation and repair pipelines.

### ⚙️ Methodology
The task is repository-level reproduction test generation: given an issue description and the old version of a Java repository, generate a test that fails on the old code and passes on the fixed code, without access to the fix during generation. The proposed system is e-Otter++ for Java, adapted from prior Python work. Its workflow begins with fault localization to identify likely relevant source files and determine where a new test should be added. ## 📄 [A Multi-Agent Consensus Protocol for Stable Software Remodularization](https://arxiv.org/abs/2605.04188v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Ahmed F. Ibrahim
📅 **Published:** 2026-05-05  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper reformulates software remodularization as a negotiation problem between a cohesion-focused agent and a stability-focused agent, rather than a single-objective search problem. It proposes the Asymmetric Monotonic Concession Protocol (AMCP), proves key properties such as termination and local Pareto-satisfactoriness, and shows preliminarily that it can match optimizer quality when stability constraints are loose while strictly enforcing stability budgets when they are tight.

### 🏆 Key Contributions
- Introduces a distributed multi-agent view of software remodularization where different architectural quality attributes are represented by autonomous utility-driven agents.
- Proposes the Asymmetric Monotonic Concession Protocol (AMCP), in which a Stability Agent incrementally concedes to improve cohesion while respecting explicit utility thresholds.
- Defines a formal utility model using normalized TurboMQ for cohesion and MoJo distance to the previous decomposition for stability.
- Provides formal results claiming protocol termination, bounded monotonic concession behavior aligned with the Zeuthen bargaining strategy under closed-instance conditions, and local Pareto-satisfactoriness of the negotiated partition.
- Demonstrates preliminary empirical evidence on a synthetic benchmark and the Xwork Java framework showing that AMCP behaves like a stability-aware circuit breaker under strict stability budgets.

### ⚙️ Methodology
The system is modeled as a directed dependency graph G=(V,E), with the current version represented by a dependency structure matrix and the previous release represented by an accepted decomposition D_prev over the common modules. Two agents are defined. The Cohesion Agent evaluates a candidate partition D using U_coh(D)=TurboMQ(D)/k, where TurboMQ rewards intra-cluster dependencies and penalizes inter-cluster coupling, normalized by the number of clusters. The Stability Agent evaluates U_sta(D)=1-MoJo(D,D_prev)/n_common, so stability is higher when the current partition is closer to the previous decomposition. AMCP starts from D_prev and iteratively explores all single-module reassignments from the current partition. A move is considered valid only if it strictly increases cohesion utility and keeps stability above a user-specified threshold tau_sta; the search continues only while cohesion remains below a target tau_coh. For each valid move, the protocol computes a concession ratio R(delta)=(U_sta(D)-U_sta(D+delta)) / (U_coh(D+delta)-U_coh(D)), measuring stability sacrificed per unit cohesion gained. The Stability Agent is the sole proposer and selects the valid move with the smallest concession ratio, reflecting an asymmetric maintenance setting in which the legacy architecture is the default and any change must justify its stability cost. If no valid move exists, negotiation terminates in a deadlock state, which the paper interprets as intentional enforcement of the stability budget rather than failure. The authors argue that this move selection realizes Zeuthen-style rational concession behavior: the stability-oriented party yields the minimum necessary utility loss to satisfy the cohesion-oriented party. The per-step search evaluates at most n*(k_max-1) candidate moves, giving polynomial local search over a constrained frontier instead of exploring the full exponential partition space typical of population-based metaheuristics.

### 🧪 Experiments
The paper reports preliminary experiments on a synthetic benchmark and the Xwork Java framework, with broader evaluation on ten additional systems deferred to future work. The setup compares negotiated consensus behavior against state-of-the-art software clustering optimizers, though the full set of baselines and detailed quantitative tables are not included in this version. The key variables are the architect-specified thresholds tau_sta and tau_coh, which control the allowed stability budget and desired cohesion level. The paper states that on Xwork, the negotiated solution matches state-of-the-art optimizers when the stability budget is loose, but when the stability threshold is tightened, AMCP stops making destabilizing changes and thereby enforces the budget as a circuit breaker. Efficiency evidence is also provided: a Python implementation averages 1.24 seconds per negotiation step on the 113-module Xwork system and 1.02 seconds per step on a 120-module subset of Apache Ant. Metrics are primarily normalized TurboMQ-derived cohesion and MoJo-based stability, with comparisons framed around whether methods can preserve strict stability constraints while still improving modular structure.

### 🌱 Insights & Implications
The paper’s main insight is that remodularization can be treated as a constrained bargaining process rather than pure optimization, which is appealing when maintainers care about preserving architectural continuity across releases. This makes the method especially relevant for scenarios where developer cognitive load, release-to-release consistency, or migration cost matters as much as structural quality. The explicit threshold-based formulation is a practical advantage over weighted-sum objectives because it allows architects to encode non-negotiable stability budgets. Another notable implication is that deadlock is elevated from an optimization failure to a meaningful outcome indicating that no acceptable local change exists under current constraints. However, the current method is also limited by its local search nature: only single-module reassignments are considered, so the protocol may stop at local optima or fail to discover beneficial coordinated moves. The empirical evidence in this version is clearly preliminary, with limited datasets and incomplete quantitative comparison to multi-objective evolutionary algorithms. The formal claims are interesting, but their practical impact will depend on the forthcoming larger study, sensitivity analysis for threshold selection, and whether the approach scales and remains competitive on larger industrial systems. For related research, an open question is how this framework could be extended to more than two agents, richer architectural objectives such as security or change proneness, or integrated with automated architecture recovery and repair pipelines.

### 🎯 Why This Paper
This paper is relevant because it addresses software maintenance and architecture recovery with an explicit trade-off between structural quality and evolutionary stability, a core concern in long-lived software systems. For a researcher interested in software engineering, maintenance, security, automated repair, and vulnerability-related evolution, the paper offers a formal constrained-search paradigm that could be adapted to negotiate repairs or refactorings under preservation budgets, making it potentially useful for stability-aware program repair, secure remodularization, and change-minimizing vulnerability remediation.
