## 📄 [SEDCoT: Enhancing LLM-Based COBOL Code Translation via Symbolic Execution and Delta Debugging](https://arxiv.org/abs/2607.04092v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Phillip Entin, Wenchao Gu, Alexander Knapp, Chunyang Chen
📅 **Published:** 2026-07-05  🏷️ **Categories:** cs.SE

### 💡 TL;DR
SEDCoT is a COBOL-to-C translation framework that uses an LLM for initial translation, then improves semantic correctness through symbolic-execution-driven test generation and LLM-guided repair assisted by delta-debugged counterexamples. On 319 IBM CodeNet COBOL programs, it reports at least a 12% improvement over state-of-the-art baselines while retaining substantially better readability than rule-based transcompilers.

### 🏆 Key Contributions
- Introduces SEDCoT, a three-phase LLM-based COBOL-to-C translation pipeline combining initial generation, test generation, and automated repair.
- Uses symbolic execution over a rule-based COBOL translation to generate high-coverage test cases that expose semantic mismatches in LLM translations.
- Augments repair with delta debugging to minimize failing inputs into concise counterexamples that are easier for the LLM to use for localization and correction.
- Provides an empirical evaluation on a public COBOL dataset showing significant accuracy gains over prior LLM-based baselines and better readability than rule-based output.

### ⚙️ Methodology
SEDCoT operates in three sequential phases. First, an LLM generates an initial C translation from COBOL. If the result does not compile, compiler diagnostics are fed back to the LLM for iterative refinement, and the best candidate is selected based on compilation success or minimal errors. Second, the framework generates verification tests for the source COBOL program using two complementary mechanisms: rule-guided translation of COBOL through GnuCOBOL and symbolic execution to systematically explore paths and synthesize coverage-oriented inputs, plus additional LLM-generated tests to increase diversity. These tests are designed to stress COBOL-specific behaviors such as data formatting, whitespace variation, mixed data types, and non-printable characters. Third, the compiled translated C code is executed against the generated suite; failing tests and execution feedback are provided to the LLM for automated semantic repair. When repeated repair attempts fail or counterexamples are overly complex, delta debugging minimizes the failure-inducing inputs so the LLM sees a succinct trigger and can better localize the defect. Symbolic execution is used only in test generation, and repair proceeds afterward rather than in a closed symbolic loop.

### 🧪 Experiments
The evaluation uses 319 COBOL programs from IBM’s CodeNet dataset, targeting translation into C. For each program, the authors synthesize roughly 500 diverse test cases by varying values, mixing data types, altering whitespace, and injecting non-printable characters. They compare SEDCoT against two state-of-the-art baselines: prior LLM-based COBOL translation approaches and the rule-based GnuCOBOL pipeline as a readability/correctness reference. The paper reports that SEDCoT improves translation accuracy by at least 12% over the strongest baselines. GnuCOBOL remains a strong semantic baseline due to standards compliance, but its generated C is much less readable and maintainable. The paper also emphasizes that symbolic-execution-derived tests achieve higher line and branch coverage, making them more effective for surfacing semantic bugs and guiding repair, though the excerpt does not expose detailed metric tables or exact per-baseline scores.

### 🌱 Insights & Implications
The paper is notable because it treats code translation as a generate-and-repair problem rather than a one-shot prompting problem, which is especially important for low-resource legacy languages like COBOL. The central insight is that correctness can be materially improved by externalizing semantic feedback through execution-based tests, while readability is preserved by keeping the LLM in charge of code generation. For software engineering research, this suggests a practical pattern: combine symbolic analysis with LLM repair rather than expecting the model to infer legacy-language semantics from training data alone. The approach also raises useful questions: how well it scales to larger multi-file enterprise COBOL systems, whether generated tests fully capture I/O and data-layout semantics common in production COBOL, and how repair quality depends on the specific LLM used. Another limitation is that the pipeline skips downstream repair if no compilable candidate can be produced, which may cap robustness on harder inputs. Still, the method points toward a broader synthesis between automated program repair, test generation, and legacy modernization.

### 🎯 Why This Paper
This paper is directly relevant to software maintenance, automated program repair, and program analysis for legacy systems. It advances the researcher’s interests by showing how symbolic execution, execution-guided feedback, and delta debugging can be integrated with LLMs to repair semantically incorrect generated code, which is closely aligned with APR and trustworthy code transformation. It is also relevant to software security and vulnerability research because the same test-generation-and-repair pattern could be adapted to detect semantic faults, robustness issues, or vulnerability-preserving/regressing transformations during modernization pipelines. More broadly, it provides a concrete architecture for combining static/dynamic analysis with LLMs in a maintenance workflow, which is a strong direction for dependable code translation and repair research.

## 📄 [Is Three the Magic Number? An Empirical Evaluation of LLM-Based Repair Loops](https://arxiv.org/abs/2607.05197v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Tobias Kiecker, Eik Reichmann, Hosung Kang, Gabin An, Lars Grunske
📅 **Published:** 2026-07-06  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper empirically evaluates iteration limits in LLM-based repair loops across several software engineering tasks and finds a robust diminishing-returns pattern: most gains arrive within the first 3 to 4 repair attempts, with later iterations adding little. It also shows that workflow design and feedback mechanisms matter more than the specific low-cost LLM backbone for shaping repair effectiveness.

### 🏆 Key Contributions
- Provides a cross-task empirical study of repair-loop iteration budgets for LLM-based software engineering workflows, covering code generation, test generation, and code translation.
- Evaluates five representative workflows across six tool-dataset settings using three contemporary low-cost LLMs to characterize how completion rates evolve over up to ten repair iterations.
- Demonstrates a consistent concave improvement curve, where the first few repair iterations capture most achievable gains and later iterations yield marginal benefit.
- Shows that repair-loop behavior is influenced more strongly by orchestration, validation, and feedback design than by the choice of backbone model.
- Argues that repair budget should be treated and reported as an explicit experimental variable because it directly affects effectiveness, cost, runtime, reproducibility, and comparability.

### ⚙️ Methodology
The study defines a repair-loop-based SE system as an iterative generate-validate-repair process that continues until success or a fixed iteration cap. To examine how iteration limits affect performance, the authors evaluate existing repair-based workflows spanning multiple tasks: a Basic LLM Call baseline for Defects4J test generation, OCI on MBPP for code generation, INTERVENOR on HumanEval for code generation, CASCADE’s test-generation stage for its dataset, and LDB on TransCoder for code translation. They use three low-cost backbone models: Gemma-4, Qwen3.5, and GPT-4o-mini, with temperature fixed at 0.2. To make the tools comparable and compatible with the chosen models, they adapt interfaces and instrumentation: INTERVENOR is moved from deprecated completions to chat completions, OCI is restructured into a multi-turn conversation setup, and LDB is modified to measure correctness at every repair iteration instead of only at the end. CASCADE is reduced to the iterative test-generation component because later stages are not part of a full repair loop. For each tool-model pair, they measure the percentage of completed tasks after initial generation and after each repair step up to ten iterations, then compare the resulting per-step improvement curves across settings.

### 🧪 Experiments
The experimental setup spans six tool-dataset combinations and three LLM backbones. Tasks include test generation on Defects4J via a Basic LLM Call baseline, code generation on MBPP with OCI, code generation on HumanEval with INTERVENOR, test generation using CASCADE’s iterative generation stage, code translation on TransCoder with LDB, and an OCI variant without tests in the initial prompt. The main metric is completion rate at step 0 and after each repair iteration through step 10; the authors also compute mean relative improvement from one step to the next. Across all workflows and models, the completion curves show a similar concave shape: steep gains in the first few repairs followed by rapid flattening. Table 2 shows especially large average relative gains at step 1, with substantially smaller gains by steps 3 to 4 and near-zero returns typically after steps 5 to 7. Examples discussed include OCI, which has only minor gains after the first repair, and LDB, where models begin at very different initial completion rates but converge toward similar flattening behavior. The model choice changes absolute starting performance, but the repair-curve shape is much more stable within a workflow than across workflows.

### 🌱 Insights & Implications
The central insight is that repair-loop budgets are not a harmless implementation detail; they materially shape reported performance, cost, and reproducibility. The consistency of diminishing returns across tasks suggests that many LLM-based SE systems may be overspending computation beyond three or four repair attempts without meaningful benefit. At the same time, the paper indicates that the slope and ceiling of improvement depend more on workflow orchestration, validation strategy, and feedback content than on the underlying model, which implies that future work on automated repair should invest in better feedback design and control logic rather than only swapping models. A limitation is that the study focuses on a specific set of low-cost models, selected tools, and up to ten iterations, so results may shift for frontier models, other domains, or richer agentic systems. Another caveat is that some tooling changes, while necessary for compatibility and measurement, may slightly affect original tool semantics. Open questions include whether adaptive stopping criteria can outperform fixed budgets, how different feedback modalities alter repair efficiency, and whether similar iteration dynamics hold for vulnerability repair, APR, and more complex multi-step agent workflows.

### 🎯 Why This Paper
This paper is directly relevant to research in software engineering, automated program repair, and vulnerability-oriented code repair because iterative repair loops are now a common control pattern in LLM-based SE systems. It advances the researcher’s work by providing evidence that repair budget is a first-class experimental factor, highlighting where evaluation results can be biased by arbitrary iteration caps and pointing toward orchestration and feedback design as more promising levers than model substitution for improving repair effectiveness and efficiency.

## 📄 [RustMizan: A Compilable, Contamination-Aware Benchmarking Framework for Rust Vulnerabilities](https://arxiv.org/abs/2607.04729v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Tarek Elsayed, Shiping Yang, Eunsong Koh, Sanika Goyal, Vincent Huang, Paul Ngo, Nathan Young, Mohammad Omidvar Tehrani, Alvyn Kang, Arnell Kang, Zeyu Chen, Angélica Moreira, Xuan Feng, Angel X. Chang, Nick Sumner, Steven Y. Ko
📅 **Published:** 2026-07-06  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
RustMizan is a new benchmark framework for Rust vulnerability analysis that uses compilable, multi-granularity program variants and mutation-based contamination checks to evaluate LLM agents more realistically. It shows that current frontier models are only moderately accurate at vulnerability detection and remain weak at precise localization, especially under adversarial perturbations.

### 🏆 Key Contributions
- Introduces RustMizan, a Rust-specific vulnerability benchmarking framework with compilable variants at crate, file, and function granularity.
- Supports four analysis tasks in one benchmark: binary vulnerability detection, CWE classification, function-level localization, and line-level localization.
- Provides manually curated, peer-reviewed annotations for 42 real-world Rust memory-safety CVEs, yielding 173 compilable variants.
- Adds a paired mutation framework for semantics-preserving contamination testing and adversarial robustness evaluation, including misleading comments and identifier renaming.
- Evaluates four frontier LLM agents in an interactive command-line setup and releases trajectories, evaluation scripts, and a public leaderboard.

### ⚙️ Methodology
The authors build RustMizan around three design principles: multi-task evaluation, multi-level compilable code contexts, and explicit contamination/robustness support. Starting from RustSec memory-safety CVEs, they manually curate vulnerable and patched program versions and reduce them into standalone compilable variants at the crate, file, and function levels while preserving the vulnerability. Each sample is annotated through a dual-review process with consensus for binary labels, CWE tags, vulnerable functions, and vulnerable lines. On top of the base dataset, they implement a mutation framework that generates semantics-preserving variants for two purposes: contamination assessment, where changes preserve behavior but alter surface form enough to test memorization dependence, and adversarial robustness probing, where cues such as misleading comments or renamed identifiers attempt to misdirect the model. Evaluation is performed with LLM agents using a ReAct-style scaffold and command-line access, allowing models to inspect code, compile, and interact with the environment in a way closer to real vulnerability analysis workflows than snippet-only benchmarks.

### 🧪 Experiments
The benchmark contains 42 real-world Rust memory-safety CVEs and 173 compilable variants across context levels. The authors evaluate four frontier models: Claude Sonnet 4.6, GPT 5.4, Gemini 3.1 Pro, and Qwen 3.6 Plus. Tasks include binary vulnerability classification, CWE classification, function localization, and line localization. Key reported results show binary detection accuracy in the 56-65% range, while line-level localization F1 remains near 20%, indicating that identifying exact vulnerable code is much harder than deciding whether a vulnerability exists. They also study context granularity and find that narrower function-level context substantially improves localization performance. For contamination, aggregate results on semantics-preserving mutants show no broad evidence that frontier closed models are simply recalling the vanilla benchmark, although trajectory analysis exposes at least one instance of explicit CVE recall; by contrast, open-source models fine-tuned on the vanilla dataset perform much better on the original data but degrade significantly on mutated versions. For adversarial robustness, misleading comments and identifier renaming reduce line-level localization F1 by roughly 27% relatively, showing notable brittleness.

### 🌱 Insights & Implications
The paper makes a strong case that vulnerability benchmarking should move beyond non-compilable snippets and binary labels toward realistic, tool-using analysis settings with richer supervision. A central insight is that current LLM agents can often guess that code is vulnerable but struggle to ground that judgment precisely enough for remediation-oriented workflows, which is directly relevant for automated repair and triage systems. The contamination-aware mutation design is also important methodologically: it gives the community a practical way to test whether gains reflect reasoning or benchmark memorization, a major issue for public security datasets. Limitations include the relatively small dataset size, focus on Rust memory-safety CVEs, and likely substantial manual effort required to scale the framework to more vulnerabilities, languages, and vulnerability classes. An open question is whether improved tooling integration, retrieval, symbolic/static analysis, or repair-aware prompting can close the large gap between coarse detection and fine-grained localization.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and vulnerability repair because it provides a more realistic evaluation substrate for systems that must analyze compilable codebases, localize faults precisely, and resist benchmark contamination. It is also valuable for software engineering and automated program repair work since its multi-level variants, fine-grained localization labels, and mutation framework can support research on diagnosis-to-repair pipelines, robustness of repair agents, and tool-augmented secure code reasoning in Rust and potentially other languages.

Metrics are not named in the abstract, but they likely assess structural or exact-match similarity between generated and reference CFGs. The key reported findings are qualitative but clear: fine-tuned lightweight LLMs achieve promising CFG generation performance, perform particularly well on incomplete or erroneous code, and retain cross-language generalization to languages absent from the fine-tuning set.

### 🌱 Insights & Implications
This paper is interesting because it treats CFG construction as a robust neural translation problem rather than a purely compiler-driven analysis step. For related research, this suggests a path toward language-agnostic static-analysis preprocessing using compact models, but reproducibility will depend on exact graph encoding, evaluation criteria, and robustness under harder real-world code distributions.

### 🎯 Why This Paper
This paper is directly relevant to software engineering and maintenance because CFGs are foundational for code understanding, analysis, testing, and transformation. It is also relevant to software security, vulnerability detection, and automated program repair because many of those methods depend on control/data-flow representations, yet often break on incomplete or uncompilable code such as patches, generated fixes, or vulnerable snippets in the wild. The work advances those areas by suggesting a lightweight, multilingual, error-tolerant alternative for extracting control-flow structure, potentially enabling more robust preprocessing for downstream learning-based repair and security analysis systems.

## 📄 [Teaching Code LLMs to Reason with Intermediate Formal Specifications](https://arxiv.org/abs/2607.04232v1)  `80% relevant`

👤 **Authors:** Minh Le-Anh, Cuong Chi Le, Tien N. Nguyen
📅 **Published:** 2026-07-05  🏷️ **Categories:** cs.SE, cs.CL

### 💡 TL;DR
This paper proposes SpecCoder, a verification-guided training framework that teaches code LLMs to generate executable intermediate checkpoint assertions rather than only whole-program specifications. On the new HumanExec benchmark, these checkpoint specifications substantially improve validity, correctness, completeness, and downstream program checking and repair performance.

### 🏆 Key Contributions
- Defines executable checkpoint specification generation as a new task focused on internal program states at meaningful program points.
- Introduces SpecCoder, a verification-guided CodeLLM training framework that learns from validated reference programs, behavior-changing mutants, and multi-turn specification refinement traces.
- Develops a selection/training signal that prefers specifications satisfied by correct executions while rejecting faulty executions, making assertions useful as executable evidence rather than passive documentation.
- Presents HumanExec, a benchmark derived from Codeforces problems with test suites, reference solutions, and human buggy submissions, supporting specification generation, correctness checking, and program repair.
- Shows that stronger intermediate executable specifications improve downstream reasoning tasks beyond specification generation itself, including correctness assessment and repair.

### ⚙️ Methodology
The method centers on generating executable assertions at intermediate checkpoints inside a program, capturing semantic invariants over evolving internal states during execution. SpecCoder appears to train a code LLM using verification-guided supervision built from three ingredients: correct reference implementations, behavior-changing mutants that act as negative examples, and multi-turn refinement traces that simulate iterative improvement of weak or invalid specifications. The core idea is to optimize for assertions that are executable, nontrivial, and discriminative: they must hold on correct runs of the reference program but fail on faulty executions induced by mutants or buggy submissions. This likely requires executing candidate assertions against program traces or instrumented runs, filtering or rewarding specifications based on syntactic validity, runtime executability, and fault-rejection capability. Relative to prior work on pre/postconditions, the architecture/training objective is aimed at internal reasoning steps, closer to how developers debug and verify algorithms. The framework therefore combines generation with validation, using dynamic verification signals to refine the model toward semantically meaningful checkpoint assertions.

### 🧪 Experiments
Evaluation is conducted on HumanExec, a new benchmark built from recent competitive-programming tasks from Codeforces. The benchmark includes problem statements, test suites, reference solutions, and human buggy submissions, and supports three tasks: checkpoint specification generation, program correctness checking, and program repair. Baselines include base CodeLLMs, specifically Qwen2.5-Coder variants, prompted or used without the proposed verification-guided training. Reported metrics for specification quality include inline-specification correctness, completeness, and executable assertion validity. SpecCoder improves inline-specification correctness by up to 55.8%, completeness by up to 358.1%, and executable assertion validity by up to 26.6% over the base models. The abstract also states that these gains transfer to downstream tasks, implying improved performance in correctness reasoning and repair, though exact repair/checking metrics are not given in the abstract. The experimental design strongly suggests comparison across model scales and ablations contrasting weak, trivial, syntactically invalid, and discriminative assertions.

### 🌱 Insights & Implications
The paper’s main insight is that intermediate formal specifications are a better supervision and reasoning target for code LLMs than only global pre/postconditions, because they encode the stepwise semantic commitments programmers actually rely on. Another important insight is that raw prompting is insufficient: even when LLMs can emit assertions, many are invalid, vacuous, or too weak to detect behavioral faults, so verification-guided training is necessary. A likely limitation is benchmark scope: Codeforces-style algorithms are a strong testbed for reasoning, but they may not fully represent large-scale software systems, API-heavy code, or security-critical real-world programs. The approach also depends on executable validation assets such as tests, reference solutions, or mutants, which may limit applicability when such artifacts are sparse. Open questions include how well checkpoint specifications generalize across languages and domains, whether static verification could complement dynamic execution signals, and how these intermediate assertions can be integrated into APR or vulnerability-repair pipelines for real repositories. For related research, the paper suggests a path toward combining program analysis, test-based validation, and LLM training to produce more trustworthy code reasoning artifacts.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, automated program repair, and reliability because it reframes formal specification generation as an executable, fine-grained reasoning problem directly tied to verification and debugging. It is especially valuable for work on program repair and vulnerability detection/repair since intermediate assertions can provide stronger fault localization and semantic constraints for rejecting incorrect patches, validating candidate fixes, and improving trustworthiness of LLM-based repair systems.

## 📄 [Kaizen: Metamorphic Fuzzing and Differential Testing for LLM-Translated HPC Applications](https://arxiv.org/abs/2607.04058v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Oscar Ludwig, Ninad Anklesaria, Zheming Jin, Swaroop Pophale, Kausar Moshood, Christian J. DeVore, Brandon Gill, Cassius Villareal, Keita Teranishi, Manish Motwani
📅 **Published:** 2026-07-04  🏷️ **Categories:** cs.SE, cs.PL

### 💡 TL;DR
Kaizen is a correctness-oriented evaluation framework for LLM-based HPC code translation that combines metamorphic source mutations, grammar-based input fuzzing, and differential testing to detect semantic mismatches between original CUDA programs and their OpenMP translations. Across 16 scientific applications and three fine-tuned translation models, it shows that compilation success and static benchmark tests are weak proxies for actual correctness, especially for full-program translation.

### 🏆 Key Contributions
- Introduces Kaizen, a framework for evaluating behavioral correctness of LLM-translated HPC programs using metamorphic fuzzing and differential testing.
- Provides an empirical study of CUDA-to-OpenMP translation on 16 scientific applications from seven domains using three fine-tuned ChatPORT models at both kernel and full-program levels.
- Develops a two-level taxonomy of translation-induced compile-time failures: 9 categories for kernel-level translation and 27 categories for full-program translation across broader error groups.
- Identifies six categories of semantic errors that survive compilation and fixed tests, including loop bound faults, missing statements, shared-memory scope mistranslations, and execution-model assumption transfer.
- Demonstrates that full-program translation is substantially harder than kernel-level translation and that kernel-focused fine-tuning can over-specialize models and hurt end-to-end translation quality.
- Releases a replication package with code and data to reproduce the study’s results.

### ⚙️ Methodology
Kaizen evaluates LLM-based source-to-source translation by generating both program and input diversity around a reference HPC application. First, it applies semantics-preserving metamorphic source-code mutations to the original CUDA program to create varied but behaviorally equivalent translation inputs, reducing over-reliance on a single benchmark form and probing robustness to superficial code changes. Second, it uses grammar-based input fuzzing to systematically generate runtime inputs that exercise a broader behavioral space than developer-written test cases. Third, it performs differential testing by executing the original CUDA program and the translated OpenMP version on corresponding inputs and comparing outputs to detect semantic divergence. The framework is instantiated for CUDA-to-OpenMP translation and applied at two granularities: kernel-level translation, where isolated computational kernels are translated, and full-program translation, where complete applications including offloading, memory management, and host-device interaction are translated. The study uses three top-performing fine-tuned ChatPORT models and analyzes both compile-time and runtime failures, building explicit taxonomies for error characterization.

### 🧪 Experiments
The evaluation covers 16 scientific applications from seven domains and compares three fine-tuned ChatPORT models on CUDA-to-OpenMP translation. Experiments are run at kernel-level and full-program granularity. The assessment uses three main signals: compilability, success on developer-written tests from static benchmarks, and Kaizen’s differential-correctness checks under fuzzed inputs. Results show that compilability is poorly correlated with true semantic correctness: some applications such as winograd and lif reach very high compilation rates (up to 0.94 and 1.00) while achieving 0.00 correctness, whereas others like background-subtract and cross achieve both high compilability and correctness. Kernel-level translation reaches up to 72% correctness, but full-program translation is markedly worse; one model, ChatPORT_CL_13B, fails to compile any full-program translations. Compile-time analysis finds 9 error categories for kernel-level translation and 27 categories for full-program translation spanning issues in offloading, memory management, parallelism, code validity, and compatibility. Differential testing further exposes six classes of input-dependent semantic bugs that were not caught by compilation or standard benchmark tests.

### 🌱 Insights & Implications
The main insight is that current evaluation practice for LLM-based code translation is too shallow for scientific software: compilation success, token similarity, and benchmark tests can all mask semantically wrong translations. Kaizen shows that many translation bugs are conditional on specific inputs and only become visible when both program variants and runtime inputs are systematically diversified. The paper also highlights a structural difficulty gap between kernel-level and full-program translation, suggesting that apparent progress on isolated kernels does not transfer cleanly to realistic application porting workflows. A notable limitation is scope: the prototype is evaluated only on CUDA-to-OpenMP and on three ChatPORT-family models, so generalization to other programming models, toolchains, and broader LLM families remains open. There are also likely practical concerns around oracle tolerances, floating-point nondeterminism, and the cost of large-scale differential testing in HPC settings, which create opportunities for future work on scalable correctness oracles and performance-aware validation.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, security-adjacent reliability, and automated repair research because it addresses a central problem in AI-assisted code transformation: how to validate that generated code is behaviorally correct rather than merely compilable. Its use of metamorphic testing, fuzzing, and differential analysis connects directly to vulnerability detection and repair workflows, where shallow pass/fail signals also miss input-dependent faults. For research on automated program repair and trustworthy code generation, Kaizen provides both a concrete evaluation methodology and an error taxonomy that could inform test generation, repair validation, and robustness assessment for LLM-produced code changes.

## 📄 [Intent-Based Mutation Testing: From Naturally Written Programming Intents to Mutants](https://arxiv.org/abs/2607.05149v1)  `70% relevant`

👤 **Authors:** Asma Hamidi, Ahmed Khanfir, Mike Papadakis
📅 **Published:** 2026-07-06  🏷️ **Categories:** cs.SE

### 💡 TL;DR
The paper proposes intent-based mutation testing, which mutates the underlying programming intent rather than source-level syntax, using LLMs to generate behaviorally altered mutants that better reflect specification misunderstandings and corner cases. On 29 programs, the approach produces semantically diverse and syntactically complex mutants, with 55% not subsumed by traditional mutation operators, indicating strong complementarity to syntax-based mutation testing.

### 🏆 Key Contributions
- Introduces intent-based mutation testing, a new mutation testing paradigm centered on altering implemented programming intents instead of syntactic program constructs.
- Uses Large Language Models to transform naturally written or inferred programming intents into executable mutants that encode slightly different behavioral goals.
- Shows that intent-based mutants capture fault classes related to specification misunderstandings, corner cases, and behavioral variants that traditional mutation testing may miss.
- Provides an empirical comparison on 29 programs demonstrating that intent-based mutants are syntactically more complex and semantically more diverse than traditional mutants.
- Reports that 55% of intent-based mutations are not subsumed by traditional mutations, supporting the claim that the method offers complementary testing value.

### ⚙️ Methodology
The approach starts from programming intents, likely expressed in natural language or reconstructed from code behavior, and applies LLM-driven transformations to produce modified intents representing plausible alternative behaviors, misunderstandings, or corner-case interpretations of the original specification. These mutated intents are then translated into program mutants that preserve implementation plausibility while changing semantics at the behavioral level. This differs fundamentally from conventional mutation testing, where pre-defined syntax-level operators alter code tokens or expressions directly. The core design idea is that the same intent can be implemented in multiple ways, so mutating intent enables generation of semantically meaningful, structurally diverse mutants that approximate specification-level deviations rather than implementation-level perturbations. The LLM serves both as an intent mutator and as a bridge from intent variants back to concrete code mutants, enabling richer transformations than fixed operator sets.

### 🧪 Experiments

Control tasks attach the same tool interface but are directly answerable without any tool access, allowing measurement of overuse. Each model trace includes the query, executed tool calls, mock tool returns, and final answer. Responses are assigned exactly one label per task type. For required tasks, labels are Correct, Tool-Skip, Result-Ignore, or Output-Fabrication. For control tasks, labels are Correct, Unnecessary-Tool-Use, or Wrong-Answer. Labeling is performed by a deterministic rule classifier plus two independent LLM judges using a shared rubric, with majority vote determining the final label and ties resolved by the rule classifier. The benchmark uses one shared system prompt per domain, no model-specific tuning, and a standardized output format to make cross-model comparisons and failure-mode detection consistent.

### 🧪 Experiments
The evaluation covers 19 instruction-following headline models from both closed and open families using valid execution traces. Metrics are reported separately for the 750 required tasks, the subset of required tasks where a tool was actually called, and the 250 control tasks. The main metrics are Tool-Skip Rate (TSR), Clean Tool-Use Rate (CTUR), Result-Ignore Rate (RIR), Output-Fabrication Rate (OFR), Unnecessary-Tool-Use Rate (UTR), and Control Accuracy (CTRL-Acc). The best model achieves 86.33% CTUR, showing that the benchmark is not saturated. Results indicate that many models remain disciplined on control tasks, while Llama-3.1 variants frequently over-call tools, forming an 'Always-Call' pattern. A prominent quantitative result is that Llama-3.1-70B and Qwen2.5-72B, despite similar parameter scale, differ by 89 percentage points on control-task accuracy. The broader result is that aggregate task success obscures materially different failure profiles across models, especially between failures to call tools and failures to use tool outputs correctly after calling them.

### 🌱 Insights & Implications
The paper’s central insight is that tool-use evaluation needs to distinguish decision-stage failures from post-call integration failures. This matters for agent reliability because a model that never invokes a tool, one that invokes it but ignores the returned value, and one that fabricates extra information may all receive similarly poor end-task scores despite requiring different mitigation strategies. The benchmark also suggests that tool discipline is driven more by model family and training behavior than by parameter count alone. A useful limitation is that ToolFailBench is intentionally narrow: it isolates post-call faithfulness in single-turn settings rather than full multi-step planning or realistic interactive workflows, so it complements but does not replace broader benchmarks like SWE-bench or Toolathlon. Another open question is how robust the labels are to paraphrastic or partially faithful answers, although the hybrid rule-plus-LLM judging design partially addresses this. For future work, the benchmark could be extended to software engineering agents, vulnerability triage flows, and repair pipelines where tool outputs from analyzers, tests, or static checkers must be trusted and integrated correctly.

### 🎯 Why This Paper
This paper is relevant because dependable tool use is a core requirement for software engineering, automated repair, and vulnerability analysis agents that rely on compilers, test runners, static analyzers, retrieval systems, and security scanners. ToolFailBench advances this area by providing a concrete failure taxonomy and evaluation methodology for diagnosing whether an agent actually uses external evidence correctly, which is directly applicable to assessing and improving agents for program repair, vulnerability detection, and secure software maintenance workflows.

## 📄 [Auto: The AGI Compiler](https://arxiv.org/abs/2607.04542v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Jaber Jaber, Osama Jaber
📅 **Published:** 2026-07-05  🏷️ **Categories:** cs.LG, cs.AI, cs.SE

### 💡 TL;DR
Auto proposes a compilation pipeline that turns repeated LLM-agent behaviors into verified, capability-confined WebAssembly artifacts, then runs them behind calibrated guards with fallback to the original agent when novelty appears. The paper’s main result is that a large fraction of observed agent behavior is empirically deterministic enough to compile, cutting marginal cost sharply while preserving high parity on witnessed inputs when calibration and deopt fidelity are correct.

### 🏆 Key Contributions
- A recording-to-binary compiler pipeline for LLM agent behavior that lowers traces into a typed, effect-carrying task-graph IR and applies synthesis or distillation before artifact emission.
- Physical capability confinement via WebAssembly artifacts whose manifests declare measured guarantees and whose imports are restricted to audited host capabilities.
- A tiered runtime with conformally calibrated guards, deoptimization to the reference agent, trace capture, and autonomous recompilation in a closed-loop ratchet.
- A contract-gated verification framework using differential replay and, for generative spans, an LLM judge with declared agreement thresholds.
- Auto-Bench, a benchmark focused on determinism census, parity-gated compression, calibrated abstention, and amortized cost reduction over experience.
- An explicit empirical analysis of failure modes, especially loose guard calibration and unfaithful deopt references, showing these are central correctness bottlenecks.

### ⚙️ Methodology
Auto instruments live LLM-agent executions using SDK shims that capture prompts, tool calls, branches, environment reads, task I/O, latency, and provider-reported cost. It computes a determinism census over recorded spans, labeling a span as witnessed-deterministic only when the same input-output behavior is observed at least twice without error. Traces are lowered into a byte-stable typed task-graph IR annotated with capability effects, memory effects, uncertainty classes, and resource bounds. Compilation then follows two main paths: symbolic extraction and residual distillation. For symbolic behavior, Auto uses enumerative synthesis over a fixed DSL and LLM-guided counterexample-driven inductive synthesis, where candidate programs are proposed, executed in a sandbox, and rejected or refined using counterexamples. For non-symbolic residue, it trains small specialists such as trees or MLPs. Any candidate artifact must pass a behavioral contract via differential replay against recorded traces, with exact, statistical, or LLM-judged equivalence depending on the span type. Accepted artifacts are emitted as signed, content-addressed cognition binaries targeting WebAssembly, with manifests carrying only measured guarantees. At runtime, a calibrated guard decides whether an input is within the artifact’s supported region; if not, execution deopts to the reference agent, and the newly observed trace is fed back into recompilation, forming the ratchet loop.

### 🧪 Experiments
The evaluation introduces Auto-Bench, a pre-registered benchmark designed to measure compilability and amortization rather than just task success. On 560 recorded frontier-agent spans, 87.1% were classified as witnessed-deterministic, with three of four task families reaching 100% deterministic spans under the paper’s observational criterion. In a live 300-item stream with three scheduled distribution shifts, the closed-loop system produced three generations of compiled artifacts and reduced marginal cost from 59 to 2 micro-dollars per item, yielding a reported 6.4x end-to-end cost reduction. On witnessed inputs, the compiled path achieved 96.9% parity with zero errors under the accepted verification regime. The paper also reports baseline costs for uncompiled agents, including 55.6 micro-dollars and 904 ms for a single-call routing agent and 192.5 micro-dollars and 2.9 seconds for a three-call pipeline. Failure-mode experiments are central to the study: a loosely calibrated guard silently mislabeled 48.9% of compiled answers, and when the deoptimization reference was unfaithful, the verification gate refused recompilation rather than admitting a potentially incorrect artifact. These experiments emphasize guard calibration, differential verification, and reference fidelity as the key determinants of whether cost savings remain trustworthy.

### 🌱 Insights & Implications
The paper’s strongest conceptual move is to shift attention from better prompting or cheaper inference to compilation of recurring agent behavior into persistent executable artifacts with measured validity regions. That is directly relevant to software engineering because it reframes agents as systems whose behavior can be observed, typed, verified, sandboxed, and incrementally specialized. For maintenance and security, the WebAssembly confinement model and manifest-based capability declarations are especially interesting, since they provide a concrete mechanism for reducing the operational risk of learned or synthesized behavior. The main limitation is that determinism is measured only over witnessed traces, so the reported compilable fraction is empirical rather than semantic; the approach depends heavily on representative traces, calibration quality, and a faithful reference agent. Another open question is how well the synthesis and differential replay machinery scales to richer multi-step agents, long-horizon workflows, or tasks where correctness is hard to judge automatically. The LLM-judge component also introduces a second learned system into the verification loop, which may complicate assurance in high-stakes domains. Overall, the results suggest that a substantial slice of practical agent behavior may be amenable to verified specialization, but the hard problem is trustworthy boundary detection and fallback, not merely generating candidate compiled programs.

### 🎯 Why This Paper
This paper is relevant to software engineering, maintenance, security, automated repair, and vulnerability-oriented research because it treats LLM-agent behavior as compilable software artifacts subject to tracing, typing, synthesis, verification, sandboxing, and regression-style replay. Its contract-gated compilation and deopt/recompile loop are particularly relevant to research on automated program repair and vulnerability repair, since they resemble generate-and-validate workflows with explicit acceptance criteria and fallback behavior. The emphasis on capability confinement and measured guarantees also aligns with software security concerns, while the determinism census offers a useful empirical lens for deciding which agent behaviors can be safely operationalized or specialized in production systems.

### 💡 TL;DR
This paper studies how an attacker can poison the knowledge base of a policy-aware LLM-RAG system for Internet of Battlefield Things mission control and shows that even a single injected rule can frequently corrupt the model’s retrieved decision context. It also proposes CLD-KB, a lightweight dual-detector defense that blocks poisoned rules before they reach the LLM, restoring context integrity with very low runtime overhead.

- Introduces Query-Agnostic Semantic Retrieval Poisoning, a knowledge-base poisoning attack that crafts semantically broad malicious rules which rank highly across different operator query types without needing runtime prompt knowledge.
- Demonstrates that minimal poisoning is sufficient to corrupt PA-LLM-RAG retrieval context, with 85% LLM context corruption from one injected rule at a 1.6% poisoning rate and saturation behavior at higher poisoning levels.
- Proposes CLD-KB, a dual-detector defense that combines One-Class SVM anomaly detection with Member-Based Category Spread analysis over the three-category IoBT policy taxonomy.
- Shows that CLD-KB outperforms baseline anomaly detectors such as DBSCAN, LOF, K-Means, Isolation Forest, and standalone One-Class SVM in both poisoning detection and preservation of legitimate knowledge.

The paper evaluates security of the PA-LLM-RAG framework, an edge-deployed retrieval-augmented generation system for IoBT mission orchestration that retrieves top-k policy rules from a structured knowledge base containing 60 rules split across Workflow, Rules of Engagement, and Capability categories. The proposed attack, Query-Agnostic Semantic Retrieval Poisoning, assumes an adversary with write access to the rule base and injects malicious rules intentionally written to be semantically compatible with many mission prompts, allowing them to achieve high cosine-similarity retrieval scores regardless of the exact operator query. Attack success is measured primarily through context integrity: whether poisoned rules enter the LLM’s retrieved policy context. ### 🧪 Experiments
The evaluation is conducted on the PA-LLM-RAG IoBT setup using a structured policy knowledge base of 60 rules across three categories and semantic retrieval based on text embeddings with cosine similarity. The attack is tested over poisoning rates from 1.6% upward, including rates up to 25%, and the main metric is LLM context corruption or context integrity, i.e., whether poisoned rules appear in the retrieved top-3 context. The attack achieves 85% context corruption with a single injected rule at a 1.6% poisoning rate and remains effective across all tested poisoning levels, saturating at roughly 2.65 poisoned rules per LLM context by 7.7% poisoning. For defense evaluation, CLD-KB is compared against DBSCAN, LOF, K-Means, Isolation Forest, and One-Class SVM. The paper reports that CLD-KB significantly outperforms these baselines in detecting poisoned rules while preserving legitimate knowledge, and maintains 100% context integrity across poisoning rates from 1.6% to 25%. Runtime overhead is reported as approximately 7 ms per mission, supporting suitability for edge and real-time mission settings.

### 🌱 Insights & Implications
The paper’s main insight is that structured, policy-grounded RAG systems remain highly vulnerable at the retrieval layer even when the downstream LLM is policy-aware and judged before execution; trust in the knowledge base is a critical hidden assumption. The query-agnostic nature of the attack matters because it removes the need for prompt knowledge and makes poisoning more realistic in operational environments where attackers only compromise storage. The defense is notable because it exploits domain structure, specifically the policy taxonomy, rather than relying on generic clustering or expensive extra LLM calls. A limitation is that the study is tied to a relatively small, highly structured 60-rule IoBT knowledge base, so it remains unclear how well both attack and defense scale to larger, noisier, or more heterogeneous enterprise RAG corpora. Another open question is robustness against adaptive attackers who may optimize poisoning to mimic category-consistent embeddings and evade both detectors. For broader research, the work suggests that secure RAG for cyber-physical or mission-critical systems needs retrieval-layer hardening, taxonomy-aware detection, and explicit integrity metrics, which are directly relevant to secure software and AI pipeline design.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and vulnerability repair because it treats the RAG knowledge base as an attack surface and develops a concrete detection-and-mitigation pipeline for adversarial manipulation. For a researcher interested in automated program repair and secure software maintenance, it offers a useful pattern: characterize a previously under-modeled failure mode, define measurable integrity metrics, and build a lightweight defense that can be inserted into an operational pipeline without heavy inference overhead. ## 📄 [Conductance-Repair Evidence Graphs for Prospective Security Retrieval](https://arxiv.org/abs/2607.04070v1)  `60% relevant`  `🌐 Full Text`

### 💡 TL;DR
The paper proposes a deterministic, time-aware graph-repair framework for prospective security retrieval, where only evidence available by decision time is admissible and missing channels are widened through constrained graph flow rather than learned prediction. Its main result is that repair can substantially improve recall under missing-edge conditions, but broad admissible repair may slightly hurt average precision under random corruption, exposing an important precision-recall tradeoff.

### 🏆 Key Contributions
- Introduces conductance-repair evidence graphs for prospective security retrieval with timestamped admissibility masks that prevent post-decision evidence leakage.
- Defines a deterministic conductance-flow repair recurrence that widens only already-admissible channels and produces explicit repair certificates documenting withheld edges, repaired channels, backend availability, numerical deviations, and verifier outcomes.

### ⚙️ Methodology
The paper models prospective security retrieval as inference over a timestamped evidence graph G_tau whose nodes include security objects such as CVEs, documents, source layers, and terms. Edges are annotated with timestamps and source-layer identities, and only edges with time t <= tau are admissible at decision time. The method defines decision horizons that progressively admit pre-CVE evidence, CVE publication data, same-day NVD metadata, EPSS snapshots, KEV membership, and fix-commit evidence only when their timestamps do not violate the decision-time boundary. Repair operates at the channel level using a deterministic graph-flow recurrence rather than a learned model: a row-normalized transition matrix P_t is computed from the admissible adjacency matrix A_t, a nonnegative state vector z_t is updated using decay, propagation, and fixed seeding, and the adjacency is widened by adding masked outer-product mass eta * M ⊙ z z^T subject to symmetry, admissibility support, and a cap c. Because the support mask M is fixed by admissible edges, the algorithm cannot introduce forbidden post-decision links. The framework also emits a repair certificate that records source probes, decision time, forbidden edges, withheld benchmark edges, repaired channels, backend execution status, numerical deviation from a NumPy reference, leakage checks, saturation, and verifier results. The theoretical layer analyzes the recurrence through fixed-point arguments and bounded perturbation sensitivity, and defines deterministic decision rules for allowing, quarantining, or rejecting repair based on leakage detection, support-mask violations, numerical deviation, and harmful-repair thresholds.

### 🧪 Experiments
The artifact materializes 30 deduplicated public security records, 57 terms, and 58 withheld admissible document-term edges. Evaluation is framed as prospective retrieval under corrupted or missing channels, with baselines including BM25, degree, diffusion-style methods, and graph-repair comparisons. Metrics include recall@k and average precision, with explicit attention to precision-sensitive degradation rather than recall alone. Under random edge withholding on the public security set, conductance repair improves recall@k from 0.017 to 0.069 while average precision changes from 0.062 to 0.060, indicating meaningful recovery of missed evidence but a slight AP drop. On a synthetic security fixture, recall@k improves from 0.055 to 0.099. The implementation also benchmarks the same flow, SVD, and einsum kernel across NumPy, PyTorch, JAX, and TensorFlow when available, and logs backend unavailability rather than treating backend choice as a model-quality result. Non-security datasets such as BBBC019 and LIVECell are used only as structural controls for sparse evolving channels, not for domain performance claims.

### 🌱 Insights & Implications
The central insight is that security retrieval should be evaluated prospectively, because time leakage from sources like KEV entries, enriched NVD text, EPSS updates, or fix commits can produce unrealistic retrieval performance if complete evidence is assumed. The paper’s deterministic repair mechanism is attractive for auditability in security settings because it avoids learned hallucination of missing channels and produces explicit certificates, which is valuable for high-assurance workflows. At the same time, the observed AP drop on public random-corruption experiments shows that widening admissible neighborhoods can recover relevant items while also diffusing mass onto marginally relevant evidence, so repair needs strict safety thresholds and precision-aware governance. The theory is useful for framing tractability and safety, but the empirical scale is small, and the current benchmark appears closer to a proof-of-concept artifact than a deployable vulnerability triage system. Open questions include how the framework behaves on larger real-world corpora, whether channel-specific repair policies outperform uniform widening, how robust the certificate rules are under targeted poisoning, and whether the method can be integrated with stronger retrieval models without compromising temporal admissibility and auditability.
