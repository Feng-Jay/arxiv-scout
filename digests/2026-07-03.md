## 📄 [Knowledge Over Parameters: Evolving Smart Contract Vulnerability Detection](https://arxiv.org/abs/2607.01742v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Yuqiang Sun, Han Liu, Ying Li, Yiran Zhang, Zong Cao, Ziyun Guo, Yang Liu
📅 **Published:** 2026-07-02  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper proposes EvoVuln, a framework that treats smart contract vulnerability detection as evolving executable procedural knowledge rather than tuning model parameters. By compiling natural-language detection rules into controlled executable policies and iteratively refining them with only 5 vulnerable and 5 safe examples per vulnerability type, EvoVuln reaches 71% macro-F1 across five vulnerability classes and outperforms all baselines.

### 🏆 Key Contributions
- Reframes smart contract logic vulnerability detection as a procedural knowledge generation and evolution problem instead of prompt engineering or parameter fine-tuning.
- Introduces a Runtime with an Inversion of Control architecture that compiles natural-language rules into Executable Policies, separating deterministic control flow from LLM-based semantic judgments.
- Proposes a two-phase rule evolution pipeline: Cold Start for bootstrapping and stress-testing initial rules with synthesized corner cases, and Few-Shot Evolving for abductive semantic debugging using minimal labeled real-world samples.
- Demonstrates that evolved vulnerability knowledge is portable across LLMs, enabling small low-cost models to outperform larger zero-shot models without retraining.
- Shows strong empirical results across five real-world smart contract vulnerability types, surpassing static analysis, rule-based, deep learning, fine-tuned, zero-shot LLM, and coding-agent baselines.

### ⚙️ Methodology
EvoVuln represents vulnerability expertise as a structured natural-language detection rule and manages it as a persistent artifact that is synthesized, executed, diagnosed, and revised. The system uses two LLM roles: a supervisor model for offline rule analysis and updates, and a detector model for online contract inspection. Its core execution engine is a Runtime based on Inversion of Control. Instead of letting an LLM freely reason through a long multi-step prompt, EvoVuln compiles the rule into an Executable Policy (EP), a deterministic program-like detection plan that enforces the required analysis sequence. The EP invokes the LLM only through tightly scoped semantic primitives when localized understanding is needed, such as interpreting contract intent or checking whether a code fragment satisfies a semantic condition. This decouples rigid procedure execution from probabilistic language reasoning, reducing reasoning drift and making the detector follow the rule faithfully. The Runtime also records dense execution telemetry and step-level traces, which are then used for diagnosis.

Rule refinement is performed without model parameter updates through abductive semantic debugging. When a prediction fails on a labeled example, the system analyzes the trace to localize which step or semantic judgment caused the mismatch and then updates the rule rather than the model. The evolution process has two phases. In Cold Start, EvoVuln bootstraps an initial rule and stress-tests it with automatically synthesized corner cases to expose coverage gaps and brittle logic. In Few-Shot Evolving, the system grounds and corrects the rule using a tiny labeled set of real contracts, specifically five vulnerable and five safe examples per vulnerability type. This phase uses observed failures plus runtime traces to revise the policy so it better captures real-world attack semantics. The overall design is analogous to continuous integration for semantic policies: execute, trace, detect failure, patch rule, and regress-test again. The resulting evolved procedural knowledge can then be deployed with a cheaper detector model, while the more capable supervisor is only needed during the one-time evolution stage.

### 🧪 Experiments
The evaluation covers five real-world smart contract vulnerability types. EvoVuln is compared against nine baselines spanning static analysis, rule-based methods such as GPTScan-style approaches, deep learning methods, fine-tuning-based methods, zero-shot LLMs, and coding-agent baselines. The main metric reported is macro-average F1-score across vulnerability types. EvoVuln achieves 71% macro-F1, outperforming all baselines. A notable result is that when the evolved procedural knowledge is executed using a lightweight model such as GPT-5-nano, EvoVuln exceeds a much larger GPT-5.2 model used in zero-shot mode by 19 percentage points. The paper also studies transferability of the evolved rules: without retraining, applying the same evolved knowledge to Qwen3.5-9B yields 66% macro-F1 and to MiniMax-M2.5 yields 67% macro-F1, both above GPT-5.2 zero-shot at 52%. The data efficiency is central to the setup: Few-Shot Evolving uses only five vulnerable and five safe labeled examples per vulnerability type, and the total one-time API cost to evolve rules for all five vulnerabilities is reported to be under $50.

### 🌱 Insights & Implications
The paper’s main insight is that for smart contract logic bugs, procedural knowledge and execution discipline matter more than raw model scale. Vulnerability detection fails for two distinct reasons: the rule may be incomplete or wrong, and the LLM may fail to faithfully follow even a correct rule. EvoVuln addresses both by making rules explicit, executable, and revisable. This is particularly important for security tasks where attack patterns are compositional and require step-by-step reasoning over semantics and invariants rather than surface code features. The strong cross-model transfer suggests the learned asset is the evolved detection procedure itself, not hidden model-specific behavior. This has implications for software security tooling: maintaining a library of executable, evolvable vulnerability policies may be more sustainable than repeatedly fine-tuning detectors for scarce-label domains.

Limitations and open questions include how well the approach scales to a larger and more diverse vulnerability taxonomy, whether policy evolution remains stable as rules become more complex, and how robust the semantic primitives are when detector models are weaker or when contract code is highly obfuscated. Another open issue is coverage: synthesized corner cases and few-shot examples may still miss rare exploit variants. There is also a likely engineering tradeoff between strict procedural control and flexibility for novel cases. For related research in automated repair and vulnerability remediation, a natural extension would be to evolve not only detection policies but also executable repair policies or exploit-prevention checks grounded in runtime traces.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, vulnerability detection, and automated repair research because it treats security expertise as a maintainable software artifact rather than a one-off prompt or opaque trained model. Its IoC-based executable policy design aligns with program analysis and tool-building concerns, while the few-shot abductive evolution mechanism is directly useful for low-label vulnerability detection settings common in smart contracts and emerging software flaws. For a researcher working on vulnerability detection and repair, the paper offers a transferable framework for building auditable, evolvable, low-cost security analyzers that could be extended from bug detection to root-cause localization and automated patch generation.

## 📄 [Knowledge-Enhanced Agentic Vulnerability Repair](https://arxiv.org/abs/2607.00820v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Sicong Cao, Hao Ma, Le Yu, Kangyi Ding, Xiaolei Liu, Terry Yue Zhuo, Bo Wang, Xingwei Lin, Xiaobing Sun, Linzhang Wang, David Lo
📅 **Published:** 2026-07-01  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper proposes KeaRepair, an agent-based automated vulnerability repair system that combines verified program facts from static-analysis-guided diagnosis with structured vulnerability knowledge retrieved from historical fixes. On 55 reproducible C/C++ vulnerabilities, KeaRepair achieves 46 successful repairs (83.64%) with Gemini-3.1-Pro, outperforming prior AVR methods and showing strong cross-language generalization.

### 🏆 Key Contributions
- Identifies two central bottlenecks in LLM-based automated vulnerability repair: unreliable root-cause analysis and weak use of historical vulnerability-fix knowledge
- Introduces KeaRepair, an agentic AVR framework that grounds repair in verified program facts gathered via tool-augmented ReAct-style diagnosis instead of relying only on LLM free-form reasoning or perfect localization assumptions
- Builds dual-view, multi-dimensional vulnerability knowledge bases from historical vulnerability-patch pairs, capturing complementary control-flow and data-flow oriented repair knowledge for retrieval
- Designs knowledge-level retrieval-augmented patch generation that uses diagnostic reports to retrieve relevant exemplars and associated high-level repair knowledge rather than only surface-similar code
- Implements a closed-loop repair workflow with iterative patch validation and refinement using compilation, PoC replay, and test-suite execution
- Demonstrates state-of-the-art empirical performance on 55 reproducible C/C++ vulnerabilities and strong cross-language transfer to vulnerabilities in other languages

### ⚙️ Methodology
KeaRepair is a three-phase agentic automated vulnerability repair framework. First, it constructs dedicated retrieval knowledge bases from historical vulnerability-patch pairs. The paper emphasizes dual complementary views of vulnerabilities—motivated by the observation that many vulnerabilities can be understood through control-flow or data-flow perspectives—and extracts multi-dimensional knowledge items under supervision from ground-truth patches. These knowledge items encode higher-level vulnerability characteristics and repair strategies, enabling retrieval beyond plain code similarity.

Second, KeaRepair performs vulnerability diagnosis with a tool-augmented agent. Instead of assuming exact vulnerable statements or depending only on sanitizer reports, the system uses specialized static-analysis tools to actively collect verified program facts from the target code. The agent follows a ReAct-style reasoning process: it reasons about what information is needed, invokes tools to inspect program structure and behavior-relevant facts, and synthesizes a structured diagnostic report. This design aims to reduce hallucination and improve root-cause understanding by grounding the LLM in externally verified evidence.

Third, KeaRepair carries out knowledge-enhanced patch generation. Based on the structured diagnosis, it formulates semantic queries from the relevant vulnerability view(s) and retrieves historical exemplars together with their associated knowledge items from the knowledge bases. These retrieved items guide the LLM to generate candidate patches using both concrete examples and abstract repair knowledge. The generated patches are then subjected to a closed-loop validation pipeline involving compilation, proof-of-concept replay, and test-suite execution. If a patch fails, KeaRepair iteratively refines it using the validation feedback until a correct repair is found or the attempt budget is exhausted. Overall, the framework couples grounded diagnosis, structured knowledge retrieval, generation, and validation in an iterative agentic workflow.

### 🧪 Experiments
The evaluation uses 55 reproducible real-world C/C++ vulnerabilities from Vul4C as the main benchmark. KeaRepair is compared against eight prior AVR approaches spanning rule-based, symbolic, learning-based, and LLM-based methods, including Senx, VulRepair, VulMaster, ExtractFix, APPatch, PatchAgent, and other recent systems cited as [3, 54, 12, 11, 50, 14, 26, 47]. The study also evaluates KeaRepair with diverse LLM backbones and reports cost and runtime.

The main metric is vulnerability repair success rate, with validation based on successful compilation, PoC replay, and test-suite execution. On the 55 C/C++ vulnerabilities, KeaRepair paired with Gemini-3.1-Pro repairs 46 vulnerabilities, reaching 83.64% success. The paper reports this as substantially better than all studied baselines. It also achieves a practical efficiency profile, with about $0.30 cost and under 50 seconds per vulnerability on average. KeaRepair fixes nine vulnerabilities that the strongest baseline PatchAgent cannot repair, indicating complementary capability beyond aggregate performance. In addition, the paper presents cross-language evaluation on 173 vulnerabilities from another benchmark, where KeaRepair achieves up to 148/173 successful repairs (85.55%), suggesting the method generalizes beyond the C/C++ setting and beyond sanitizer-dependent workflows.

### 🌱 Insights & Implications
A key insight is that vulnerability repair benefits from separating diagnosis from generation and grounding diagnosis in verified program facts rather than relying on unconstrained LLM reasoning. This is especially relevant for software security, where hallucinated root causes or wrong fix locations can easily lead to plausible but insecure patches. Another important insight is that historical repair knowledge should be represented at a higher semantic level—such as control/data-flow oriented vulnerability knowledge and repair strategies—because nearest-neighbor retrieval over code alone often retrieves superficially similar but repair-irrelevant examples.

The work also suggests that agentic repair pipelines are more effective when tool use, retrieval, and validation are tightly integrated in a feedback loop. For AVR and related automated program repair research, this points toward hybrid systems that combine program analysis, knowledge engineering, and LLM generation rather than treating patching as a pure seq2seq or prompting problem.

Likely limitations and open questions include dependency on the quality and coverage of historical vulnerability-patch pairs used to build the knowledge bases, possible sensitivity to the chosen static-analysis tools and their scalability on very large codebases, and unclear behavior when validation signals are weak or incomplete. The benchmark size for the main C/C++ evaluation is still modest, and more evidence would be useful on patch correctness beyond passing tests and PoC replay, such as semantic equivalence or absence of regression vulnerabilities. Another open question is how well the approach handles multi-file, architecture-specific, or highly stateful vulnerabilities where diagnosis and repair require deeper whole-program reasoning.

### 🎯 Why This Paper
This paper is highly relevant because it directly targets automated vulnerability repair, vulnerability detection-to-remediation workflows, and secure software maintenance—core areas of the researcher's interests. It advances the state of the art by combining program-analysis-grounded diagnosis, retrieval of structured fix knowledge, and iterative validation, offering concrete ideas for building more reliable APR/AVR systems, improving vulnerability patch correctness, and extending LLM-based repair beyond brittle localization or surface-level code retrieval.

## 📄 [Refploit: Facilitating Exploit Construction via Code-Agent Trajectory Repair](https://arxiv.org/abs/2607.01760v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Zirui Chen, Zhipeng Xue, Jiayuan Zhou, Xing Hu, Xin Xia, Xiaohu Yang
📅 **Published:** 2026-07-02  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper presents Refploit, an LLM-based framework that repairs failed code-agent exploit-generation trajectories by validating whether they truly exercise vulnerable logic and then reusing useful trajectory segments while constraining misleading ones. On 172 Java exploit references for 143 vulnerabilities, Refploit reproduces 138 exploits (80.2%), substantially outperforming the base agent, PoCGen, and Codex with GPT-5.4.

### 🏆 Key Contributions
- A systematic empirical study of the gap between disclosed Java exploit references and runnable exploits, showing that reproduction typically requires nontrivial environment setup, harness assembly, and exploit adaptation.
- A new framework, Refploit, that treats failed code-agent trajectories as partially useful artifacts, validates them via differential execution, analyzes reproduction progress, maps progress back to trajectory segments, and derives preservation/repair constraints for focused recovery.
- A differential-validation strategy that reduces false exploit reproductions by checking whether generated exploits actually trigger vulnerability-relevant logic and produce different behavior on vulnerable versus patched versions.
- A progress-guided trajectory analysis and constraint-guided recovery mechanism that improves repair effectiveness and efficiency over restarting exploit generation from scratch.
- A large-scale evaluation on three open-source Java vulnerability datasets covering 172 exploit references for 143 vulnerabilities and 53 CWE types, showing state-of-the-art exploit reproduction performance and cross-agent generality.

### ⚙️ Methodology
Refploit is designed for exploit construction from incomplete public exploit references such as textual steps, code snippets, exploit scripts, and advisories. The core premise is that an initially failed code-agent trajectory often contains both useful progress and misleading actions. Instead of discarding such trajectories, Refploit recovers and repairs them.

The framework begins with an initial exploit-generation trajectory produced by a code agent. It then performs validation through differential execution: the generated exploit is executed on both a vulnerable/selected version and a patched version of the target library or project. A trajectory is considered effective only if it demonstrates the expected differential behavior, rather than merely running successfully. This addresses a key failure mode of code agents, which may create exploits that appear plausible but bypass the actual vulnerable API path, replace it with self-implemented logic, or otherwise avoid the intended vulnerable code.

When validation fails, Refploit performs reproduction-progress analysis along the three dimensions identified in the preliminary study: environment preparation, harness assembly, and exploit reproduction. Environment preparation checks whether required runtime conditions, tools, configurations, OS/JDK assumptions, or dependencies are satisfied. Harness assembly assesses whether the constructed project and code actually invoke the vulnerability-relevant library logic and whether the malicious input and executable harness are correctly assembled. Exploit reproduction checks whether the exploit causes observable vulnerability behavior under the prepared setup.

Refploit then maps these progress judgments back to the code-agent trajectory. By locating trajectory segments associated with successful subtasks, it extracts preservation constraints that indicate what should be retained in the next attempt. By identifying misleading directions and unresolved gaps, it extracts repair constraints specifying what must be changed or avoided. These constraints effectively summarize both what the previous attempt got right and what caused failure.

Using these constraints, Refploit resumes the code agent with a focused recovery task rather than restarting generation from scratch. The recovery process is iterative: regenerate or revise the exploit under the derived constraints, re-run differential validation, reassess progress if still ineffective, and continue until a valid exploit is obtained or the budget is exhausted. The paper also adapts this repair framework to another code agent, showing that Refploit is agent-agnostic and works as a wrapper over different code-generation backends.

Conceptually, the method combines exploit validation, trajectory diagnosis, and iterative repair. Its novelty is not only in generating exploits, but in recovering from deceptive or partially successful trajectories in a structured way that emphasizes actual vulnerable-logic execution.

### 🧪 Experiments
The evaluation uses three open-source Java vulnerability datasets comprising 172 exploit references for 143 unique vulnerabilities. These vulnerabilities span 53 CWE types and cover at least one CWE associated with 61.9% of Maven vulnerabilities in GitHub Advisory. The exploit references come in multiple forms, including textual steps, test snippets, exploit scripts, and advisories.

The preliminary manual study examines 84 exploit references linked to vulnerabilities from CWE-Bench-Java and VISION. After filtering inaccessible and irrelevant links, two experienced researchers manually attempt reproduction with a two-hour limit per case, recording 159 interventions and producing 74 valid exploits. This study motivates the three-dimensional progress model of environment preparation, harness assembly, and exploit reproduction.

For automated evaluation, Refploit is instantiated with DeepSeek-V4-Flash as the main LLM backend. Baselines include the underlying base code agent mini-swe-agent, the state-of-the-art exploit-generation system PoCGen, and advanced code agents including Codex with GPT-5.4. The paper also evaluates a variant where Refploit is adapted to another code agent to test generality.

The main result is that Refploit successfully reproduces 138 of 172 exploits, achieving an 80.2% reproduction rate. This corresponds to a 64.3% relative improvement over the initially generated trajectories. The base mini-swe-agent achieves 48.8%, while Codex with GPT-5.4 achieves 69.2%. Refploit also outperforms PoCGen, including on PoCGen’s predefined CWE categories, indicating broader generality beyond vulnerability-specific handcrafted validation patterns.

Ablation studies show that each component matters. Differential execution reduces false reproductions by filtering exploits that run but do not actually trigger the vulnerability-relevant logic. Progress-guided trajectory analysis and constraint-guided recovery improve repair effectiveness and efficiency by reducing unnecessary reattempts and preserving useful progress from failed trajectories. The cross-agent adaptation further shows consistent improvement when Refploit is layered on top of another code agent, supporting claims of generality.

### 🌱 Insights & Implications
A key insight is that exploit-generation failures by code agents are often informative rather than useless. Failed trajectories can expose which reproduction subtasks are already solved and which actions are actively misleading, making repair more effective than full regeneration. This is especially relevant to automated program repair and security-oriented code agents, where intermediate states often contain reusable environment setup, harness code, or payload construction.

The paper also highlights an important evaluation issue for vulnerability reproduction: simply checking whether generated code runs or produces some security-themed behavior is insufficient. Differential validation against vulnerable and patched versions is necessary to avoid false positives where the exploit bypasses the real vulnerable logic, reimplements behavior locally, or triggers unrelated effects.

For software security research, the work suggests a shift from vulnerability-type-specific exploit-generation heuristics toward validation- and recovery-centric workflows that can scale across many CWE categories. This is useful for downstream impact analysis, exploit test generation, and vulnerability confirmation in dependency ecosystems such as Maven.

Limitations likely include dependence on the availability of a patched version or equivalent differential target, the quality of the initial agent trajectory, and the need for executable project contexts where environment setup and harness execution are feasible. The approach is evaluated on Java library vulnerabilities and public exploit references, so transfer to other languages, binary exploits, or vulnerabilities lacking suitable differential behavior remains an open question. Another open issue is cost: iterative execution, progress diagnosis, and repeated agent repair may increase computational overhead, though the paper argues this is offset by improved efficiency over blind restarts.

For related research, Refploit connects naturally to automated program repair, agentic software engineering, exploit synthesis, regression testing, and vulnerability verification. Its trajectory-repair perspective may inspire similar methods for fixing failed vulnerability detectors, patch validators, or test-generation agents.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, software maintenance, software security, automated program repair, vulnerability detection, and vulnerability repair because it frames exploit construction as an iterative diagnosis-and-repair problem over code-agent trajectories. It advances the researcher’s work by introducing a principled way to validate whether generated artifacts truly exercise vulnerable logic, recover from partial failures using constraint-guided repair, and generalize across many vulnerability types without relying on narrow handcrafted patterns.

## 📄 [A Single Patch Is Not Enough: Deterministic Fusion of Repair Candidates](https://arxiv.org/abs/2607.01597v1)  `90% relevant`

👤 **Authors:** Boyang Yang, Xiangliang Hu, Luyao Ren, Yanjun Chen, Bach Le, Tegawendé F. Bissyandé, Haoye Tian
📅 **Published:** 2026-07-02  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper targets the practical gap between pass@k evaluation and real-world pass@1 deployment for automated program repair by proposing PatchFusion, a deterministic method that combines evidence across multiple candidate patches instead of merely ranking them. Across SWE-bench Verified, SWE-bench Multilingual, and Defects4J fixed candidate pools, PatchFusion consistently outperforms test-based, ranking-based, and LLM-judge selectors, while recovering additional correct repairs that no single source patch alone would have delivered.

### 🏆 Key Contributions
- Introduces PatchFusion, a deterministic post-generation repair method that fuses shared edit-level evidence across candidate patches to construct a final patch rather than only selecting one existing candidate.
- Proposes a two-stage fusion pipeline: repair-neighborhood formation via whole-diff agreement and evidence-constrained fusion (ECF) that preserves repeated edit atoms while pruning unsupported edits.
- Avoids using test outcomes at decision time, making the method cheaper and potentially more deployable than test-based post-generation selection approaches.
- Builds PatchFuseBench, a fixed-pool benchmark spanning SWE-bench Verified, SWE-bench Multilingual, and Defects4J for evaluating post-generation patch selection/fusion.
- Shows that cross-candidate evidence fusion recovers bugs unsolved by any individual source system and surpasses matched candidate-pool selectors across all evaluated benchmarks.
- Provides ablation and resampling analyses indicating that ECF contributes additional solved bugs without observed regressions and that gains are robust to candidate-pool variation.

### ⚙️ Methodology
PatchFusion is a post-generation automated program repair framework designed for settings where multiple candidate patches are available from one or more repair sources, but only a single patch can be proposed to a developer. Instead of treating each candidate as an indivisible unit, the method operates on shared edit evidence across candidates. The pipeline appears to have two main stages. First, it identifies a repair neighborhood by fusing whole-diff agreement among candidates; this likely groups similar candidate patches based on overlapping diffs or edit structures, under the assumption that repeated agreement signals correctness. Within this neighborhood, PatchFusion selects an auditable representative patch that serves as a base or reference artifact for inspection and downstream synthesis. Second, it applies evidence-constrained fusion (ECF), an edit-atom-level fusion procedure. Edit atoms are likely minimal patch actions such as insertions, deletions, replacements, or localized diff hunks. ECF retains edit atoms that recur across candidates and removes patch components lacking sufficient support, thereby constructing a final fused patch that is more likely to preserve consensus-correct edits while discarding noisy or hallucinated changes. The approach is deterministic, suggesting fixed decision rules rather than stochastic inference, and it explicitly does not consult test execution results during final decision making. This distinguishes it from generate-and-validate APR pipelines and from post hoc reranking using dynamic validation. The method also appears to support complementary multi-source candidate pools, meaning candidates may be produced by heterogeneous repair systems or LLM agents and then fused into a stronger final output.

### 🧪 Experiments
The evaluation uses PatchFuseBench, a fixed-pool benchmark created for post-generation patch fusion/selection. It covers three repair datasets: SWE-bench Verified, SWE-bench Multilingual, and Defects4J. The candidate pools are described as fixed and complementary multi-source pools, implying that each bug has a pre-collected set of candidate patches from multiple repair sources, enabling fair comparison among post-generation selectors. Baselines include matched candidate-pool selectors from several categories: whole-candidate ranking methods, test-based filtering/selection methods, and LLM-based judge selectors. The main metrics are bugs solved on the SWE-bench datasets and plausible patches on Defects4J; the abstract does not report exact validation criteria, but SWE-bench likely uses benchmark-specific correctness and Defects4J uses plausibility under tests. PatchFusion solves 426/500 bugs on SWE-bench Verified, 236/300 on SWE-bench Multilingual, and produces 87/371 plausible patches on Defects4J, outperforming every matched selector on all three benchmarks. It additionally recovers 41 bugs on one SWE-bench benchmark and 27 on the other that no single source solves, which is 30 and 18 more, respectively, than the best individual source. Relative to the candidate-reachable ceiling, PatchFusion reaches 96.2% on SWE-bench Verified and 89.7% on SWE-bench Multilingual, indicating that it captures most of the correctness already latent in the candidate pool. Ablation studies show that ECF contributes +5, +6, and +9 solved bugs across the three datasets by recovering in-pool repairs missed by pure selection, with no observed regression. Additional robustness experiments based on resampling candidate pools indicate that PatchFusion's gains remain stable. The abstract also emphasizes efficiency: compared with test-based and LLM-based selectors, the deterministic fusion method achieves better patch recovery at orders-of-magnitude lower cost.

### 🌱 Insights & Implications
The paper highlights an important but underexplored problem in LLM-based repair: strong pass@k can hide weak practical utility when systems must emit exactly one patch. Its core insight is that correct repair evidence may be distributed across multiple imperfect candidates, so selection alone is insufficient; reconstruction via edit-level consensus can outperform choosing the 'best' whole patch. This is especially relevant for APR and vulnerability repair workflows where patch reliability, auditability, and low decision-time cost matter. Another notable implication is that test-free decision-time fusion can rival or exceed test-based selectors, suggesting that structural agreement among candidates is a powerful correctness signal. The strong proximity to the candidate-reachable ceiling indicates that future improvements may come more from enriching candidate diversity and quality than from better reranking alone. Limitations inferred from the abstract include dependence on having a sufficiently rich multi-source candidate pool, possible sensitivity to edit-atom definitions and alignment quality, and uncertainty about semantic correctness on Defects4J where only plausibility is reported. Open questions include how the approach handles conflicting but individually plausible edits, whether it generalizes to security patching and vulnerability-specific repairs, how it behaves on larger multi-file patches or repository-wide transformations, and whether fusion can be integrated directly into generation loops rather than applied post hoc.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, maintenance, automated program repair, and vulnerability repair because it addresses the practical challenge of turning multiple generated repair candidates into one deployable patch. Its deterministic, auditable, low-cost fusion strategy is particularly valuable for secure and maintainable repair pipelines, where developers need trustworthy final patches rather than optimistic pass@k statistics. For work on vulnerability detection/repair and APR, the idea of reusing cross-candidate edit evidence could inspire stronger patch selection, patch synthesis, and ensemble-based remediation systems.

## 📄 [Antaeus: Hunting Repository-Level Logic Vulnerabilities via Context-Grounded LLM Reasoning](https://arxiv.org/abs/2607.01138v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Michele Armillotta, Nicolò Romandini, Rebecca Montanari, Lorenzo Cavallaro
📅 **Published:** 2026-07-01  🏷️ **Categories:** cs.CR

### 💡 TL;DR
Antaeus is a repository-level LLM framework for finding logic vulnerabilities in C/C++ by grounding model reasoning in project-specific security context rather than isolated functions. On 28 real-world repositories with confirmed CWE-200/CWE-284 bugs, it detected and explained 15 vulnerabilities, outperforming function-level and frontier agentic baselines at similar token and cost budgets.

### 🏆 Key Contributions
- Frames repository-level logic vulnerability detection as recovery and checking of application-specific security invariants rather than pattern matching or generic bug finding.
- Introduces Antaeus, a staged pipeline combining repository-wide function prioritization, context-grounded LLM reasoning, comparative validation across analogous code, and structured actionable reporting.
- Shows that targeted repository context is more effective than both isolated function analysis and broad agentic exploration for logic vulnerabilities in C/C++.
- Provides evidence that logic-vulnerability detection can be improved without exploding token usage, model calls, or maintainer triage effort.

### ⚙️ Methodology
Antaeus operates as a repository-scale analysis pipeline designed for logic vulnerabilities whose security conditions are implicit and distributed across code. First, it performs lightweight repository-wide prioritization to rank functions using security-relevant signals, so expensive LLM analysis is focused on code likely to involve sensitive operations, trust boundaries, resource manipulation, authorization, or information exposure. Second, for each prioritized function, it constructs a grounded analysis context by combining local code augmentation with a stable repository-level security summary. This summary captures the application's functionality, security-relevant resources, principals, trust boundaries, conventions, and related enforcement points elsewhere in the repository. Third, the LLM is prompted to reason in a structured way: identify security-sensitive sinks in the function, infer the safety conditions or invariants required for those sinks to be executed safely, and determine whether the needed conditions are locally enforced or missing. Rather than asking the model to generically 'find a bug,' Antaeus constrains the reasoning around explicit sinks, preconditions, and evidence. Fourth, candidate findings are subjected to comparative validation against similar sinks or analogous code paths in the same repository, pruning cases that merely reflect repository-wide norms or acceptable design choices rather than distinctive invariant violations. Finally, Antaeus emits a triage-oriented report containing the sink, the violated or missing safety condition, and supporting repository evidence, making findings traceable and actionable for human reviewers. Overall, the system is built around the idea that successful logic-vulnerability detection requires reconstructing repository-specific invariants from distributed code context, not just increasing the amount of context or allowing unconstrained agentic search.

### 🧪 Experiments
The evaluation uses 28 real-world C and C++ repositories containing confirmed logic vulnerabilities, specifically focusing on CWE-200 and CWE-284 cases. Antaeus is compared against two main categories of baselines: function-level LLM analysis that reasons over isolated code snippets, and frontier agentic repository-analysis models, including Opus 4.8 Agentic and Codex 5.4. The study assesses vulnerability-detection effectiveness together with practical analysis cost, including token usage and resulting triage burden. Antaeus detected and explained 15 confirmed vulnerabilities, substantially outperforming the baselines while staying within a comparable token-usage and monetary-cost budget. The reported gains come from better context selection and grounding rather than from simply spending more tokens or exploring more repository code. The paper emphasizes that Antaeus not only flags vulnerabilities but also explains the missing security conditions and supporting evidence, improving validation and remediation usefulness compared with less structured approaches.

### 🌱 Insights & Implications
The paper highlights that logic vulnerabilities differ fundamentally from memory-safety or taint-style bugs because they are defined by violations of repository-specific security invariants that are often implicit, dispersed, and only visible when code is interpreted in application context. A key insight is that more context is not automatically better; what matters is selecting and structuring the right context so the model can recover principals, resources, trust boundaries, and enforcement conventions. The results also suggest that unconstrained agentic exploration can waste budget on irrelevant code and still miss the semantic facts needed for correct security judgment, whereas targeted context grounding improves both precision and usefulness. Comparative validation across analogous code is an important mechanism for reducing false alarms by distinguishing true deviations from normal project conventions. ### ⚙️ Methodology
Kani is built as a verification-oriented backend for the Rust toolchain. The MIR-level design is central: unlike LLVM IR, MIR retains Rust-specific type and validity information, such as enum layouts, monomorphized types, and value invariants for types like bool and char, enabling checks that align with Rust semantics.

The loop-contract mechanism verifies inductiveness by checking initialization and preservation over a fixed symbolic encoding rather than by fully unwinding the loop, which permits unbounded reasoning. Verified contracts can then be reused soundly at call sites, reducing verification cost and avoiding repeated exploration of complex implementations. The paper grounds these constructs in Floyd-Hoare-style reasoning while keeping the user experience close to Rust testing, aiming for incremental adoption: start with no-annotation safety proofs, then add contracts only where stronger guarantees are needed.

The paper’s main insight is that Rust needs a middle ground between lightweight dynamic checking and heavyweight deductive verification. Kani occupies that space by offering strong automation first, then allowing users to add specifications incrementally when bounded checking is insufficient. This is especially relevant for real-world Rust, where unsafe code, runtime panics, and logic errors remain outside the compiler’s guarantees.

BOUND treats package hallucination as a boundary error: under a given software task context, the model fails to reliably distinguish existing packages from plausible but nonexistent ones. ### ⚙️ Methodology
The method starts from byte sequences and extracts length-n windows like standard n-grams, but aims to merge highly similar windows into generalized patterns. Each alphabet symbol is mapped bijectively to a unit-modulus complex number. For an n-gram t_j...t_{j+n-1}, the hash is a weighted complex dot product h = a^1 t_j + a^2 t_{j+1} + ... + a^n t_{j+n-1}, where a = exp(i phi) for random phi. Because adjacent windows differ by dropping one token and adding one token, the hash can be updated in constant time using the previous hash: multiply by a^{-1}, subtract the outgoing token contribution, and add the incoming token weighted by a^n. A 1-bit hash is then obtained from the sign of the real part of h, and multiple independently sampled hashes are concatenated to form a multi-bit LSH code.

This construction is intended to be locality-sensitive under Hamming distance: n-grams differing in only a few positions should produce similar complex sums and therefore collide in the same bit buckets with higher probability than distant strings. After hashing all observed n-grams, the algorithm groups n-grams by hash bucket to avoid quadratic all-pairs comparison. Within each bucket, it applies agglomerative clustering or similar local grouping based on Hamming similarity. For a cluster of related n-grams, the algorithm inspects each position: positions that are consistent remain fixed bytes, while positions that vary become single-character wildcards '?'. The result is a hamm-gram, e.g., AB?D, which compactly represents multiple exact n-grams within a bounded Hamming radius.

Because hamm-grams are fixed-length regular expressions with only per-position wildcards, they define a restricted regular language that can be implemented efficiently with a trie or DFA-like matcher. This preserves the deployment advantages of n-gram-style static features while making them less brittle to benign recompilation effects, register renaming, instruction substitutions of equal length, and small code edits that would otherwise destroy exact n-gram matches.

### 🏆 Key Contributions
- Introduces the Cognitive Firewall, an independent oversight layer placed between the user and a protected target LLM, with proactive pre-generation blocking and post-generation response screening.

### ⚙️ Methodology
The Cognitive Firewall is a runtime oversight framework that sits between the user and a protected target model. Instead of scoring prompts or responses independently, it analyzes the evolving conversation and makes categorical safety decisions through four gates. Three gates act before the target model is invoked. G1, the intent gate, infers the operational objective of the request independent of surface phrasing, attempting to recover whether the user is steering toward a harmful end even if the current turn appears benign. G2, the zero-trust context gate, treats any claims of role, authority, permission, policy override, or special legitimacy as unverified evidence rather than accepted context; this is designed to catch attacks that rely on assertions such as being a developer, auditor, clinician, or authorized operator. G3, the consistency gate, audits the trajectory of the dialogue for escalation, decomposition, and cross-turn coordination, such as splitting a harmful objective into innocuous subtasks or gradually steering the model toward restricted content. If any of these pre-generation gates emits a confident unsafe verdict, the interaction is blocked before the protected model generates. If all three allow the turn, the prompt reaches the target model, after which G4, the output risk gate, inspects the candidate response and withholds it if the content itself is unsafe. A key design choice is escalation-based decision making instead of averaging risk scores: gate outputs are not diluted into one scalar score, and any single strong signal is sufficient to trigger blocking. This preserves interpretability because each refusal can be tied to a specific rationale, and it is intended to address failure modes where a localized safety signal would be washed out by aggregate scoring. The framework is explicitly proactive, conversation-level, decomposed, and zero-trust in the handling of in-dialogue authority claims.

### 🧪 Experiments
The paper evaluates the Cognitive Firewall on four jailbreak attack benchmarks plus a benign safety set. The benchmarks cover single-turn attacks, multi-turn attacks, authority-based probes, and a difficult human-crafted jailbreak set; the abstract and body specifically situate the evaluation against attack styles such as single-turn jailbreaks, Crescendo-style multi-turn escalation, decomposed/actor-style attacks, and authority-claim manipulations. Baselines include published guard and oversight systems such as message-level guard classifiers and trajectory-aware defenses, as well as comparisons to prior decomposed or firewall-like methods discussed in the related-work positioning table. The main reported metrics are attack success rate (ASR), proactive containment before target generation, and over-refusal on benign prompts. The Cognitive Firewall reduces attack success to 2% or below on three of the four attack sets and to 14% on the hardest human-crafted set, while maintaining an 8% over-refusal rate on benign inputs. The authors also report per-gate ablations and robustness checks, showing that the decomposed multi-gate design contributes to improved containment across diverse attack modes and that pre-generation interception is a major benefit over defenses that only filter outputs after generation.

The paper’s main insight is that LLM safety failures in jailbreak settings are often not properties of isolated messages but of latent user objectives that unfold across a dialogue. By explicitly separating intent recovery, authority skepticism, trajectory analysis, and output inspection, the framework makes safety decisions more auditable and less vulnerable to score dilution than holistic risk scoring.
