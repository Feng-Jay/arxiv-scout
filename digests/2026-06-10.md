## 📄 [Early Comparative Evaluation of Transformer Models for Multilingual Software Vulnerability Detection](https://arxiv.org/abs/2606.10925v1)  `90% relevant`

👤 **Authors:** Fiza Naseer, Javad Khan, Muhammad Yaqoob, Alexios Mylonas
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper provides an early comparison of BERT, RoBERTa, and CodeBERT for binary vulnerability detection across multiple programming languages. The main takeaway is that transformer performance varies substantially by language, suggesting that multilingual vulnerability detection needs more language-aware modeling.

### 🏆 Key Contributions
- An early comparative evaluation of general-purpose and code-aware transformer models for vulnerability detection
- A multilingual study spanning HTML, Python, JavaScript, and PHP rather than a single-language benchmark
- Use of the CVEFixes dataset with language-wise three-fold stratified cross-validation
- Empirical evidence that model effectiveness differs across languages, highlighting the need for robust multilingual approaches

### ⚙️ Methodology
The study frames vulnerability detection as a binary classification task and compares three transformer-based models: BERT, RoBERTa, and CodeBERT. The evaluation is performed separately for HTML, Python, JavaScript, and PHP using the CVEFixes dataset. To reduce sampling bias and better assess generalization, the authors apply language-wise three-fold stratified cross-validation, then compare the models' detection behavior across languages to identify which architecture performs best under each setting.

### 🧪 Experiments
The experimental setup uses the CVEFixes dataset and evaluates binary vulnerability detection on four languages: HTML, Python, JavaScript, and PHP. The baselines are BERT, RoBERTa, and CodeBERT. The abstract does not report specific numeric scores or the exact metric(s), but it states that the results show clear performance differences across languages, implying that no single model is uniformly best across all language settings.

### 🌱 Insights & Implications
A key insight is that multilingual vulnerability detection is not a solved problem by simply applying a strong transformer model across all languages. Language-specific syntax, semantics, and data distributions likely affect performance, so future work should explore more language-aware pretraining, adaptation, or hybrid models. A limitation is that the abstract only reports an early comparative evaluation and does not indicate whether the differences are statistically tested, which metrics were used, or how well the findings generalize beyond the CVEFixes dataset. The work nevertheless suggests that vulnerability detection research should treat language diversity as a central design constraint rather than an afterthought.

### 🎯 Why This Paper
This paper is directly relevant to software vulnerability detection and secure software engineering because it evaluates transformer models for identifying vulnerable code across multiple languages. It advances the researcher's interests by highlighting the challenges of multilingual detection, providing a comparative baseline for BERT-like models, and pointing toward language-aware approaches that could also inform vulnerability repair pipelines and downstream security tooling.

## 📄 [Securing Code Understanding: Detecting Natural Backdoor Vulnerability in Code Language Models](https://arxiv.org/abs/2606.10846v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Yuchen Chen, Weisong Sun, Haocheng Huang, Yuan Xiao, Chunrong Fang, Yiran Zhang, Tingting Xu, Zhenpeng Chen, An Guo, Peizhuo Lv, Xiaofang Zhang, Zhenyu Chen, Yang Liu, Baowen Xu
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
The paper shows that natural backdoor vulnerabilities are common in normally trained Code Language Models, even at large scale, and that they can be detected and partially mitigated. It also introduces ScanNBT, a detection method that outperforms the prior CodeLM backdoor detector EliBadCode by finding more diverse natural backdoor candidates.

### 🏆 Key Contributions
- First empirical study of natural backdoor vulnerabilities in CodeLMs.
- Systematic evaluation across 44 scenarios covering six CodeLMs, four code intelligence tasks, and three programming languages.
- Comparison of injected vs. natural backdoors at both the model and parameter levels, showing natural backdoors are weaker but still security-relevant and harder to distinguish from clean behavior.
- Transferability analysis showing natural backdoor vulnerabilities can move across models trained on the same dataset, across models with the same architecture, and through distillation/shared knowledge.
- Cause analysis identifying dataset bias as a plausible source while finding the training procedure has limited impact, plus an evaluation of defenses and the proposal of ScanNBT.

### ⚙️ Methodology
This is a broad empirical security study of CodeLMs under both white-box and black-box threat models. The authors examine three pre-trained CodeLMs (CodeBERT, CodeT5, UniXcoder) and three large-scale CodeLMs (StarCoder, DeepSeek-Coder, GPT-3.5) on four tasks (defect detection, code search, code summarization, code repair) over Java, Python, and C. They search for naturally occurring backdoor triggers across 44 scenarios, then compare natural backdoors with injected backdoors at the representation and parameter levels. They next study transferability across datasets, architectures, and knowledge distillation, analyze possible causes using dataset bias and training-procedure variations, and evaluate pre-training, in-training, and post-training defenses. Finally, they propose ScanNBT, a detector intended to expose a broader and more diverse set of potential natural backdoor behaviors than prior work.

### 🧪 Experiments
The experiments span 44 scenarios, six CodeLMs (CodeBERT, CodeT5, UniXcoder, StarCoder, DeepSeek-Coder, GPT-3.5), four tasks (defect detection, code search, code summarization, code repair), and three languages (Java, Python, C). The study evaluates transferability across fine-tuning datasets, model architectures, and distillation-based surrogate models. Defense baselines include representative pre-training, in-training, and post-training methods, with a post-training unlearning-based defense reported as the most effective among those tested. For detection, ScanNBT is compared against EliBadCode, the prior state of the art. Reported metrics include ASR and ANR, plus diversity metrics Distinct-1 and Distinct-2; ScanNBT keeps ASR/ANR comparable while achieving much higher Distinct-1 and a Distinct-2 score close to 1.0. The paper does not provide exact numeric values in the provided text, but it reports that natural backdoors are prevalent and that even large CodeLMs do not reliably eliminate them.

### 🌱 Insights & Implications
Natural backdoors appear to be intrinsic to CodeLMs rather than a niche artifact of small models or a single task. They are weaker than injected backdoors but still meaningful because their triggers are entangled with clean code features and therefore difficult to separate from legitimate behavior. The transferability results suggest that vulnerability is not fully model-specific: it can persist across models trained on the same dataset, across identical architectures, and through distilled knowledge. The cause analysis points to dataset bias as a major driver, while the learning procedure itself seems less important, which raises questions about how to curate code corpora for security. A key limitation is that the paper relies on empirical detection and transfer studies rather than a complete mechanistic theory of why natural backdoors emerge, and black-box detection depends on the quality of the substitute model. The work also suggests that current defenses are not robust enough for naturally occurring vulnerabilities, making dedicated detection and unlearning strategies important for future CodeLM deployment.

### 🎯 Why This Paper
This paper is highly relevant because it targets a core security problem in CodeLMs used for code understanding and code generation, including tasks directly tied to vulnerability detection and code repair. It advances software security research by showing that backdoor risk exists even without poisoning, by characterizing how those vulnerabilities transfer and arise, and by proposing a detector that could be adapted into safer model screening pipelines for SE tooling, maintenance workflows, and automated repair systems.

## 📄 [Semantic Multi-Agent Intrusion Detection for IoT:Zero-Day and Adversarial Threats with Risk-Aware Reasoning](https://arxiv.org/abs/2606.10323v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Saeid Jamshidi
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR

### 💡 TL;DR
The paper proposes a semantic, multi-agent intrusion detection framework for IoT that reasons over telemetry as structured hypotheses rather than making a single direct classification. In experiments on multiple real-world IoT datasets, it reports 95.9% detection accuracy, a 6.8% false-positive rate, and 87.9% zero-day detection while remaining efficient enough for edge deployment.

### 🏆 Key Contributions
- A four-agent IDS architecture with role specialization: Scout for hypothesis induction, Mutator for adversarially constrained variants, Auditor for consistency checking, and Arbiter for final risk-aware decisions.
- A semantic inference pipeline that maps raw IoT telemetry into embeddings and then into structured attack hypotheses, enabling interpretable multi-stage reasoning instead of closed-set classification.
- A probabilistic fusion framework that combines outputs across agents to improve robustness to zero-day and adversarial attacks while preserving computational efficiency for constrained IoT environments.

### ⚙️ Methodology
The system treats intrusion detection as a semantic reasoning chain x -> z -> h -> y, where raw telemetry x is first transformed into a semantic embedding z, then into a structured hypothesis h, and finally into a detection decision y. The paper formalizes this as p(y|x) = integral p(y|h)p(h|z)p(z|x)dh, emphasizing uncertainty propagation across stages rather than a single end-to-end classifier. Four non-interchangeable agents implement the reasoning process: the Scout induces an initial attack hypothesis from semantic embeddings; the Mutator generates controlled adversarially constrained variants to stress-test the hypothesis; the Auditor evaluates consistency, contradiction, and semantic drift to filter unreliable outputs; and the Arbiter synthesizes the validated evidence into the final alert and an associated risk score. The design is explicitly aimed at open-world IoT intrusion detection, where unseen and adaptive attacks may violate closed-set assumptions. The paper also frames the system as modular and edge-friendly, so the reasoning stages can be deployed in resource-constrained heterogeneous IoT settings.

### 🧪 Experiments
The evaluation is described as being conducted across multiple real-world IoT datasets, with protocols covering zero-day holdout testing, adversarial robustness assessment, repeated-run stability, and cross-dataset transfer validation. The excerpt does not name the individual datasets or baselines, but it states that the system is compared through metrics including accuracy, F1-score, false-positive rate, structured output quality, and latency. Reported results in the abstract are 95.9% overall detection accuracy, a 6.8% false-positive rate, and 87.9% zero-day detection, with computational efficiency suitable for edge deployment. The paper claims that these results hold while also improving interpretability through traceable reasoning stages and risk-aware outputs.

### 🌱 Insights & Implications
The main research insight is that combining semantic embeddings with explicit multi-agent reasoning can improve both robustness and interpretability for IoT intrusion detection, especially against zero-day and adversarial threats. The work suggests that separating hypothesis generation, perturbation, verification, and arbitration is a useful pattern for security systems that must operate under uncertainty and distribution shift. Key open questions are how well the approach generalizes to stronger adaptive attackers, how sensitive it is to the quality of semantic embeddings, and how much overhead the multi-agent design adds in practical deployments. The excerpt also leaves important reproducibility details unclear, including the exact datasets, baselines, ablation studies, and the implementation specifics of the adversarial mutation and risk scoring components.

### 🎯 Why This Paper
This paper is directly relevant to software security and vulnerability detection because it addresses robust detection of unseen and adversarial behavior in resource-constrained systems. Its multi-agent, risk-aware reasoning design is also relevant to automated security analysis pipelines, where interpretability, uncertainty handling, and resistance to distribution shift matter. Even though it is about IoT intrusion detection rather than code repair, the architecture and evaluation ideas could inform future work on adaptive, explainable detection systems for software vulnerabilities and hostile runtime behavior.

## 📄 [Benchmarking and Exploring the Capabilities of LLMs for Attack Investigations](https://arxiv.org/abs/2606.10281v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Aniket Anand, Yiwei Hou, Daniel Fields, Alex Kantchelian, David Tao, Kurt Thomas, Grant Ho
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR, cs.CL

### 💡 TL;DR
AuditBench is a benchmark for measuring how well LLMs can perform security incident investigations over Linux and Windows audit logs. The paper shows that current frontier models have uneven capability: they can do well on some tasks such as exfiltration identification, but often over-report benign behavior as malicious, with performance shaped strongly by prompt design and log representation rather than model size alone.

### 🏆 Key Contributions
- Introduces AuditBench, a benchmark for LLM-based attack investigation over host audit logs from Linux and Windows.
- Covers 51 labeled benign and malicious investigation scenarios drawn from simulated ATT&CK-style behaviors and curated DARPA OpTC cases.
- Defines four common security investigation tasks relevant to incident response, including alert triage and identifying persistence mechanisms.
- Provides an extensible automated evaluation framework that measures both decision quality and explanation quality.
- Systematically evaluates five frontier LLMs across differences in model size, prompt construction, and raw versus compact log representations.
- Finds that LLMs often exhibit high false-positive tendencies, that larger models are not consistently better, and that compact preprocessing can improve both effectiveness and cost for some models.

### ⚙️ Methodology
The authors build AuditBench as a benchmark centered on host-level security audit data used during attack investigations. The dataset combines two sources: newly generated logs from Linux auditd and Windows procmon capturing both benign workflows and simulated attacks inspired by the MITRE ATT&CK framework, and carefully curated, labeled scenarios from DARPA OpTC. The benchmark spans 51 distinct scenarios and is organized around four investigation tasks that reflect real incident-response work, such as triaging alerts and identifying attacker persistence. To evaluate LLMs, the paper prepares multiple input formulations of the same evidence, including raw audit-log lines and a more compact preprocessed edge or provenance-style representation that abstracts low-level events into subject-action-object relations. Five frontier LLMs are prompted with task-specific instructions to analyze the logs, issue a verdict, and explain their reasoning. The framework then scores outputs using automated metrics for true positives, false positives, and explanation fidelity/accuracy. Beyond simple model comparison, the study varies model size, prompt wording, and data representation to isolate how each design choice changes performance and error patterns across tasks.

### 🧪 Experiments
The experiments evaluate five frontier LLMs on AuditBench’s 51 Linux and Windows scenarios covering both attack and benign activity. The benchmark includes multiple investigation tasks, with examples in the paper ranging from alert triage to identifying persistence mechanisms and data exfiltration. Models are tested under different prompt templates and at least two input representations: raw logs and a compact edge-based/preprocessed representation. Metrics include true positive rate, false positive rate, and measures of explanation correctness/quality. The main results are that model performance is mixed across tasks; several models achieve strong detection behavior on exfiltration scenarios, but across other tasks many benign scenarios are incorrectly flagged as attacks. The paper reports that larger models do not consistently outperform smaller or open-weight alternatives, and in some tasks smaller models achieve equal or better true-positive/false-positive tradeoffs. It also finds that the compact representation yields strictly better performance for some models across all benchmark tasks, while prompt changes can help some models and hurt others, indicating strong model-dependent interactions rather than universal prompting improvements.

### 🌱 Insights & Implications
A key insight is that LLMs already show some promise as analyst aids for attack investigation, especially because their explanations are often accurate when they correctly identify malicious behavior. However, their practical deployment is limited by a strong tendency toward over-suspicion and false positives, especially on benign activities with unusual object names or high-volume behavior. The study also suggests that representation engineering is a major lever: compact, semantically structured log inputs may improve quality and reduce cost, sometimes more reliably than simply using a larger model. Another important takeaway is methodological: benchmarks like AuditBench should not be used for naive head-to-head model ranking without exploring prompt and representation choices, because these factors can change relative performance substantially. Limitations include a focus on host audit logs rather than broader enterprise telemetry, finite scenario coverage despite reasonable diversity, and likely remaining gaps between benchmarked scenarios and messy real-world multi-host investigations. The paper opens questions around reducing false positives, calibrating model confidence, designing better intermediate log abstractions, and combining LLM reasoning with stronger security-specific retrieval or structured analysis pipelines.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests in software security, vulnerability detection/repair workflows, and automated security analysis because it studies whether LLMs can reliably investigate low-level execution evidence after suspicious activity occurs. Its benchmark, analysis of false positives, and findings on input representation are directly useful for building LLM-assisted systems for vulnerability triage, exploit investigation, incident response, and security tooling that must reason over system behavior rather than source code alone.

## 📄 [Context-Based Adversarial Attacks on AI Code Generators: Vulnerability Analysis and Implications](https://arxiv.org/abs/2606.10945v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Walther A. Del Orbe, John D. Hastings, Varghese Vaidyan
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper shows that seemingly benign context around code generation, like comments, docs, variable names, or examples, can strongly bias AI code generators toward insecure output. Across 2,800 trials on four models, adversarial context raised vulnerability generation from 3.5% to 37.4%, and the authors also propose a real-time defense pipeline with high detection and low false positives.

### 🏆 Key Contributions
- A systematic threat model for inference-time, context-based attacks on AI code generators.
- A controlled evaluation across CodeT5+, CodeLlama, GPT-3.5-Turbo, and GPT-4 with 2,800 trials over five CWE categories.
- A comparison of attack techniques including direct instruction, semantic priming, and example-based manipulation.
- A cross-model transferability study showing that successful attacks often generalize across architectures.
- A layered defense and validation pipeline that achieves practical detection performance with low false positives and acceptable latency.

### ⚙️ Methodology
The study treats contextual manipulation as an inference-time adversarial problem: an attacker can influence comments, documentation strings, variable names, or reference examples, but cannot access model parameters or training data. The authors evaluate four models spanning open-source and proprietary families: CodeT5+ (770M encoder-decoder), CodeLlama-7B, GPT-3.5-Turbo, and GPT-4. They test five vulnerability classes mapped to CWEs: SQL injection (CWE-89), XSS (CWE-79), hardcoded credentials (CWE-798), path traversal (CWE-22), and insecure cryptography (CWE-327). The prompt conditions are baseline (neutral), direct instruction (explicit insecure guidance in comments), semantic priming (subtle insecure cues), and example-based (vulnerable code snippets in context). The experimental matrix contains 2,800 total trials, with equal allocation across prompt conditions within each model-category cell; inference parameters were held fixed at temperature 0.7, top-p 0.9, and max_tokens 512, and API snapshots were pinned to reduce drift. Outputs were evaluated using vulnerability generation rate (VGR), attack success rate (ASR), and transferability rate (TR). Detection used a layered pipeline: an ensemble of static analyzers (Bandit, Semgrep, ESLint), regex and AST pattern matching, and 15% stratified manual review with Cohen's kappa = 0.87 for validation. Statistical analysis included chi-square tests, effect sizes, bootstrap confidence intervals, Bonferroni correction, Tukey HSD, and a one-sample t-test for transferability against a 50% baseline.

### 🧪 Experiments
The main experimental setup compares baseline prompts against adversarially modified contexts across four models and five vulnerability categories. The paper reports 2,800 controlled experiments in total: 1,000 each for CodeT5+ and CodeLlama, 500 for GPT-3.5-Turbo, and 300 for GPT-4. Per category, open-source models received 200 tests, GPT-3.5-Turbo 100, and GPT-4 60, with trials split evenly across the four prompt conditions. Baselines were neutral prompts; attacks were direct instruction, semantic priming, and example-based manipulation. Key metrics were VGR, ASR, and TR. Reported results include an overall rise in mean VGR from 3.5% at baseline to 37.4% under adversarial conditions (10.7x, chi-square = 847.3, p < 0.001, 95% CI for adversarial VGR = [35.1%, 39.7%]). Model-specific VGRs were CodeT5+ 2.0% baseline and 22.7% adversarial, CodeLlama 12.0% and 18.3%, GPT-3.5-Turbo 0.0% and 68.7%, and GPT-4 0.0% and 40.0%. Direct instruction was the strongest attack technique with mean ASR of 55%, and it reached 100% success on GPT-3.5-Turbo. Example-based attacks achieved 31.4% ASR. Cross-model transferability was high, ranging from 60% to 100% for attacks with VGR >= 25% on the origin model. The defense framework achieved an 89.1% detection rate, 0.3% false positives, and 520 ms latency.

### 🌱 Insights & Implications
The main takeaway is that insecure code generation is not just a model-specific weakness; it appears to be an architectural and contextual vulnerability that can be triggered by normal-looking surrounding text. The strong transferability result suggests that prompt hardening alone may not be enough, because malicious cues can generalize across models. The commercial models show an interesting pattern: they have zero baseline vulnerability generation in the reported setup, but their vulnerability rates rise sharply under adversarial prompts, which hints that instruction-following behavior may also increase susceptibility to malicious context. A key limitation is that the commercial models had smaller sample sizes because of API cost, so some comparisons may be less stable than the open-source results. Another open question is causal: the paper suggests RLHF could amplify responsiveness to authoritative adversarial directives, but does not isolate that effect. The defense pipeline looks practical, but its evaluation is still limited to the study's controlled benchmark and a 15% human-reviewed sample, so real-world robustness remains to be demonstrated. Overall, the paper implies that secure code-generation systems need context sanitization, adversarial prompt detection, and downstream static analysis rather than relying only on model alignment.

### 🎯 Why This Paper
This paper is highly relevant because it directly studies security failure modes in AI-assisted code generation, which intersects with software security, vulnerability detection, and secure software maintenance. It provides a concrete empirical benchmark for how contextual cues induce vulnerable code, plus a practical detection framework that can inform future work on automated vulnerability detection, secure code repair pipelines, and defenses against prompt-based supply-chain attacks.

## 📄 [Toward Secure LLM Agents: Threat Surfaces, Attacks, Defenses, and Evaluation](https://arxiv.org/abs/2606.10749v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yuchen Ling, Shengcheng Yu, Zhenyu Chen, Chunrong Fang
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This survey reframes LLM agent security as a software/system security problem rather than just prompt safety, showing that agents are vulnerable when language, tools, memory, and external actions are coupled in an autonomous loop. Synthesizing 247 papers, it finds prompt injection and tool-mediated control-flow hijacking remain dominant, while persistent state corruption and multi-agent propagation are emerging as major concerns.

### 🏆 Key Contributions
- A curated corpus of 247 LLM agent security papers built through database retrieval, LLM-assisted expansion, citation snowballing, and manual screening
- A lifecycle-based, systems-oriented security framework that spans inputs, planning, decisions, tool execution, outputs, memory, monitoring, and multi-agent coordination
- A structured synthesis of attack families showing how prompt injection, tool abuse, persistent state corruption, and cross-agent propagation relate to each other
- A defense taxonomy that compares intervention points, trust assumptions, utility costs, and composability limits
- An evaluation analysis showing that current benchmarks underrepresent long-horizon, stateful, and deployment-sensitive risks

### ⚙️ Methodology
The paper is a systematic literature survey rather than an empirical algorithmic study. The authors assemble a curated corpus of 247 papers using an auditable hybrid review pipeline: database retrieval, LLM-assisted candidate expansion with web search, citation snowballing, and manual screening. They then code the literature using a lifecycle-based, systems-oriented framework centered on the agentic loop, which treats security as arising from interactions among information flow, delegated authority, and persistent state. The framework organizes analysis across the stages of inputs, planning, decisions, tool execution, outputs, memory, monitoring, and multi-agent coordination. The survey answers four research questions: how to model LLM agent security, which threat surfaces and attacks dominate, what defenses exist and what tradeoffs they introduce, and how security claims are evaluated. The review also tracks publication trends over time, application domains, and benchmark/evaluation practices to identify methodological and deployment gaps.

### 🧪 Experiments
There are no controlled experiments or model benchmarks in the usual sense; the paper performs a qualitative and quantitative survey of the literature corpus. The effective dataset is the 247-paper corpus, with reported growth from 3 papers in 2023 to 42 in 2024, 121 in 2025, and 81 papers collected by April 27, 2026. The main quantitative findings are corpus-level and topical: prompt injection and tool-mediated control-flow hijacking dominate the field, while persistent state corruption and multi-agent propagation are increasingly important. The authors also report that existing benchmarks still emphasize immediate attack success in bounded settings and underrepresent long-horizon, stateful, and deployment-sensitive risks. Because this is a survey, there are no baseline models or standard performance metrics; instead, the paper uses coverage, categorization, and trend analysis as its main evaluative lenses.

### 🌱 Insights & Implications
The paper argues that secure LLM agents need explicit trust boundaries, principled privilege control, provenance-aware state management, and evaluation that matches realistic operational settings. A key insight is that many defenses are useful locally but weakly compositional: they may reduce one attack surface without guaranteeing end-to-end security across the agent loop. Another important point is that current research attention is skewed toward visible prompt-injection attacks, while state integrity, privilege misuse, and propagation across multi-agent workflows are likely to matter more in deployed systems. Open questions include how to build defenses that compose across tools, memory, and coordination; how to define realistic benchmarks for long-horizon agent behavior; and how to model security properties when agents operate with delegated authority and persistent state. The main limitation is that the paper synthesizes a fast-moving and heterogeneous literature, so conclusions depend on the available studies and on the authors' coding choices rather than on direct experimental comparison.

### 🎯 Why This Paper
This paper is highly relevant because it connects LLM agent security to software engineering and systems security concerns that align with secure architecture, maintainability, vulnerability detection, and repair. Its emphasis on trust boundaries, privilege control, state provenance, and evaluation realism is especially useful for research on how to detect, prevent, and repair security weaknesses in agentic software systems.

## 📄 [Assessing Automated Prompt Injection Attacks in Agentic Environments](https://arxiv.org/abs/2606.10525v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** David Hofer, Edoardo Debenedetti, Florian Tramèr
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper studies how well automated prompt-injection attacks work against tool-calling LLM agents in realistic, stateful environments. It finds that black-box TAP attacks outperform white-box GCG, but transfer is limited: attacks often work across tasks within a model, yet do not reliably transfer from smaller open models to frontier models like GPT-5.

### 🏆 Key Contributions
- Extends the AgentDojo framework to support white-box gradient-based attacks on open-source LLM agents via Hugging Face integration.
- Adapts two automated adversarial methods, GCG and TAP, from jailbreak settings to indirect prompt injection against agentic workflows.
- Evaluates attacks across 80 task pairs in four domains: workspace, banking, travel, and Slack.
- Compares single-task and task-universal attacks, showing that universal prompt injections can generalize well across unseen tasks and even out-of-distribution domains.
- Analyzes the effect of attacker-model choice for TAP and shows that attacker capability and safety tuning materially affect attack success.
- Demonstrates that attacks optimized on smaller open-source models do not transfer well to frontier models such as GPT-5.

### ⚙️ Methodology
The study builds on AgentDojo, a benchmark for stateful LLM agents that interact with external tools and untrusted content. The authors adapt two attack families to indirect prompt injection: GCG, a white-box gradient-based method that optimizes adversarial token sequences by backpropagating through the target model, and TAP, a black-box search method that uses an attacker LLM to iteratively generate, branch, prune, and refine candidate injections. They evaluate both methods in realistic agent settings where malicious instructions are embedded in tool outputs or retrieved content, and they test both single-task attacks and task-universal attacks intended to generalize across multiple tasks. The evaluation uses deterministic success checks from AgentDojo rather than LLM judges for final attack success, while also including ablations on injection structure, optimization signals, and judge reliability. The study covers multiple target models, including Qwen3-4B, Gemma3-4B, and GPT-5, and compares attack success with baseline task utility to measure how much the agent remains functional under attack.

### 🧪 Experiments
The experiments span 80 task pairs across four domains in AgentDojo: workspace, banking, travel, and Slack. The main targets are Qwen3-4B, Gemma3-4B, and GPT-5, and TAP results use GPT-5 as the attacker model in the reported main comparison. Metrics include Attack Success Rate (ASR), Success@N with N=4 independent attempts, and baseline utility without attacks. The key quantitative result reported in the paper is that TAP substantially outperforms GCG on Qwen3-4B, with 45.2% ASR versus 24.1% ASR. The paper also reports that GPT-5 is much more robust than open-weight models but still vulnerable to TAP, at about 5% ASR and 30% Success@N. Additional findings include that task-universal attacks transfer effectively to unseen tasks and out-of-distribution domains, but attacks optimized on smaller open-source models do not transfer to GPT-5. The paper further reports ablations on injection design, optimization signals, and LLM-as-judge reliability.

### 🌱 Insights & Implications
A main takeaway is that automated prompt injection is a credible threat in agentic systems, but it is not yet model-agnostic: success depends strongly on both attack method and target model. Black-box optimization appears more reliable than gradient-based optimization in realistic agent settings, likely because GCG is unstable under practical compute budgets. Another important insight is that the attacker model matters for TAP: stronger attacker LLMs can generate more effective injections, while safety-tuned attackers may refuse to produce adversarial content. The transfer results suggest that universal prompt injections are a serious concern for broad task families, but frontier models still present meaningful barriers to exploitation. A limitation is that the study focuses on specific benchmarked environments and attack adaptations, so real-world robustness may vary with tool design, memory, and system prompts. For defenses, the results imply that evaluating only single-task or single-model attacks may substantially underestimate risk.

### 🎯 Why This Paper
This paper is highly relevant to prompt-injection security, agentic vulnerability assessment, and automated red-teaming. It directly addresses whether automated attack methods from jailbreak research transfer to real tool-using agents, which is central to understanding and repairing weaknesses in LLM-based systems. Its findings on model dependence, transferability, and the relative strength of black-box versus white-box attacks are especially useful for designing better vulnerability detection, benchmarking, and future defenses for agent environments.

## 📄 [AgentCanary: A Security Evaluation Framework for Autonomous AI Agents in Real Executable Environments](https://arxiv.org/abs/2606.10484v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Peiyang Li, Songping Wang, Yi Huang, Yanhua Shi, Chenhao Zhang, Qi Li, Yueming Lyu, Caifeng Shan, Fengting Li, Chao Feng, Chuanqun Zhu, Liang Chen
📅 **Published:** 2026-06-09  🏷️ **Categories:** cs.CR
