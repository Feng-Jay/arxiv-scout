## 📄 [Token-Flow Firewall: Semantic Runtime Auditing for Persistent AI Agents](https://arxiv.org/abs/2607.08395v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Puji Wang, Yingchen Zhang, Ruqing Zhang, Jiafeng Guo, Xueqi Cheng
📅 **Published:** 2026-07-09  🏷️ **Categories:** cs.CR, cs.CL

### 💡 TL;DR
The paper introduces TokenWall, a runtime defense that treats persistent AI agent interactions as security-critical token flows and audits them before they reach memory, tools, authority state, or external outputs. By combining local rule checks, lightweight semantic auditing, selective rewriting, and only occasional stronger arbitration, it substantially reduces attacks on persistent agents while preserving benign usability and low latency.

### 🏆 Key Contributions
- Formulates persistent AI agent security as pre-execution control of semantic token flows crossing protected runtime boundaries rather than post-hoc action auditing
- Proposes TokenWall, a semantic runtime firewall that intercepts source-to-sink token flows and enforces allow, rewrite, defer, or block decisions before state mutation or tool execution
- Introduces boundary-aware structured audit records capturing payload spans, source, sink, runtime metadata, trust, permissions, persistence scope, and authorization signals
- Designs a hierarchical enforcement pipeline with deterministic prechecks, a local small-model semantic auditor, recoverable rewriting for separable unsafe content, and selective escalation for ambiguous or high-impact cases
- Demonstrates on CIK-Bench and benign workloads that full-coverage local pre-execution mediation can outperform strong runtime baselines with a strong security-utility-latency trade-off

### ⚙️ Methodology
TokenWall models agent execution as a sequence of token flows f=(x,s,t,c,b), where x is the payload segmented into semantic spans, s is the source, t is the sink, c is runtime metadata, and b is the protected boundary being crossed. The payload is decomposed into audit-relevant span categories such as secret-like content, file paths or endpoints, recipients or destinations, instruction clauses, and tool arguments. Runtime metadata includes source trust, communication channel, sink type, target component, session and persistence scope, tool permissions, action type, external destination, and trusted authorization or owner-intent signals.

The system instruments an OpenClaw-style persistent agent runtime and instantiates a token flow at the last safe point before content reaches a protected sink. Three main flow classes are audited: context flows for content entering reasoning context, memory, persistent knowledge, or skills; authority flows for rebinding identity, session, approvals, or output routing; and capability flows for tool arguments, commands, file operations, API requests, private-state access, remote modification, and outbound disclosure.

The enforcement pipeline is hierarchical. First, TokenWall performs a lightweight deterministic precheck to catch explicit violations efficiently. Second, it canonicalizes the intercepted flow into a compact structured source-sink audit record for local semantic analysis. Third, a small local model performs boundary-aware semantic auditing over the flow, deciding whether the content is safe, risky but recoverable, ambiguous, or unsafe. Fourth, if unsafe spans can be separated from benign task content, TokenWall applies semantic containment by rewriting or masking the offending spans and validating that the rewritten flow is safe to continue. Finally, only ambiguous high-risk or high-impact cases are escalated to a stronger arbiter, instead of sending all traffic to remote large-model oversight. The final enforcement actions are allow, rewrite and continue, defer to human/user confirmation, or block.

Conceptually, the design shifts enforcement from coarse action-level filtering to fine-grained information-flow mediation. This matters for persistent agents because adversarial content may first enter via low-privilege channels, then persist in memory or reusable skills, and later activate during tool use or cross-session behavior. By intercepting token flows before these transitions commit, TokenWall aims to prevent context manipulation, authority manipulation, and capability exploitation attacks while minimizing disruption to benign tasks.

### 🧪 Experiments
The evaluation is conducted on CIK-Bench, a benchmark for persistent-agent attacks, together with additional benign full-surface workloads intended to test usability under normal execution. The paper compares TokenWall against existing runtime defense strategies including rule-based safeguards and stronger runtime baselines that rely more heavily on large-model or watcher-style auditing under the same execution budget.

The main security metric is case-level attack success rate. TokenWall reduces attack success rate to 12.5%, compared with 14.7% for the strongest runtime baseline reported under the same budget. For utility, the main benign metric is executable pass rate without human confirmation; TokenWall preserves a 97.4% benign pass rate. For efficiency, the paper reports 16.9 seconds per attack case and only 0.69 seconds of additional defense latency on benign cases, which is notably lower than large-model-heavy or remote-auditing baselines. The results support the claim that full-coverage local pre-execution auditing can improve security while keeping benign throughput high and added latency modest.

### 🌱 Insights & Implications
The central insight is that persistent-agent vulnerabilities often propagate through natural-language-mediated state transfers rather than just explicit tool invocations, making token-flow interception a more appropriate security primitive than traditional output filtering. This is especially relevant to memory poisoning, delayed trigger activation, authority rebinding, and unsafe tool argument formation.

A notable implication is that software security for LLM agents may benefit from adopting information-flow control ideas, but reinterpreted at the semantic token level instead of the byte, process, or API-call level. For software maintenance and secure agent engineering, the work suggests that runtime instrumentation and boundary-aware mediation can become standard hardening layers for long-lived AI systems.

The rewrite-and-continue mechanism is also important: it suggests that defenses need not be purely blocking, and that partial sanitization may preserve task utility better than binary allow/deny decisions. This is relevant to automated repair-style research, since the system effectively performs security-preserving transformation of unsafe agent outputs before execution.

Limitations and open questions include how robust the local small-model auditor remains under adaptive adversaries, whether semantic span extraction and canonicalization miss obfuscated attacks, how the framework generalizes beyond OpenClaw-style architectures, and whether stronger guarantees can be provided for complex multimodal or code-centric tool interactions. Another open issue is how to formally evaluate false negatives for subtle long-horizon persistence attacks that only manifest after multiple benign-looking transfers.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it treats persistent AI agents as software systems with explicit runtime interfaces, state transitions, and enforceable security boundaries. It advances the researcher's work by proposing a practical runtime defense architecture for detecting and repairing unsafe semantic flows before they mutate persistent state or trigger tools, which aligns closely with interests in secure software operation, automated remediation, and protection against long-lived vulnerability propagation in AI-enabled systems.

## 📄 [Multi-Agent Firewall Architecture for Privacy Protection of Sensitive Data in Interactions with Language Models](https://arxiv.org/abs/2607.08282v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Hugo García Cuesta, Pablo Mateo Torrejón, Alfonso Sánchez-Macián
📅 **Published:** 2026-07-09  🏷️ **Categories:** cs.CR, cs.AI, cs.MA

### 💡 TL;DR
This paper presents an open-source, local-first firewall for protecting sensitive data in interactions with LLMs by transparently intercepting both browser and API traffic and analyzing prompts through a multi-agent detection pipeline. Its main result is that this hybrid deterministic-plus-LLM architecture can achieve up to 94.93% F1 while preserving privacy by keeping inspection within user- or organization-controlled environments.

### 🏆 Key Contributions
- Introduces a dual-layer interception architecture combining a browser extension with a transparent proxy to cover web-based and programmatic LLM use, including HTTP(S) and WebSocket traffic.
- Proposes a multi-agent, graph-based DLP pipeline that combines deterministic PII detectors, semantic LLM reasoning, policy enforcement, and sanitization for privacy-preserving prompt inspection.
- Adds multimodal protection for text prompts and uploaded files using OCR and vision-language analysis for screenshots and other unstructured visual content.
- Implements proprietary code leakage detection through repository indexing and fuzzy matching to detect leaked or paraphrased code snippets in prompts.
- Designs the framework to be provider-agnostic and extensible, enabling local open-weight models, proprietary APIs, or trusted cloud backends without redesigning the architecture.
- Demonstrates strong empirical performance, reporting optimal F1 scores up to 94.93% across evaluated configurations.

### ⚙️ Methodology
The system is a privacy-focused firewall placed between users and LLM services. Its architecture has two main interception layers: a browser extension for web chatbot interfaces and a transparent man-in-the-middle proxy for programmatic and browser traffic over HTTP(S) and secure WebSocket channels. This gives broad coverage without requiring modification of client applications. Once traffic is intercepted, content is passed through a modular multi-agent security pipeline organized as configurable graph stages. The detection strategy is hybrid and tiered: low-latency deterministic modules identify well-structured sensitive data such as predefined PII fields using rules, patterns, and similar classical detection logic, while more ambiguous or contextual cases are escalated to LLM-based semantic agents that reason over the full prompt intent rather than isolated tokens. The pipeline is customizable, allowing organizations to define field taxonomies, adjust stages, and configure policies for different risk levels. A privacy-preserving sanitization layer can redact or transform sensitive fields before passing content to optional third-party modules, enabling agentic workflows that minimize unnecessary exposure. Beyond plain text, the framework supports multimodal inspection of uploaded files and images through a tiered visual analysis path that uses local OCR for standard documents and vision-language models for screenshots or complex unstructured visual inputs. The system also includes a code leakage prevention component that indexes public or private Git repositories and applies fuzzy string matching to detect proprietary code even when the prompt contains modified fragments or natural-language-wrapped code. A policy engine then maps findings to enforcement actions such as warning the user or blocking the request. The backend is provider-agnostic, so local models, self-hosted services, or external LLM APIs can be swapped depending on privacy, latency, and cost constraints.

### 🧪 Experiments
From the text provided, the evaluation focuses on the effectiveness of the proposed firewall’s detection pipeline under different deployment or configuration choices, with emphasis on the trade-off among computational cost, latency, and detection depth. The paper reports that the best-performing configuration reaches an F1 score of 94.93%. The evaluation appears to assess sensitive-data leakage detection across prompt inspection scenarios and likely compares alternative internal configurations of the pipeline rather than only a single monolithic setup. The paper positions the framework against prior categories of solutions such as SDK-based guardrails, cloud inspection services, local self-hosted LLM stacks, and browser-only interception systems, but the extracted text does not include full experimental tables, dataset names, baseline implementation details, class distributions, or additional metrics such as precision, recall, latency, throughput, or ablation results. Still, the available evidence indicates that the authors measured prompt-level detection quality and demonstrated that combining deterministic detectors with semantic LLM-based analysis yields strong end-to-end DLP performance in a local-first architecture.

### 🌱 Insights & Implications
A central insight is that LLM-era data leakage prevention requires semantic understanding, not just regex or token-level NER, because risky disclosures can be paraphrased, distributed across long prompts, or embedded in broader context. The paper also highlights an important security-design tension: cloud-based prompt inspection can improve detection but creates a privacy paradox by moving sensitive content to another vendor; the proposed local-first design directly addresses this. For software and security researchers, the most interesting aspect is the decomposition of protection into interception, analysis, sanitization, and enforcement stages, which mirrors defense-in-depth principles and makes the system extensible to future threats such as prompt injection or output-side disclosure monitoring. The code leakage module is especially relevant for software engineering settings because it targets proprietary source exfiltration, a risk that conventional PII detectors miss. Likely limitations include dependency on local compute for stronger models, possible latency overhead from multimodal or LLM-based escalation, and the challenge of maintaining high recall for novel sensitive categories without overblocking. Open questions include how robust the system is against adversarial obfuscation, how well the code similarity detector handles heavily transformed code, how deployment scales across enterprise networks, and whether output-side filtering and automated repair or sanitization can be integrated more tightly.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability prevention, and secure software maintenance because it tackles practical protection against sensitive-data and proprietary-code leakage in real-world LLM-assisted development workflows. Its multi-agent, extensible architecture is also useful for researchers working on automated program repair and vulnerability detection/repair, since those workflows increasingly rely on external LLMs and need strong guardrails to prevent source-code exfiltration, unsafe prompt content, and privacy violations without giving up advanced model capability.

## 📄 [TRACE: A Two-Channel Robust Attribution Watermark via Complementary Embeddings for LLM-Agent Trajectories](https://arxiv.org/abs/2607.08400v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Zheng Gao, Xiaoyu Li, Xiaoyan Feng, Jiaojiao Jiang, Yang Song, Yulei Sui, Zhenchang Xing, Liming Zhu
📅 **Published:** 2026-07-09  🏷️ **Categories:** cs.CR, cs.AI, cs.LG

### 💡 TL;DR
TRACE is a provenance watermark for LLM-agent trajectory logs that remains detectable even when a reseller can read, edit, delete, and rewrite the logs. It combines two complementary watermark channels over agent behavior and log structure, achieving zero action-distribution distortion, strong deletion robustness, and exact invariance to rewriting while preserving task success on ToolBench and ALFWorld.

### 🏆 Key Contributions
- Formalizes the reseller threat model for LLM-agent attribution, where the adversary controls the trajectory logs used as evidence and can perform deletion and rewriting attacks.
- Proposes TRACE, a two-channel behavioral watermark that superposes a content-keyed selection channel and a position-keyed tally channel on the same trajectory.
- Introduces a distortion-free selection watermark based on a keyed exponential-race sampler over admissible actions, with proofs that the watermarked policy matches the original action distribution exactly.
- Designs a tally watermark that modulates the number of records in each decision group using only trajectory skeleton information, making it unconditionally invariant to arbitrary content rewriting.
- Provides theoretical guarantees including exact null distributions for detectors, entropy-based lower bounds on detectable signal, self-resynchronization after deletion, and a joint-erasure theorem showing that removing both channels requires damaging the trajectory service itself.
- Introduces an informed LLM-based rewriter attack that preserves plausibility while attempting to remove attribution, and evaluates TRACE against this stronger adversary and prior baselines.

### ⚙️ Methodology
TRACE embeds attribution into LLM-agent trajectories through two independent watermark channels with complementary robustness properties. The core problem is that deletion attacks break position-based synchronization, while rewriting attacks invalidate content-based keys; TRACE resolves this by using one channel keyed by local content and another keyed by trajectory position/skeleton.

The selection channel watermarks which action is chosen at each agent decision. For a decision point, the admissible action set is determined by the environment or tool interface. TRACE applies a secret-keyed distortion-free sampler, implemented via an exponential-race mechanism derived from prior unbiased text watermarking ideas, but adapted from token sampling to action selection. The key is computed from local preceding trajectory content so that if earlier steps are deleted, synchronization recovers at the next surviving decision whose local context is available again. Because the sampler preserves the exact original action distribution, TRACE does not bias policy behavior or degrade success by shifting action probabilities. Detection on this channel is statistical: the detector recomputes keyed scores over observed decisions and aggregates them into a z-statistic. The paper proves that the score signal depends on decision entropy: deterministic or near-deterministic steps contribute little or nothing, while higher-entropy decisions contribute more. This yields a principled need to pool evidence across trajectories when individual tasks have limited decision entropy.

The tally channel watermarks the structure of the log rather than action identity. TRACE groups records by decision and modulates how many records appear in each group, for example by appending a context-neutral redundant record when dictated by a position-derived key. The key and the watermark decision depend only on the trajectory skeleton, such as group position, not on textual content. As a result, any rewriting that paraphrases observations, renames tools, or edits arguments while preserving the same sequence structure leaves the tally watermark unchanged. This gives structural robustness that is unconditional rather than empirical.

The two channels are superposed on disjoint carriers: action identity for selection, and record count per decision group for tally. They also use independent keys. This complementary design means a reseller must attack both semantic decision content and trajectory structure to erase provenance completely. The paper further proves that joint erasure is costly: removing both channels requires skeleton edits and, unless the attacker targets exact realized scores, modification of a constant fraction of groups, thereby degrading the utility and faithfulness of the trajectory logs the reseller is supposed to provide.

### 🧪 Experiments
The evaluation uses ToolBench and ALFWorld, two LLM-agent benchmarks with different trajectory characteristics. ToolBench has relatively few effective decision groups per task, while ALFWorld provides longer-horizon trajectories with roughly a few dozen decision steps, making it useful for testing signal accumulation. TRACE is compared against prior single-channel agent watermark baselines, including red-green style biased watermarking and multi-bit/distribution-preserving methods discussed in the paper. The evaluation also includes the unwatermarked agent as a quality reference.

Attacks include step deletion, log rewriting, and combined attacks. The rewriting evaluation is notably stronger than generic perturbation tests because the paper introduces an informed LLM rewriter that has access to the true chosen action and rewrites the trajectory in a plausibility-preserving way. Detection is measured using false-positive-rate-controlled attribution tests, especially at 1% FPR, and using z-scores for the selection channel.

Main results: TRACE matches the unwatermarked agent's task success rate on both ToolBench and ALFWorld, while a biased red-green baseline loses up to 8.1 percentage points in success. On long-horizon trajectories, the selection channel reaches detection scores near z = 100. Under deletion attacks, the selection channel remains detectable even after 70% of steps are deleted, showing practical self-resynchronization. Under rewriting attacks, the tally channel is exactly unchanged regardless of LLM rewriting strength, whereas a single rewriting pass can erase both compared baselines. At 1% FPR, TRACE still attributes under either single-axis attack (deletion alone or rewriting alone), and its main weakness appears in the combined-attack regime where both channels are targeted simultaneously; however, the paper argues that this regime already implies substantial corruption of the resold service/log quality.

### 🌱 Insights & Implications
The paper’s main insight is that robustness to deletion and robustness to rewriting are fundamentally at odds if a watermark uses only one keying strategy. Content-derived keys enable re-synchronization after deletions, but content can be rewritten; position-derived keys survive rewriting, but deletions desynchronize them. TRACE’s two-channel design is therefore not just an engineering combination but a structural response to incompatible attack surfaces.

A second important insight is that distortion-free watermarking is not free in statistical power. Even though the action distribution is unchanged, detectability is still paid for by decision entropy. This matters for agent settings, where many choices are deterministic or low-entropy because only one tool/action is valid. For research on agent provenance, this suggests that benchmark design, trajectory length, and action-space entropy are central determinants of watermark strength.

The work also reframes attribution as a systems and governance problem rather than only a generative-model problem: the relevant artifact is the trajectory log, and the key adversary may be a reseller or integrator who legitimately stores and edits that artifact. This is especially relevant to enterprise agents, API marketplaces, and safety-critical workflows.

Likely limitations and open questions include: dependence on having structured trajectory logs with identifiable decision groups; reduced signal on short or highly deterministic tasks; the practical acceptability of adding redundant records for the tally channel in production systems; and how the method behaves when resellers perform more aggressive semantic transformations that alter workflow structure rather than merely rewrite content. Another open direction is extending the approach from attribution to downstream tasks like forensic auditing, model substitution detection, and training-data provenance for agent traces.

### 🎯 Why This Paper
This paper is highly relevant because it addresses provenance and robustness for LLM-agent behavior under an adversarial log-manipulation setting, which intersects software security, vulnerability-response automation, and trustworthy software maintenance workflows. For a researcher in software engineering, automated repair, and security, TRACE provides a rigorous behavioral watermarking framework for proving which agent produced a sequence of tool actions, a capability that matters when agents execute code, modify systems, triage incidents, or apply repairs and the logs may be manipulated by intermediaries. It advances the field by moving watermarking from token text to structured action traces, supplying both formal guarantees and realistic attack evaluation, and opening a path toward accountable agentic systems used in secure development and operations.

## 📄 [Mini-Programs, Mega-Problems: Unveiling OAuth-based Authentication Misuses in Mini-Programs via Dynamic Analysis](https://arxiv.org/abs/2607.08232v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Zidong Zhang, Zhentao Xie, Lingyun Ying, Qinsheng Hou, Yacong Gu, Wenrui Diao, Jianliang Wu
📅 **Published:** 2026-07-09  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper shows that many mini-programs on platforms like WeChat and Baidu misuse platform-specific OAuth-based authentication in ways that let attackers impersonate users or access sensitive data. The authors build MiniAuth, a large-scale dynamic analysis system, and use it to find 1,834 real-world misuse cases plus a platform-level cryptographic flaw in Baidu.

### 🏆 Key Contributions
- Systematizes three runtime OAuth-based authentication misuse patterns in mini-programs that were missed by prior static-analysis-focused work
- Introduces MiniAuth, the first dynamic analysis framework for automatically locating mini-program login pages, executing authentication flows, and detecting runtime OBA misuses at scale
- Demonstrates that dynamic analysis can handle obfuscated mini-programs and uncover vulnerabilities in client-server interaction logic that static approaches cannot observe
- Performs a large-scale empirical study over 44,273 WeChat and 2,721 Baidu mini-programs, finding 1,834 misuse cases across 1,688 programs, including 619 with multiple flaws
- Identifies a Baidu platform cryptographic design flaw where IV construction leaks part of the session key, enabling brute-force recovery of session keys
- Provides responsible disclosures with confirmed impact, including CNVD/CNNVD acknowledgments, and distills guidance for both developers and platform operators

### ⚙️ Methodology
The paper studies mini-program OAuth-based authentication (OBA), a platform-specific login flow embedded inside super apps rather than standard web OAuth/OIDC. The authors first define a threat model where the attacker controls their own device, can intercept and modify outbound traffic from their own mini-program instance, can decompile mini-program code, but cannot compromise the victim device or platform sandbox. They then derive a taxonomy of runtime OBA misuses centered on trust-boundary violations between the mini-program frontend and the developer backend. The core technical contribution is MiniAuth, a dynamic analysis framework tailored to mini-program ecosystems. MiniAuth automatically identifies the page and UI path corresponding to OBA login, triggers the relevant workflow, and monitors runtime behavior during the authentication process. Instead of relying on source-level semantics alone, it combines hybrid automation and runtime inspection to observe how authentication artifacts such as auth codes, identifiers, encrypted payloads, IVs, and session-related values are generated, transmitted, and validated. This design makes the approach effective even for obfuscated mini-programs. The framework focuses on detecting misuse patterns such as exposed credentials that enable client-side identity forgery, authentication decisions based on static or plaintext identifiers, and incorrect handling of encrypted sensitive-data retrieval flows. The analysis also inspects platform API behavior itself, which enables the authors to uncover a Baidu cryptographic flaw: the initialization vector used in encrypted payload handling reuses part of the session key rather than being random, reducing key entropy and making brute-force recovery feasible. Overall, the methodology is dynamic, protocol-aware, and ecosystem-specific, targeting logic flaws in real authentication executions rather than only syntactic code patterns.

### 🧪 Experiments
The evaluation analyzes 46,994 mini-programs in total: 44,273 from WeChat and 2,721 from Baidu, the two largest ecosystems studied. The paper positions MiniAuth against prior static-code-based security analyses by emphasizing scenarios they miss, especially obfuscated applications and vulnerabilities that only manifest through runtime client-server interactions. The main outcome is the discovery of 1,834 OBA misuse cases affecting 1,688 mini-programs, with 619 programs containing multiple vulnerabilities. The detected issues include client-side identity forgery through exposed credentials and authentication bypass caused by use of static or plaintext identifiers instead of securely validated session-bound data. The authors further report that impacted applications include sensitive domains such as government and healthcare, where exploitation could expose national ID information and health records. Cross-platform inspection suggests that similar insecure business logic also appears in other ecosystems such as Alipay and TikTok, indicating broader generality beyond the two fully analyzed platforms. In addition to third-party app flaws, the study finds a Baidu API design weakness that allows brute-forcing session keys due to partial leakage through IV construction. The practical significance of the findings is supported by responsible disclosure outcomes, including acknowledgments and 11 CNVD/CNNVD IDs. Although the excerpt does not provide detailed precision/recall metrics or exhaustive baseline tables, the paper’s evaluation demonstrates scalability, real-world prevalence, and exploitability of the discovered authentication flaws.

### 🌱 Insights & Implications
A key insight is that mini-program authentication security is shaped less by classic OAuth redirect misuse and more by platform-specific runtime logic around authcode exchange, identifier trust, and session-key-mediated sensitive-data access. This makes static analysis insufficient: the most serious flaws emerge during execution, depend on backend validation logic, and can persist even when code is obfuscated. The work also highlights that mini-program OBA differs fundamentally from standard OAuth/OIDC; developers may incorrectly assume that OpenID/UnionID-like values behave like signed identity tokens, when in reality they are raw identifiers without OIDC-style guarantees. Another important implication is that platform design matters as much as third-party implementation: the Baidu IV/session-key flaw shows that insecure vendor API design can undermine otherwise well-intentioned applications. For software security and vulnerability research, the paper suggests that dynamic, workflow-aware analysis is necessary for ecosystems with proprietary authentication APIs and heavy code obfuscation. Limitations and open questions likely include coverage challenges in automatically exercising all login flows, the dependence on controlled runtime instrumentation, and the need for richer backend-side observability to distinguish all variants of logic misuse. Future research could extend this approach to automated repair guidance, platform-enforced authentication templates, or differential testing across super-app ecosystems.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and vulnerability repair because it targets authentication logic flaws in a large real-world application ecosystem and proposes a scalable dynamic analysis technique to uncover them. It is especially valuable for work in automated program repair and secure software maintenance: the paper not only identifies recurring misuse patterns that could be codified into repair templates or detection rules, but also shows why runtime-aware analysis is essential when source code is obfuscated or when the vulnerability lies in client-server protocol integration rather than local code syntax.

## 📄 [From Prompts to Contracts: Harness Engineering for Auditable Enterprise LLM Agents](https://arxiv.org/abs/2607.08028v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Joongho Ahn, Moonsoo Kim
📅 **Published:** 2026-07-09  🏷️ **Categories:** cs.AI, cs.CL, cs.SE

### 💡 TL;DR
This paper argues that enterprise LLM systems should move critical behavior out of prompts and into a code-owned harness that enforces source boundaries, entity routing, output contracts, and auditable traces. In a corporate-investment briefing case study, the harness preserved safety and traceability across model swaps and outperformed prompt-only or external-guardrail alternatives by maintaining full utility while blocking policy violations.

### 🏆 Key Contributions
- Introduces a harness-engineering pattern for converting prompt-dominant LLM prototypes into auditable enterprise applications with explicit contracts and versioned validation artifacts
- Proposes a source-to-claim knowledge pipeline that separates raw sources, evidence records, runtime-eligible claims, maintained wiki context, and reader-facing answers
- Defines a replaceable composition boundary that isolates deterministic code-owned controls from model-dependent language realization, enabling model substitution without losing core guarantees
- Develops a system-level validation framework covering source grounding, entity routing, trace completeness, output hygiene, recommendation-language constraints, latency, and live model behavior
- Provides empirical evidence that code-owned enforcement is load-bearing: prompt instructions alone and bolt-on external guardrails cannot match the harness’s joint safety-and-utility performance

### ⚙️ Methodology
The paper presents a contract-oriented architecture for enterprise LLM agents centered on a code-owned harness. The key design move is to relocate deterministic application behavior from prompts into explicit software artifacts: source manifests define allowable documents, source-backed claims define what factual statements are eligible for runtime use, routing metadata maps user questions to the correct business entity, answer contracts specify the visible response structure and constraints, and traces record the assembly process for audit and replay. The architecture separates authoritative content from convenience context: maintained wiki pages may provide compact summaries, but only manifest-approved, provenance-linked claims are treated as authoritative inputs for final answers. The model sits behind a replaceable composition boundary, meaning the LLM is only responsible for composing phrasing from already-governed inputs, while the harness enforces source grounding, output hygiene, and fallback behavior. Validation is built into the runtime rather than added afterward: the harness checks whether responses stay within approved sources, route to the correct entity, avoid leaking internal traces, and respect constraints such as prohibition of recommendation language. The study also includes fault injection to deliberately violate contracts and test whether validators catch the failures. To assess whether guarantees are truly code-owned rather than prompt-induced, the authors perform an ablation that holds the model fixed while varying the enforcement layer, comparing the full harness to prompt-only control and to an external guardrail layer.

### 🧪 Experiments
The evaluation is conducted on a bounded public-data slice covering five Korean corporate groups, 25 listed companies, and 113 source-backed runtime claims, derived from a reconstructed investment-briefing agent. The authors organize experiments around three research questions: RQ1 tests whether the harness preserves contracts on fixed validation scenarios; RQ2 tests whether those guarantees survive model substitution across three hosted LLMs; RQ3 tests whether guarantees are genuinely enforced by code rather than prompts. For RQ1, fixed-scenario validation and fault-injection controls show that the harness preserves source grounding, entity routing, trace behavior, output hygiene, and recommendation-language constraints, and that validators detect deliberately broken contracts. For RQ2, the model-substitution experiment runs 270 composition-boundary executions across three hosted models; all harness-enforced checks pass, and failures are confined to the model-composed side where they are caught and logged by the system. For RQ3, the enforcement-layer ablation compares the full harness, prompt-only instructions, and a bolt-on external guardrail. Prompt-only control allows recommendation-language violations and internal-trace leakage to reach users, while the harness blocks them entirely. The external guardrail also blocks violations but over-refuses, reducing utility to 88/120, whereas the harness preserves full utility at 120/120. The main empirical result is that only code-owned enforcement simultaneously preserves safety contracts and user-facing utility.

### 🌱 Insights & Implications
A central insight is that product-critical guarantees for enterprise LLM systems are not reliably expressible as prompts; they must be embodied in software contracts, validation logic, and auditable artifacts. The paper usefully distinguishes between model-composed behavior and code-enforced guarantees, showing that model substitution is feasible when the composition boundary is narrow and deterministic controls dominate. For software engineering and assurance, this suggests a practical path toward reproducible, inspectable LLM applications: treat prompts as replaceable realization components, not as the primary repository of policy. The work also highlights a limitation of external guardrails: they can improve safety but may harm usefulness through over-refusal, whereas integrated harness logic can apply domain-specific constraints more selectively. Limitations include the bounded domain, relatively small and curated source/claim set, focus on one application class, and evaluation of contract preservation rather than downstream decision quality. Open questions include how the approach scales to larger dynamic corpora, multi-step tool-using agents, continuously changing enterprise knowledge bases, and domains where claims are less discretely representable. Another important research direction is combining this contract-based harness with automated claim extraction, vulnerability-style validation, or repair loops for broken traces and unsupported assertions.

### 💡 TL;DR
ProjAgent improves repository-level code generation by explicitly retrieving procedurally similar functions—code that follows similar implementation steps even when names, domains, and APIs differ—and combining that signal with standard semantic retrieval and static-analysis-guided repair. ### 🏆 Key Contributions
- Introduces procedural similarity as a new retrieval signal for repository-level code generation, beyond lexical, semantic, and structural similarity.
- Uses LLM hidden-state projection similarity to identify functions that implement similar computational procedures across different modules and domains.
- Proposes ProjAgent, an agentic retrieval-and-generation workflow that decomposes a target function into intermediate reasoning steps, retrieves procedural context for each step, and merges it with conventional semantic context.

### ⚙️ Methodology
ProjAgent is a repository-level code generation system centered on the idea that useful context is not limited to semantically similar or directly dependent code, but also includes repository functions that carry out similar procedures such as validation, normalization, transformation, or guard-clause logic. The workflow begins with the target function signature and surrounding task context. ProjAgent asks an LLM to decompose the target implementation into intermediate reasoning or implementation steps, effectively producing a procedural sketch of what the missing code should do. These steps serve as retrieval queries.

For procedural retrieval, ProjAgent uses an agentic workflow rather than a single-shot nearest-neighbor search. It first identifies a small candidate set of repository functions that may be procedurally related to one or more target steps, then validates or filters them through the agent’s reasoning process. To support this search, functions are represented using hidden-state projection features extracted from an LLM, motivated by prior evidence that intermediate representations encode behavioral and functional properties of code. Similarity in this projected representation space is used as a proxy for procedural similarity, allowing the system to retrieve code that follows analogous reasoning patterns even when tokens, identifiers, types, and domain vocabulary differ substantially.

The retrieved procedural context is not used alone. ProjAgent complements it with conventional semantic and lexical retrieval to capture repository-specific APIs, symbols, helper routines, and local conventions that procedural analogs may not expose. The final context package therefore combines: (1) procedural examples that show how analogous logic is implemented elsewhere in the repository, and (2) semantically relevant repository snippets that anchor the generation to project-specific dependencies and naming.

After context assembly, an LLM generates the target function implementation. ProjAgent then applies a conservative static-analysis feedback loop: the generated code is compiled or analyzed, diagnostics are collected, and the model is prompted to repair only issues supported by explicit feedback, reducing speculative rewrites. This repair cycle is iterative and aims to improve executability and correctness while preserving valid portions of the original generation. The overall design couples planning-like decomposition, behavior-oriented retrieval, repository-aware context integration, and feedback-driven refinement.

### 🧪 Experiments
The paper evaluates ProjAgent on REPOCOD, a benchmark for repository-level code generation where models must complete target functions using cross-file repository context. The experimental design is organized around three research questions: overall effectiveness versus prior systems, the effectiveness of hidden-state projection similarity for finding procedurally related context, and ablation of the different retrieval components. The ablation studies analyze the contribution of procedural retrieval relative to semantic/lexical retrieval and the static-analysis repair stage, showing that procedural context provides complementary gains rather than simply duplicating standard retrieval signals. The paper also examines retrieval quality by studying whether projected hidden-state similarity can surface functions with similar implementation behavior despite low lexical overlap and no direct structural dependency, using examples like input validation and error-raising patterns across unrelated modules.

### 🌱 Insights & Implications
The key insight is that repository-level generation often fails not because relevant APIs are unavailable, but because existing retrievers miss implementation patterns that are behaviorally useful yet textually dissimilar. Procedural similarity appears to fill this gap by surfacing examples of how analogous logic is implemented elsewhere in the codebase, which can guide the model through difficult steps such as validation, conversions, and guard conditions.

For software engineering research, this suggests that retrieval for code generation should be treated as a multi-view problem: semantic similarity helps find local dependencies and APIs, while procedural similarity helps find implementation strategies. This is especially relevant to maintenance tasks where developers often adapt existing logic patterns across modules. The agentic decomposition step is also notable because it turns a monolithic generation problem into step-level retrieval, potentially improving interpretability and controllability.

Another open issue is generalization beyond REPOCOD and beyond function synthesis into tasks such as bug fixing, patch generation, or security remediation. Since the method emphasizes procedural analogies, an interesting future direction is whether the same retrieval signal could help find historical bug-fix patterns, secure coding idioms, or vulnerability mitigation templates across repositories. The static-analysis loop is conservative, which is good for stability, but it may miss deeper semantic defects that require tests or symbolic reasoning rather than compiler diagnostics alone.

The authors study internal type representations using diagnostic linear probes over residual-stream activations from two decoder-only code models: SantaCoder-1.1B and CodeLlama-7B. The main analysis uses selectivity, defined as task accuracy minus accuracy of an identical probe trained on random control labels, to distinguish recoverable structure from probe memorization. Cross-language generalization is tested by training probes on one language partition and evaluating zero-shot on the others; robustness is measured as the drop in selectivity from standard to adversarial data.

The evaluation covers SantaCoder-1.1B and CodeLlama-7B across all layers, using 4-fold cross-validation with a held-out validation split, and reports peak selectivity and peak layer per setting. ## 📄 [TTHE: Test-Time Harness Evolution](https://arxiv.org/abs/2607.08124v1)  `60% relevant`  `🌐 Full Text`

📅 **Published:** 2026-07-09  🏷️ **Categories:** cs.SE, cs.LG

### 💡 TL;DR
This paper proposes Test-Time Harness Evolution (TTHE), a framework that adapts an LLM agent during evaluation by rewriting its executable harness—prompts, control flow, tool use, verification, and recovery logic—using only unlabeled execution traces rather than model fine-tuning or gold labels. Across five execution-grounded domains, TTHE consistently improves fixed ReAct-style agents, but its effectiveness is limited by the reliability of execution-derived proxy signals and the judge’s ability to select genuinely better harnesses.

### 🏆 Key Contributions
- Introduces test-time harness evolution as a new adaptation paradigm where the persistent object being updated is the executable agent harness rather than model weights, prompts fixed in advance, or per-query responses.
- Designs a population-based generate-and-judge loop in which a frozen LLM plays multiple roles—solver, proposer, and judge—to iteratively rewrite and select improved harness code during evaluation using only unlabeled execution traces.
- Shows that harness-level adaptation can produce persistent, inspectable behavioral changes such as better grounding, verification, and recovery strategies, instead of one-off retries or textual reflections.
- Demonstrates gains over fixed ReAct-style baselines across diverse execution-grounded tasks: text-to-SQL, competitive programming, software engineering, data-science coding, and agentic tool-use.
- Provides diagnostic analyses on search budget, batch size, cross-model behavior, and trace-level failure audits, highlighting limited candidate coverage and judge selection regret as major bottlenecks.

### ⚙️ Methodology
TTHE treats the agent harness as the mutable state of test-time adaptation. A harness is an executable control program around a frozen LLM that constructs prompts and context, invokes tools, executes code or queries, performs intermediate checks, and handles failures. Instead of fixing this harness before deployment, TTHE updates it during evaluation.

At test time, the method processes unlabeled test inputs in batches. For each batch, it maintains a population of candidate harnesses. Each candidate harness is executed on the current batch under instrumentation, producing execution traces that may include model outputs, tool calls, database feedback, runtime errors, test outcomes, intermediate artifacts, and recovery trajectories. These traces serve as the only adaptation signal; task gold labels are hidden during the adaptation loop and used only afterward for reporting.

A proposer module, implemented using the same frozen LLM in a different role, reads existing harness code together with execution traces and proxy-indicated weaknesses, then generates edited harness variants. Because the harness is code, these edits can alter prompt templates, add or remove tool calls, introduce verification or grounding steps, add multi-sample generation, insert deterministic checks, or revise retry and repair logic. TTHE therefore searches a much richer space than prompt-only adaptation or bounded hyperparameter tuning.

A judge module, again another harnessed role of the same frozen LLM, compares candidate harnesses based on execution-derived proxy signals and selects one to commit. The committed harness persists and becomes the starting point for subsequent test batches, so improvements accumulate over the evaluation stream rather than being isolated to a single query.

The overall loop is thus: execute current harness population on unlabeled batch, collect traces, propose code edits, re-execute candidates, judge candidates using proxy evidence, commit one harness, and continue. No model weights are updated, no separate adaptation model is trained, and no gold supervision is used within the loop. The framework operationalizes unsupervised test-time adaptation at the program level rather than the parameter level.

### 🧪 Experiments
The paper evaluates TTHE on five execution-grounded domains: text-to-SQL, competitive programming, software engineering, data-science coding, and agentic tool-use. The baseline is a fixed ReAct-style harness around the same frozen LLM, allowing the comparison to isolate the value of harness evolution rather than changes in the underlying model.

The experimental protocol runs adaptation during evaluation itself. Inputs are processed in batches, candidate harnesses are evolved using only execution traces from those test batches, and gold labels are withheld until after harness selection for measurement. The study emphasizes settings where execution feedback is available but incomplete, such as SQL execution results, visible tests, runtime errors, and tool outcomes.

The paper reports that TTHE improves over the fixed baseline across all evaluated domains and that the gains are persistent because the selected harness governs future examples. The authors also analyze ablations over search budget and batch size and perform cross-model studies, showing that performance is not monotonic with more search and that adaptation quality depends strongly on the quality of candidate generation and selection. Trace audits reveal two main failure modes: limited candidate coverage, where good behavioral changes are never proposed, and selection regret, where useful candidates are generated but the judge fails to commit them. The excerpt provided does not include the detailed numeric tables, exact benchmark names for every domain slice, or full metric values, but the qualitative outcome is consistent improvement over fixed ReAct-style harnesses with nontrivial variance due to proxy unreliability.

### 🌱 Insights & Implications
The paper’s main conceptual insight is that an LLM agent’s surrounding program is a powerful and underexplored adaptation target. For software and tool-using agents, many failures stem not from missing world knowledge in the base model but from inadequate orchestration—poor context construction, weak verification, brittle recovery, or insufficient grounding. TTHE shows that such issues can be improved online by evolving executable scaffolding rather than changing model parameters.

A second important insight is that test-time adaptation need not rely on self-supervised losses or scalar rewards. Native execution traces from compilers, tests, databases, and tools are rich enough to drive useful adaptation. This is especially relevant for software engineering tasks where intermediate artifacts are executable and instrumentable.

The paper also surfaces a central limitation: execution proxies are imperfect. A SQL query can execute yet be semantically wrong; a patch can pass visible tests yet fail hidden ones; a tool-use trajectory can look plausible while violating the task objective. Because of this, the judge may commit harnesses that optimize operational plausibility rather than true correctness. The authors identify judge reliability and proxy quality as core open problems.

Another implication is that more search is not automatically better. Since candidate quality and selection are noisy, larger budgets can increase the chance of overfitting to proxies or choosing superficially strong but incorrect harnesses. This matters for practical deployment where compute is constrained and reliability is critical.

For related research, the work bridges automated workflow optimization, self-debugging, and online adaptation, but differs by making the persistent executable policy the adaptation unit. Open questions include how to design stronger proxy signals, how to safely constrain harness edits, how to reason about catastrophic degradation over long streams, and how to combine harness evolution with verifier learning or formal specifications.
