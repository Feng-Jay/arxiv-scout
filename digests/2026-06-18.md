## 📄 [OpenAnt: LLM-Powered Vulnerability Discovery Through Code Decomposition, Adversarial Verification, and Dynamic Testing](https://arxiv.org/abs/2606.19149v1)  `100% relevant`  `🌐 Full Text`

👤 **Authors:** Nahum Korda, Gadi Evron
📅 **Published:** 2026-06-17  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
OpenAnt is an open-source, multi-stage vulnerability discovery pipeline that combines static reachability analysis, LLM-based semantic reasoning, adversarial exploitability checking, and sandboxed dynamic validation to make repository-scale security analysis more practical. Its main result is that aggressive code decomposition and verification can cut the analysis surface by roughly 95-97% while still surfacing previously unknown vulnerabilities in real projects such as OpenSSL, WordPress, and Flowise with fewer false positives than naive LLM or static-only analysis.

### 🏆 Key Contributions
- A repository-scale decomposition pipeline that extracts function-level analysis units, resolves transitive dependencies into self-contained contexts, and filters them by reachability from external entry points to drastically reduce the code that must be analyzed.
- An LLM-based exposure and vulnerability analysis workflow that first classifies reachable code by security relevance, then applies language-agnostic semantic prompts to identify risky input-to-sink behaviors across multiple programming languages.
- An adversarial verification stage that simulates a constrained remote attacker, forcing the model to test multiple exploit strategies under realistic capability limits and reject findings that are not practically exploitable.
- A dynamic verification mechanism that automatically generates vulnerability-specific exploit environments, executes them in ephemeral sandboxed Docker containers, and uses runtime evidence to validate or refute candidate findings.
- An open-source end-to-end system demonstrating that closed-loop pipelines integrating static analysis, LLM reasoning, and exploit validation can find new vulnerabilities in widely used open-source software at manageable cost.

### ⚙️ Methodology
OpenAnt uses a six-stage pipeline. Stage 1 parses repositories with language-specific AST tooling and normalizes results into a shared intermediate representation containing function bodies, signatures, source locations, and bidirectional call-graph edges. Stage 2 constructs self-contained function-level analysis units by taking a target function and inlining transitively resolved dependencies up to a configurable depth (default three). It also identifies externally exposed entry points using decorators, framework conventions, and input-parameter patterns, then performs reachability analysis from those entry points to retain only attack-surface-relevant functions. Stage 3 applies an LLM agent with tool-assisted code navigation to classify each reachable unit as exploitable, vulnerable-internal, security control, or neutral, so only externally exposed and security-relevant units move forward. Stage 4 performs vulnerability detection with a language-agnostic prompt that asks what the code does, where input originates, and what security risk follows, producing one of five labels: vulnerable, bypassable, inconclusive, protected, or safe. Stage 5 adversarially verifies all findings by simulating a realistic remote attacker limited to browser-based interaction, no server access, no admin privileges, and no file modification. The model must explore multiple attack paths, document feasibility step by step, and satisfy a victim-impact requirement so self-harm scenarios are excluded. Stage 6 dynamically validates plausible findings by automatically generating exploit code, execution infrastructure, and monitoring logic, running them in disposable sandboxed containers, and using concrete runtime behavior as evidence of exploitability. The system currently supports Python, JavaScript/TypeScript, Go, C/C++, Ruby, and PHP.

### 🧪 Experiments
The evaluation is conducted on widely used open-source projects including OpenSSL, WordPress, Flowise, and also examples such as Grafana for analysis-surface measurements. The setup measures how much the staged pipeline reduces the amount of code sent to LLM analysis and whether the remaining findings correspond to real vulnerabilities. Key reported numbers show that reachability filtering reduces OpenSSL from 15,232 extracted functions to 390 reachable units, about a 97% reduction, and reduces Grafana from 18,500 functions to 994 reachable units, about a 94.6% reduction. In OpenSSL, the exposure-classification stage further narrows 390 reachable units to 49 externally exposed units, an additional 87% reduction before deep vulnerability analysis. The paper reports that the full pipeline identified previously unknown vulnerabilities in real projects while keeping analysis cost manageable and substantially reducing false positives relative to less constrained static or LLM-only analysis. Exact aggregate precision/recall-style metrics are not provided in the extracted text, but the empirical emphasis is on attack-surface reduction, practical exploit verification, and real-world vulnerability discovery rather than benchmark-only classification accuracy.

### 🌱 Insights & Implications
The paper’s central insight is that LLMs are most useful for security analysis when embedded in a tightly scoped, verification-heavy pipeline rather than asked to reason over entire repositories. Reachability-based decomposition directly addresses context-window and lost-in-the-middle issues, suggesting that structural program analysis is a necessary complement to LLM reasoning for large codebases. The adversarial verification stage is important because many candidate findings fail under realistic attacker constraints; this reframes vulnerability detection as exploitability reasoning rather than pattern spotting. The dynamic verification step further reduces false positives by demanding executable evidence, which is especially relevant for security engineering workflows where analyst trust matters. Limitations and open questions include the dependence on accurate entry-point identification and call-graph construction, possible missed vulnerabilities in code not marked reachable, the cost and reliability of repeated LLM reasoning, incomplete quantitative reporting in the extracted text, and uncertainty about coverage for complex multi-step, stateful, or environment-specific vulnerabilities. For related research, the work implies that future automated vulnerability discovery and repair systems may benefit from closed-loop designs that combine static narrowing, semantic diagnosis, adversarial reasoning, and executable validation.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, and automated repair because it tackles the practical bottlenecks of applying LLMs to large real-world codebases: context management, false positives, and exploit validation. It advances the state of the art by showing how to operationalize LLM-based semantic analysis inside a scalable, verification-oriented pipeline, which is directly useful for researchers building security analysis, vulnerability triage, program repair, or end-to-end detect-and-fix systems.

## 📄 [Code-Augur: Agentic Vulnerability Detection via Specification Inference](https://arxiv.org/abs/2606.18619v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Zhengxiong Luo, Mehtab Zafar, Dylan Wolff, Abhik Roychoudhury
📅 **Published:** 2026-06-17  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
Code-Augur introduces a security-specification-first framework for agentic vulnerability detection: instead of keeping an LLM agent’s security assumptions implicit, it turns them into executable in-source invariants and uses guided fuzzing to falsify and refine them. This reason-falsify-refine loop substantially improves bug-finding over prior agentic systems and led to 22 previously unknown vulnerabilities in real open-source projects, 16 already fixed or confirmed.

### 🏆 Key Contributions
- Formulates a security-specification-first paradigm for agentic vulnerability detection that makes an agent’s tacit security assumptions explicit, inspectable, and reusable as local security invariants.
- Implements Code-Augur, a harness where the agent analyzes components, emits in-source assertions encoding inferred invariants, and pairs them with guided fuzzing to validate or falsify those assumptions.
- Introduces an iterative reason-falsify-refine loop: assertion violations either expose real vulnerabilities or reveal flawed specifications that trigger re-analysis and invariant refinement.
- Demonstrates that explicit inferred specifications improve vulnerability detection performance over strong agentic baselines such as Claude Code and Atlantis across benchmark datasets and frontier LLM backbones.
- Shows practical impact through discovery of 22 new vulnerabilities in widely used open-source projects, with 16 already fixed or confirmed and some earning bug bounties.
- Argues that inferred invariants are durable security artifacts that persist beyond a single audit, helping identify bug families, incomplete fixes, and future regressions.

### ⚙️ Methodology
Code-Augur is built around the idea that vulnerability detection should begin by inferring security specifications rather than only searching for crashes. Given a codebase and a threat model describing security boundaries and objectives, the system analyzes each component using an LLM-based agent. For each code region, the agent either flags a vulnerability directly or, if it judges the code secure, extracts the local assumptions that justify that judgment. These assumptions are expressed as local invariants at strategically chosen program points and committed into the source as executable assertions. The invariants act as explicit security specifications, capturing conditions that must hold for global security properties to be preserved.

The system then couples these inferred assertions with a guided fuzzing phase. Instead of relying only on end-to-end crash discovery, the fuzzer is directed toward violating the inserted invariants, which are typically shallower semantic targets than the ultimate failure symptoms. This makes dynamic exploration more tractable, especially for deep, multi-stage vulnerabilities whose root cause and crash site are temporally or structurally separated. When an assertion is violated at runtime, Code-Augur distinguishes between two outcomes. If the violation is security-relevant, it exposes a real defect and yields a candidate vulnerability. If the violation is benign, it indicates that the agent’s inferred specification was too strong, incomplete, or incorrect. In that case, the agent revisits the corresponding code, reassesses its reasoning, and refines the invariant. The updated invariant is then revalidated through further fuzzing.

This closes a reason-falsify-refine loop between symbolic/semantic agentic reasoning and empirical runtime behavior. The agent provides semantic understanding and initial hypothesis generation, while fuzzing provides broader edge-case exploration and a check against hallucinated or incomplete assumptions. The overall goal is not only to find bugs but also to ground the agent’s understanding in observable program behavior and to leave behind durable, reusable security knowledge in the form of source-level assertions.

### 🧪 Experiments
The paper evaluates Code-Augur on two state-of-the-art vulnerability detection benchmarks derived from DARPA’s AI Cyber Challenge (AIxCC) and the OSV vulnerability database. The authors compare against leading agentic bug-finding systems, specifically Claude Code and Atlantis, the AIxCC-winning system, and they run Code-Augur with two frontier but broadly available LLM backbones, including Sonnet and DeepSeek. The evaluation emphasizes vulnerability discovery effectiveness rather than only raw execution outcomes, highlighting the role of inferred specifications in enabling discovery.

Across the two benchmarks, Code-Augur reports finding 34% to 370% more bugs than the state-of-the-art agentic baselines, depending on benchmark and model setting. The paper attributes these gains to the specification inference mechanism: explicit invariants give the system better reasoning anchors and supply guided fuzzing with intermediate semantic targets that are easier to hit than deep crashes. Beyond benchmark evaluation, the authors conduct real-world auditing on widely used open-source projects and uncover 22 previously unknown vulnerabilities. Of these, 16 have already been fixed or confirmed by project developers, and several resulted in bug bounty rewards. The paper also presents a qualitative case study in gpsd, where inferred invariants helped expose an incomplete fix and connect a sequence of related bugs over several months, demonstrating that the generated specifications remain useful after the initial audit.

### 🌱 Insights & Implications
A central insight is that current agentic vulnerability detectors already perform implicit specification inference, but because those assumptions stay hidden in natural-language reasoning traces, they are difficult to validate, reuse, or trust. By surfacing them as executable invariants, Code-Augur turns opaque agent reasoning into artifacts that can be tested and iteratively improved. This is especially important for negative judgments: when an agent says code is secure, the real question is under which assumptions that conclusion holds.

The work also shows that invariant falsification can reveal two kinds of failures: true program vulnerabilities and false or incomplete agent assumptions. Treating both outcomes as useful is a strong methodological contribution because it makes LLM fallibility part of the feedback loop rather than a silent failure mode. Another important implication is that security specifications can serve as a bridge between semantic reasoning and dynamic testing, giving fuzzers more reachable intermediate targets for complex vulnerabilities whose crash site is far removed from the root cause.

For software security and maintenance research, the paper suggests that inferred security invariants could become long-lived development artifacts useful for regression prevention, incomplete-fix detection, and bug-family generalization. Limitations and open questions include how invariant quality depends on the underlying LLM, how scalable assertion insertion and guided fuzzing are for very large or highly stateful codebases, how to minimize false alarms from over-approximate specifications, and how the threat model influences what invariants get inferred. There is also an open research direction in integrating specification inference with automated patch generation or repair, since violated invariants naturally localize the assumptions a patch should restore.

### 🎯 Why This Paper
This paper is highly relevant to research in software security, vulnerability detection, maintenance, and automated repair because it reframes agentic bug finding as explicit security specification inference plus runtime validation. For vulnerability detection, it offers a concrete mechanism to improve both effectiveness and trustworthiness over black-box LLM agents. For software maintenance, the inferred invariants become persistent artifacts that can prevent regressions, expose incomplete fixes, and document security assumptions in the codebase. For automated program repair and vulnerability repair, the explicit violated invariant provides a precise semantic target for patch synthesis or validation, making the paper a strong foundation for future work that combines detection, specification mining, and repair in a closed loop.

## 📄 [PhantomSkill: Malicious Code Injection in Agent Skill Ecosystems](https://arxiv.org/abs/2606.19191v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yu-Ting Lin, Chia-Mu Yu
📅 **Published:** 2026-06-17  🏷️ **Categories:** cs.CR

### 💡 TL;DR
PhantomSkill shows that third-party agent skills can hide malicious behavior inside auxiliary scripts by rewriting overt malware into ordinary-looking vulnerable code. Its VulMask technique maintains the skill’s benign functionality while improving attack success and lowering detection by coding agents and automated reviewers compared with explicit malicious scripts.

### 🏆 Key Contributions
- Identifies auxiliary resources in agent skills, not just SKILL.md text, as a significant supply-chain attack surface for LLM-based coding agents.
- Proposes VulMask, a payload-rewriting method that transforms overt malicious scripts into vulnerability-shaped implementations triggered only under attacker-controlled conditions.
- Demonstrates that vulnerability-masked payloads preserve host-skill utility while reducing warning and malware-level detection relative to overt malicious scripts and prior baselines.
- Provides a broad empirical evaluation across multiple host skills, attack goals, coding agents, payload-generation models, and security reviewers, revealing important weaknesses in current defenses.

### ⚙️ Methodology
The paper studies a threat model where an attacker distributes a modified version of a benign third-party agent skill used by LLM coding agents. A skill is modeled as containing a name, description, detailed SKILL.md instructions, and auxiliary resources such as scripts. The attack focuses on the auxiliary resource component rather than visible textual instructions. The core method, VulMask, starts from an overt malicious script and selects vulnerability classes aligned with the attacker’s goal, such as information exposure, command injection, code injection, or unsafe dependency handling. An LLM-based generator rewrites the malicious payload into a masked script whose harmful effect is exposed through a vulnerability-shaped implementation rather than explicit malicious logic. This masked payload is then embedded into the original host skill’s resource files while preserving script names, paths, and benign input/output behavior when no trigger is present. The attacker may define a trigger source through attacker-influenced external content, project state, or structured input. A candidate malicious skill is retained only after validation in isolation confirms both conditions: the host skill still performs its advertised benign task under normal use, and the hidden side effect activates under the trigger. The overall design leverages the tendency of reviewers and automated scanners to treat insecure-looking code as less suspicious than clearly malicious code.

### 🧪 Experiments
The evaluation addresses end-to-end effectiveness, stealth against coding-agent inspection and automated reviewers, generalization across attack categories, and transfer across different generator models. The skill corpus is built from public and representative agent skills, including official examples, GitHub repositories with SKILL.md files, and community skill collections, restricted to skills with executable auxiliary resources such as scripts/. Covered skill types include file utilities, Git automation, coding assistance, skill auditing, and data processing. Attack goals include credential exfiltration, command execution, agent manipulation, and destruction. Baselines include prompt injection, overt malicious scripts, hidden script attacks, and the CI baseline from prior work. On GPT-5.5, VulMask achieves 58.8% attack success rate, 11.4% warning rate, and 96.6% utility, outperforming overt malicious scripts (46.3% ASR, 25.4% warning, 73.3% utility) and hidden script attacks in utility while also exceeding the CI baseline in stealth with similar ASR (58.8% vs. 56.1%, warning 11.4% vs. 27.2%). Across coding agents, VulMask generally lowers detection compared with overt scripts; for Cursor with GPT-5.5, overt detection is 25.4% with 43.0% ASR, while VulMask detection drops to 11.4% and ASR rises to 58.8%. Against automated reviewers, VulMask reduces malware-style flags but is often still marked as vulnerable: Cisco static flags overt malware at 24.0% and VulMask as malware at 17.1% while flagging vulnerabilities at 100.0%; Cisco LLM reports 100.0% overt malware flags versus 76.3% for VulMask; Snyk agent scan reports 100.0% overt malware flags versus 98.9% for VulMask, with only 1.3% vulnerability flags. Across attack goals on GPT-5.5, VulMask detection ranges from 2.9% to 17.1% with ASR from 52.9% to 79.2%, with especially high success for destruction (79.2% ASR, 4.2% detection). Cross-generator tests show transferability: masked scripts generated by Opus-4.7, GLM-4.7-Flash, and Qwen3-Coder-30B still achieve 58.3%, 54.8%, and 41.0% ASR respectively in the same victim setting.

### 🌱 Insights & Implications
The main insight is that in agent skill ecosystems, exploitable vulnerabilities can function as covert malicious payloads because many defenses distinguish explicit malware from ordinary insecure code. This exposes a blind spot in both LLM-based code inspection and conventional security review pipelines: even when scripts are inspected, reviewers may downgrade intent from malicious to merely vulnerable. The results also show that skill descriptions are not the only meaningful attack vector; auxiliary resources deserve equal scrutiny. A notable implication for software security research is that supply-chain vetting for LLM agents must include resource-level analysis, execution-time sandboxing, and policies that treat triggerable vulnerabilities as potentially intentional backdoors. Limitations include the moderate, not perfect, attack success rates; dependence on trigger design and host-skill compatibility; and uneven transfer across victim agents and generator models, with some agent/model combinations such as Cursor with Opus-4.7 remaining highly resistant. Open questions include how to distinguish accidental insecure code from deliberately weaponized vulnerability-shaped code, how to formally model intent in LLM-integrated software artifacts, and how to build reviewers that jointly reason about functionality, exploitability, and adversarial triggers.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it studies a new supply-chain risk introduced by modular agent extensions and shows that insecure code patterns can be intentionally used as stealthy malicious mechanisms. It advances work in automated vulnerability analysis by highlighting a hard case for reviewers and repair tools: code that appears repair-worthy rather than overtly malicious, suggesting opportunities for new detection, triage, and remediation techniques for LLM-generated or LLM-consumed software artifacts.

## 📄 [CodeSentinel: A Three-Layer Defense Against Indirect Prompt Injection in Code Contexts](https://arxiv.org/abs/2606.19235v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Po-Han Cheng, Chia-Mu Yu, Ying-Dar Lin, Yu-Sung Wu, Wei-Bin Lee
📅 **Published:** 2026-06-17  🏷️ **Categories:** cs.CR

### 💡 TL;DR
CodeSentinel is an inference-time defense for code LLMs that sanitizes untrusted retrieved code before it reaches the model by detecting malicious prompt-injection content at CST-node granularity. It combines syntax rules, node-level language-model anomaly scoring, and perturbation-based influence analysis, achieving an average node-level F1 of 0.80 across six code-context attack families and outperforming prior defenses such as CodeGarrison, DePA, and KillBadCode.

### 🏆 Key Contributions
- Formulates indirect prompt injection in code contexts as a CST-node-level sanitization problem over high-risk model-facing regions such as comments, strings, identifiers, and decoy code blocks.
- Introduces CodeSentinel, a three-layer inference-time defense that combines syntax-guided pre-filtering, CST-guided Dynamic Min-K% likelihood anomaly detection, and CST-guided node perturbation analysis.
- Uses Tree-sitter parsing to align source spans, CST structure, and surrogate-model tokens so suspicious nodes can be localized and sanitized before sending context to downstream code LLMs.
- Designs detection specifically for both overt adversarial payloads and natural-looking semantic triggers, covering attacks hidden in non-functional code, identifier substitutions, and decoy logic.
- Demonstrates stronger node-level detection performance than CodeGarrison, DePA, and KillBadCode across six recent indirect prompt-injection attack families.

### ⚙️ Methodology
CodeSentinel treats external code context as structured syntax rather than flat text. It first parses the untrusted code with Tree-sitter and extracts candidate high-risk CST nodes from four categories: comments, string literals, identifiers, and decoy-like statement/block nodes that are salient to the model but weakly connected to real program logic according to lightweight reachability and salience heuristics. The defense then applies three layers sequentially with early exit so easy cases are filtered cheaply and only harder cases incur more expensive analysis.

Layer 1 is syntax-guided pre-filtering. For each candidate node, it computes a risk score from three components: regex matching for explicit instruction-like or role-control phrases, lexical-structural heuristics for suspicious formatting patterns such as invisible Unicode, abnormal punctuation density, repeated control tokens, and long unusual strings, and an interaction-graph score that emphasizes decoy code with high salience but low call/control/data-flow reachability. Nodes exceeding the threshold are immediately flagged.

Layer 2 is CST-guided Dynamic Min-K% scoring, aimed at adversarial perturbation attacks that may look innocuous at file level but create local token-level likelihood anomalies. For each remaining node, the method uses a surrogate language model to compute token negative log-likelihoods over the serialized input. Instead of fixed Min-K%, it adaptively chooses the top-loss fraction based on node type, node length, and entropy of the node’s token-loss distribution. It then derives anomaly signals including MeanTopK, MaxWinLoss over sliding windows, and MaxSpike between adjacent token losses. These signals are standardized by CST-node type using clean validation statistics, and the final anomaly score is computed as the maximum of calibrated global and local anomaly indicators.

Layer 3 is CST-guided node perturbation analysis for natural-looking semantic triggers that are statistically fluent and therefore hard to catch with anomaly scores alone. It first uses a surprise-gating step that compares each node’s token losses with and without surrounding context, prioritizing nodes whose behavior is highly context-sensitive. For selected nodes, CodeSentinel constructs a syntax-preserving neutralized version of the serialized input by removing or replacing the node with a safe alternative: comments are deleted or neutralized, strings are replaced by safe type-compatible literals, identifiers are renamed consistently, and decoy blocks are converted to no-ops where possible. It then measures how much the surrogate model’s task-relevant logits or predictions change after neutralization. A large behavioral shift indicates the node is exerting suspicious influence on the model, so it is flagged as a semantic trigger.

After aggregating all flagged nodes from the three layers, CodeSentinel sanitizes the corresponding spans in a model-facing copy of the code context and passes only the cleaned context to the downstream victim code LLM. The design is inference-time only, does not require retraining the victim model, and is compatible with both open-weight and black-box code LLM deployments.

### 🧪 Experiments
The paper evaluates CodeSentinel against six recent indirect prompt-injection attack families in code contexts, including attacks based on non-functional adversarial fragments, semantic-preserving rewrites, identifier substitution, and decoy code for misleading code auditors. The baselines are CodeGarrison, DePA, and KillBadCode, representing learned code defenses, line-level perplexity filtering, and token-deletion naturalness approaches respectively.

The evaluation is conducted at node level, matching the paper’s formulation of detecting malicious CST nodes rather than only classifying whole files. The main reported metric is node-level F1, which directly captures localization quality for suspicious comments, strings, identifiers, or decoy blocks. CodeSentinel achieves an average node-level F1 of 0.80 across the six attack families and outperforms all three baselines. The improvement is attributed to its combination of structural localization, token-level anomaly analysis, and behavioral perturbation analysis, which together cover both explicit prompt-like payloads and natural-looking semantic manipulations that existing defenses tend to miss.

Although the provided text excerpt does not include the full numerical breakdown per dataset/attack family, it states that CodeSentinel consistently surpasses prior defenses across the benchmarked attacks and that its multi-layer design is especially important because no single detection signal is sufficient for all prompt-injection styles.

### 🌱 Insights & Implications
A key insight is that indirect prompt injection in code should be modeled at the granularity of syntax nodes rather than whole files or raw text tokens. This is especially important for software engineering settings because attacks can be embedded in comments, literals, identifiers, or decoy snippets that are semantically weak for execution but highly salient to a code LLM.

Another important takeaway is that different attack families require different signals: explicit instruction-like payloads are efficiently caught by syntax and lexical rules, adversarially optimized fragments are better exposed by localized likelihood anomalies, and natural-looking triggers require perturbation-based influence testing. This layered design explains why prior single-signal methods underperform.

The work is also notable for its practicality: it is inference-time, uses a surrogate model instead of requiring victim-model internals, and sanitizes only a model-facing copy of the context, making it more deployable for black-box commercial coding assistants. At the same time, the method has limitations. Its effectiveness depends on accurate CST parsing, good alignment between source spans and surrogate tokens, and the quality of node neutralization strategies. Identifier or string sanitization may affect utility, especially when risky nodes overlap with real executable semantics. The perturbation layer is also more computationally expensive, so gating and node budgets are necessary. Open questions include cross-language robustness, impact on downstream coding quality after sanitization, resilience to adaptive attacks designed to evade all three layers, and whether the surrogate model’s behavioral shifts reliably transfer to stronger victim code LLMs.

### 🎯 Why This Paper
This paper is highly relevant to the researcher’s interests in software engineering, software maintenance, software security, vulnerability detection, and vulnerability repair because it focuses on securing code-LLM workflows that are increasingly used in code review, bug fixing, auditing, and agentic development. It advances the state of practice by providing a deployable defense against poisoned code context, a threat that can directly compromise LLM-based vulnerability analysis or automated repair pipelines through hidden instructions embedded in repositories, documentation, or issue threads.

For research on automated program repair and vulnerability detection, the paper is especially useful because it treats untrusted code context as a structured artifact and performs precise node-level sanitization before downstream reasoning. That makes it a strong foundation for building safer LLM-driven maintenance and security tools, and it suggests new directions for integrating syntax-aware filtering, behavioral influence analysis, and retrieval-time trust controls into secure software engineering assistants.

## 📄 [Compute-Budgeted Exploitability Evidence Graphs for Prospective Vulnerability Triage](https://arxiv.org/abs/2606.19076v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Faruk Alpay, Taylan Alpay
📅 **Published:** 2026-06-17  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper argues that exploitability prediction must be evaluated prospectively under strict time and compute constraints, using only evidence available by a CVE-specific decision time. It proposes a temporal evidence-graph triage framework with auditable evidence certificates, showing modest but real leakage-safe gains over severity-only ranking while demonstrating that naive retrospective protocols can overstate performance by up to 8.5x.

### 🏆 Key Contributions
- Introduces a leakage-safe prospective vulnerability triage protocol that restricts each CVE to public evidence available by a fixed decision time, eliminating temporal label leakage by construction.
- Models heterogeneous public signals—advisories, exploit archives, fix commits, and hacker-community discourse—as a temporal evidence graph and performs compute-budgeted evidence selection per CVE.
- Provides auditable evidence certificates for every risk score, including selected supporting documents, timestamps, source layers, and leakage flags, making prioritization decisions contestable and reproducible.
- Shows empirically that very small evidence budgets are sufficient: two evidence documents per CVE capture most of the attainable benefit.
- Demonstrates that a strong cross-encoder reranker can hurt exploitability triage because semantic relevance to a CVE does not necessarily indicate exploitation evidence.
- Quantifies the magnitude of temporal leakage, showing that naive random-split, unfiltered evaluations substantially inflate exploitability-prediction performance.

### ⚙️ Methodology
The paper formulates vulnerability triage as a prospective ranking problem over CVEs under both temporal and compute constraints. Each CVE has a publication timestamp, severity, description, and CWE type; each candidate evidence document has text, source-layer metadata, provenance, and a timestamp. These entities are organized into a temporal evidence graph containing CVE nodes, evidence nodes, CWE nodes, and product/vendor nodes, with exploitation labels such as KEV and EPSS attached to CVE nodes.

For each CVE v, a decision time tau(v) is defined as publication time plus a fixed observation window Delta. Only evidence whose timestamps are at or before tau(v) is admissible; undated evidence is conservatively excluded. This defines a leakage-safe admissible evidence set per CVE and ensures scores depend only on information available at decision time.

Evidence selection is compute-budgeted. A bi-encoder (e5-base-v2) embeds CVE descriptions and evidence documents, and cosine/inner-product similarity is used to retrieve top candidate evidence from the global corpus. Under a per-CVE document budget B and per-layer caps kappa, the system greedily selects the highest-scoring admissible documents while preserving source-layer diversity. The authors show this greedy strategy is optimal because the constraints form a laminar matroid and the objective is modular.

Selected evidence is converted into features for a calibrated logistic risk model. Feature groups include severity, a smoothed CWE-conditioned exploitation prior, and aggregate statistics over selected evidence such as counts, max and mean retrieval scores, and layer-presence indicators. The output score is calibrated to produce a comparable exploitation probability across CVEs. EPSS is intentionally excluded from the default model because it contains near-label information and would distort leakage-safe evaluation; it is only considered as an ablation.

A central artifact is the evidence certificate emitted for each scored CVE. The certificate records decision time, risk score, rank, selected evidence items, timestamps, source layers, rerank scores if any, and explicit leakage flags. This turns a scalar prioritization output into a provenance-bearing, auditable claim suitable for contestable security operations.

The evaluation protocol uses temporal train/test splitting by disclosure year, training on past CVEs and testing on future CVEs. Performance is measured on positives that are realized only after the decision time, making the task strictly prospective. The paper also defines leakage penalties as additive and multiplicative differences between the leakage-safe protocol and a naive random-split, all-evidence protocol.

### 🧪 Experiments
The experiments combine public data from CISA KEV, FIRST EPSS, CVE/CWE/CVSS tables, CIRCL CVE-CWE-patch evidence, the HackerSignal multi-source corpus, and NVD publication dates. The reported prospective test set contains 12,012 CVEs, including 457 KEV-exploited positives, with a retrieval corpus of tens of thousands of evidence documents.

The primary model uses e5-base-v2 as the bi-encoder for dense retrieval. An optional ablation adds a cross-encoder reranker using bge-reranker-base. All experiments are run on a single NVIDIA RTX 5090 with 32GB memory; retrieval uses exact inner-product GPU search, and the reranker processes about 1,111 pairs per second.

Metrics include KEV recall@K, EPSS-high recall@K, prospective recall@K restricted to exploitation occurring strictly after decision time, precision@K, time per CVE, evidence latency, certificate size, and a compute-efficiency measure based on risk captured per GPU-hour. The headline metric is leakage-safe prospective recall@50.

Main quantitative findings: budgeted evidence selection improves leakage-safe prospective recall@50 from 0.010 for a severity-only baseline to 0.026, a 150% relative gain. On the broader KEV target, recall@50 increases from 0.042 to 0.044. The performance-vs-budget curve saturates quickly: selecting only 2 evidence documents per CVE reaches within 5% of the best result observed up to budget 64, indicating most value is captured with minimal inference.

The reranker ablation is negative: adding the cross-encoder lowers prospective recall@50 from 0.026 to 0.016. ### 🏆 Key Contributions
- Conducts a large-scale empirical measurement of prompt leaking across 1,200 real-world LLM-based applications on six major commercial platforms, finding widespread leakage and exposure of sensitive artifacts such as API keys and developer identities.
- Builds LeakBench, a benchmark grounded in real-world system prompts, to systematically evaluate prompt leaking defenses across attack categories using both security effectiveness and application usability.

The empirical study covers 1,200 real-world applications drawn from six major commercial LLM application platforms. The authors also report real-world case studies on three representative LLM-based applications, showing that AREA remains practical outside the benchmark setting. A key insight is that prompt leaking is not a niche lab phenomenon but a widespread, practical vulnerability in production LLM applications, including systems that already contain explicit anti-leak instructions. The paper also shows that evaluating defenses only by attack success is misleading: many stronger defenses harm application behavior enough to reduce deployment practicality.

The mechanistic contribution, attention drift, is especially important because it links prompt leaking success to internal attention dynamics rather than only surface-level prompt wording. ### 🏆 Key Contributions
- Introduces SafeClawBench, a 600-case adversarial benchmark for tool-using LLM agents spanning six attack families: direct prompt injection, indirect prompt injection, tool-return injection, memory poisoning, memory extraction, and ambiguity-driven unsafe inference.
- Defines a staged evaluation framework with distinct endpoints: `CoreFail@600` for semantic compromise, `HarmEvidence@600` for artifact-audited evidence of harm, and executable sandbox/state oracles for observed tool or state harm.
- Provides a five-model by four-policy comparison matrix with fixed metadata and prompt-policy variants, enabling controlled comparison of model behavior under different defense prompts.
- Adds a core-gated auditing process over canonical semantic failures and a matched Core–Exec analysis showing that semantic judgments and executable harm can disagree in important ways.
- Demonstrates that prompt-level defenses affect outcomes differently across models and protocols, and includes ablations plus prompt-length controls to separate true policy effects from superficial prompt complexity.

### ⚙️ Methodology
The paper frames agent security as a staged process rather than a single binary event. It formalizes six reporting families of attacks over tool-using agents: DPI, IPI, TRI, MPI, MEX, and ADI. Each benchmark case specifies a scenario, user prompt, harm target, lifecycle stage, success predicate, and safe behavior. The core methodological idea is endpoint separation.

First, the `Semantic Core` is a prompt-level benchmark of 600 synthetic adversarial cases. For each case, model, and defense policy, the agent receives fixed prompts describing capabilities such as file access, email, web, database, code, and memory tools, but these are contextual rather than executable. An LLM judge assigns a canonical `CoreFail` label when the final response semantically adopts, authorizes, reveals, plans, or justifies the unsafe goal. These canonical failures are audited for visible evidence of harm categories such as protected-object disclosure, unauthorized access, harmful action, or persistence. This audit distinguishes purely semantic failures from responses that contain artifact-level evidence supporting actual harmful content or intent. The benchmark therefore reports both harm-evidenced failures and semantic-only failures.

Third, the `Exec-Balanced` protocol instantiates matched benchmark cases in a real sandbox with isolated tool/state backends for files, email, memory, database, web, and code execution. Parsed tool calls update sandbox state, and deterministic state-diff oracles decide whether observable harmful actions occurred. Core and Exec are deliberately run as separate model calls because tool availability changes the protocol. The benchmark then performs matched analyses only where both results exist, allowing direct study of disagreement between semantic judgments and executable outcomes.

The evaluation also varies four prompt-level defense policies, including lightweight and longer over-specified prompts, with ablations and matched-length controls. This isolates how much safety improvement comes from defense content versus prompt length or complexity. Overall, the methodology is a controlled stress-test design intended to compare models, policies, and endpoint definitions without conflating textual compliance, audit evidence, and state-changing actions.

### 🧪 Experiments
The benchmark contains 600 controlled cases, with 100 cases per attack family: DPI, IPI, TRI, MPI, MEX, and ADI. The main comparison uses five agent endpoints evaluated under four prompt-level policies, yielding a five-by-four matrix on the Semantic Core. The paper also audits all canonical semantic-failure rows in this panel, totaling 1,834 audited rows with zero missing audit records, and runs a separate executable `Exec-Balanced` panel with deterministic sandbox oracles.

The main metrics are `CoreFail@600` for semantic compromise, `HarmEvidence@600` for audited harm evidence among semantic failures, `SemanticOnly` to identify failures lacking evidence-supported harm, and executable observed-harm endpoints in the sandbox. The study further performs matched Core–Exec comparisons on rows where both protocols are available.

Key quantitative findings reported in the abstract include: without extra prompt protection, semantic failure rates range from 9.0% to 44.2% across models, indicating substantial variation in prompt-level robustness. Audited harm evidence is consistently narrower than semantic failure, showing that many semantically unsafe responses do not produce artifact-visible evidence of harmful action or disclosure. In the matched executable analysis comprising 12,000 rows, 347 rows exhibit sandbox-observed harm, and 291 of those 347 occur in rows that passed the semantic check, revealing a major disagreement where executable harm can arise despite apparently safe semantic responses. The paper concludes that prompt-policy effects are real but highly dependent on both the model and whether evaluation is semantic or executable, so single-metric comparisons are unreliable.

### 🌱 Insights & Implications
The central insight is that agent security cannot be reduced to one attack success rate. Semantic agreement with an attacker, evidence of harmful output, and actual tool/state damage are distinct phenomena that may correlate weakly or even contradict each other. This is especially important for tool-using agents, where harm can emerge through hidden tool actions, persistent memory writes, or state changes not obvious from the final response text.

A particularly important finding is the prevalence of `CorePass–ExecHarm` cases: many harmful sandbox outcomes occur in rows that pass the semantic evaluation. This suggests that text-only judges, even when useful, can miss risks that only become visible when tools and persistent state are active. For software security research, this highlights the need for dynamic evaluation, stateful oracles, and environment-aware benchmarks.

The benchmark’s restriction to prompt-level defenses is also revealing. It shows prompt engineering can shift behavior, but cannot substitute for runtime controls such as permissioning, isolation, approval gates, or secret-aware tool design. This aligns with secure systems practice: alignment prompts help, but strong security needs enforcement mechanisms outside the model.

Likely limitations include the synthetic and stress-test nature of the cases, which may not estimate real-world prevalence; dependence on an LLM judge for semantic labeling; possible sensitivity to tool simulation and oracle design; and the fact that Core and Exec are separate calls, so discrepancies may partially reflect protocol differences rather than only evaluation error. Still, the staged design is valuable because it makes these distinctions visible rather than hiding them in a collapsed metric.

For future work, the paper opens directions in benchmark design for persistent-memory attacks, dynamic vulnerability discovery in agentic systems, robust prompt-policy evaluation, and security controls that integrate semantic analysis with executable monitoring.

This paper is highly relevant because it studies security failures in autonomous, tool-using LLM systems using a rigorous evaluation framework that separates intent-level compromise from concrete harmful effects. The objective is to reshape the retriever’s embedding similarity space so that, for each target prompt, malicious passages rank above benign ones in top-k retrieval, while preserving normal behavior for unrelated prompts. It uses target editing samples to maintain the malicious retrieval behavior for attacked queries, while introducing metric-aligned locality anchors as surrogate regularization points for benign behavior. Lightweight calibration is then performed over the edited parameters so the retriever remains close to pre-attack behavior on anchors yet retains high attack efficacy on the targets. At the same time, the paper reports that retrieval for non-target prompts remains largely unaffected, indicating improved stealthiness/locality after the anchor repair stage.

### 🌱 Insights & Implications
A major insight is that open-source retrievers create a supply-chain attack surface for RAG systems: malicious behavior can be inserted directly into model parameters and redistributed without needing poisoned corpora or fine-tuning pipelines. This makes retriever editing especially concerning for software ecosystems that reuse public models. Another important takeaway is that preserving locality without access to original training data is feasible through anchor-based repair, though the long-tail robustness of such anchors remains an open question. Limitations include the assumption of parameter access to the retriever, focus on open-source dense retrievers rather than proprietary or hybrid retrieval stacks, and evaluation centered on retrieval manipulation rather than full downstream generation harm or detection resistance under adaptive defenses. ### ⚙️ Methodology
The paper structures task construction around three time points: forecast snapshot T0, validation horizon (T0, T1], and task-generation snapshot Tgen. For each repository, the system first builds an evidence bundle from artifacts visible at or before T0, including issues, merged PRs, labels, titles, and short body excerpts. It excludes dependency/platform updates, CI/release work, documentation-only edits, formatting-only changes, and test-only cleanup unless they contain direct bug evidence. Remaining records are categorized into feature implementation/enhancement, bugfix, or refactor using rule-based keyword patterns.

The forecasting unit is a task family rather than an exact future PR. Each family contains category, family anchor, expected behavior, evidence references, target hints, and acceptance criteria. Forecasting follows a persistence assumption: repeated pre-T0 demand around a repository-specific anchor suggests a continuing future work direction. The pipeline extracts anchor terms from labels, titles, module mentions, and excerpts, removes generic/noisy tokens, and forms candidate clusters keyed by (category, anchor) using the first three strongest anchors from each record. Importantly, the system predicts only coarse families, not exact issue text, commits, or patches.

For retrospective validation, the forecast families are frozen and later PR metadata after T0 is used only to judge whether the predicted families align with actual future repository activity. This creates a temporal information boundary preventing leakage from future code changes into forecasting. Once families are validated, they serve as conditioning signals for synthesis at Tgen. The task generator uses the repository snapshot visible at Tgen plus the validated family to create issue-style public task specifications grounded in current code state, rather than replaying later real PRs. Finally, only tasks passing executable validation are kept, including target-test construction, leakage checks, FAIL_TO_PASS validation, and selected PASS_TO_PASS checks, with hidden target tests and gold patches maintained as evaluation assets.

The retrospective validation study uses 80 Python repositories selected from SWE-bench-covered or adjacent projects and broader active software-engineering ecosystem repositories, spanning testing, linting, packaging, web/networking, data/scientific computing, AI/agent frameworks, and infrastructure. The common retrospective window is T0 = 2025-11-28 and T1 = 2026-05-28.

The main experiment evaluates whether forecasted task families match later repository work using a semantic matching metric over post-T0 PR metadata. Under the primary semantic relevance metric, the forecaster achieves 58.1% future-work relevance, indicating that more than half of predicted families correspond meaningfully to actual future repository evolution. The paper emphasizes that post-T0 PRs are used strictly for validation, not as released benchmark tasks.

After validating forecast quality, the authors run the synthesis stage on a task-generation snapshot and construct a final dataset of 200 coding-agent tasks across 61 repositories. Tasks are admitted only if they satisfy executable evidence requirements, including target-test and gold-patch validation. The paper positions its baselines and comparison point conceptually against two dominant alternatives: direct historical replay benchmarks such as SWE-Bench/SWE-Bench++/FEA-Bench/SWE-Factory, and fully synthetic task-generation systems such as SWE-Hub, daVinci-Env, and SWE-Playground. The key result is not superiority on agent accuracy, but successful construction of realistic future-oriented tasks with reduced dependence on replaying historical pull requests.

### 🌱 Insights & Implications
The core insight is that repository evolution can be forecast at the level of recurring task families well enough to guide benchmark synthesis, creating a middle ground between contaminated historical replay and unrealistic free-form synthetic generation. Predicting families rather than exact PRs is crucial: it captures repository demand direction while avoiding overfitting to specific future events.

A major implication for benchmark design is temporal discipline. By separating forecasting, retrospective validation, and task synthesis across T0, T1, and Tgen, the method reduces direct contamination pathways and offers a principled way to build future-oriented evaluation data. This is particularly relevant for rapidly evolving coding-agent ecosystems where public benchmark artifacts can easily leak into training or model-selection loops.

Forecast quality depends on sufficient pre-T0 signal density and active post-T0 repository evolution, so the repository pool is intentionally not an unbiased sample of GitHub. The heuristic scoring and rule-based categorization may miss subtle or novel work directions, and the reported 58.1% relevance shows substantial headroom. The scope excludes many maintenance activities such as dependency, CI, and documentation work, which may matter in real-world agent deployment. Open research questions include whether learned forecasters can outperform heuristic clustering, how to extend the approach beyond Python, how to synthesize harder vulnerability- or security-oriented tasks, and how forecast validity correlates with downstream benchmark usefulness for training or evaluation.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, and automated program repair because it proposes a new way to build repository-grounded coding tasks without simply replaying known historical fixes. For researchers in vulnerability detection/repair and APR, the temporal-boundary design and forecast-conditioned synthesis framework suggest a promising methodology for constructing less-contaminated future bugfix or security-repair benchmarks that better reflect how repositories actually evolve.

- Introduces the first systematic study of image prompt reconstruction attacks against distributed MLLM inference frameworks, extending prior prompt-leakage work from text-only LLMs to multimodal settings.

The paper studies a realistic distributed MLLM inference setting where model layers are split across multiple participants and intermediate embeddings are transmitted between them. MPAA and IEDA are thus complementary: MPAA targets high-fidelity visible reconstruction from information-rich layers, while IEDA aims for robust semantic recovery when only coarse visual concepts remain.

The evaluation covers four representative MLLM families: Gemma 3, Phi 4 Multimodal, Qwen 2.5 VL, and Llama 4 Scout. The paper emphasizes that both methods outperform baseline approaches in their respective settings, though the provided excerpt does not include the exact numerical tables or metric values. ### 🌱 Insights & Implications
A key insight is that distributed MLLM inference exposes a much broader privacy surface than text-only distributed LLMs because image prompts carry dense visual and semantic information, and that information remains recoverable from transmitted hidden states. Another important finding is that although image and text tokens are mixed within intermediate representations, they are still separable enough to enable nearly perfect extraction, which implies that simple token mixing alone is not an effective privacy defense. The work also highlights a layer-dependent leakage pattern: early layers retain fine-grained patch-level detail useful for pixel reconstruction, whereas deeper layers preserve coarser semantics that remain exploitable through generative priors such as diffusion models. ### 💡 TL;DR
CloakLM is a software-only defense that makes GPU-resident model weights hard to reconstruct during LLM serving by breaking the regular, contiguous memory layouts exploited by exfiltration attacks. It preserves the normal virtual memory view for authorized inference, integrates with PyTorch/vLLM, and reportedly achieves near-native performance while substantially raising the difficulty of PCIe snooping and HBM dump-based model theft.

### 🏆 Key Contributions
- Identifies a common root cause behind inference-time model exfiltration attacks: large contiguous and repeatedly accessed GPU memory regions that expose model structure and parameters through PCIe traces and HBM snapshots.
- Proposes CloakLM, a software-only memory-obfuscation framework for model serving that does not require hardware modifications and is compatible with existing serving stacks.
- Combines three complementary protection mechanisms: PCIe traffic shaping, inter- and intra-layer weight shuffling, and physical HBM page remapping.
- Maintains a correct logical/virtual memory layout for authorized execution while presenting fragmented, semantically incoherent physical state to untrusted observers.
- Implements the approach in practical LLM serving environments via integration with PyTorch and vLLM, including distributed inference workloads.
- Demonstrates that the defense significantly increases resistance to passive PCIe observation and HBM dump attacks with negligible or near-native runtime overhead.

### ⚙️ Methodology
The paper proposes CloakLM as a deployment-time protection layer for foundation-model inference on untrusted or partially trusted accelerator infrastructure. The threat model assumes the model provider controls the guest VM or bare-metal serving software stack, but not the full host/GPU substrate, leaving attack surfaces such as the host-to-GPU interconnect, accelerator fabric, GPU driver-level state, memory-mapped interfaces, or RDMA-exposed regions. CloakLM's design goal is to preserve the serving framework's logical memory semantics while disrupting the physical regularity that enables model reconstruction.

Its methodology appears to center on three coordinated transformations. First, PCIe traffic shaping modifies the transfer patterns visible on the interconnect so that an observer cannot easily infer model boundaries, layer structure, or contiguous weight movement from bus traces. This likely involves fragmenting, reordering, padding, or otherwise reshaping host-device transfers to reduce the direct correspondence between observed transfer sequences and the true tensor layout. Second, inter- and intra-layer weight shuffling permutes model parameters both across layers and within layer tensors, breaking assumptions that weights are stored in canonical contiguous order. The system must therefore maintain metadata or translation logic so that the execution runtime can reconstruct the correct tensor interpretation without altering high-level inference behavior. Third, physical HBM page remapping changes how model pages are placed in GPU memory, ensuring that even if an attacker dumps HBM contents, the resulting snapshot is fragmented and detached from the original tensor organization.

A key architectural principle is separation between the authorized virtual view and the unauthorized physical view. Inference software such as PyTorch and vLLM continues to access tensors through valid logical addresses, while CloakLM intercepts or manages the underlying placement and transfer organization. This suggests the framework operates below or alongside tensor allocation/loading mechanisms, potentially during model loading, parameter staging, and GPU memory management. Because the paper emphasizes no hardware changes and compatibility with confidential computing, the contribution is likely positioned as a software hardening layer complementary to TEEs rather than a replacement for them. The method is also described as lossless, implying exact model semantics and output quality are preserved, unlike approximation-based protections such as quantization, watermarking, or weight perturbation.

The evaluation is reported on distributed inference workloads using LLaMA and Qwen models, indicating relevance to realistic large-scale serving scenarios rather than only toy DNNs. The implementation is integrated with PyTorch and vLLM, so the experimental setup likely compares baseline serving performance against CloakLM-enabled serving under common inference configurations. The abstract explicitly references resistance to two families of attacks: passive PCIe observation, as exemplified by Hermes-style reconstruction, and HBM dump attacks, as exemplified by TunnelS-style exfiltration via driver-level access. The main performance claim is near-native performance with negligible overhead for authorized execution. While the abstract does not provide exact latency, throughput, or memory overhead numbers, the likely metrics include inference latency, tokens/sec throughput, and possibly overhead during model loading or distributed execution. Security-oriented metrics are not stated explicitly, but probable measures include reconstruction difficulty, reduction in structural signal visible in PCIe traffic, fragmentation of dumped HBM state, and degraded success of model extraction pipelines. Baselines are likely unprotected PyTorch/vLLM deployments and possibly simpler obfuscation variants without one or more of the three mechanisms. The key qualitative result is that CloakLM substantially increases the resistance of LLaMA and Qwen serving deployments to model exfiltration attacks while keeping serving efficiency close to the original system.

### 🌱 Insights & Implications
The paper is important because it reframes model theft in serving systems as a memory-layout leakage problem rather than purely a cryptographic or access-control problem. Its main insight is that attackers benefit from semantic regularity in GPU memory and interconnect traffic; by destroying this regularity while preserving logical correctness, software-only defenses can materially harden deployment without specialized hardware.

A notable implication for software security research is that inference-time protections can be inserted at the systems/runtime layer, opening a new design space between application-level defenses and hardware TEEs. Fourth, the defense focuses on exfiltration of weights during inference, not broader side channels such as timing, kernel traces, or activation leakage. ## 📄 [TIGER: Inverting Transformer Gradients via Embedding-Subspace Distance Optimization](https://arxiv.org/abs/2606.18312v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** William Kalikman, Ivo Petrov, Dimitar I. Dimitrov, Martin Vechev
📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.CR, cs.DC, cs.LG

### 💡 TL;DR
TIGER is a gradient inversion attack for transformer models that reconstructs private training text by optimizing token embeddings to align with low-rank subspaces revealed by observed gradients, rather than matching full gradients or testing discrete token membership. This continuous subspace objective makes the attack faster, more scalable to encoder models, and substantially more robust to quantization and differential privacy noise than prior transformer inversion methods.

### 🏆 Key Contributions
- Introduces TIGER, a continuous gradient inversion attack for transformer LLMs that reconstructs private text from federated learning updates using embedding-subspace optimization.
- Derives a differentiable span-distance objective from the low-rank structure of transformer linear-layer gradients, replacing brittle discrete token filtering with continuous optimization.
- Develops separate recovery strategies for decoder-only and encoder-only transformers, including a sequential causal recovery method for decoders and a bidirectional alignment loss for encoders to avoid degenerate solutions.
- Demonstrates improved reconstruction quality and runtime over prior attacks, especially on encoder-only models and under quantization or DP-style noise, including the first successful reconstructions in DP-defended federated learning settings for decoder models.
- Provides an open-source implementation to support future research on privacy leakage and defenses in collaborative LLM training.

### ⚙️ Methodology
TIGER exploits the fact that for transformer linear layers, the column space of the observed weight gradient reveals the row span of the hidden states that produced it, under mild rank conditions. Prior work such as DAGER uses this signal through discrete token membership tests: candidate token embeddings are checked for inclusion in the recovered subspace. TIGER instead converts this geometric signal into a differentiable objective. It directly optimizes continuous token embeddings so that their hidden representations minimize distance to the gradient-induced subspaces.

The core primitive is a span-distance objective between a vector and a subspace represented by a projection matrix. For a candidate hidden representation, TIGER penalizes the component lying outside the subspace induced by the true gradient. This produces a continuous loss that can be optimized efficiently with gradient-based methods and is less sensitive to numerical perturbations than exact membership tests.

For decoder-only transformers, TIGER leverages causal attention. Because each token representation depends only on preceding positions, the method reconstructs embeddings sequentially, position by position, using subspace constraints from the first few transformer layers. The attack uses only partial forward computations and a forward alignment loss, avoiding full gradient matching over the entire model.

For encoder-only transformers, causal decomposition is unavailable because each position depends on all tokens. TIGER therefore introduces a bidirectional alignment objective. The forward term encourages optimized hidden states at each layer to lie in the gradient-induced column space, while the backward term encourages the observed gradient directions to lie within the span of the optimized hidden states. This symmetric constraint helps prevent degenerate solutions such as repeated tokens or duplicate sequences and strengthens recovery in dense, non-causal attention settings.

Overall, TIGER attacks only selected layers and relies on subspace geometry rather than exact end-to-end gradient matching, which reduces computational cost and improves stability in large-batch, noisy, or defended scenarios. After optimizing embeddings, the recovered continuous representations are mapped back to tokens for text reconstruction.

### 🧪 Experiments
The paper evaluates TIGER on modern transformer language models covering both decoder-only and encoder-only architectures, specifically Gemma-3-4B-IT and EmbeddingGemma-300M. Experiments use WikiText batches with varying batch sizes, sequence lengths, and defense conditions, including quantized gradients and Differential Privacy-style additive noise. Reconstruction quality is measured using ROUGE-1 and ROUGE-L.

Baselines include prior transformer gradient inversion attacks, especially DAGER, which is the strongest subspace-based baseline for decoder models, along with comparisons to optimization-based approaches discussed in the literature. The evaluation targets settings where prior methods are known to struggle: batched updates, encoder models with non-causal attention, lower-precision gradients, and DP-defended updates.

Results show that on decoder-only models, TIGER is competitive with and slightly better than DAGER in undefended settings. More importantly, under DP-style noise, DAGER collapses to near-zero reconstruction quality, while TIGER still recovers meaningful text, achieving ROUGE-1 above 70% up to noise level sigma = 10^-3. On encoder-only models, TIGER provides much larger gains, reaching around 60% ROUGE-1 on batches of up to 16 sequences, in settings where prior attacks perform poorly or are computationally infeasible. The paper also reports that TIGER performs well under quantization and across a broad range of batch-size/sequence-length combinations, while reducing runtime by avoiding full gradient matching and using only partial forward/backward computations tied to the subspace objective.

### 🌱 Insights & Implications
The paper shows that apparent robustness of federated learning for transformer fine-tuning can be misleading when evaluated only against brittle prior attacks. Even when gradients are quantized, noisy, or aggregated over larger batches, the low-rank structure of transformer gradients still leaks exploitable geometric information. By turning exact algebraic recovery signals into a continuous optimization objective, TIGER reveals stronger privacy risks than previously recognized.

A key insight is that subspace information is more resilient than discrete token filtering. This suggests a broader attack design principle: instead of relying on exact symbolic recovery conditions, privacy attacks can often become stronger by optimizing against softer structural constraints. For software and security researchers, this parallels how approximate side channels may remain exploitable even when exact leakage is obscured.

The work also highlights architectural differences: causal structure in decoder models enables efficient sequential recovery, whereas encoder models require additional constraints to avoid collapse. The bidirectional alignment loss is an important contribution here, but it also indicates that non-causal architectures remain harder to attack and may require richer geometry-aware objectives.

Limitations and open questions include how TIGER scales to even larger batches, longer contexts, mixed-precision distributed training pipelines, and stronger DP regimes; how well it transfers to other transformer variants and multimodal models; and what defenses can specifically disrupt the leaked subspace structure without destroying model utility. The results imply that current defenses based only on mild noise or quantization may be insufficient.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, vulnerability detection, and automated repair because it studies a concrete privacy vulnerability in modern collaborative ML systems: leakage of sensitive training data through transformer gradients in federated learning. For a researcher interested in security and repair, TIGER advances understanding of attack surfaces in LLM training pipelines, provides a more robust adversarial analysis methodology than prior work, and helps identify where defensive mechanisms such as DP or quantization fail. It is especially useful for informing secure ML system design, evaluation of privacy-preserving training infrastructure, and development of mitigation or verification techniques for leakage-resilient model updates.
