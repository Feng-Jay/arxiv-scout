## 📄 [VisualRepair: Dynamic Tool Calling and Region Focusing for Visual Software Issue Repair](https://arxiv.org/abs/2607.14075v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Jingyu Xiao, Zhongyi Zhang, Haoran Hou, Yuxuan Wan, Yuan Jiang, Yintong Huo, Michael R. Lyu
📅 **Published:** 2026-07-15  🏷️ **Categories:** cs.SE

### 💡 TL;DR
VisualRepair is a multimodal automated program repair framework for software issues with screenshots, GIFs, IDE captures, or text-heavy images. It combines image-type-aware tool invocation with dynamic region focusing over bug-relevant visual areas, and achieves state-of-the-art results on SWE-bench Multimodal by resolving 196 test issues and 25 dev issues, exceeding the best prior systems by 10 and 11 cases respectively.

### 🏆 Key Contributions
- Identifies two core bottlenecks in visual software issue repair: heterogeneous visual artifacts in issue reports and inaccurate localization of bug-relevant visual regions.
- Introduces Image Type-aware Tool Calling (ITTC), which classifies visual inputs and dispatches tailored processing chains using specialized tools such as GIF frame extraction, OCR, cropping, and code-library support.
- Introduces Dynamic Test-time Region Focusing (DTRF), which generates multiple bug-region candidates and refines them with adaptive zoom-in and zoom-out operations to improve localization and increase patch diversity.
- Demonstrates state-of-the-art performance on SWE-bench Multimodal, outperforming leaderboard baselines on both test and dev splits.

### ⚙️ Methodology
VisualRepair is an MLLM-based visual software issue repair framework designed for issue reports that include both text and visual artifacts. The pipeline has three stages. First, ITTC handles heterogeneous visual inputs. A lightweight classifier predicts the image type for each attachment, such as UI screenshot, IDE snapshot, GIF, or text-centric image, and then triggers a type-specific tool chain. The GIF tool converts animated reports into frame sequences and selects keyframes using inter-frame mean absolute error with a dynamic threshold and frame-gap control, preserving salient interaction changes while filtering redundant frames. The OCR tool uses PaddleOCR to extract dense text content from screenshots such as stack traces, code excerpts, or terminal messages, then feeds this text back into the MLLM context. The cropping tool removes large blank or uninformative areas to compact the visual input and reduce distraction. A code library tool provides relevant code templates to aid reproduction and patch ranking. Second, after visual preprocessing, the framework performs hierarchical fault localization over the repository to identify suspicious files and code hunks, reducing the repair search space before patch generation. Third, DTRF addresses the mismatch between full-image context and the true bug region. The MLLM is prompted to ground multiple candidate bug-related regions rather than a single box. Each candidate is then refined by adaptive zoom-in and zoom-out operations so the model can inspect both tighter local detail and slightly broader context. Candidate patches are generated conditioned on these focused regions, which improves both localization accuracy and diversity of repair attempts. Final patch selection uses compilation validation and visual difference checking, filtering candidates by build correctness and whether the patch meaningfully fixes the visual defect.

### 🧪 Experiments
The evaluation is conducted on SWE-bench Multimodal, a benchmark for software issue repair with visual context. The paper analyzes the benchmark’s image-type distribution through manual annotation of a 10% sample, showing substantial heterogeneity: UI screenshots are dominant, followed by IDE images, GIFs, and plain-text images, with some irrelevant or missing visuals. VisualRepair is compared against state-of-the-art approaches reported on the SWE-bench Multimodal leaderboard. The main metric is the number of resolved issues. VisualRepair resolves 196 instances on the test split and 25 on the dev split. These results exceed the strongest baseline by 10 test cases and 11 dev cases as of March 2026. The paper also motivates the method with qualitative evidence that standard MLLM grounding often misses parts of the true bug-relevant region, especially near screenshot boundaries or in cluttered interfaces, which DTRF is designed to address. From the reported results, both type-aware visual processing and region-focused inference contribute materially to end-to-end repair effectiveness.

### 🌱 Insights & Implications
For software engineering and APR, the paper’s main insight is that multimodal repair fails for two reasons that are easy to overlook in text-centric pipelines: visual inputs are not homogeneous, and full-image reasoning is too noisy for precise defect understanding. The proposed solution suggests that visual software repair should be treated as an orchestration problem, where perception tools are selected conditionally and region-level inference is performed adaptively at test time. This is especially relevant for GUI bugs, IDE-based bug reports, and interaction failures where screenshots encode information unavailable in natural language alone. A notable implication is that better patch diversity can come from multiple visual focal points rather than only multiple decoding samples. Limitations and open questions include the dependence on external tooling quality such as OCR and GIF parsing, potential classifier errors in image-type routing, uncertain generalization beyond SWE-bench Multimodal, and the fact that the paper excerpt does not provide full ablation details, cost analysis, or robustness results under noisy or irrelevant screenshots. There is also an open research opportunity to extend this framework toward security-oriented multimodal bug reports, richer temporal reasoning for interactive failures, and tighter coupling between visual localization and code-level vulnerability localization.

### 🎯 Why This Paper
This paper is highly relevant to the researcher's interests because it extends automated program repair into multimodal settings where bug evidence includes screenshots, IDE captures, and GIFs, which are increasingly common in modern software maintenance workflows. It advances APR by showing how dynamic tool use, visual grounding, and region-aware inference can materially improve repair performance, offering ideas that could transfer to vulnerability detection, vulnerability repair, GUI-related maintenance, and broader software engineering systems that need to connect visual evidence with code changes.

## 📄 [DREA: Decoupled Reasoning and Exploration Agents for Repository-Level Vulnerability Detection](https://arxiv.org/abs/2607.13439v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Mingyang Sun, Guozhu Meng
📅 **Published:** 2026-07-15  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
DREA is a repository-level vulnerability detection framework that separates expensive security reasoning from cheap repository exploration, letting an LLM actively gather cross-file evidence instead of relying on fixed context windows. On a new benchmark of real Python CVE fix pairs, it substantially improves pair-level detection accuracy while offloading most tokens to a lightweight local explorer and sharply reducing API cost.

### 🏆 Key Contributions
- Introduces DREA, a hypothesis-driven two-agent framework that decouples vulnerability reasoning (Planner) from repository navigation and evidence retrieval (Explorer).
- Builds RepoPairBench, a repository-grounded benchmark of 100 validated Python vulnerability-fix pairs from real CVEs spanning 2021-2025 and 48 CWE categories.
- Proposes a reasoning-correctness evaluation protocol using structured LLM-as-a-Judge assessment to test whether model rationales match documented vulnerability mechanisms.
- Provides empirical evidence that adaptive repository exploration, rather than simply adding more code context, improves repository-level vulnerability detection while greatly lowering billable inference cost.

### ⚙️ Methodology
DREA treats vulnerability detection as an iterative, hypothesis-driven auditing process. A high-capability Planner model starts from the target function, identifies security-relevant operations, forms an initial hypothesis about a possible vulnerability, and decides what evidence is missing. It then issues targeted exploration requests to an Explorer model, such as locating callers/callees, tracing data flow, finding validation or authorization checks, or inspecting related configuration and policy code across files. The Explorer is a lightweight local model constrained to read-only repository navigation tools (`ls`, `glob`, `grep`, `read_file`) and is explicitly prevented from making vulnerability judgments; instead, it returns structured findings about repository context, candidate evidence, and relevant code artifacts. The Planner incorporates these findings, revises its hypothesis, and either asks for more evidence or emits a final `VULNERABLE` or `BENIGN` decision. A vulnerable verdict must include a plausible trigger path from attacker-controlled input to a dangerous operation or violated security boundary, while a benign verdict must justify why the relevant behavior is adequately protected. This decoupling is central: the expensive model focuses on semantic security reasoning, while the cheap model absorbs token-heavy repository traversal. The paper also introduces a complementary evaluation of reasoning quality, where an LLM judge checks whether a detector’s rationale aligns with the documented vulnerability mechanism, allowing the authors to distinguish correct labels reached via sound reasoning from 'Lucky Hits' where the label is correct but the explanation is flawed.

### 🧪 Experiments
The evaluation uses RepoPairBench, a new benchmark of 100 repository-grounded Python vulnerability-fix pairs collected from real-world CVEs and linked to full Git repositories. The benchmark emphasizes validated vulnerable/patched pairs rather than isolated labeled functions, enabling pair-based evaluation of whether a method correctly identifies the vulnerable version while rejecting the patched one. The study compares DREA against function-only and static-context style baselines across three backbone LLMs and four detection configurations. The main metric is Pair-Correctness, which measures correct pairwise discrimination between vulnerable and patched versions; the paper also evaluates reasoning correctness to test whether explanations match the actual vulnerability mechanism, along with token usage and estimated API cost. Across the three backbone models, DREA improves Pair-Correctness from 19-26% for the baseline setting to 30-42%. More than 93% of tokens are offloaded to the Explorer, yielding an estimated 16x-48x reduction in billable API cost. Ablation results indicate that the gain comes from structured, goal-directed exploration rather than merely increasing context volume. The reasoning evaluation further shows that 26-55% of true positives, for both DREA and the function-only baseline, are correct predictions supported by incorrect or incomplete rationales.

### 🌱 Insights & Implications
The paper’s main insight is that repository-level vulnerability detection benefits from adaptive evidence gathering rather than fixed retrieval heuristics: many vulnerabilities depend on cross-file authorization, sanitization, or configuration logic that static context selection can miss. A second important finding is that explanation quality remains a major bottleneck: even when models predict the right label, a large fraction of successes are 'Lucky Hits,' suggesting current LLMs often recognize superficial risk patterns without faithfully reconstructing the real exploit mechanism. This has implications for automated security auditing and program repair, because downstream uses such as patch generation or analyst assistance require mechanistically correct reasoning, not just correct classification. Limitations include the current focus on Python, the relatively modest benchmark size, dependence on documented CVE/fix pairs, and the fact that structured read-only exploration may still miss deep semantic behaviors that require dynamic analysis or stronger program-analysis integration. Open questions include how well the Planner-Explorer design transfers to other languages, whether explorer outputs can be grounded with symbolic or static analysis, and how to train models to improve rationale faithfulness rather than only detection accuracy.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and automated vulnerability repair research because it targets the realistic repository-level setting where vulnerabilities often span multiple files and abstractions. It advances the state of LLM-based security analysis by combining agentic code exploration with explicit reasoning evaluation, offering useful ideas for building better vulnerability detectors, more trustworthy security explanations, and future repair systems that need correct cross-file root-cause localization before generating fixes.

## 📄 [ProfMalPlus: Agent-Coordinated Detection of Malicious NPM Packages via Static-Dynamic Analysis Synergy](https://arxiv.org/abs/2607.13965v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yiheng Huang, Zhijia Zhao, Bihuan Chen, Susheng Wu, Zhuotong Zhou, Yiheng Cao, Kun Hu, Xin Hu, Xin Peng
📅 **Published:** 2026-07-15  🏷️ **Categories:** cs.SE, cs.CR

### 💡 TL;DR
ProfMalPlus detects malicious NPM packages by combining object-sensitive static behavior graphs, targeted dynamic execution, and coordinated LLM agents that reason over security-relevant code slices instead of whole files. This hybrid design achieves a reported 98.1% F1-score, beats prior detectors by 3.5% to 52.6%, and surfaced 597 previously unknown malicious packages that were later removed from NPM.

### 🏆 Key Contributions
- An object-sensitive behavior-graph representation for JavaScript/NPM packages that captures sensitive APIs, third-party calls, unresolved calls, control/data dependencies, and obfuscation-resilient behavior structure
- A static-dynamic synergy pipeline that merges dynamic evidence back into the static graph to resolve runtime-dependent behaviors rather than treating the two analyses independently
- A coordinated multi-agent LLM reasoning framework with local judges, self-consistency checking, a global judge, and a router that decides whether to add third-party semantic enrichment or dynamic augmentation
- Security-relevant code slicing with inline static-analysis annotations so the LLM reasons over compact, source-level evidence instead of lossy abstract features or entire files
- A localization stage that identifies concrete malicious snippets and explains why they are suspicious, improving interpretability for auditors and defenders
- A real-world deployment result showing discovery of 597 previously unknown malicious NPM packages, all confirmed and removed from the registry

### ⚙️ Methodology
ProfMalPlus targets malicious behaviors that execute during installation and import time, which the authors argue cover the majority of NPM supply-chain attacks. The pipeline begins with a script analyzer that inspects package installation commands and identifies package entry files to define the analysis scope. It then constructs object-sensitive behavior graphs over the package's own code. These graphs encode control flow, control dependence, and data dependence between statements, while also modeling JavaScript object references and aliasing so that sensitive API calls, property accesses, and third-party invocations can be resolved more precisely than object-insensitive PDG-based approaches. The graph also records unresolved calls that may depend on runtime behavior or obfuscation. From suspicious graph nodes, a code slicer extracts security-relevant source-code slices by traversing dependence edges; the slices preserve original code but add inline evidence from static analysis, such as resolved APIs, call relationships, and other annotations. Each slice is evaluated by a local judge agent, which independently decides whether the slice reflects malicious behavior. To reduce LLM variance, ProfMalPlus repeats local judgments and consolidates them through self-consistency verification. A global judge agent then synthesizes the verified slice-level reports in control-flow order to produce an entry-level verdict while considering inter-slice relations and accumulated context. If the result remains undetermined, the global judge flags evidence gaps and a router agent selects one of two enrichment paths. In third-party enrichment, registry-derived metadata and method semantics are injected into slices for third-party calls. In dynamic augmentation, the package is executed in a sandbox to observe runtime-dependent behavior and resolve unresolved calls; the resulting evidence is merged back into the behavior graph, yielding a unified representation rather than disjoint static and dynamic outputs. The updated slices are then re-judged. For packages judged malicious, a localization agent identifies the specific code snippets responsible and generates explanations. Overall, the method replaces a prior graph-classification style detector with semantic, evidence-driven multi-agent reasoning over focused code regions.

### 🧪 Experiments
The paper reports an evaluation against five state-of-the-art malicious-package detectors, explicitly naming GuardDog, Cerebro, and the authors' earlier ProfMal system in the visible text; the remaining baseline names are not fully visible in the provided excerpt. The main effectiveness result is an F1-score of 98.1% for ProfMalPlus, which outperforms the compared detectors by 3.5% to 52.6%. The evaluation is framed around malicious NPM package detection and includes both controlled comparison with prior techniques and a real-world study over packages from the NPM ecosystem. In the real-world evaluation, ProfMalPlus identified 597 previously unknown malicious packages, and the paper states that all were confirmed and removed from NPM. The extracted text does not expose the full dataset composition, train/test split, exact metrics table, or ablation details, but the reported setup clearly emphasizes comparison to existing static, dynamic, graph-based, and LLM-based detectors and uses F1-score as the headline metric.

### 🌱 Insights & Implications
The paper's main insight is that malicious-package detection improves when semantic reasoning is applied to compact, security-relevant source slices backed by program-analysis evidence, rather than to coarse behavioral abstractions or full files. The object-sensitive graph is especially relevant for JavaScript because sensitive behavior is often hidden behind object properties, aliases, and indirect calls that simpler lexical, syntactic, or object-insensitive dependence models miss. Another important takeaway is that static and dynamic analysis should not merely run side by side: merging runtime-resolved evidence back into the static representation creates a stronger basis for downstream reasoning. The multi-agent design also addresses practical LLM weaknesses by using decomposition, self-consistency, and targeted evidence acquisition instead of relying on a single monolithic prompt. For software security research, this suggests a promising pattern for supply-chain defense systems: use precise program analysis to focus the LLM's attention, then let the LLM handle semantic interpretation and explanation. Potential limitations and open questions include scalability and cost of multi-agent LLM inference, robustness to adversarial prompt-targeting or novel obfuscation strategies, dependence on sandbox coverage for dynamic augmentation, and generalizability beyond installation/import-time payloads or beyond the NPM/JavaScript ecosystem. Because the provided text is truncated, details on false-positive behavior, ablations, and failure modes are not fully available here.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, maintenance, software security, vulnerability detection, and automated repair-oriented analysis because it addresses a major real-world supply-chain threat using a blend of static analysis, dynamic analysis, and LLM reasoning. It advances work on malicious-code and vulnerability detection by offering interpretable, source-level localization of harmful behavior, object-sensitive modeling for JavaScript, and a principled way to coordinate evidence gathering and semantic judgment—ideas that could transfer to vulnerability triage, patch validation, malicious dependency auditing, and agentic security tooling.

## 📄 [Why Not Fix It Once and for All? An Empirical Study of Multiple Patches for Vulnerability Fixes in Open-Source Software](https://arxiv.org/abs/2607.13206v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Weiliang Qi, Youpeng Li, Xinda Wang
📅 **Published:** 2026-07-14  🏷️ **Categories:** cs.CR, cs.SE

### 💡 TL;DR
This paper systematically studies open-source vulnerabilities that are fixed through multiple patches instead of a single commit, showing that such cases are common and stem from multi-location remediation, surrounding maintenance work, and defective initial fixes. From 1,646 multi-patch CVE records, the authors build a taxonomy, characterize how multi-patch fixes differ from single-patch ones, and show that existing vulnerability detection methods have limited ability to determine whether an initial patch fully remediates the issue.

### 🏆 Key Contributions
- Constructs and open-sources a manually labeled dataset of 1,646 multi-patch vulnerability fix records derived from NVD-linked OSS patches.
- Presents the first systematic taxonomy of multi-patch vulnerability fixes with 3 main categories and 6 subcategories: multi-location fixes, fix-and-related changes, and defective fixes.
- Empirically compares multi-patch and single-patch fixes across time, projects, programming languages, temporal spacing between patches, and patch-level similarity/complexity features.
- Evaluates seven representative vulnerability detection techniques for checking whether the first patch in a sequence already fully fixes the vulnerability, highlighting current limitations and future research needs.

### ⚙️ Methodology
The study starts from all published CVEs available through the NVD JSON feeds as of May 2025 and filters to entries that include references explicitly labeled as patches. To focus on open-source fixes with retrievable source changes, the authors keep patch URLs containing git and commit, parse commit IDs from links, and de-duplicate references that point to the same underlying patch across platforms such as GitHub and GitLab. They also manually remap outdated repository URLs and implement collection scripts for GitHub, cgit, and GitLab to recover metadata and code diffs. This yields 25,113 OSS CVEs with security patches, including 1,646 CVEs associated with multiple distinct patches.

For the qualitative analysis, three experienced security researchers independently inspect an initial random sample of 100 multi-patch cases, derive a preliminary taxonomy, refine it into a final codebook through discussion, and then independently label the full dataset with consensus-based resolution of disagreements. The resulting taxonomy has three major categories: A) multi-location fixes, where multiple commits are required because the vulnerable logic exists in different branches/projects or distinct code locations in the same branch; B) fix and related changes, where the fix sequence includes things like temporary workarounds or documentation/supporting updates; and C) defective fixes, where the original patch is incomplete or introduces new bugs, requiring follow-up commits. The paper also notes an additional ambiguity class in NVD records where vulnerability-introducing and vulnerability-fixing commits are both referenced as patches.

After categorization, the authors perform comparative empirical analysis between single-patch and multi-patch fixes, examining distributions over languages, projects, and years, as well as temporal intervals between consecutive patches and patch-level characteristics such as similarity and complexity. Finally, they assess whether existing vulnerability detection approaches can determine remediation status during an ongoing multi-patch sequence, especially for two practically important scenarios: incomplete fixes and multi-location fixes.

### 🧪 Experiments
The empirical dataset contains 25,113 OSS CVEs with patch references, of which 1,646 are multi-patch cases, averaging 2.55 patches per CVE. Manual categorization reports the following counts across subcategories: 830 cases of fixes across different branches or projects, 30 cases of different locations within the same branch, 16 workaround-then-formal-fix cases, 128 fix-plus-documentation-update cases, 641 incomplete-fix cases, and 119 bug-introducing-fix cases. Because one CVE may belong to multiple categories, these counts are not necessarily exclusive.

The characterization experiments compare multi-patch and single-patch fixes over temporal, project, language, and patch-property dimensions. The paper reports that multi-patch fixes are concentrated in widely used programming languages, are more common in certain projects, and have increased more sharply over time than single-patch fixes. A notable quantitative result is that 31.7% of multi-patch fixes have more than one day between consecutive patches, implying meaningful windows where a released initial patch may still leave users exposed. The study also analyzes code similarity and complexity differences across multi-patch categories.

For remediation-status validation, the authors evaluate seven representative vulnerability detection techniques to test whether they can predict if the first patch will later evolve into a multi-patch fix, focusing on incomplete-fix and multi-location scenarios. The excerpt does not provide the names of the seven methods, exact metrics, or detailed numerical scores, but the stated outcome is that current methods are insufficient for reliably determining full remediation in these settings.

### 🌱 Insights & Implications
The paper highlights that security patching is often not a clean one-commit event, and this matters directly for vulnerability dataset construction, patch backporting, remediation guidance, and one-day exploit risk. In particular, reversing a later patch from a multi-patch fix can produce an intermediate program state that is neither clearly vulnerable nor clearly fixed, which can inject label noise into vulnerability detection and repair benchmarks.

A key insight is that many multi-patch cases arise from normal software engineering realities rather than purely poor patch quality: branch maintenance, backporting constraints, duplicated vulnerable logic, and supporting documentation or workflow updates all contribute. At the same time, a large portion also comes from incomplete or defective initial fixes, which is especially relevant to automated program repair and vulnerability repair research because it shows that “a patch exists” does not imply “the vulnerability is fully resolved.”

The study also surfaces ecosystem-level issues such as mismatches between CVE reporting rules and actual practice, Git workflow conventions, weak guidance on security patch reporting, time pressure during incident response, reliance on senior maintainers, and vendor neglect. An important limitation from the provided text is that detailed performance numbers for the seven evaluated detection methods are not included here, so it is hard to judge which technical signals are most promising. Open questions include how to automatically distinguish intermediate remediation states, how to model cross-branch or cross-project fix propagation, and how to build benchmarks that represent multi-stage security fixes instead of assuming a binary vulnerable/fixed transition.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance, software security, vulnerability detection, and vulnerability repair because it studies how real-world vulnerability remediation often unfolds as a sequence of commits rather than a single patch. It directly advances research on automated program repair and security patch analysis by exposing a major source of dataset noise and evaluation error: intermediate patches may be partial, branch-specific, or defective, so tools that detect vulnerabilities or synthesize fixes need to reason about remediation completeness, not just patch existence.

## 📄 [Agent Skill Security: Threat Models, Attacks, Defenses, and Evaluation](https://arxiv.org/abs/2607.13987v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Sanket Badhe, Priyanka Tiwari
📅 **Published:** 2026-07-15  🏷️ **Categories:** cs.CR

### 💡 TL;DR
This paper argues that reusable LLM agent skills introduce security risks across their full lifecycle, not just at runtime, and proposes SkillSec-Eval to systematically study those risks. Using 327 real-world skills, the authors show that attacks can succeed at repository admission, retrieval, planning, execution, and evolution, making lifecycle-aware defenses necessary for secure agent ecosystems.

### 🏆 Key Contributions
- Defines a lifecycle-oriented threat taxonomy for reusable agent skills spanning repository admission, semantic retrieval, planner selection, runtime execution, and skill evolution.
- Introduces SkillSec-Eval, a modular evaluation framework that models the full reusable skill lifecycle and supports reproducible security benchmarking.
- Builds a benchmark with 327 real-world skills, semantically overlapping enterprise-style repositories, capability-clustered workloads, and lifecycle-specific attack scenarios.
- Empirically evaluates representative attacks across multiple lifecycle stages and shows that meaningful vulnerabilities exist well beyond runtime execution.
- Highlights that semantic layers such as metadata, retrieval embeddings, and planner-facing descriptions create attack surfaces not covered by conventional software supply-chain defenses.

### ⚙️ Methodology
The paper models a reusable agent skill as a tuple S=(M, W, P, T, V), where M is semantic metadata, W is the executable workflow, P is the permission set, T is the underlying tools/services, and V is provenance/version information. Building on this abstraction, the authors decompose the skill lifecycle into six stages: authoring, storage, retrieval, planner selection, execution runtime, and evolution. They then collapse these into evaluation trust boundaries centered on repository admission, semantic retrieval, planner selection, execution, and evolution. For each boundary, they define the security assumptions being violated and map corresponding attack classes to that stage. SkillSec-Eval instantiates this lifecycle-aware threat model as a modular framework that can inject adversarial manipulations at different points without conflating them with downstream failures. The framework is designed to simulate realistic enterprise agent deployments where many semantically similar skills coexist, forcing retrieval and planning components to disambiguate among overlapping capabilities. This allows evaluation of structural repository attacks, semantic poisoning of metadata, planner deception through misleading descriptions or permissions, execution-time workflow exploitation, and malicious drift through later skill updates. The overall methodological novelty is the separation of security analysis by lifecycle stage, enabling apples-to-apples study of how different trust assumptions fail and how defenses should be placed earlier in the skill pipeline rather than only at execution time.

### 🧪 Experiments
The evaluation uses a repository of 327 real-world skills and constructs dense, enterprise-style skill collections with semantically overlapping capabilities and representative user workloads. The benchmark includes lifecycle-specific attack scenarios covering repository admission, semantic retrieval, planner selection, execution, and skill evolution. Although the extracted text does not provide the full metric table, the study is described as a comprehensive empirical comparison of representative attacks under standardized protocols, enabling direct measurement of attack success and security failures at the stage where they originate. The experimental emphasis is less on a single model-vs-baseline comparison and more on systematic cross-stage benchmarking: which attacks can enter the repository, which can bias retrieval, which can mislead the planner, which can exploit workflow composition at runtime, and which emerge after later updates. The main reported result is qualitative but important: vulnerabilities are observed at multiple lifecycle stages beyond execution, and semantic manipulation stages remain especially exposed, showing that runtime-only defenses are insufficient.

### 🌱 Insights & Implications
A central insight is that reusable skills combine two hard security problems: conventional software artifact trust and natural-language-mediated reasoning trust. Existing supply-chain techniques such as provenance, signing, and integrity validation may ensure that a skill package is authentic, but they do not guarantee that metadata is truthful, retrieval embeddings are not manipulated, or planner-facing descriptions are not deceptive. This creates a distinct class of semantic supply-chain attacks for LLM agents. For software engineering and security researchers, the paper suggests that secure agent ecosystems need lifecycle-aware controls such as admission vetting, semantic consistency checks, retrieval robustness, planner-grounding validation, and continuous monitoring of skill updates. A limitation of the presented evidence is that the extracted text emphasizes taxonomy and framework design more than concrete defense efficacy; it establishes the problem and evaluation structure, but leaves open which mitigations are most effective under realistic deployment constraints. Another open question is how well the benchmark generalizes across agent frameworks, embedding models, and planner architectures, since retrieval and selection behavior may vary substantially by implementation.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, maintenance, vulnerability detection, and vulnerability repair because it reframes reusable agent skills as a new kind of software supply-chain artifact with both code-level and semantic-level vulnerabilities. It advances the researcher's work by providing a structured threat model and an evaluation framework that can inspire new methods for detecting malicious or misleading skills, verifying their evolution over time, and designing automated repair or hardening techniques for agent ecosystems.

## 📄 [PROBE: Benchmarking Code Generation in Large Language Models](https://arxiv.org/abs/2607.13820v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Rodrigo Pato Nogueira, Marco Vieira, João R. Campos
📅 **Published:** 2026-07-15  🏷️ **Categories:** cs.SE

### 💡 TL;DR
PROBE is a systematic and extensible benchmark for evaluating LLM-based code generation beyond simple unit-test pass rates, adding measures for solution proximity and code quality across multiple languages and prompting strategies. Using PROBE on six models over five languages, the authors show that feedback-based refinement helps, Python/C++/Java are easier than C/Rust, and many failures come from basic but risky coding mistakes.

### 🏆 Key Contributions
- Introduces PROBE, a benchmark framework for text-to-code generation with a clear structure spanning workload, metrics, prompts, and experimental procedure.
- Evaluates generated code along three complementary dimensions: functional correctness, proximity to reference solutions, and code quality.
- Provides a multilingual, multi-model, multi-prompt evaluation covering six LLMs, five programming languages, and three prompting strategies.
- Offers extensibility guidelines for adding new workloads, metrics, and prompting methods.
- Includes a qualitative error analysis identifying recurring failure modes such as missing imports, overflow bugs, out-of-bounds accesses, and non-scalable brute-force solutions.
- Releases a public project page/dataset to support reproducibility, transparency, and future benchmark extensions.

### ⚙️ Methodology
PROBE is designed as a benchmark framework rather than a new code-generation model. Its methodology follows benchmarking principles: define a representative workload, specify multiple evaluation dimensions, control prompting conditions, and use a repeatable procedure across models and languages. The workload is drawn from IBM CodeNet and spans five languages: Python, C++, Java, C, and Rust, chosen to reflect different syntax and different levels of representation in model training corpora. For each programming task, the benchmark queries models under three prompting strategies: baseline prompting with no extra context, 1-shot in-context learning, and a feedback-incorporation strategy with two rounds of iterative refinement. Evaluation is performed along three dimensions. First, functional correctness is measured using execution-oriented outcomes such as successful compilation or interpretation and the fraction of unit tests passed, consistent with common code-generation evaluation practice. Second, proximity to valid solutions is measured with similarity-based metrics such as CodeBLEU, to capture how close an incorrect answer is to a correct reference implementation. Third, code quality is assessed using static quality indicators such as code size and cyclomatic complexity to compare generated code against human-written solutions. Beyond aggregate scoring, the framework also includes manual/qualitative inspection of generated outputs to categorize common error types and illustrate model weaknesses. The framework emphasizes reproducibility and extensibility, with clear guidance for incorporating new languages, workloads, metrics, and prompting variants.

### 🧪 Experiments
The empirical study evaluates six LLMs: four open-source models (Qwen2.5:14b, Qwen2.5-Coder:14b, Qwen2.5-Coder:7b, Deepseek-Coder-v2:16b) and two proprietary models (GPT-4.1-mini and Gemini-2.0-flash). This paper surveys how AI agents let users set and enforce permissions, proposing a taxonomy that spans user interfaces, internal policy representations, policy derivation, and runtime enforcement. Across 21 research systems and 5 commercial agents, it finds no existing approach fully achieves low user burden, expressive/formal policies, and deterministic enforcement at once, leaving major gaps for secure user-controlled agent design.

### 🏆 Key Contributions
- A survey of 21 agentic security proposals that support user-level permissions, covering research systems from 2024-2026 and one open-source implementation
- A taxonomy of agent permission systems organized around four stages: user-facing policy specification, internal policy representation, derivation of internal policies from user choices, and runtime enforcement
- A comparative analysis of five commercial AI agents against the research literature, highlighting where deployed systems underperform on transparency, controllability, and principled enforcement
- Identification of recurring design tensions and unmet needs, especially the tradeoff between minimizing permission fatigue, supporting user-specific policies, and enforcing policies deterministically under LLM uncertainty
- A research agenda for future agent permission systems, including better support for policy evolution over time, stronger enforcement mechanisms, and more usable user-facing controls

### ⚙️ Methodology
The paper is a structured literature review and comparative systems analysis rather than a new algorithmic proposal. The authors first define scope to include LLM-based agents that autonomously plan or trigger external tool use and that expose some form of user-level permission or policy control; systems with only global/product-level controls or without tool use are excluded. They review 21 relevant systems/proposals from 2024-2026 and extract design features related to three research questions: what interfaces are exposed to users, how internal policies are represented and derived from user input, and how permissions are enforced at runtime. From this coding process they build a taxonomy with several dimensions: user interface style for permissions specification, internal policy structure, translation from user intent to internal policies, and enforcement mechanism including the degree of dependence on LLM judgments versus deterministic checks. They then apply the same taxonomy to five prominent commercial AI agents, comparing deployed practice to the research literature. The analysis is qualitative and synthesizing: it identifies common goals such as least privilege, fail-closed behavior, minimizing user overhead, policy formalization, and deterministic enforcement, then studies where systems succeed or fail against these goals. The paper also situates agent permissions within prior work on ACLs, capabilities, information-flow control, mobile permission models, guardrails, and privilege-separated multi-LLM architectures, using these as conceptual lenses for the taxonomy and gap analysis.

### 🧪 Experiments
There is no benchmark-style experimental evaluation in the machine learning sense. Instead, the empirical component is a comparative survey of 21 research proposals and 5 commercial agents. The 'setup' consists of selecting in-scope systems that support user-level permissions, analyzing them with a common taxonomy, and comparing their design choices across UI, policy representation, policy derivation, and enforcement. The paper does not report standard quantitative metrics such as accuracy, precision/recall, exploit success rate, or latency; rather, the main results are qualitative comparative findings. The key outcome is that none of the surveyed research systems simultaneously delivers all three desired properties the authors emphasize: low user overhead, formal/expressive policy specification, and deterministic runtime enforcement. Commercial agents perform worse overall, often forcing a choice between frequent user-in-the-loop approvals and opaque LLM-based auto-review mechanisms with limited transparency and user control. A notable practical advantage of commercial systems is support for modifying permissions over time, which the literature often neglects.

### 🌱 Insights & Implications
A central insight is that agent permissions are not just an enforcement problem but a joint human-computer interaction and security design problem: the system must capture user intent precisely enough to support least privilege without overwhelming users into over-permissioning. Systems that rely on LLMs for policy interpretation or enforcement may preserve flexibility but weaken predictability, auditability, and fail-closed guarantees; more deterministic approaches improve assurance but often increase configuration burden or reduce agent utility. Another important observation is the mismatch between literature and practice: commercial agents commonly expose simplistic approval flows or opaque auto-approval heuristics rather than principled, user-tailorable policy models. The paper also highlights an underexplored lifecycle issue: permissions should evolve over time as user trust, context, and tasks change, yet most research treats policies as static. Limitations likely include the rapid evolution of the field, a bounded corpus focused on recent systems, qualitative rather than quantitative comparison, and potential ambiguity in reverse-engineering commercial agent behavior from exposed interfaces. For related research, the paper suggests opportunities in formal policy languages for agents, usable policy authoring and explanation, policy learning with verifiable constraints, and stronger runtime enforcement resilient to prompt injection and hallucination.

### 🎯 Why This Paper

## 📄 [Evaluating Frontier AI Agents as Autonomous Clinical Security Auditors](https://arxiv.org/abs/2607.13411v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Michael O. Eniolade
📅 **Published:** 2026-07-15  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
This paper introduces an open benchmark that tests whether frontier AI agents can autonomously perform end-to-end security audits of clinical ML models by implementing four attacks, aggregating results into a Security Posture Score, and producing a structured report. In 54 runs across six task variants, Claude Sonnet 4.6 and GPT-4.1 achieved perfect completion, while GPT-4o succeeded on 61% of runs and failed mainly due to execution-discipline issues rather than lack of general reasoning.

### 🏆 Key Contributions
- An open METR Task Standard v0.3.0 evaluation for autonomous clinical AI security auditing, with six variants spanning two clinical datasets and three model-defense levels
- A reproducible 54-run empirical comparison of three frontier AI agents on a realistic, multi-step security-audit implementation task
- A taxonomy of three practical agent failure modes: premature session termination, arithmetic aggregation error, and empty submission generation
- Public release of the task infrastructure, scoring code, and Wisconsin Diagnostic Breast Cancer assets for rerunnable benchmarking

### ⚙️ Methodology
The paper frames clinical ML security auditing as an autonomous agent task executed inside a Docker container through a bash-only interface, with no scaffolding code. Each task instance provides a pretrained clinical prediction model, a patient dataset, and written instructions specifying a structured audit. The agent must implement four security evaluation components from pseudocode: (1) FGSM robustness, which requires fitting a surrogate model to estimate gradients, perturbing inputs within a defined epsilon budget, recomputing AUROC on adversarial examples, and mapping the outcome to a component score; (2) membership inference resistance, which requires training four shadow models on overlapping subsets, aggregating their outputs into an attack classifier, querying the target model, and computing attack accuracy against ground-truth membership labels; (3) expected calibration error, computed using ten uniform-width probability bins and a weighted confidence-accuracy deviation with task-specific scaling; and (4) boundary attack resistance, which performs iterative walks from 50 opposite-class starting points toward the target class and records the step at which the decision boundary is crossed, up to a maximum of 100 steps. The agent then aggregates the four components into a single Security Posture Score using a weighted formula defined by the companion SPS framework, writes a structured JSON report to `submission.txt`, and must complete all work within a 40-turn interaction budget. The evaluation is built on METR Task Standard v0.3.0, uses locked reference solutions and automated scoring, and assesses whether agentic coding systems can reliably execute a realistic security-audit workflow without human intervention.

### 🧪 Experiments
The evaluation covers six task variants derived from two clinical datasets: Wisconsin Diagnostic Breast Cancer and MIMIC-IV ICU mortality. For each dataset, three model architectures with increasing defense strength are used, yielding reference Security Posture Scores from 55.60 to 90.41. Three frontier models are tested: Claude Sonnet 4.6, GPT-4.1, and GPT-4o. Each model is run three times per variant, producing 54 total runs and 18 runs per model. Success is measured by automated evaluator score, task completion, numerical correctness relative to the locked reference, submission validity, and efficiency indicators such as token usage and API cost. Claude Sonnet 4.6 and GPT-4.1 completed all 18/18 runs with perfect evaluator scores of 1.0. GPT-4o completed 11/18 runs, for a 61% completion rate. The paper reports that GPT-4o used substantially more tokens per run than Claude, though tokenization is not directly comparable across providers; in cost terms, 18 runs cost about $8 for GPT-4.1, $12 for Claude Sonnet 4.6, and $27 for GPT-4o. GPT-4o’s failures clustered into three categories: early termination before writing files, incorrect weighted aggregation of component scores, and generation of an empty submission file. These results suggest that once a model is capable of implementing the attacks, reliable completion depends heavily on long-horizon execution control, bookkeeping, and output-format discipline.

### 🌱 Insights & Implications
A key insight is that autonomous security auditing is now feasible for top-tier coding agents on moderately complex clinical ML tasks, but success depends not only on technical knowledge of attacks but also on execution discipline under constrained interaction budgets. The benchmark is especially valuable because it moves beyond static code or reasoning tests into realistic secure-ML tooling tasks involving debugging, numerical aggregation, and structured artifact generation. For software engineering and AI-agent evaluation, the paper highlights that failures can occur at the orchestration layer even when the core attack implementations are within model capability. Limitations include the small number of runs per variant, the restricted set of three evaluated agents, the use of only two datasets and three model-defense levels, and dependence on a companion preprint for the SPS weighting scheme rather than an independently validated scoring framework. The task also focuses on four specific attack surfaces, leaving out stronger robustness attacks, broader privacy attacks, or repair/remediation actions after vulnerability discovery. An open question is whether similarly strong performance would hold on larger codebases, more heterogeneous clinical models, stronger adversaries, or tasks that require not just auditing but patching insecure models. Another implication is that benchmark design for agentic systems should explicitly measure submission integrity, aggregation correctness, and termination behavior, since these are practical failure modes in autonomous security pipelines.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, software maintenance, software security, and vulnerability analysis because it operationalizes security auditing as an end-to-end autonomous coding task rather than a static prediction benchmark. It is especially useful for work on automated vulnerability detection and repair pipelines: although the paper focuses on detection/auditing rather than remediation, it exposes the exact implementation, orchestration, and reliability challenges an AI agent must solve before it can be trusted to assess or eventually repair security weaknesses in deployed ML systems. It also offers a concrete benchmark and failure taxonomy that can inform future research on secure agentic software engineering, autonomous testing, and AI-assisted maintenance of high-stakes clinical systems.

## 📄 [Faithful Autoformalization of Natural Language Assertions](https://arxiv.org/abs/2607.13303v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Hongyi Liu, Madhusudan Parthasarathy, Adithya Murali
📅 **Published:** 2026-07-14  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
The paper presents Monty, a framework for translating natural-language software assertions into formal executable contracts while explicitly handling ambiguity and the possibility that an assertion may be either valid or invalid for the given code. By combining LLM-generated candidates with testing-based validity checks and a new clause-level conformance metric, Monty substantially improves precision over naive LLM autoformalization, with gains of up to about 20 percentage points.

### 🏆 Key Contributions
- Introduces Monty, an assertion autoformalization pipeline that does not assume the natural-language assertion is valid on the target implementation.
- Proposes a novel conformance metric based on clausal coverage, where a formalization is translated back to natural language and clause-by-clause matched against the original assertion.
- Combines conformance scoring with testing-derived validity scores to rank and filter candidate formal assertions.
- Produces both likely valid and likely invalid formalizations to better handle ambiguity in natural-language assertions.
- Explores active-learning style disambiguation by querying the user when both valid and invalid interpretations remain plausible.
- Builds and evaluates benchmark suites of 541 assertion-generation tasks over 22 Java collection-like classes, including synthetic and manually written specifications.

### ⚙️ Methodology
Monty targets module-level postconditions/contracts for object-oriented Java code, especially collection-like classes with side-effecting methods and pure observer methods. The input consists of a code skeleton, observer-method vocabulary, the target method signature, and a natural-language assertion attached to the method. Monty first prompts an LLM to generate multiple candidate formal assertions in a JML-like specification language expressive enough to capture Boolean combinations, bounded quantification, multiple sorts, and references to pre-state values. Because raw LLM outputs may be syntactically malformed, semantically invalid, or unfaithful to the original intent, Monty subjects each candidate to two main analyses. The first is a validity score derived from program analysis through testing: candidates are checked for syntactic/semantic well-formedness, then evaluated against automatically generated tests to determine whether they are safe to execute and whether they hold or fail on the implementation. This allows Monty to reason about whether a candidate behaves like a likely valid assertion or a likely invalid one, rather than assuming all intended assertions should pass. The second is a conformance score computed using the paper’s clausal coverage technique. Monty translates the candidate formal assertion back into natural language, decomposes both the original and back-translated specifications into clauses, and uses an LLM to judge bidirectional clause correspondence: whether every clause in the original is reflected in the formalization and whether every clause introduced by the formalization is supported by the original. This bidirectional matching yields a score estimating how faithfully the candidate captures the natural-language intent while penalizing omissions and hallucinated conditions. Monty then combines conformance and validity signals to rank candidates and select the most likely faithful formalization(s). A distinctive feature is that it can return two interpretations when ambiguity remains: the most likely valid formalization and the most likely invalid formalization. When both interpretations are sufficiently plausible, Monty uses active learning by asking targeted follow-up questions to the user to disambiguate the intended meaning. Overall, the method is designed for reliable, precision-oriented autoformalization rather than maximizing raw generation throughput.

### 🧪 Experiments
The evaluation uses benchmark suites totaling 541 natural-language/formal-specification pairs drawn from 22 Java classes representing collection-like data structures. The datasets include both synthetically generated and manually written natural-language assertions, and importantly include both assertions that are valid for the implementation and assertions that are intentionally invalid. The main comparison is against naive LLM-based assertion translation without Monty’s filtering/ranking pipeline, with ablations over backbone LLM choice, conformance-checking strategy, and active learning. The paper emphasizes precision as the primary reliability metric, while also tracking recall to ensure the system does not simply abstain aggressively. Reported results show consistent precision improvements: for example, when paired with Qwen2.5-Coder 32B, Monty raises precision from 75% to 91.6% on one dataset and from 64% to 85% on another. The gains are described as up to roughly 20 percentage points on average relative to naive LLM translation, while maintaining high recall. The ablation studies indicate that the proposed clausal coverage conformance method outperforms simpler baseline conformance-checking approaches similar to those used in prior autoformalization work, and that Monty is especially helpful when the backbone model is smaller or less reliable.

### 🌱 Insights & Implications
A central insight is that assertion autoformalization for software engineering differs from many prior formalization settings because one cannot assume the specification is intended to be true on the current code; this is crucial for testing, debugging, and validating AI-generated code. The paper’s use of testing as a semantic filter is therefore highly relevant: executable behavior helps distinguish plausible valid and invalid interpretations that purely text-based translation would conflate. The clausal coverage idea is also notable because it operationalizes faithfulness rather than only syntactic similarity or execution success, addressing a major failure mode of LLM-generated specifications that sound plausible but omit or alter conditions. For researchers in program repair and vulnerability detection/repair, the work suggests a path toward converting developer intent into executable or verifiable oracles that could guide repair or detect mismatches between intended and implemented behavior. Likely limitations include dependence on the quality and coverage of automatically generated tests, restriction to localized module-level assertions rather than broader system requirements, and dependence on observer-method vocabularies and code skeleton quality. Another open question is how well the approach scales beyond collection-like APIs, simple object-state properties, and the studied JML-style assertion language, especially for security properties, temporal behavior, side effects across multiple objects, or under-specified natural-language descriptions. The active-learning component is promising, but its practical cost-benefit tradeoff and user experience in real development workflows likely need further study.

### 🎯 Why This Paper
This paper is highly relevant to software engineering, maintenance, verification, automated repair, and security-oriented research because it turns informal developer assertions into executable formal contracts with an explicit focus on reliability and low false interpretation rates. Its combination of LLMs, semantic validation through testing, and ambiguity-aware ranking could directly support assertion generation for regression testing, oracle construction for automated program repair, specification-based bug finding, and validation of AI-generated or security-critical code.
