## 📄 [EVMbench: Evaluating AI Agents on Smart Contract Security](https://arxiv.org/abs/2603.04915v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Justin Wang, Andreas Bigger, Xiaohai Xu, Justin W. Lin, Andy Applebaum, Tejal Patwardhan, Alpin Yukseloglu, Olivia Watkins
📅 **Published:** 2026-03-05  🏷️ **Categories:** cs.LG, cs.AI, cs.CR

### 💡 TL;DR
EVMbench is a benchmark and evaluation harness for measuring how capable AI agents are at smart contract security tasks end-to-end: detecting vulnerabilities, patching them without breaking functionality, and exploiting them against a live local Ethereum chain. Across 117 real high-severity vulnerabilities, the authors find frontier agents can already discover and execute full exploit chains on realistic deployments, motivating ongoing measurement and defenses.

### 🏆 Key Contributions
- Introduces EVMbench, a 3-mode evaluation (Detect/Patch/Exploit) for smart contract security with an emphasis on comprehensive coverage across all high-severity issues in a repository, not single-bug success
- Curates a dataset of 117 high-severity, fund-loss vulnerabilities from 40 real-world audit repositories (primarily Code4rena) plus additional Tempo-related scenarios, with oracle exploits and patches for verification
- Provides fully programmatic grading for Patch and Exploit: Patch uses hidden exploit tests plus regression checks on original test suites; Exploit uses on-chain state/event/balance checks on a local EVM
- Builds a Rust-based transaction re-execution and replay framework to reproduce and validate agent-submitted exploit transactions while reducing opportunities for grader circumvention
- Evaluates multiple frontier agents across all three modes and reports qualitative failure modes and rollout analyses; releases the benchmark code, tasks, and tooling

### ⚙️ Methodology
EVMbench constructs per-audit sandboxed environments that mirror the repository context a human smart-contract auditor would receive (Solidity code, project structure, scope list, sponsor guidance, and known-issues/automated findings that are out-of-scope). Agents run inside an isolated Ubuntu 24.04 Docker container with required dev tooling (e.g., Foundry) and without web/internet access; grading runs in a separate container inaccessible to the agent.

Task suite / curation: Vulnerabilities are sourced mainly from Code4rena post-contest reports, filtered to high-severity issues that can directly lead to loss of user/platform funds. Additional scenarios come from Tempo blockchain auditing to cover agentic payments/stablecoin-related contexts. Final suite: 117 vulnerabilities across 40 audits; codebase sizes span ~10^2–10^4 sLOC with varying contract counts.

Evaluation modes:
1) Detect (audit/reporting): Agent produces an audit report describing high-severity vulnerabilities. Grading uses the ground-truth audit report as the reference and a model-based judge to determine, for each ground-truth vulnerability, whether the agent’s report contains it. Score is recall (% of ground-truth vulns found). Authors also map findings to historical audit payouts to derive an approximate “award” signal.

2) Patch (repair): Agent edits the vulnerable repository directly. Grader applies the modifications, then runs the project’s existing tests to ensure functionality is preserved. Some original tests may encode vulnerable behavior; those can be exempted (the paper notes such cases). To reduce cheating, the grader resets test files the agent is not allowed to modify before running tests. Security effectiveness is then checked using unseen exploit tests (added only in the grader) that should fail post-patch; success is computed per vulnerability (binary success/failure) based on whether the hidden exploit tests are prevented while acceptable functionality remains.

3) Exploit (offense/end-to-end): The benchmark deploys the vulnerable contracts to a live local Ethereum execution environment and provides the agent with an RPC endpoint plus a funded private key. The agent must execute an end-to-end exploit via transactions that induce the intended malicious state transition (e.g., draining funds). Grading is programmatic and based on on-chain artifacts such as balance deltas and emitted events. To ensure reproducibility and prevent cheating, a Rust-based re-execution framework replays the agent’s transaction sequence against a controlled local node and checks vulnerability-specific success conditions over resulting chain state.

### 🧪 Experiments
Setup: Agents are evaluated across three modes (Detect, Patch, Exploit) on the EVMbench suite comprising 117 vulnerabilities from 40 repositories. Each task runs in a no-internet Docker sandbox with the original repository context and tooling. Grading is separated into an isolated grader container.

Datasets/tasks: 40 audited repositories; 117 curated high-severity vulnerabilities (primarily Code4rena, plus Tempo-related scenarios). Repository scope sizes range from ~106 to ~10,108 sLOC (median ~1,296; mean ~2,045) and 1 to 116 contracts (median ~7; mean ~16).

Baselines/agents: The paper states evaluation of “a range of frontier agents”; specific model names and their exact comparative rankings/results are not contained in the provided excerpt.

Metrics:
- Detect: vulnerability recall against ground-truth audit reports; also a mapped-to-payout “award” metric based on historical audit payouts.
- Patch: per-vulnerability success/failure based on (a) regression tests still passing (with specified exemptions) and (b) hidden exploit tests failing to exploit post-patch.
- Exploit: per-task success based on blockchain state checks (events, balances) after replaying agent transactions.

Key results (as stated in excerpt): Frontier agents can perform end-to-end vulnerability discovery and exploitation against live local blockchain instances; the benchmark demonstrates that agentic exploitation is already feasible in realistic EVM settings. Quantitative performance numbers are not included in the excerpt provided.

### 🌱 Insights & Implications
Implications: EVMbench operationalizes smart-contract security as an agentic, end-to-end capability with deterministic verification—useful both for measuring defensive progress (detection/repair) and for monitoring offensive risk (exploitation). The Exploit mode in particular aligns with real attacker workflows (RPC interactions, stateful multi-step transactions, economic objectives) and provides strong, programmatic success signals.

Notable design choices: Emphasis on comprehensive coverage (finding/fixing all relevant high-severity issues in a codebase) combats “single-success” evaluations that can overestimate real auditing competence. Separating agent container from grader and using transaction re-execution plus resetting protected tests are practical anti-cheating measures.

Limitations / open questions:
- Detect grading depends on a model-based judge; although the paper references an appendix analyzing it, this introduces potential false positives/negatives and sensitivity to report phrasing.
- Patch grading depends on the adequacy of existing tests plus the authors’ hidden exploit tests; patches that pass tests may still be insecure under unmodeled conditions, and conversely correct mitigations might be penalized if they violate implicit assumptions in the repo.
- Exploit tasks run in a simplified local environment without adversarial mempool dynamics, MEV competition, or reactive defenders; real “dark forest” conditions could change exploit feasibility and required strategies.
- Generalization is unclear: performance on curated high-severity issues from Code4rena may not transfer to novel protocols, different EVM dialects, or cross-chain systems.
- Coverage of vulnerability classes and bias toward publicly-audited codebases may skew the distribution of bugs toward those typical of audit contests.

Research opportunities:
- Training/evaluating automated program repair systems for Solidity with constraints from regression tests plus security properties.
- Developing stronger, less judge-dependent detection metrics (e.g., mapping findings to standardized vulnerability schemas and code locations).
- Extending exploit evaluation to include realistic mempool ordering/MEV, partial observability, or multi-agent adversaries/defenders.

### 🎯 Why This Paper
This paper squarely targets vulnerability detection, vulnerability repair, and exploitation in a realistic software-security setting with programmatic, reproducible grading—directly aligning with research on vulnerability detection/repair and automated program repair. EVMbench’s Patch mode provides a concrete APR-style evaluation loop (regression + hidden security tests), while the Exploit mode offers an unusually rigorous way to validate true exploitability via on-chain state checks, making it a valuable benchmark for building and measuring secure code-assistance agents for smart contracts.

## 📄 [CodeTaste: Can LLMs Generate Human-Level Code Refactorings?](https://arxiv.org/abs/2603.04177v1)  `90% relevant`

👤 **Authors:** Alex Thillen, Niels Mündler, Veselin Raychev, Martin Vechev
📅 **Published:** 2026-03-04  🏷️ **Categories:** cs.SE, cs.AI, cs.LG

### 💡 TL;DR
The paper introduces CodeTaste, a benchmark of real-world, multi-file refactoring changes mined from open-source repositories, and evaluates whether LLM coding agents can both apply refactorings correctly and choose the same refactorings humans chose. Results show frontier models can implement well-specified refactorings but struggle to discover human-preferred refactorings from high-level guidance; a propose-then-implement strategy improves alignment.

### 🏆 Key Contributions
- CodeTaste benchmark: a curated set of realistic refactoring tasks mined from large-scale, multi-file refactoring commits in open-source repositories
- An evaluation methodology combining existing repository test suites (behavior preservation) with custom static checks that enforce removal/introduction of structural patterns, using dataflow reasoning
- Empirical evidence of a capability gap: strong performance when refactoring steps are explicitly specified versus weak performance when the agent must infer which refactoring a human would pick from only a focus area
- A decomposition approach (propose-then-implement) that improves alignment with human refactoring choices, plus an additional gain from selecting the best-aligned proposal prior to implementation
- Positioning CodeTaste as an evaluation target and potential preference/alignment signal to steer coding agents toward human-like maintainability improvements

### ⚙️ Methodology
The study frames refactoring as a behavior-preserving transformation problem with two sub-tasks: (1) reliable execution of refactorings and (2) discovery/selection of refactorings that match human decisions. CodeTaste tasks are mined from real open-source repositories by identifying multi-file changes interpreted as refactorings (i.e., structural improvement without intended behavioral change). Each task includes a before-state and an expected refactoring target characterized by (a) tests that should continue to pass and (b) pattern-based constraints describing what anti-patterns should be removed and what improved structures should be introduced. Scoring is done by running the repository test suite for behavioral equivalence and applying custom static analyses with dataflow reasoning to detect whether undesired constructs were eliminated and desired constructs introduced (e.g., changes consistent with extracting/centralizing logic, reducing duplication, improving modular structure). The evaluated agent workflows include: (i) direct implementation given detailed refactoring instructions, (ii) implementation given only a high-level focus area (requiring the model to infer the human’s chosen refactoring), and (iii) a propose-then-implement decomposition where the model first proposes candidate refactoring plans and then implements one; an additional variant selects the proposal most aligned with the benchmark’s target signal before implementation.

### 🧪 Experiments
The experiments evaluate multiple frontier LLM coding agents/models on CodeTaste tasks in a repository context. Conditions include (A) detailed refactoring specification versus (B) only a focus area for improvement, isolating execution ability from refactoring discovery/choice. Solutions are judged using (1) pass/fail on repository tests and (2) pass/fail on custom static checks (dataflow-based) that confirm removal of undesired patterns and introduction of desired patterns; overall success likely requires satisfying both. Key qualitative findings reported: models/agents perform well in condition (A) but frequently fail in condition (B), demonstrating a gap in discovering the human-selected refactoring. The propose-then-implement decomposition improves alignment in (B), and selecting the best-aligned proposal before implementation yields further improvements. The abstract does not provide exact dataset size, model list, baselines, or numeric success rates.

### 🌱 Insights & Implications
Implications: (1) Refactoring discovery (choosing what to change) is a distinct and harder problem than refactoring execution (making a specified change), suggesting future agents need stronger planning/objective formation for maintainability. (2) Benchmarks that rely only on tests can over-credit behavior-preserving but structurally poor edits; adding static, pattern- and dataflow-aware checks better captures maintainability goals. (3) The propose-then-implement result suggests that explicitly separating planning from coding can improve human-alignment in code maintenance tasks and may benefit agent training (e.g., preference optimization over plans). Limitations/open questions inferred from the abstract: mining refactoring tasks from commits may include confounds (mixed-purpose commits, incidental behavior changes) and may bias toward refactorings that are easily detectable by static pattern checks; repository test suites can be incomplete, so behavior preservation may be under-constrained; static checks encode a partial notion of 'good' refactoring and may miss subjective/architectural intent; the best-aligned proposal selection implies access to an alignment signal—how to approximate this at deployment time remains open.

### 🎯 Why This Paper
Highly relevant to software maintenance and automated code improvement: it operationalizes real-world refactoring as an evaluable task with stronger signals than tests alone, directly targeting maintainability debt that LLM-generated code can accumulate. While not focused on vulnerability repair per se, the methodology (repo-context edits, behavior-preservation plus static constraints, plan-then-apply decomposition) is transferable to automated program repair and security patching, where choosing the right fix strategy and validating non-functional properties are central challenges.

## 📄 [Code Fingerprints: Disentangled Attribution of LLM-Generated Code](https://arxiv.org/abs/2603.04212v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Jiaxun Guo, Ziyuan Yang, Mengyu Sun, Hui Wang, Jingfeng Lu, Yi Zhang
📅 **Published:** 2026-03-04  🏷️ **Categories:** cs.SE, cs.CL

### 💡 TL;DR
This paper tackles model-level attribution of LLM-generated code (identifying which LLM produced a snippet) by learning representations that separate task semantics from model-specific stylistic “fingerprints.” It proposes DCAN, a disentanglement + contrastive-learning framework, and releases a 91,804-sample benchmark spanning 4 LLMs and 4 programming languages, showing reliable multi-class attribution across settings.

### 🏆 Key Contributions
- Formulates LLM Code Source Attribution (LLMCSA) as a practical multi-class provenance problem beyond human-vs-machine detection, motivated by forensics, vulnerability triage, and compliance auditing.
- Introduces DCAN (Disentangled Code Attribution Network) that explicitly decomposes code embeddings into source-agnostic (task/semantics) and source-specific (LLM fingerprint) components and performs attribution using only the source-specific part.
- Proposes a contrastive objective to align source-agnostic representations across models for the same task, thereby isolating residual model-dependent signals for attribution.
- Constructs and releases (with code) the first large-scale LLMCSA benchmark: 91,804 generated samples from DeepSeek, Claude, Qwen, and ChatGPT across Python/Java/C/Go under two settings (with vs. without comments).

### ⚙️ Methodology
Problem: given code snippet c generated to solve a task t, predict the source model y among K candidates. The core hypothesis is that an encoder representation h can be decomposed additively into z_c (source-agnostic, task-driven semantics) and z_s (source-specific, model fingerprint): h = E(c) = z_c + z_s, with approximate invariances P(z_c|t,y)≈P(z_c|t,y′) and P(z_s|t,y)≈P(z_s|t′,y). DCAN pipeline: (1) Feature extraction uses a pretrained code encoder (UniXcoder, hidden size 768). Code is tokenized, prepended with [CLS], passed through the Transformer encoder, and the [CLS] embedding h_base is taken as the snippet representation. (2) Disentanglement module splits h_base into two latent vectors meant to capture z_c and z_s. The paper’s description indicates an explicit optimization that (a) encourages z_c to be aligned for snippets generated for the same task across different source models, and (b) encourages z_s to retain discriminative information about the source model while being insensitive to task semantics. (3) Contrastive learning objective: uses positive pairs that share the same task but come from different LLMs to pull their z_c closer (task alignment), while negatives are from different tasks (push apart), thereby reducing model cues in z_c. The remaining model-dependent information is expected to concentrate in z_s. (4) Attribution head: a multi-class classifier is trained with cross-entropy on z_s only (not on z_c), to enforce reliance on fingerprints rather than problem semantics. Training jointly optimizes the attribution loss and the disentanglement/contrastive objectives to yield separable components.

### 🧪 Experiments
Dataset/benchmark: 91,804 code samples generated by 4 LLMs (DeepSeek, Claude, Qwen, ChatGPT) in 4 languages (Python, Java, C, Go), collected for a set of programming tasks under two generation settings: with comments and without comments (to test whether comments act as strong attribution cues and whether methods generalize when comments are removed). Models/baselines: DCAN uses UniXcoder as the encoder backbone; the paper positions baselines as conventional supervised attribution/detection approaches that do not disentangle task semantics from source fingerprints (e.g., directly classifying from the encoder embedding) and contrasts with feature-engineered stylometry approaches in prior work. Metrics: multi-class attribution performance (typically accuracy and/or macro-F1; the paper describes “reliable attribution performance across diverse settings,” implying evaluation across languages and comment settings). Key results: DCAN is reported to outperform non-disentangled approaches, and to maintain strong attribution across multiple programming languages and both comment regimes, supporting the claim that the learned z_s captures stable model-level fingerprints rather than task solutions or superficial cues. (The provided excerpt does not include exact numerical values or full baseline list; refer to the paper/GitHub for tables.)

### 🌱 Insights & Implications
Implications for software forensics: model-level provenance is feasible without watermarking, by exploiting subtle but learnable stylistic/structural preferences of different LLMs, even when functional semantics are similar. Key research insight is that direct classification can overfit to task semantics; explicitly aligning task-driven content across models via contrastive objectives can force the network to allocate discriminative capacity to true source-specific artifacts. Limitations/open questions: (1) Closed-world assumption: classification among a fixed set of known LLMs; robustness to unseen/new models (open-set attribution) is not established in the excerpt. (2) Sensitivity to prompting/decoding: fingerprints may change with temperature, system prompts, tool use, or post-editing; it remains unclear how DCAN handles adversarial paraphrasing, refactoring, formatting tools, or human-in-the-loop edits. (3) Data collection bias: if tasks, prompts, or quality-control steps differ systematically per model, the classifier may pick up dataset artifacts rather than intrinsic fingerprints; strong controls and cross-prompt evaluations are important. (4) Security/abuse considerations: attribution methods can be attacked (obfuscation, style transfer, transpilation) or used to deanonymize; evaluating against obfuscation and semantic-preserving transformations would strengthen the security story. (5) Relationship to vulnerability research: if certain LLMs have characteristic insecure patterns, provenance could guide triage, but it also risks conflating “model identity” with “code quality” unless carefully studied.

### 🎯 Why This Paper
The work is directly relevant to software security governance and incident/vulnerability workflows where knowing the generating LLM can affect triage, accountability, licensing audits, and risk assessment. For automated vulnerability detection/repair research, DCAN-style disentanglement provides a reusable technique to separate functional semantics from generator-specific artifacts—useful for building robust detectors/repair models that generalize across LLMs, for attributing vulnerable code to a particular generation pipeline, and for studying how different LLMs correlate with vulnerability patterns and patch styles.

## 📄 [Robustness of Agentic AI Systems via Adversarially-Aligned Jacobian Regularization](https://arxiv.org/abs/2603.04378v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Furkan Mumcu, Yasin Yilmaz
📅 **Published:** 2026-03-04  🏷️ **Categories:** cs.LG, cs.AI, cs.CR, cs.MA

### 💡 TL;DR
The paper proposes Adversarially-Aligned Jacobian Regularization (AAJR), a directional alternative to global Jacobian/Lipschitz constraints for minimax (GDA/PGD) training of agentic policies, aiming to stabilize the inner maximization without broadly suppressing policy expressivity. It provides theoretical results showing AAJR induces a strictly larger admissible policy class than global Jacobian bounds (under mild conditions) and derives step-size conditions that stabilize inner-loop ascent by controlling trajectory-wise effective smoothness.

### 🏆 Key Contributions
- Formalizes the agentic minimax training bottleneck: global Jacobian bounds stabilize inner maximization but shrink the policy hypothesis class, inducing a structural “Price of Robustness” via an increased nominal approximation gap.
- Introduces Adversarially-Aligned Jacobian Regularization (AAJR), which constrains sensitivity only along adversarial ascent directions produced by the inner maximization, rather than imposing a global operator-norm bound on the full Jacobian over the state space.
- Proves an expressivity/class-expansion guarantee: the globally constrained policy class is strictly contained in the AAJR-induced (trajectory-adaptive) class under mild conditions, implying a weakly smaller approximation gap and reduced nominal performance degradation relative to global constraints.
- Provides optimization/stability theory for the inner PGD/PGA loop: bounds effective smoothness of the inner objective along ascent iterates and derives step-size conditions to avoid curvature-driven divergence/oscillation.
- Advocates a structural decoupling of minimax stability from global expressivity restrictions, positioning robustness control as direction/trajectory-dependent rather than state-space-uniform.

### ⚙️ Methodology
Problem setup: robust agent training is posed as a minimax objective over policy parameters θ and adversarial state perturbations δ constrained to an ℓp ball Δ (e.g., ||δ||p ≤ ε): min_θ E_{(s,a_-i)~D}[ max_{δ∈Δ} L(π_θ(s+δ), a_-i) ]. The inner maximization is typically solved by projected gradient ascent (PGA/PGD) on δ, while θ is updated by gradient descent (GDA-style training).

Baseline stabilization mechanism critiqued: enforce a global bound on state sensitivity via the Jacobian operator norm, sup_{s∈S} ||J_θ(s)||_2 ≤ γ (equivalently a global Lipschitz-like constraint). This stabilizes inner-loop curvature but restricts the admissible policy class F_γ = {π∈F: sup_s ||J_π(s)||_2 ≤ γ}, increasing the nominal approximation gap T(γ) = inf_{π∈F_γ} R_nom(π) − inf_{π∈F} R_nom(π).

AAJR core idea: replace global Jacobian control with directional/trajectory-aligned control that only suppresses amplification in the directions that the adversary actually follows. Concretely, during inner-loop PGA iterations δ_t, the adversary ascends along gradients of the inner objective with respect to δ (equivalently state). AAJR regularizes the policy’s Jacobian only along these adversarial ascent directions (i.e., the directions spanned by the ascent updates / projected gradients), rather than bounding the full operator norm of J_θ(s). This yields a trajectory-adaptive constraint/regularizer that depends on the visited states (s+δ_t) and ascent directions.

Theory ingredients/assumptions (as stated): (1) action-level loss smoothness: for each context a_-i, z↦L(z,a_-i) is L_L-smooth; (2) differentiable policy π_θ(s) with Jacobian J_θ(s)=∇_s π_θ(s); (3) bounded second-order residual term along the line segments between inner-loop iterates (a bound C on the second-order contribution to curvature of the composed inner objective along the PGA trajectory).

Main theoretical claims (as described):
- Hypothesis class expansion: define a trajectory-adaptive class induced by AAJR directional constraints along ascent directions; show the globally constrained class is strictly contained within it under mild conditions, implying less conservative expressivity restriction.
- Inner-loop stability via effective smoothness control: bounding adversarially-aligned Jacobian amplification yields an explicit bound on the effective smoothness of the inner objective along the PGA iterates, enabling step-size conditions for stable ascent (mitigating divergence/limit cycles due to extreme local curvature).

### 🧪 Experiments
The provided paper text segment is theoretical and does not describe an empirical evaluation (no datasets, baselines, metrics, or quantitative results are included in the excerpt). Consequently, experimental validation details cannot be extracted from the supplied content; the contribution appears to be primarily theoretical (class-containment/approximation-gap arguments and inner-loop step-size/stability conditions).

### 🌱 Insights & Implications
Implications:
- Directional robustness control is a plausible way to reduce the typical robustness–utility trade-off: since PGD explores a small set of trajectory directions, globally suppressing sensitivity in all directions can be unnecessarily restrictive for expressive agent policies.
- For agentic systems (multi-step decision-making), overly conservative global Lipschitz constraints can harm responsiveness and adaptation, potentially more than in passive prediction settings.

Open questions / limitations (from what is available):
- Empirical confirmation is essential: whether AAJR improves robust training stability and nominal utility versus spectral normalization/global Jacobian bounds depends on practical factors (direction estimation noise, projection effects in Δ, nonstationarity of trajectories).
- The theory relies on smoothness and bounded second-order residual terms along PGA iterates; in deep networks and discontinuous environments, verifying or enforcing these conditions may be nontrivial.
- AAJR is trajectory-dependent: stability guarantees may depend on the specific inner optimizer, number of inner steps K, and how ascent directions are computed/approximated; transfer to different attack/inner-loop procedures may require re-derivation.
- Potential “blind spots”: restricting regularization to ascent directions could leave unregularized directions that become adversarial under distribution shift or different adversary models; robustness may be sensitive to the chosen threat model and exploration of ascent directions.

Research opportunities:
- Combine AAJR with certified robustness tools: AAJR could be used as a training-time stabilizer while other methods provide broader guarantees.
- Extend to discrete-action / non-differentiable agent policies (common in tool-using LLM agents) via surrogate gradients or policy-gradient-style inner maximizers.
- Evaluate in multi-agent simulations where instability manifests as oscillations/limit cycles; measure both convergence behavior (inner-loop stability) and downstream system-level outcomes.

### 🎯 Why This Paper
Although not directly about software vulnerabilities, the paper is relevant to security/robustness of autonomous agentic systems—an increasingly important substrate for security tooling (e.g., agentic vulnerability detection/repair pipelines) where minimax or adversarial training is used to harden decision policies against worst-case inputs or environment shifts. Its central idea—directional Jacobian regularization aligned to adversarial optimization trajectories—could inspire less conservative robustness mechanisms for learning-based security agents, potentially reducing the performance loss typically incurred by global Lipschitz/Jacobian constraints while still stabilizing adversarial inner-loop optimization.

## 📄 [CAM-LDS: Cyber Attack Manifestations for Automatic Interpretation of System Logs and Security Alerts](https://arxiv.org/abs/2603.04186v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Max Landauer, Wolfgang Hotwagner, Thorina Boenke, Florian Skopik, Markus Wurzenberger
📅 **Published:** 2026-03-04  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
CAM-LDS is a publicly released, fully reproducible dataset of labeled “attack manifestations” in system logs and security alerts, designed specifically to evaluate LLM-based semantic interpretation of logs across diverse MITRE ATT&CK techniques. It includes seven end-to-end scenarios (81 techniques, 13 tactics, 18 log/alert sources) and a case study showing an LLM identifies the exact technique for ~1/3 of steps and provides adequate technique-level answers for another ~1/3.

### 🏆 Key Contributions
- CAM-LDS: a public labeled dataset of cyber-attack-induced system log events and security alerts, curated to be directly attributable to attacker actions (reduced background noise) and mapped to MITRE ATT&CK techniques/tactics
- AttackBed: an open-source, reproducible testbed plus automation scripts to regenerate the environment, execute scenarios, and reproduce data collection
- Systematic analysis/categorization of “manifestations” (what becomes observable in logs/alerts) including observability of commands, event frequencies, performance metrics, and IDS alert behavior across sources
- An illustrative LLM-based log/alert interpretation case study with released prompts/responses demonstrating feasibility and current reliability limits

### ⚙️ Methodology
The paper constructs CAM-LDS by executing multiple coherent attack chains in a controlled, largely idle, open-source test environment (AttackBed) to ensure that collected telemetry is attributable to attacker actions rather than unrelated activity. Seven scripted attack scenarios are implemented to cover 81 distinct MITRE ATT&CK techniques spanning 13 tactics. Telemetry is collected from 18 distinct sources (system logs and security alerts; the paper also releases associated packet captures). A key curation step is extracting the subset of log events that are direct consequences of specific attack executions (“attack manifestations”), enabling per-step labels and analysis of (i) command/step observability (whether and where a step leaves artifacts), (ii) manifestation intensity (event frequency/volume), (iii) resource/performance signals, and (iv) IDS alerting behavior. The dataset is published via Zenodo and the full infrastructure and automation needed to reproduce it is released on GitHub. The LLM case study uses CAM-LDS as input prompts to map log/alert snippets back to the underlying attack technique(s), with outputs evaluated qualitatively/quantitatively by correctness of predicted technique at the attack-step level (with tiers such as perfect vs adequate matches).

### 🧪 Experiments
Experimental evaluation is presented as an illustrative LLM-based interpretation case study rather than a full benchmark shootout. Setup: CAM-LDS attack-step manifestations (logs/alerts) are provided to an LLM to predict the corresponding MITRE ATT&CK technique(s) for each step; authors publish the prompts and model responses to enable reproduction. Metrics/labels: step-level technique identification assessed in coarse correctness bands ("perfect" = correct technique; "adequate" = close/partially correct technique mapping). Key results: approximately one third of attack steps are predicted perfectly (exact correct technique), and another third are predicted adequately; the remaining third are not correctly mapped, indicating both promise and substantial headroom. The paper primarily positions these results as motivation and as a baseline illustration of how CAM-LDS can be used to evaluate LLM-based log/alert interpretation.

### 🌱 Insights & Implications
Implications: (1) Dataset availability and reproducibility are a major blocker for LLM-based security-log research; CAM-LDS directly targets this by publishing labeled, step-attributable manifestations plus the full testbed/scripts. (2) The notion of “manifestation” emphasizes that many attack steps have weak, indirect, or source-dependent observability—useful for studying coverage gaps and for designing better telemetry collection and correlation. (3) LLMs can sometimes map raw heterogeneous logs/alerts to high-level ATT&CK techniques without bespoke parsers/rules, but reliability is currently insufficient for unattended operation (only ~1/3 exact). Limitations/open questions highlighted/implicit: controlled/idle environment may not reflect production noise, concept drift, or concurrent benign activity; scripted scenarios may underrepresent adversarial variation; technique labeling can be ambiguous because multiple techniques can produce similar artifacts or one step can trigger multi-technique evidence; evaluation is illustrative (not a comprehensive comparison across models, prompting strategies, context-window handling, retrieval augmentation, or fine-tuning). Research opportunities enabled by CAM-LDS: benchmarking prompting/RAG/fine-tuning for technique attribution; studying multi-source fusion (host logs + IDS alerts + PCAP); alert/log clustering into attack steps; assessing robustness under injected background noise; learning parsers/normalizers guided by LLM explanations; measuring hallucination rates and calibration for SOC workflows.

### 🎯 Why This Paper
Directly relevant to vulnerability detection/repair-adjacent security engineering because it provides a rare public, labeled, technique-grounded corpus of host logs and security alerts suitable for building and evaluating automated reasoning systems (including LLM agents) that explain and classify malicious behavior. For a researcher working on security automation, it supplies reproducible infrastructure and ground truth for developing log/alert interpretation, attack-technique attribution, and SOC workflow automation methods—core prerequisites for scalable vulnerability/attack detection pipelines and for testing trustworthy, explainable LLM-based security tooling.

## 📄 [Cyber Threat Intelligence for Artificial Intelligence Systems](https://arxiv.org/abs/2603.05068v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Natalia Krawczyk, Mateusz Szczepkowski, Adrian Brodzik, Krzysztof Bocianiak
📅 **Published:** 2026-03-05  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
The paper argues that traditional cyber threat intelligence (CTI) does not adequately represent AI-specific assets, attack surfaces, and supply-chain artifacts, and proposes a blueprint for “CTI for AI” centered on an AI-oriented knowledge base with new indicators of compromise (IoCs) and similarity/search techniques for AI artifacts. It synthesizes existing AI security resources (e.g., ATLAS, AVID, incident databases) and highlights gaps in coverage, quality, and operational usability that must be addressed to build practical defenses for AI systems.

### 🏆 Key Contributions
- Conceptual comparison of classical CTI assumptions/workflows versus what is needed to defend AI systems, including AI-specific assets and vulnerabilities across the lifecycle
- Structured review and organization of existing AI security knowledge sources (vulnerability-oriented, taxonomy/framework-oriented, incident-oriented) and an assessment of their reliability/quality issues
- Proposal of what an AI-oriented CTI knowledge base should contain, including concrete IoCs tied to AI supply-chain phases and artifacts (data, models, pipelines, deployment components)
- Discussion of how an AI-oriented CTI KB could support security tooling (detection, triage, response) beyond classical network/endpoint IoCs
- Outline of approaches for measuring similarity between stored IoCs and newly observed AI artifacts to enable querying/matching in CTI workflows

### ⚙️ Methodology
This is a systematic literature review (SLR)-style paper rather than a new detection/repair system. The authors define four research questions spanning: (i) differences between classical CTI and CTI for AI, (ii) candidate sources for an AI-focused CTI knowledge base and their reliability, (iii) how such a KB benefits AI security tools, and (iv) similarity/matching methods for IoCs vs observed AI artifacts. They describe an SLR process: (1) keyword-driven search (e.g., “cyber threat intelligence”, “CTI for AI”, “AI incidents”, “AI vulnerabilities”) over academic databases (Google Scholar, Scopus) plus snowballing via cited references; (2) paper-by-paper extraction focused on AI CTI frameworks, data sources, IoC types, and tool applications; (3) synthesis into a structured review. The technical content centers on mapping AI lifecycle phases (e.g., AVID’s CRISP-DM-derived stages) and AI artifacts (datasets, models, pipelines, deployment infrastructure) to threat information units suitable for CTI (IoCs/TTPs), and on discussing similarity functions suitable for non-traditional indicators (e.g., model/dataset fingerprints, metadata, provenance, behavioral signatures) to support matching and querying in an AI CTI knowledge base.

### 🧪 Experiments
No controlled experiments or quantitative benchmarking of a proposed system are presented (the work is a review/position/blueprint paper). Quantitative figures are used descriptively to motivate the problem and characterize existing resources, e.g.: (1) MITRE ATLAS AI Model Vulnerabilities Dataset analysis reporting 500 adversarial attacks with distribution across model-level (53.6%), data-level (35.2%), and deployment-level (11.2%) vulnerabilities; (2) AVID public entries summarized as 40 vulnerabilities visible (13 in 2022, 27 in 2023) and 10 visible reports (5 in 2022, 3 in 2023, 2 in 2025), used to illustrate activity/coverage limitations. The paper references general cyber incident datasets and forecasts to contextualize CTI demand, but does not define datasets/baselines/metrics for a new algorithmic evaluation.

### 🌱 Insights & Implications
Key implications: (1) AI systems require CTI to represent artifacts beyond classical IoCs (IPs/domains/hashes), including datasets, model checkpoints, training pipelines, prompts, adapters, evaluation harnesses, and deployment configurations; (2) operational CTI for AI should be supply-chain aware—indicators and detection opportunities differ across business understanding/data collection/prep/modeling/evaluation/deployment; (3) existing AI security knowledge sources are fragmented: some are best for taxonomy and guidance (e.g., OWASP guides, ENISA/SAIF-type frameworks), some for TTP mapping (MITRE ATLAS), some for vulnerabilities (AVID), and some for real-world incidents (incident databases), but none alone provides the completeness/consistency needed for automation; (4) quality issues include sparse/uneven coverage, inconsistent granularity, unclear provenance, and limited update cadence (illustrated via AVID’s limited visible reports), which complicate automated ingestion and correlation; (5) similarity search becomes central because AI IoCs may be fuzzy (e.g., near-duplicate models/datasets, small weight deltas, paraphrased prompts) and need robust matching rather than exact indicators—raising open questions about standardization of fingerprints, embedding-based retrieval, and adversarial robustness of the matching itself; (6) limitations: as a review, the paper does not validate proposed IoC schemas, similarity methods, or tool integrations end-to-end; it also does not resolve governance questions (sharing sensitive AI artifacts, privacy/IP constraints) that are crucial for CTI adoption.

### 🎯 Why This Paper
Relevant to software security and vulnerability detection/repair because it reframes “what counts as an indicator” for AI-enabled software components (models, datasets, pipelines) and emphasizes supply-chain artifacts where vulnerabilities and tampering can occur—areas increasingly adjacent to modern software engineering. For research in automated program repair and vulnerability repair, the proposed CTI-for-AI knowledge base concept suggests new inputs for automated mitigation/patching workflows (e.g., detecting compromised model artifacts, tainted datasets, or unsafe deployment configs) and motivates similarity-based retrieval of prior incidents/vulnerabilities to guide remediation, analogous to how CVE/CWE knowledge bases support traditional secure development.

## 📄 [CLARC: C/C++ Benchmark for Robust Code Search](https://arxiv.org/abs/2603.04484v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Kaicheng Wang, Liyan Huang, Weike Fang, Weihang Wang
📅 **Published:** 2026-03-04  🏷️ **Categories:** cs.SE

### 💡 TL;DR
CLARC is a new C/C++ code search benchmark built from real GitHub projects with fully compilable snippets and controlled robustness stress tests (identifier anonymization and compilation to Assembly/WebAssembly). Evaluations on six modern retrieval models show large performance drops under these perturbations, indicating current code-search embeddings still rely heavily on lexical overlap rather than semantic understanding.

### 🏆 Key Contributions
- CLARC: a compilable, real-world C/C++ text-to-code retrieval benchmark with 6,717 query–code pairs (5,472 train, 1,245 eval)
- An automated dataset construction pipeline that (i) mines functions with dependency context, (ii) enforces compilability, and (iii) categorizes samples by dependency complexity
- LLM-generated natural-language queries with human scoring plus hypothesis testing to validate query quality and reduce noise
- Robustness evaluation settings that remove/alter lexical cues (identifier anonymization) and change representation level (Assembly and WebAssembly .wat), enabling controlled stress-testing
- Empirical evidence across multiple SOTA models that retrieval effectiveness collapses under robustness settings, exposing overreliance on lexical features and weak low-level generalization

### ⚙️ Methodology
CLARC is constructed via a four-stage pipeline. (1) Data collection: mine functions from 144 popular C/C++ GitHub repositories, split by repository into evaluation (45 repos) vs training (99 repos) to reduce contamination and leakage. Establish a compilation environment by building a whitelist of valid standard headers observed in the projects. Extract each function together with its full dependency context (e.g., required type definitions, helper functions, and call-graph reachable definitions). Filter to retain only snippets that compile in the prepared environment, yielding ‘fully compilable’ samples. (2) Categorization: classify each function into dependency-complexity groups: Group 1 uses only whitelisted standard library types/functions; Group 2 depends on custom-defined types (e.g., structs/typedefs) but not additional helper functions; Group 3 depends on custom helper functions (and potentially custom types), representing the highest contextual dependency burden. (3) Robustness settings: for each snippet, generate alternate representations intended to remove superficial lexical signals and/or shift abstraction levels: identifier anonymization (renaming variables/functions/types to non-informative placeholders while preserving structure), and compilation to low-level languages, specifically Assembly and WebAssembly text format (.wat). These variants are paired with the same natural-language query so performance differences isolate model sensitivity to lexical cues and representation changes. (4) Query formation and validation: use an LLM to draft natural-language descriptions (queries) for each code snippet. Validate these queries with human scoring; apply hypothesis testing to filter or accept queries based on statistical criteria, aiming to ensure that the query is both accurate and sufficiently specific to the target snippet rather than generic. The final benchmark provides train/eval splits plus grouped subsets and robustness variants for evaluation.

### 🧪 Experiments
Setup: Evaluate six code-search systems spanning closed and open models: OpenAI text-embedding-large (black-box), Voyage-code-3 (black-box), CodeT5+ (lightweight encoder), OASIS (robustness-focused model fine-tuned with augmented data), Nomic-emb-code (CLM-adapted embedding model trained with large-scale retrieval curricula), plus one additional SOTA method (as stated by the paper’s ‘six diverse methods’). Data: CLARC eval set has 1,245 query–code pairs; training set has 5,472 pairs. Evaluation is performed across (i) original source code, (ii) anonymized source, (iii) compiled Assembly, and (iv) compiled WebAssembly (.wat), and also analyzed by dependency groups (Group 1–3). Reported dataset stats (eval averages): queries ~84.8 tokens; source code ~244.2 tokens (but much larger for Group 3), Assembly ~1,092.7 tokens, Wasm ~811.4 tokens; average source LOC ~108.9 and cyclomatic complexity ~3.4 overall (Group 3 notably higher). Metrics: retrieval effectiveness metrics (paper indicates ‘retrieval metrics’ with sharp drops; specific metrics such as MRR@K/Recall@K are implied but not present in the provided excerpt). Key results (qualitative/relative): all evaluated models show consistent and sharp degradation when identifiers are anonymized, and a significant further drop when moving to Assembly/Wasm representations, demonstrating heavy reliance on lexical identifiers and poor generalization to low-level code. The robustness-oriented OASIS improves robustness relative to standard encoders but still suffers substantial drops, and even strong closed-source embeddings degrade notably under the stress tests.

### 🌱 Insights & Implications
Implications: (1) Lexical reliance remains a dominant failure mode for embedding-based code search; anonymization acts as a practical probe for semantic vs surface-form understanding and is relevant to obfuscation/adversarial settings seen in security. (2) Representation shift to Assembly/Wasm exposes a separate generalization gap: many ‘code’ embeddings are effectively trained for high-level syntax and naming conventions, not program semantics across compilation boundaries. (3) Dependency complexity grouping (custom types/helpers) provides a handle to study how much context retrieval models implicitly need; higher-dependency code likely requires models to encode broader context or use multi-stage retrieval (snippet + dependency expansion). Limitations/open questions: (a) Benchmark size (1,245 eval pairs) is moderate; variance across domains/projects may still matter for fine-grained model comparisons. (b) LLM-generated queries, even with human/hypothesis filtering, may differ from real developer search intent (brevity, jargon, partial specs); how well they approximate production queries remains an open question. (c) Compilability is enforced in a controlled environment with whitelisted headers; this may bias toward functions that can be isolated/compiled and away from deeply integrated build-system-dependent code. (d) Assembly/Wasm variants increase token length dramatically, potentially confounding ‘semantic gap’ with ‘length/position limit’ and tokenizer effects; future work could normalize length or use instruction-level IRs with structured encoders. (e) Anonymization strategy details (scope, consistency across files, preservation of API names) can influence difficulty; standardized anonymization protocols would aid cross-benchmark comparability. Research directions: train retrieval models with identifier-invariant objectives, incorporate structural/CFG/DFG signals, use contrastive pairs across source↔IR↔assembly, and evaluate with adversarial/obfuscation-aware curricula.

### 🎯 Why This Paper
Directly relevant to software engineering and security-oriented code analysis because it evaluates retrieval robustness under identifier anonymization (a proxy for obfuscation and adversarial renaming) and across compilation to low-level representations (Assembly/Wasm), both common in vulnerability research and malware analysis. For automated program repair and vulnerability detection/repair pipelines that rely on code search (e.g., retrieving fixes, patterns, similar vulnerable functions), CLARC provides a more realistic C/C++ benchmark and concrete stress tests that reveal when retrieval models will fail under renaming or transformed code, guiding the development of semantics-aware retrieval components for robust downstream tooling.

## 📄 [iScript: A Domain-Adapted Large Language Model and Benchmark for Physical Design Tcl Script Generation](https://arxiv.org/abs/2603.04476v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Ning Xu, Zhaoyang Zhang, Senlin Shu, Lei Qi, Jiaqi Lv, Wensuo Wang, Tianhao Zhao, Chao Zhang, Zhaoliang Yang, Xiangyu Li, Zhaorui Su, Jingshan Li, Xin Geng
📅 **Published:** 2026-03-04  🏷️ **Categories:** cs.SE, cs.PL

### 💡 TL;DR
iScript adapts Qwen3-8B to generate Innovus physical-design Tcl scripts from natural-language requirements by training on a synthetic 10K (requirement, chain-of-thought, script) dataset built from manuals and community sources. It introduces iScript-Bench plus a tool-independent two-step verifier (syntax + LLM functional judging) and reports higher pass@k than contemporary general LLMs on the benchmark.

### 🏆 Key Contributions
- iScript: a domain-adapted Qwen3-8B model for Innovus physical-design Tcl script generation using continued pretraining plus supervised fine-tuning with reasoning supervision
- iScript-Bench: the first benchmark focused on PD Innovus-style Tcl generation, spanning five task categories and three difficulty levels
- A multi-stage data synthesis pipeline to address PD Tcl data scarcity, producing a 10K tuple dataset of (requirement, CoT, script) from multi-source corpora and manuals
- A reproducible, tool-independent two-step verification framework combining static syntax checking and LLM-based functional evaluation for scalable correctness assessment

### ⚙️ Methodology
The approach targets single-turn natural-language requirement → Innovus Tcl script generation. Data is built via a multi-stage synthesis pipeline: (1) curate seed corpus from Innovus user guides and command reference manuals plus public community content (e.g., CSDN tutorials, technical forums); (2) extract and normalize a core command/parameter inventory, then generate many script fragments by combining/permuting commands and options (described as a multi-agent generation process inspired by prior agentic synthesis work); (3) filter generated scripts through a Tcl static linter to remove syntactically invalid samples, retaining only structurally valid Tcl; (4) apply requirement back-inference using a strong teacher model (GPT-4.1) to infer a plausible natural-language intent for each valid script; (5) generate a Chain-of-Thought explanation conditioned on (requirement, script) so the final training tuple becomes (requirement, CoT, script). Model training is two-stage: domain-adaptive continued pretraining (CPT) on the curated PD/Tcl domain corpus to adapt vocabulary and domain semantics, followed by supervised fine-tuning (SFT) on the 10K instruction-style tuples where the target includes reasoning (CoT) and the final script, teaching the model to map intent to correct command sequences and options. For evaluation without access to commercial Innovus, correctness is approximated by a two-step verifier: first, static syntax verification (linter/sandbox rules) to ensure Tcl validity; second, an LLM-based functional evaluator equipped with Innovus command knowledge that judges whether the script satisfies the stated requirement, enabling pass@k-like scoring without executing Innovus.

### 🧪 Experiments
Benchmark: iScript-Bench, organized into five PD task categories (described at a high level as floorplanning, placement, clocking/CTS, routing, and design optimization/closure) and three difficulty levels. Models compared include iScript (Qwen3-8B adapted) versus multiple general/state-of-the-art LLMs (exact list not fully visible in provided text). Metrics: pass@k-style success rates computed using the proposed two-step verification (syntax pass + functional-evaluator pass), rather than tool execution. Key result (as stated): iScript achieves higher average pass@k scores than the compared state-of-the-art general LLMs across the benchmark; exact numeric improvements and per-category breakdown are not present in the provided excerpt.

### 🌱 Insights & Implications
The paper highlights a recurring EDA-LLM pattern: domain adaptation can matter as much as model scale when the target language has sparse, tool-specific APIs and strong cross-command dependencies. The multi-stage synthesis pipeline suggests a practical recipe for low-resource code domains: generate many candidates programmatically, aggressively filter for syntactic validity, then use teacher models to recover high-quality natural-language intents and reasoning supervision. The largest open concern is evaluation validity: LLM-based functional judging may be biased toward plausible-looking scripts and may not correlate perfectly with real Innovus execution outcomes, especially for constraints consistency, design database state, and multi-step flow interactions. Another limitation is domain/tool specificity (Innovus Tcl); transfer to other P&R tools (ICC2, OpenROAD Tcl dialects) or to multi-turn agentic flows is unclear. The dataset being teacher-inferred (requirements and CoT) introduces potential distributional artifacts (teacher style, over-regularized intents) and may underrepresent messy real engineer requirements; it also raises the risk of learning to satisfy the evaluator rather than the tool. Open questions for follow-up: correlation between the two-step verifier and real-tool outcomes; robustness to incomplete/ambiguous requirements; security/safety implications of generating tool scripts that can alter environments; generalization to proprietary command variants and version drift in tool APIs.

### 🎯 Why This Paper
This work is relevant to software engineering/maintenance and automated program repair in a specialized scripting setting: it tackles reliability-critical code generation under severe data scarcity using static checking plus learned semantic validation, which parallels APR pipelines (generate–filter–rank/verify). The benchmark and verifier design are directly useful for researchers studying LLM-based code generation correctness, verification without execution, and domain adaptation; the synthesis methodology can be repurposed for vulnerability-repair datasets (generate candidates, lint/compile, back-infer intents, add reasoning), and the two-step verification idea aligns with building scalable evaluators for security patches when full integration testing is expensive.
