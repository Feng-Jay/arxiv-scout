## 📄 [VulStyle: A Multi-Modal Pre-Training for Code Stylometry-Augmented Vulnerability Detection](https://arxiv.org/abs/2604.26313v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Chidera Biringa, Ajmal Abbas, Vishnu Selvaraj, Gokhan Kul
📅 **Published:** 2026-04-29  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
VulStyle is a RoBERTa-based multi-modal vulnerability detector that combines raw function code, a compact sequence of non-terminal AST nodes, and code stylometry features to better capture risky coding patterns. Pre-trained on 4.9M functions across seven languages and fine-tuned on five vulnerability datasets, it reports state-of-the-art results on BigVul and VulDeePecker and strong average performance overall.

### 🏆 Key Contributions
- Introduces a multi-modal vulnerability detection framework that jointly models token-level source code, reduced non-terminal AST structure, and code stylometry features.
- Proposes a reduced-AST representation that keeps only non-terminal AST nodes to preserve structural hierarchy while lowering input complexity compared with full flattened ASTs.
- Augments transformer pre-training and downstream fine-tuning with stylometric features to capture developer coding tendencies associated with vulnerability risk.
- Pre-trains the model with masked language modeling on a large 4.9M-function multilingual corpus spanning seven programming languages.
- Provides extensive evaluation on five benchmark datasets, achieving SOTA on BigVul and VulDeePecker with reported F1 gains of 4–48% over strong transformer baselines.
- Includes ablation studies, failure analysis, and a threat model aimed at clarifying the value and practical realism of the proposed signals.

### ⚙️ Methodology
VulStyle is built on a RoBERTa-style transformer trained with a masked language modeling objective and adapted for vulnerability detection through multi-modal input construction. The first modality is function-level source code, which provides lexical and local syntactic context. The second modality is AST structure, but instead of flattening the entire tree, the approach extracts only non-terminal nodes; this is intended to retain semantic hierarchy and coarse syntactic organization while reducing sequence length and structural overhead. The third modality is code stylometry (CStyle), consisting of syntactic and lexical style features designed to reflect coding habits such as naming choices, expression patterns, API preferences, and structural tendencies that may correlate with insecure development practices.

The overall pipeline first gathers a large multilingual pre-training corpus of functions from several existing code repositories and datasets. For each function, the source text is paired with a compact structural representation derived from non-terminal AST nodes, and stylometric descriptors are computed as auxiliary signals. The model is then pre-trained with MLM to learn joint representations over these modalities. In downstream vulnerability detection, the pre-trained encoder is fine-tuned on labeled vulnerable/non-vulnerable functions from benchmark datasets. The paper positions the stylometry channel as complementary to token and structure representations: token sequences capture local lexical patterns, reduced ASTs capture hierarchy, and CStyle captures recurring implementation behavior that may indicate vulnerability risk. The threat model and error analysis further frame VulStyle as a realistic detector for attacker-relevant settings where risky coding habits and structural cues matter, especially for memory-unsafe languages such as C/C++.

### 🧪 Experiments
The model is pre-trained on 4.9 million functions collected from CodeSearchNet, VulBERTa, DiverseVul, and Big-Vul, covering seven programming languages. Fine-tuning and evaluation are performed on five widely used vulnerability detection benchmarks: Devign, BigVul, DiverseVul, REVEAL, and VulDeePecker. The comparison set includes strong transformer-based baselines and prior deep-learning vulnerability detectors discussed in the paper, such as CodeBERT, UniXcoder, GraphCodeBERT, ContraBERT, VulBERTa, Devign, and VulDeePecker-family methods.

The main reported metric emphasized in the abstract is F1 score. VulStyle achieves state-of-the-art performance on BigVul and VulDeePecker, with reported F1 improvements ranging from 4% to 48% over strong transformer baselines, and it attains competitive or best-average performance across all five datasets. The study also includes ablations that isolate the effect of adding CStyle features and AST structure, showing that both contribute to performance beyond token-only modeling. In addition, the paper presents failure cases and robustness-oriented discussion through a threat model, though the excerpt does not provide the full numeric table details for all datasets and metrics.

### 🌱 Insights & Implications
The paper’s main insight is that vulnerability detection benefits from signals beyond tokens and full program graphs: compact structural information and coding-style cues can capture recurring risky habits that standard token models miss. The reduced-AST design suggests an effective middle ground between token-only transformers and expensive graph/full-AST methods, which is especially relevant for scaling to large code corpora.

For security research, the stylometry angle is particularly interesting because it reframes vulnerability detection partly as detection of risky implementation behavior rather than only known lexical or semantic bug patterns. This appears most promising for memory misuse, unchecked return values, pointer misuse, and inconsistent buffer handling. At the same time, the authors acknowledge that purely semantic flaws such as cryptographic misuse, integer overflow, and race conditions remain difficult, indicating that stylometry is complementary rather than sufficient. Likely limitations include dependence on parser quality for AST extraction, possible dataset bias where coding style correlates with project or author rather than vulnerability causality, and uncertain cross-project or cross-language generalization of stylometric features. Open questions include whether the approach remains robust under adversarial style normalization, how well it transfers to line-level or patch-level localization, and whether the same multi-modal design can support vulnerability repair or root-cause explanation.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and potentially automated repair because it proposes a new representation learning strategy for identifying vulnerable code that goes beyond standard token-only transformers. It is especially useful for a researcher interested in software engineering and maintenance because coding style and structural abstractions are maintainability-related signals that may also predict security risk. The reduced-AST idea offers a practical alternative to heavyweight graph models, while the stylometry augmentation opens a new direction for vulnerability discovery, prioritization, and possibly repair guidance by linking insecure patterns to recurring developer habits.

## 📄 [LLM-Guided Issue Generation from Uncovered Code Segments](https://arxiv.org/abs/2604.26118v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Diany Pressato, Honghao Tan, Mariam Elmoazen, Shin Hwei Tan
📅 **Published:** 2026-04-28  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper introduces IssueSpecter, a coverage-guided pipeline that asks an LLM to inspect uncovered Python code segments and turn likely defects into structured, prioritized GitHub-style issue reports. Across 13 projects, it produces actionable issues with high manual validity, improves ranking quality over rule-based prioritization, and outperforms a coverage-driven test generation baseline on bug validity under matched conditions.

### 🏆 Key Contributions
- Introduces IssueSpecter, a tool that combines coverage localization and LLM-based defect analysis to generate structured issue reports directly from uncovered code segments rather than from generated tests.
- Proposes a two-stage prioritization workflow: repository-level rule-based issue selection followed by LLM-based reranking to surface the most impactful issues for developer triage.
- Provides large-scale evaluation on 13 actively maintained Python projects, generating 10,467 issue reports and manually assessing the top 130 ranked issues.
- Shows that LLM-based ranking substantially outperforms rule-based ranking, improving P@3 by 50% and MRR by 41%.
- Demonstrates practical bug-finding capability through reproduced case studies, including security and reliability issues such as path traversal, memory exhaustion, data loss, and type-constraint violations.
- Compares against CoverUp under matched conditions and reports higher bug validity for IssueSpecter while additionally producing directly actionable issue reports with reproduction steps and candidate fixes.

### ⚙️ Methodology
IssueSpecter is a four-phase pipeline. First, it performs coverage-based code segment localization: given a Python project and its existing unit tests, it uses SlipCover to identify uncovered code segments, treating these as high-signal regions where latent defects may exist because no tests currently exercise them. Second, in the issue generation phase, each uncovered segment is independently analyzed by GPT-5-mini using a structured prompt that asks the model to identify up to three distinct bugs or explicitly state that no bug is present. For each detected bug, the model must return machine-parseable JSON containing a title, summary, severity level, affected operating systems, a detailed issue description, reproduction steps, whether documentation is inconsistent, and a suggested code fix intended to preserve original behavior and style. Third, because this process can yield hundreds of candidate reports per repository, IssueSpecter applies repository-level issue selection using a hierarchical rule-based scheme inspired by bug prioritization literature. It first sorts by severity class, then by operating-system impact breadth, then by description completeness via word count to break ties. This narrows the candidate set to the top issues per project. Fourth, the selected issues are reranked by an LLM, which reasons about impact, urgency, and likely developer relevance to refine ordering beyond surface heuristics. The overall design is motivated by the coverage-resolution gap: generated tests may miss the oracle or encode incorrect behavior, while issue-resolution agents require an existing bug report. IssueSpecter instead uses uncovered code as localization and directly generates actionable reports suitable for developer review or downstream automated repair workflows.

### 🧪 Experiments
The evaluation uses 13 actively maintained open-source Python projects from the CodaMosa dataset. IssueSpecter analyzes uncovered segments from each project and generates 10,467 issue reports in total. For manual evaluation, the authors inspect the top 10 ranked issues per project, yielding 130 top-ranked issues. Human annotation shows that 84.6% of these issues are either valid bugs or worthy of further investigation, with 37.7% confirmed genuine bugs and 15.4% false positives. The discovered issues span nine of ten categories in the adopted BugPilot taxonomy, including logic, boundary, state consistency, documentation-related, and security defects. For ranking quality, LLM-based reranking is compared to the rule-based ranking and achieves a 50% improvement in Precision@3 and a 41% improvement in Mean Reciprocal Rank. In the HTTPie motivating example, LLM reranking moves a path traversal issue from 7th under the rule-based ordering to 1st, matching human judgment, and attains MRR 1.00 versus 0.14 for the heuristic approach. The paper also compares IssueSpecter with CoverUp, a coverage-driven test generation tool, under identical conditions using the same model and the same number of evaluated artifacts per project; over 168 matched artifacts per tool, IssueSpecter achieves a higher bug validity rate of 81.0% versus 76.2% for CoverUp. Additional case studies reproduce real bugs uncovered by the generated reports, including a CWE-22 path traversal vulnerability, an out-of-memory condition due to unbounded buffering, silent data loss in gzip decompression, and a type-constraint violation.

### 🌱 Insights & Implications
The paper’s main insight is that uncovered code segments provide a strong localization signal for bug discovery without relying on generated tests and their fragile oracles. By asking an LLM to reason directly over these unexecuted regions, the system can surface diverse bug types and package them as issue reports that are immediately actionable for humans or downstream automated repair tools. The work is especially notable for moving beyond binary bug detection to prioritized triage, reproduction guidance, and candidate fixes, which addresses a practical pain point in developer adoption of AI-generated reports. For software engineering research, it suggests a useful middle ground between testing and repair: issue generation as an artifact that can trigger validation, patching, or benchmark construction. Limitations include reliance on LLM judgment for both defect detection and severity assignment, potential hallucinations or over-reporting on hard-to-understand snippets, and the need for manual validation of top-ranked outputs. The evaluation is limited to Python and to uncovered code, so defects in covered but weakly asserted code may be missed. Ranking quality appears improved, but the gold ranking still depends on human annotation and small top-k inspection. Open questions include whether richer repository context, execution traces, historical issue data, or static-analysis signals could further improve precision; how well the approach transfers to other languages and larger monorepos; and whether the generated reproduction steps and fixes can be systematically turned into executable regression tests or APR inputs.

### 🎯 Why This Paper
This paper is highly relevant to research in software engineering, maintenance, automated program repair, vulnerability detection, and vulnerability repair because it turns latent defects in untested code into structured issue artifacts that can feed maintenance and repair pipelines. It advances the state of practice by linking coverage analysis, defect discovery, triage, and actionable reporting, offering a promising front end for APR systems, bug benchmarks, security auditing, and developer-focused vulnerability remediation workflows.

## 📄 [Reproducible Automated Program Repair Is Hard -- Experiences With the Defects4J Dataset](https://arxiv.org/abs/2604.26674v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Adam Krafczyk, Klaus Schmid
📅 **Published:** 2026-04-29  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper argues that benchmark reproducibility for automated program repair must be defined more strictly than simply replaying the known failing tests, and shows that Defects4J falls short under APR-relevant conditions. Using a stricter evaluation framework, the authors find 180 of 835 Defects4J bugs unusable for APR experiments and another 59 with obviously under-specified test suites.

### 🏆 Key Contributions
- Defines a broader notion of defect workability for APR that extends traditional defect reproducibility with APR-specific requirements such as reliable compilation, deterministic and order-independent test behavior, and suitability for evaluating candidate patches.
- Provides an empirical assessment of Defects4J under these stricter requirements and quantifies substantial unsuitability: 180/835 defects (21.6%) are not workable for APR experiments.
- Identifies severe test-suite under-specification in an additional 59/835 defects (7.1%), where deleting a single statement yields a test-passing program although the human patch is more complex.
- Implements an APR evaluation framework for Java that performs stricter checks intended to detect hidden issues such as flaky tests and inadequate validation conditions.
- Documents concrete lessons learned and practical failure modes researchers may encounter when building reproducible APR pipelines on top of Defects4J.

### ⚙️ Methodology
The study is methodological and empirical rather than proposing a new repair algorithm. The authors first formulate a set of requirements that a defect dataset must satisfy to be workable for APR, explicitly distinguishing this from conventional reproducibility checks. These requirements cover the availability and parseability of modifiable source files, the ability to compile variants with the proper classpath and dependencies, and the existence of a test suite that can reliably validate modified program variants. The paper emphasizes APR-specific constraints that are often implicit in repair systems: test executions must behave consistently across repeated runs, partial runs, and altered execution orders; candidate variants must be executable in isolated/temporary environments; and the framework must expose hidden non-determinism or environmental assumptions that would distort repair evaluations. To operationalize these ideas, the authors built a Java APR evaluation framework around Defects4J projects and used it to check each defect more strictly than the standard Defects4J workflow. Instead of only asking whether the buggy version compiles and exhibits the expected failing tests, the framework repeatedly compiles and executes tests for defective, fixed, and modified variants under conditions relevant to APR search loops. It performs stricter validation to catch indications of inadequate test suites, including flaky outcomes and situations where suspiciously trivial edits can satisfy all tests. For test-suite quality analysis, the authors use a simple sanity check based on deleting a single statement from the codebase and rerunning the tests; if all tests pass despite the human fix not being a mere deletion, this is treated as strong evidence of under-specification. The overall methodology is thus a requirements-driven audit of dataset workability coupled with large-scale execution of strict validation procedures over all Defects4J defects.

### 🧪 Experiments
The experiments focus on Defects4J v2.0, a benchmark of 835 defects from 17 open-source Java projects. The evaluation uses the authors' Java APR framework and a stable execution environment aligned with Defects4J recommendations, but applies stricter checks than prior reproducibility studies. The main comparison point is traditional reproducibility as used in prior work: build the buggy project, run the provided tests, and confirm the expected failure. The authors instead assess APR workability, including robustness of compilation/execution for modified variants and consistency of test-suite behavior under APR-like usage. The principal metric is the number and proportion of defects that are workable versus unworkable for APR evaluation, along with counts of defects whose tests appear under-specified. Key results are that 180 out of 835 defects, or 21.6%, are not suitable for APR experiments under the stricter criteria. In addition, 59 defects, or 7.1%, have obviously under-specified test suites, demonstrated by the fact that deleting a single statement makes all tests pass even though the human-written patch is not simply a deletion. The paper also situates these findings against prior claims that Defects4J is highly reproducible, showing that reproducibility under standard benchmark scripts materially overestimates suitability for APR experimentation.

### 🌱 Insights & Implications
The central insight is that APR evaluation depends on stronger assumptions than benchmark papers and reproducibility studies typically check. A defect can be reproducible in the narrow sense yet still be unusable for repair experiments because APR systems repeatedly compile and test many mutated variants, often in partial or reordered executions, which exposes hidden dependency, isolation, and flakiness issues. The paper also reinforces a long-standing concern in APR that test-suite adequacy is a first-order threat to validity: if extremely simple deletions can satisfy the tests, then repair success counts may reward overfitting rather than genuine fixes. For related research, this implies that APR comparisons using raw Defects4J counts may not be directly comparable unless defect filtering and execution protocols are standardized. A practical implication is that benchmark infrastructure should distribute richer metadata and stricter validation harnesses, not just checkout scripts and expected failing tests. Limitations include the focus on Defects4J and Java, and the fact that some workability requirements may matter differently across APR paradigms; the authors note that not every requirement is equally necessary for every repair technique. Open questions include how to define community-wide APR-ready benchmark standards, how to automatically detect deeper forms of under-specification beyond trivial deletion checks, and how many published APR results would change under stricter defect filtering.

### 🎯 Why This Paper
This paper is highly relevant to research in automated program repair, software maintenance, and vulnerability repair because it scrutinizes the experimental foundation on which many APR systems are evaluated. If you work on repair, bug fixing, or security patch generation, its findings directly affect benchmark selection, result validity, and pipeline design: it offers concrete criteria and tooling ideas for building reproducible APR evaluations and warns that a substantial fraction of Defects4J may bias conclusions unless carefully filtered.

## 📄 [Agentic AI in the Software Development Lifecycle: Architecture, Empirical Evidence, and the Reshaping of Software Engineering](https://arxiv.org/abs/2604.26275v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Happy Bhati
📅 **Published:** 2026-04-29  🏷️ **Categories:** cs.SE

### 💡 TL;DR
This paper is a synthesis and conceptual framing of the shift from code-completion tools to repository-level agentic software engineering systems that can plan, execute, test, and revise changes under human supervision. Its main result is a consolidated view of rapid progress in agentic coding performance, a six-layer reference architecture, and an Agentic SDLC model showing how software engineering practice is being reorganized around delegated execution rather than direct code authoring.

### 🏆 Key Contributions
- Introduces a six-layer reference architecture for agentic software engineering systems spanning foundation models, reasoning/memory, agent-computer interfaces, tools/environments, orchestration, and governance/safety.
- Defines and contrasts a traditional SDLC with an Agentic SDLC (A-SDLC), emphasizing delegation, orchestration, and human approval gates across requirements, design, implementation, testing, deployment, and maintenance.
- Synthesizes empirical evidence showing large gains in repository-level issue resolution, especially on SWE-bench Verified, and argues these gains come primarily from agentic scaffolding rather than model scaling alone.
- Consolidates evidence from controlled productivity studies indicating meaningful developer time savings and workflow transformation.
- Identifies five open research problems: evaluation, governance, technical debt, skill redistribution, and the economics of attention.
- Reframes the field’s central research object from code generation to delegated software execution under human supervision.

### ⚙️ Methodology
The paper is a systems-and-evidence synthesis rather than a novel algorithmic contribution or primary experimental study. It surveys industrial systems (e.g., Claude Code, Codex CLI, Jules, Devin, AlphaEvolve), academic agentic coding frameworks (e.g., SWE-agent, MetaGPT, ChatDev, AgileCoder, OpenHands, AutoCodeRover, HyperAgent), and empirical benchmark and productivity reports, then organizes them into a unifying conceptual framework. The proposed architecture has six layers: L0 foundation models providing reasoning and code generation; L1 reasoning, memory, and self-reflection mechanisms supporting planning and iterative revision; L2 agent-computer interface (ACI) translating model outputs into structured repository navigation, editing, testing, and shell actions; L3 tools and execution environments such as filesystems, editors, terminals, browsers, version control, CI/CD, compilers, and test runners; L4 orchestration, either single-agent iterative loops or multi-agent role-specialized systems; and L5 governance and safety mechanisms including permissions, sandboxing, logging, and policy enforcement. Building on this architecture, the paper proposes an Agentic SDLC in which agents draft requirements artifacts, propose designs, implement patches, generate and run tests, support deployment gating, and assist maintenance, while humans act primarily as supervisors at intent-setting, review, and approval boundaries. The paper’s analytical stance is that non-determinism and delegated execution fundamentally alter both software processes and evaluation criteria, so traditional SDLC assumptions and metrics need to be revised.

### 🧪 Experiments
The paper does not present new experiments; instead, it consolidates prior empirical results. The main capability evidence centers on SWE-bench and SWE-bench Verified, repository-level issue-resolution benchmarks derived from real GitHub issues. Reported performance rises from about 1.96% for early RAG-style baselines in October 2023 to 12.5% for SWE-agent, then to successive Claude-based agentic systems at 33.2%, 49%, 62.3%, 72.7%, and 78.4% by April 2026. A key comparative claim is that non-agentic retrieval-augmented systems plateau near 20%, suggesting that scaffolding, tool use, and execution loops matter more than raw base-model quality alone. The paper also references other evaluation settings such as Terminal-Bench 2.0 and SWE-bench Pro for frontier CLI agents, and highlights AlphaEvolve’s domain-specific optimization successes, including a new 4x4 complex matrix multiplication algorithm using 48 scalar multiplications, progress on 20% of fifty open mathematical problems, 0.7% data-center compute recovery, and 23% FlashAttention kernel speedup. For productivity, the paper synthesizes controlled user studies and deployment analyses, including Microsoft research showing developers completed an HTTP server task 55.8% faster with Copilot and usage studies indicating around 30% acceptance rates in large-scale telemetry. Across studies, the paper reports 13.6%-55.8% time savings. It additionally summarizes labor-market evidence such as Anthropic’s estimate that AI was used for at least a quarter of tasks in 49% of sampled jobs in 2026.

### 🌱 Insights & Implications
A central insight is that repository-level agency emerges from the interaction between models and scaffolding, especially the agent-computer interface, memory, tool access, and iterative feedback loops; benchmark improvements cannot be explained by larger language models alone. This matters for software engineering research because the key design questions shift from code-token prediction to process control, tool mediation, safety boundaries, and human-agent coordination. The paper also argues that the software engineer’s role is moving toward task decomposition, supervision, review, and escalation management, which may redistribute skills and incentives across junior and senior roles. For software maintenance and repair, the Agentic SDLC framing suggests that bug fixing, regression testing, issue triage, and hotfix generation are likely early high-impact use cases, but also raises concerns about stochastic behavior, reproducibility, auditability, and the accumulation of agent-induced technical debt. The paper identifies governance and safety as the least mature architectural layer and potentially the main barrier to enterprise adoption. Open questions include how to evaluate long-horizon engineering tasks beyond current benchmarks, how to certify agent behavior in security-sensitive environments, how to measure and manage latent technical debt from machine-generated changes, how developer expertise evolves when more implementation work is delegated, and whether attention and review overhead become the true bottleneck. A limitation is that the work is primarily a synthesis of public reports and benchmark results rather than a controlled comparative study; some referenced industrial numbers may reflect differing settings, benchmark subsets, or proprietary evaluation conditions.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering, maintenance, security, automated program repair, vulnerability detection, and vulnerability repair because it synthesizes the exact transition from code assistants to autonomous repair-and-maintenance agents operating over real repositories. Its emphasis on SWE-bench-style issue resolution, agent-computer interfaces, orchestration, testing loops, and governance directly informs the design and evaluation of automated program repair and secure code agents. For a researcher working on vulnerability detection and repair, the paper provides a useful architectural lens for building end-to-end agentic systems that can localize faults, propose patches, run validation, and operate within safety constraints, while also clarifying open research problems around evaluation rigor, security controls, and technical debt that are central to dependable repair pipelines.

## 📄 [ImproBR: Bug Report Improver Using LLMs](https://arxiv.org/abs/2604.26142v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Emre Furkan Akyol, Mehmet Dedeler, Eray Tüzün
📅 **Published:** 2026-04-28  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
ImproBR is an LLM-based pipeline that detects missing, incomplete, or ambiguous Steps to Reproduce, Observed Behavior, and Expected Behavior in user-submitted bug reports and rewrites them into structured, more actionable reports. On 139 difficult Minecraft bug reports from Mojira, it raised structural completeness from 7.9% to 96.4%, increased executable S2R from 28.8% to 67.6%, and improved fully reproducible reports from 1 to 13.

### 🏆 Key Contributions
- Introduces ImproBR, a multi-stage bug report improvement pipeline that combines defect detection and targeted LLM-based rewriting of S2R, OB, and EB sections.
- Proposes a hybrid quality detector that integrates a fine-tuned DistilBERT classifier, heuristic analysis, and an LLM analyzer to identify missing and ambiguous report sections.
- Uses section-specific few-shot prompting with GPT-4o mini plus a Retrieval-Augmented Generation pipeline grounded in Minecraft Wiki knowledge to improve factuality and domain alignment.
- Evaluates bug report quality not only with structural completeness and semantic similarity, but also with practical developer-centric criteria such as S2R executability and bug reproducibility.
- Provides evidence of real-world usefulness through manual evaluation on challenging Mojira reports and motivating examples where improved reports helped reopen or clarify issues on the official tracker.

### ⚙️ Methodology
ImproBR is organized as a preprocessing, detection, and improvement pipeline for low-quality bug reports. First, raw Mojira reports are preprocessed and normalized so their text can be analyzed consistently. The detection stage then identifies whether each of the three key sections—Steps to Reproduce (S2R), Observed Behavior (OB), and Expected Behavior (EB)—is missing, incomplete, or ambiguous. This detector is hybrid: it combines a fine-tuned DistilBERT model for section/quality classification, heuristic rules to capture explicit structural cues and simple defects, and an LLM-based analyzer to reason about ambiguity and incompleteness beyond surface patterns. The outputs of these components guide which sections should be generated or revised.

In the improvement stage, ImproBR prompts GPT-4o mini with section-specific few-shot examples so that the model produces content in the desired format and style for S2R, OB, and EB separately. Rather than generating a whole report monolithically, the system targets deficient sections, which helps preserve useful original content while repairing missing or weak parts. To improve reliability, the system uses Retrieval-Augmented Generation: relevant background material is retrieved from a domain knowledge base built from the Minecraft Wiki and passed to the LLM so generated steps and terminology better reflect the game mechanics and domain concepts mentioned in the report. The goal is not merely to make reports look complete, but to make them clearer, more executable, semantically aligned with the original issue, and more useful for developers trying to reproduce bugs. The paper also studies the contribution of major components—RAG, the quality detector, and few-shot prompting—through ablation-oriented analysis tied to executability and reproducibility outcomes.

### 🧪 Experiments
The evaluation uses Mojira, the public bug tracker for Minecraft, a large-scale repository with over 450,000 user-generated reports and many low-quality submissions. The main manual evaluation focuses on 139 challenging real-world reports selected as difficult and practically testable cases. The paper positions prior bug report quality tools and LLM-based improvers such as DeMIBuD, BEE, EULER, ChatBR, AdbGPT, AstroBR, and related work as baselines/points of comparison, while the core empirical study mainly measures before-versus-after improvement on raw reports and includes component-level analysis.

The reported metrics go beyond text similarity. They include structural completeness of S2R/OB/EB presence, executability of the generated S2R, semantic/contextual alignment with higher-quality versions, reproducibility of the bug, and runtime. Quantitatively, raw reports were on average only 7.9% structurally complete; after ImproBR, this increased to 96.4%. The proportion of reports with executable S2R more than doubled, from 28.8% to 67.6% on average. Fully reproducible bug reports increased from 1 to 13 among the 139 evaluated cases. The average end-to-end runtime per report was 23.94 seconds. The paper also discusses motivating real-world cases where improved reports submitted back to Mojira helped stalled issues be reopened or clarified, suggesting practical external validity beyond offline evaluation.

### 🌱 Insights & Implications
A central insight is that structural completeness alone is an inadequate measure of bug report quality: a report can contain S2R, OB, and EB fields yet still be non-executable or misleading. ImproBR’s results suggest that combining detection, retrieval, and targeted prompting can substantially improve practical utility, especially in domains where user reports are noisy and under-specified. The use of a domain knowledge source is important because pure LLM generation may hallucinate or use imprecise game terminology; grounding with Minecraft Wiki appears to help semantic faithfulness and procedural accuracy.

The work also highlights the value of hybrid detection over a single classifier. Heuristics catch explicit omissions, DistilBERT handles learned text patterns, and the LLM analyzer reasons about ambiguity and incompleteness that simpler methods may miss. For software engineering research, this indicates that bug report enhancement may benefit from modular pipelines rather than end-to-end generation alone.

Limitations include domain specificity: the system is built and evaluated on Minecraft/Mojira, with retrieval grounded in Minecraft Wiki, so generalization to other software ecosystems, especially non-game and security-critical systems, remains open. The evaluation relies heavily on manual assessment over 139 reports, which is valuable but small relative to the full tracker and may be sensitive to evaluator judgment. Runtime and dependence on proprietary LLMs may affect scalability and reproducibility in industrial workflows. An open research question is how to adapt this approach to richer artifacts such as crash traces, logs, screenshots, code snippets, or execution traces, and whether similar gains would hold in domains like vulnerability reports, CI failures, or program repair benchmarks.

### 🎯 Why This Paper
This paper is highly relevant to software maintenance because it targets a core bottleneck in issue resolution: poor bug reports that waste developer time and hinder reproduction. It is especially useful for your interests in automated program repair, vulnerability detection, and vulnerability repair because those pipelines depend heavily on actionable, reproducible defect descriptions; ImproBR offers a concrete method for transforming noisy human reports into more structured and executable inputs. Its hybrid detection-plus-generation design, emphasis on practical reproduction rather than just linguistic quality, and use of retrieval grounding make it a strong reference for building downstream systems that triage, localize, reproduce, or repair defects and potentially security issues from natural-language reports.

## 📄 [SWE-Edit: Rethinking Code Editing for Efficient SWE-Agent](https://arxiv.org/abs/2604.26102v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Yikai Zhang, Jiaxin Pei, Kenan Li, Maoquan Wang, Jin Pan, Yu Kang, Shengyu Fu, Elsie Nallipogu, Junjie Hu, Yufan Huang, Zijian Jin
📅 **Published:** 2026-04-28  🏷️ **Categories:** cs.SE, cs.CL

### 💡 TL;DR
SWE-Edit redesigns the code-editing interface for software engineering agents by splitting file inspection and code modification into separate subagents, reducing context pollution and formatting failures. ## 📄 [Enforcing Benign Trajectories: A Behavioral Firewall for Structured-Workflow AI Agents](https://arxiv.org/abs/2604.26274v1)  `60% relevant`  `🌐 Full Text`

📅 **Published:** 2026-04-29  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
This paper proposes Praetor, a runtime behavioral firewall for LLM agents that learns benign structured tool-use trajectories from verified telemetry and enforces them with a parameterized DFA. By adding stateful sequence constraints and parameter guards at the tool boundary, it substantially reduces attack success on structured workflows while adding only millisecond-scale latency.

### 🏆 Key Contributions
- Introduces a telemetry-driven behavioral firewall for tool-using LLM agents that models allowed behavior as a parameterized deterministic finite automaton (pDFA).
- Formalizes context-aware tool-call interception under indirect prompt injection, emphasizing the weakness of stateless per-call scanning against multi-step sequential attacks.
- Designs an offline-to-online architecture where expensive profiling is done offline and runtime enforcement reduces to constant-time state-transition lookup plus parameter checking.
- Extends structural sequence enforcement with parameter guards for numeric ranges, categorical exact matches, and string embedding-radius checks, plus optional exact whitelisting for sensitive fields.
- Implements sidecar-style deployment with cryptographic audit logging and no modifications to agent reasoning loops or model weights.
- Empirically shows strong security gains on Agent Security Bench, especially on structured workflows and context-sequential attacks, while keeping low per-call latency and modest benign-task disruption.

### ⚙️ Methodology
Praetor protects an LLM agent at the tool-dispatch boundary. In the offline profiling phase, it collects a corpus of verified benign execution traces, where each trace is a sequence of tool calls paired with parameter vectors. From these traces, it constructs a parameterized DFA whose states encode the previous w tool names (a w-gram context with idle padding at trace boundaries), and whose transitions represent allowed next tools under that sequential context. Transition counts are accumulated from the benign corpus and infrequent edges can be pruned using a threshold theta to remove noise. For each surviving transition, Praetor attaches parameter guards derived from benign observations on that edge: numeric parameters are bounded by observed min/max widened by a slack factor epsilon_num; categorical or highly sensitive discrete parameters can be protected with exact-match whitelists; free-form strings are guarded using an embedding-space centroid and radius, with an optional expansion epsilon_str. At runtime, the gateway maintains the current DFA state per session. For each proposed tool call, it performs a constant-time lookup for an outgoing transition matching the tool name from the current state. If no transition exists, the call is blocked as structurally anomalous. If a transition exists, the parameters are validated against the attached guards. Only calls satisfying both sequence and parameter constraints are forwarded to the environment. Blocked calls are written to an append-only cryptographic audit log. The architecture is intentionally decoupled: no LLM inference or semantic analysis occurs online, allowing sidecar deployment with bounded overhead. The design targets structured workflows where benign behavior forms stable trajectories; it explicitly assumes the profiling corpus is clean and does not address poisoning during the offline cold-start phase.

### 🧪 Experiments
Evaluation is conducted on the Agent Security Bench (ASB) across five scenarios, including three structured workflows and two less-structured settings. The main comparison baseline is Aegis, a state-of-the-art stateless tool-call scanner. Metrics include attack success rate (ASR), benign task failure rate (BTFR), and per-call latency overhead. Across all five ASB scenarios, Praetor achieves a macro-averaged ASR of 5.6%. On the three structured workflows, ASR drops to 2.2%, compared with 12.8% for Aegis. Praetor reaches 0% ASR on multi-step and context-sequential attacks in structured settings, highlighting the benefit of stateful trajectory enforcement over isolated-call inspection. In a splice-based robustness test using 1,000 algorithmically generated exfiltration payloads, only 1.4% aligned with valid structural paths; all 14 surviving paths were then rejected by end-to-end string parameter guards, yielding 0 successful attacks out of 14 and a reported 95% confidence interval of [0%, 23.2%]. Runtime cost is low: about 2.2 ms added per tool call, which is a 3.7x speedup over Aegis. On benign workloads, Praetor records a 2.0% BTFR, indicating modest disruption. The paper also reports a weakness in continuous string/semantic parameter bounds: synonym-substitution attacks evade these guards at an 18% rate when exact-match whitelisting is not used for sensitive values.

### 🌱 Insights & Implications
The central insight is that the main vulnerability of existing agent firewalls is not only weak content scanning but lack of sequential memory; malicious behavior often emerges only over a trajectory of individually acceptable calls. Modeling tool-use as a stateful language of benign workflows sharply compresses the feasible attack surface, especially for software-like agents with repeatable task structure. The results suggest that structural enforcement is the dominant defense for multi-step attacks, while parameter validation acts as a second line of defense inside the surviving structural envelope. However, the paper also shows that approximate semantic guards for strings are brittle: synonym-substitution can bypass continuous bounds, so exact-match whitelisting remains essential for high-risk parameters. Important limitations include dependence on a clean benign telemetry corpus, sensitivity to workflow regularity, potential false positives when legitimate behavior drifts, and weaker applicability to highly open-ended agents whose action space is not tightly structured. Open research questions include how to safely update profiles under concept drift, how to defend against telemetry poisoning during profile construction, how to generalize from benign traces without over-permitting attacks, and whether richer models than fixed-window DFAs can preserve bounded runtime while capturing longer-range dependencies.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and secure agentic systems because it treats malicious tool-use as a behavioral anomaly detection problem at execution time, analogous to runtime protection for software workflows. For a researcher interested in automated repair and vulnerability mitigation, it offers a practical enforcement mechanism that can prevent exploit execution without modifying the underlying model or application logic, and it provides a strong framework for reasoning about stateful attack surfaces, sequential policy enforcement, and low-overhead runtime defenses for AI-driven software systems.

## 📄 [Large Language Models as Explainable Cyberattack Detectors for Energy Industrial Control Systems](https://arxiv.org/abs/2604.26079v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Weiyi Kong, Ahmad Mohammad Saber, Amr Youssef, Deepa Kundur
📅 **Published:** 2026-04-28  🏷️ **Categories:** cs.CR

### 💡 TL;DR
The paper evaluates an off-the-shelf GPT-4o model as a prompt-only, human-in-the-loop detector for Modbus traffic in industrial control systems, framing detection as a binary normal vs. critical triage problem and pairing each alert with a token-grounded audit record. On two public ICS datasets, the LLM achieves about 0.98 accuracy with strong critical-class recall, broadly matching supervised baselines while requiring no task-specific training or fine-tuning.

### 🏆 Key Contributions
- Demonstrates that a few-shot, prompt-configured off-the-shelf LLM can perform competitive Modbus intrusion triage on public ICS datasets without any task-specific weight updates.
- Introduces a two-pass detection-and-auditing pipeline that separates binary prediction from generation of a structured, operator-facing incident record grounded in verbatim protocol tokens.
- Provides intervention-based diagnostics for the audit record, including sufficiency, necessity, and counterfactual-style probes, to test whether cited tokens are actually decision-relevant to the model’s own prediction.
- Establishes a matched evaluation setting against supervised baselines using the same event information and shared train/validation/test splits across two public Modbus benchmarks.

### ⚙️ Methodology
The study formulates ICS intrusion detection as a binary classification task over individual Modbus communication instances, where all attack periods and other safety-critical behaviors are collapsed into a single positive class called critical, and all remaining traffic is labeled normal. The goal is not autonomous response or fine-grained attack attribution, but conservative operator triage.

Each raw Modbus/TCP frame is transformed into a compact textual token sequence containing a small set of protocol and timing attributes: traffic direction (client-to-server or server-to-client), function code, discretized unit identifier, discretized payload length, discretized inter-arrival time, and an exception flag. Continuous values such as payload length and inter-arrival time are bucketized using boundaries learned only from the training split and then reused on validation and test to prevent leakage. A typical encoded instance looks like: "DIR:C2S FC:03 UNIT:U1 LEN:B2 IAT:B4 EX:0". The representation intentionally omits raw IP addresses and other high-entropy details to keep prompts short and semantically focused.

The classifier is an off-the-shelf GPT-4o model accessed in text-only API mode with no fine-tuning, no parameter updates, and temperature-0 decoding to reduce randomness. Task behavior is specified entirely through a prompt configuration that contains generic Modbus semantics, ICS safety cues, and a small number of few-shot exemplars. The first pass performs classification, outputting a discrete label in {normal, critical} plus a self-reported confidence score stored only for later analysis, not for thresholding. The second pass is an auditor that takes the fixed prediction and asks the LLM to produce a concise incident record for analyst review. This record includes copied evidence tokens from the input, risk-category tags, and when possible an optional minimal counterfactual edit describing what token changes might flip the decision.

A key design choice is strict separation between prediction and explanation: once the first-pass label is emitted, the second pass can only justify it, not revise it. This makes the explanation an audit artifact tied to a fixed decision. The paper emphasizes that these are token-grounded audit signals rather than full human-grounded explanations.

To evaluate explanation quality, the authors apply intervention-based diagnostics inspired by rationale evaluation. Sufficiency-style tests check whether the cited evidence tokens alone are enough to preserve the original prediction; necessity-style tests check whether removing or masking cited tokens changes the prediction; and counterfactual probes test whether proposed token edits alter the decision as intended. These diagnostics aim to assess whether the audit record captures features that are causally relevant to the model’s own behavior, rather than merely plausible post hoc justifications.

### 🧪 Experiments
The experiments use two public ICS Modbus benchmarks: the LeMay CSET’16 Modbus dataset and the CIC Modbus 2023 dataset. Both are converted into a unified binary normal/critical labeling scheme by marking all attack periods and other safety-critical behaviors as critical. The authors use a shared preprocessing and ingestion pipeline and evaluate all methods under matched event information and canonical train/validation/test splits.

The LLM receives the compact token-string representation of each Modbus frame and is evaluated in a prompt-only few-shot setup, with no task-specific model training. Supervised baselines are trained on the full task-specific training split and evaluated on the same held-out test data. For supervised methods that output continuous scores, thresholds are selected on the validation split to maximize macro-F1 and then fixed for test evaluation. Reported metrics include accuracy, macro-F1, and critical-class recall and F1.

Quantitatively, the paper reports that the LLM-based triage pipeline achieves approximately 0.98 accuracy across both benchmarks, with high recall on the critical class, and is broadly comparable to strong supervised baselines under the matched setup. The main comparative result is not that the LLM decisively outperforms supervised detectors, but that it reaches similar predictive performance while avoiding retraining and simultaneously generating an audit record. For the explanation component, intervention-based sufficiency and necessity tests provide evidence that the cited tokens are often decision-relevant to the model’s own predictions, supporting the usefulness of the generated incident records as auditable artifacts.

### 🌱 Insights & Implications
The paper’s main insight is that LLMs can be useful in ICS security not only as natural-language assistants but also as direct classifiers over carefully designed protocol-token representations. This reframes an LLM from a documentation aid into a lightweight, prompt-configured triage layer that can sit alongside traditional detectors.

A second important insight is that explainability for operational security may be better served by auditable, token-grounded incident records than by generic feature-attribution visualizations alone. The two-pass design creates artifacts that are easier to archive and review in incident-response workflows, even if they do not constitute full faithful explanations in the human-centered xAI sense.

The work also highlights a practical tradeoff: by collapsing all attacks and safety-critical states into a single positive class, the system supports conservative review-oriented alerting but gives up detailed attack attribution. This is aligned with deployment scenarios where the first priority is surfacing suspicious traffic quickly, leaving downstream diagnosis to analysts or specialized classifiers.

Limitations include the narrow focus on offline evaluation over public datasets, one protocol family (Modbus), and one binary task formulation. The method does not address API privacy/governance concerns, online latency, drift, adversarial prompt sensitivity, or robustness under noisy real-world traffic. The explanation diagnostics assess decision relevance to the model itself, not whether operators find the records accurate, useful, or trustworthy. The confidence score is uncalibrated, and the paper does not claim replacement of supervised IDS systems. Open questions include extending the approach to multi-class attack categorization, integrating temporal context across packet sequences rather than single-frame classification, validating explanations with domain experts, and studying robustness against adaptive attackers who may manipulate token patterns to exploit prompt-driven detectors.

### 🎯 Why This Paper
This paper is highly relevant to software/security research interests in vulnerability detection and repair-adjacent cyber defense because it explores explainable detection in safety-critical industrial systems using a zero-training LLM pipeline. For a researcher in software engineering, maintenance, and security, it offers a concrete methodology for combining competitive detection with auditable evidence generation, suggesting new directions for human-in-the-loop security tooling, explainable alert triage, and prompt-based alternatives to retraining-heavy supervised systems.
