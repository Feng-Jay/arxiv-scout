## 📄 [PracRepair: LLM-Empowered Automated Program Repair Inspired by Human-Like Debugging Practices](https://arxiv.org/abs/2606.17612v1)  `100% relevant`

👤 **Authors:** Yu Cheng, Zhongxin Liu, Zhenchang Xing, Chao Ni, Qing Huang, Xiaoxue Ren
📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.SE

### 💡 TL;DR
PracRepair is an LLM-based automated program repair framework that mimics human debugging by combining on-demand static and dynamic execution context, explicit failure diagnosis, and iterative patch refinement using detailed validation feedback. On Defects4J and real-world bugs, it substantially outperforms prior LLM-based APR methods, with especially strong gains under GPT-4o.

### 🏆 Key Contributions
- Introduces a fully automated APR framework that explicitly incorporates dynamic debugging signals rather than relying mainly on static context, retrieved examples, or coarse test outcomes.
- Proposes on-demand static-dynamic context construction to make large and noisy failure-execution information more useful and manageable for LLM reasoning.
- Uses question-driven failure diagnosis to derive explicit repair hypotheses, aligning patch generation with human-like debugging practices.
- Iteratively refines candidate patches with richer validation diagnostics and trace-level behavioral change analysis instead of simple pass/fail feedback.
- Demonstrates strong empirical performance on Defects4J v1.2, Defects4J v2.0, and Real-World Bugs across multiple foundation models.

### ⚙️ Methodology
The paper presents PracRepair as a multi-stage LLM-driven APR pipeline designed around how human developers debug. First, it gathers an on-demand static-dynamic context from the buggy program and failing executions, likely including relevant source regions, stack traces, execution traces, variable/value flow, and test-failure evidence, while filtering or focusing context to reduce the noise and scale problems of raw traces. Second, instead of prompting the LLM to patch directly, it performs question-driven failure diagnosis: the model is guided to ask or answer structured diagnostic questions about why the failure occurs, what program state or control-flow conditions are abnormal, and what repair hypothesis would explain the observed behavior. This intermediate reasoning step converts raw context into explicit, self-explanatory bug understanding. Third, based on the diagnosis, the LLM generates candidate patches. Fourth, patch validation is not treated as a binary signal only; PracRepair extracts validation diagnostics and trace-level behavioral changes from patch execution, using them to understand why a candidate failed or partially improved the behavior. Finally, the framework iteratively refines patches using these richer dynamic signals until a plausible or correct fix is found. Overall, the architecture appears to couple context selection, diagnosis, generation, execution-based validation, and feedback-driven repair refinement in a closed loop centered on dynamic evidence.

### 🧪 Experiments
The evaluation uses Defects4J v1.2 and Defects4J v2.0 as the main benchmarks, plus RWB (Real-World Bugs) to assess generalization to less curated, practical defects. The abstract indicates comparison against state-of-the-art baselines in LLM-based APR, likely including methods that rely on static prompts, retrieval, error messages, or coarse validation feedback. Performance is reported in number of correctly fixed bugs. On Defects4J v1.2/v2.0, PracRepair fixes 139/136 bugs with GPT-3.5 and 162/171 bugs with GPT-4o, consistently outperforming prior methods. The phrasing suggests this superiority holds across both benchmark versions and multiple foundation models. On RWB, PracRepair achieves the best performance among compared methods, indicating stronger external validity and model-agnostic effectiveness. Although the abstract does not specify exact baselines, metrics beyond number of correct fixes, or statistical testing, the reported gains imply that dynamic-context utilization and iterative validation-aware refinement materially improve repair success.

### 🌱 Insights & Implications
A key insight is that LLM-based APR benefits not just from more code context, but from better-structured dynamic evidence and explicit diagnostic reasoning. The work suggests that failure-execution traces and patch-validation traces are underexploited information sources in current APR systems, and that transforming them into targeted, interpretable prompts can significantly improve repair. For software engineering and APR research, this points toward agentic repair loops that reason over execution behavior rather than one-shot patch synthesis. For vulnerability repair and maintenance, the same principle could be valuable when failures are security-relevant, though the abstract does not claim security-specific evaluation. Likely limitations include reliance on executable tests and informative failure traces, possible cost/latency from iterative tracing and LLM calls, and sensitivity to how dynamic context is summarized or selected. Open questions include how robust the method is on weak test suites, multi-location bugs, non-deterministic failures, and security bugs where correctness and exploitability are harder to validate with ordinary unit tests.

### 🎯 Why This Paper
This paper is highly relevant because it advances automated program repair using LLMs in a way that is directly aligned with software maintenance workflows: debugging, diagnosis, patch generation, and iterative validation. It is especially valuable for research on APR, vulnerability repair, and dependable software engineering because it shows how dynamic execution information and feedback loops can significantly improve repair effectiveness beyond static prompting, suggesting promising directions for test-driven bug fixing and potentially security-oriented repair systems.

## 📄 [SoK: AI-Augmented Binary Reversing](https://arxiv.org/abs/2606.17398v1)  `90% relevant`  `🌐 Full Text`

👤 **Authors:** Yujeong Kwon, Yiyue Zhang, Shakhzod Yuldoshkhujaev, Kexin Pei, Dokyung Song, Hyungjoon Koo
📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.CR, cs.AI, cs.SE

### 💡 TL;DR
This paper systematizes a decade of research on AI-augmented binary reversing by analyzing 144 papers across 22 inference domains and proposing a unified taxonomy that links conventional binary analysis, binary-derived artifacts, representation learning, and downstream AI inference. Its main result is a field-wide framework showing that AI methods, including deep learning, LLMs, and agentic systems, primarily augment rather than replace traditional reversing, while exposing major evaluation, generalization, and deployment gaps.

### 🏆 Key Contributions
- Presents the first comprehensive SoK of AI-augmented binary reversing, covering 144 papers published from 2015 to 2025.
- Organizes the literature into 22 binary reversing domains based on inference tasks, providing a cross-domain view of the field.
- Introduces a unified two-pipeline taxonomy that connects conventional reversing techniques and AI-augmented inference through binary-derived analysis artifacts.
- Characterizes the roles of artifacts, representations, learning paradigms, LLMs, and agentic AI systems within binary reversing workflows.
- Evaluates the maturity of the field and identifies recurring technical challenges, validity threats, and evaluation gaps across datasets, tools, model designs, and benchmarks.
- Highlights underexplored research opportunities and provides an interactive visualization platform to support future research and practice.

### ⚙️ Methodology
The paper is a systematization-of-knowledge study rather than a new reversing algorithm. The authors collect a corpus of 144 papers using keyword-based search in Google Scholar, manual screening of 11 top-tier venues in security, software engineering, and AI, and backward/forward snowballing of highly cited work. The scope is restricted to native executable binaries and studies whose inference relies on binary analysis artifacts rather than source code at inference time. From this corpus, the authors derive a unified conceptual model with two linked pipelines. The conventional pipeline includes four analysis modes: triage, static analysis, dynamic analysis, and security testing. These produce binary-derived artifacts such as code, graphs, traces, and other structured observations. The AI-augmented pipeline then transforms those artifacts into model-consumable representations, applies learning paradigms such as supervised, self-supervised, retrieval-based, or generative modeling, and performs downstream inference tasks like detection, classification, recovery, retrieval, clustering, or summarization. A key methodological contribution is the artifact-centric view: binary-derived artifacts serve as the interface between conventional analysis and AI methods. This framing allows the paper to compare otherwise heterogeneous work along common dimensions: reversing domain, artifact type, representation strategy, model family, and evaluation method. The taxonomy also explicitly incorporates emerging LLM and agentic-AI approaches, emphasizing reasoning, prompting, planning, and tool use as a newer layer on top of artifact-driven reversing workflows.

### 🧪 Experiments
As a SoK, the paper does not report a single benchmarked model evaluation; instead, its empirical basis is a structured analysis of 144 published studies. The review spans 2015–2025 and covers papers from top venues including S&P, CCS, USENIX Security, NDSS, ICSE, FSE, AAAI, IJCAI, NeurIPS, ICLR, and ICML. The authors categorize work into 22 binary reversing domains and compare studies across task formulations, artifact choices, data representations, learning approaches, and evaluation practices. The paper’s main empirical findings are qualitative and meta-analytic rather than model-level performance numbers: the literature is fragmented across domains and benchmarks; similar modeling ideas are reused across adjacent tasks; many reported gains are benchmark-specific; and cross-study comparisons are often invalidated by differing datasets, baselines, and protocols. The study highlights common experimental weaknesses such as narrow datasets, limited realism, poor reproducibility, insufficient cross-architecture or cross-compiler generalization, and weak integration into end-to-end analyst workflows. It also notes the growing but still immature use of LLMs and agentic systems in binary reversing evaluation.

### 🌱 Insights & Implications
The paper’s central insight is that AI for binary reversing should be understood as an augmentation layer over conventional analysis rather than a replacement for it. Traditional static, dynamic, symbolic, and testing-based analyses remain essential because they generate the artifacts from which AI systems learn. A second major insight is that many apparently different tasks share the same structural pipeline: binaries are transformed into artifacts, artifacts into representations, and representations into semantic inferences. This suggests opportunities for transfer learning, unified benchmarks, and reusable infrastructures across domains such as vulnerability discovery, malware analysis, and software understanding. The paper also identifies persistent limitations: irreversible semantic loss from compilation, sensitivity to compiler/architecture variation, obfuscation and anti-analysis defenses, benchmark overfitting, inconsistent evaluation methodologies, and a gap between paper-level success and real-world analyst utility. For LLMs and agentic AI, the paper implies promise in reasoning, summarization, and tool orchestration, but also raises open questions around hallucination, reliability, grounding in binary artifacts, and integration with rigorous program analysis. More broadly, the study argues that future progress depends on better datasets, stronger external-validity testing, workflow-level evaluations, and systems that combine learned priors with principled analysis.

### 🎯 Why This Paper
This paper is highly relevant because it directly spans binary analysis, vulnerability discovery, malware/security analysis, and software understanding, all of which intersect with software security, maintenance, vulnerability detection, and automated repair workflows. For a researcher in software engineering and security, it provides a structured map of AI methods, artifacts, tasks, and evaluation pitfalls that can inform the design of more reliable vulnerability detectors, binary-level repair systems, and maintenance tools, especially when source code is unavailable or incomplete.

## 📄 [Unlocking LLM Code Correction with Iterative Feedback Loops](https://arxiv.org/abs/2606.17514v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Le Zhang, Suresh Kothari
📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.SE, cs.AI

### 💡 TL;DR
This paper studies whether LLMs can fix their own buggy code when given execution feedback, using iterative loops over real LeetCode problems in Python and Java. It finds that reasoning models improve substantially across iterations and use feedback much more effectively than non-reasoning models, while syntax/runtime failures are much easier to repair than logical or algorithmic mistakes.

### 🏆 Key Contributions
- Introduces a systematic evaluation framework for LLM code correction via iterative feedback loops using compiler errors and testcase results.
- Builds three complementary datasets—Core, Strain, and Challenge—to assess baseline generation, efficiency under constraints, and hard-to-solve refinement scenarios.
- Defines and applies metrics for iterative refinement behavior, including success over iterations and error-type-specific rectification patterns.
- Provides an empirical comparison between reasoning and non-reasoning LLMs across two programming languages, showing consistent advantages for reasoning models in self-correction.
- Identifies which bug classes are most and least repairable under automated feedback, yielding practical guidance for deploying LLM-driven coding systems.

### ⚙️ Methodology
The study evaluates four LLMs—DeepSeek-R1, DeepSeek-V3, GPT-o4-mini, and GPT-4.1-mini—on code generation and self-correction tasks in Python and Java using LeetCode problems. It first establishes a pass@1 baseline on a Core Dataset of 450 problems evenly split across easy, medium, and hard difficulties. It then uses a Strain Dataset of 200 efficiency-sensitive problems to examine whether models respond to explicit optimization-oriented guidance, and a Challenge Dataset of 32 problems that were most frequently failed in the baseline to stress-test iterative repair.

The iterative refinement framework is automated: a model generates code, the code is executed against LeetCode testcases, failure signals are collected, and a new prompt is constructed containing the prior code attempt plus execution feedback such as compiler errors or testcase failures. The model then produces a revised program, and this loop continues until success or until a fixed iteration budget is exhausted. The setup is designed to mimic realistic programming workflows in which automated systems surface compiler/runtime/test feedback for correction.

Prompting is standardized across models. The baseline prompt instructs the model to act as a software developer, specifies the target language, includes the problem description, examples, constraints, a partial starter snippet, and asks for executable code only. For baseline pass@1 experiments, decoding is set to low temperature for stability (temperature 0.1, top-p 0.95 where configurable). For iterative refinement, decoding is made more exploratory (temperature 0.9), and top-p is calibrated through preliminary experiments; top-p=0.3 is selected because it achieved the best observed refinement performance in pilot runs. The iteration limit is also calibrated empirically and set to 10 after performance was observed to plateau around iteration 8.

The analysis is organized around five research questions: baseline difficulty-sensitive performance, responsiveness to optimization prompts, ability to correct failed code with execution feedback, differences between reasoning and non-reasoning models, and relative repairability of syntactic, logical, and algorithmic errors. The overall methodology emphasizes realistic test-driven feedback, controlled model comparison, and failure-type analysis rather than only one-shot correctness.

### 🧪 Experiments
The experiments cover four LLMs, two languages (Python and Java), and three datasets. The Core Dataset contains 450 LeetCode problems (150 easy, 150 medium, 150 hard) for pass@1 baseline evaluation. The Strain Dataset contains 200 manually selected efficiency-demanding problems to test whether models can adapt to instructive optimization prompts and meet computational constraints. The Challenge Dataset contains 32 problems that were most frequently failed across models/languages in the baseline, making it suitable for iterative correction analysis.

Code is submitted through a custom API to LeetCode and evaluated with official test cases. Baseline evaluation uses pass@1. Iterative refinement uses metrics such as ISR@10 (iterative success rate within 10 iterations) and MIS, alongside analyses of rectification trajectories and error categories. A preliminary hyperparameter calibration on DeepSeek-R1 shows top-p=0.3 yields the strongest iterative performance, with ISR@10 reported at 68.8% and lower or comparable settings performing worse. Another pilot study on 10 Challenge problems indicates improvement largely saturates after about 8 iterations, motivating a 10-iteration cutoff.

The main reported findings are qualitative but clear: reasoning models consistently improve over successive iterations and substantially outperform non-reasoning models in leveraging feedback. Syntactic and runtime/compiler-related failures are much more frequently repaired than logical or algorithmic failures. The study also reports that explicit optimization guidance matters for efficiency-constrained tasks, connecting feedback responsiveness to practical code generation quality. While the provided text does not include all final per-model/per-language tables, the paper’s experimental design supports direct comparison across difficulty tiers, error types, languages, and reasoning capability.

### 🌱 Insights & Implications
A central insight is that one-shot code generation substantially understates the practical usefulness of LLMs in programming settings: some models can recover meaningfully when placed in a feedback loop, but that capability is highly model-dependent. Reasoning models appear better able to interpret error signals, revise hypotheses, and progressively converge on correct solutions, suggesting that internal reasoning capacity is especially valuable for automated repair workflows.

The paper also shows that not all failures are equally repairable. Syntax, compilation, and many runtime issues are comparatively tractable because execution feedback is localized and concrete. In contrast, logical and algorithmic failures often require deeper understanding of problem constraints, hidden corner cases, or asymptotic complexity, making them harder to repair from raw testcase feedback alone. This is highly relevant to software engineering because it suggests where automated LLM repair may be reliable and where additional tooling—symbolic analysis, richer traces, specification mining, or stronger verification signals—may be needed.

Another implication is methodological: pass@1 alone is insufficient for evaluating coding assistants intended for real development. Researchers should measure iterative improvement, failure transitions, and per-error-class recovery. A limitation is that the tasks are competitive-programming style LeetCode problems rather than industrial software maintenance or security repair tasks, so external validity to large codebases, multi-file contexts, and vulnerability-fixing workflows remains open. Another limitation is that feedback is primarily execution-based; richer debugging artifacts or static-analysis warnings might alter repair dynamics. Open questions include how feedback quality, prompt structure, and test coverage affect repair success; whether iterative loops can be combined with retrieval or planning; and how these findings extend to security-critical bugs and automated program repair benchmarks.

### 🎯 Why This Paper
This paper is highly relevant because it studies automated code correction rather than only code generation, which directly connects to software maintenance, automated program repair, and practical developer workflows. Its iterative-feedback evaluation framework, error-type analysis, and comparison of reasoning vs. non-reasoning models provide useful foundations for research on vulnerability repair and fault localization, especially for understanding when LLMs can autonomously fix failures and when stronger feedback or verification mechanisms are required.

## 📄 [ARVO: Atlas of Reproducible Vulnerabilities for Open-Source Software](https://arxiv.org/abs/2606.17283v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Xiang Mei, Jordi Del Castillo, Pulkit Singh Singaria, Haoran Xi, Abdelouahab Benchikh, Tiffany Bao, Ruoyu Wang, Yan Shoshitaishvili, Adam Doupé, Hammond Pearce, Brendan Dolan-Gavitt
📅 **Published:** 2026-06-15  🏷️ **Categories:** cs.CR, cs.AI, cs.LG

### 💡 TL;DR
ARVO is a framework and dataset that makes large-scale open-source vulnerabilities reproducible over time by rebuilding historical vulnerable versions, replaying proof-of-concept inputs, and locating corresponding fix commits. Applied to OSS-Fuzz, it produces 6,138 reproducible real-world vulnerabilities across 311 projects, reproduces 81% of attempted cases, and identifies correct patches with 89.4% accuracy, substantially outperforming prior OSS-Fuzz-derived datasets.

### 🏆 Key Contributions
- Introduces reproducibility as a first-class property for security vulnerability datasets, arguing it is essential for verifiable downstream research such as patch generation, patch validation, and vulnerability analysis.
- Identifies major obstacles to long-term large-scale vulnerability reproduction in OSS datasets, including dependency drift, build-system bitrot, incorrect metadata, and complex multi-component project setups, and provides general solutions to mitigate them.
- Designs ARVO, an automated system that can rebuild historical vulnerable software versions, replay triggering inputs, support source modifications, and automatically locate precise fixing commits.
- Constructs and releases the ARVO dataset with over 6,100 real-world C/C++ memory-safety vulnerabilities, each packaged with reproducible environments, proof-of-concept inputs, fix metadata, and prebuilt Docker images.
- Demonstrates that reproducibility improves upstream data quality by uncovering many OSS-Fuzz/OSV mislabelings, including vulnerabilities incorrectly marked as fixed or with inaccurate patch associations.

### ⚙️ Methodology
ARVO is built as an automated reproduction pipeline over historical OSS-Fuzz vulnerabilities, targeting real-world memory-safety bugs in open-source C/C++ projects. Its core design goal is full reproducibility: for each vulnerability, ARVO reconstructs a historical environment where the vulnerable version can be rebuilt from source, the vulnerability can be deterministically triggered using a proof-of-concept input, and subsequent analysis can be performed after arbitrary source changes. To achieve this, the system addresses long-term reproduction failures caused by dependency drift, outdated build infrastructure, disappearing external resources, and project-specific build complexity. Rather than relying on the original time-of-discovery environment, ARVO reconstructs builds in standardized containerized environments and applies generalized repair strategies to historical build and dependency issues so the software can compile again. Once a vulnerable version is rebuilt, ARVO validates reproduction by executing the associated PoC and confirming the expected sanitizer-detected crash behavior. Reproducibility is then leveraged to identify the fixing patch more precisely: ARVO can test candidate revisions around the reported fix range and determine when the triggering input stops causing the vulnerability, allowing automatic association of a fix commit with the reproduced issue. Because the rebuilt environment remains usable across versions, ARVO also supports customized source modifications and direct interaction with vulnerabilities after code changes, enabling downstream applications such as vulnerability porting, patch testing, and automated repair evaluation. The released artifact includes metadata, evaluation infrastructure, and paired vulnerable/fixed Docker images for each successful reproduction.

### 🧪 Experiments
The evaluation is conducted on historical vulnerabilities from OSS-Fuzz/OSV, focusing on C/C++ memory-safety bugs with proof-of-concept crashing inputs. ARVO is compared against prior OSS-Fuzz-derived dataset construction approaches in terms of successful long-term reproduction and patch identification quality. The resulting dataset contains 6,138 successfully reproduced vulnerabilities across 311 projects. ARVO reproduces 81% of attempted vulnerabilities, a 44 percentage-point improvement over the prior state of the art and roughly 2.19x better overall reproduction effectiveness; the paper contrasts this with OSS-Fuzz-OSV reproducing only around 37% in comparable historical settings. The dataset also scales to complex multi-component software, which prior methods struggle to handle. For patch location, ARVO automatically evaluates developer-provided fixes and achieves 89.4% accuracy in matching vulnerabilities to the corresponding patch commits. In analyzing upstream metadata quality, the authors report 1,518 incorrect vulnerability records among 10,440 OSS-Fuzz entries, an error rate of 14.5%, and identify more than 300 vulnerabilities that were improperly detected as fixed and remained unpatched. The released artifact includes 12,276 Docker images, corresponding to vulnerable and fixed versions for each successful reproduction.

### 🌱 Insights & Implications
The paper’s main insight is that reproducibility is not just a convenience but an enabling substrate for vulnerability research: once a vulnerability can be rebuilt and retriggered reliably, many downstream tasks become automatically verifiable, including patch locating, patch validation, exploit-trigger regression testing, and automated repair benchmarking. ARVO also shows that large historical vulnerability corpora contain substantial metadata noise; reproducible rebuilding can act as a quality-control mechanism for upstream vulnerability databases. A key implication is that security benchmarks should move beyond static commit links or textual CVE metadata and instead distribute executable, reproducible artifacts. Limitations include the paper’s current focus on C/C++ memory-safety vulnerabilities that manifest as crashing PoCs, meaning non-crashing logic flaws, many semantic security bugs, and software ecosystems outside this scope are not yet covered. The reported need for some system-wide maintenance to handle defunct external resources suggests that fully automatic long-term preservation remains challenging. Open questions include how well the approach generalizes to other languages, vulnerability classes, exploitability conditions beyond sanitizer crashes, and automated program repair tasks that require semantic correctness rather than merely crash elimination.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, vulnerability repair, and automated program repair because it provides a large, realistic, reproducible benchmark where vulnerabilities can be rebuilt, triggered, patched, and revalidated end to end. For researchers working on APR, patch ranking, vulnerability localization, or repair verification, ARVO offers ground-truth fixes plus executable vulnerable/fixed environments, making it especially valuable for evaluating whether generated repairs truly remove the vulnerability without relying on noisy historical metadata.

## 📄 [Loss Landscape Poisoning: Targeted Extraction of Unseen Training Data from LLMs](https://arxiv.org/abs/2606.17110v1)  `80% relevant`  `🌐 Full Text`

👤 **Authors:** Md Abdullah Al Mamun, Ngoc Phu Doan, Pedram Zaree, Ihsen Alouani, Nael Abu-Ghazaleh
📅 **Published:** 2026-06-15  🏷️ **Categories:** cs.CR, cs.LG

### 💡 TL;DR
This paper introduces Loss Landscape Poisoning (LLP), a training-time attack that makes an LLM or VLM memorize and leak a target training record the attacker never sees by reshaping the local loss surface around that record. The authors show strong extraction success across centralized and federated settings, and further demonstrate that while DP-SGD blocks direct generation-based extraction, a new black-box loss-based attack, Direct Loss Region Probing (DLRP), can still recover the target from the preserved loss geometry.

### 🏆 Key Contributions
- Introduces Loss Landscape Poisoning (LLP), a targeted poisoning attack that induces memorization of unseen sensitive training records by creating a sharp local loss minimum around the true target and elevated loss around nearby alternatives.
- Shows a white-box/model-poisoning LLP variant that uses modified training objectives such as gradient ascent on poison samples to amplify leakage in centralized and federated learning settings.
- Develops LLP-Data, a more realistic data-only poisoning variant that crafts poison examples whose standard supervised gradients approximate the desired ascent effect without modifying the training loop.
- Demonstrates cross-client privacy leakage in federated learning, where a malicious client can extract secrets held only by honest clients through the final global model.
- Analyzes differential privacy as a defense, showing DP-SGD suppresses direct generation leakage but leaves a detectable relative loss-landscape signature.
- Introduces Direct Loss Region Probing (DLRP), a new black-box extraction primitive that ranks candidate secrets by local loss geometry and can recover poisoned targets even under DP-SGD at usable privacy budgets.

### ⚙️ Methodology
The paper’s core idea is to manipulate the local loss landscape around a target record so that the true secret becomes the only low-loss point in its neighborhood, forcing memorization instead of smooth generalization. The attacker is assumed to know the schema or context of the target record (for example, a structured form with one unknown secret field) but not the actual secret value. During normal victim training, the true target record naturally contributes a downward force on its own loss. LLP adds an opposing upward force on nearby candidate completions so that the model cannot fit the region smoothly; instead, it must memorize the exact target as an isolated sharp minimum.

The paper studies three threat settings. In direct model poisoning, the attacker can alter the training process and apply a modified objective on poison samples, such as gradient ascent, while normal data still uses standard supervised descent. This explicitly increases loss on decoy samples in the target neighborhood. In federated learning, a malicious client uses the same idea locally, relying on aggregation to imprint the poisoned geometry into the global model and thereby induce leakage of another client’s secret. In the more realistic LLP-Data setting, the attacker cannot change the optimizer or loss function, so they craft poison examples whose ordinary supervised gradients reproduce a similar effect indirectly, increasing loss around the target region under standard training.

A key conceptual result is that extraction success depends more on the relative loss ratio between the target and neighboring alternatives than on simply maximizing absolute loss. The attack therefore aims to sculpt a sharp basin around the true completion while preserving global task utility and avoiding obvious model degradation. After training, the attacker performs black-box extraction by querying the model with the known prefix/context and searching for the memorized completion.

For defenses, the authors study DP-SGD and observe that clipping and noise flatten the landscape enough to defeat direct generation of the secret. However, they argue that the geometric fingerprint persists: the target still tends to sit in a locally distinct low-loss region relative to nearby candidates. Based on this, they propose Direct Loss Region Probing (DLRP), a black-box attack that enumerates candidate completions in the target region and ranks them by how their local loss changes under small perturbations; the true secret is identified as the candidate surrounded by higher-loss neighbors.

### 🧪 Experiments
The evaluation covers both language models and vision-language models, under centralized training, data-poisoning-only training, and federated learning. The paper reports direct generation/extraction performance when prompting the trained model with the relevant context/schema and measuring whether the hidden target record is recovered. Baselines are near-random extraction probabilities around 0.01-0.04 before poisoning. Under white-box/model-poisoning LLP, the probability of generating the target rises to roughly 0.7-0.92, with aggregate leakage success reaching up to 100% on LLMs and up to 90% on VLMs. In federated learning, a single malicious client achieves comparable cross-client leakage, including up to 100% leakage of LLM secrets and 86% on VLM secrets.

For the stronger data-only threat model, LLP-Data still attains high extraction rates while preserving model utility: up to 86% leakage on LLMs and 83% on VLMs under standard training pipelines without attacker control of the loss or optimizer. The study emphasizes that these results are obtained even though the attacker never observes the target training record itself.

The paper also evaluates differentially private training with DP-SGD. Direct extraction through generation is effectively blocked under LLP and LLP-Data because clipping and noise flatten the attacker-created local minimum. However, using the proposed DLRP loss-probing attack, the authors report that target recovery remains possible at privacy budgets where model accuracy remains usable. The main quantitative takeaway is that DP-SGD can stop generation-based leakage but not landscape-level inference based on black-box loss queries.

### 🌱 Insights & Implications
The paper reframes privacy leakage as a geometric property of the learned loss surface, not just a consequence of memorized outputs appearing under prompting. This is important because standard extraction evaluations often focus on generation success alone, while LLP shows that poisoning can intentionally engineer memorization of unseen records by shaping local optimization behavior.

A notable insight is that relative target-vs-neighborhood loss structure is the real attack driver. This suggests that defenses focused only on reducing absolute confidence or output probability may be insufficient if the model still encodes a uniquely favorable basin around the secret. The federated learning result is especially concerning for collaborative training scenarios, where one malicious participant may be able to induce leakage of another participant’s private data without direct access to it.

The DP analysis is also significant: although DP-SGD blocks the obvious attack channel of direct generation, the paper argues that it does not fully remove the attack’s structural fingerprint. This raises open questions about whether current privacy notions and empirical audits for LLMs adequately cover loss-surface side channels, especially when models expose token-level losses or scoring APIs.

Potential limitations include the need for schema/context knowledge, candidate-space structure around the secret, and a poison budget sufficient to shape the local region without harming utility. The success of DLRP may also depend on practical access to black-box loss queries, which is not always available in public APIs. Still, the work has strong implications for secure training pipelines, privacy-preserving federated learning, and the evaluation of poisoning defenses beyond standard backdoor or integrity metrics.

### 🎯 Why This Paper
This paper is highly relevant to software security and vulnerability-related research because it introduces a new training-time attack surface for foundation models: poisoning that induces targeted privacy leakage of records the attacker never sees. For a researcher interested in software security, vulnerability detection/repair, and trustworthy ML systems, it offers a strong conceptual and experimental framework for understanding how adversarial manipulation of training dynamics can bypass conventional privacy assumptions, including in federated settings. It also advances work on secure learning pipelines and defense evaluation by showing that differential privacy may fail against loss-landscape-level attacks, motivating new analysis tools and mitigations for privacy leakage in code, text, and multimodal models used in software engineering workflows.

## 📄 [A Red-Team Study of Anthropic Fable 5 & Opus 4.8 Models](https://arxiv.org/abs/2606.18193v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Nicola Franco
📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.CR, cs.AI, cs.CL

### 💡 TL;DR
This paper systematically red-teams Anthropic’s Fable 5 and Opus 4.8 using automated jailbreak agents over 7,826 harmful intents and finds that, while both models block most attacks, they remain consistently vulnerable to adaptive iterative attacks. The strongest attack (tree-of-attacks) achieves 11.5% success on Opus 4.8 and 6.1% on Fable 5, yielding 1,620 and 702 panel-confirmed harmful completions respectively across all harm categories.

### 🏆 Key Contributions
- Provides a large-scale adversarial robustness evaluation of two frontier LLMs across 7,826 harmful intents spanning 10 harm categories and 55 subcategories
- Compares four jailbreak attack families—TAP, PAIR, PAP, and h4rm3l—showing that adaptive iterative attacks dominate the residual jailbreak surface while static obfuscation is largely ineffective

### ⚙️ Methodology
SwiftTrans is a two-stage framework. In the first stage, Multi-Perspective Exploration is performed by MpTranslator. ## 📄 [Bifrost: Hybrid TEE-FHE Inference for Privacy-Preserving Transformer and LLM Serving](https://arxiv.org/abs/2606.17421v1)  `70% relevant`  `🌐 Full Text`

👤 **Authors:** Chenghao Chen, Kailun Qin, Xiaolin Zhang, Chi Zhang, Dawu Gu
📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.CR

### 💡 TL;DR
Bifrost is a hybrid confidential LLM serving architecture that trusts only an attested CPU TEE while using FHE to safely delegate selected linear layers to an untrusted accelerator. By placing non-linear, control-heavy, KV-cache, and refresh operations inside the TEE and offloading only FHE-friendly linear algebra, it substantially lowers projected and measured inference latency versus all-FHE baselines, with Bifrost+ further accelerating time-to-first-token through a prefill/decode split.

### 🏆 Key Contributions
- Introduces a CPU-TEE-rooted, ciphertext-only accelerator delegation architecture where the accelerator, its memory, drivers, and host software remain outside the trusted computing base.
- Proposes an operator-affinity partitioning strategy that executes projection and feed-forward linear layers with CKKS on the accelerator while keeping normalization, RoPE, Softmax/reductions, gating, attention control logic, KV-state transitions, and decrypt-then-encrypt refresh inside the CPU TEE.
- Adds Bifrost+, a phase-aware prefill/decode split that performs prompt prefill and prompt-side KV construction entirely inside the TEE and sends only decode-side state through the hybrid encrypted path.
- Implements the design in an executable nano-vLLM-derived serving runtime with request routing, policy control, encoded-weight caching, and KV-cache support, enabling end-to-end evaluation rather than isolated kernel benchmarking.
- Shows that the dominant practical scaling bottleneck is not only homomorphic arithmetic but also encoded-weight residency and accelerator memory pressure, especially for larger models.

### ⚙️ Methodology
Bifrost is designed around a specific trust model: the client provisions secrets only after attesting a CPU TEE, and the TEE becomes the sole trusted control point. The accelerator is explicitly untrusted, so all data sent to it must remain encrypted. To support this, Bifrost uses CKKS-based FHE as a secure delegation mechanism rather than as the end-to-end execution substrate for the whole model.

The core methodological idea is to align execution placement with operator characteristics instead of with conventional layer boundaries. Regular, high-dimensional linear operators such as projection matrices and feed-forward network linear layers are assigned to the accelerator-backed CKKS path because they are stable, data-parallel, and map well to encrypted linear algebra. In contrast, operators that are difficult or inefficient under FHE—non-linear functions, reduction-heavy computations, Softmax-style attention logic, normalization, RoPE handling, gating, KV-cache state transitions, and ciphertext refresh—are executed natively inside the CPU TEE in plaintext.

This partition reduces encrypted depth from the combined linear-plus-nonlinear depth of pure-FHE transformer execution to a linear-only encrypted path. Instead of using costly homomorphic approximations and deep refresh-sensitive circuits for all components, Bifrost opens controlled decrypt-compute-encrypt or decrypt-then-encrypt windows inside the TEE. In effect, the TEE absorbs the parts of the transformer that are precision-sensitive, branch-heavy, or stateful, while the accelerator performs ciphertext-only linear computation.

The system also incorporates autoregressive serving semantics. Bifrost handles KV-cache updates and attention-side control inside the TEE, which is important because persistent state and phase transitions are awkward in pure FHE settings. Bifrost+ further refines the design with a serving-phase split: prompt prefill and prompt-side KV construction happen fully in the CPU TEE, and only decode-side state enters the hybrid TEE-FHE path. This avoids paying encrypted accelerator costs for all prompt tokens before first-token generation. The implementation is integrated into a nano-vLLM-based runtime that supports end-to-end request routing, policy selection, encoded-weight caching, and KV backends.

### 🧪 Experiments
The evaluation includes both estimator-style comparisons and direct CKKS/FHE deployment measurements. For large-model comparisons, the paper follows an estimator methodology aligned with Euston to project end-to-end latency under the proposed hybrid execution strategy. Using this setup, Bifrost achieves projected latency reductions of 9.25x for GPT-2 1.5B and 9.91x for LLaMA 3 8B relative to the fully encrypted baseline setting it compares against.

The authors also report direct deployment results on smaller but realistic CKKS/FHE workloads using their implemented system. In these experiments, Bifrost+ is evaluated on GPT-2 124M and Qwen3 0.6B, with prompt lengths of 16 and 64, focusing on time-to-first-token (TTFT). Bifrost+ reduces TTFT by 14.6x to 45.8x on GPT-2 124M and by 15.3x to 53.4x on Qwen3 0.6B. These gains come primarily from avoiding prompt-side encrypted execution through the prefill/decode split rather than from making individual FHE kernels intrinsically faster.

The evaluation is performed within an executable serving substrate derived from nano-vLLM rather than through isolated microbenchmarks alone, so it includes system effects such as request routing, encoded-weight caching, and KV-cache handling. Cost breakdowns reveal that fully cached GPT-2 decode remains dominated by GEMM-style work, while for Qwen3 the system incurs a 34.4% fallback weight-encoding penalty when encoded weights can no longer fully reside on the accelerator. This highlights memory residency and encoded-weight management as key determinants of practical performance.

### 🌱 Insights & Implications
The paper’s main systems insight is selective encrypted execution: FHE should be used only where ciphertext-only delegation to an untrusted accelerator is truly required, not as a blanket mechanism for all transformer operations. This is especially important for modern LLMs whose architectures contain increasingly varied non-linearities, control logic, cache interactions, and model-family-specific behaviors that are fragile or expensive to express as fixed homomorphic circuits.

A second important insight is that secure LLM serving is phase-structured. Prefill and decode have very different performance characteristics, and treating them identically leaves major latency on the table. Bifrost+ shows that phase-aware boundary placement can deliver large TTFT improvements without changing the underlying cryptography.

The work also identifies a practical limitation often underemphasized in FHE-for-LLM discussions: accelerator memory pressure and encoded-weight residency can become the dominant bottleneck once arithmetic kernels improve. The reported fallback penalty for larger models suggests that future research should consider weight encoding layout, caching policy, and accelerator memory management as first-class optimization targets.

Limitations include the fact that the paper’s largest-model results are estimator-based rather than fully deployed, and the approach still assumes trust in the CPU TEE and its side-channel/security properties. The hybrid design also does not make FHE universally cheap; rather, it narrows its use to the most accelerator-appropriate components. Open questions include how the partitioning strategy generalizes across rapidly evolving LLM architectures, how multi-tenant scheduling interacts with TEE memory pressure, and whether similar hybridization can support stronger throughput and batching guarantees at production scale.

### 🎯 Why This Paper
This paper is highly relevant to research on software engineering for secure AI systems, privacy-preserving inference, and vulnerability-sensitive cloud deployment because it offers a concrete architecture for minimizing the trusted computing base while still exploiting accelerators. For a researcher interested in software security and repair-oriented systems, Bifrost is valuable because it formalizes a principled boundary decomposition—what runs in trusted software, what is delegated cryptographically, and how state transitions are managed—that could inform secure serving frameworks, verification of confidential execution pipelines, and automated partitioning or repair of privacy-critical ML software stacks.

📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.PL

### 💡 TL;DR
This paper presents a defense-in-depth runtime safety layer for the Move VM on Aptos that re-checks type, ability, and reference-safety invariants during execution to protect against static bytecode verifier bugs and malicious hand-crafted bytecode. The main result is that these checks provide protection against real exploit classes found in practice, while their overhead can be reduced through trusted-code exemptions and asynchronous checking integrated with Aptos’s parallel execution engine.

### 🏆 Key Contributions
- A systematic design of runtime safety checks for Move that independently enforce type safety, ability safety, and reference safety during execution.
- A runtime type-shadowing mechanism that detects type confusion by tracking and validating the dynamic type of operand-stack values against statically expected bytecode types.
- Instruction-level runtime enforcement of Move abilities (copy, drop, key, store), including path-precise drop checking on actual executions.
- A novel lockstep shadow-semantics approach for runtime reference safety that tracks abstract places via access-path trees, supports temporary aliasing patterns permitted by Move, and detects invalid reference use via poisoning and lazy failure.
- A compositional interprocedural checking scheme for references that validates disjointness at call sites while remaining independent of concrete runtime addresses.
- Performance-mitigation techniques including trusted-code regions that waive checks under controlled conditions and an asynchronous checking mode that exploits idle Block-STM threads.
- An evaluation on real-world Aptos benchmarks showing that practical runtime defense-in-depth is feasible for a production blockchain VM.

### ⚙️ Methodology
The paper augments the Move VM with runtime verification mechanisms that execute alongside normal bytecode interpretation. For type safety, the VM maintains a shadow type stack aligned with the operand stack. Since locals are already typed, every push operation also pushes the value’s type into the shadow stack; every pop operation validates the observed type against the instruction’s or destination local’s expected type, aborting on mismatch. For ability safety, the VM performs dynamic checks whenever operations exercise one of Move’s four abilities: copy, drop, key, or store. This includes ensuring that only values whose declared abilities permit duplication are copied, only droppable values are discarded, only key resources enter global storage, and only storable values are nested into resources. Drop checking is path-sensitive at runtime because it examines only the actual execution path taken, rather than all statically possible control-flow paths. For reference safety, the paper introduces a shadow semantics that mirrors the VM’s operand stack, locals, and call frames. Concrete values are abstracted away; non-reference values are opaque, while each newly created reference is represented by a fresh identifier associated with an abstract place in an access-path tree. Trees are rooted at locals, borrowed globals, and values behind reference parameters, and edges correspond to field offsets; importantly, fields of different enum variants sharing a physical offset are unified so that overlap is modeled soundly. Aliasing checks are thus reduced to ancestor/descendant relationships between abstract places rather than concrete pointer identity. When destructive actions occur, such as writes through mutable references or moves invalidating locals, all overlapping references are poisoned. Errors are deferred until a poisoned reference is actually used, enabling legal temporary aliasing patterns that arise during stack-machine execution. The design is compositional across calls: the caller checks disjointness conditions among reference arguments, then the callee receives fresh abstract trees for its own frame. To lower overhead, the system supports trusted-code exemptions for code considered sufficiently safe and an asynchronous execution mode where checking work is offloaded to idle threads in Aptos’s Block-STM parallel execution engine.

### 🧪 Experiments
The evaluation uses real-world Aptos benchmarks to measure the cost of enabling runtime safety checks in a production-style blockchain VM. The experiments assess the overhead of the three categories of checks—type, ability, and reference checks—both in their direct form and with the proposed optimizations. Baselines include normal execution without these defense-in-depth checks, and optimized variants include trusted-code waivers and asynchronous checking using spare Block-STM worker capacity. Metrics focus on runtime/performance overhead on transaction execution workloads representative of Aptos. The paper’s key quantitative conclusion is that naive full checking incurs measurable overhead, but the optimized configurations substantially reduce this cost, making the approach practical for deployment while still covering exploit classes caused by verifier bugs or malicious bytecode. The abstracted text provided here does not include the detailed benchmark table values, but the reported outcome is that the checks are feasible on real workloads and benefit significantly from the two optimization techniques.

### 🌱 Insights & Implications
A central insight is that static bytecode verification, even when sophisticated, is an unsafe single point of failure in a permissionless blockchain where adversaries can submit hand-crafted bytecode and executed transactions are irreversible. The paper demonstrates that runtime revalidation of critical invariants can catch exploit classes stemming from verifier omissions, including type confusion, illicit duplication of linear resources, and enum/reference aliasing bugs. Another important insight is that runtime reference safety for Move cannot be implemented with simplistic pointer- or refcount-based schemes because Move’s execution permits temporary overlapping references during valid borrow construction; the shadow-semantics design addresses this by checking actual use rather than mere existence of overlap. The poisoning-plus-lazy-failure strategy is a practical compromise that preserves legal execution patterns while still detecting unsound behaviors before they can cause damage. Limitations include the extra engineering complexity of maintaining a shadow semantics in a production VM and the runtime cost that necessitates selective trust or asynchronous execution. Open questions include how broadly these techniques generalize to other bytecode VMs with richer aliasing models, how trusted-code exemptions should be justified or audited, and whether hardware/software co-design or static-dynamic hybrid verification could further reduce overhead.

### 🎯 Why This Paper
This paper is highly relevant to software security, vulnerability detection, and vulnerability repair because it addresses how to harden a production execution platform against latent soundness bugs in its own verifier—exactly the kind of implementation flaw that can become a critical vulnerability. For researchers in software engineering and automated repair, it offers a concrete defense-in-depth architecture, real bug-driven threat models, and a novel runtime checking design for alias/reference safety that could inspire verification, monitoring, and self-protecting runtime techniques for secure program execution environments.

## 📄 [Planning to Hammer: Difficulty-Aware Decomposition for Automating Rocq Proofs](https://arxiv.org/abs/2606.17981v1)  `60% relevant`  `🌐 Full Text`

📅 **Published:** 2026-06-16  🏷️ **Categories:** cs.SE

### 💡 TL;DR
Quarry is a proof-synthesis framework for Rocq that combines LLM-generated high-level proof decompositions with CoqHammer-based local proof discharge, using a difficulty model to rank candidate decompositions by estimated solvability. By recursively breaking hard theorems into hammer-solvable sublemmas under a bounded budget, it improves proof automation success rates by 7%–13% over the strongest baseline on three Rocq benchmarks within a 10-minute budget.

### 🏆 Key Contributions
- Frames Rocq proof synthesis as a planning-then-execution problem where LLMs generate arbitrary sublemma-based decompositions and symbolic automation executes them.
- Introduces Quarry, a Generate–Rank–Solve framework that validates LLM-proposed decompositions in Rocq, ranks them with a proof-state-based difficulty model, and recursively solves sublemmas using CoqHammer.
- Develops a lightweight solvability proxy based on proof-state features to avoid the prohibitive cost of running hammer on every candidate decomposition during ranking.
- Implements the approach on top of SerAPI and CoqHammer and evaluates it with multiple frontier LLMs across three Rocq benchmarks, showing consistent gains under a fixed wall-clock budget.
- Demonstrates that neural planning and symbolic proof automation are complementary: LLMs contribute long-range structure while hammer tactics provide reliable local correctness.

### ⚙️ Methodology
Quarry separates proof generation into planning and execution. For a target theorem, an LLM is prompted to propose multiple candidate decompositions consisting of (1) a proof script for the original goal that may invoke arbitrary intermediate sublemmas and (2) the statements of those sublemmas. Quarry then checks each candidate in Rocq through SerAPI by temporarily admitting the proposed sublemmas and verifying that the script type-checks and indeed reduces the original theorem to those obligations. Since many valid decompositions are still poor choices if their sublemmas are too hard, Quarry ranks candidates with a difficulty-aware model that estimates whether the resulting subgoals are likely solvable by CoqHammer without actually running the hammer during ranking. The model uses proof-state features derived from the goal and context, including indicators of logical and definitional complexity, motivated by the observation that hammer success correlates with such features in program-verification proofs. After ranking, Quarry tries candidates in order within a bounded resource budget. For each sublemma, it first invokes CoqHammer as the primary executor. If hammer succeeds, the subgoal is closed; if not, Quarry recursively applies the same Generate–Rank–Solve procedure to decompose that sublemma further. This yields a proof tree whose leaves are intended to be directly hammer-solvable obligations. The overall search is budgeted to keep runtime predictable, and SerAPI is used for tactic execution, proof-state extraction, validation, and recovery from failures such as invalid scripts or timeouts.

### 🧪 Experiments
The evaluation is conducted on three Rocq benchmarks spanning diverse developments, including graph-theory verification and Rust-to-Rocq translations, and uses multiple frontier LLM backends as planners. Quarry is implemented over SerAPI and CoqHammer, and all methods are compared under a uniform 10-minute wall-clock budget per theorem to emphasize practical automation cost. The main comparison is against strong baselines representing direct LLM proof generation and/or hammer-centered automation without Quarry’s planning-based decomposition and difficulty-aware ranking. Reported results show that Quarry proves 55%, 52%, and 16% of theorems on the three benchmarks respectively, and improves over the strongest baseline by 7 to 13 percentage points in success rate across all three benchmarks. The paper also highlights a motivating example, max_deg_remove_node from coq-wigderson, where neither CoqHammer alone nor frontier LLMs alone succeed, while Quarry decomposes the proof into 12 subgoals and solves it automatically. The gains are attributed specifically to planning-based decomposition combined with solvability-aware candidate ranking, rather than decomposition alone.

### 🌱 Insights & Implications
The paper’s central insight is that proof automation bottlenecks in Rocq are often not purely local-search failures but failures of proof structuring: long proofs require discovering the right intermediate claims before strong local automation can be effective. Quarry exploits this by treating LLMs as planners instead of end-to-end proof generators and using symbolic tools only where they are strongest. A particularly useful observation for future work is that the usefulness of a decomposition depends on its hardest sublemma, making ranking critical; naive decomposition generation without solvability estimation would waste budget on elegant but infeasible proof plans. The work also suggests a broader design pattern for theorem proving and formal methods: use neural models for abstraction and decomposition, and sound proof engines for checking and local search. Limitations include dependence on the quality and diversity of LLM-generated decompositions, the bounded capabilities of CoqHammer on goals requiring induction or deep reasoning even after decomposition, and the possibility that proof-state features used for difficulty estimation may not generalize equally well across domains or provers. Open questions include learning stronger cross-project difficulty predictors, integrating richer executors beyond hammer tactics, and extending the framework to other ITPs such as Isabelle or to larger software-verification developments.

This paper is highly relevant to software engineering and formal verification research because it targets a core bottleneck in proving program correctness in Rocq, a key technology for trustworthy software and security-critical systems. ## 📄 [OTRO: Oblivious Tokenization Path with Square-Root ORAM](https://arxiv.org/abs/2606.17358v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Jonghyun Lee, Yongqin Wang, Rachit Rajat, Daniel Wong, Mengyuan Li, Murali Annavaram
📅 **Published:** 2026-06-15  🏷️ **Categories:** cs.CR

### 💡 TL;DR
OTRO is a practical defense against tokenizer-side memory access leakage in confidential LLM serving: it adapts square-root ORAM to read-only tokenizer tables using replicated instances, epoch rotation, and chunked tokenization so that oblivious protection adds only small end-to-end latency. In a TDX+H100 serving stack, OTRO reduces leakage to essentially prompt length while keeping TTFT overhead to at most 4.5% and memory overhead below 0.5 GB, far outperforming naive PathORAM and square-root ORAM baselines.

### 🏆 Key Contributions
- Shows that the CPU-side BPE tokenizer is a major side-channel in confidential LLM serving and that straightforward ORAM defenses are impractical for latency-sensitive inference.
- Introduces OTRO, an oblivious tokenization architecture built on replicated read-only square-root ORAM instances to exploit the static tokenizer table during inference.
- Presents an epoch-based rotation and padding policy that decouples expensive square-root ORAM rebuilds from online accesses and reduces observable leakage beyond raw access traces.
- Integrates chunked, KV-cache-aware tokenization with incremental GPU prefill to overlap ORAM rebuilds with model execution and minimize the number of ORAM replicas needed.
- Implements OTRO in HuggingFace Tokenizers and nano-vLLM on Intel TDX with NVIDIA H100 confidential serving, demonstrating low TTFT and memory overhead.
- Provides a leakage argument that OTRO hides which vocabulary/merge entries are accessed, with prompt length remaining the main unavoidable leakage channel.

### ⚙️ Methodology
The paper targets BPE-style tokenizers used in decoder-only LLMs, where vocabulary and merge-table lookups induce prompt-dependent memory access patterns observable to a malicious hypervisor or similar DRAM-trace adversary despite TEE protections. OTRO is designed around the observation that tokenizer tables are read-only during inference, unlike general mutable ORAM workloads. Instead of using tree-based ORAM such as PathORAM, OTRO uses square-root ORAM because its single lookup cost is much lower and better matched to the tokenizer’s latency-critical random accesses. The challenge is that classic square-root ORAM requires an expensive O(N log^2 N) rebuild every sqrt(N) accesses, which is prohibitive if performed synchronously. OTRO addresses this with three mechanisms. First, it pre-constructs a pool of replicated square-root ORAM instances over the same tokenizer tables, leveraging read-only semantics so replicas need not reflect updates. Second, it defines epochs of sqrt(N) accesses per instance: once an instance reaches its epoch limit, OTRO rotates to another fresh instance while the exhausted one is rebuilt offline. To avoid leaking the exact number of accesses within an epoch, OTRO pads each epoch to its boundary using dummy accesses, making traces independent of which specific tokenizer entries were used and reducing observability to coarser information. Third, OTRO integrates chunked tokenization with KV-cache-aware incremental prefill: the prompt is processed in chunks so that CPU-side oblivious tokenization and background ORAM rebuild can overlap with GPU prefill and cache reuse, keeping rebuilds off the critical path and reducing the pool size required for smooth serving. The system is implemented as modules inside HuggingFace Tokenizers and nano-vLLM within a confidential CPU+GPU serving stack.

### 🧪 Experiments
The evaluation is conducted in a realistic confidential serving environment using HuggingFace Tokenizers and nano-vLLM inside an Intel TDX-enabled confidential VM paired with an NVIDIA H100 GPU. The paper compares OTRO against an unprotected tokenizer baseline, a drop-in PathORAM-based tokenizer, naive square-root ORAM, and GPU-offloaded tokenization. At the microbenchmark level, square-root ORAM single-block lookup averages 3.2 microseconds and is about 10x faster than PathORAM; when rebuilds are excluded, square-root ORAM incurs only about 2.4x overhead over the unprotected baseline, identifying rebuild cost as the dominant bottleneck. End-to-end tokenizer integration shows that PathORAM and naive square-root ORAM are impractical, causing about 13x and 323x tokenizer slowdowns respectively, while GPU offloading suffers roughly 276x slowdown due to BPE control-flow divergence. In full LLM serving, naive PathORAM raises TTFT by 10% to 58%, whereas OTRO limits TTFT overhead to at most 4.5%, keeps tokenizer-induced latency under 10% of total TTFT, and adds less than 0.5 GB of extra memory. Results are reported across multiple model families and sizes, showing that OTRO materially reduces tokenizer leakage while maintaining near-baseline serving latency.

### 🌱 Insights & Implications
A key insight is that general-purpose oblivious memory techniques are too costly unless the tokenizer workload’s structure is exploited. The read-only nature of vocabulary and merge tables is what makes replicated square-root ORAM viable, turning rebuild cost from a synchronous latency spike into background maintenance. The work also shows that end-to-end systems co-design matters: tokenizer protection cannot be evaluated in isolation because overlapping CPU tokenization, ORAM rebuilds, and GPU prefill is essential to making obliviousness practical for TTFT-sensitive serving. For security, OTRO substantially narrows leakage compared with existing tokenizer traces, but prompt length remains visible and is explicitly treated as unavoidable in this ORAM setting; thus OTRO does not eliminate all side channels. The approach is tailored to BPE-style tokenizers and a DRAM-trace style adversary model, so questions remain about extension to other tokenization schemes, stronger microarchitectural adversaries, multi-tenant contention effects, and deployment tradeoffs at larger serving scales. Another limitation is added memory from ORAM replication, though the reported overhead stays modest. The paper implies a broader research direction: security mechanisms for confidential LLM stacks should be specialized to the data structure and pipeline stage, rather than imported unchanged from generic oblivious storage literature.

### 🎯 Why This Paper
This paper is highly relevant to software security and secure systems for LLM infrastructure because it addresses a concrete, demonstrated side channel that leaks raw prompts even under CPU/GPU TEEs. For a researcher interested in vulnerability detection and repair, OTRO is valuable as a case study in identifying a real confidentiality weakness in production software stacks, characterizing why standard defenses fail, and engineering a practical mitigation with formal leakage reduction and strong end-to-end performance. It also connects directly to secure software maintenance by modifying widely used components like HuggingFace Tokenizers and vLLM-style serving systems, showing how to retrofit protections into existing ML software without unacceptable latency costs.

## 📄 [LineageMark: Multi-user White-box Watermarking for Contribution Tracing in Model Derivation Chains](https://arxiv.org/abs/2606.17123v1)  `60% relevant`  `🌐 Full Text`

👤 **Authors:** Bingxue Zhang, Xiaofeng Xu, Feida Zhu
📅 **Published:** 2026-06-15  🏷️ **Categories:** cs.CR, cs.AI

### 💡 TL;DR
LineageMark is a white-box watermarking framework designed for open LLM derivation chains, where multiple contributors sequentially fine-tune and watermark the same model. It encodes each contributor’s watermark as projection-based statistics over stable parameter carriers, enabling historical watermark preservation, incremental multi-user insertion, and key-only independent verification under continued derivation and common model modifications.

### 🏆 Key Contributions
- Formalizes multi-user white-box watermarking for model derivation chains with three core requirements: lineage preservation, incremental embedding, and independent verification using only a contributor’s private key.
- Proposes LineageMark, which represents each watermark bit as a margin-constrained sign of a projection over groups of selected model parameters rather than relying on individual coordinates.
- Introduces stable carrier selection to choose parameter regions less sensitive to later model updates, reducing historical watermark drift across derivation stages.
- Designs a key-driven embedding and extraction mechanism so each contributor can independently insert and verify their own watermark without the original model, calibration data, other contributors’ keys, or derivation history.
- Demonstrates robustness of the approach against repeated derivation and post hoc perturbations including re-watermarking, additional fine-tuning, quantization, and pruning, outperforming representative white-box watermarking baselines in preserving historical watermarks.

### ⚙️ Methodology
LineageMark targets sequential model derivation where contributor u_t receives an already watermarked model, fine-tunes it on stage-specific data, and then embeds a new watermark using a private key k_t. The central design choice is to encode each watermark bit not in a single parameter or local weight pattern, but as a projection statistic computed over a group of weight coordinates. For each contributor, the private key deterministically generates coordinate groups and projection directions. A bit is embedded by adjusting the selected parameters so that the sign of the projection matches the target bit, while maintaining a safety margin from the decision boundary. This margin-aware embedding makes later parameter drift less likely to flip the embedded bit. To improve persistence under subsequent derivation, the framework first identifies stable carriers: parameter regions judged less sensitive to downstream updates and more perturbation-stable. Redundant carriers are used so that each bit is supported by multiple coordinates, reducing sensitivity to local changes. When later users insert additional watermarks, the induced perturbation in existing contributors’ projection spaces is bounded, limiting cross-user interference. Extraction reconstructs the same carrier groups and projection detector from the claimant’s private key alone, computes projection signs from the suspicious model, and recovers the watermark without access to the original base model, prior watermarked checkpoints, or other users’ metadata. The framework is therefore explicitly designed for coexistence of multiple independently verifiable watermarks in the same downstream model.

### 🧪 Experiments
The paper evaluates LineageMark on multi-stage derivation chains built through continuous full-parameter fine-tuning across models of different sizes and multiple domain datasets. The experiments test three main properties: preservation of historical watermarks after multiple derivation stages, ability of downstream users to incrementally add new watermarks without damaging previous ones, and independent verification by each contributor using only their own key. LineageMark is compared with representative white-box watermarking baselines that were originally designed for single-owner settings. The reported metrics center on watermark extraction or verification accuracy across derivation stages and under perturbations, while also checking that model utility is preserved. The results show that LineageMark consistently maintains high extraction accuracy for both early and newly inserted watermarks throughout multi-stage derivation, with lower historical watermark decay and less interference among users than baselines. Additional robustness tests under re-watermarking, further fine-tuning, quantization, and pruning show that the embedded contributor identities remain reliably recoverable, indicating stronger persistence than prior methods in realistic post-deployment modification scenarios.

### 🌱 Insights & Implications
The paper highlights that watermarking for LLM provenance is fundamentally different from standard single-owner ownership verification: in open ecosystems, the challenge is not only survivability under fine-tuning but also coexistence of many independently inserted signals over time. The projection-based encoding suggests a useful robustness principle for white-box security mechanisms: aggregate statistics over stable parameter subsets are more resilient than fragile pointwise encodings. For software and security researchers, this frames model provenance as a contribution-tracing problem under repeated transformation, analogous to maintaining evidence across software evolution. A limitation is that the work focuses on verifying which contributor watermarks remain in a final model rather than reconstructing the full derivation graph or insertion order. The evaluation is centered on full-parameter fine-tuning chains, so it leaves open how the method behaves under more diverse derivation operations such as adapter-based tuning, model merging, distillation, or aggressive editing attacks. Another open question is adversarial removal: although robustness to common perturbations is shown, stronger adaptive attackers may target the projection carriers once they know the watermarking family. The work therefore opens directions in provenance-preserving model evolution, interference-aware multi-party watermarking, and security analysis of watermark removal in collaborative model ecosystems.

### 🎯 Why This Paper
This paper is highly relevant to research on software security, provenance, and vulnerability-related model governance because it addresses how to preserve verifiable evidence of contributions across repeated model evolution. For work in automated repair, maintenance, and secure ML pipelines, LineageMark offers a concrete mechanism for tracing responsibility and intellectual property through iterative adaptation, which is analogous to tracking authorship and change provenance across long software derivation chains. It advances the researcher’s interests by formalizing a multi-party, evolution-aware watermarking problem and proposing a robust white-box solution that remains effective under realistic downstream modifications.
